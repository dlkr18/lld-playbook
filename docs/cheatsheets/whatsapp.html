01 — Requirements

Functional

  • User management — register, update profile/status, track online/offline, block/unblock
  • One-on-one messaging — text, media attachments, message history, delete
  • Group chats — create groups, add/remove participants, admin privileges, leave group
  • Message status — Sent (✓) → Delivered (✓✓) → Read (✓✓)
  • Real-time features — typing indicators, online/offline status, last seen
  • Message features — reply, forward, star/favorite, search messages

Non-Functional

  • Performance — O(1) user lookup, fast message delivery, efficient retrieval
  • Scalability — support millions of users, large group sizes, efficient message history
  • Reliability — guaranteed delivery, message ordering, idempotency
  • Security — message privacy, user authentication, block functionality

03 — Design Patterns

Five patterns working together — each earns its place.

Composite Pattern

Chat interface treats DirectChat and GroupChat uniformly. Service code calls chat.addMessage() without caring if it's 1:1 or group. DirectChat auto-delivers; GroupChat fans out to all participants.

Observer Pattern

MessageObserver interface for real-time notifications. Users observe message received/delivered/read events. Decouples delivery from business logic.

State Pattern

Message status progression: SENT → DELIVERED → READ. canTransitionTo() enforces legal transitions. DELETED is a terminal state from any non-deleted state.

Strategy Pattern

Message delivery varies by chat type. DirectChat delivers to the single recipient. GroupChat fans out to all participants except sender. Same addMessage() interface, different delivery logic.

Builder Pattern

Message.Builder for flexible construction — handles text, media, replies, forwards. Auto-infers message type from content. Clean API: new Builder(senderId, content).replyTo(id).build()

Factory Pattern

MessageContent auto-infers MessageType from attachment MIME type: image/* → IMAGE, video/* → VIDEO, audio/* → AUDIO, else DOCUMENT. No text attachment = TEXT.

04 — Message State Machine

Why track status per-user? In group chats, a message is DELIVERED when any recipient receives it, and READ when any recipient reads it. The deliveredTo and readBy maps track per-user granularity.

MessageStatus Enum with Transition Guard

public enum MessageStatus {
    SENT("Sent", "✓"),
    DELIVERED("Delivered", "✓✓"),
    READ("Read", "✓✓"),       // blue ticks
    DELETED("Deleted", "");

    public boolean canTransitionTo(MessageStatus newStatus) {
        if (this == DELETED) return false;  // terminal
        if (newStatus == DELETED) return true; // always deletable
        return newStatus.ordinal() > this.ordinal();
    }
}

Per-User Delivery Tracking (Group Messages)

public class Message {
    private final Map<UserId, LocalDateTime> deliveredTo = new HashMap<>();
    private final Map<UserId, LocalDateTime> readBy = new HashMap<>();

    public void markDelivered(UserId userId) {
        deliveredTo.put(userId, LocalDateTime.now());
        updateOverallStatus();
    }

    public void markRead(UserId userId) {
        readBy.put(userId, LocalDateTime.now());
        deliveredTo.putIfAbsent(userId, LocalDateTime.now());
        updateOverallStatus();
    }

    private void updateOverallStatus() {
        if (!readBy.isEmpty()) status = MessageStatus.READ;
        else if (!deliveredTo.isEmpty()) status = MessageStatus.DELIVERED;
    }
}

DirectChat — Auto-Deliver on Send

public class DirectChat implements Chat {
    public void addMessage(Message message) {
        messages.add(message);
        UserId recipient = getOtherUser(message.getSenderId());
        message.markDelivered(recipient); // auto-deliver to 1 recipient
    }
}

GroupChat — Fan-Out Delivery

public class GroupChat implements Chat {
    private final Map<UserId, Participant> participants;

    public void addMessage(Message message) {
        messages.add(message);
        // Fan-out: mark delivered to ALL participants except sender
        for (UserId userId : participants.keySet()) {
            if (!userId.equals(message.getSenderId())) {
                message.markDelivered(userId);
            }
        }
    }
}

07 — Key Algorithms

Message Builder (Flexible Construction)

public class Message {
    public static class Builder {
        private final MessageId id = MessageId.generate();
        private final UserId senderId;
        private final MessageContent content;
        private MessageType type;
        private MessageId repliedToId, forwardedFromId;

        public Builder(UserId senderId, MessageContent content) {
            this.senderId = senderId;
            this.content = content;
            this.type = content.inferType(); // auto from MIME
        }

        public Builder replyTo(MessageId id)      { repliedToId = id; return this; }
        public Builder forwardFrom(MessageId id) { forwardedFromId = id; return this; }
        public Message build() { return new Message(this); }
    }
}

MessageContent — Type Inference (Factory)

public final class MessageContent {
    private final String text;
    private final Attachment attachment;

    public MessageType inferType() {
        if (!hasAttachment()) return MessageType.TEXT;
        String ft = attachment.getFileType().toLowerCase();
        if (ft.startsWith("image/")) return MessageType.IMAGE;
        if (ft.startsWith("video/")) return MessageType.VIDEO;
        if (ft.startsWith("audio/")) return MessageType.AUDIO;
        return MessageType.DOCUMENT;
    }
}

Unread Count Calculation

// DirectChat: count messages from other user after lastRead
public int getUnreadCount(UserId userId) {
    LocalDateTime lastRead = lastReadAt.getOrDefault(userId, createdAt);
    return (int) messages.stream()
        .filter(msg -> !msg.getSenderId().equals(userId))
        .filter(msg -> msg.getTimestamp().isAfter(lastRead))
        .filter(msg -> !msg.isDeleted())
        .count();
}

// GroupChat: uses Participant.lastReadAt for per-user tracking

Message Search

public List<Message> searchMessages(ChatId chatId, String query) {
    Chat chat = getChatOrThrow(chatId);
    String lowerQuery = query.toLowerCase();

    return chat.getMessages().stream()
        .filter(msg -> !msg.isDeleted())
        .filter(msg -> msg.getContent().getText()
            .map(text -> text.toLowerCase().contains(lowerQuery))
            .orElse(false))
        .collect(Collectors.toList());
}

PhoneNumber Validation (Value Object)

public final class PhoneNumber {
    private static final Pattern PHONE_PATTERN =
        Pattern.compile("^\\+?\\d{8,15}$");

    private PhoneNumber(String number) {
        String normalized = number.replaceAll("[\\s-]", "");
        if (!PHONE_PATTERN.matcher(normalized).matches())
            throw new IllegalArgumentException("Invalid phone");
        this.number = normalized;
    }
}

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap All data stores: chats, groups, userChats, messages, starredMessages, typingUsers Safe concurrent reads/writes without global lock
ConcurrentHashMap.newKeySet() userChats values, starredMessages values, typingUsers values Thread-safe Set backed by ConcurrentHashMap
computeIfAbsent userChats, starredMessages, typingUsers Atomic insert-if-missing for index maps
ReadWriteLock Chat service (alternative approach) Multiple readers concurrent; writers exclusive

ReadWriteLock Approach

public class ConcurrentChatService implements ChatService {
    private final ReadWriteLock lock = new ReentrantReadWriteLock();

    public MessageId sendMessage(ChatId chatId, UserId senderId,
                                  MessageContent content) {
        lock.writeLock().lock();
        try {
            // send message logic
            return messageId;
        } finally { lock.writeLock().unlock(); }
    }

    public List<Message> getMessages(ChatId chatId) {
        lock.readLock().lock();
        try {
            return messages;
        } finally { lock.readLock().unlock(); }
    }
}
"The primary implementation uses ConcurrentHashMap for lock-free reads. For operations that span multiple maps (e.g. creating a chat and updating user index), computeIfAbsent provides atomic semantics per-key. At scale, I'd add fine-grained per-chat locks."

12 — Interview Follow-up Q&A

How would you implement end-to-end encryption?
Each user generates a public/private key pair on registration. The sender encrypts with the recipient's public key; only the recipient's private key can decrypt. For groups, use a shared group key rotated when members change. The MessageContent value object would store encrypted bytes. Decrypt happens at the client layer, not in the service.
How does read receipt work differently in groups vs 1:1?
The readBy map on Message tracks per-user read timestamps. In 1:1, a single read turns the message READ. In groups, the overall status updates when any recipient reads. To show "read by 3 of 5", query readBy.size() vs participants count. WhatsApp shows individual read times on long-press.
How would you handle message delivery for offline users?
Messages stay in SENT status. On reconnect, the server pushes all pending messages (status SENT for that user) and marks them DELIVERED. Use a persistent message queue (e.g. Kafka) per user. The deliveredTo map naturally supports this — missing users simply haven't been marked yet.
Why Composite pattern for Chat instead of just one class with a type field?
DirectChat has exactly 2 participants with symmetric logic (getOtherUser). GroupChat has N participants with admin roles, promotion, and fan-out delivery. Composite lets each implementation encapsulate its own logic. The service works through the Chat interface — chat.addMessage() does the right thing polymorphically. No switch/case on type.
How would you implement disappearing messages?
Add a ttl field on Message (Duration or epoch). A background ScheduledExecutorService periodically scans and calls message.delete() for expired messages. Or use a lazy approach: check TTL on read and delete if expired. The State pattern already supports DELETED as a terminal state.
How do you prevent duplicate direct chats between two users?
createDirectChat first calls findDirectChat(user1, user2) which scans user1's chat list for an existing DirectChat with user2. If found, returns the existing ChatId. This ensures idempotency — calling createDirectChat multiple times returns the same chat.
How would you scale message search?
Current implementation does linear scan with String.contains(). At scale: (1) Elasticsearch for full-text indexing with fuzzy matching, (2) inverted index per chat, (3) search only within a time window. The searchMessages() interface stays the same — swap the implementation behind it.
How would you handle large groups (1000+ members)?
Fan-out on write becomes expensive. Switch to fan-out on read: store message once, each user pulls from the group on access. Use a message queue for async delivery. Read receipts become aggregated ("read by 500+") instead of per-user. Limit admin operations with rate limiting.
Why Builder pattern for Message instead of constructors?
Message has many optional fields: repliedToId, forwardedFromId, explicit type override, custom timestamp. Telescoping constructors would need 8+ variants. Builder gives a fluent API: new Builder(sender, content).replyTo(id).build(). Only senderId and content are required; everything else has sensible defaults.
How would you implement typing indicators at scale?
Current: ConcurrentHashMap<ChatId, Set<UserId>> with set/remove. At scale: typing state is ephemeral — use Redis with TTL (auto-expire after 5s). Client sends periodic heartbeats. No persistence needed. Use WebSocket channels per-chat for real-time push.
What happens when the last admin leaves a group?
GroupChat.removeParticipant() checks if the removed user was admin and if no admins remain. If so, it auto-promotes the first remaining member to ADMIN. If the group becomes empty after leave, the service deletes it entirely from both the chats and groups maps.
How does the forward message work internally?
forwardMessage() reads the original message's content, creates a new Message with the forwarder as sender and the original messageId in forwardedFromId. The new message has its own ID, timestamp, and SENT status. message.isForwarded() returns true. The original is unmodified.
