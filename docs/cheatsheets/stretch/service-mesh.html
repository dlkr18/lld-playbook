Service Mesh // Stretch

Sidecar proxy layer that handles service-to-service networking — mTLS, retries, traffic shaping, observability — outside application code. Istio + Envoy is the canonical stack. Costs CPU + complexity; pays off only above a microservice count threshold.

Envoy Istio mTLS Sidecar Linkerd

Kubernetes cluster with Envoy sidecars in every pod; control plane (Istiod) pushes config via xDS; mTLS between every pod; features (mTLS, traffic split, circuit breaker, retry budget, tracing); cost table; ambient mesh alternative.

01 — Identity

A service mesh is a network-layer concern moved out of the application. A small proxy (sidecar) sits next to every workload and intercepts all traffic in and out; a control plane configures all proxies centrally. The proxies provide mTLS, retries, timeouts, traffic splitting, RBAC, and auto-emitted metrics & traces — all without app code changes.

The mesh is "L7 networking as code, deployed alongside the app instead of baked into it." Application teams gain reliability + observability + zero-trust for free; the platform team owns the mesh.

01b — API / Control Surface (Istio examples)

# Traffic split for canary
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata: { name: orders }
spec:
  hosts: [orders]
  http:
  - route:
    - destination: { host: orders, subset: stable }   # 90%
      weight: 90
    - destination: { host: orders, subset: canary }   # 10%
      weight: 10

# Outlier ejection + circuit breaker
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata: { name: orders }
spec:
  host: orders
  trafficPolicy:
    connectionPool:
      tcp: { maxConnections: 100 }
      http: { http2MaxRequests: 1000, maxRequestsPerConnection: 10 }
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50

# AuthZ: only billing can call orders.checkout
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata: { name: orders-checkout }
spec:
  selector: { matchLabels: { app: orders } }
  rules:
  - from: [ { source: { principals: ["cluster.local/ns/billing/sa/billing"] } } ]
    to:   [ { operation: { paths: ["/checkout"] } } ]

Most teams arrive at a mesh from the "we keep writing the same Hystrix/retry/auth code in 6 languages" pain. If you haven't felt that, you don't need it yet.

  ┌─────── POD A ────────┐  ┌─────── POD B ─────────┐
  │ app container        │  │ app container        │
  │  |  (localhost)      │  │  |  (localhost)       │
  │  v                   │  │  v                    │
  │ envoy sidecar  <------> envoy sidecar           │
  │  - mTLS              │  │  - mTLS               │
  │  - retries           │  │  - retries            │
  │  - circuit break     │  │  - circuit break      │
  └──────────────────────┘  └───────────────────────┘
              | iptables redirects pod traffic through sidecar

Sidecar is injected automatically (admission webhook). Apps see plaintext on localhost; everything between pods is mTLS-encrypted. App identity = ServiceAccount = SPIFFE URI.

04 — mTLS & Identity

  • Per-workload certs — each service account gets an X.509 cert via SPIFFE/SPIRE. Identity = spiffe://cluster.local/ns/billing/sa/billing-sa
  • Short TTL — certs rotate every 24h (often shorter); compromise window bounded
  • Strict / permissive / disabled mode per namespace (PeerAuthentication CRD)
  • Migration — permissive mode allows both plaintext + mTLS during rollout
  • End-user JWT propagation — mesh verifies JWT at ingress, passes principal to downstream policies
Result: zero-trust network with effectively no app code involvement. Every call is authenticated and encrypted; the application sees plaintext localhost as if it's 1999.

07 — Observability

  • Sidecar emits RED metrics per route + per source/dest pair (Prometheus, Datadog)
  • Auto-distributes trace context (B3 / W3C) and creates spans for in/out hops — app only needs to propagate headers
  • Access logs in JSON to stdout; usually shipped via Fluent Bit / Vector
  • Service graph auto-discovered (Kiali, Datadog Service Map)

Tying back to Observability: a mesh gives you the entire layer for free, regardless of language. Worth it if you have 5+ languages in production.

09 — Operational Concerns

  • CPU overhead: +0.5 vCPU per pod is typical for sidecar; 1k pods = 500 vCPU just for proxies.
  • Latency tax: 5-10 ms p99 added per hop; deep call chains feel it.
  • Upgrade path: control plane + each sidecar; needs careful rollout (sidecar restart = pod restart for app).
  • Debugging is harder — "is it the app, the sidecar, or the network?" gets asked daily.
  • Multi-cluster + multi-region — Istio supports federated mesh; complex setup.
  • Egress policy — restricting external calls helps security but makes incident debugging harder.
  • Init container vs CNI plugin — iptables setup. Istio offers Istio CNI to avoid privileged init.
  • Pod startup races — app starts before sidecar ready → first calls fail; needs holdApplicationUntilProxyStarts.

12 — Interview Q&A

What is a service mesh?
A network infrastructure layer for service-to-service communication. A small proxy (sidecar, typically Envoy) is injected next to every pod; a control plane configures all proxies centrally. The mesh handles mTLS, retries, timeouts, traffic splitting, RBAC, observability — without application code changes. Istio + Envoy is the canonical stack.
Why mTLS in the mesh instead of just JWTs?
mTLS proves the caller is a specific workload via cryptographic identity bound to a cert (SPIFFE URI), and encrypts at the channel level. JWTs prove the caller has a token but don't bind to network identity and require app-level handling. In practice you do both: mTLS at the network for workload identity, JWT in headers to carry end-user context.
What's the data plane vs control plane split?
Data plane = the proxies (Envoy) actually moving bytes; performance-critical, replicated everywhere. Control plane (Istiod) = configures the data plane via xDS gRPC; manages cert issuance, service discovery, policy compilation. Data plane stays up if control plane is briefly down — running config keeps serving. Control plane outage just stops new configs from propagating.
When would you NOT use a service mesh?
Small fleet (<15-50 services), single-language stack where libraries (Resilience4j, Polly) suffice, tight latency budgets (mesh adds 5-10ms p99 per hop), no platform team to operate it, or early-stage product where complexity cost outweighs benefit. Many teams adopt too early and regret it.
How does Istio do canary deploys?
VirtualService with weighted routing across two DestinationRule subsets (stable, canary). Start 95/5, monitor RED metrics (auto-emitted), gradually shift to 50/50 then 0/100. Can also use header-based routing: send only employees or beta users to canary. Combine with mirroring (shadow traffic to canary without serving its responses) for safe verification.
What's the latency cost?
~5-10ms p99 per hop is typical (in + out through Envoy = 2 hops per call). For deep call chains (5+ services) you can see 30-50ms p99 overhead vs direct. Plus +0.5 vCPU and 50-200 MB RAM per pod. Ambient mesh reduces this significantly by sharing one ztunnel per node.
How is config pushed to all the proxies?
xDS — a family of gRPC streaming APIs (CDS for clusters, EDS for endpoints, LDS for listeners, RDS for routes, ADS aggregates). Each Envoy opens a long-lived stream to the control plane and receives config diffs. Push model, near-instant propagation. xDS is an open spec used by many proxies, not just Istio's.
What's the difference between Istio and Linkerd?
Istio: Envoy-based (C++), feature-rich (L7 policies, federation, multi-cluster), heavy. Linkerd: Rust-based linkerd-proxy, smaller (~30 MB per sidecar), simpler model, less feature-rich. Linkerd is the choice when "do you really need all that?" is yes. Istio is the choice for big enterprises with diverse needs.
What's ambient mesh and why does it matter?
Istio's no-sidecar mode. Per-node ztunnel handles L4 + mTLS; a "waypoint" proxy adds L7 only where you need it. Lower resource cost (no per-pod sidecar), smaller blast radius for upgrades (DaemonSet vs pod restart), pay-for-what-you-use L7. Still maturing but it's the direction many shops want to go.
How do you authenticate end users through a mesh?
RequestAuthentication CRD verifies JWTs at the sidecar (no app code needed). AuthorizationPolicy CRD enforces "service X can call /Y only if JWT has scope Z". The mesh handles transport identity (mTLS, who-the-workload-is); JWTs carry user identity end-to-end. Two layers stacked.
How do you debug a request that's failing somewhere in the mesh?
(1) Auto-emitted access logs in each sidecar show in/out per hop. (2) Distributed traces (auto-injected) show latency per service. (3) Envoy admin endpoint exposes config + stats per sidecar. (4) istioctl proxy-status + istioctl proxy-config show what config each Envoy thinks it has. The hardest debugging is "is the app or the sidecar saying no?" — access logs are the first stop.
