01 — Requirements

Functional

  • Directory ops — mkdir, ls, rmdir (empty only), cd, pwd
  • File ops — touch, write, read, append, rm
  • Absolute paths (/home/user/file.txt) and relative paths (../file.txt, ./docs/)
  • Path normalization — resolve . and .. segments
  • Case-sensitive names
  • Metadata: size, creation timestamp, last modified, permissions (rwx)
  • Move / Copy files and directories to new locations

Non-Functional

  • Performance — O(1) direct access via HashMap, O(d) for path traversal where d = depth
  • Memory efficient — share common paths, no duplicate content
  • Thread-safe — support concurrent operations via ConcurrentHashMap
  • Scalable — handle 100K+ files and directories

03 — Design Patterns

The interview differentiator — Composite is the backbone, supported by clean separation of concerns.

Composite Pattern

Abstract FileSystemNode base class with File (leaf) and Directory (composite). Uniform treatment of files and directories. getSize() on a directory recursively sums children. Natural tree hierarchy.

Strategy Pattern

Permission checking as a pluggable strategy. Flexible access control — swap from basic rwx to ACLs without changing core logic.

Singleton Pattern

Root directory is a single source of truth. The entire tree hangs from one root node. Ensures consistent filesystem state.

Template Method

Abstract FileSystemNode defines getPath() (walks up parent pointers) and delete(). Subclasses override getSize() for polymorphic behavior.

Why Composite?

  • Uniform interface — client code calls node.getSize() without caring if it's a file or directory
  • Easy tree traversal — recursive operations (size, search, delete) are natural
  • Open/Closed — add SymLink as a new leaf without modifying existing code
  • Polymorphic ops — each node type defines its own behavior for shared operations

05 — Path Resolution

Why a separate PathResolver? Clean separation of concerns. Path parsing is complex (absolute vs relative, ., .., trailing slashes). Isolated class is easy to test independently and reuse across operations.

Resolution Algorithm

Algorithm: ResolvePath(path, currentDir)
Input:  path string, current directory
Output: target Directory

1. if path is absolute (starts with '/'):
      current = root
      path = path.substring(1)
   else:
      current = currentDir

2. if path is empty:
      return current

3. segments = path.split('/')

4. for each segment in segments:
      if segment == '' or segment == '.':
         continue
      if segment == '..':
         current = current.parent
         if current == null: current = root
      else:
         child = current.getChild(segment)
         if child == null: throw FileNotFoundException
         if child is not Directory: throw NotADirectoryException
         current = child

5. return current

Time: O(d) where d = path depth  •  Space: O(1)

06 — CRUD Operations

Create File

public boolean createFile(String path, String content) {
    if (files.containsKey(path)) return false;

    String fileName = getFileName(path);
    FileNode file = new FileNode(fileName, false);
    file.setContent(content);
    files.put(path, file);

    String parentPath = getParentPath(path);
    directories.computeIfAbsent(parentPath,
        k -> ConcurrentHashMap.newKeySet()).add(path);
    return true;
}

Create Directory

public boolean createDirectory(String path) {
    if (files.containsKey(path)) return false;

    String dirName = getFileName(path);
    FileNode dir = new FileNode(dirName, true);
    files.put(path, dir);
    directories.put(path, ConcurrentHashMap.newKeySet());

    String parentPath = getParentPath(path);
    directories.computeIfAbsent(parentPath,
        k -> ConcurrentHashMap.newKeySet()).add(path);
    return true;
}

Read & Write

public Optional<String> readFile(String path) {
    FileNode file = files.get(path);
    if (file == null || file.isDirectory())
        return Optional.empty();
    return Optional.ofNullable(file.getContent());
}

public boolean writeFile(String path, String content) {
    FileNode file = files.get(path);
    if (file == null || file.isDirectory()) return false;
    file.setContent(content);
    return true;
}

Delete (File or Directory)

public boolean delete(String path) {
    if ("/".equals(path)) return false; // cannot delete root

    FileNode node = files.get(path);
    if (node == null) return false;

    if (node.isDirectory()) {
        Set<String> children = directories.get(path);
        if (children != null && !children.isEmpty())
            throw new DirectoryNotEmptyException("...");
        directories.remove(path);
    }

    files.remove(path);
    String parentPath = getParentPath(path);
    Set<String> siblings = directories.get(parentPath);
    if (siblings != null) siblings.remove(path);
    return true;
}

Move & Copy

public boolean move(String src, String dest) {
    if (!files.containsKey(src) || files.containsKey(dest))
        return false;
    FileNode node = files.remove(src);
    files.put(dest, node);
    // update parent directories: remove from source, add to dest
    return true;
}

public boolean copy(String src, String dest) {
    FileNode sourceNode = files.get(src);
    if (sourceNode == null || files.containsKey(dest))
        return false;
    FileNode newNode = new FileNode(getFileName(dest), sourceNode.isDirectory());
    if (!sourceNode.isDirectory())
        newNode.setContent(sourceNode.getContent());
    files.put(dest, newNode);
    return true;
}

07 — Key Algorithms

Directory Size (Recursive Composite)

Algorithm: GetDirectorySize(directory)

1. if directory.children is empty: return 0
2. totalSize = 0
3. for each child in directory.children:
      if child is File:
         totalSize += child.size
      elif child is Directory:
         totalSize += GetDirectorySize(child) // recursive
4. return totalSize

Time: O(n) total files in subtree  •  Space: O(h) recursion stack, h = tree height

File Write Operation

Algorithm: WriteFile(path, content)

1. directory = resolvePath(parentPath(path), currentDir)
2. fileName  = basename(path)
3. file      = directory.getChild(fileName)
4. if file == null:
      file = new File(fileName)
      directory.addChild(file)
5. if file is not File: throw InvalidOperationException
6. if !file.permissions.canWrite(): throw AccessDeniedException
7. file.content.clear()
8. file.content.append(content)
9. file.modifiedAt = now()

Time: O(d) for path resolution  •  Space: O(1)

Directory Listing with Metadata

Algorithm: ListDirectory(path)

1. directory = resolvePath(path, currentDir)
2. result = []
3. for each child in directory.children:
      info = { name, type, size, modified }
      result.add(info)
4. return result sorted by name

Time: O(n log n) for sorting  •  Space: O(n) where n = children count

Helper: getParentPath & getFileName

private String getParentPath(String path) {
    if ("/".equals(path)) return null;
    int lastSlash = path.lastIndexOf('/');
    return lastSlash == 0 ? "/" : path.substring(0, lastSlash);
}

private String getFileName(String path) {
    if ("/".equals(path)) return "/";
    return path.substring(path.lastIndexOf('/') + 1);
}

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap files map, directories map Safe concurrent reads/writes to the path-to-node store
ConcurrentHashMap.newKeySet() Directory children sets Thread-safe sets for parent-child tracking
computeIfAbsent() createFile, createDirectory Atomic check-and-insert for parent directory tracking
ReadWriteLock Per node (production extension) Multiple concurrent readers, exclusive writer per file/directory
"I use ConcurrentHashMap for the global path store. For per-node locking at scale, I'd add ReadWriteLock per FileSystemNode — multiple readers can access a file concurrently, but writes are exclusive."

Concurrency Considerations

  • Atomic operations — computeIfAbsent prevents race conditions when multiple threads create files in the same directory
  • Delete safety — check-then-act on directory emptiness must be atomic to prevent orphaned children
  • Move atomicity — remove-from-source + add-to-dest should be in a critical section to prevent inconsistent state
  • At scale — use StampedLock for optimistic reads on frequently-accessed directories

10 — Happy Path Flow

Trace this end-to-end in the interview.

Initialize FileSystem
fs = new InMemoryFileSystem()
Root "/" created automatically. files map and directories map initialized with ConcurrentHashMap.
Create Directory Hierarchy
fs.mkdir("/home")
fs.mkdir("/home/user")
fs.mkdir("/home/user/docs")
Each mkdir: check exists → create FileNode(name, isDir=true) → add to files map → register in directories map → update parent's children set.
Create & Write File
fs.touch("/home/user/docs/readme.txt")
fs.write("/home/user/docs/readme.txt", "Hello World!")
touch: creates empty FileNode. write: looks up in files map, sets content. Both O(1) operations.
Read File
content = fs.read("/home/user/docs/readme.txt") → "Hello World!"
O(1) HashMap lookup. Returns Optional.empty() if not found or if path is a directory.
Navigate Directories
fs.cd("/home/user")
fs.pwd() → "/home/user"
cd: resolves path, updates currentDir pointer. pwd: walks parent pointers to reconstruct full path.
List Directory
fs.ls("docs") → ["readme.txt"]
Relative path resolved from currentDir. Returns child names from the directory's children set.
Append Content
fs.append("/home/user/docs/readme.txt", "\nLine 2")
Appends to existing StringBuilder content. Updates modifiedAt timestamp.
Copy & Move
fs.copy("/home/user/docs/readme.txt", "/home/backup.txt")
fs.move("/home/backup.txt", "/home/user/backup.txt")
Copy: creates new FileNode with cloned content. Move: removes from source, inserts at dest, updates parent sets.
Delete
fs.rm("/home/user/docs/readme.txt")
fs.rmdir("/home/user/docs")
rm: removes file from maps. rmdir: checks directory is empty first, throws DirectoryNotEmptyException if not.

12 — Interview Follow-up Q&A

Why Composite pattern for the filesystem?
Files and directories share common behavior (name, path, permissions, delete). Composite gives uniform treatment — client code calls node.getSize() without knowing the type. Directory's getSize() recursively aggregates children. Adding a new node type like SymLink is a new class, zero changes to existing code (Open/Closed Principle).
How does path resolution handle edge cases?
The algorithm handles: (1) absolute vs relative paths by checking the leading /, (2) . segments are skipped, (3) .. navigates to parent (clamped at root), (4) empty segments from double slashes are ignored, (5) trailing slashes are handled by the split. Time is O(d) where d = path depth.
Why store children in HashMap instead of ArrayList?
O(1) lookup by name vs O(n) scan. Fast insertion and deletion. Duplicate names are automatically prevented (same key overwrites). The tradeoff is no ordering — use LinkedHashMap for insertion order or TreeMap for sorted order if needed.
How would you handle file permissions?
The design includes a Permissions value object (read/write/execute). Check permissions before each operation: canRead() before read, canWrite() before write/append. For Unix-style: add user/group ownership, implement umask, support setuid/setgid. Use Strategy pattern for permission checking to swap between basic rwx and full ACLs.
How would you add symbolic links?
Add a SymLink class extending FileSystemNode that stores a target path string. During path resolution, when encountering a SymLink, resolve its target path recursively. Detect circular links with a visited set during resolution. Limit symlink depth (e.g., 40 hops like Linux).
How would you support file search (find)?
BFS/DFS traversal from a starting directory with predicate filters (name pattern, size range, modified date). For glob patterns (*.txt), use regex matching on filenames. At scale, maintain an inverted index of filenames to paths for O(1) lookup. The Composite pattern makes tree traversal natural.
How to handle very large files?
Instead of a single StringBuilder, chunk files into fixed-size blocks (4KB like real filesystems). Use a block list per file. Support streaming read/write via InputStream/OutputStream abstractions. For the in-memory case, use byte[][] (array of blocks) with lazy allocation. This also enables efficient append without copying.
How do you ensure thread safety?
ConcurrentHashMap for the global path-to-node store ensures safe concurrent access. computeIfAbsent() provides atomic check-and-insert. For per-node locking, add ReadWriteLock per FileSystemNode: multiple readers concurrent, exclusive writer. Move operations need to lock both source and destination parent (use ordered locking to prevent deadlocks).
Flat HashMap vs Composite Tree — which to choose?
HashMap approach: O(1) random access by full path, simpler implementation, but stores full path strings (more memory) and doesn't naturally support tree traversal. Composite tree: O(d) traversal but shared path prefixes save memory, natural recursive operations (size, search), parent pointers enable .. and pwd. For interview: start with Composite (shows pattern knowledge), mention HashMap as optimization for high-throughput random access.
How would you add journaling / crash recovery?
Log every operation to a write-ahead log (WAL) before executing. Format: [timestamp, op, path, args]. On crash, replay the log. Support checkpoints to truncate old log entries. Implement undo by writing inverse operations. This is how ext4's journal and NTFS's change journal work.
How would you persist the filesystem to disk?
Serialize the tree to a file (JSON or binary format) periodically. Use write-ahead log for durability between snapshots. On startup, load from the latest snapshot + replay WAL. For efficient serialization, use BFS to write level-by-level with parent references as indices.
What about directory rename/move?
Implement mv(source, dest): (1) remove node from source parent's children map, (2) update node's parent pointer, (3) add to destination parent's children map, (4) for the flat-map approach, recursively update all child paths (expensive). The tree approach only needs to move one node since paths are computed dynamically via parent pointers. This is a key advantage of the Composite tree.
Why parent pointers instead of path-only navigation?
Parent pointers enable: (1) getPath() by walking up to root — no path storage needed, (2) efficient .. navigation in O(1), (3) pwd implementation, (4) orphan detection (parent == null means detached). The tradeoff is bidirectional references that must stay consistent during moves and deletes.
