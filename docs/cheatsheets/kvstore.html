01 — Requirements

Functional

  • Basic Operations — put(key, value), get(key), delete(key), exists(key)
  • TTL (Time-To-Live) — setWithTTL(key, value, ttl), getTTL(key), persist(key), auto-expiration
  • Atomic Operations — increment(key), decrement(key), compareAndSwap(key, expected, new)
  • Transactions — begin(), commit(), rollback() with isolation
  • Pattern Matching — keys(pattern) with wildcard support
  • Persistence — Snapshots (periodic full dump), Write-Ahead Log (WAL), load from disk on restart

Non-Functional

  • Performance — O(1) operations, 100K+ QPS
  • Thread-Safety — concurrent reads/writes via ConcurrentHashMap
  • Memory Efficiency — eviction policies (LRU, LFU) when memory limit reached
  • Durability — WAL + snapshots for crash recovery
  • Availability — 99.99% uptime target

03 — Design Patterns

Five patterns, each earning its place in the design.

Strategy Pattern

EvictionPolicy interface with LRU and LFU implementations. Pluggable — swap eviction algorithms without changing the store. Also used for PersistenceManager.

Command Pattern

WAL operations (PUT, DELETE, INCREMENT) are replayable Command objects. Each mutation is logged before applying. On crash recovery, replay the log to rebuild state.

Memento Pattern

Snapshot captures a full copy of the store at a point in time. Restore reverts to that saved state. Periodic background snapshots + WAL replay = full durability.

Singleton Pattern

Single KVStore instance — one source of truth. All clients reference the same in-memory data. No inconsistency from multiple store instances.

Observer Pattern

Key expiry events trigger notifications to registered listeners. Subscribers react to expiration without polling. Decouples TTL cleanup from business logic.

05 — Get / Put / Delete

All O(1). ConcurrentHashMap provides lock-free reads and segment-level locking for writes. WAL append is sequential I/O (fast).

put(key, value)

// 1. Write to WAL first (durability before visibility)
operation = new WALEntry(PUT, key, value)
wal.append(operation)
wal.sync()  // force to disk

// 2. Update in-memory store
entry = new Entry(value, null, version++)
data.put(key, entry)

// 3. Trigger async snapshot if threshold reached
if operationsSinceSnapshot > SNAPSHOT_THRESHOLD:
    scheduleSnapshot()

Time: O(1) memory + O(1) amortized WAL  ·  Space: O(1)

get(key)

entry = data.get(key)

if entry == null:
    return null                // cache miss

if entry.isExpired():
    data.remove(key)           // lazy expiration
    ttlManager.cancelExpiry(key)
    return null

evictionPolicy.recordAccess(key)  // update LRU/LFU tracking
return entry.getValue()

Time: O(1)  ·  Space: O(1)  ·  Lazy TTL check on every read

delete(key)

// 1. Log deletion to WAL
wal.append(new WALEntry(DELETE, key, null))
wal.sync()

// 2. Remove from memory + cancel any TTL task
data.remove(key)
ttlManager.cancelExpiry(key)

Actual Implementation (InMemoryKVStore)

public void put(String key, String value) {
    KeyValue kv = new KeyValue(key, value);
    store.put(key, kv);
}

public String get(String key) {
    KeyValue kv = store.get(key);
    if (kv == null) return null;
    if (kv.isExpired()) {
        store.remove(key);
        return null;
    }
    return kv.getValue();
}

public void delete(String key) {
    store.remove(key);
}

06 — TTL Expiration

Two-pronged strategy: Lazy expiration on read (cheap, always correct) + scheduled background cleanup (prevents memory leaks from unread keys).

setWithTTL Algorithm

Algorithm: SetWithTTL(key, value, ttlSeconds)

1. expiryTime = currentTime() + ttlSeconds * 1000

2. entry = new Entry(value, expiryTime, version++)
3. data.put(key, entry)

4. // Schedule automatic removal
   task = scheduler.schedule(() -> {
       currentEntry = data.get(key)
       if currentEntry != null and currentEntry.expiryTime == expiryTime:
           data.remove(key)
   }, ttlSeconds, SECONDS)

5. ttlManager.scheduleExpiry(key, task)

Time: O(1) + O(log n) scheduler  ·  Space: O(1)

KeyValue.isExpired()

public boolean isExpired() {
    if (ttl == null) return false;
    return LocalDateTime.now().isAfter(timestamp.plusSeconds(ttl));
}

07 — Persistence (WAL + Snapshots)

Belt and suspenders. WAL guarantees no data loss between snapshots. Snapshots provide fast full-state recovery. Together: replay WAL from last snapshot.

Write-Ahead Log (WAL)

Every mutation is logged before being applied to memory. Sequential writes to disk (fast). On crash, replay the log to rebuild state.

interface WAL {
    void append(Operation op);   // log before apply
    List<Operation> replay();  // crash recovery
    void sync();                // fsync to disk
}

enum Operation { PUT, DELETE, INCREMENT }
PropertyDetail
Write patternSequential append-only (SSD-friendly)
Durabilityfsync after each write (configurable)
RecoveryReplay from last checkpoint
RotationTruncate after successful snapshot

Snapshots (Memento)

Periodic full dump of in-memory state. Defensive copy ensures snapshot isolation.

public class Snapshot {
    private final String id;
    private final Map<String, String> data;
    private final LocalDateTime timestamp;

    public Snapshot(String id, Map<String, String> data) {
        this.id = id;
        this.data = new HashMap<>(data);  // defensive copy
        this.timestamp = LocalDateTime.now();
    }
}

Recovery Flow

flowchart LR
    A[Startup] --> B{Snapshot exists?}
    B -- Yes --> C[Load Snapshot]
    B -- No --> D[Empty Store]
    C --> E[Replay WAL from snapshot timestamp]
    D --> E
    E --> F[Store Ready]

createSnapshot & restoreSnapshot

public String createSnapshot() {
    String snapshotId = UUID.randomUUID().toString();
    Map<String, String> data = store.entrySet().stream()
        .collect(Collectors.toMap(
            Map.Entry::getKey, e -> e.getValue().getValue()));
    snapshots.put(snapshotId, new Snapshot(snapshotId, data));
    return snapshotId;
}

public void restoreSnapshot(String snapshotId) {
    Snapshot snapshot = snapshots.get(snapshotId);
    if (snapshot != null) {
        store.clear();
        for (Map.Entry<String,String> e : snapshot.getData().entrySet())
            put(e.getKey(), e.getValue());
    }
}

08 — Atomic Operations & CAS

Lock-free concurrency. Versioned entries enable optimistic locking. CAS fails and retries rather than blocking threads.

Compare-And-Swap (CAS)

Algorithm: CompareAndSwap(key, expectedValue, newValue)

1. entry = data.get(key)

2. if entry == null:
       if expectedValue != null: return false
       entry = new Entry(newValue, null, 1)
       data.put(key, entry)
       return true

3. currentValue = entry.getValue()

4. if currentValue != expectedValue:
       return false          // CAS failed -- stale read

5. // Atomic update with version increment
   newEntry = new Entry(newValue, entry.expiryTime, entry.version + 1)
   success = data.replace(key, entry, newEntry)  // atomic

6. if success:
       wal.append(WALEntry(PUT, key, newValue))

7. return success

Time: O(1)  ·  Space: O(1)  ·  No locks — retry on conflict

Increment / Decrement

// Atomic increment using CAS loop
public long increment(String key) {
    while (true) {
        String current = get(key);
        long val = (current == null) ? 0 : Long.parseLong(current);
        String next = String.valueOf(val + 1);
        if (compareAndSwap(key, current, next))
            return val + 1;
        // CAS failed, retry with fresh value
    }
}

09 — Transactions

Optimistic concurrency. Buffer writes in a local map. On commit, validate read-set versions, then apply atomically. Rollback = discard the buffer.

Transaction Lifecycle

stateDiagram-v2
    [*] --> Active: begin()
    Active --> Active: put(key, value)
    Active --> Committed: commit()
    Active --> RolledBack: rollback()
    Committed --> [*]
    RolledBack --> [*]

Transaction Class

public class Transaction {
    private final String id;
    private final Map<String, String> changes;  // write buffer
    private boolean committed;

    public void put(String key, String value) {
        changes.put(key, value);
    }

    public void commit()   { committed = true; }
    public void rollback() { changes.clear(); }
}

Commit Algorithm (Optimistic Locking)

Algorithm: CommitTransaction(transaction)

// Phase 1: Validate (check versions haven't changed)
for each (key, entry) in transaction.readSet:
    currentEntry = data.get(key)
    if currentEntry.version != entry.version:
        return false  // read conflict, abort

// Phase 2: Prepare (log to WAL)
for each (key, entry) in transaction.writeSet:
    wal.append(WALEntry(PUT, key, entry.value))
wal.sync()

// Phase 3: Commit (apply to memory)
for each (key, entry) in transaction.writeSet:
    newEntry = new Entry(entry.value, null, currentVersion++)
    data.put(key, newEntry)

transaction.status = COMMITTED
return true

Time: O(w + r) where w = writes, r = reads  ·  Space: O(w + r)

Service-Level Transaction Methods

public String beginTransaction() {
    String txnId = UUID.randomUUID().toString();
    transactions.put(txnId, new Transaction(txnId));
    return txnId;
}

public void commitTransaction(String txnId) {
    Transaction txn = transactions.get(txnId);
    if (txn != null && !txn.isCommitted()) {
        for (Map.Entry<String,String> e : txn.getChanges().entrySet())
            put(e.getKey(), e.getValue());
        txn.commit();
    }
}

public void rollbackTransaction(String txnId) {
    Transaction txn = transactions.remove(txnId);
    if (txn != null) txn.rollback();
}

10 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap store, transactions, snapshots Lock-free reads, segment-level write locking. O(1) ops stay O(1) under concurrency.
Entry.version Every entry Optimistic locking for CAS. Detect stale reads without locks.
replace(key, old, new) CAS operation ConcurrentHashMap's atomic compare-and-replace. Only succeeds if entry unchanged.
ScheduledExecutorService TTL expiry Thread-safe task scheduling. Non-blocking delayed removal of expired keys.
volatile PricingStrategy ref Cross-thread visibility for runtime strategy swaps without full lock.
HashMap (defensive copy) Snapshot, Transaction.getChanges() Isolation. Snapshot data can't be mutated by ongoing writes.
"ConcurrentHashMap gives us lock-free reads at the cost of no total ordering. For a KV store, this is the right tradeoff — we optimize for throughput over global consistency."

14 — Interview Follow-up Q&A

Why ConcurrentHashMap instead of synchronized HashMap?
ConcurrentHashMap provides lock-free reads and segment-level write locking. Synchronized HashMap locks the entire map on every operation. For a KV store doing 100K+ QPS, the difference is massive — CHM allows concurrent readers without blocking, while synchronized blocks all threads on any write.
How does lazy vs active TTL expiration work?
Lazy: Check isExpired() on every get(). Free, always correct, but expired keys linger in memory if never read. Active: ScheduledExecutorService fires removal tasks at exact expiry time. Guarantees timely cleanup. Use both: active for memory hygiene, lazy as a safety net.
How would you handle memory limits and eviction?
Set a maxEntries or maxMemoryBytes limit. On every put(), check if over limit. If so, call evictionPolicy.evict() to get the victim key and remove it. LRU uses a LinkedHashMap in access order — the eldest entry is the least-recently-used. LFU uses a frequency counter — evict the key with the lowest access count.
Why WAL before memory write?
If we write to memory first and crash before persisting, data is lost. WAL guarantees that every committed mutation is on disk. On recovery, replay the WAL to rebuild the in-memory state. The write order must be: WAL → fsync → memory update. This is the same strategy used by PostgreSQL, SQLite, and Redis AOF.
How do snapshots and WAL work together?
Snapshots capture full state periodically (e.g., every 10 minutes or every N operations). WAL logs every mutation. On recovery: (1) load the latest snapshot, (2) replay only the WAL entries after the snapshot timestamp. This bounds recovery time — you never replay the entire WAL from the beginning. After a successful snapshot, truncate the WAL.
How would you implement LRU eviction?
Use LinkedHashMap with access-order mode (new LinkedHashMap(16, 0.75f, true)). Every get() moves the key to the tail. The head is always the least-recently-used key. Override removeEldestEntry() or call iterator().next() to find the eviction victim. O(1) for both access and eviction.
How would you scale beyond a single node?
Consistent hashing for partitioning keys across nodes. Each node owns a range of the hash ring. Replication (master-slave or multi-master) for fault tolerance. Gossip protocol for cluster membership. Use vector clocks for conflict resolution in multi-master setups. The KVStore interface stays the same — the routing layer is transparent.
How does CAS enable lock-free concurrency?
Each entry has a version number. compareAndSwap() reads the current version, computes the new value, and atomically replaces only if the version hasn't changed. If another thread modified the entry, CAS fails and the caller retries with the fresh value. No locks needed — threads never block each other. This is the same approach used by java.util.concurrent.atomic classes.
How would you handle transaction isolation?
The current design uses optimistic concurrency control. Each transaction reads entries with their version numbers (read set). On commit, validate that all read-set versions are unchanged. If any version changed, abort — another transaction modified the data. This provides snapshot isolation without locks. For stricter isolation (serializable), use pessimistic locking with a lock manager.
Why defensive copies in Snapshot and Transaction?
new HashMap<>(data) creates a shallow copy, isolating the snapshot from ongoing mutations. Without it, a snapshot's data would change as the live store mutates — breaking the Memento pattern's guarantee. Same for Transaction.getChanges() — callers get a copy, not a reference to the internal buffer.
How would you support complex data types like Lists and Sets?
Store typed collections as values: Map<String, List<String>> for lists, Map<String, Set<String>> for sets, Map<String, TreeMap> for sorted sets. Add type-specific commands: lpush, rpop, sadd, zadd. The value field becomes Object with runtime type checking, or use a tagged union (enum + value).
How would you improve write performance?
(1) Batch WAL writes — group multiple operations into a single fsync. (2) Async fsync — trade durability for speed (configurable). (3) SSD for WAL — sequential writes on SSD are extremely fast. (4) Compression — compress WAL entries to reduce I/O. (5) Pipeline — accept multiple commands and flush once, like Redis pipelining.
What real-world extensions would you add?
Pub/Sub: Channel subscriptions with pattern matching. Streams: Append-only log with consumer groups. Geo-Spatial: lat/long with radius queries using Geohash indexing. Lua Scripting: Server-side atomic script execution. Cluster Mode: Sharding + replication for horizontal scaling.
