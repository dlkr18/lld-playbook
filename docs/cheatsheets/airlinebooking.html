Airline Seat Booking // LLD

Search flights by route/date, hold a specific seat with a TTL, confirm the hold into a PNR, or cancel — with lock-free per-seat CAS so two passengers can never book the same seat. State lifecycle, pluggable Strategy pricing + allocation, Observer notifications.

01 — Requirements

Functional

  • Register flights — origin, destination, departure/arrival times, operated by an aircraft with a seat map (cabins of rows × columns).
  • Search — flights by (origin, destination, date).
  • View seat map — available seats per cabin class (ECONOMY / BUSINESS / FIRST).
  • Hold a seat — a specific seat (holdSeat("12A")) or any seat in a class (holdAnySeat(ECONOMY)), reserved with a TTL; issues a PNR immediately.
  • Confirm — turn a valid hold into a ticketed booking (post-payment step).
  • Cancel — release a held/confirmed seat; terminal.
  • Expire — holds that exceed their TTL free their seat automatically.
  • Price — per-class fare via a pricing strategy. Notify — confirmation/cancellation to the passenger.

Non-Functional

  • Correctness — no double booking. Two confirmed bookings on the same seat is the single hard invariant.
  • Concurrency — lock-free per-seat reservation; holds on different seats never contend, so throughput scales with seats, not with a global lock.
  • Liveness — stale holds must not permanently block a seat (TTL + eager sweep + lazy takeover).
  • Extensibility — pricing, allocation, notification channels all pluggable without touching the orchestrator.
  • Testability — expiry drivable deterministically (manual sweepExpiredHolds()) as well as by a background daemon thread.

03 — Design Patterns

Six patterns, each earning its place — the interview differentiator.

State — Booking Lifecycle

Booking is the context; HeldState / ConfirmedState / CancelledState / ExpiredState encode legal transitions. Illegal transitions throw from the state itself. Alternative rejected: a switch(status) smears rules across methods.

Strategy — Pricing + Allocation

FarePricingStrategy (ClassBased vs Dynamic) and SeatAllocationStrategy (FrontToBack vs WindowPreferred). Policies change independently; the orchestrator never knows which is active. OCP.

Observer — Notifications

AirlineBooking is the subject; Email/SMS observers subscribe. Adding a channel (push, WhatsApp) never touches booking logic. Fan-out with per-observer failure isolation.

Lock-free CAS (concurrency idiom)

Per-seat AtomicReference<Occupancy>; reserve via compareAndSet. "Exactly one winner" with no lock and no lock-registry bookkeeping. Alternative rejected: SeatLockManager (the BookMyShow approach).

Builder — Aircraft Seat Map

Aircraft.builder("A320").addCabin(FIRST,1,4)... numbers rows continuously and enforces column ≤ 26. A seat map is tedious to hand-construct.

Facade — Orchestrator

AirlineBooking is the single entry point wiring search, inventory, strategies, and observers — the god object an interviewer sees first.

04 — Booking State Machine

Why State? Transition legality is asked in four places (confirm / cancel / expire from each state). One class per state keeps the table cohesive and makes illegal transitions throw at the source — a confirmed booking rejects expire, a terminal one rejects everything. 
stateDiagram-v2
    [*] --> HELD: holdSeat() [CAS AVAILABLE->HELD, PNR issued]
    HELD --> CONFIRMED: confirm() [CAS HELD->BOOKED]
    HELD --> CANCELLED: cancel() [seat released]
    HELD --> EXPIRED: TTL elapsed + sweep [seat released]
    CONFIRMED --> CANCELLED: cancel() [seat released]
    CANCELLED --> [*]
    EXPIRED --> [*]

Booking as State Context

public final class Booking {
    private final String pnr;          // issued at hold time, stable thru confirm
    private final Seat seat;
    private final Instant holdExpiry;
    private BookingState state = HeldState.INSTANCE;

    // Transitions are synchronized so a user confirm and the
    // background sweeper cannot corrupt the state field.
    public synchronized void confirm() { state.confirm(this); }
    public synchronized void cancel()  { state.cancel(this); }
    public synchronized void expire()  { state.expire(this); }

    // package-private hook used by state objects
    void setState(BookingState next) { this.state = next; }
}
The state field is guarded, but the authoritative winner is the seat CAS — state is only advanced by the thread that won the seat, so confirm and sweep can't both "succeed" on the same seat.

05 — Strategy Pattern

Why Strategy? Pricing and seat-preference policies change independently and often. New pricing model = new class. New allocation logic = new class. Swappable at construction without touching the orchestrator. Open/Closed Principle.

06 — Key Algorithm: Lock-free CAS Seat Hold + TTL

The signature detail. Each seat maps to its own AtomicReference<Occupancy> where Occupancy is an immutable snapshot {status, ownerPnr, holdExpiry}. The only way to reserve a seat is a compare-and-set AVAILABLE → HELD, so exactly one racer wins. No mutex, no lock registry, no cross-seat contention.

1. Per-seat compare-and-set (double-booking guard)

public boolean tryHold(String seatNumber, String bookingId, Instant expiry) {
    AtomicReference<Occupancy> ref = require(seatNumber);
    Instant now = Instant.now();
    for (;;) {                                // lock-free retry loop
        Occupancy current = ref.get();
        boolean eligible = current.status == AVAILABLE
                || current.isExpiredHold(now);    // lazy takeover
        if (!eligible) return false;             // BOOKED or live hold
        Occupancy next = Occupancy.held(bookingId, expiry);
        if (ref.compareAndSet(current, next)) return true;  // we won
        // lost the race -> re-read and re-evaluate eligibility
    }
}

Because compareAndSet is atomic and Occupancy is never mutated in place, when N threads race the same seat exactly one CAS succeeds; losers observe the changed reference and either back off (taken) or retry (still eligible but the ref moved). confirm (HELD→BOOKED, owner-checked) and release (HELD/BOOKED→AVAILABLE, owner-checked) use the same idiom.

2. Confirm — owner-checked CAS HELD → BOOKED

public boolean confirm(String seatNumber, String bookingId) {
    AtomicReference<Occupancy> ref = require(seatNumber);
    Instant now = Instant.now();
    for (;;) {
        Occupancy current = ref.get();
        boolean ownedActiveHold = current.status == HELD
                && bookingId.equals(current.bookingId)
                && !current.isExpiredHold(now);
        if (!ownedActiveHold) return false;    // expired / not owner
        if (ref.compareAndSet(current, Occupancy.booked(bookingId)))
            return true;
    }
}

3. Hold TTL — eager sweep + lazy takeover

// EAGER: daemon ScheduledExecutorService calls this on a fixed interval.
// CASes every expired hold back to AVAILABLE, returns freed PNRs so the
// orchestrator can drive their bookings to EXPIRED (via State).
public List<String> sweepExpired(Instant now) {
    List<String> freed = new ArrayList<>();
    for (AtomicReference<Occupancy> ref : occupancy.values()) {
        Occupancy current = ref.get();
        if (current.isExpiredHold(now)
                && ref.compareAndSet(current, Occupancy.available()))
            freed.add(current.bookingId);
    }
    return freed;
}

// LAZY: tryHold (above) already treats an expired hold as eligible and
// CASes straight over it -- a new booker is never blocked between sweeps.
The two paths are consistent: whoever wins the CAS owns the seat; a sweep that loses the CAS simply doesn't free that slot. For deterministic tests the sweeper is optional — the no-arg constructor omits the background thread and tests call sweepExpiredHolds() explicitly.

4. Confirm vs. sweeper race (ordering matters)

Algorithm: confirm(pnr)

1. booking = bookings.get(pnr); if already CONFIRMED -> return  // idempotent
2. if !inventory.confirm(seat, pnr):        // seat CAS FIRST
       booking.expire() (best-effort); throw SeatUnavailableException
3. booking.confirm();                     // HELD -> CONFIRMED (only after CAS win)
4. notifyConfirmed(booking)              // Observer fan-out

Booking is registered in the map BEFORE its tryHold, so a concurrent
sweep can always find the booking it is expiring. State is only ever
advanced by the thread that won the seat CAS.

07 — Thread Safety

MechanismWhereWhy
AtomicReference<Occupancy> one per seat in SeatInventory Lock-free CAS for AVAILABLE→HELD→BOOKED. Exactly one winner; different seats never contend.
immutable Occupancy seat snapshot Never mutated in place — that's what makes compare-and-set semantics correct.
ConcurrentHashMap inventories (per flight), bookings (by PNR) Safe concurrent registry reads/writes. putIfAbsent prevents duplicate flight registration.
CopyOnWriteArrayList observers Read-heavy fan-out; safe iteration during notify without locking.
synchronized methods Booking.confirm/cancel/expire Guards the state field so a user confirm and the sweeper can't corrupt it. Authoritative winner is still the seat CAS.
daemon ScheduledExecutorService hold sweeper Single-thread background TTL sweep; optional (omitted for deterministic tests).
"Why not just synchronize the whole booking method? A global lock serialises all seats on a flight — a full A380 would bottleneck on one monitor. Per-seat CAS keeps unrelated bookings fully parallel, with no lock registry to clean up."

Why CAS over a lock manager

ApproachTrade-off

// Unique 6-char record locator [A-Z0-9], retried on collision
private String newPnr() {
    for (;;) {
        StringBuilder sb = new StringBuilder(6);
        for (int i = 0; i < 6; i++)
            sb.append(PNR_ALPHABET[ThreadLocalRandom.current()
                .nextInt(PNR_ALPHABET.length)]);
        String candidate = sb.toString();
        if (!bookings.containsKey(candidate)) return candidate;
    }
}

11 — How This Differs from Movie-Ticket Booking

They look similar (both hold-then-confirm a seat under contention), but the interesting modelling is different — lean into that.
DimensionAirline (this)BookMyShow
Seat model Seat MAP with classes — rows × columns grouped into heterogeneous cabins (ECONOMY/BUSINESS/FIRST) with very different per-class fares. One homogeneous grid on a screen.
Record PNR issued at hold time, survives through confirmation — a first-class domain concept. Just a booking id.
Hold semantics Headline requirement. Hold is both eagerly swept and lazily self-healing (CAS over an expired hold before the sweeper runs). Lock timeout is a secondary detail.
Booking granularity Single seat per booking + auto-assign strategy (window vs front-to-back). All-or-nothing multi-seat selection.
Concurrency mechanism Lock-free per-seat CAS on an AtomicReference — clean "exactly one winner", no lock bookkeeping. Lock-manager registry (Map<seat, lock>).

12 — Interview Follow-up Q&A

How would you make it distributed / multi-node?
Replace the in-process AtomicReference with an atomic store per seat: Redis SET seat:{flight}:{seat} {pnr} NX PX {ttl} gives CAS-hold + TTL in one op — Redis auto-expires the hold, so no sweeper needed. Confirmation writes a DB row with a UNIQUE(flight_id, seat_number) WHERE status='BOOKED' constraint as the ultimate double-booking backstop.
Why per-seat CAS instead of synchronizing the whole booking method?
A global lock serialises all seats on a flight — a full A380 would bottleneck on one monitor. The only thing that must be race-free is a single seat's status transition. Per-seat CAS gives "exactly one winner" with no lock, no lock registry to clean up, and no cross-seat contention, so throughput scales with the number of seats.
How do you prevent a stale hold from permanently blocking a seat?
Two mechanisms, reconciled by CAS. Eager: a daemon scheduler runs sweepExpiredHolds(), CASing every expired hold back to AVAILABLE and moving its booking to EXPIRED (so status changes even if nobody re-requests the seat). Lazy: tryHold treats an expired hold as eligible and CASes straight over it, so a new booker is never blocked between sweeps. Whoever wins the CAS owns the seat; a sweep that loses simply doesn't free that slot.
What about overbooking?
Allow holds/bookings up to capacity × factor per class using a per-class atomic counter instead of per-seat status for the count, while still assigning concrete seats at check-in. Add a bump / reaccommodation policy for when everyone shows up.
How would you book a party of N seats atomically?
Extend holdSeat to a seat list; acquire all-or-nothing by CAS-ing seats in a canonical (sorted) order and rolling back released seats on any failure. The sorted order avoids deadlock and partial holds.
How would you add a waitlist on a sold-out class?
A queue per (flight, class). On cancel/expire, an Observer pops the next waiter and offers the freed seat with a fresh TTL — reusing the existing notification fan-out. No orchestrator changes.
Are confirm and hold idempotent?
confirm is already idempotent for an already-CONFIRMED PNR (early return). For hold, accept a client idempotency key and return the existing booking on retry, so a network retry doesn't create a second PNR.
How would you support blocked / maintenance seats?
Add an Occupancy status BLOCKED that tryHold treats as ineligible; an admin sets it directly. The CAS idiom is unchanged — BLOCKED is just another non-eligible state.
Why State pattern for only four states?
Transition legality is checked in confirm / cancel / expire from each state — that's a table. One class per state keeps rules cohesive and self-validating (a confirmed booking rejects expire). The alternative, a switch(status) in the service, smears the rules across methods and every new status touches every switch. A bit of class overhead, but far clearer.
