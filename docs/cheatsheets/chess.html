01 — Requirements

Functional

  • 8×8 board with standard chess piece placement (32 pieces)
  • Piece-specific movement validation (6 piece types, each with unique rules)
  • Path obstruction checking (pieces can't jump over others, except Knight)
  • Turn management — White moves first, alternating turns
  • Check detection — king is under attack by opponent piece
  • Checkmate detection — king in check + no legal moves = game over
  • Stalemate detection — NOT in check + no legal moves = draw
  • Special moves — castling, en passant, pawn promotion
  • Move history in algebraic notation (e4, Nf3, O-O)
  • Undo/redo and PGN export

Non-Functional

  • Performance — move validation < 10ms, check detection < 50ms
  • Thread-safe — synchronized state mutations for multiplayer
  • Extensible — easy to add chess variants (Chess960, Bughouse)
  • Pluggable AI — minimax with alpha-beta pruning
  • Testable — unit test each piece type and special move independently

03 — Design Patterns

The interview differentiator — multiple patterns, each earning its place.

Strategy / Polymorphism

Each piece type has unique movement rules. The isValidPieceMove() method delegates to piece-specific logic via switch on PieceType. In the enhanced version, each piece is an abstract subclass with its own isValidMove().

Template Method

Common validation steps shared across all pieces: (1) bounds check, (2) own piece check, (3) friendly capture check, (4) piece-specific validation, (5) king safety check. Steps 1-3 and 5 are fixed; step 4 varies.

Command Pattern

Move operations encapsulated as commands for undo/redo. Each Move stores: piece, from, to, captured piece, and special move flags. Enables game replay and PGN export.

Memento Pattern

Board state saved before move simulation. wouldLeaveInCheck() simulates a move, checks king safety, then undoes the mutation. Production version would use board.clone() for immutable snapshots.

Factory Pattern

Board.initialize() acts as a factory, creating and placing all 32 pieces in their standard starting positions. Extensible for Chess960 random placement.

Observer Pattern

Game events (move, check, checkmate, draw) fire notifications. In production: UI updates, move logging, spectator broadcasting, and real-time sync via WebSocket.

04 — Board Representation

Key insight: The board is a 2D array Piece[8][8]. Row 0 = rank 8 (Black back rank), row 7 = rank 1 (White back rank). Column 0 = file 'a', column 7 = file 'h'. Null cells = empty squares.

Board.java — Core Operations

public class Board {
    private final Piece[][] grid = new Piece[8][8];

    public void initialize() {
        for (int i = 0; i < 8; i++) {
            grid[1][i] = new Piece(PieceType.PAWN, Color.BLACK);
            grid[6][i] = new Piece(PieceType.PAWN, Color.WHITE);
        }
        grid[0][0] = new Piece(PieceType.ROOK, Color.BLACK);
        grid[0][7] = new Piece(PieceType.ROOK, Color.BLACK);
        // ... Knights at [0][1],[0][6], Bishops at [0][2],[0][5]
        // ... Queen at [0][3], King at [0][4], same for White on row 7
    }

    public Piece getPiece(Position pos) { return grid[pos.getRow()][pos.getCol()]; }
    public void setPiece(Position pos, Piece p) { grid[pos.getRow()][pos.getCol()] = p; }

    public void movePiece(Position from, Position to) {
        Piece piece = getPiece(from);
        setPiece(to, piece);
        setPiece(from, null);
        if (piece != null) piece.setMoved();
    }
}

Position.java — Value Object

public class Position {
    private final int row, col;

    public boolean isValid() {
        return row >= 0 && row < 8 && col >= 0 && col < 8;
    }

    public String toString() {
        char file = (char) ('a' + col);  // col 0 → 'a', col 7 → 'h'
        int rank = 8 - row;                // row 0 → 8, row 7 → 1
        return "" + file + rank;          // e.g. "e4"
    }
}

Piece-Specific Validation (switch dispatch)

private boolean isValidPieceMove(Piece piece, Position from, Position to) {
    int dr = to.getRow() - from.getRow();
    int dc = to.getCol() - from.getCol();
    int absDr = Math.abs(dr), absDc = Math.abs(dc);

    switch (piece.getType()) {
        case PAWN:   return isValidPawnMove(piece, from, to, dr, dc, absDr, absDc);
        case ROOK:   return (dr != 0 && dc != 0) ? false : isPathClear(from, to);
        case KNIGHT: return (absDr == 2 && absDc == 1) || (absDr == 1 && absDc == 2);
        case BISHOP: return (absDr != absDc) ? false : isPathClear(from, to);
        case QUEEN:  return isValidRookMove(..) || isValidBishopMove(..);
        case KING:   return absDr <= 1 && absDc <= 1;
    }
}

Pawn — The Trickiest Piece

private boolean isValidPawnMove(Piece piece, Position from, Position to,
                                int dr, int dc, int absDr, int absDc) {
    int direction = (piece.getColor() == Color.WHITE) ? -1 : 1;
    boolean isCapture = board.getPiece(to) != null;

    // Normal: forward 1 square, no capture
    if (dc == 0 && !isCapture && dr == direction) return true;

    // Double move from starting row
    int startRow = (piece.getColor() == Color.WHITE) ? 6 : 1;
    if (dc == 0 && !isCapture && from.getRow() == startRow && dr == 2 * direction) {
        Position mid = new Position(from.getRow() + direction, from.getCol());
        return board.getPiece(mid) == null;
    }

    // Diagonal capture
    if (absDc == 1 && dr == direction && isCapture) return true;

    return false;
}

Path Obstruction Check

private boolean isPathClear(Position from, Position to) {
    int dr = Integer.signum(to.getRow() - from.getRow());
    int dc = Integer.signum(to.getCol() - from.getCol());

    int r = from.getRow() + dr, c = from.getCol() + dc;
    while (r != to.getRow() || c != to.getCol()) {
        if (board.getPiece(new Position(r, c)) != null) return false;
        r += dr; c += dc;
    }
    return true;
}

Works for Rook, Bishop, and Queen. Uses Integer.signum() to step one square at a time in the movement direction. Knight skips this check entirely.

06 — Move Validation Pipeline

Template Method: 5-step pipeline. Steps 1–3 and 5 are shared across all piece types. Step 4 is piece-specific. A move must pass ALL steps.

5-Step Validation

#CheckFails When
1Bounds & identity!from.isValid() or from.equals(to)
2Own piece at sourcepiece == null or piece.color != currentPlayer
3Friendly fireTarget square has own-color piece
4Piece-specific rulesPawn/Rook/Knight/Bishop/Queen/King movement violated
5King safetyMove would leave own king in check

Full isValidMove() Implementation

public boolean isValidMove(Position from, Position to) {
    // Step 1: Bounds
    if (!from.isValid() || !to.isValid() || from.equals(to)) return false;

    // Step 2: Own piece
    Piece piece = board.getPiece(from);
    if (piece == null || piece.getColor() != currentPlayer) return false;

    // Step 3: Friendly fire
    Piece target = board.getPiece(to);
    if (target != null && target.getColor() == currentPlayer) return false;

    // Step 4: Piece-specific movement
    if (!isValidPieceMove(piece, from, to)) return false;

    // Step 5: King safety — move doesn't leave own king in check
    return !wouldLeaveInCheck(from, to, currentPlayer);
}

Check Detection — O(n) where n ≤ 16

public boolean isInCheck(Color color) {
    Position kingPos = findKing(color);
    if (kingPos == null) return false;
    Color opponent = (color == Color.WHITE) ? Color.BLACK : Color.WHITE;
    return isSquareAttackedBy(kingPos, opponent);
}

private boolean isSquareAttackedBy(Position target, Color attackerColor) {
    for (int r = 0; r < 8; r++) {
        for (int c = 0; c < 8; c++) {
            Piece piece = board.getPiece(new Position(r, c));
            if (piece != null && piece.getColor() == attackerColor
                    && isValidPieceMove(piece, new Position(r, c), target)) {
                return true;
            }
        }
    }
    return false;
}

Scans all 64 squares, checks if any opponent piece can legally reach the king's position. Worst-case 16 opponent pieces × their move logic.

Move Simulation (Memento / Undo)

private boolean wouldLeaveInCheck(Position from, Position to, Color color) {
    Piece saved = board.getPiece(to);   // save captured piece
    board.movePiece(from, to);           // simulate move
    boolean inCheck = isInCheck(color); // check king safety
    board.movePiece(to, from);           // undo move
    board.setPiece(to, saved);           // restore captured piece
    return inCheck;
}

Mutate-then-undo approach. Production alternative: deep-clone board before simulation. Trade-off: mutation is faster, cloning is safer for complex undo chains.

Checkmate & Stalemate Detection

private boolean hasAnyLegalMove(Color color) {
    for (int r = 0; r < 8; r++)
      for (int c = 0; c < 8; c++) {
        Piece piece = board.getPiece(new Position(r, c));
        if (piece != null && piece.getColor() == color)
          for (int tr = 0; tr < 8; tr++)
            for (int tc = 0; tc < 8; tc++) {
              Position from = new Position(r, c);
              Position to   = new Position(tr, tc);
              if (isValidPieceMove(piece, from, to)
                  && !wouldLeaveInCheck(from, to, color))
                return true;  // Early exit on first legal move
            }
      }
    return false;
}
ConditionResult
isInCheck(color) && !hasAnyLegalMove(color)Checkmate — game over, opponent wins
!isInCheck(color) && !hasAnyLegalMove(color)Stalemate — draw
isInCheck(color) && hasAnyLegalMove(color)Check — must escape on next move

Complexity: O(P × 64 × 64) per check, but early exit makes average case fast. P = pieces of one color (max 16).

08 — Special Moves

Castling (O-O / O-O-O)

King moves 2 squares toward a rook; rook jumps to the other side of king.

public boolean canCastle(boolean kingside) {
    if (king.hasMoved()) return false;

    int rookCol = kingside ? 7 : 0;
    Piece rook = board.getPiece(new Position(king.getRow(), rookCol));
    if (rook == null || rook.hasMoved()) return false;

    // Check path is clear
    int start = Math.min(king.getCol(), rookCol);
    int end = Math.max(king.getCol(), rookCol);
    for (int col = start + 1; col < end; col++)
        if (board.isOccupied(new Position(king.getRow(), col)))
            return false;

    // King not in check, doesn't pass through check
    if (isInCheck(king.getColor())) return false;
    int step = kingside ? 1 : -1;
    return !isSquareAttackedBy(
        new Position(king.getRow(), king.getCol() + step),
        opponent);
}

Pawn Promotion

When a pawn reaches the opposite end of the board (row 0 for White, row 7 for Black), it must be promoted.

Queen, Rook, Bishop, KnightQueen chosen ~97% of the time ("queening")

private void handlePromotion(Position to, Color color, PieceType promotion) {
    int promotionRow = (color == Color.WHITE) ? 0 : 7;
    if (to.getRow() == promotionRow) {
        board.setPiece(to, new Piece(promotion, color));
    }
}

09 — Thread Safety

MechanismWhereWhy
synchronized makeMove() Only one thread can mutate game state at a time — prevents race conditions between validation and execution
volatile currentPlayer, gameOver Cross-thread visibility for read-only access (current turn display)
Piece[8][8] Board grid Array is final reference; mutations gated by synchronized makeMove()
Immutable Position All position params Positions are value objects — row/col are final, inherently thread-safe
"The synchronized makeMove() serializes all game mutations. For a two-player game, this is sufficient. For a chess server hosting thousands of games, each ChessGameImpl instance has its own lock — games are independent."
public synchronized boolean makeMove(Position from, Position to) {
    if (gameOver || !isValidMove(from, to)) return false;

    board.movePiece(from, to);
    currentPlayer = (currentPlayer == Color.WHITE) ? Color.BLACK : Color.WHITE;

    if (isInCheck(currentPlayer)) {
        if (!hasAnyLegalMove(currentPlayer)) {
            gameOver = true;  // Checkmate!
        }
    }
    return true;
}

13 — Interview Follow-up Q&A

How do you validate if a move puts your own king in check?
Simulate the move on the board (move piece, save captured), then run isInCheck(myColor) which scans all opponent pieces to see if any can reach the king. Undo the move afterward. This "try-and-revert" approach is O(n) where n ≤ 16 opponent pieces.
What's the difference between checkmate and stalemate?
Checkmate: King IS in check AND no legal moves exist → game over, opponent wins. Stalemate: King is NOT in check AND no legal moves exist → draw. The only difference is whether the king is currently under attack.
How do you implement castling validation?
Five conditions: (1) King hasn't moved, (2) Rook hasn't moved, (3) All squares between king and rook are empty, (4) King is not currently in check, (5) King doesn't pass through or land on a square attacked by an opponent piece. The hasMoved flag on Piece tracks conditions 1 and 2.
How would you detect check efficiently?
Find the king's position, then iterate over all opponent pieces (max 16). For each, call isValidPieceMove(piece, piecePos, kingPos). If any returns true, the king is in check. O(16 × pathCheck). For engines: use bitboard attack maps for O(1) check detection.
Why polymorphism (or switch) for piece movement instead of if/else chains?
(1) Each piece type's rules are isolated and self-documenting. (2) Adding a new piece (e.g. for fairy chess) is adding a new case/class, not modifying existing code (Open/Closed). (3) Each piece can be unit-tested independently. (4) Queen's rule is literally isValidRookMove || isValidBishopMove — clean composition.
How would you add an AI opponent?
Minimax algorithm with alpha-beta pruning. The evaluation function scores: material (piece values), position (center control, king safety), mobility (legal move count). Depth 4-6 for real-time play. Enhancement: iterative deepening with transposition tables. Alternatively, integrate Stockfish via UCI protocol behind an AIEngine interface.
How do you handle en passant?
Track the last move. If a pawn moved 2 squares, store the "en passant target" square (the square it passed through). On the next move only, an adjacent enemy pawn can capture diagonally to that target square. The enPassantTarget resets to null after any other move.
How do you handle pawn promotion?
After a pawn moves to the last rank (row 0 for White, row 7 for Black), replace it with a new Piece of the chosen type (Queen/Rook/Bishop/Knight). The makeMove() method takes an optional promotionType parameter. Default to Queen if not specified.
Why Piece[8][8] instead of a HashMap or List?
Array gives O(1) access by position — critical since every move validation queries multiple squares. HashMap would add overhead for hashing/boxing. The 64-element array is cache-friendly and tiny in memory. For professional engines, bitboards (64-bit longs) are even faster for batch operations via bitwise ops.
How would you implement undo/redo?
Command pattern: each Move object stores {piece, from, to, capturedPiece, specialMoveFlags}. Undo = reverse the move and restore the captured piece. Redo = re-execute. A Stack<Move> for undo and a Stack<Move> for redo. The redo stack clears when a new move is made.
How would you handle threefold repetition?
Zobrist hashing: assign a random 64-bit number to each (piece, square) pair. XOR all values for the current board state to get a hash. Maintain a Map<Long, Integer> counting occurrences. If any position occurs 3 times, it's a draw. XOR is incremental: updating the hash for a move is O(1).
How would you scale this for online multiplayer?
Each game is an independent ChessGameImpl with its own lock. WebSocket for real-time move sync. Server-side validation (never trust the client). GameServer manages a ConcurrentHashMap<String, ChessGame>. Chess clock with ScheduledExecutorService for time-out handling. Elo rating updates on game completion.
