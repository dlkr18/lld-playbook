01 — Requirements

Functional

  • User management — register with unique username, suspend users, track reputation
  • Questions — ask with title/body/tags, edit by author, close (no new answers), view count on read
  • Answers — post on open questions, list sorted by accepted-first then vote score
  • Voting — upvote/downvote questions and answers; block self-votes and duplicate votes; allow vote switching
  • Acceptance — question author accepts one answer; switching acceptance reverses prior rep bonus
  • Discovery — keyword search in title/body, browse by tag, sort by vote score

Non-Functional

  • Concurrency — ConcurrentHashMap stores; synchronized per-vote-key for idempotent voting
  • Extensibility — pluggable ReputationPolicy and SearchStrategy
  • Domain invariants — title 15–150 chars, body ≥ 30 chars, 1–5 tags enforced in Question
  • Observability hooks — ReputationListener for badges, notifications, audit logs
Out of scope in this impl: comments, bounties, badges, full-text search engine, auth/session — mention as follow-ups in the interview.

03 — Design Patterns

Strategy — ReputationPolicy

StackOverflowReputationPolicy maps votes to rep (+10/-2) and +15 for accepted answer. Swap for gamification experiments without touching vote logic.

Strategy — SearchStrategy

VoteScoreSearchStrategy filters by keyword (case-insensitive title/body) and sorts by vote count. Replace with Elasticsearch adapter at scale.

Observer — ReputationListener

LoggingReputationListener fires on every rep change. Hook for badge awards, email digests, analytics pipelines.

Facade + Repository

StackOverflow facade wraps StackOverflowService. In-memory ConcurrentHashMap indexes act as repositories.

Domain Invariants (Question aggregate)

  • Title: 15–150 characters after trim
  • Body: minimum 30 characters
  • Tags: 1–5 required; indexed lowercase in service layer
  • Only author can edit; cannot edit when CLOSED
  • Only question author can accept; acceptance blocked on CLOSED questions

05 — Voting & Reputation

Three rules interviewers probe: no self-votes, no duplicate same-direction votes, vote switching must reverse old rep before applying new.

Vote Key & Idempotency

// Composite key: voterId + targetType + targetId
String voteKey = voterId + ":Q:" + questionId;
synchronized (voteKey.intern()) {
    VoteType existing = votes.get(voteKey);
    if (existing == voteType) throw new DuplicateVoteException();
    if (existing != null) {
        target.apply(-existing.getValue());
        applyReputation(authorId, -policy.reputationForVote(existing));
    }
    target.apply(voteType.getValue());
    votes.put(voteKey, voteType);
    applyReputation(authorId, policy.reputationForVote(voteType));
}

Accept Answer (with rep reversal)

synchronized (question) {
    question.getAcceptedAnswerId().ifPresent(prevId -> {
        prev.unmarkAsAccepted();
        applyReputation(prevAuthor, -15);
    });
    question.acceptAnswer(answerId);
    answer.markAsAccepted();
    applyReputation(answerAuthor, +15);
}

06 — Key Algorithms

Answer listing sort

Accepted answer first, then descending vote count — O(n log n) over answers for one question.

Tag browse

tagIndex maps tag name → Set<questionId>. Lookup O(1), then hydrate and sort by votes via SearchStrategy.

Keyword search

Linear scan of in-memory questions filtering title/body contains keyword (case-insensitive). At production scale: inverted index or Elasticsearch with BM25 ranking.

12 — Interview Q&A

How do you prevent a user from voting twice?
Store votes in a map keyed by voterId:targetType:targetId. Before applying, check if the same VoteType exists (DuplicateVoteException). Use synchronized block on the key to prevent TOCTOU races between two concurrent requests from the same voter.
What happens when a user switches from upvote to downvote?
Reverse the old vote's score delta and reputation change first, then apply the new vote. The author sees net rep change of -12 (+10 reversed, -2 applied). Same vote key is updated in place.
Why Strategy for reputation instead of hardcoding?
Stack Overflow's real rules vary by action type, daily caps, and privilege levels. Strategy lets you inject different policies (strict, generous, A/B test) without modifying InMemoryStackOverflowService. Open/Closed Principle.
How would you scale search beyond in-memory scan?
Keep SearchStrategy interface; add ElasticsearchSearchStrategy posting question documents on ask/edit. Tag browse becomes a terms aggregation. Rank by BM25 + vote score boost. Async index updates via message queue.
How do you model comments and bounties?
Comment: child entity under Question/Answer with its own rep-neutral voting. Bounty: value object on Question with expiry; deduct rep on create, award to answer author on accept or auto-expire. Both extend service interface without breaking existing aggregates.
What DB schema would back this?
Tables: users, questions, answers, tags, question_tags (M:N), votes (unique on voter_id + target_type + target_id). Accept answer as nullable FK on questions. Denormalize vote_count for sort or compute via aggregation. Use transactions for accept + rep updates.
How do you handle hot questions with millions of views?
Separate view counter in Redis (INCR), flush to DB periodically. Don't increment on every read in the hot path — sample or batch. CDN-cache rendered HTML for anonymous users.
Why Observer for reputation?
Vote and accept logic shouldn't know about badges, emails, or leaderboards. Listeners subscribe once; service fires onReputationChanged. Add LoggingReputationListener in demo; swap for BadgeAwardListener in production.
Can a suspended user still vote?
Current impl checks suspension on ask/answer only. Mention you'd also block voteQuestion/voteAnswer for suspended users — easy guard at top of castVote(). Shows you think about privilege levels (rep ≥ 125 to downvote, etc.).
How is this different from Reddit LLD?
Stack Overflow has accepted answers, reputation privileges, and tag-based discovery. Reddit is subreddit-scoped feeds with karma but no "accepted" concept. SO emphasizes Q&A quality; Reddit emphasizes community/topic channels.
What about eventual consistency for rep?
In-memory demo is strongly consistent. At scale, emit VoteCast events to Kafka; rep worker aggregates per user. UI shows approximate rep with refresh. Trade-off: simpler write path vs instant rep accuracy.
How do you test voting edge cases?
Unit test castVote: self-vote, duplicate, switch UP→DOWN, concurrent votes from different users, accept then switch accept rep math. Use CountDownLatch for concurrency tests on the vote key lock.
