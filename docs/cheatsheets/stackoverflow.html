01 — Requirements

Functional (MVP)

  • User Management — Register/authenticate, profiles with reputation & badges, activity tracking
  • Question Management — Ask with title, body, tags; edit/delete own; pagination; close by author or moderators
  • Answer Management — Post answers; edit/delete own; accept best answer (author only); multiple answers per question
  • Voting System — Up/downvote Q&A; one vote per user per item; change vote; reputation impact
  • Comment System — Comment on Q&A; edit/delete own; chronological ordering
  • Tagging System — Tag questions (1-5 tags); browse by tag; popular tags tracking
  • Reputation System — +10 question upvote, +10 answer upvote, +15 accepted answer, −2 downvote
  • Search — Keyword search, filter by tags, sort by votes/activity/date

Advanced Features (Extensions)

  • Badge System — Bronze/Silver/Gold achievement awards
  • Bounty System — Offer reputation for answers
  • Moderation — Flag inappropriate content
  • Notifications — Alert users of activity on their content
  • Follow System — Follow questions/users/tags

Non-Functional

  • Scalability — Millions of users, thousands of questions/day, efficient search
  • Performance — Question page < 200ms, search < 500ms, instant vote registration
  • Consistency — Accurate reputation, consistent vote counts, no duplicate votes
  • Availability — 99.9% uptime, graceful degradation
  • Extensibility — Pluggable reputation rules, customizable badges, new content types
  • Data Integrity — Audit trail for votes, no orphaned content, referential integrity

Out of Scope

  • Real-time chat, video/image hosting (external CDN), email notifications (external service)
  • Payment processing for bounties, complex spam detection (external service), mobile apps

03 — Design Patterns

Clean DDD patterns applied to the Q&A domain.

Aggregate Root

User and Question are aggregate roots. Question owns its Answers and Comments. All mutations go through the aggregate root to enforce invariants.

Type-Safe IDs

UserId, QuestionId, AnswerId — immutable final classes wrapping long. Prevents accidentally passing a QuestionId where a UserId is expected. Validated positive at construction.

Value Objects

Tag, Vote, Comment, Badge are immutable value objects. Tag enforces naming rules (lowercase, alphanumeric + hyphens, 2-35 chars). Equality by value, not identity.

Rich Domain Model

Entities contain validation logic: Question.validateTitle(), User.validateUsername(), Answer.validateBody(). Business rules enforced at domain layer, not in services.

QuestionStatus Enum

public enum QuestionStatus {
    OPEN,     // accepting answers, votes, comments; searchable
    CLOSED,   // no new answers; can still vote/comment; still searchable
    DELETED   // removed from all views
}

Key Transitions in Question.java

public void edit(String newTitle, String newBody) {
    if (status != QuestionStatus.OPEN)
        throw new IllegalStateException("Cannot edit closed or deleted question");
    this.title = validateTitle(newTitle);
    this.body = validateBody(newBody);
    this.updatedAt = LocalDateTime.now();
}

public void acceptAnswer(AnswerId answerId) {
    if (status == QuestionStatus.CLOSED)
        throw new IllegalStateException("Cannot accept answer on closed question");
    this.acceptedAnswerId = answerId;
}

public void close()   { this.status = QuestionStatus.CLOSED; }
public boolean isOpen() { return status == QuestionStatus.OPEN; }
public boolean hasAcceptedAnswer() { return acceptedAnswerId != null; }

06 — Tags & Search System

ADR: Tags as Separate Entities — Tags are first-class navigation. Separate Tag entity with many-to-many relationship enables tag pages, statistics, suggestions, and consistent naming.

Tag Validation

private String validateName(String name) {
    String trimmed = name.trim().toLowerCase();
    if (trimmed.length() < 2 || trimmed.length() > 35)
        throw new IllegalArgumentException("Tag name must be 2-35 characters");
    if (!trimmed.matches("^[a-z0-9-]+$"))
        throw new IllegalArgumentException("Lowercase letters, numbers, and hyphens only");
    return trimmed;
}

Tags are equality-by-name value objects. equals() and hashCode() based on name field only.

07 — Key Algorithms & Code

Question Validation (Aggregate Root)

public Question(QuestionId id, String title, String body,
               UserId authorId, Set<Tag> tags) {
    this.id       = Objects.requireNonNull(id, "QuestionId cannot be null");
    this.title    = validateTitle(title);   // 15-150 chars
    this.body     = validateBody(body);     // min 30 chars
    this.authorId = Objects.requireNonNull(authorId);
    this.tags     = validateTags(tags);     // 1-5 tags
    this.voteCount = 0;
    this.viewCount = 0;
    this.status    = QuestionStatus.OPEN;
}

private Set<Tag> validateTags(Set<Tag> tags) {
    if (tags == null || tags.isEmpty())
        throw new IllegalArgumentException("Must have at least one tag");
    if (tags.size() > 5)
        throw new IllegalArgumentException("Cannot have more than 5 tags");
    return new HashSet<>(tags); // defensive copy
}

public void markAsAccepted() {

User Validation & Registration

private String validateUsername(String username) {
    String trimmed = username.trim();
    if (trimmed.length() < 3 || trimmed.length() > 20)
        throw new IllegalArgumentException("Username must be 3-20 characters");
    if (!trimmed.matches("^[a-zA-Z0-9_]+$"))
        throw new IllegalArgumentException("Letters, numbers, and underscores only");
    return trimmed;
}

private String validateEmail(String email) {
    String trimmed = email.trim().toLowerCase();
    if (!trimmed.matches("^[A-Za-z0-9+_.-]+@(.+)$"))
        throw new IllegalArgumentException("Invalid email format");
    return trimmed;
}

Type-Safe ID Pattern

public final class UserId {
    private final long value;

    public UserId(long value) {
        if (value <= 0)
            throw new IllegalArgumentException("UserId must be positive");
        this.value = value;
    }

    // equals() and hashCode() based on value
    // Same pattern for QuestionId, AnswerId, CommentId, etc.
}

Why? Compile-time safety. void acceptAnswer(QuestionId qId, AnswerId aId) cannot have its arguments swapped by accident.

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap All in-memory stores (users, questions, votes, tags) Safe concurrent reads/writes without global lock
synchronized(question) Vote application, answer acceptance Fine-grained lock per question — two different questions can be modified concurrently
Collections.unmodifiableSet Question.getTags() Defensive copy prevents callers from modifying internal tag set
final fields All ID classes (UserId, QuestionId, AnswerId) Immutable by design — thread-safe without synchronization
Objects.requireNonNull All constructors Fail-fast on null arguments, prevents NPE downstream
UNIQUE constraint (DB) votes(user_id, target_type, target_id) Database-level prevention of duplicate votes
"Vote count is denormalized on Question/Answer for fast reads, but the individual Vote entities are the source of truth. This allows one vote per user enforcement and audit trails."

+15 accepted answer bonus. Expert reputation: 51 + 15 = 66.
Handle Downvote
expert.addReputation(VoteType.DOWNVOTE.getReputationChange())
−2 reputation. Expert reputation: 66 − 2 = 64. Floor at 0 (can never go negative).
Check Privileges
expert.hasReputation(100) → false
expert.hasReputation(50) → true

    id BIGSERIAL PRIMARY KEY,

    email VARCHAR(255) UNIQUE NOT NULL,

CREATE TABLE answers (
    id BIGSERIAL PRIMARY KEY,

    body TEXT NOT NULL,

);

CREATE TABLE votes (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES users(id),

);

CREATE TABLE comments (
    id BIGSERIAL PRIMARY KEY,

    author_id BIGINT REFERENCES users(id),
    parent_type VARCHAR(20) NOT NULL,

);

Performance Optimizations

  • Denormalization — vote_count on Question/Answer, answer_count on Question, question_count on Tag
  • Composite indexes — (tag_id, vote_count DESC) for popular questions by tag
  • Full-text index — on question title + body for search
  • Read replicas — route reads to replicas, master for writes only
  • Sharding — by user_id for user data, by question_id for questions

12 — Interview Follow-up Q&A

Why store votes as separate entities instead of just a counter?
ADR: Business rules require (1) preventing duplicate votes via unique constraint, (2) showing user's own vote status, (3) supporting vote changes (upvote ↔ downvote), and (4) audit trail. Counter alone can't enforce these. We denormalize vote_count on Question/Answer for fast reads, but individual Vote entities are the source of truth.
How would you prevent duplicate votes at scale?
Database UNIQUE constraint on (user_id, target_type, target_id). In-memory: composite key in ConcurrentHashMap. At the application layer, check-then-act inside synchronized(question) to prevent race conditions. The database constraint is the ultimate safeguard.
Why incremental reputation instead of on-demand calculation?
User profiles are displayed frequently — recalculating by aggregating all votes every time is O(n) and DB-intensive. Incremental updates are O(1) per event. Trade-off: needs careful event handling and periodic reconciliation to correct drift from edge cases (e.g., deleted content).
Why type-safe IDs instead of just long/String?
Prevents accidentally passing a QuestionId where a UserId is expected — caught at compile time. Encapsulates validation (must be positive). Makes method signatures self-documenting: acceptAnswer(QuestionId, AnswerId, UserId) is unambiguous.
How would you handle simultaneous answer acceptance?
synchronized(question) — only one thread can modify a question at a time. The first thread sets acceptedAnswerId; the second finds it already set and either throws or replaces. At the DB level, an optimistic lock (version column) can also prevent lost updates.
Why are Tags separate entities instead of plain strings on Question?
ADR: Tags are first-class navigation. Separate entity enables: (1) tag metadata (description, count), (2) consistent naming (validation rules), (3) efficient tag-based queries via junction table, (4) tag auto-suggestions, (5) tag statistics pages. Worth the schema complexity.
How would you implement search at scale?
ADR: Dedicated search engine (Elasticsearch). Provides excellent relevance ranking, fast full-text search, faceted search by tags, and rich query capabilities. Index questions asynchronously. Accept eventual consistency — a newly posted question may take seconds to appear in search. The SearchService interface stays the same regardless of backend.
How do you handle the reputation floor?
addReputation(int points) clamps to 0: reputation = Math.max(0, reputation + points). This prevents negative reputation from griefing/abuse. All users start at 1 reputation. The floor is enforced in the User domain object, not in the service layer.
Why answers as separate entities instead of embedded in Question?
ADR: Questions can have 100+ answers. Embedded storage causes (1) unbounded document size, (2) hard to query answers independently, (3) update conflicts when multiple users answer simultaneously. Separate table allows bounded reads, independent operations, and better concurrency.
How would you add a bounty system?
New Bounty entity with amount, expiry, and status. Question gets an optional bountyId. User's reputation is deducted when bounty is created (escrowed). On expiry, auto-award to highest-voted answer or return to asker. Fits naturally into the existing reputation system.
How would you handle content moderation and flagging?
New Flag entity (flaggerId, contentId, reason, status). When flag count exceeds threshold, content enters a review queue visible to users with moderator reputation (10,000+). Moderators can close, delete, or dismiss flags. Automated spam detection can be a pluggable strategy.
What about the "only author can accept" rule — how is it enforced?
At the service layer: if (!question.getAuthorId().equals(userId)) throw new AuthorizationException("Only question author can accept"). The domain model enforces the "cannot accept on closed question" rule. The service enforces the "only author" rule because it requires knowing the calling user's identity.
How do you ensure data integrity with denormalized vote counts?
The Vote entity is the source of truth. Denormalized voteCount on Question/Answer is updated atomically inside the same synchronized block as the Vote creation/update. Periodic reconciliation job: UPDATE questions SET vote_count = (SELECT SUM(value) FROM votes WHERE target_id = question_id).

14 — Patterns Summary

PatternWhereWhy
Aggregate Root User, Question Enforce invariants; all mutations through root entity
Type-Safe ID UserId, QuestionId, AnswerId Compile-time safety, no ID mixing, self-documenting APIs
Value Object Tag, Vote, Comment, Badge, all IDs Immutable, equality by value, thread-safe by design
Rich Domain Model Validation in entities (User, Question, Answer, Tag) Business rules at domain layer, not leaked to services
Repository ConcurrentHashMap stores In-memory, easy to swap for JPA/DB implementation
Defensive Copy Question.getTags() → unmodifiableSet Protect internal state from external mutation
Fail-Fast Validation Objects.requireNonNull, all validate*() methods Catch invalid state at construction, not at use time
