Cricinfo System // Live Scoreboard

Low-level design for a live cricket scoring platform with ball-by-ball updates, player statistics, match lifecycle, and real-time commentary. Complex domain modeling at its best.

01 — Requirements

Functional

  • Match Management — Create match (teams, venue, date, format), start, toss, innings, declare winner
  • Score Tracking — Record ball outcome (runs, wicket, extra, boundary), track overs (6 legal balls), manage innings
  • Player Statistics — Batting (runs, balls faced, 4s, 6s, strike rate), bowling (overs, maidens, wickets, economy), fielding (catches, run-outs)
  • Commentary & Updates — Ball-by-ball commentary, key events (wicket, boundary, milestone), partnerships, fall of wickets
  • Team Management — Playing XI selection, squad management, batting order, bowling rotation
  • Match Formats — Test (4 innings), ODI (50 overs), T20 (20 overs) with different rules

Non-Functional

  • Performance — Real-time score updates < 1 second latency
  • Concurrency — Handle millions of concurrent viewers
  • Scalability — Support 1000+ matches per day globally
  • Reliability — 99.99% uptime during live matches

Key Challenges

  • Complex Scoring — Runs, wickets, extras (byes, leg-byes, wides, no-balls)
  • Match States — Innings transitions, overs, partnerships, fall of wickets
  • Hierarchical Data — Match → Innings → Over → Ball
  • Tournament Management — Series, leagues, knockout formats

03 — Design Patterns

Five patterns, each solving a specific cricket domain challenge.

Observer Pattern

Real-time score updates to millions of subscribers. Scoreboard, commentary feed, and notification channels subscribe to match events. Ball delivered → all observers notified.

State Pattern

Match states (SCHEDULED, IN_PROGRESS, COMPLETED, CANCELLED). Each state defines which transitions are legal. Invalid transitions throw immediately.

Strategy Pattern

Different match formats (Test, ODI, T20) have different rules — max overs, innings count, DLS calculations. Swap format strategy without changing scoring logic.

Composite Pattern

Match → Innings → Overs → Balls. Hierarchical tree where each level aggregates stats from its children. Total score = sum of innings scores = sum of over scores = sum of ball scores.

Builder Pattern

Build complex Match objects step by step — set teams, venue, format, date, umpires. Prevents incomplete Match construction.

04 — Match Lifecycle

Why State? A match transitions through well-defined states. Each state determines legal operations. Starting a COMPLETED match? Throws. Scoring on a SCHEDULED match? Throws. Clean, self-documenting.

Match State Guards (from CricinfoServiceImpl)

public void startMatch(String matchId) {
    Match match = matches.get(matchId);
    if (match == null)
        throw new IllegalArgumentException("Match not found");
    if (match.getStatus() != MatchStatus.SCHEDULED)
        throw new IllegalStateException("Cannot start -- status: " + match.getStatus());
    match.start();
}

public void endMatch(String matchId, String winnerId) {
    Match match = matches.get(matchId);
    if (match.getStatus() != MatchStatus.IN_PROGRESS)
        throw new IllegalStateException("Cannot end -- status: " + match.getStatus());
    Team winner = teams.get(winnerId);
    match.end(winner); // sets COMPLETED, records win/loss
}

Over Completion Logic

public boolean isComplete() {
    int legalBalls = 0;
    for (Ball ball : balls) {
        if (ball.getExtra() != ExtraType.WIDE
            && ball.getExtra() != ExtraType.NO_BALL) {
            legalBalls++;
        }
    }
    return legalBalls >= 6;
}

// Wides and no-balls are NOT legal deliveries
// They add to team score but don't count toward over completion

End-of-Over Actions

  • Change bowler (same bowler cannot bowl consecutive overs)
  • Swap striker & non-striker
  • Check if innings is complete (all overs bowled in limited overs)
  • Check for maiden over (0 runs conceded off bat)

06 — Scoring System

Ball Delivery Flow: Every ball triggers a cascade of updates — team score, batsman stats, bowler stats, partnership, commentary, and over tracking.

Recording a Ball

public void recordBall(Ball ball) {
    currentOver.addBall(ball);

    // 1. Update team score
    currentInnings.addRuns(ball.getRuns());

    // 2. Update batsman score
    ball.getStriker().getBattingStats().addRuns(ball.getRuns(), ball.isBoundary());

    // 3. Update bowler stats
    ball.getBowler().getBowlingStats().addBall(ball);

    // 4. Handle wicket
    if (ball.isWicket()) {
        handleWicket(ball.getWicket());
        currentInnings.incrementWickets();
    }

    // 5. Handle extras (wide, no-ball, bye, leg-bye)
    if (ball.getExtra() != null) {
        handleExtra(ball);
    }

    // 6. Check over complete (6 legal balls)
    if (currentOver.isComplete()) { endOver(); }

    // 7. Check innings complete (all out or overs done)
    if (isInningsComplete()) { endInnings(); }
}

Ball.java — The Atomic Unit

public class Ball {
    private final int overNumber;
    private final int ballNumber;
    private final Player bowler;
    private final Player batsman;
    private int runs;
    private boolean isWicket;

    public String toString() {
        String result = overNumber + "." + ballNumber + ": "
            + batsman.getName() + " scored " + runs;
        if (isWicket) result += " (OUT!)";
        return result;
    }
}

08 — Live Updates (Observer Pattern)

Why Observer? Millions of viewers need real-time updates. Scoreboard, commentary panel, mobile notifications, and analytics systems all subscribe to match events. Adding a new subscriber = zero changes to scoring logic.

Observer Architecture

interface ScoreObserver {
    void onScoreUpdate(Match match, Ball ball);
    void onWicket(Match match, Wicket wicket);
    void onOverComplete(Match match, Over over);
    void onInningsComplete(Match match, Innings innings);
    void onMatchComplete(Match match);
}

// Subscribers
class ScoreboardDisplay  implements ScoreObserver { /* live TV overlay */ }
class CommentaryFeed     implements ScoreObserver { /* text commentary */ }
class MobileNotifier     implements ScoreObserver { /* push notifications */ }
class StatisticsEngine   implements ScoreObserver { /* analytics pipeline */ }
EventWho Observes
Ball scoredScoreboard (update total), Commentary (add line), Stats (recalculate)
Wicket fallsScoreboard (update wickets), Mobile (push "WICKET!"), Commentary (detailed)
Over completeScoreboard (update overs), Commentary (over summary)
Innings completeAll observers (innings break, target set)
Match completeAll observers (result, man of the match)

Firing Events

// In Match (the Subject):
private final List<ScoreObserver> observers = new CopyOnWriteArrayList<>();

public void recordBall(Ball ball) {
    // ... scoring logic ...
    for (ScoreObserver obs : observers) {
        obs.onScoreUpdate(this, ball);
    }
    if (ball.isWicket()) {
        for (ScoreObserver obs : observers) {
            obs.onWicket(this, ball.getWicket());
        }
    }
}

Key insight: Scoring logic is decoupled from display logic. Add WebSocket push, Kafka events, or SMS alerts without touching the scoring engine.

09 — Key Algorithms

Required Run Rate

public double getRequiredRunRate(int target, int currentScore, double remainingOvers) {
    if (remainingOvers <= 0) return Double.MAX_VALUE;
    return (target - currentScore) / remainingOvers;
}

// Example: Target 280, current 120/3 after 25 overs
// Required RR = (280 - 120) / 25 = 6.40 runs/over

Current Run Rate

public double getCurrentRunRate(int totalRuns, double oversBowled) {
    if (oversBowled == 0) return 0.0;
    return totalRuns / oversBowled;
}

Maiden Over Detection

public boolean isMaidenOver(Over over) {
    return over.isComplete() && over.getRuns() == 0;
}

// All 6 legal balls, 0 runs off the bat
// Updates bowler's maidens count

Innings Completion Check

public boolean isInningsComplete() {
    // All out (10 wickets in normal cricket)
    if (currentInnings.getWickets() >= 10) return true;

    // Overs exhausted (limited overs format)
    if (matchType != MatchType.TEST
        && currentInnings.getOversCompleted() >= maxOvers) return true;

    // Target chased (2nd innings)
    if (isSecondInnings()
        && currentInnings.getTotalRuns() > target) return true;

    // Declaration (Test match only)
    if (currentInnings.isDeclared()) return true;

    return false;
}

Strike Rotation (Odd/Even Runs)

// After each ball, determine if striker and non-striker swap
if (runs % 2 == 1) {
    swapStrikerNonStriker(); // 1 or 3 runs: change ends
}
// At end of over, they swap again
// Net effect: odd runs in last ball = same striker next over

10 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap teams, players, matches stores Safe concurrent reads/writes for millions of viewers
synchronized(match) recordBall, startMatch, endMatch Only one scorer updates a specific match at a time
CopyOnWriteArrayList Match.observers Safe iteration while adding/removing score observers
volatile Match.status Cross-thread visibility of match state changes
Immutable objects Ball, Wicket, BattingStats snapshots Thread-safe reads without locking
"Score updates come from a single scorer per match (sequential via synchronized), but millions of read requests are served concurrently. This read-heavy workload is ideal for ConcurrentHashMap."

14 — Interview Follow-up Q&A

How do you handle concurrent score updates from multiple scorers?
synchronized(match) ensures only one scorer thread updates a specific match at a time. Reads via ConcurrentHashMap serve millions of viewers concurrently without blocking the scorer. In production, a message queue (Kafka) serializes score events per match partition.
How do you calculate the required run rate?
(Target - CurrentScore) / RemainingOvers. If remainingOvers is 0, the match is over. In DLS (rain-affected) matches, the calculation adjusts based on resources remaining — that would be a separate DLSStrategy implementing a scoring strategy interface.
How do you detect a maiden over?
An over is maiden if isComplete() && getRuns() == 0. Only runs off the bat count — byes and leg-byes don't break a maiden, but wides and no-balls do. The bowler's BowlingStats.maidens counter increments.
Why Composite pattern for Match → Innings → Over → Ball?
Each level aggregates from its children: Match total = sum of Innings totals = sum of Over totals = sum of Ball runs. You can query at any level — "score at over 15" or "runs in this partnership" — without recalculating from raw balls. It also maps naturally to the cricket domain hierarchy.
How would you handle different match formats (Test vs ODI vs T20)?
Strategy Pattern. A MatchFormatStrategy interface defines getMaxOvers(), getMaxInnings(), getPowerplayRules(), isInningsComplete(). Test has unlimited overs and 4 innings, ODI has 50 overs and 2 innings, T20 has 20 overs. Inject the right strategy at match creation.
How do you handle extras (wides, no-balls)?
Extras add to the team score but wides and no-balls are not legal deliveries — they don't count toward the 6-ball over limit. Byes and leg-byes count as legal deliveries but don't credit the batsman. The Ball entity stores an ExtraType enum and the scoring logic branches accordingly.
How would you implement ball-by-ball commentary?
Each Ball entity has a commentary field. A CommentaryGenerator (Strategy) produces text based on ball outcome. The Observer pattern pushes commentary to subscribers. Key events (wicket, 50, 100, boundary) trigger richer commentary templates.
How would you track partnerships?
A Partnership entity tracks two batsmen, runs scored, balls faced, and which batsman scored what. When a wicket falls, the current partnership ends and a new one starts. Fall of wickets timeline is a List<Partnership> on the Innings.
How would you scale to millions of concurrent viewers?
Write path: single scorer per match (serialized). Read path: CQRS — separate read replicas. Score updates published to Kafka, consumed by read-side projections. WebSocket or Server-Sent Events push to clients. CDN for static scorecards. Redis cache with <1s TTL for hot matches.
How would you handle DLS (Duckworth-Lewis-Stern) in rain-affected matches?
DLS is a Strategy: DLSScoringStrategy adjusts the target based on overs lost and wickets in hand. The resource table maps (overs remaining, wickets lost) → percentage of resources. Inject DLS strategy when rain interrupts. The match's target is recalculated dynamically.
Why Observer over polling for live scores?
Polling wastes bandwidth when no updates. Observer (push) delivers updates instantly on every ball. At scale, use WebSocket for persistent connections or SSE for one-way push. Kafka decouples the scorer from millions of consumers. Each consumer group processes at its own pace.
How would you store historical match data?
In-memory for live matches (hot data). After completion, persist to a relational DB (Match, Innings, Over, Ball tables with foreign keys). Player career stats aggregated in a materialized view. Historical queries use read replicas. Ball-by-ball data enables replay and deep analytics.
