Security in HLD // Concept

Who's calling, what are they allowed to do, and is the data encrypted in transit and at rest. Layered defense: TLS at the edge, mTLS inside the mesh, AuthN at the gateway, AuthZ at every service, secrets in a vault, audit at the end.

AuthN / AuthZ OAuth 2.0 / OIDC JWT mTLS Zero Trust

Layered defense: WAF + TLS at edge → AuthN at gateway → mTLS in mesh → AuthZ at service → encryption at rest via KMS → immutable audit log feeding SIEM.

01 — The Problem

Every distributed system has three irreducible security questions:

  1. AuthN — who is calling? (humans, services, devices)
  2. AuthZ — what are they allowed to do? (per resource, per action)
  3. Confidentiality / integrity — can anyone read or tamper with the data on the wire or at rest?
You can't bolt this on after launch. The blast radius of getting it wrong is regulatory, financial, and reputational at once.

03 — AuthN Approaches

ApproachHowBest for
Session cookieServer creates session, returns opaque ID in Set-Cookie; cookie sent on every request; server looks up session store.Classic web apps. Needs CSRF defense.
JWT bearerSigned token in Authorization: Bearer .... Stateless — receiver validates signature.SPAs, mobile, service-to-service.

Session vs JWT trade-off: session cookies are easy to revoke (delete row) but require a shared session store; JWTs are stateless and distributed but hard to revoke before expiry (need a blocklist or short TTL).

04 — AuthZ: RBAC vs ABAC

RBAC: user -> role(s) -> permissions
  user Alice has role "editor"
  role "editor" can [read, write] on resource type "doc"

ABAC: policy is a predicate
  allow if subject.department == resource.department
         and action in ["read","write"]
         and request.time.hour between 9 and 18

ReBAC (Zanzibar / Google Docs / GitHub):
  edges in a graph: user --(editor)--> doc
  check = reachability query
RBACABACReBAC
ComplexityLowMediumHigh
FlexibilityCoarseFineGraph-native (sharing)
Used byK8s, AWS IAM (mostly)OPA / Cedar policiesGoogle Zanzibar, GitHub, Auth0 FGA
For interviews: name RBAC by default, mention ABAC for fine-grained policy, mention ReBAC/Zanzibar if the system is sharing-heavy (docs, repos, photos).

06 — JWT Deep Dive

header.payload.signature   (base64url, dot-separated)

header  = { "alg": "RS256", "kid": "key-2026-04" }
payload = { "sub": "user-42", "iss": "auth.example.com",
            "aud": "api.example.com",
            "iat": 1700000000, "exp": 1700003600,
            "scope": "orders:read orders:write" }
signature = RSA-sign(SHA256(header.payload), private_key)

Receiver:
  1. fetch JWKS from /.well-known/jwks.json (cache it)
  2. pick key by `kid`
  3. verify signature
  4. check exp, iss, aud
  5. enforce scope / claims

Pros

  • Stateless — no DB lookup per request
  • Distributed — any service can verify with the public key
  • Self-contained — carries claims, scopes

Cons

  • Revocation is hard — can't take back a signed token before exp without a blocklist (which kills statelessness)
  • Size — sent on every request; 1–4 KB easily
  • Key rotation — kid + JWKS endpoint with overlapping validity
  • Footgun: alg: none — old libraries accepted unsigned tokens. Always pin allowed algorithms.
Best practice: short TTL (15–60 min) + refresh tokens. If you need instant revocation (logout, password change), maintain a small blocklist of jti claims, TTL = max access-token lifetime.

07 — TLS & mTLS

TLS handshake (1.3, simplified):
  client hello (ciphers, key_share)
  server hello (cert chain, key_share, finished)
  client (finished)
  -> encrypted application data

mTLS adds:
  server requests client cert
  client sends cert + signs nonce
  server verifies cert chain (and SPIFFE ID)

Where to terminate TLS

StrategyHowTrade-off
Edge terminationTLS ends at the load balancer; HTTP inside the trust boundary.Simple, cheap. Network must be trusted.
End-to-end TLSLB re-establishes TLS to the backend.Slightly slower; defends against in-VPC sniffing.
mTLS in service meshSidecar (Envoy) handles cert rotation; app sees plaintext locally.Zero-trust networking. See Service Mesh.

Cert rotation: SPIFFE/SPIRE or cert-manager rotates every ~24h automatically; long-lived certs are an antipattern.

08 — Secrets & KMS

Where NOT to put secrets

  • Source code (Git history is forever)
  • Plain env vars baked into images
  • Slack, Notion, wikis

Where to put secrets

ToolNotes
HashiCorp VaultDynamic secrets (DB creds generated per app), broker-style
AWS Secrets Manager / Parameter StoreIAM-gated; auto rotation hooks
K8s Secrets + Sealed SecretsEncrypted at rest in etcd; sealed-secrets uses asymmetric encryption to commit safely
Cloud KMS (AWS KMS, GCP KMS, Azure Key Vault)Hosts the KEK; FIPS 140-2 HSMs available

Envelope encryption (the canonical pattern)

1. Generate DEK (data encryption key) per object
2. Encrypt object with DEK
3. Encrypt DEK with KEK (in KMS)  -> wrapped_DEK
4. Store: ciphertext + wrapped_DEK

To decrypt:
  KMS.Decrypt(wrapped_DEK) -> DEK
  Decrypt(ciphertext, DEK) -> plaintext

Why: KMS only sees small wrapped DEKs, never the full data. KEK never leaves HSM.

11 — Zero Trust

Old model: "castle and moat" — the LAN is trusted, the internet is hostile. Zero trust says nothing is trusted; identity and policy decide every call.

  • Authenticate every call (mTLS or signed JWT), even inside the VPC
  • Authorize every call with policy as code (OPA, Cedar)
  • Continuously verify — certs rotate, sessions are short, posture is checked
  • Least privilege — IAM scoped per service, no wildcard * roles
  • Audit everything — immutable append-only log to a separate account
The principle that earns the most respect in an interview: "Assume the network is compromised. mTLS between every service. Audit every call."

13 — Common Pitfalls

  • Confusing AuthN with AuthZ — "logged in" is not "allowed". Always check both.
  • JWT alg: none — pin allowed algorithms in the verifier.
  • Long-lived bearer tokens — 1-year tokens get exfiltrated. Short TTL + refresh.
  • Secrets in env vars committed to Helm values — use sealed-secrets or external secret operators.
  • Encrypting only disk — doesn't stop an attacker with DB creds.
  • Rolling your own crypto — use libsodium, AWS Encryption SDK, Tink. Never custom AES wiring.
  • Trusting X-Forwarded-For from the open internet — only trust it from your own LB.
  • Forgetting to revoke on logout — pure stateless JWT can't, so add a small blocklist.
  • No cert rotation plan — the 2020 outages were certificates that expired.

14 — Interview Q&A

Session cookie vs JWT — which would you pick?
For a classic web app with a small number of backends, session cookie with SameSite=Lax + CSRF token + Redis session store. For a multi-service SPA or mobile app spanning many backends, JWT with short TTL (15 min) + refresh token rotation. The deciding question is "how do I revoke?" — if instant revocation matters, sessions or short JWT + blocklist.
How do you handle JWT revocation?
Three options: (1) Keep TTL short (15 min) so worst case is 15 min of valid stolen token. (2) Maintain a blocklist of jti claims keyed by user, TTL = max access-token lifetime. (3) Switch to opaque tokens with introspection (RFC 7662) — every request checks with the auth server. Pick (1)+(2) most commonly.
Why mTLS instead of just JWTs between services?
JWTs prove "the caller has a token"; mTLS proves "the caller is this specific workload" via cryptographic identity bound to a cert. mTLS also encrypts at the channel level — no need to terminate TLS at the LB and re-encrypt. In practice you do both: mTLS for transport + identity, JWT in the body to carry user context end-to-end.
Walk through envelope encryption.
Generate a per-object data encryption key (DEK). Encrypt the data with DEK. Encrypt the DEK with the key encryption key (KEK) in KMS. Store ciphertext + wrapped DEK side by side. To read: ask KMS to unwrap DEK (audit logged), then decrypt locally. KEK never leaves the HSM, KMS sees only small DEKs — low blast radius if any one DEK leaks.
What's the difference between RBAC and ABAC?
RBAC: users have roles, roles have permissions. Coarse, easy to audit. ABAC: policies are predicates over attributes (subject, resource, action, environment). Fine-grained, harder to audit. Real systems use RBAC for the 80% and ABAC for the long tail (location, time, owner). For social/sharing systems (Docs, GitHub), ReBAC (Zanzibar) models access as a graph.
How do you store passwords?
argon2id (or bcrypt cost 12+ / scrypt) with per-user random salt and optionally a server-side pepper. Tune cost so a single hash takes ~100ms on the production CPU. Never MD5, SHA-1, or raw SHA-256. Migrate by rehashing on next login.
How do you protect against CSRF?
First defense: SameSite=Lax (or Strict) on session cookies — browsers won't send them on cross-site state-changing requests. Second: anti-CSRF token in form / header that the server validates. Third: for SPAs using bearer tokens in Authorization, CSRF is structurally impossible because cross-origin code can't read the token from localStorage across origins.
How does PCI affect your design?
Card data must be encrypted in transit and at rest, accessible only to PCI-scoped components. The trick is tokenisation: a small isolated vault stores real PANs and returns opaque tokens; the rest of your system handles tokens and is descoped from PCI. See Payment System.
What's a JWKS endpoint and why use one?
JSON Web Key Set — an HTTPS endpoint (e.g. /.well-known/jwks.json) listing the issuer's public keys keyed by kid. Verifiers fetch and cache it. Key rotation is just publishing a new kid and signing new tokens with it while old ones remain verifiable until they expire. No code redeploy, no shared secrets to distribute.
How do you handle secret rotation?
(1) Use a secrets manager (Vault, AWS SM) that supports rotation hooks. (2) Issue overlapping credentials so old and new are both valid during the rollout window. (3) Apps fetch secrets at startup AND on a timer (e.g. every 1h); on rotation, new requests use new secrets while in-flight finish with old. (4) Force rotation on every leak suspicion — cheap if automated.
