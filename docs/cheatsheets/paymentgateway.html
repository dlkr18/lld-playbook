01 — Requirements

Functional

  • Process payments via multiple methods: Credit/Debit Card, UPI, Wallet, Net Banking
  • Transaction lifecycle: PENDING → SUCCESS / FAILED → REFUNDED
  • Merchant and customer management (registration, lookup)
  • Refund processing — full or partial refunds against completed transactions
  • Transaction status queries and payment history
  • Idempotent payments — prevent double-charge on retries
  • Notification support (webhook callbacks, email confirmations)
  • Reporting and reconciliation

Non-Functional

  • Performance — response time < 100ms for critical payment operations
  • Security — PCI-DSS compliance, card data encryption, tokenization
  • Scalability — support 10,000+ concurrent transactions
  • Reliability — 99.9% uptime, no lost transactions
  • Data Consistency — ACID for payment writes, eventual consistency for notifications
  • Idempotency — every payment request carries an idempotency key

03 — Design Patterns

The interview differentiator — four patterns, each earning its place.

Strategy Pattern

Payment method processing. Each PaymentMethod (Card, UPI, Wallet, Net Banking) is a strategy with its own validation, authorization, and settlement logic. New payment method = new class.

State Pattern

Transaction lifecycle. Each TransactionStatus defines exactly which transitions are legal. PENDING can → SUCCESS or FAILED. SUCCESS can → REFUNDED. Invalid ops throw immediately.

Observer Pattern

Notifications on payment events. Merchant gets webhook callback, customer gets email/SMS confirmation, analytics service logs the event. All decoupled from core processing.

Factory Pattern

Payment processor creation. Given a PaymentMethod enum, the factory returns the correct processor (CardProcessor, UPIProcessor, etc.) without exposing creation logic.

04 — Transaction State Machine

Why State? A payment can only transition through well-defined states. State pattern eliminates scattered if/else, makes illegal transitions impossible, and adding a new status (e.g., PROCESSING, ON_HOLD) = adding a new class. Open/Closed Principle. 
stateDiagram-v2
    [*] --> PENDING: processPayment()
    PENDING --> SUCCESS: authorize + capture
    PENDING --> FAILED: decline / timeout
    SUCCESS --> REFUNDED: processRefund()
    FAILED --> [*]
    REFUNDED --> [*]
    SUCCESS --> [*]

Refund State Machine

stateDiagram-v2
    [*] --> PENDING: refund requested
    PENDING --> PROCESSED: bank confirms
    PENDING --> FAILED: bank rejects
    PROCESSED --> [*]
    FAILED --> [*]

public enum TransactionStatus {

    FAILED,      // declined, timeout, insufficient funds

06 — Retry & Idempotency

Critical for payment systems: Network failures, timeouts, and retries MUST NOT result in double charges. Every payment request must carry an idempotency key.

Idempotency Key Flow

// Client sends: POST /payments { idempotency_key: "abc-123", ... }

public Transaction processPayment(String idempotencyKey, ...) {
    // 1. Check if we've already processed this key
    Transaction existing = idempotencyStore.get(idempotencyKey);
    if (existing != null) {
        return existing;  // return cached result, no re-processing
    }

    // 2. Process payment normally
    Transaction txn = doProcess(merchantId, customerId, amount, method);

    // 3. Store result keyed by idempotency key
    idempotencyStore.put(idempotencyKey, txn);
    return txn;
}

Retry Strategy

AttemptDelayAction
1st0msImmediate attempt
2nd1sExponential backoff
3rd4sExponential backoff
4th16sFinal attempt
After max—Mark FAILED, alert ops

Key: Retries use the SAME idempotency key, so duplicate processing is impossible even if multiple retry threads fire concurrently.

07 — Key Algorithms

Payment Processing Pipeline

Time Complexity: O(1) per payment · Space Complexity: O(n) transactions stored

public Transaction processPayment(String mid, String cid,
                                  double amt, PaymentMethod method) {
    // 1. Generate unique transaction ID
    String id = UUID.randomUUID().toString();

    // 2. Create transaction in PENDING state
    Transaction t = new Transaction(id, mid, cid, amt);

    // 3. Validate payment method (Strategy)
    validatePaymentMethod(method, details);

    // 4. Authorize with payment processor
    boolean authorized = authorize(method, amt);

    // 5. Transition state
    t.setStatus(authorized
        ? TransactionStatus.SUCCESS
        : TransactionStatus.FAILED);

    // 6. Store and return
    transactions.put(id, t);
    return t;
}

Refund Processing

public Refund processRefund(String transactionId, double amount) {
    // 1. Lookup original transaction
    Transaction txn = transactions.get(transactionId);
    if (txn == null) throw new TransactionFailedException("Not found");

    // 2. Validate: only SUCCESS transactions can be refunded
    if (txn.getStatus() != TransactionStatus.SUCCESS)
        throw new TransactionFailedException("Cannot refund");

    // 3. Validate: refund amount ≤ original amount
    if (amount > txn.getAmount())
        throw new InsufficientFundsException("Exceeds original");

    // 4. Create refund entity
    Refund refund = new Refund(generateId(), transactionId, amount);

    // 5. Process with bank and update status
    refund.setStatus(RefundStatus.PROCESSED);
    txn.setStatus(TransactionStatus.REFUNDED);

    return refund;
}

Gateway Facade (Simplified)

public class PaymentGateway {
    private final Map<String, PaymentResult> transactions;

    public PaymentResult processPayment(PaymentMethod method,
                                        double amount,
                                        Map<String, String> details) {
        String txnId = UUID.randomUUID().toString();
        PaymentStatus status = amount > 0
            ? PaymentStatus.SUCCESS
            : PaymentStatus.FAILED;
        PaymentResult result = new PaymentResult(txnId, status, "Processed");
        transactions.put(txnId, result);
        return result;
    }

    public PaymentResult getStatus(String transactionId) {
        return transactions.get(transactionId);
    }
}

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap Transaction store, idempotency store Safe concurrent reads/writes without global lock
synchronized(txn) State transitions on transactions Per-transaction lock — only one thread transitions a specific txn at a time
putIfAbsent() Idempotency key insertion Atomic check-and-insert prevents duplicate processing
AtomicLong Refund ID generation Lock-free incrementing for sequential IDs
volatile Payment processor references Cross-thread visibility when swapping processor strategies
CopyOnWriteArrayList Observer/webhook listener list Safe iteration while registering/removing listeners
"I use fine-grained locking on the Transaction object, not a global lock. Two different payments process concurrently. Only concurrent operations on the same transaction (e.g., refund vs status check) are serialized."

12 — Interview Follow-up Q&A

How do you ensure idempotency to prevent double charges?
Every payment request carries an idempotency key (UUID generated client-side). Before processing, we check ConcurrentHashMap<String, Transaction> for the key. If found, return the cached result. If not, process and store atomically using putIfAbsent(). In production, use Redis with TTL (24h) so keys expire. The idempotency key + atomic check ensures that even concurrent retries with the same key process exactly once.
How do you prevent double charges when the network times out?
Three layers: (1) Idempotency key — client retries with the same key, server returns cached result. (2) Transaction status check — before authorizing, verify the txn is still PENDING. (3) Distributed lock — in production, acquire a lock on the idempotency key in Redis before processing. The combination ensures at-most-once processing even with aggressive retries.
How would you handle refunds?
Refund is a separate entity with its own lifecycle (PENDING → PROCESSED / FAILED). Validation: original txn must be SUCCESS, refund amount ≤ original amount. We support partial refunds — track total refunded amount per transaction. The original txn transitions to REFUNDED only when fully refunded. In production, refunds go through the same payment processor (Visa/Mastercard reversal) and can take 5-10 business days.
How do webhook callbacks work for merchant notifications?
Observer pattern: merchants register webhook URLs for events (payment.success, payment.failed, refund.processed). On state transitions, we fire HTTP POST to registered URLs with signed payloads (HMAC-SHA256). Webhooks include retry with exponential backoff (5 attempts over 24h). The merchant endpoint must return 2xx to acknowledge. We store delivery attempts for debugging. In production, use a message queue (SQS/Kafka) for reliable delivery.
How do you handle PCI-DSS compliance?
Never store raw card data. Use tokenization — the Card object holds a token, not actual PAN. In production: (1) Use a PCI-compliant vault (Stripe, Braintree) for card storage. (2) Client-side encryption with the vault's public key. (3) Our servers never see raw card numbers — only tokens. (4) All communication over TLS 1.2+. (5) Audit logs for all access. The Card class in our LLD represents the token, not raw data.
How would you handle reconciliation between your system and banks?
Daily batch job compares our transaction records against bank settlement files. Three-way match: our DB, payment processor records, bank statement. Discrepancies flagged for manual review. Key fields: transaction ID, amount, timestamp, status. In production, maintain a ledger table with double-entry bookkeeping (debit merchant, credit customer on refund). Reconciliation runs as a scheduled job with alerting for mismatches > threshold.
Why State pattern instead of if/else on TransactionStatus?
(1) Each state is self-documenting — you see exactly which transitions are legal. (2) Adding a new state (e.g., ON_HOLD, PROCESSING) is a new class, no modification to existing code (Open/Closed Principle). (3) Invalid transitions throw immediately with clear messages. (4) Payment systems have strict compliance requirements — the state machine makes illegal flows impossible at compile time, not runtime surprises.
How would you handle payment timeouts from external processors?
Set aggressive timeouts on external calls (5s for authorization). On timeout: (1) Mark transaction as PENDING (not FAILED). (2) Schedule an async status-check job that polls the processor. (3) The processor's response resolves it to SUCCESS or FAILED. (4) Use the idempotency key when querying so the processor returns the actual outcome. Never assume timeout = failure — the charge may have succeeded.
How do you handle different currencies and exchange rates?
Add Currency field to Transaction. Use Strategy pattern for currency conversion — ExchangeRateStrategy interface with implementations (FixedRate, LiveRate from forex API). Settlement is always in the merchant's base currency. Store amounts as BigDecimal (never double) to avoid floating-point errors. Record both customer currency and merchant settlement currency.
How would you scale this to millions of transactions per day?
(1) Horizontal scaling: stateless service behind load balancer. (2) Database sharding by merchant_id or transaction_id range. (3) Read replicas for status queries. (4) Redis for idempotency keys (TTL 24h) and hot transaction cache. (5) Kafka for async notifications/webhooks. (6) CQRS: separate write (payment processing) and read (dashboards, reports) paths. (7) Connection pooling to payment processors.
What happens if the server crashes mid-transaction?
Write-Ahead Log (WAL): before calling the external processor, write the intent to a durable log. On restart, replay unresolved entries. The idempotency key ensures the processor handles retries safely. In production: (1) Use DB transactions for state changes. (2) Outbox pattern for notifications — write event to outbox table in the same DB transaction, background worker publishes to Kafka. (3) Saga pattern for multi-step payments (auth + capture as separate steps).
How would you implement subscription/recurring payments?
New entity: Subscription (id, merchantId, customerId, amount, interval, nextBillingDate, tokenizedPaymentMethod). A ScheduledExecutorService triggers billing on nextBillingDate. Each billing cycle creates a normal Transaction via processPayment() with a stable idempotency key (subscription_id + billing_period). Failed attempts retry with backoff. After N failures, pause subscription and notify customer. The tokenized card/mandate means no re-authentication needed.
