Car Rental System // LLD

Future-dated, time-boxed booking of a specific car with a pick-up → return lifecycle. Strategy (pricing + late-fee) with Decorator layering, State machine for the reservation lifecycle, Observer notifications, and a per-car lock making interval-overlap checks atomic.

01 — Requirements

Functional

  • Fleet Inventory — maintain cars, each with a CarType (ECONOMY/SUV/LUXURY), a home Location (pickup branch), and a service CarStatus (ACTIVE/MAINTENANCE/RETIRED)
  • Search — find available cars filtered by type, location and a date range [pickup, dropoff)
  • Reserve — book a specific car for a window; reject if it overlaps an existing active reservation for that car (no overbooking)
  • Pick up & Return — start the rental, then end it computing charges = base pricing + late fee if returned after the scheduled drop-off
  • Cancel — cancel a reservation before pickup; this frees the car instantly (availability is derived)
  • Pluggable Pricing — daily rate by car type, weekly discount, seasonal surge, and a swappable late-fee policy (standard / grace-period)
  • Notifications — confirmation / reminder / receipt on lifecycle events via observers

Non-Functional

  • Correctness under concurrency (headline NFR) — two threads must never both reserve the same car over overlapping dates. The concurrency test fires 64 threads at one car+window and asserts exactly one wins
  • Low contention — unrelated cars must not block each other on the reserve path (per-car lock, not a global lock)
  • Extensibility — new pricing / late-fee / notification rules without editing the orchestrator (Open–Closed)
  • Deterministic money — BigDecimal everywhere, never double

03 — Design Patterns

Five patterns, each earning its place — the interview differentiator.

Strategy — Pricing & Late-Fee

Two independent axes: PricingStrategy (Daily/Weekly/Seasonal) and LateFeePolicy (Standard/Grace). Base tariff and penalty rule change for different reasons — marketing swaps either without touching reservations.

Decorator

SeasonalPricingStrategy wraps any PricingStrategy and applies a peak-month surcharge; GracePeriodLateFeePolicy wraps any LateFeePolicy and forgives small lateness. Avoids a Daily×Weekly×Seasonal subclass matrix.

State Machine

Reservation lifecycle: RESERVED → PICKED_UP → RETURNED, plus RESERVED → CANCELLED. Each state allows only its legal verbs and throws IllegalStateException otherwise. Stateless singletons.

Observer

ReservationObserver (Email, SMS) subscribes to lifecycle events via default methods. Orchestrator fires confirmations/reminders/receipts without knowing about channels.

Facade

CarRental is a single, readable entry point that hides per-car locking, the fleet registries and the strategy wiring.

04 — Reservation State Machine

Why State? Four phases with a strict transition matrix. Each state class allows only its legal verbs, so "return a car that was never picked up" is impossible rather than a status-field bug. The states are stateless singletons and live in the model package next to Reservation. 
stateDiagram-v2
    [*] --> RESERVED: reserve()
    RESERVED --> PICKED_UP: pickUp(at)
    RESERVED --> CANCELLED: cancel()
    PICKED_UP --> RETURNED: returnCar(at, lateFee)
    RETURNED --> [*]
    CANCELLED --> [*]

State delegation in Reservation

public final class Reservation {
    private final BigDecimal baseCost;        // frozen at reserve time
    private final LocalDateTime scheduledDropoff; // dropoff.atStartOfDay()
    private ReservationState state = ReservedState.INSTANCE;

    public synchronized void pickUp(LocalDateTime at) {
        state.pickUp(this, at);           // delegate to current state
    }
    public synchronized Charges returnCar(LocalDateTime at, BigDecimal lateFee) {
        return state.returnCar(this, at, lateFee);
    }
    public synchronized void cancel() { state.cancel(this); }

    public boolean isActive() {          // RESERVED or PICKED_UP
        ReservationStatus s = getStatus();
        return s == ReservationStatus.RESERVED || s == ReservationStatus.PICKED_UP;
    }
}

PickedUpState — the return transition

public Charges returnCar(Reservation r, LocalDateTime at, BigDecimal lateFee) {
    Charges charges = new Charges(r.getBaseCost(), lateFee); // frozen base + late fee
    r.markReturned(at, charges);                             // -> ReturnedState
    return charges;
}
// ReservedState.returnCar(..) throws IllegalStateException("not picked up")
// PickedUpState.cancel(..)     throws IllegalStateException("already picked up")

05 — Strategy Pattern

Why two Strategy interfaces? The base tariff (daily / weekly / seasonal) and the penalty rule (flat / grace-period) change for different reasons and at different times. Modeling them as two interfaces lets either be swapped without a code change to reservations. Decorators layer surge and grace on top of any base rule.

The two interfaces

interface PricingStrategy {
    BigDecimal price(CarType type, DateRange period);   // base cost
}
interface LateFeePolicy {
    BigDecimal lateFee(CarType type,
                    LocalDateTime scheduledDropoff,
                    LocalDateTime actualReturn);   // ZERO if on time
}

Both are stateless / thread-safe: a single instance is shared across all concurrent reservations.

Pricing implementations

ImplementationLogic
DailyPricingStrategyFlat daily tariff: dailyRate(type) × days
WeeklyPricingStrategyEvery full 7-day block billed at billedDaysPerWeek (default 6 — "rent a week, pay for six"); remainder days at full rate
SeasonalPricingStrategyDecorator — wraps any base strategy, multiplies by peakMultiplier when the window hits a peak month

Late-fee implementations

ImplementationLogic
StandardLateFeePolicydailyRate(type) × penaltyMultiplier for every started late day (ceil)
GracePeriodLateFeePolicyDecorator — returns within graceHours of due are free; beyond grace, delegates (measured from the original due time so grace forgives without shifting the meter)

Decorator layering (from the demo)

PricingStrategy daily  = new DailyPricingStrategy(rates);
PricingStrategy weekly = new WeeklyPricingStrategy(rates, 6);

// Seasonal WRAPS daily -> surge composes on top of the base tariff
PricingStrategy holidaySurge = new SeasonalPricingStrategy(
        daily, EnumSet.of(Month.DECEMBER), new BigDecimal("1.25"));

// Grace WRAPS standard -> forgive < N hours, else the standard rule
LateFeePolicy lenient = new GracePeriodLateFeePolicy(
        new StandardLateFeePolicy(rates, new BigDecimal("1.5")), 2);

06 — Key Algorithms: Overlap + Late-Fee

1. Date-Range Overlap (the crux)

// DateRange is HALF-OPEN: pickup inclusive, dropoff exclusive.
// Two intervals overlap iff each starts strictly before the other ends.
public boolean overlaps(DateRange other) {
    return this.pickup.isBefore(other.dropoff)
        && other.pickup.isBefore(this.dropoff);
}
The half-open choice is deliberate: it makes back-to-back rentals legal for free. [10,13) and [13,15) don't overlap, so a car returned on the 13th is bookable from the 13th — no +1 day fudging. days() = ChronoUnit.DAYS.between(pickup, dropoff), always ≥ 1.

2. Per-car overlap probe (availability is derived)

// Caller MUST hold carId's lock. No stored RENTED flag anywhere.
private boolean hasOverlap(String carId, DateRange period) {
    for (Reservation r : reservationsByCar.get(carId)) {
        if (r.isActive() && r.getPeriod().overlaps(period)) {
            return true;
        }
    }
    return false;
}
Time: O(reservations for that car) | Space: O(1)
Scale-up: per-car sorted interval list / interval tree -> O(log n)

3. Late-Fee — scheduled vs actual, per started day

// StandardLateFeePolicy.lateFee(type, scheduledDropoff, actualReturn)
long minutesLate = Duration.between(scheduledDropoff, actualReturn).toMinutes();
if (minutesLate <= 0) return BigDecimal.ZERO;   // on time / early

long lateDays = (minutesLate + 1439) / 1440; // ceil: any part-day = full day
return dailyRate(type)
        .multiply(penaltyMultiplier)
        .multiply(BigDecimal.valueOf(lateDays));
scheduledDropoff = dropoff.atStartOfDay() (the exclusive end day). Base cost is frozen at reservation time so a mid-rental tariff change can't alter an existing quote; the late fee is computed only at return. The GracePeriodLateFeePolicy decorator returns 0 within N hours of due and otherwise delegates.

4. Check-then-reserve (atomic)

Algorithm: Reserve(customerId, carId, period)

1. lock = lockFor(carId); lock.lock()   // per-car ReentrantLock
2. if car.status != ACTIVE:        throw CarUnavailableException
3. if hasOverlap(carId, period):    throw CarUnavailableException
4. base = pricingStrategy.price(car.type, period)  // freeze quote
5. reservation = new Reservation(id, car, customer, period, base)
6. reservationsByCar.get(carId).add(reservation)  // publish under lock
7. lock.unlock()  (finally)
8. notifyReserved(reservation)     // AFTER lock released

Check + insert are ONE atomic step -> no check-then-act race.

07 — Thread Safety

MechanismWhereWhy
ReentrantLock per car Map<carId, ReentrantLock> Makes overlap-check + insert atomic. Held across reserve / search / pickUp / returnCar / cancel so checks see a consistent snapshot
ConcurrentHashMap cars, customers, locations, reservationsById, reservationsByCar Lock-free reads and safe publication of the registries
CopyOnWriteArrayList observers Safe iteration while notifying without locking the reserve path
AtomicLong reservationSeq Lock-free unique reservation IDs: R1, R2, ...
volatile status Car.status Visible status changes (maintenance/retire) across threads
model has no service dep state objects receive a computed lateFee One-directional dependency arrow (service → model); no import cycle
"I use a per-car ReentrantLock, not a global lock. Customers booking different cars proceed in parallel; only threads racing for the same car serialize. Observer notifications happen after the lock is released so a slow listener never widens the critical section."

Contention isolation in reserve()

ReentrantLock lock = lockFor(carId);
lock.lock();
try {
    if (car.getStatus() != CarStatus.ACTIVE)  throw new CarUnavailableException(...);
    if (hasOverlap(carId, period))          throw new CarUnavailableException(...);
    reservationsByCar.get(carId).add(new Reservation(...)); // publish under lock
} finally { lock.unlock(); }
notifyReserved(reservation); // lock already released

11 — vs Parking Lot & Hotel Booking

It borrows two familiar sub-problems, but the combination and the distinctive mechanics are new. Say this out loud so the interviewer knows you're not re-skinning a template.

vs Parking Lot

Parking is spot allocation for an open-ended, present-tense stay — you drive in now, availability is a boolean "is this spot occupied right now", pricing is duration-after-the-fact. Car rental is future-dated, time-boxed booking: you reserve a window that may start days from now, so availability is an interval-overlap question, not a boolean, and multiple non-overlapping reservations coexist on one car.

vs Hotel Booking

Hotel rooms are fungible within a room type — you book "a deluxe room" and the system hands you any free one. Car rental reserves a specific identified car (by plate) at a specific branch, so the lock granularity is per-car and multi-location inventory matters.

What's genuinely this problem's own

The pick-up → return lifecycle (a rental isn't done at booking; it's collected, then returned) and the return-time late-fee computed from scheduled-vs-actual timestamps. Neither parking (no reservation lifecycle) nor a basic hotel booking (charge fixed at booking) has a two-phase collect/return flow with penalty math at the end. That lifecycle is exactly why the State pattern earns its place, and future-dated windows are why the per-car interval lock is the core concurrency story rather than a counting semaphore.

12 — Interview Follow-up Q&A

How do you guarantee two threads never double-book the same car?
A ReentrantLock per car held across the overlap-check and the insert makes it one atomic step — killing the classic check-then-act race where two threads both pass the check and both insert. It's per-car, not global, so threads targeting different cars never contend. The concurrency test fires 64 threads at one car+window and asserts exactly one wins.
Why half-open [pickup, dropoff) intervals?
Half-open makes back-to-back rentals legal for free: [10,13) and [13,15) don't overlap, so a car returned on the 13th is bookable from the 13th with no +1 day fudging. The overlap test collapses to a.pickup < b.dropoff && b.pickup < a.dropoff, and days() is a clean ChronoUnit.DAYS.between(pickup, dropoff).
Why is there no RENTED car status?
Availability is derived, not stored. Whether a car is free for a window is computed from its active reservations, which avoids a whole class of stale-flag bugs (double-book because a RENTED boolean wasn't reset). It's why cancel instantly frees the car — the cancelled reservation is simply no longer isActive(). The cost is recomputing overlaps per query, which is acceptable and cacheable later.
Why freeze base cost at reserve but compute late fee at return?
The quoted price must be honored, so baseCost is frozen on the Reservation at reserve time — a mid-rental tariff change can't alter an existing quote. The late fee depends on the actual return time, which isn't known until return, so a separate LateFeePolicy owns that math and PickedUpState assembles new Charges(baseCost, lateFee).
How would you support one-way rentals (drop at a different branch)?
Add a dropoffLocation to the reservation and a relocation fee to the pricing strategy; the car's home location updates on return. Availability search would then need to consider in-transit cars.
How do you persist this and survive restarts / scale horizontally?
Replace the in-memory maps with a repository interface. Move the per-car lock into the DB as SELECT ... FOR UPDATE on the car row, or a Postgres EXCLUDE exclusion constraint on (car_id, tsrange) so the DB itself enforces non-overlap — correct even if a lock service fails. For multiple app servers, in-process locks no longer suffice; prefer the DB constraint over a distributed lock (Redis/ZooKeeper). The domain logic is unchanged.
How would you handle no-shows / auto-cancel?
A scheduled sweeper transitions stale RESERVED reservations past a grace window to CANCELLED. The State pattern already permits RESERVED → CANCELLED, so no new transition logic is needed.
Search is O(cars) — how do you make it fast for a huge fleet?
Index cars by (type, location) and keep a per-car sorted interval list (or an interval tree) so the overlap probe is O(log n) instead of scanning all reservations. Add a ReservationPolicy strategy in reserve() (max active per customer, deposit) to bound abuse, and a waitlist via a new observer + a cancellation hook that offers the freed car to the next customer.
