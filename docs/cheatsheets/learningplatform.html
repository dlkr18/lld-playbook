01 — Requirements

Functional

  • User management — Student / Instructor registration, profile, auth
  • Course management — Create/update/delete courses, add sections & lessons, publish/unpublish
  • Enrollment — Browse & search courses, enroll (free/paid), track enrollment status, unenroll
  • Learning experience — Watch videos / read content, mark lessons complete, track progress (%), resume from last position

Non-Functional

  • Security — Secure video streaming, payment processing, DRM
  • Thread-safe — Concurrent enrollments, progress updates

03 — Design Patterns

Five patterns — each solving a distinct problem in the learning domain.

Composite Pattern

Course → Section → Lesson hierarchy. Uniform traversal for progress calculation, content listing, and prerequisite checking.

Strategy Pattern

Grading strategies (auto vs. manual). Injected into assessments. New grading logic = new class, zero changes to service.

Template Method

Abstract Assessment.evaluate() defines the grading flow. Quiz overrides with auto-grade; Assignment overrides with manual review.

Observer Pattern

Progress updates trigger notifications. When a lesson completes, observers fire: update enrollment progress, check certificate eligibility, notify student.

Factory Pattern

Assessment creation — factory decides whether to instantiate Quiz or Assignment based on type. Extensible for new assessment types.

Repository Pattern

ConcurrentHashMap stores act as in-memory repositories. Easy to swap for JPA, JDBC, or NoSQL without touching service logic.

06 — Assessment System

Why Template Method? Both Quiz and Assignment follow the same grading flow (validate → evaluate → record). The how of evaluation differs — Quiz auto-grades, Assignment awaits manual review.

Abstract Assessment (Design)

abstract class Assessment {
    protected String id, title;
    protected int passingScore;

    // Template method: defines the grading skeleton
    public final AssessmentResult evaluate(String studentId, List<String> answers) {
        validate(answers);
        int score = grade(answers);    // subclass-specific
        boolean passed = score >= passingScore;
        return new AssessmentResult(studentId, id, score, passed);
    }

    protected abstract int grade(List<String> answers); // hook
}

Quiz — Auto Grading

class Quiz extends Assessment {
    private List<Question> questions;
    private int timeLimit;

    protected int grade(List<String> answers) {
        int correct = 0;
        for (int i = 0; i < questions.size(); i++) {
            if (answers.get(i).equals(questions.get(i).correctAnswer))
                correct++;
        }
        return (correct * 100) / questions.size();
    }
}

Assignment — Manual Grading

class Assignment extends Assessment {
    private String instructions;
    private LocalDateTime deadline;

    protected int grade(List<String> answers) {
        // Queued for instructor review; returns 0 pending manual grade
        return 0;
    }
}

07 — Key Algorithms

Course Progress Calculation

// O(n) where n = total lessons in course
public double getProgress(String studentId, String courseId) {
    Course course = courses.get(courseId);
    List<Lesson> lessons = course.getLessons();
    if (lessons.isEmpty()) return 0.0;

    Map<String, Boolean> studentProgress = lessonProgress.get(studentId);
    if (studentProgress == null) return 0.0;

    long completed = lessons.stream()
        .filter(l -> studentProgress.getOrDefault(l.getId(), false))
        .count();

    return (completed * 100.0) / lessons.size();
}

Time: O(n) · Space: O(n) · Uses stream filter against per-student progress map

Quiz Auto-Grading

// O(q) where q = number of questions
int autoGrade(Quiz quiz, Map<String, String> userAnswers) {
    int correct = 0;
    for (Question q : quiz.questions) {
        String answer = userAnswers.get(q.id);
        if (answer != null && answer.equals(q.correctAnswer))
            correct++;
    }
    return (correct * 100) / quiz.questions.size();
}

Time: O(q) · Space: O(1) · Supports MULTIPLE_CHOICE, TRUE_FALSE, MULTIPLE_SELECT

Course Search & Ranking

// O(n log n) where n = candidate courses
public List<Course> searchCourses(String keyword) {
    String lower = keyword.toLowerCase();
    return courses.values().stream()
        .filter(c -> c.getTitle().toLowerCase().contains(lower) ||
                    c.getDescription().toLowerCase().contains(lower))
        .collect(Collectors.toList());
}

// Production ranking adds relevance scoring:
// title exact match (+10), description match (+5),
// rating boost (+rating), popularity boost (+log(enrollments))

Certificate Eligibility Check

// O(n + m) where n=lessons, m=assessments
boolean isCertificateEligible(String studentId, String courseId) {
    // 1. Check enrollment exists and is ACTIVE
    Enrollment e = getEnrollment(studentId, courseId);
    if (e == null || e.getStatus() != ACTIVE) return false;

    // 2. All lessons completed?
    if (getProgress(studentId, courseId) < 100) return false;

    // 3. All assessments passed?
    for (Assessment a : getAssessments(courseId)) {
        Submission s = getSubmission(studentId, a.getId());
        if (s == null || s.getScore() < a.getPassingScore())
            return false;
    }
    return true;
}

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap courses, enrollments, studentEnrollments, lessonProgress Safe concurrent reads/writes across all data stores
ConcurrentHashMap.newKeySet() studentEnrollments values Thread-safe Set for each student's enrollment IDs
computeIfAbsent lessonProgress, studentEnrollments Atomic check-and-create for nested maps/sets
AtomicLong enrollmentIdGenerator Lock-free unique ID generation
Defensive copies Course.getLessons() Returns new ArrayList<>(lessons) to prevent external mutation
"All stores use ConcurrentHashMap with computeIfAbsent for atomic nested-map creation. No global locks — concurrent enrollments in different courses proceed in parallel."

Key Concurrency Code

// Atomic nested map creation for progress tracking
Map<String, Boolean> studentProgress = lessonProgress.computeIfAbsent(
    studentId, k -> new ConcurrentHashMap<>()
);
studentProgress.put(lessonId, true);

// Thread-safe enrollment set per student
studentEnrollments.computeIfAbsent(studentId, k -> ConcurrentHashMap.newKeySet())
                  .add(enrollmentId);

// Lock-free ID generation
String enrollmentId = "ENR-" + enrollmentIdGenerator.incrementAndGet();

11 — Data Structures

// Primary in-memory stores
Map<String, Course>      courses             = new ConcurrentHashMap<>();
Map<String, Enrollment> enrollments         = new ConcurrentHashMap<>();

// Index: studentId → set of enrollmentIds
Map<String, Set<String>> studentEnrollments = new ConcurrentHashMap<>();

// Nested progress: studentId → { lessonId → completed }
Map<String, Map<String, Boolean>> lessonProgress = new ConcurrentHashMap<>();

// Lock-free ID generation
AtomicLong enrollmentIdGenerator = new AtomicLong(0);

// On Course entity:
List<Lesson> lessons = new ArrayList<>();

12 — Interview Follow-up Q&A

How would you handle video streaming at scale?
Use a CDN (CloudFront, Akamai) for video delivery with adaptive bitrate streaming (HLS/DASH). Store raw videos in S3, transcode to multiple resolutions via a media pipeline. Track playback position in the Progress entity for resume capability. DRM (Widevine/FairPlay) prevents piracy.
How do you prevent content piracy?
DRM (Digital Rights Management) for encrypted video streams. Watermarking with student ID embedded in video frames. Limit concurrent sessions per account. Signed URLs with expiration for video access. Server-side rendering prevents easy download.
How would you scale course search?
Replace in-memory filtering with Elasticsearch or Apache Solr. Index title, description, tags, instructor name. Add relevance scoring: title match (+10), description (+5), rating boost, popularity (log of enrollments). The searchCourses() interface stays the same — swap implementation.
How would you implement course recommendations?
Collaborative filtering: "Students who enrolled in X also enrolled in Y." Content-based: similar topics, same instructor, matching difficulty level. Trending: enrollment spike detection. A RecommendationStrategy interface keeps it pluggable.
How do you scale assessment grading?
Quizzes auto-grade immediately (O(q) per submission). Assignments go into a grading queue (SQS/Kafka) for instructor review. For large classes, enable peer grading where 3+ students review each submission. ML-based plagiarism detection (cosine similarity on code, TF-IDF on essays).
Why separate Progress from Enrollment?
Enrollment tracks the relationship (student ↔ course). Progress tracks granular lesson-level completion. Separation gives: (1) Granular analytics — time per lesson, drop-off points. (2) Resume capability — pick up exactly where you left off. (3) Independent queries — "which lessons are most completed?" without touching enrollments.
Why Template Method for assessments instead of Strategy?
The grading skeleton is fixed: validate → grade → record result. Only the grading logic varies. Template Method is the right fit when the algorithm structure is shared and only specific steps differ. Strategy would be better if the entire evaluation pipeline varied.
How would you handle concurrent enrollment in a popular course?
ConcurrentHashMap with computeIfAbsent handles concurrent enrollment atomically. For courses with a max capacity, use AtomicInteger for seat count with compareAndSet to prevent overselling. In production, a distributed lock (Redis) or optimistic locking (DB version column).
How would you implement prerequisites?
Each Course holds a List<String> prerequisiteIds. On enrollment, check that the student has COMPLETED status for all prerequisites. This forms a DAG (directed acyclic graph). Validate no cycles when setting prerequisites using topological sort.
How would you add a discussion forum per course?
New entities: Thread (title, courseId, authorId, createdAt) and Post (threadId, authorId, content, upvotes). Index by courseId for listing. Add createThread() and replyToThread() to the service. Upvoting uses AtomicInteger. Instructor answers get a visual badge.
How would you implement gamification (badges, streaks)?
Observer pattern: when a lesson completes, fire a GamificationObserver that checks badge criteria. Store Badge (type, earnedAt) and Streak (startDate, currentCount). Daily login check: if lastActivityDate == yesterday, increment streak; else reset. Leaderboard via sorted set (Redis ZSET).
What database would you use in production?
PostgreSQL for courses, enrollments, users (relational integrity). Redis for progress tracking (fast writes, TTL for sessions). Elasticsearch for course search. S3 + CDN for video/file storage. Kafka for async events (grading queue, notifications, analytics).
