01 — Requirements

Functional (MVP)

  • Music Catalog — Browse songs, albums, artists; search by title/artist/album; view details (title, artist, duration, genre)
  • User Management — Create accounts, profiles with preferences, follow/unfollow artists, subscription tiers (Free, Premium)
  • Playlist Management — Create/rename/delete playlists, add/remove songs, public/private visibility, share playlists
  • Playback — Play/pause/resume, next/previous, shuffle mode, repeat modes (NONE, ONE, ALL), queue management
  • Library — Liked songs, saved albums, recently played, listening history
  • Search — Search by title/artist/album, filter by genre, autocomplete

Non-Functional

  • Performance — Search < 100ms, playlist load < 200ms, playback start < 500ms
  • Scalability — Millions of songs/users, billions of streams/month
  • Availability — 99.99% uptime, graceful degradation, offline support
  • Consistency — Playlist updates immediately visible, accurate play counts, no duplicate songs

Out of Scope

  • Audio file storage/streaming (assume CDN), payment processing, royalty calculations
  • ML recommendation algorithms, social media integration, podcasts, live radio

03 — Design Patterns

Patterns that earn their place in the interview.

State Pattern

Playback lifecycle: STOPPED → PLAYING → PAUSED. Each state defines legal transitions. PlaybackSession delegates play(), pause(), resume() to current state.

Strategy Pattern

Search (simple text scan vs indexed lookup) and Recommendation (genre-based, history-based). Swappable algorithms without changing service code.

Aggregate Root

Playlist and User are aggregate roots. All mutations go through the root — playlist.addSong() auto-updates timestamps, user manages own library.

Value Object

Type-safe IDs (SongId, AlbumId, etc.) are immutable with equals()/hashCode(). Prevents accidental ID type mixing at compile time.

04 — Playback State Machine

Why State? Playback has well-defined legal transitions. You can't pause a stopped session. You can't resume a playing session. The state machine enforces this cleanly. 
stateDiagram-v2
    [*] --> Idle: Create session
    Idle --> Loading: play()
    Loading --> Playing: audio loaded
    Loading --> Error: load failed
    Playing --> Paused: pause()
    Playing --> Idle: stop()
    Playing --> Loading: next/previous song
    Playing --> Ended: song finished
    Paused --> Playing: resume()
    Paused --> Idle: stop()
    Paused --> Loading: next/previous song
    Ended --> Loading: auto next (queue has more)
    Ended --> Idle: queue finished
    Error --> Idle: reset
    Error --> Loading: retry()

PlaybackSession — State Transitions

public class PlaybackSession {
    private Song currentSong;
    private int currentPositionSeconds;
    private PlaybackState state;          // PLAYING, PAUSED, STOPPED
    private final PlaybackQueue queue;

    public void play(Song song) {
        this.currentSong = song;
        this.currentPositionSeconds = 0;
        this.state = PlaybackState.PLAYING;
        song.play();                        // increment play count
    }

    public void pause() {
        if (state == PlaybackState.PLAYING)
            this.state = PlaybackState.PAUSED;
    }

    public void resume() {
        if (state == PlaybackState.PAUSED && currentSong != null)
            this.state = PlaybackState.PLAYING;
    }

    public void next() {
        Song nextSong = queue.next();      // respects repeat mode
        if (nextSong != null) play(nextSong);
        else stop();
    }

    public void seek(int positionSeconds) {
        if (currentSong != null
            && positionSeconds >= 0
            && positionSeconds <= currentSong.getDurationSeconds())
            this.currentPositionSeconds = positionSeconds;
    }
}

Auto-Advance on Song End

public void advanceTime(int seconds) {
    if (state == PlaybackState.PLAYING && currentSong != null) {
        currentPositionSeconds += seconds;
        if (currentPositionSeconds >= currentSong.getDurationSeconds()) {
            if (queue.hasNext()) next();
            else stop();
        }
    }
}

When current song finishes, the session checks the queue. If more songs exist (respecting repeat mode), it auto-advances. Otherwise, it stops.

        songs.add(song);
        this.updatedAt = LocalDateTime.now();

    public void removeSong(int index) {
        if (index >= 0 && index < songs.size()) {

06 — Queue & Algorithms

Core interview topic. The queue handles shuffle (Fisher-Yates), repeat modes (NONE/ONE/ALL), and preserves original order for unshuffle.

Fisher-Yates Shuffle

public void setShuffle(boolean enabled) {
    if (enabled == shuffleEnabled) return;
    this.shuffleEnabled = enabled;

    if (enabled) {
        originalOrder = new ArrayList<>(queue);     // save order
        Song currentSong = getCurrentSong();

        // Fisher-Yates: O(n), uniform distribution
        for (int i = queue.size() - 1; i > 0; i--) {
            int j = random.nextInt(i + 1);
            Collections.swap(queue, i, j);
        }

        // Keep current song at front
        if (currentSong != null) {
            int newIndex = queue.indexOf(currentSong);
            if (newIndex > 0) Collections.swap(queue, 0, newIndex);
            currentIndex = 0;
        }
    } else {
        // Restore original order, find current song's position
        Song currentSong = getCurrentSong();
        queue.clear();
        queue.addAll(originalOrder);
        currentIndex = queue.indexOf(currentSong);
    }
}

Key insight: The currently playing song is moved to position 0 after shuffle, so playback is seamless. Original order is saved for when the user disables shuffle.

Repeat Mode Logic

public Song next() {
    if (queue.isEmpty()) return null;

    // REPEAT ONE: stay on same song
    if (repeatMode == RepeatMode.ONE)
        return getCurrentSong();

    // More songs ahead: advance
    if (currentIndex < queue.size() - 1)
        return queue.get(++currentIndex);

    // At end of queue
    if (repeatMode == RepeatMode.ALL) {
        currentIndex = 0;                   // loop to start
        return queue.get(0);
    }

    return null;                            // NONE: end playback
}

Previous Song Logic

public Song previous() {
    if (queue.isEmpty()) return null;
    if (currentIndex > 0)
        return queue.get(--currentIndex);

    // At start: wrap to end if repeat ALL
    if (repeatMode == RepeatMode.ALL) {
        currentIndex = queue.size() - 1;
        return queue.get(currentIndex);
    }
    return getCurrentSong();     // stay at first song
}

07 — Search Implementation

Simple Text Search (O(n))

List<Song> searchSongs(String query) {
    String lowerQuery = query.toLowerCase();
    return songs.values().stream()
        .filter(song ->
            song.getTitle().toLowerCase().contains(lowerQuery) ||
            song.getArtistName().toLowerCase().contains(lowerQuery))
        .collect(Collectors.toList());
}

Good enough for demo. Scans all songs with substring matching on title and artist name.

12 — Interview Follow-up Q&A

How does shuffle work without losing the original order?
Before shuffling, we save the current queue into originalOrder. The Fisher-Yates algorithm shuffles in-place (O(n), uniform distribution). The currently-playing song is moved to index 0 so playback isn't interrupted. When shuffle is disabled, we restore from originalOrder and find the current song's original position.
How do repeat modes interact with shuffle?
They're orthogonal. Shuffle controls the order of the queue; repeat controls what happens at the boundary. With SHUFFLE + REPEAT ALL, when the queue ends it wraps to the first song of the shuffled order. With REPEAT ONE, the current song replays regardless of shuffle state.
Why type-safe IDs instead of plain Strings?
Prevents accidental mixing — you can't pass a SongId where an AlbumId is expected. Compiler catches bugs at build time. Each ID validates non-null/non-empty in its constructor. They're immutable value objects with proper equals()/hashCode() for use as Map keys.
How would you scale search to millions of songs?
Replace in-memory scan with Elasticsearch or Apache Solr. Use inverted indexes for title/artist/album. Add fuzzy matching for typo tolerance, n-gram tokenization for partial matches, and TF-IDF for relevance ranking. The MusicLibrary interface stays the same — just swap the implementation.
How would you implement recommendations?
Start simple: recommend songs from the same genre or by the same artist as recently played tracks. At scale, use collaborative filtering (users who liked X also liked Y) or content-based filtering (audio features, tempo, key). Behind a RecommendationStrategy interface for easy swapping.
How does the subscription tier affect functionality?
FREE tier: ads between songs, limited skips per hour, no offline downloads, lower audio quality. PREMIUM tier: ad-free, unlimited skips, offline downloads, high-quality audio. Implemented via the SubscriptionTier enum checked in the service layer before allowing restricted operations.
Why is Playlist an aggregate root?
All mutations (add/remove songs, rename, visibility changes) go through Playlist methods. This ensures invariants are always enforced — name length validation, timestamp updates, unmodifiable external views. External code can't bypass these rules by manipulating the internal list directly.
How would you handle concurrent playlist edits (collaborative playlists)?
Add synchronized blocks on the playlist object for add/remove operations. Use optimistic locking with a version field: read version, apply change, compare-and-swap. If version mismatch, retry. At scale, use a CRDT (Conflict-free Replicated Data Type) for eventual consistency across replicas.
How would you handle offline playback?
Premium users download encrypted audio files to local storage. A DownloadManager tracks download state (QUEUED, DOWNLOADING, COMPLETE, EXPIRED). Offline playback reads from local cache. Licensing requires DRM and periodic online check-ins to verify subscription status.
Why HashMap for sessions instead of just storing state on User?
Separation of concerns. User is a domain entity (profile, library). PlaybackSession is transient runtime state (what's playing now). Sessions can be garbage-collected when inactive without affecting the user. Also enables multiple session types in future (mobile + desktop simultaneously).
How would you add crossfade between songs?
The PlaybackSession.advanceTime() method would trigger a crossfade when remainingTime < crossfadeDuration. Pre-load the next song's audio, start playing it at low volume while fading out the current song. This is a UI/audio-engine concern, but the queue's next() method provides the upcoming song for pre-loading.
What's the difference between addToQueue and playPlaylist?
playPlaylist() clears the queue and loads all playlist songs — it's a full queue replacement. addToQueue() appends a single song to the existing queue without interrupting current playback. If nothing is playing, addToQueue also starts playback automatically.
