01 — Requirements

Functional

  • Beverage Menu — Espresso, Cappuccino, Latte, Americano, Mocha, Hot Chocolate
  • Ingredient Management — Track coffee beans, milk, water, sugar, chocolate with capacity limits
  • Order Processing — Select → Customize → Pay → Brew → Dispense → Receipt
  • Customization — Extra shot (+$0.50), milk type (whole/skim/almond/soy), sugar level (0-3)
  • Payment Methods — Cash (with change), Card, UPI, Mobile Wallet
  • Low Stock Alerts — Notify when ingredient < 10% capacity
  • Machine States — IDLE, SELECTING, PAYMENT, BREWING, DISPENSING, MAINTENANCE, OUT_OF_SERVICE

Non-Functional

  • Performance — Beverage dispensing < 30s, support 100+ orders/hour, queue up to 5 concurrent users
  • Thread-safe — Atomic ingredient deduction, no overselling under concurrency
  • Extensible — Add new beverages via recipe config, new payment methods via interface
  • Reliable — Graceful ingredient shortage handling, automatic rollback on brewing failure

03 — Design Patterns

Seven patterns, each earning its place in the design.

State Pattern

Machine states (IDLE, BREWING, MAINTENANCE, etc). Each MachineState defines legal transitions. Invalid ops throw immediately. Adding CLEANING state = adding a class.

Strategy Pattern

Payment processing — Cash, Card, UPI each implement PaymentStrategy. Injected at runtime, swappable without restart.

Factory Pattern

BeverageRecipe creates beverages from a static recipe map. createBeverage(LATTE) returns a fully-configured Beverage. Centralized creation logic.

Builder Pattern

Fluent API for custom orders: .extraShot(true).milkType(ALMOND).sugarLevel(2).build(). Decouples construction from representation.

Observer Pattern

Low stock alerts — IngredientContainer notifies observers when stock drops below 10% capacity. Decouples inventory from alerting logic.

Singleton + Command

Singleton: Single machine instance for global state. Command: Order operations for undo/redo and logging.

IngredientContainer — Thread-safe Inventory

public class IngredientContainer {
    private final Map<Ingredient, Integer> quantities;   // ConcurrentHashMap
    private final Map<Ingredient, Integer> maxCapacity;

    public synchronized boolean hasIngredients(Map<Ingredient, Integer> required) {
        for (Map.Entry<Ingredient, Integer> entry : required.entrySet()) {
            if (quantities.getOrDefault(entry.getKey(), 0) < entry.getValue())
                return false;
        }
        return true;
    }

    public synchronized boolean consume(Map<Ingredient, Integer> required) {
        if (!hasIngredients(required)) return false;
        for (Map.Entry<Ingredient, Integer> e : required.entrySet()) {
            quantities.put(e.getKey(), quantities.get(e.getKey()) - e.getValue());
        }
        return true;
    }

    public synchronized void refill(Ingredient ingredient, int amount) {
        int current = quantities.getOrDefault(ingredient, 0);
        int max = maxCapacity.get(ingredient);
        quantities.put(ingredient, Math.min(current + amount, max));
    }
}

Cash Payment with Change Calculation

public Payment processCashPayment(double totalPrice, double cashGiven) {
    if (cashGiven < totalPrice)
        throw new InsufficientPaymentException();

    double change = cashGiven - totalPrice;

    // Greedy: largest denominations first
    Map<Integer, Integer> denominations = calculateChange(change);

    if (!canProvideChange(denominations))
        throw new InsufficientChangeException("Please provide exact change");

    deductChange(denominations);
    return new Payment(CASH, totalPrice, change, true);
}

08 — Key Algorithms

Atomic Ingredient Deduction (Critical!)

The single most important algorithm — must be thread-safe and all-or-nothing.

Naive Approach (Race Condition!)

// BAD: Two threads can both pass check and oversell
if (inventory.get(COFFEE_BEANS) >= 7) {
    inventory.put(COFFEE_BEANS, inventory.get(COFFEE_BEANS) - 7);
}

Correct: Synchronized All-or-Nothing

public synchronized boolean deductIngredients(Beverage beverage) {
    // Phase 1: Check ALL ingredients
    for (Map.Entry<Ingredient, Integer> entry : beverage.getRecipe().entrySet()) {
        IngredientContainer container = inventory.get(entry.getKey());
        if (!container.isSufficient(entry.getValue()))
            return false;  // Abort: insufficient
    }

    // Phase 2: All available → deduct atomically
    for (Map.Entry<Ingredient, Integer> entry : beverage.getRecipe().entrySet()) {
        IngredientContainer container = inventory.get(entry.getKey());
        container.deduct(entry.getValue());
    }
    return true;
}

synchronized prevents race conditions. Two-phase check-then-deduct ensures no partial deduction.

Low Stock Detection

public boolean isLowStock() {

// After every consume(), check and alert
if (container.isLowStock()) {

Refill with Capacity Cap

public synchronized void refill(Ingredient ingredient, int amount) {

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap IngredientContainer.quantities, orders store Safe concurrent reads/writes on the map level
synchronized hasIngredients(), consume(), refill() Atomic check-then-deduct prevents overselling
Math.min() cap refill() Prevents overfilling beyond container capacity
UUID Order ID generation Globally unique, no counter contention
FIFO Queue User request queue Fairness — first-come, first-served for concurrent users
"The synchronized keyword on the IngredientContainer methods means only one thread can check-and-deduct at a time. Two concurrent orders for the last Espresso — one succeeds, one gets an error. No overselling."

13 — Interview Follow-up Q&A

How do you handle concurrent orders for limited ingredients?
synchronized on the IngredientContainer methods. The consume() method checks ALL ingredients first, then deducts atomically. Two threads can't both pass the check for the last serving — only one enters the synchronized block at a time.
What if brewing fails mid-way?
Rollback ingredient deduction (re-add consumed amounts), refund payment, set order status to FAILED, log the error, and transition machine to MAINTENANCE state. The two-phase approach makes rollback straightforward — we know exactly what was deducted.
How to calculate optimal change with limited denominations?
Greedy algorithm with largest denominations first. If greedy can't provide exact change, fall back to DP (dynamic programming) for optimal coin combination. If impossible, throw InsufficientChangeException and prompt for exact payment.
How to add a new beverage type?
Add to BeverageType enum, define recipe in the static recipe map (ingredients and quantities), set a price. No other code changes needed — the Factory pattern picks it up automatically. Open/Closed Principle in action.
Why synchronized instead of ReentrantLock or AtomicInteger?
synchronized is simpler and sufficient for our use case. We need atomicity across multiple ingredients (all-or-nothing), not just single-variable CAS. AtomicInteger only helps per-ingredient. ReentrantLock would work but adds complexity without benefit here.
Why ConcurrentHashMap if you also use synchronized?
ConcurrentHashMap ensures safe reads from non-synchronized code paths (e.g., checkIngredients() and getQuantity() are unsynchronized for read performance). The synchronized blocks protect the check-then-deduct compound operation.
How would you handle multiple machines in a fleet?
Each machine is an independent CoffeeMachineImpl instance. A central FleetManager tracks all machines, aggregates inventory status, routes refill trucks. Cloud sync for analytics and remote monitoring.
How would you support pre-ordering via a mobile app?
Expose the CoffeeMachine interface as a REST API. Mobile app calls placeOrder() remotely. Order enters a queue. Machine processes queue in FIFO order. App receives push notification when beverage is ready.
Why State pattern for the machine instead of if/else?
(1) Each state class is self-documenting — legal transitions are explicit. (2) Adding CLEANING or WARMING_UP state is a new class, no modification to existing code. (3) Invalid transitions throw immediately with clear messages. (4) Scales much better than nested switch/if-else.
How do you handle ingredient expiry tracking?
Add expiryDate to each IngredientContainer entry. A background ScheduledExecutorService periodically checks expiry. Expired ingredients are marked unavailable. Machine enters MAINTENANCE if critical ingredients expire.
Why BigDecimal for prices instead of double?
double has floating-point precision issues (0.1 + 0.2 != 0.3). BigDecimal provides exact decimal arithmetic — critical for financial calculations. Always use compareTo() instead of equals() for BigDecimal comparisons.
How would you implement usage analytics?
Observer pattern on order completion. An AnalyticsObserver tracks: popular beverages (count by type), peak hours (timestamp histogram), revenue per beverage, ingredient consumption rates. Data stored in a time-series structure for trend analysis.
