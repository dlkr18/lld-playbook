01 — Requirements

Functional

  • Users register with name and email, manage profile (bio, picture)
  • Friend request lifecycle: Send → Accept / Reject / Cancel
  • Follow/unfollow system (asymmetric, separate from friends)
  • Block/unblock users (auto-removes friendship)
  • Create, update, delete posts with visibility (PUBLIC, FRIENDS_ONLY, PRIVATE)
  • Like/unlike posts and comments, comment with nested replies
  • Share posts to your feed, tag users in posts
  • News feed generation from friends + followed users
  • Direct messaging with delivery status (SENT → DELIVERED → READ)
  • Notifications for friend requests, likes, comments, follows
  • Search users by name/email, posts by content

Non-Functional

  • Performance — Response time < 100ms for critical operations
  • Scalability — Support 10,000+ concurrent users
  • Reliability — 99.9% uptime
  • Data Consistency — ACID transactions where needed
  • Extensible feed — Pluggable feed algorithms (chronological, ranked)
  • Observable events — Notifications on all key interactions

03 — Design Patterns

Four patterns working in concert to keep the design clean and extensible.

Strategy Pattern

FeedAlgorithm interface with ChronologicalFeedAlgorithm implementation. Swap feed ranking at runtime — add RankedFeedAlgorithm or EngagementFeedAlgorithm without changing the service.

Observer Pattern

NotificationService fires notifications on friend requests, likes, comments, and follows. Decouples event producers from delivery channels (console, push, email).

Repository Pattern

In-memory HashMap stores for all entities. Service layer never touches storage details directly. Easy to swap for database-backed implementation.

Interface Segregation

SocialNetworkService defines the full contract. FeedAlgorithm and NotificationService are separate focused interfaces. Each can evolve independently.

04 — Friend Request Lifecycle

Why a state model? Friend requests transition through well-defined states. Each transition triggers side effects (mutual friendship creation, notifications). Prevents invalid operations like accepting an already-rejected request. 
stateDiagram-v2
    [*] --> PENDING: sendFriendRequest()
    PENDING --> ACCEPTED: acceptFriendRequest()
    PENDING --> REJECTED: rejectFriendRequest()
    PENDING --> CANCELLED: cancel()
    ACCEPTED --> [*]: mutual friendship created
    REJECTED --> [*]
    CANCELLED --> [*]

FriendRequest Model

public class FriendRequest {
    private final String requestId, senderId, receiverId;
    private FriendRequestStatus status;        // PENDING initially
    private LocalDateTime sentAt, respondedAt;

    public void accept() {
        this.status = FriendRequestStatus.ACCEPTED;
        this.respondedAt = LocalDateTime.now();
    }

    public void reject() {
        this.status = FriendRequestStatus.REJECTED;
        this.respondedAt = LocalDateTime.now();
    }
}

Accept Flow — Mutual Friendship

public void acceptFriendRequest(String requestId) {
    FriendRequest request = friendRequests.get(requestId);
    if (request == null)
        throw new InvalidRequestException("Friend request not found");

    request.accept();

    // Bidirectional friendship
    User sender   = users.get(request.getSenderId());
    User receiver = users.get(request.getReceiverId());
    sender.addFriend(receiver.getUserId());
    receiver.addFriend(sender.getUserId());
}

Friendship is always bidirectional — both users get each other in their friendIds set. Following is asymmetric (A can follow B without B following A).

Block → Auto-Remove Friendship

public void blockUser(String userId, String blockedUserId) {
    User user = getUser(userId);
    user.blockUser(blockedUserId);
    removeFriend(userId, blockedUserId); // auto-unfriend on block
}

05 — News Feed Generation

Strategy Pattern — The feed algorithm is injected via the FeedAlgorithm interface. Swap from chronological to engagement-ranked without touching the service layer.

FeedAlgorithm Interface

public interface FeedAlgorithm {
    List<Post> generateFeed(User user, List<Post> allPosts, int limit);
}

ChronologicalFeedAlgorithm

public class ChronologicalFeedAlgorithm implements FeedAlgorithm {
    public List<Post> generateFeed(User user, List<Post> allPosts, int limit) {
        return allPosts.stream()
            .filter(p -> shouldShowInFeed(user, p))
            .sorted((a, b) -> b.getCreatedAt().compareTo(a.getCreatedAt()))
            .limit(limit)
            .collect(Collectors.toList());
    }

    private boolean shouldShowInFeed(User user, Post post) {
        return user.getUserId().equals(post.getAuthorId()) ||  // own posts
               user.isFriend(post.getAuthorId()) ||              // friends' posts
               user.isFollowing(post.getAuthorId());            // following posts
    }
}

Visibility Filtering (Service Layer)

private boolean isVisibleToUser(Post post, User viewer) {
    if (post.getVisibility() == PostVisibility.PUBLIC) return true;
    if (post.getAuthorId().equals(viewer.getUserId())) return true;
    if (post.getVisibility() == PostVisibility.FRIENDS_ONLY) {
        return viewer.isFriend(post.getAuthorId());
    }
    return false; // PRIVATE: only author sees it
}

Two-layer filtering: Service applies visibility rules first, then FeedAlgorithm applies relevance + sorting. Separation of concerns.

Like System — Set-Based Idempotency

// In Post.java — HashSet ensures no duplicate likes
private Set<String> likeUserIds = new HashSet<>();

public void like(String userId)    { likeUserIds.add(userId); }
public void unlike(String userId)  { likeUserIds.remove(userId); }
public int getLikesCount()          { return likeUserIds.size(); }
public boolean isLikedBy(String id) { return likeUserIds.contains(id); }

Key insight: Using Set<String> instead of a counter provides O(1) like/unlike/check and automatic idempotency — liking twice has no effect.

Nested Comment Replies

public class Comment {
    private final String commentId, postId, authorId;
    private String content;
    private Set<String> likeUserIds = new HashSet<>();
    private List<Comment> replies = new ArrayList<>();  // tree structure
    private boolean edited;

    public void addReply(Comment reply) { replies.add(reply); }
}

// Service creates reply and links to parent
public Comment replyToComment(String userId, String commentId, String content) {
    Comment parent = comments.get(commentId);
    Comment reply = new Comment(replyId, parent.getPostId(), userId, content);
    parent.addReply(reply);
    return reply;
}

Comments form a tree via composition. Each Comment holds a List of child replies. Supports arbitrary nesting depth.

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap All entity stores (production) Safe concurrent reads/writes on the data stores
HashSet<String> User.friendIds, Post.likeUserIds Idempotent add/remove, O(1) contains check
defensive copies All getter methods on collections Return new HashSet<>(friendIds) to prevent external mutation
immutable IDs userId, postId, commentId, etc. Declared final — set once in constructor, never changes
UUID.randomUUID() All entity creation Thread-safe ID generation without coordination
"All getters return defensive copies of internal collections. External code cannot corrupt the entity's state. In production, entity stores would use ConcurrentHashMap and critical sections would use synchronized blocks on the entity."

charlie.followingIds.add(alice), alice.followerIds.add(charlie). NotificationService fires USER_FOLLOWED to Alice.
Create Posts

post2 = createPost(bob, "Having a great day!", FRIENDS_ONLY)
Posts stored with UUID, timestamp, and visibility.
Like & Comment
likePost(bob, post1) — bob added to post1.likeUserIds
commentOnPost(charlie, post1, "Great post!") — Comment linked to post
Notifications: POST_LIKE to Alice, POST_COMMENT to Alice.
Generate News Feed

1. Filter by visibility (isVisibleToUser). 2. FeedAlgorithm filters by relationship (own + friends + following). 3. Sort by createdAt descending. 4. Limit to 10.

conversationId = sorted(alice, bob) joined by "_". Messages appended via computeIfAbsent.

11 — Interview Follow-up Q&A

How would you generate a ranked news feed instead of chronological?
Implement a new RankedFeedAlgorithm implementing FeedAlgorithm. Score posts by engagement (likes + comments + recency + author closeness). Inject it via constructor — zero changes to InMemorySocialNetworkService. The Strategy pattern makes this a one-class addition.
How would you handle the news feed at Facebook scale (billions of users)?
Two approaches: (1) Fan-out on write: when a user posts, push to all friends' pre-computed feed caches. Fast reads, expensive writes. Good for users with few friends. (2) Fan-out on read: build feed at read time by merging friends' posts. Slow reads, cheap writes. Good for celebrities. Facebook uses a hybrid: fan-out on write for normal users, fan-out on read for high-follower accounts.
Why use Set<String> for likes instead of a counter?
A Set<String> gives us: (1) Idempotency — liking twice has no effect. (2) Unlike support — remove by userId. (3) isLikedBy() — O(1) check for UI state. (4) Who liked — can retrieve the full list. A simple counter loses all this information.
How would you handle real-time notifications at scale?
Replace SimpleNotificationService with a WebSocket/SSE-based implementation. Use a message queue (Kafka/RabbitMQ) between event producers and notification delivery. The NotificationService interface stays the same — just swap the implementation.
How do you prevent duplicate friend requests?
Before creating a new request, check: (1) Is there already a PENDING request between these users? (2) Are they already friends? (3) Is either user blocked? Add these validations in sendFriendRequest(). The current implementation validates user existence; production would add these checks.
How would you implement the graph for mutual friends?
Each User's friendIds is a Set. Mutual friends = user1.getFriendIds().retainAll(user2.getFriendIds()) — set intersection in O(min(n,m)). At scale, use a graph database (Neo4j) or precompute with adjacency lists in Redis.
Why is friendship bidirectional but following asymmetric?
Friendship is a mutual relationship — both users consent via the request/accept flow, so both get each other in friendIds. Following is one-directional — A follows B without B's consent. This mirrors real platforms: Facebook friends vs Twitter follows.
How does the conversation ID work for messaging?
getConversationId() sorts both userIds alphabetically and joins with "_". This ensures Alice-to-Bob and Bob-to-Alice always map to the same conversation key. No duplication, no lookup confusion. computeIfAbsent() lazily creates the message list.
How would you add privacy controls for who can message you?
Add a privacySettings map on User with keys like MESSAGE_PERMISSION: FRIENDS_ONLY | EVERYONE | NOBODY. Check in sendMessage() before creating the message. Block list is already implemented — blocked users can't message. Extend with follower-only or mutual-friends-only.
How would you handle post deletion with comments and likes?
Current implementation removes the post from the map. In production: (1) Soft delete — mark as deleted, filter from feeds, but keep data for auditing. (2) Cascade delete — remove associated comments from the comments map. (3) Orphan cleanup — background job to clean up orphaned notifications referencing deleted posts.
How would you implement hashtags and trending topics?
Extract hashtags from post content via regex. Maintain a Map<String, List<String>> mapping hashtag to postIds (inverted index). For trending: use a sliding window counter (e.g., Redis sorted set with timestamp scores) to track hashtag frequency over the last N hours.
Why defensive copies on all collection getters?
Without defensive copies, external code could do user.getFriendIds().clear() and destroy the internal state. Returning new HashSet<>(friendIds) means the caller gets a snapshot — mutations don't affect the original. Essential for encapsulation and thread safety.
