01 — Requirements

Functional

  • User Management — register as buyer/seller/both, profiles, verified seller status
  • Item Listing — create auction with starting price, reserve price, buy-now price, duration (1–10 days)
  • Bidding — place bid > current highest + increment, proxy bidding, bid retraction
  • Auction Lifecycle — SCHEDULED → ACTIVE → CLOSED → SOLD/UNSOLD
  • Soft Close — extend auction if bid placed in last minute
  • Winner Determination — highest bidder wins if reserve met, notify winner + seller
  • Payment — winner pays within 48h, escrow service, seller paid after shipment
  • Search — keyword, category, price range, sort by ending soon / popular / new

Non-Functional

  • Performance — bid placement < 100ms, 10,000+ concurrent auctions, real-time updates (WebSocket)
  • Concurrency — thread-safe bid placement, atomic highest-bid updates, optimistic locking
  • Reliability — 99.9% uptime, no lost bids, accurate winner determination
  • Scalability — horizontal scaling, DB sharding by auction_id, Redis caching for active auctions
  • Fraud Prevention — shill bidding detection, bid pattern analysis, verified sellers

03 — Design Patterns

Six patterns, each solving a distinct auction concern.

State Pattern

Auction lifecycle: SCHEDULED → ACTIVE → CLOSED → SOLD/UNSOLD. Each status defines legal transitions. Invalid ops throw immediately. Adding a new status = new class.

Observer Pattern

Notify bidders on outbid, auction end, winner announcement. Real-time push via WebSocket or console fallback.

Strategy Pattern

Different bidding strategies (proxy, sniping, manual). Pluggable bid logic swappable at runtime without modifying service code.

Command Pattern

Bid operations (Place, Retract, Cancel) encapsulated as commands. Enables undo/redo and a full audit trail of every action.

Scheduler Pattern

ScheduledExecutorService auto-closes auctions at their end time. Guarantees lifecycle completion even without user action.

Factory Pattern

Centralized creation of Auction and Bid objects. Assigns UUIDs, timestamps, and initial states consistently.

04 — Auction Lifecycle

Why State Pattern? Eliminates scattered if/else. Each status class defines EXACTLY which transitions are legal. Adding a new status (e.g., PAUSED) = adding a new class. Open/Closed Principle.

AuctionStatus Enum

public enum AuctionStatus {
    SCHEDULED,   // before start time
    ACTIVE,      // accepting bids
    COMPLETED,   // end time reached, no more bids
    CANCELLED    // seller cancelled before start
}

Auction State Transitions (model/Auction.java)

public void start() {
    this.status = AuctionStatus.ACTIVE;
}

public void end() {
    this.status = AuctionStatus.COMPLETED;
}

public boolean isActive() {
    LocalDateTime now = LocalDateTime.now();
    return status == AuctionStatus.ACTIVE && now.isBefore(endTime);
}

All mutable state lives on Auction. Status transitions are guarded — placeBid() checks status == ACTIVE before accepting.

05 — Bid Validation & Placement

Critical path. Bid placement must be thread-safe, validate amount > current highest, and update state atomically. This is the most-asked code in interviews.

Synchronized placeBid (model/Auction.java)

public synchronized boolean placeBid(Bid bid) {
    if (status != AuctionStatus.ACTIVE) {
        return false;
    }

    if (bid.getAmount().compareTo(currentPrice) <= 0) {
        return false;   // must be strictly greater
    }

    bids.add(bid);
    currentPrice = bid.getAmount();
    winningBidderId = bid.getBidderId();
    return true;
}

Service-Level Orchestration (AuctionServiceImpl)

public boolean placeBid(String auctionId, String bidderId, BigDecimal amount) {
    Auction auction = auctions.get(auctionId);
    if (auction == null) return false;

    String bidId = UUID.randomUUID().toString();
    Bid bid = new Bid(bidId, auctionId, bidderId, amount);

    if (auction.placeBid(bid)) {
        bid.accept();   // PENDING → ACCEPTED
        return true;
    }
    bid.reject();       // PENDING → REJECTED
    return false;
}

Race Condition: Why Synchronized?

// ❌ BAD: Check-then-update without lock
if (bidAmount > auction.getCurrentPrice()) {
    auction.setCurrentPrice(bidAmount); // RACE CONDITION!
}

// ✅ GOOD: synchronized on the Auction object
public synchronized boolean placeBid(Bid bid) {
    // only one thread enters at a time per auction
    if (bid.getAmount().compareTo(currentPrice) <= 0) return false;
    bids.add(bid);
    currentPrice = bid.getAmount();
    return true;
}

Two users see same highest bid and both think they won — synchronized prevents this. Fine-grained lock per Auction object.

06 — Proxy Bidding Algorithm

What is proxy bidding? User sets a maximum. System auto-bids the minimum increment above each competitor until the max is reached or the user wins.

Proxy Bid Logic

public Bid handleProxyBid(User bidder, double maxAmount, Auction auction) {
    double currentBid = auction.getCurrentHighestBid();
    double nextIncrement = MIN_INCREMENT;

    double bidAmount = currentBid + nextIncrement;

    while (bidAmount <= maxAmount &&
           auction.getCurrentHighestBid() > bidAmount) {
        bidAmount = auction.getCurrentHighestBid() + nextIncrement;
        if (bidAmount <= maxAmount) {
            auction.placeBid(bidder, bidAmount);
        }
    }
    return auction.getHighestBid();
}

07 — Auto-Close Scheduling

Why scheduled? Auctions must close at their exact end time, even if no one is actively watching. ScheduledExecutorService guarantees this.

Schedule Auction Close

public void scheduleAuctionClose(Auction auction) {
    long delay = ChronoUnit.SECONDS.between(
        LocalDateTime.now(),
        auction.getEndTime()
    );

    scheduler.schedule(() -> {
        closeAuction(auction);
    }, delay, TimeUnit.SECONDS);
}

Close Auction + Winner Determination

private void closeAuction(Auction auction) {
    auction.setStatus(AuctionStatus.CLOSED);

    User winner = auction.determineWinner();
    if (winner != null &&
        auction.getCurrentHighestBid() >= auction.getReservePrice()) {
        auction.setStatus(AuctionStatus.SOLD);
        notifyWinner(winner, auction);
        notifySeller(auction.getSeller(), auction);
    } else {
        auction.setStatus(AuctionStatus.UNSOLD);
    }
}

Soft Close (Anti-Sniping)

If a bid is placed in the last 60 seconds, extend the auction by 60 seconds. Prevents last-second sniping and gives all bidders a fair chance.

if (ChronoUnit.SECONDS.between(now, endTime) < 60) {
    auction.setEndTime(endTime.plusSeconds(60));
    rescheduleClose(auction);
}

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap auctions store in AuctionServiceImpl Safe concurrent reads/writes for auction lookups
synchronized(this) Auction.placeBid() Fine-grained lock per auction — only one thread bids on a specific auction at a time
BigDecimal Bid.amount, Auction.currentPrice Immutable, precise decimal — no floating-point errors for monetary values
UUID ID generation for auctions and bids Globally unique, no coordination needed
ScheduledExecutorService Auto-close scheduling Thread-pool based scheduling, handles concurrent auction closures
ArrayList (guarded) Auction.bids Accessed only inside synchronized placeBid — no external mutation
"I use fine-grained locking on each Auction object, not a global lock. Two different auctions process bids concurrently. Only concurrent bids on the same auction are serialized."

12 — Winner Determination

Algorithm

public void endAuction(String auctionId) {
    Auction auction = auctions.get(auctionId);
    if (auction != null) {
        auction.end();  // ACTIVE → COMPLETED

        if (auction.getWinningBidderId() != null) {
            System.out.println("Winner: " + auction.getWinningBidderId()
                + " with bid: $" + auction.getCurrentPrice());
        }
    }
}

Winner Logic

ScenarioResult
Bids exist AND highest ≥ reserve priceSOLD — highest bidder wins
Bids exist BUT highest < reserve priceUNSOLD — reserve not met
No bids at allUNSOLD — no interest
Buy-now price clickedSOLD immediately — auction closes early

getWinningBid (simple Auction.java)

public Bid getWinningBid() {
    return bids.isEmpty() ? null : bids.get(bids.size() - 1);
}

public double getCurrentPrice() {
    return bids.isEmpty() ? startingPrice : bids.get(bids.size() - 1).getAmount();
}

Since bids are append-only and validated on insert, the last element is always the highest. O(1) winner lookup.

13 — Interview Q&A

How do you handle concurrent bids on the same auction?
synchronized on the Auction object — fine-grained locking. Only one thread enters placeBid() at a time for a specific auction. Two different auctions process bids concurrently. In production, use optimistic locking (@Version) for database-level concurrency.
What is proxy bidding and how does it work?
User sets a maximum amount ($500). System places the minimum bid ($100). When another user bids higher ($110), the system auto-bids $120 on the proxy user's behalf. This continues until the max is exceeded or the auction ends. Implemented as a while-loop with increment logic.
How do you prevent a seller from bidding on their own auction (shill bidding)?
Check bidderId != auction.getSellerId() before accepting the bid. Flag and log any attempts. In production: cross-reference IP addresses, payment methods, and bid patterns across related accounts. Repeated violations = account suspension.
What happens if the auction close is delayed due to system load?
ScheduledExecutorService provides best-effort scheduling. For production: use a distributed scheduler (Quartz, or a time-ordered Kafka topic). The auction's isActive() method also checks now < endTime as a guard — no bids accepted after the deadline even if the close task is delayed.
How do you handle the "last-second sniping" problem?
Soft close — if a bid is placed in the last 60 seconds, extend the end time by 60 seconds. Reschedule the auto-close task. This gives all bidders a fair chance to respond. eBay uses a similar mechanism called "automatic extension."
Why BigDecimal instead of double for monetary values?
double has floating-point precision issues: 0.1 + 0.2 = 0.30000000000000004. In an auction, a $0.01 error could determine the winner. BigDecimal provides exact decimal arithmetic. All monetary comparisons use compareTo(), not ==.
How would you scale this for Black Friday with 100x traffic?
(1) Bid queue: Kafka to absorb bid spikes, process asynchronously. (2) Redis caching: Cache active auction data, current price. (3) DB sharding: Shard by auction_id. (4) WebSocket: Push updates instead of polling. (5) Horizontal scaling: Stateless service behind load balancer. (6) Read replicas: Separate DB for search/browse.
Why State pattern for auction lifecycle instead of if/else?
(1) Each state class is self-documenting — you see exactly what's legal. (2) Adding a new state (e.g., PAUSED, EXTENDED) = new class, no modification to existing code (Open/Closed). (3) Invalid transitions throw immediately with clear messages. (4) No forgotten else branches or duplicated checks across methods.
How do you handle reserve prices?
Reserve price is set at auction creation and is hidden from bidders. When the auction closes, compare the highest bid to the reserve. If highest ≥ reserve → SOLD. If highest < reserve → UNSOLD. The seller is protected from selling too cheaply.
How would you implement real-time bid notifications?
WebSocket for real-time push: each auction has a "room" that watchers subscribe to. On each new bid, broadcast the new price to all subscribers. SSE (Server-Sent Events) as an alternative for one-way push. Polling as fallback for older clients. The Observer pattern in the codebase maps directly to this.
What's the difference between the two Auction classes in the codebase?
Auction.java (root package) is a simplified version with double prices and basic enum. model/Auction.java is the production-grade version with BigDecimal, full state machine, synchronized bidding, and proper getters. In production, you'd keep only model/Auction.java.
How do you ensure no bids are lost under high concurrency?
(1) synchronized ensures serial bid processing per auction. (2) Bids are append-only — never deleted, forming an audit trail. (3) ConcurrentHashMap for safe auction lookups. (4) In production: Kafka provides durable message storage — bids persist even if the service crashes.

Key Takeaway

The bid placement and auction close mechanisms are the most critical components — they must be thread-safe, accurate, and handle high concurrency during peak times (last-minute bidding rush). Everything else is wiring.
