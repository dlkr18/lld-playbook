01 — Requirements

Functional

  • CRUD — Create, update, delete feature flags
  • Enable/Disable — Toggle flags globally without code deployment
  • Rollout strategies — Percentage-based, user-targeted, attribute-based, ring-based
  • Flag types — Boolean, String, Number, JSON
  • Rule engine — Priority-based rules with conditions and operators
  • Real-time updates — Changes propagate immediately with cache invalidation
  • Fast evaluation — < 10ms with fallback to defaults on errors

Non-Functional

  • Performance — Flag evaluation < 10ms, support 100K+ QPS
  • Availability — 99.99% uptime
  • Consistency — Eventually consistent across regions
  • Scalability — Millions of flags, billions of evaluations
  • Audit — Track all flag changes and evaluations

03 — Design Patterns

Five patterns, each solving a specific concern in the feature flag system.

Strategy Pattern

RolloutStrategy interface with PercentageRollout, UserTargetRollout, and AttributeRollout implementations. Each defines shouldApply(context) independently. Adding a new rollout type = new class, zero changes elsewhere.

Cache-Aside Pattern

FlagCache sits between service and repository. On miss: load from repo, populate cache. On update: invalidate cache entry, publish event. TTL-based expiration prevents stale reads.

Observer Pattern

Flag update events propagate to subscribers via FlagUpdatedEvent. Each subscriber calls onFlagUpdate(flagKey) to invalidate local caches across services.

Builder Pattern

FeatureFlag, Rule, and EvaluationContext all use fluent builders. Clean API: FeatureFlag.builder().key("x").type(BOOLEAN).build()

Template Method

Rule evaluation follows a fixed template: check conditions → check rollout strategy → return value. Subclasses customize condition matching and strategy application.

05 — Rollout Strategies

Why Strategy? Rollout algorithms change independently of the flag system. New rollout type = new class implementing RolloutStrategy. Injected at flag creation, swappable without touching the evaluation engine.

TargetingRule Interface

public interface TargetingRule {
    boolean matches(User user);
}

Implementations

StrategyLogicUse Case
PercentageRollout hash(userId) % 100 < percentage Gradual rollout: 1% → 5% → 25% → 100%
UserTargetRollout userIds.contains(userId) Internal testing, beta users
AttributeRollout context attributes match required attributes Premium users, specific regions
GroupTargetingRule user.getGroup().equals(targetGroup) Group-based targeting in this implementation

GroupTargetingRule Implementation

public class GroupTargetingRule implements TargetingRule {
    private final String targetGroup;

    public GroupTargetingRule(String targetGroup) {
        this.targetGroup = targetGroup;
    }

    public boolean matches(User user) {
        return user.getGroup().equals(targetGroup);
    }
}

RolloutStrategy Value Object

public class RolloutStrategy {
    private final String type;        // "percentage", "userTarget", "attribute"
    private final int percentage;     // 0-100 for percentage rollout
    private final String targetGroup; // group to target
}

Core Evaluation Logic

public boolean isFeatureEnabled(String featureId, User user) {
    Feature feature = features.get(featureId);
    if (feature == null || !feature.isEnabled()) {
        return false;
    }

    for (RolloutStrategy strategy : feature.getStrategies().values()) {
        if (strategy.getTargetGroup().equals(user.getGroup())) {
            int hash = Math.abs(user.getId().hashCode() % 100);
            return hash < strategy.getPercentage();
        }
    }

    return true; // enabled with no matching strategy = on for all
}

Key insight: If the feature is enabled and no rollout strategy matches the user's group, the feature is ON for that user. Strategies act as restrictors, not enablers.

07 — Key Algorithms

Algorithm 1: Flag Evaluation

Algorithm: EvaluateFlag(key, context)
Input:  flag key (string), evaluation context
Output: flag value (any type)

1. flag = cache.get(key)
2. if flag == null:
      flag = repository.findByKey(key)
      cache.put(key, flag)

3. if !flag.enabled:
      return flag.defaultValue

4. rules = flag.rules sorted by priority DESC

5. for each rule in rules:
      if rule.matches(context):
         if rule.strategy.shouldApply(context):
            return rule.value

6. return flag.defaultValue

Time: O(R × C) where R = rules, C = conditions per rule. Space: O(1) for evaluation.

Algorithm 2: Percentage Rollout (Consistent Hashing)

Algorithm: PercentageRollout(userId, percentage)
Input:  user ID, target percentage (0-100)
Output: boolean (should apply)

1. hash = murmur3(flagKey + userId)
2. bucket = hash % 100
3. return bucket < percentage
PropertyGuarantee
ConsistentSame user always gets same result for a given flag
UniformEven distribution across buckets
DeterministicNo randomness — reproducible across evaluations

Algorithm 3: Rule Matching

Algorithm: RuleMatches(rule, context)
Input:  rule with conditions, evaluation context
Output: boolean (rule matches)

1. if rule.conditions is empty:
      return true

2. for each condition in rule.conditions:
      attribute = context.getAttribute(condition.attribute)
      if !condition.operator.evaluate(attribute, condition.value):
         return false

3. return true // All conditions matched (AND semantics)

Algorithm 4: Cache Invalidation

Algorithm: InvalidateCache(flagKey)
Input:  flag key that was updated

1. cache.invalidate(flagKey)
2. publishEvent(new FlagUpdatedEvent(flagKey))
3. for each subscriber:
      subscriber.onFlagUpdate(flagKey)
TriggerAction
Flag enabled/disabledInvalidate + publish event
Rules added/removedInvalidate + publish event
Default value changedInvalidate + publish event
Flag deletedInvalidate + publish event

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap FeatureFlagServiceImpl.features Safe concurrent reads/writes to the feature store
HashMap (per Feature) Feature.strategies Strategies are added at config time; reads during evaluation
boolean volatile Feature.enabled, FeatureToggle.enabled Cross-thread visibility of enable/disable without full lock
Immutable fields User (id, email, group), RolloutStrategy (type, %, group) Thread-safe by design — no mutable shared state
LocalDateTime.now() AuditLog, FeatureToggle timestamps Thread-safe clock reads for audit trail
"ConcurrentHashMap for the feature store ensures safe concurrent flag lookups. Individual Feature fields use immutability (final) and volatile where needed. RolloutStrategy is an immutable value object — no locks required."

10 — Happy Path Flow

Trace this end-to-end in the interview.

Create Feature
service.createFeature("new-ui", "New UI Design")
Feature stored in ConcurrentHashMap, enabled = false
Enable Feature
service.enableFeature("new-ui")
feature.enable() → enabled = true. Feature is now ON globally.
Add Rollout Strategy
service.addRolloutStrategy("new-ui", "beta", new RolloutStrategy("percentage", 50, "beta-users"))
50% of users in "beta-users" group will see the new UI.
Create Users
user1 = new User("U1", "alice@test.com", "beta-users")
user2 = new User("U2", "bob@test.com", "regular")
Alice is in "beta-users" group, Bob is in "regular" group.
Evaluate Flag — Alice
service.isFeatureEnabled("new-ui", user1)
Feature enabled? Yes. Strategy matches "beta-users"? Yes. hash("U1") % 100 < 50? Depends on hash. Alice gets a deterministic result.
Evaluate Flag — Bob
service.isFeatureEnabled("new-ui", user2)
Feature enabled? Yes. Strategy matches "regular"? No matching strategy → return true. Bob sees the feature (no restriction applies).
Audit Trail
AuditLog records: [timestamp] Feature new-ui: ENABLED for user U1
Every evaluation can be logged with featureId, userId, result, and timestamp.

12 — Interview Follow-up Q&A

How would you handle flag evaluation at scale (100K+ QPS)?
Multi-level caching: L1 in-process ConcurrentHashMap (per JVM, <1ms), L2 Redis (shared, <5ms), L3 database (source of truth). Pre-compute flag states for known user segments. Use bloom filters for negative lookups. Batch evaluations for bulk requests.
How to ensure consistency across microservices?
Use event sourcing for flag updates. Publish changes via Kafka/Redis pub-sub. Each service subscribes to flag-update events and invalidates its local L1 cache. TTL-based expiration as a safety net ensures eventual consistency even if events are missed.
How do you prevent flag sprawl (too many stale flags)?
Enforce a flag lifecycle: creation → rollout → stable → cleanup. Set mandatory expiration dates. Automated cleanup after N days at 100%. Flag usage analytics dashboard. Required documentation (owner, purpose, expected removal date) for each flag.
Why consistent hashing for percentage rollout instead of random?
Determinism — same user always gets same result. Consistency — increasing from 10% to 20% doesn't change existing users' assignments, it only adds new ones. Reproducibility — easy to debug. No state — no need to store per-user assignments.
How would you implement A/B testing on top of this?
Use String flags with variants ("control", "variant_A", "variant_B"). Assign variants via consistent hashing with bucket ranges (0-33: control, 34-66: A, 67-100: B). Track metrics per variant. Calculate statistical significance before declaring a winner.
How to handle emergency kill switches?
A kill switch is a boolean flag with no rules — just enabled/disabled. disableFeature() immediately sets enabled=false. The ConcurrentHashMap ensures the change is visible to all evaluation threads. In production, push the change to Redis pub-sub for instant cross-service propagation.
Why Builder pattern for FeatureFlag and Rule?
(1) Both have many optional parameters — telescoping constructors become unreadable. (2) Fluent API reads like English: FeatureFlag.builder().key("x").type(BOOLEAN).enabled(true).build(). (3) Immutable after construction — the builder collects params, then constructs a frozen object. Thread-safe by design.
How do you test feature flags?
Unit tests: Override flags in test config. Integration tests: Mock the evaluation context. Statistical tests: Verify percentage rollouts are uniform (run 10K users, assert ~50% get flag). E2E: Use UserTargetRollout with test user IDs to force specific paths.
Why ConcurrentHashMap and not synchronized HashMap?
ConcurrentHashMap uses lock striping (segments), allowing concurrent reads and writes to different segments. A synchronized HashMap locks the entire map for every operation. For a feature flag system with 100K+ QPS reads and rare writes, ConcurrentHashMap's read performance is critical.
How would you handle flag dependencies (flag A requires flag B)?
Model dependencies as conditions in rules. A rule for flag A includes a condition that evaluates flag B's state. Use EvaluationContext to carry evaluated flag states. Detect circular dependencies at flag creation time using topological sort. Evaluate in dependency order.
Why separate TargetingRule interface from RolloutStrategy?
TargetingRule answers "WHO qualifies?" (group membership, attributes). RolloutStrategy answers "WHAT percentage of qualified users?" They're orthogonal concerns. A flag can target "premium" users (targeting) AND roll out to 10% of them (rollout). Separation enables flexible composition.
How to handle flag evaluation failures gracefully?
Always fall back to the default value. The evaluation algorithm's final line is return flag.defaultValue. Wrap the entire evaluation in try-catch. Log failures for alerting. In production, circuit-breaker pattern: if the flag service is down, use cached values or defaults for the entire TTL period.
