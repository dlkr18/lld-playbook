Restaurant Management // LLD

Thread-safe in-memory restaurant system with table management, reservations, order lifecycle, menu handling, and billing. 14 files, clean separation of concerns.

Repository Pattern Strategy Pattern Factory Pattern Observer Pattern Fine-grained Locking

01 — Requirements

Functional

  • Table management — add tables with capacity, track status (AVAILABLE / OCCUPIED / RESERVED)
  • Reservation system — reserve tables by customer, party size, date/time; cancel reservations
  • Order lifecycle — PENDING → PREPARING → READY → SERVED → PAID
  • Menu system — add items with id, name, price; attach items to orders
  • Billing — generate itemized bill with 10% tax; auto-free table on payment
  • Table finder — find the smallest available table that fits a given party size
  • Search & filtering — get all available tables, query by status

Non-Functional

  • Thread-safe — ConcurrentHashMap for stores, synchronized per-table for reservations & orders
  • Performance — O(1) table/order lookups via HashMap, O(n) best-fit table search
  • Extensible — interface-based service, easy to swap in-memory for DB-backed impl
  • ACID-ready — per-table locking ensures consistency for concurrent operations
  • Scalable — stateless service layer, 10,000+ concurrent users ready

03 — Design Patterns

Four patterns, each solving a specific restaurant domain concern.

Repository Pattern

All data access goes through ConcurrentHashMap stores. Service interface abstracts storage — swap InMemoryRestaurantService for a DB-backed implementation with zero changes to callers.

Strategy Pattern

Table allocation uses best-fit strategy: find the smallest available table that fits the party size. Pricing and allocation algorithms are pluggable.

Factory Pattern

ID generation via AtomicLong counters with prefixes: RES-1, ORD-1, BILL-1. Centralized creation ensures unique, predictable identifiers.

Observer Pattern

Notification support for status updates — kitchen gets notified of new orders, waitstaff of ready orders. The NotificationService interface is pluggable (console, SMS, push).

04 — Table Management

Why per-table locking? Multiple customers can reserve different tables concurrently. Only operations on the same table are serialized. No global lock bottleneck. 
stateDiagram-v2
    [*] --> AVAILABLE
    AVAILABLE --> RESERVED: makeReservation()
    AVAILABLE --> OCCUPIED: createOrder()
    RESERVED --> AVAILABLE: cancelReservation()
    RESERVED --> OCCUPIED: createOrder()
    OCCUPIED --> AVAILABLE: generateBill() [payment]

Table Entity

public class Table {
    private String tableId;
    private int capacity;
    private TableStatus status;  // AVAILABLE by default

    public Table(String id, int cap) {
        tableId = id;
        capacity = cap;
        status = TableStatus.AVAILABLE;
    }
}

Best-Fit Table Finder

public Table findTable(int partySize) {
    Table best = null;
    for (Table t : tables.values()) {
        if (t.getStatus() == TableStatus.AVAILABLE
                && t.getCapacity() >= partySize) {
            if (best == null || t.getCapacity() < best.getCapacity()) {
                best = t;  // smallest table that fits
            }
        }
    }
    return best;
}

Time: O(n) scan — at scale, use a TreeMap keyed by capacity for O(log n) ceiling lookup.

Order Entity

public class Order {
    private String orderId;
    private List<MenuItem> items = new ArrayList<>();
    private OrderStatus status;

    public Order(String id) {
        orderId = id;
        status = OrderStatus.PENDING;
    }

    public void addItem(MenuItem i) { items.add(i); }
    public void setStatus(OrderStatus s) { status = s; }
}

Status Guard — Items Only in PENDING

public void addItemToOrder(String orderId, MenuItem item) {
    Order order = orders.get(orderId);
    if (order == null) throw new OrderNotFoundException(...);
    if (order.getStatus() != OrderStatus.PENDING) {
        throw new IllegalStateException(
            "Cannot add items to order in state: " + order.getStatus());
    }
    order.addItem(item);
}

07 — Reservation System

Thread-safe reservations: synchronized(table) prevents two customers from reserving the same table simultaneously. The lock is per-table, not global.

Reservation Entity

public class Reservation {
    private String reservationId, customerId, tableId;
    private LocalDateTime dateTime;
    private int partySize;

    public Reservation(String id, String cid, String tid,
                        LocalDateTime dt, int size) {
        reservationId = id; customerId = cid;
        tableId = tid; dateTime = dt; partySize = size;
    }
}

Make Reservation — Validates & Locks

public Reservation makeReservation(String customerId, String tableId,
                                    LocalDateTime dateTime, int partySize) {
    Table table = getTable(tableId);

    synchronized (table) {
        if (table.getStatus() != TableStatus.AVAILABLE)
            throw new IllegalStateException("Table not available");
        if (partySize > table.getCapacity())
            throw new IllegalArgumentException("Party exceeds capacity");

        String resId = "RES-" + reservationCounter.incrementAndGet();
        Reservation res = new Reservation(resId, customerId, tableId, dateTime, partySize);
        reservations.put(resId, res);
        table.setStatus(TableStatus.RESERVED);
        return res;
    }
}

Cancel Reservation — Frees Table

public void cancelReservation(String reservationId) {
    Reservation res = reservations.get(reservationId);
    if (res == null) throw new ReservationNotFoundException(...);
    Table table = tables.get(res.getTableId());
    if (table != null) {
        synchronized (table) {
            if (table.getStatus() == TableStatus.RESERVED)
                table.setStatus(TableStatus.AVAILABLE);
        }
    }
    reservations.remove(reservationId);
}

08 — Billing

Bill Entity — Auto-Calculated Tax

public class Bill {
    private String billId;
    private double amount, tax, total;

    public Bill(String id, double amt) {
        billId = id;
        amount = amt;           // subtotal from items
        tax = amt * 0.1;       // 10% tax
        total = amount + tax;   // grand total
    }
    public double getTotal() { return total; }
}

Generate Bill — Sum Items + Free Table

public Bill generateBill(String orderId) {
    Order order = orders.get(orderId);
    if (order == null) throw new OrderNotFoundException(...);

    double subtotal = 0;
    for (MenuItem item : order.getItems()) {
        subtotal += item.getPrice();
    }

    String billId = "BILL-" + billCounter.incrementAndGet();
    Bill bill = new Bill(billId, subtotal);
    order.setStatus(OrderStatus.PAID);

    // Free the table via tableToOrder reverse lookup
    for (Map.Entry<String, String> entry : tableToOrder.entrySet()) {
        if (entry.getValue().equals(orderId)) {
            Table table = tables.get(entry.getKey());
            synchronized (table) {
                table.setStatus(TableStatus.AVAILABLE);
            }
            tableToOrder.remove(entry.getKey());
            break;
        }
    }
    return bill;
}

09 — Key Algorithms

Best-Fit Table Allocation

Time: O(n)    Space: O(1)

// Find smallest available table that fits the party
public Table findTable(int partySize) {
    Table best = null;
    for (Table t : tables.values()) {
        if (t.getStatus() == TableStatus.AVAILABLE
                && t.getCapacity() >= partySize) {
            if (best == null || t.getCapacity() < best.getCapacity())
                best = t;
        }
    }
    return best;
}

"At scale, use a TreeMap<Integer, List<Table>> keyed by capacity for O(log n) ceiling lookups."

Bill Calculation

Time: O(k) where k = number of items    Space: O(1)

double subtotal = 0;
for (MenuItem item : order.getItems()) {
    subtotal += item.getPrice();
}
// Or using streams:
double subtotal = order.getItems().stream()
    .mapToDouble(MenuItem::getPrice).sum();

Table-to-Order Reverse Lookup

Time: O(m) where m = active orders    Space: O(m) for the mapping

// tableToOrder: ConcurrentHashMap<String, String> (tableId -> orderId)
// On createOrder: tableToOrder.put(tableId, orderId)
// On generateBill: scan for orderId, free table, remove mapping

"In production, add an orderToTable reverse map for O(1) lookup instead of linear scan."

10 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap tables, reservations, orders, tableToOrder Safe concurrent reads/writes to all data stores
synchronized(table) makeReservation, cancelReservation, createOrder, generateBill Fine-grained lock per table — two different tables can be reserved concurrently
AtomicLong reservationCounter, orderCounter, billCounter Lock-free ID generation (RES-1, ORD-1, BILL-1)
"I use fine-grained locking on the Table object, not a global lock. Two different tables can be reserved concurrently. Only concurrent operations on the same table are serialized."

14 — Interview Follow-up Q&A

How do you prevent two customers from reserving the same table?
synchronized(table) — only one thread enters the reservation block for a specific table at a time. The first thread sets the table to RESERVED. The second thread sees the table is no longer AVAILABLE and throws IllegalStateException. Different tables can be reserved concurrently since the lock is per-table, not global.
Why use ConcurrentHashMap instead of synchronized HashMap?
ConcurrentHashMap uses segment-level locking (Java 8+ uses CAS + fine-grained node locks), allowing concurrent reads without blocking and concurrent writes to different segments. A synchronized HashMap would serialize ALL operations globally. For a restaurant with dozens of tables and orders, CHM gives much better throughput.
How would you handle time-slot based reservations?
Add a Map<String, List<Reservation>> keyed by tableId. When making a reservation, check for overlapping time slots: existingRes.dateTime within ±2 hours of requested time. This allows the same table to have reservations for different time windows (lunch vs dinner).
How would you scale the table-finding algorithm?
Replace the O(n) scan with a TreeMap<Integer, Set<Table>> keyed by capacity. Use ceilingKey(partySize) for O(log n) best-fit lookup. For multi-restaurant chains, partition by restaurant ID and use a distributed cache (Redis sorted sets by capacity).
Why is the tableToOrder reverse lookup O(n)? How would you fix it?
Currently, generateBill() iterates tableToOrder.entrySet() to find which table has the given orderId. Fix: add a second map orderToTable: Map<String, String> (orderId → tableId) for O(1) reverse lookup. Maintain both maps in sync during createOrder() and generateBill().
How would you add menu categories and search?
Add a category field to MenuItem (APPETIZER, MAIN, DESSERT, DRINK). Maintain a Map<String, List<MenuItem>> indexed by category. For text search, use a Trie or inverted index on item names. The MenuItem is already a value object — just extend it.
How would you handle order modifications after PREPARING?
The current guard (status != PENDING) prevents it. To support modifications: allow adding items until READY (not just PENDING), and track each item's individual status. Alternatively, create a separate "supplementary order" linked to the original — common in real restaurants.
How would you implement split billing?
Add a generateSplitBill(orderId, Map<String, List<String>> customerToItems) method. Each customer gets a Bill for their items. The order only transitions to PAID when ALL split bills are paid. Use an AtomicInteger countdown or a Set<String> of unpaid bill IDs.
How would you add tips and discounts to the billing system?
Use the Strategy pattern: define a BillingStrategy interface with calculateTotal(subtotal). Implementations: StandardBilling (10% tax), DiscountBilling (decorator with percentage off), TipBilling (decorator adding gratuity). Composable: new TipBilling(new DiscountBilling(standard, 0.15), 0.18).
How would you add a waitlist when all tables are full?
Add a PriorityQueue<WaitlistEntry> sorted by arrival time. When findTable() returns null, add the customer to the waitlist. When a table becomes AVAILABLE (in generateBill() or cancelReservation()), poll the waitlist and notify the next eligible party via NotificationService.
How would you persist this to a database?
The RestaurantService interface stays the same. Create a JpaRestaurantService or JdbcRestaurantService implementation. Use @Entity annotations on model classes. Replace ConcurrentHashMap with JPA repositories. The synchronized(table) becomes a database-level SELECT ... FOR UPDATE or optimistic locking with @Version.
How would you handle kitchen order display (KDS)?
Implement the Observer pattern: add a KitchenDisplay that observes order status changes. When an order moves to PREPARING, the KDS shows it. When it moves to READY, remove from KDS and alert waitstaff. Use CopyOnWriteArrayList for thread-safe observer iteration.

16 — Patterns Summary

PatternWhereWhy
Repository ConcurrentHashMap stores behind RestaurantService interface Abstracts storage; swap in-memory for DB with zero caller changes
Strategy findTable() best-fit algorithm; extensible to BillingStrategy Swap allocation / pricing algorithm without changing service
Factory AtomicLong counters (RES-N, ORD-N, BILL-N) Centralized, predictable, lock-free ID generation
Observer NotificationService for status updates (extensible) Kitchen display, waitstaff alerts without coupling to service
Fine-grained Locking synchronized(table) on reservation, order, billing Per-table concurrency; no global bottleneck
Value Object MenuItem, Bill Immutable, thread-safe, shareable without defensive copies
Entity Table, Order, Customer, Reservation Mutable with identity (ID-based equality)
