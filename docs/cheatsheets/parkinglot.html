01 — Requirements

Functional

  • Vehicle Entry — validate vehicle type, find suitable space via allocation strategy, generate unique ticket with entry time, update occupancy in real-time
  • Vehicle Exit — calculate fee based on duration + pricing strategy, process payment, free parking space, generate receipt
  • Space Management — support MOTORCYCLE, COMPACT, LARGE, DISABLED space types mapped to vehicle types (Motorcycle → Motorcycle/Compact, Car → Compact/Large, Truck/Bus → Large)
  • Flexible Pricing — hourly with grace period, flat daily rate, peak/off-peak differential, weekend vs weekday, early bird specials
  • Payment Processing — multiple methods (Cash, Credit Card, Debit Card, Mobile), validate before exit, handle failures gracefully
  • Reporting — real-time occupancy per space type, revenue analytics, average duration per vehicle type, peak utilization tracking

Non-Functional

  • Performance — O(1) availability check per vehicle type, O(log n) find available space, < 500ms entry/exit, support 1000+ spaces
  • Scalability — multi-floor support, distributed parking lots (multiple locations), horizontal scaling of payment processing
  • Reliability — atomic ticket generation (no duplicates), consistent occupancy tracking, ACID payment transactions, lost ticket handling
  • Usability — clear error messages for full lot, estimated wait time if full, show nearest available space on entry

03 — Design Patterns

Four patterns, each earning its place — the interview differentiator.

Strategy — Pricing

PricingStrategy interface with pluggable implementations: HourlyPricing, FlatRate, PeakOffPeak, EarlyBird. Injected at construction, swappable at runtime. Different locations need different pricing.

Strategy — Allocation

SpaceAllocationStrategy interface for space selection: NearestFirst (minimize walking), CompactFirst (maximize utilization), CheapestFirst, FloorBalanced. Easy to A/B test.

State Machine

Ticket lifecycle: ACTIVE → PAID / CANCELLED / LOST. Prevents double-payment, handles lost ticket penalties. Clear terminal states.

Repository Pattern

ConcurrentHashMap stores for spaces, tickets, payments. In-memory implementation easily swappable for database-backed persistence.

04 — Ticket State Machine

Why track ticket status? Prevents double-payment, handles lost ticket scenario, enables partial payments (pay at kiosk, exit later). Each state transition is validated.

Ticket Lifecycle in Code

public final class ParkingTicket {
    private final String ticketId;
    private final Vehicle vehicle;
    private final ParkingSpace parkingSpace;
    private final LocalDateTime entryTime;
    private LocalDateTime exitTime;
    private boolean isActive;

    public void markExit(LocalDateTime exitTime) {
        if (!isActive) throw new IllegalStateException("Ticket already inactive");
        if (exitTime.isBefore(entryTime)) throw new IllegalArgumentException(...);
        this.exitTime = exitTime;
        this.isActive = false;
    }

    public Duration calculateDuration() {
        LocalDateTime endTime = exitTime != null ? exitTime : LocalDateTime.now();
        return Duration.between(entryTime, endTime);
    }

    public boolean isValid() { return isActive && exitTime == null; }
}

05 — Strategy Pattern

Why Strategy? Algorithms change independently of the service. New pricing model = new class. New allocation logic = new class. Swappable at runtime without restart. Open/Closed Principle.

Pricing Strategy

interface PricingStrategy {
    Money calculateFee(ParkingTicket ticket);
    String getDescription();
}
ImplementationLogic
HourlyPricingStrategyPer-type rates: Motorcycle $10/hr, Car $20/hr, Truck $40/hr, Bus $50/hr. 15-min grace period. Min charge $5. Rounds up partial hours.
FlatRate$20/day regardless of duration
PeakOffPeakPricing$5/hr peak (17:00-21:00), $2/hr off-peak
EarlyBirdSpecial$15 flat if enter before 9am and exit before 6pm

Hourly Pricing Implementation

public class HourlyPricingStrategy implements PricingStrategy {
    private final Map<VehicleType, Money> hourlyRates;
    private final Money minimumCharge;           // $5.00
    private final Duration gracePeriod;          // 15 minutes

    public Money calculateFee(ParkingTicket ticket) {
        Duration duration = ticket.calculateDuration();

        // Free if within grace period
        if (duration.compareTo(gracePeriod) <= 0)
            return Money.ofMinor(0, currency);

        long minutes = duration.toMinutes();
        long hours = (minutes + 59) / 60; // Round up

        Money totalFee = hourlyRate.times(hours);
        return totalFee.compareTo(minimumCharge) < 0 ? minimumCharge : totalFee;
    }
}

06 — Key Algorithms

1. Find Available Space

Algorithm: FindAvailableSpace(vehicleType, allocationStrategy)

1. compatibleTypes = getCompatibleSpaceTypes(vehicleType)
   // e.g. CAR -> [COMPACT, REGULAR]

2. candidates = allSpaces.stream()
       .filter(space -> !space.isOccupied() && space.canFit(vehicle))
       .collect(Collectors.toList())

3. if candidates is empty:
       throw ParkingFullException(vehicleType)

4. optimalSpace = allocationStrategy.selectSpace(candidates, vehicleType)
   // Nearest: min floor, then spaceId
   // Compact: smallest compatible space
   // Cheapest: lowest hourly rate

5. return optimalSpace

Time: O(k) where k = available spaces | Space: O(1)
Optimization: PriorityQueue per type -> O(log k)

2. Vehicle Entry

Algorithm: EnterVehicle(vehicle)

1. validateVehicle(vehicle)  // null check, license plate check

2. // Prevent duplicate parking
   if activeTickets.anyMatch(t -> t.vehicle.license == vehicle.license):
       throw InvalidVehicleException("Already parked")

3. availableSpaces = allSpaces.filter(!occupied && canFit(vehicle))

4. space = allocationStrategy.selectSpace(availableSpaces, vehicleType)

5. synchronized(space) {       // Prevent double-booking
       space.occupy(vehicle);    // returns false if race condition
   }

6. ticket = new ParkingTicket(generateId(), vehicle, space, now())
   activeTickets.put(ticket.id, ticket)

7. return ticket

Time: O(log k) with PQ, O(k) with set | Space: O(1)

3. Calculate Parking Fee

Algorithm: CalculateParkingFee(ticketId)

1. ticket = activeTickets.get(ticketId)
   if ticket == null: throw InvalidTicketException

2. if !ticket.isValid(): throw IllegalStateException("Already processed")

3. fee = pricingStrategy.calculateFee(ticket)
   // Internally: duration = between(entry, now)
   // If duration <= 15min grace -> $0
   // Else: ceil(hours) * hourlyRate[vehicleType]
   // Apply min($5) floor

4. return fee

Time: O(1) | Space: O(1)

4. Vehicle Exit

Algorithm: ExitVehicle(ticketId, paymentMethod)

1. ticket = activeTickets.get(ticketId)
   validateTicket(ticket)

2. fee = pricingStrategy.calculateFee(ticket)

3. payment = new Payment(generateId(), ticket, fee, paymentMethod)

4. success = paymentProcessor.processPayment(payment)
   if !success: payment.markFailed(); throw PaymentFailedException
   payment.markCompleted("TXN-" + timestamp)

5. ticket.markExit(LocalDateTime.now())  // ACTIVE -> PAID

6. synchronized(space) {
       space.vacate();  // clears vehicle, sets occupied=false
   }

7. activeTickets.remove(ticketId)
   completedTickets.put(ticketId, ticket)

8. return payment

Key: Payment processed BEFORE freeing space (fraud prevention)
Time: O(1) + payment latency | Space: O(1)

5. Occupancy Report

Algorithm: GenerateOccupancyReport()

1. for each spaceType in [MOTORCYCLE, COMPACT, LARGE, DISABLED]:
       available = allSpaces.count(type == spaceType && !occupied)
       occupied  = allSpaces.count(type == spaceType && occupied)
       availableByType.put(spaceType, available)
       occupiedByType.put(spaceType, occupied)

2. overallRate = totalOccupied / totalSpaces

3. return new OccupancyReport(timestamp, total, occupied, maps)

Time: O(n) where n = total spaces | Space: O(1)
Output example:
  COMPACT:  45/50 available  (10% utilization)
  REGULAR:  12/100 available (88% utilization)
  LARGE:    5/20 available   (75% utilization)
  Overall:  28% utilization

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap allSpaces, activeTickets, completedTickets Safe concurrent reads/writes for all data stores
synchronized(space) occupy(), vacate() on ParkingSpace Fine-grained lock per space — prevents double-booking same space
AtomicLong ticketCounter, paymentCounter Lock-free unique ID generation: TICKET-00000001, PAY-00000001
Objects.requireNonNull All constructors Fail-fast validation on all inputs
final fields Vehicle, ParkingTicket, Payment Immutable identity fields, thread-safe by design
"I use fine-grained locking on the ParkingSpace object, not a global lock. Two different spaces can be occupied/vacated concurrently. Only concurrent operations on the same space are serialized."

Double-Booking Prevention

// In ParkingSpace.java -- synchronized on the individual space
public synchronized boolean occupy(Vehicle vehicle) {
    if (isOccupied || !canFit(vehicle.getVehicleType())) {
        return false;  // Race condition guard
    }
    this.isOccupied = true;
    this.currentVehicle = vehicle;
    return true;
}

public synchronized Vehicle vacate() {
    if (!isOccupied) return null;
    Vehicle v = this.currentVehicle;
    this.isOccupied = false;
    this.currentVehicle = null;
    return v;
}

12 — Interview Follow-up Q&A

How would you extend to support multiple floors?
Already supported — each ParkingSpace has a floorNumber field. The NearestSpaceAllocationStrategy sorts by floor number (lower first). To make floors explicit: create a ParkingFloor class with its own space map, and ParkingLot iterates floors ground-up. Optimization: use elevator proximity for allocation priority.
How would you implement real-time display boards?
Three approaches: Push-based (Observer) — DisplayBoard implements an observer interface, ParkingService notifies all boards on entry/exit. Pull-based (Polling) — DisplayBoard uses ScheduledExecutorService to call getOccupancyReport() every 5 seconds. Message Queue (Scalable) — ParkingService publishes events to Kafka topic, display boards consume asynchronously. For LLD interview, Observer is the cleanest answer.
How would you manage parking across multiple lots in a city?
Create a CityParkingRegistry that holds Map<String, ParkingLot> by location ID. The registry exposes findLotsWithAvailability(userLocation, vehicleType, maxDistance) which filters by availability and sorts by distance. Mobile app integration: show lots on map, reserve space in advance (hold for 15 min), navigate with GPS. Each lot still uses its own ParkingService instance.
How would you implement surge/dynamic pricing?
Create a DynamicPricingStrategy that wraps a base strategy. It queries the current OccupancyReport: >90% full = 2x multiplier, >75% = 1.5x, otherwise 1x. This is essentially the Decorator pattern — same approach as ride-hailing surge pricing. The strategy reference can be swapped at runtime since it's injected. A background job could monitor utilization and toggle surge pricing automatically.
How do you prevent two vehicles from getting the same space?
ParkingSpace.occupy() is synchronized. Even if two threads select the same space from the available list, only the first to enter the synchronized block will succeed (returns true). The second thread finds isOccupied=true and returns false, triggering the service to throw ParkingFullException or retry with another space. This is fine-grained locking — two different spaces can be occupied concurrently.
How would you handle disabled/handicapped parking?
Already implemented. SpaceType.DISABLED has capacity 2 (fits car-size vehicles). ParkingSpace.canFit(Vehicle) checks vehicle.hasDisabledPermit() for DISABLED spaces — only vehicles with a permit can park there. The Vehicle constructor accepts an optional hasDisabledPermit boolean. Regulatory compliance: maintain minimum handicapped spaces per local code.
How would you implement a reservation system?
Add a Reservation entity with reservationId, arrivalTime, expiryTime (arrival + 15 min), reservedSpace, and status (ACTIVE/USED/EXPIRED/CANCELLED). ReservationService.reserveSpace() marks the space as reserved (not available for walk-ins). Use ScheduledExecutorService to auto-cancel expired reservations and free the space. Charge a $5 non-refundable reservation fee, deducted from parking fee. Require at least 30 min advance booking.
How would you implement valet parking?
Extend ParkingTicket into ValetTicket with valetId, keyLocation, and retrievalTimeMinutes. ValetService.acceptVehicle() records which valet parked it and where keys are stored. On retrieval: notify the assigned valet, estimate retrieval time based on space location (floor + distance). Valet can park anywhere (bypasses allocation strategy since they walk back). Add a valet surcharge to the pricing.
Why Strategy over simple if/else for pricing?
(1) Open/Closed Principle: new pricing model = new class, zero changes to InMemoryParkingService. (2) Runtime swappable: switch from hourly to flat rate without restart. (3) Testable: each strategy is unit-testable in isolation. (4) Composable: wrap HourlyPricing with DynamicPricing (Decorator). (5) Different lots can use different strategies simultaneously.
What are the tradeoffs of your occupancy tracking design?
Current approach streams allSpaces to count availability: simple but O(n). Alternative: maintain Map<SpaceType, Set<ParkingSpace>> for available and occupied sets separately. Tradeoffs: O(1) availability check and O(1) count, but must keep two sets synchronized on every entry/exit. In production with 1000+ spaces, the set-based approach is strictly better. For the interview, mention both and explain the migration path.
How would you add Electric Vehicle charging support?
Add SpaceType.ELECTRIC_VEHICLE with charging stations. Extend ParkingSpace with chargingStation boolean and chargingRate. Track charging time separately from parking time. Two-part billing: parking fee + charging fee (per kWh). Add a ChargingStatus enum (CHARGING, COMPLETE, NOT_APPLICABLE). Notify vehicle owner when charging completes via push notification.
How would you add license plate recognition for auto entry/exit?
Replace ticket-based flow with camera-based: on entry, OCR captures plate → create ticket automatically (no physical ticket). On exit, OCR reads plate → look up active ticket by license number → calculate fee → charge pre-registered payment method. Fallback: manual entry if OCR confidence is low. The ParkingService interface stays the same — the entry point just changes from physical ticket to automatic recognition.

14 — Patterns Summary

PatternWhereWhy
Strategy PricingStrategy (Hourly, Flat, Peak, EarlyBird) Swap pricing algorithms without modifying service
Strategy SpaceAllocationStrategy (Nearest, Compact, Cheapest) Pluggable space selection logic, A/B testable
State Machine ParkingTicket (ACTIVE → PAID/CANCELLED/LOST), Payment (PENDING → COMPLETED/FAILED/REFUNDED) Validated lifecycle transitions, prevents double-processing
Repository ConcurrentHashMap stores (allSpaces, activeTickets, completedTickets) In-memory, easy to swap for database-backed persistence
Dependency Injection InMemoryParkingService constructor takes all 4 dependencies Testable, swappable implementations, follows SOLID
Value Object Vehicle (immutable), OccupancyReport (immutable snapshot) Thread-safe by design, identity by value
Entity ParkingSpace, ParkingTicket, Payment Mutable with identity (ID-based equality)
