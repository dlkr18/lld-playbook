01 — Requirements

Functional

  • Users register with name & email, build profiles with experience, education, and skills
  • Connection system — send/accept/reject requests, bidirectional linking
  • Follow system — unidirectional follow/unfollow (asymmetric)
  • Posts with types (Article, Status, Job Posting, Poll, Video) and visibility (Public, Connections Only, Private)
  • Engagement: like, unlike, comment, share on posts
  • News feed aggregated from connections + followed users + public posts
  • Job postings — create, search by keyword/location, close
  • Skills & endorsements — add skills, endorse connections' skills
  • Search — find users by name or headline

Non-Functional

  • Performance — Response time < 100ms for critical operations
  • Security — Authentication, authorization, data encryption
  • Scalability — Support 10,000+ concurrent users
  • Reliability — 99.9% uptime
  • Data Consistency — ACID transactions where needed
  • Extensible — Pluggable strategies, interface-based design

03 — Design Patterns

Clean architecture with patterns that earn their place in interview discussion.

Repository Pattern

Abstracts data access from business logic. InMemoryLinkedInService uses HashMap stores — swap for JPA/DB without touching business logic.

Strategy Pattern

Feed generation and search/matching algorithms can be swapped. Stream-based filtering with predicates makes algorithm changes trivial.

Observer Pattern

Post interactions (likes, comments, shares) can trigger notifications. Connection acceptance fires events. Pluggable NotificationService interface ready.

Factory Pattern

UUID-based ID generation for all entities. Object creation centralized in the service layer. Easy to swap for snowflake IDs or database sequences.

04 — Connection System

Bidirectional connections vs unidirectional follows — LinkedIn supports both. Connections are mutual (both parties linked), follows are one-way. 
stateDiagram-v2
    [*] --> PENDING: sendConnectionRequest()
    PENDING --> ACCEPTED: acceptConnection()
    PENDING --> REJECTED: rejectConnection()
    ACCEPTED --> [*]: Bidirectional link created
    REJECTED --> [*]: No connection

ConnectionRequest Model

public class ConnectionRequest {
    private final String requestId;
    private final String senderId;
    private final String receiverId;
    private String message;
    private RequestStatus status;  // PENDING → ACCEPTED / REJECTED

    public void accept() { this.status = RequestStatus.ACCEPTED; }
    public void reject() { this.status = RequestStatus.REJECTED; }
}

Accept Connection — Bidirectional Linking

public void acceptConnection(String requestId) {
    ConnectionRequest request = connectionRequests.get(requestId);
    if (request == null) throw new RequestNotFoundException("Request not found");
    request.accept();
    User sender = getUser(request.getSenderId());
    User receiver = getUser(request.getReceiverId());
    sender.addConnection(receiver.getUserId());   // A → B
    receiver.addConnection(sender.getUserId());   // B → A
}

Both users' connectionIds sets are updated atomically. The Set<String> ensures no duplicates.

Follow System (Unidirectional)

public void follow(String followerId, String followingId) {
    User follower = getUser(followerId);
    User following = getUser(followingId);
    follower.follow(followingId);       // A follows B
    following.addFollower(followerId);   // B gains a follower
}

Key distinction: Connections use a Set<String> connectionIds (shared). Follows use separate followingIds and followerIds sets on each User.

// Service: endorse a connection's skill
public void endorseSkill(String userId, String skillName, String endorserId) {
    User user = getUser(userId);
    getUser(endorserId);  // validate endorser exists
    user.getSkills().stream()
        .filter(s -> s.getName().equals(skillName))
        .findFirst()
        .ifPresent(Skill::endorse);
}

Job Search — Stream-Based Filtering

public List<Job> searchJobs(String query, String location) {
    return jobs.values().stream()
        .filter(j -> j.getStatus() == JobStatus.ACTIVE)
        .filter(j -> query == null ||
            j.getTitle().toLowerCase().contains(query.toLowerCase()))
        .filter(j -> location == null ||
            j.getLocation().toLowerCase().contains(location.toLowerCase()))
        .collect(Collectors.toList());
}

Chained predicates allow nullable parameters — pass null for query or location to skip that filter.

07 — News Feed Algorithm

Feed = connections' posts + followed users' posts + public posts, sorted by recency, limited to N items. This is the fan-out-on-read approach.

Feed Generation

public List<Post> getFeed(String userId, int limit) {
    User user = getUser(userId);
    return posts.values().stream()
        .filter(p ->
            user.getConnectionIds().contains(p.getAuthorId()) ||
            user.getFollowingIds().contains(p.getAuthorId()) ||
            p.getVisibility() == PostVisibility.PUBLIC)
        .sorted((a, b) -> b.getCreatedAt().compareTo(a.getCreatedAt()))
        .limit(limit)
        .collect(Collectors.toList());
}

Post Model — Engagement Tracking

public class Post {
    private Set<String> likeUserIds;   // Set prevents double-likes
    private List<Comment> comments;    // ordered by time
    private int shares;                // share counter
    private PostVisibility visibility; // controls feed inclusion

    public void like(String userId) { likeUserIds.add(userId); }
    public void unlike(String userId) { likeUserIds.remove(userId); }
    public int getLikesCount() { return likeUserIds.size(); }
    public void incrementShares() { shares++; }
}

Using a Set<String> for likes is idempotent — calling like() twice with the same userId has no effect.

Share as Re-post

public Post sharePost(String userId, String postId) {
    getUser(userId);
    Post original = getPost(postId);
    original.incrementShares();  // bump original share count
    return createPost(userId,
        "Shared: " + original.getContent(),
        PostType.STATUS_UPDATE,
        PostVisibility.PUBLIC);   // new post appears in sharer's feed
}

08 — Key Algorithms

User Search — Name + Headline Matching

public List<User> searchUsers(String query) {
    return users.values().stream()
        .filter(u ->
            u.getName().toLowerCase().contains(query.toLowerCase()) ||
            (u.getHeadline() != null &&
             u.getHeadline().toLowerCase().contains(query.toLowerCase())))
        .collect(Collectors.toList());
}

Time: O(n) scan. At scale, replace with Elasticsearch or Lucene behind the same interface.

Pending Requests — Filtered Stream

public List<ConnectionRequest> getPendingRequests(String userId) {
    return connectionRequests.values().stream()
        .filter(r -> r.getReceiverId().equals(userId))
        .filter(r -> r.getStatus() == RequestStatus.PENDING)
        .collect(Collectors.toList());
}

User Posts — Sorted by Recency

public List<Post> getUserPosts(String userId) {
    return posts.values().stream()
        .filter(p -> p.getAuthorId().equals(userId))
        .sorted((a, b) -> b.getCreatedAt().compareTo(a.getCreatedAt()))
        .collect(Collectors.toList());
}

Time: O(n log n) sort. Use a TreeMap or index by authorId for O(log n) lookups at scale.

09 — Thread Safety

MechanismWhereWhy
HashMap All in-memory stores Simple implementation — swap to ConcurrentHashMap for thread safety
HashSet connectionIds, followerIds, followingIds, likeUserIds O(1) membership checks, prevents duplicates
UUID.randomUUID() ID generation for all entities Thread-safe, collision-resistant unique IDs
Defensive copies All getter methods returning collections new ArrayList<>(list) / new HashSet<>(set) prevents external mutation
"In production, swap HashMap to ConcurrentHashMap, add synchronized blocks on connection acceptance (two-user update), and use CopyOnWriteArrayList for observer lists."

Defensive Copies Pattern

// Every collection getter returns a copy, not the internal reference
public List<Skill> getSkills() {
    return new ArrayList<>(skills);
}
public Set<String> getConnectionIds() {
    return new HashSet<>(connectionIds);
}
public List<String> getRequiredSkills() {
    return new ArrayList<>(requiredSkills);
}

Prevents callers from modifying internal state. Essential for encapsulation and thread safety.

Engage with Post

job.setLocation("Remote") — job.addRequiredSkill("Java")

13 — Interview Follow-up Q&A

How would you scale the news feed?
Two approaches: Fan-out-on-write (precompute feeds on post creation — fast reads, expensive writes) or Fan-out-on-read (current approach — cheap writes, expensive reads). Hybrid: fan-out-on-write for users with few followers, fan-out-on-read for celebrities. Add Redis sorted sets for precomputed feeds.
How would you handle millions of connection requests?
Shard the connectionRequests map by receiverId. Use a database with an index on (receiverId, status) for fast pending-request lookups. Add pagination to getPendingRequests(). Rate-limit outgoing requests per user.
Why use Set for connectionIds instead of List?
O(1) membership check for feed filtering (connectionIds.contains(authorId)). Prevents duplicate connections without additional checks. HashSet also ensures addConnection() is idempotent.
How would you implement "People You May Know"?
Graph-based: find 2nd-degree connections (friends of friends) not already connected. Score by mutual connection count. At scale, use a graph database (Neo4j) or precompute with batch jobs. The connectionIds set makes mutual-connection counting straightforward.
How do you prevent duplicate likes?
Set<String> likeUserIds — calling likeUserIds.add(userId) is idempotent. The Set contract ensures no duplicates. getLikesCount() returns likeUserIds.size(), always accurate.
Why separate Connection and Follow?
Connections are mutual (bidirectional) — both parties agree. Follows are unidirectional — no approval needed. LinkedIn uses both: you can follow someone without being connected. Different data structures: connections use a shared connectionIds set; follows use separate followerIds and followingIds.
How would you handle post visibility at scale?
Current approach checks visibility in the feed query. At scale: precompute per-user feed with visibility filtering. CONNECTIONS_ONLY posts indexed separately. Use database views or materialized feeds. Visibility changes trigger feed recomputation for affected users.
How would you implement job recommendations?
Match user's skills with job's requiredSkills. Score by skill overlap count. Factor in location preference and job type. At scale: collaborative filtering based on what similar users applied to. The current Skill model and Job.requiredSkills list provide the foundation.
Why defensive copies on all collection getters?
Returning new ArrayList<>(list) prevents external code from modifying internal state. Without it, user.getSkills().clear() would destroy the user's skills. Essential for encapsulation and makes the code safe for concurrent access.
How would you add messaging?
New entities: Conversation (participants, messages) and Message (senderId, content, timestamp, readStatus). Add to service interface: sendMessage(), getConversations(), getMessages(). At scale: WebSockets for real-time delivery, message queue for persistence.
What's the difference between HashMap and ConcurrentHashMap here?
Current implementation uses HashMap (not thread-safe) — fine for single-threaded demo. In production, swap to ConcurrentHashMap for lock-free reads and segment-level locking on writes. The interface (Map) stays the same — one-line change.
How would you handle feed ranking beyond recency?
Extract feed ranking into a Strategy pattern: FeedRankingStrategy interface with implementations like RecencyStrategy (current), EngagementStrategy (likes + comments weight), RelevanceStrategy (skill/industry match). Swappable at runtime like pricing in ride-hailing.
