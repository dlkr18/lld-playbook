01 — The 3-Question Drill

When asked to make anything thread-safe, say these three questions out loud and answer them in order — it shows method, and it leads you straight to the right tool:

  1. What is the shared mutable state? (the field(s) two threads can touch at once — a counter, a map, a balance, a queue)
  2. What is the critical section? (the smallest span that must be indivisible — usually a check-then-act or read-modify-write)
  3. What guards it, and at what granularity? (a lock / atomic / concurrent collection — and per-entity, not one global lock)

The bug you're almost always fixing: check-then-act (TOCTOU)

if (qty <= available) available -= qty;   // two steps — a thread can slip between them
// A reads available=1, B reads available=1, both pass, both decrement -> -1  (OVERSOLD)

Fix = make check + act one indivisible operation (lock the pair, or CAS in a loop, or push the guard into the DB).

03 — Locks & Atomics

a) synchronized — the default. Guards a check-then-act.

Per-instance lock (this). Different objects don't contend — already fine-grained.

public synchronized void deduct(int qty) {
    if (qty > available) throw new InsufficientStockException();  // check
    available -= qty;                                             // + act = atomic
}
public synchronized void restock(int qty) { available += qty; }
public synchronized int  available()      { return available; }  // read under same lock

b) ReentrantLock — when you need tryLock / timeout / multiple locks

private final ReentrantLock lock = new ReentrantLock();
lock.lock();
try { /* critical section */ }
finally { lock.unlock(); }        // ALWAYS unlock in finally

if (lock.tryLock(100, MILLISECONDS)) { … }   // bounded wait, avoid deadlock

c) CAS — lock-free single-slot flip (seat/hold, counter)

"Set to NEW only if still EXPECTED." Loop to retry on contention. No lock.

// seat hold: AVAILABLE -> HELD, exactly one winner
private final AtomicReference<Status> status = new AtomicReference<>(AVAILABLE);
boolean hold() { return status.compareAndSet(AVAILABLE, HELD); }  // false if someone won first

// counter with a bounded guard needs the retry loop:
int cur;
do { cur = avail.get(); if (qty > cur) throw new InsufficientStockException(); }
while (!avail.compareAndSet(cur, cur - qty));   // retry if another thread changed it

d) ConcurrentHashMap.computeIfAbsent — atomic get-or-create

No double-init race. The classic way to mint a per-key lock or per-entity object.

private final Map<String, ReentrantLock> locks = new ConcurrentHashMap<>();
Lock forKey(String id) { return locks.computeIfAbsent(id, k -> new ReentrantLock()); }

// idempotency: run at most once per key
Txn t = store.computeIfAbsent(idemKey, k -> process(request));

e) ReadWriteLock — many readers, few writers

private final ReadWriteLock rw = new ReentrantReadWriteLock();
V get(K k){ rw.readLock().lock();  try{ return map.get(k);} finally{ rw.readLock().unlock(); } }
void put(K k,V v){ rw.writeLock().lock(); try{ map.put(k,v);} finally{ rw.writeLock().unlock(); } }
// reads run concurrently; a write is exclusive. Great for caches / filesystem trees.

04 — Async & Scheduling

a) ExecutorService + Future — run in background, await result

ExecutorService pool = Executors.newFixedThreadPool(4);
Future<Integer> f = pool.submit(() -> expensive());   // returns immediately
int result = f.get();                                // blocks until done; throws ExecutionException on failure
pool.shutdown();                                      // stop accepting; finishes queued work

b) BlockingQueue worker loop — hand-built thread pool (producer/consumer)

private final BlockingQueue<Runnable> q = new LinkedBlockingQueue<>();
private static final Runnable POISON = () -> {};

// each worker thread:
while (true) {
    Runnable r = q.take();          // blocks until work arrives
    if (r == POISON) return;       // graceful stop AFTER draining
    try { r.run(); } catch (Throwable t) { /* log, keep the worker alive */ }
}
// shutdown(): for each worker, q.add(POISON); then join() them.

c) DelayQueue scheduler — run "at a time / interval", no polling

class Task implements Delayed {
    final long runAtMillis;
    public long getDelay(TimeUnit u){ return u.convert(runAtMillis - now(), MILLISECONDS); }
    public int  compareTo(Delayed o){ return Long.compare(runAtMillis, ((Task)o).runAtMillis); }
}
// scheduler thread — take() returns ONLY when the earliest task is due:
Task t = queue.take();          // blocks until due (O(log n), zero busy-wait)
pool.execute(t::run);           // DISPATCH to workers — scheduler never runs the body
if (t.periodic) { t.runAtMillis += t.period; queue.put(t); }   // fixed-rate reschedule

d) CountDownLatch — a one-shot future / wait-for-N

// build a Future by hand: worker counts down, get() awaits.
private final CountDownLatch done = new CountDownLatch(1);
private volatile T result;
void complete(T v){ result = v; done.countDown(); }
T get() throws InterruptedException { done.await(); return result; }  // happens-before via latch

// or: wait for N parallel tasks to finish
CountDownLatch all = new CountDownLatch(n);
// each task: ... all.countDown();   main: all.await();

05 — Deadlock-free Transfer (lock two entities)

The deadlock: Thread 1 does A→B (locks A then B); Thread 2 does B→A (locks B then A). Each holds one and waits for the other — circular wait, frozen forever. Fix: always acquire in a consistent global order (e.g. by id), so a cycle can never form.

void transfer(Account from, Account to, Money amt) {
    Account first  = from.id().compareTo(to.id()) < 0 ? from : to;   // lower id first
    Account second = first == from ? to : from;
    first.lock().lock();
    try {
        second.lock().lock();
        try {
            if (from.balance() < amt) throw new InsufficientFundsException();  // check BEFORE mutate
            from.debit(amt);
            to.credit(amt);                     // debit + credit are indivisible here
        } finally { second.lock().unlock(); }
    } finally { first.lock().unlock(); }
}
// ReentrantLock is re-entrant, so debit/credit re-locking the same account is safe.

06 — No-Oversell: the 3-tier answer

ContextAnswer
In one JVM, simplesynchronized deduct — one lock per item (different products never contend)
In one JVM, hot counterAtomicInteger + CAS retry loop (lock-free)
Multiple servers / real DBDB does it — see below (app locks only guard one JVM!)
-- the guard is IN the WHERE clause; the DB row-locks atomically
UPDATE inventory SET available = available - :qty
 WHERE product_id = :id AND available >= :qty;
-- rows affected == 1 -> success ; == 0 -> insufficient stock, reject. No app lock, scales out.

07 — Muscle-Memory Snippets (non-concurrency)

The other must-type-from-memory shapes. (The concurrency ones above complete the set: BlockingQueue worker, DelayQueue scheduler, CountDownLatch future, ordered-lock transfer.)

1) Enum with behavior + transition table (state machines)

enum Status {
    PENDING, CONFIRMED, CANCELLED;
    boolean canGoTo(Status next) {
        if (this == PENDING) return next == CONFIRMED || next == CANCELLED;
        return false;                    // CONFIRMED / CANCELLED are terminal
    }
}

2) Strategy — interface field, injected

interface PricingStrategy { Money price(Ticket t); }
class ParkingService {
    private final PricingStrategy pricing;      // depend on the interface
    ParkingService(PricingStrategy pricing){ this.pricing = pricing; }  // swap = new impl
}

3) Observer — fire to all listeners

private final List<Listener> listeners = new CopyOnWriteArrayList<>();  // safe iterate
void addListener(Listener l){ listeners.add(l); }
void fire(Event e){ for (Listener l : listeners) l.on(e); }

4) Composite — leaf returns base, node recurses

interface Node { long size(); }
class File implements Node { public long size(){ return bytes; } }           // leaf
class Dir implements Node {
    private final List<Node> children = new ArrayList<>();
    public long size(){ long s=0; for(Node c:children) s += c.size(); return s; }  // recurse
}

5) LRU — HashMap + doubly-linked list, O(1) get/put

Map<K,Node> map;  Node head, tail;         // sentinels; head.next = most-recent
V get(K k){ Node n=map.get(k); if(n==null) return null; moveToHead(n); return n.v; }
void put(K k,V v){
    // upsert + moveToHead; if(map.size()>cap){ Node lru=tail.prev; remove(lru); map.remove(lru.k);} }
}
// moveToHead = removeNode(n) + addToHead(n) — pure pointer splices, no shifting.
// shortcut: new LinkedHashMap<>(cap,0.75f,true){ removeEldestEntry(e){return size()>cap;} }

6) TreeMap interval overlap — O(log n) conflict check (meeting scheduler)

NavigableMap<Integer,Integer> cal = new TreeMap<>();   // start -> end, half-open [s,e)
boolean free(int s, int e) {
    Map.Entry<Integer,Integer> f = cal.floorEntry(s);     // booking starting at/before s
    if (f != null && f.getValue() > s) return false;    // it ends after s -> overlap
    Map.Entry<Integer,Integer> c = cal.ceilingEntry(s);   // next booking
    if (c != null && c.getKey() < e) return false;     // it starts before e -> overlap
    return true;
}

08 — Gotchas (say these, don't get caught)

  • volatile vs synchronized: volatile gives visibility only (other threads see the latest write) — NOT atomicity. count++ on a volatile is still a race; you need synchronized/Atomic for read-modify-write.
  • Always unlock() in finally — an exception mid-critical-section otherwise leaks the lock forever.
  • Lock granularity: a per-entity lock (ConcurrentHashMap<Id,Lock>) lets different entities proceed in parallel; one global lock serializes everything. Prefer per-entity.
  • Don't interrupt running work to "cancel" unless the task is interrupt-aware — you can corrupt shared/IO state. Cancel only queued tasks (flip a flag checked before start).
  • App locks guard ONE JVM. Across servers, `synchronized`/Atomic do nothing — push the invariant to the DB (conditional update) or a distributed lock (Redis/ZooKeeper).
  • Prefer ConcurrentHashMap over synchronized map — lock-striped, far less contention; use computeIfAbsent for atomic get-or-create.
  • Immutable objects are automatically thread-safe — no shared mutable state, no lock needed. Make value objects (Money, Query) immutable and the whole concurrency story shrinks.
  • Thread-pool size: CPU-bound ≈ #cores; IO-bound can be much larger (threads mostly block).
