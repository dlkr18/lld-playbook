01 — Requirements

Functional

  • Return top-k suggestions (typically 5–10) for a given prefix
  • Case-insensitive matching with minimum prefix length (e.g., 2 chars)
  • Rank by popularity (query frequency), boost recent searches
  • Personalized suggestions from user history
  • Trending queries via time-weighted frequency
  • Add new search terms, update frequency, remove outdated/offensive terms
  • Track search analytics — frequency, trends, geographic distribution

Non-Functional

  • Latency — suggestion retrieval < 10ms (p99), typing is ~100ms/char
  • Throughput — 100,000+ QPS
  • Storage — 10M+ terms via memory-optimized Trie, disk backup
  • Cache hit rate > 80% with multi-level caching (L1 + L2)
  • Availability — 99.9% uptime, graceful degradation
  • Read-heavy — 99% reads, 1% writes
  • Scalable — horizontal scaling with sharding by prefix, CDN for geo-distribution

03 — Design Patterns

Six patterns, each solving a distinct problem in the autocomplete domain.

Trie Pattern

Prefix tree for O(m) prefix search where m = prefix length. Shared prefixes compress memory. Core data structure of the entire system.

Strategy Pattern

SuggestionRanker encapsulates ranking algorithms. Pluggable scoring: popularity, recency, personalization. Swap without changing service.

Cache-Aside Pattern

SuggestionCache with L1 in-memory + L2 Redis. Lazy load on miss, TTL-based invalidation. Hot prefixes pre-warmed at startup.

Observer Pattern

Notify on trending query changes. Real-time updates when popularity shifts trigger cache invalidation and re-ranking.

Singleton Pattern

Single Trie instance shared across the service. Memory efficiency — one prefix tree serves all requests.

Builder Pattern

Build Suggestion objects with score, metadata, timestamp. Clean object creation with optional fields.

04 — Trie Data Structure

TrieNode (model)

public class TrieNode {
    private final Map<Character, TrieNode> children = new HashMap<>();
    private boolean isEndOfWord;
    private int frequency;
    private String word;

    public TrieNode getChild(char c) { return children.get(c); }
    public void addChild(char c, TrieNode node) { children.put(c, node); }
    public boolean hasChild(char c) { return children.containsKey(c); }
    public void incrementFrequency() { this.frequency++; }
}

Trie Insertion

public void addWord(String word, int frequency) {
    TrieNode current = root;
    word = word.toLowerCase();

    for (char ch : word.toCharArray()) {
        current.getChildren().putIfAbsent(ch, new TrieNode());
        current = current.getChildren().get(ch);
    }

    current.setEndOfWord(true);
    current.setWord(word);
    for (int i = 0; i < frequency; i++)
        current.incrementFrequency();
}

Prefix Search + Collection

public List<Suggestion> getSuggestions(String prefix, int limit) {
    prefix = prefix.toLowerCase();
    TrieNode current = root;

    // Navigate to prefix node — O(m)
    for (char ch : prefix.toCharArray()) {
        TrieNode next = current.getChildren().get(ch);
        if (next == null) return Collections.emptyList();
        current = next;
    }

    // DFS to collect all words under prefix
    List<Suggestion> suggestions = new ArrayList<>();
    collectSuggestions(current, suggestions);

    // Sort by score (frequency) and limit
    Collections.sort(suggestions);
    return suggestions.subList(0, Math.min(limit, suggestions.size()));
}

private void collectSuggestions(TrieNode node, List<Suggestion> suggestions) {
    if (node.isEndOfWord())
        suggestions.add(new Suggestion(node.getWord(), node.getFrequency()));
    for (TrieNode child : node.getChildren().values())
        collectSuggestions(child, suggestions);
}

Recursive Removal

private boolean remove(TrieNode current, String word, int index) {
    if (index == word.length()) {
        if (!current.isEndOfWord()) return false;
        current.setEndOfWord(false);
        return current.getChildren().isEmpty();
    }
    char ch = word.charAt(index);
    TrieNode node = current.getChildren().get(ch);
    if (node == null) return false;

    boolean shouldDelete = remove(node, word, index + 1)
                           && !node.isEndOfWord();
    if (shouldDelete) {
        current.getChildren().remove(ch);
        return current.getChildren().isEmpty();
    }
    return false;
}

05 — Ranking & Scoring

Three signals combined into a weighted score: Popularity (50%), Recency (30%), Personalization (20%). The weights are tunable per deployment.

Hybrid Scoring Algorithm

public double calculateScore(Suggestion suggestion, User user) {
    // Popularity: log-normalized frequency
    double popularityScore = Math.log(1 + suggestion.getFrequency()) / 10.0;

    // Recency: exponential decay over 7 days
    long ageInHours = (System.currentTimeMillis() - suggestion.getTimestamp())
                      / (1000 * 60 * 60);
    double recencyScore = Math.exp(-ageInHours / 168.0);

    // Personalization: boost if user searched before
    double personalScore = user.hasSearched(suggestion.getText())
                           ? 1.5 : 1.0;

    return (0.5 * popularityScore + 0.3 * recencyScore
            + 0.2 * personalScore);
}

Relevance-Based Ranker

public static List<Suggestion> rankByRelevance(
        List<Suggestion> suggestions, String query) {
    suggestions.sort((s1, s2) -> {
        boolean s1Starts = s1.getWord().startsWith(query);
        boolean s2Starts = s2.getWord().startsWith(query);
        if (s1Starts && !s2Starts) return -1;
        if (!s1Starts && s2Starts) return 1;
        return Integer.compare(s2.getFrequency(), s1.getFrequency());
    });
    return suggestions;
}

Priority: Exact prefix match first, then by frequency descending. Ties broken alphabetically.

Suggestion (Comparable)

public class Suggestion implements Comparable<Suggestion> {
    private final String word;
    private final int frequency;
    private final double score;

    public Suggestion(String word, int frequency) {
        this.word = word;
        this.frequency = frequency;
        this.score = frequency * 1.0;
    }

    public int compareTo(Suggestion other) {
        return Double.compare(other.score, this.score);
    }
}

06 — Key Algorithms

Algorithm 1: Frequency-Sorted Prefix Search (AutocompleteSystem)

public List<String> getSuggestions(String prefix, int k) {
    prefix = prefix.toLowerCase();
    TrieNode node = searchPrefix(prefix);
    if (node == null) return new ArrayList<>();

    List<String> allWords = new ArrayList<>();
    collectWords(node, prefix, allWords);

    // Sort by frequency DESC, then alphabetically
    allWords.sort((a, b) -> {
        int freqA = queryFrequency.getOrDefault(a, 0);
        int freqB = queryFrequency.getOrDefault(b, 0);
        if (freqA != freqB) return freqB - freqA;
        return a.compareTo(b);
    });
    return allWords.subList(0, Math.min(k, allWords.size()));
}

Algorithm 2: DFS Word Collection

private void collectWords(TrieNode node, String prefix,
                           List<String> result) {
    if (node.isEndOfWord()) result.add(prefix);

    for (Map.Entry<Character, TrieNode> entry :
             node.getChildren().entrySet()) {
        collectWords(entry.getValue(),
                     prefix + entry.getKey(), result);
    }
}

DFS traverses all children under the prefix node. For production top-k, use a min-heap of size k instead of collecting all + sorting.

SuggestionCache Implementation

public class SuggestionCache {
    private final Map<String, List<Suggestion>> cache
        = new ConcurrentHashMap<>();
    private final int maxSize;

    public void put(String prefix, List<Suggestion> suggestions) {
        if (cache.size() >= maxSize) {
            String firstKey = cache.keySet().iterator().next();
            cache.remove(firstKey); // evict oldest entry
        }
        cache.put(prefix, suggestions);
    }

    public List<Suggestion> get(String prefix) {
        return cache.get(prefix);
    }

    public void clear() { cache.clear(); }
}

Cache Flow

Request: "goo" →
1. Check L1 (in-memory) — hit? return
2. Check L2 (Redis)     — hit? populate L1, return
3. Query Trie            — populate L1 + L2, return

Invalidation: on term update
→ invalidate "goo", "go", "g" (prefix + parents)
→ shorter TTL for trending queries
→ negative caching for "no results" prefixes

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap SuggestionCache, queryFrequency map Safe concurrent reads/writes for cache and frequency tracking
putIfAbsent() Trie insertion (TrieNode.children) Atomic child creation prevents duplicate nodes during concurrent inserts
synchronized Trie write operations Serialize structural modifications to the Trie (insert, remove)
volatile Cache entries, Trie root Cross-thread visibility for cache updates and Trie rebuilds
Read-Write Lock Trie reads vs writes (production) Multiple concurrent reads, exclusive writes — 99% read-heavy
"Read-heavy workload (99% reads, 1% writes). In production, use a ReadWriteLock on the Trie: unlimited concurrent readers, exclusive writer. Writes go through an async queue to minimize contention."

12 — Interview Follow-up Q&A

Why Trie instead of a database LIKE query?
Trie is O(m) where m = prefix length vs database LIKE is O(n log n). In-memory Trie is 1000x faster than disk-based DB. For autocomplete's sub-10ms requirement, in-memory data structures are essential.
How to handle typos (e.g., "gogle" → "google")?
Use edit distance (Levenshtein) with a threshold of 1-2. Implement fuzzy matching in the Trie by allowing character substitutions during traversal. Alternatively, integrate with Elasticsearch which has built-in fuzzy matching and phonetic analysis.
How to scale to 1 million QPS?
Shard the Trie by first letter (a-z = 26 shards). Add read replicas for each shard. Use CDN for geographic distribution. Multi-level caching (L1 in-memory, L2 Redis cluster). Client-side debouncing reduces actual QPS by 3-5x.
How to handle trending queries in real-time?
Time-windowed frequency count with sliding windows (1min, 5min, 1hr). Use exponential decay so recent queries are weighted higher. Maintain a separate trending Trie that's rebuilt every N minutes from the analytics pipeline (Kafka → Spark).
What if the Trie doesn't fit in memory?
Options: (1) Trie compaction — compress single-child chains (Patricia Trie). (2) LOUDS encoding — bit-level compression. (3) Disk-backed Trie using B-tree or memory-mapped files. (4) Shard across machines so each holds a subset. (5) Switch to Elasticsearch for very large datasets.
Why ConcurrentHashMap in the cache instead of synchronized HashMap?
ConcurrentHashMap uses segment-level locking (Java 8+: CAS + synchronized on individual bins). Multiple threads can read/write different keys concurrently. synchronized HashMap locks the entire map for every operation — catastrophic for a 100K+ QPS autocomplete service.
How would you implement personalized suggestions?
Maintain a per-user Trie or a user history map. On query, merge results from the global Trie and user's personal Trie. Apply a personalization boost (e.g., 1.5x) to terms the user has previously searched. Store user history in a separate DB with TTL-based expiry.
How does the client optimize for perceived latency?
Debouncing (300ms delay before sending request). Local cache on the client for recent prefix results. Optimistic UI — show results from previous prefix while loading. Progressive refinement — results for "go" are a superset of "goo", so filter locally.
How would you remove offensive/banned terms?
Maintain a blocklist (HashSet) checked during both insertion and retrieval. The removeWord() method already supports recursive Trie node cleanup. For real-time moderation, use a content filter service as a pre-processing step before the Trie.
Why use DFS instead of BFS for collecting suggestions?
DFS is simpler and uses O(depth) stack space vs BFS O(branching-factor) queue space. For Tries, depth is bounded by max word length (~20-30 chars). In practice, both work. For top-k optimization, use a min-heap of size k during DFS to avoid collecting all results.
How would you handle multi-language support?
Use Unicode characters as Trie keys (the HashMap<Character, TrieNode> already supports this). For CJK languages, tokenize into n-grams since there are no word boundaries. Maintain separate Tries per language or a unified Trie with language tags on suggestions. Use ICU4J for proper Unicode normalization.
What's the cache invalidation strategy?
On term update: invalidate the specific prefix AND all parent prefixes (e.g., updating "google" invalidates "g", "go", "goo", "goog", "googl", "google"). For trending changes, use shorter TTLs. Use negative caching to prevent repeated lookups for non-existent prefixes. Cache warming at startup for the top 1000 hot prefixes.
