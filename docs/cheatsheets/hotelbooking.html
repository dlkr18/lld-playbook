Hotel Booking System // LLD

Reserve a room over a continuous [checkIn, checkOut) interval. Interval-overlap availability, Strategy pricing (flat / seasonal / occupancy), State-machine reservation lifecycle, Observer notifications, and per-room locking for atomic check-then-reserve.

01 — Requirements

Functional

  • Register hotels & rooms — each hotel holds rooms grouped by RoomType (SINGLE, DOUBLE, DELUXE, SUITE); each type carries capacity + base nightly rate
  • Search availability — find rooms of a type with minimum capacity that have NO blocking reservation overlapping [checkIn, checkOut)
  • Reserve — a specific room (reserve) or any room of a type (reserveAny) for a date range; reject if it overlaps an existing reservation for that room
  • Lifecycle — check in, check out, cancel, and record no-show; illegal transitions are rejected
  • Pricing — compute stay total via a pluggable strategy (flat / seasonal per-night / occupancy-driven surge)
  • Notifications — emit lifecycle events (confirmation, check-in, check-out, cancellation, no-show) to subscribers

Non-Functional

  • Concurrency safety — a room is never double-booked over overlapping dates; a 32-thread race proves exactly one winner
  • Throughput — bookings for different rooms proceed fully in parallel (per-room lock, never a global lock)
  • Extensibility — add pricing policies / notification channels without editing core booking logic (OCP)
  • Money correctness — BigDecimal with HALF_UP scale-2 rounding
  • No lock held over I/O — notifications fire after the room lock is released so a slow channel can't stall bookings

03 — Design Patterns

Five patterns, each earning its place — the interview differentiator.

Strategy — Pricing

PricingStrategy with pluggable impls: Flat, Seasonal (per-night), Occupancy (surge). Runtime-swappable via setPricingStrategy(). Alternative rejected: if(season)… in the service violates OCP and is untestable in isolation.

State — Lifecycle

One class per ReservationStatus; each knows its legal transitions; illegal ones throw uniformly. Alternative rejected: switch(status) in Reservation — every new status touches every method.

Observer — Notifications

BookingObserver subscribers get lifecycle callbacks. Add SMS/audit by subscribing. Alternative rejected: mailer inside reserve() couples core logic to I/O and holds locks longer.

Value Object — DateRange

Overlap / nights arithmetic owned by an immutable, equatable DateRange. Alternative rejected: raw LocalDate pairs everywhere — duplicated, error-prone boundary logic.

Facade — HotelBooking

One coherent API over catalog + reservations + concurrency. Alternative rejected: exposing repositories/locks to callers leaks the concurrency contract.

04 — Reservation State Machine

Why a state machine? Transition rules per status live in exactly one class each. Illegal actions (e.g. check-out before check-in, cancel after check-in) throw IllegalStateException uniformly — no scattered if checks. States are stateless singletons shared across all reservations.

blocksAvailability() — what frees a room

A reservation blocks a room's dates only while RESERVED or CHECKED_IN. CANCELLED, NO_SHOW, and CHECKED_OUT flip blocksAvailability() to false — the room's dates are freed without deleting history.

State Pattern in Code

public final class ReservedState implements ReservationState {
    public static final ReservedState INSTANCE = new ReservedState();

    public ReservationStatus status() { return ReservationStatus.RESERVED; }

    public void checkIn(Reservation r)  { r.setState(CheckedInState.INSTANCE); }
    public void cancel(Reservation r)   { r.setState(CancelledState.INSTANCE); }
    public void markNoShow(Reservation r) { r.setState(NoShowState.INSTANCE); }

    public void checkOut(Reservation r) {
        throw new IllegalStateException("cannot check out before checking in");
    }
}

05 — Strategy Pattern (Pricing)

Why Strategy? Flat / seasonal / occupancy differ only in the total calculation. Runtime-swappable, independently unit-testable, and open for extension without modifying the orchestrator (OCP).

Pricing Strategy Interface

interface PricingStrategy {
    // occupancyRatio in [0,1]: fraction of same-type rooms occupied for the
    // requested dates. Flat/Seasonal ignore it; Occupancy surges on it.
    BigDecimal totalFor(Room room, DateRange stay, double occupancyRatio);
}
ImplementationFormulaNotes
FlatPricingStrategybaseRate × nightsBaseline; obvious expected total for tests
SeasonalPricingStrategyΣ (baseRate × monthMult) per nightPer-night so a stay can straddle a season boundary
OccupancyPricingStrategybaseRate × nights × (1 + maxSurge·occ)Demand surge; maxSurge default 0.8 (100% full → 1.8×)

Seasonal Pricing — Per-Night Across a Boundary

public BigDecimal totalFor(Room room, DateRange stay, double occ) {
    BigDecimal base = room.type().baseRate();
    BigDecimal total = BigDecimal.ZERO;
    LocalDate day = stay.checkIn();
    while (day.isBefore(stay.checkOut())) {          // half-open: exclusive checkout
        total = total.add(base.multiply(multiplierFor(day.getMonth())));
        day = day.plusDays(1);
    }
    return total.setScale(2, RoundingMode.HALF_UP);
}
// Peak (Jun-Aug, Dec) x1.5  |  Shoulder (Mar-May, Sep) x1.2  |  else x1.0
// SUITE base 350, Nov 29 -> Dec 3: 350+350+350*1.5+350*1.5 = 1750.00

06 — Key Algorithm: Date-Range Overlap

The signature detail. Stays are half-open [checkIn, checkOut). Because the check-out boundary is exclusive, adjacency (b == c) is NOT overlap — so a guest departing on the 13th and another arriving on the 13th can share the same room. This is the "back-to-back" test.

The Overlap Test

// Two half-open intervals [a,b) and [c,d) overlap iff:
//     a < d  AND  c < b

public boolean overlaps(DateRange other) {
    return this.checkIn.isBefore(other.checkOut)      // a < d
        && other.checkIn.isBefore(this.checkOut);   // c < b
}

Availability = Derived, Not a Flag

Algorithm: hasOverlap(roomId, stay)   // caller holds the room lock

1. for each existing in reservationsByRoom[roomId]:
2.     if existing.blocksAvailability()          // RESERVED or CHECKED_IN only
           && existing.stay().overlaps(stay):
3.         return true
4. return false

A room has NO boolean "available" field. Availability for a range is the
RESULT of scanning every blocking reservation for that room. Cancel / no-show /
check-out set blocksAvailability()=false, freeing the dates without losing history.

Time: O(reservations/room) linear scan | swap for interval tree at scale

07 — Thread Safety

MechanismWhereWhy
ReentrantLock per room roomLocks: roomId → lock Overlap-check + insert atomic per room; different rooms never contend
ConcurrentHashMap hotels, roomsById, reservationsById, roomLocks Safe concurrent reads/writes for all catalog + index maps
CopyOnWriteArrayList reservationsByRoom values, observers Lock-free occupancy sampling for pricing; safe iteration during notify
AtomicLong reservationSeq Lock-free unique reservation ids: R1, R2, …
volatile pricingStrategy Runtime strategy swap visible across threads without a lock
"I lock per room, not globally. Two threads targeting the same room serialize on that room's lock, so they can't both pass the 'is it free?' check — exactly one wins, the loser sees the just-inserted reservation and is rejected. Two threads targeting different rooms never contend. A 32-thread race asserts exactly 1 winner, 31 rejections."

Atomic Check-Then-Reserve

private Reservation attemptReserve(Room room, Guest guest, DateRange stay) {
    // Occupancy is advisory pricing input -- sample it LOCK-FREE, BEFORE taking
    // the room lock, to avoid any lock-ordering hazard (never hold A then grab B).
    double occ = occupancyRatioFor(room.hotelId(), room.type(), stay);
    BigDecimal total = pricingStrategy.totalFor(room, stay, occ);

    ReentrantLock lock = roomLocks.get(room.id());
    lock.lock();
    try {
        if (hasOverlap(room.id(), stay)) return null;   // caller throws / tries next
        Reservation r = new Reservation("R" + seq.incrementAndGet(),
                                        room, guest, stay, total);
        reservationsByRoom.get(room.id()).add(r);
        reservationsById.put(r.id(), r);
        return r;
    } finally {
        lock.unlock();
    }
    // Notifications fire in reserve(), AFTER the lock is released.
}

Why per-room? (vs alternatives)

  • Not global — one lock would serialize the entire hotel, killing throughput. A conflict is always within one room, so per-room is the natural granularity.
  • Not per-date — finer, but a stay spans many dates → multi-lock acquisition with ordering rules and deadlock risk; not worth it here.
  • Lifecycle ops take the same room lock so a status change (which flips blocksAvailability()) is serialized with availability checks.

11 — How This Differs from Movie/Seat Booking

The distinguishing complexity is not "book a thing" — it's what you book and how availability and concurrency are defined.
DimensionHotel BookingMovie / Seat (BookMyShow)
Booking unit A room over a continuous [checkIn, checkOut) interval A discrete seat on a single fixed show instant
Availability Derived — result of an interval-overlap scan against every reservation for the room A boolean per (show, seat)
Core arithmetic Date-range overlap with an exclusive check-out boundary (back-to-back stays) Set-membership test on a seat map
Lifecycle Real occupancy lifecycle: check-in / check-out Ticket has no check-in/out; just booked
Pricing Varies per night across a season boundary Single per-seat price
Locking Per-room lock; critical section is a range conflict test Per-seat lock (often short TTL hold before payment); set-membership test

12 — Interview Follow-up Q&A

How would this work across multiple app instances?
Replace the in-process ReentrantLock with a distributed lock keyed by room id (Redis SET NX PX), or push correctness into the DB: a reservation(room_id, checkin, checkout, status) table with a Postgres exclusion constraint EXCLUDE USING gist (room_id WITH =, daterange WITH &&), or a serializable transaction doing the overlap SELECT … FOR UPDATE before insert. The exclusion constraint makes the DB itself reject overlaps — the strongest guarantee.
How do you support hold-before-payment (like a cart)?
Add a HELD state with a TTL (as BookMyShow does for seats); a sweeper expires stale holds back to available. Here the hold is over the range, not a seat. Payment then slots in as a step before the reservation transitions to RESERVED.
How would you allow deliberate overbooking?
Hotels and airlines overbook on purpose. Allow N concurrent blocking reservations per room-type bucket up to capacity + overbookMargin. The overlap check becomes a count against a threshold rather than a boolean — count blocking reservations that overlap the range and reject only when the count reaches the margin.
How would you handle a group booking of M rooms atomically?
All-or-nothing across M rooms. Acquire the M room locks in a consistent global order (sorted by room id) to avoid deadlock, check all for overlap, reserve all, else roll back and release. Consistent lock ordering is the key to deadlock-freedom.
Search performance with many reservations?
The in-memory linear scan per room is O(reservations/room) — fine for the exercise. At scale, use a per-room interval tree / segment index keyed by date, a per-room sorted list with binary search on check-in, or a per-room per-day calendar bitmap. Availability queries then avoid scanning the full history.
Why keep a separate NO_SHOW state instead of reusing CANCELLED?
NO_SHOW is a distinct terminal state precisely so billing and analytics can charge it differently from a clean CANCELLED — a no-show typically forfeits the first night while a timely cancel may be free. Both flip blocksAvailability() to false, freeing the dates.
How do you make reserve idempotent under client retries?
Accept a client-supplied idempotency key. A dedupe map returns the same Reservation on retry instead of creating a second booking. This protects against a client re-sending after a timeout when the first request actually succeeded.
How do you handle time zones / DST?
Use LocalDate for the hotel's local calendar — a "night" is a calendar date, not an instant — which sidesteps DST entirely. Store the hotel's zone separately for display. This is why DateRange is built on LocalDate, not Instant.
Why sample occupancy lock-free and before taking the room lock?
Occupancy is an advisory input to dynamic pricing, not the availability decision (which is locked). Reading it lock-free from CopyOnWriteArrayLists is cheap and can be marginally stale without harm. Sampling it before taking the room lock guarantees no lock-ordering hazard — we never hold room A's lock while reaching for room B's.
