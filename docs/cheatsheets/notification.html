01 — Requirements

Functional

  • Send notifications via multiple channels: Email, SMS, Push, In-App
  • Support priority levels: URGENT, HIGH, MEDIUM, LOW — higher priority processed first
  • Retry failed notifications with configurable policy and exponential backoff
  • Track notification status lifecycle: PENDING → SENT / FAILED → RETRYING
  • Template-based message rendering with variable substitution (e.g. {{name}})
  • Batch processing — group notifications for bulk delivery
  • Query notification status by ID at any time
  • Async processing via background worker threads

Non-Functional

  • Thread-safe — concurrent sends, worker threads, shared priority queue
  • Low latency — response time < 100ms for enqueueing
  • Scalable — support 10,000+ concurrent notifications
  • Reliable — retry with exponential backoff, no lost notifications
  • Extensible channels — add new delivery mechanisms without modifying core
  • Pluggable templates — swap message formats per notification type

03 — Design Patterns

Four patterns, each earning its place in a production notification system.

Observer Pattern

Notification delivery is decoupled from business logic. The service enqueues a notification, and the background worker observes the queue and dispatches via channels. Status changes propagate to any listener.

Strategy Pattern

NotificationChannel determines routing. Each channel (Email, SMS, Push, In-App) is a strategy for delivery. The sendViaChannel() method dispatches based on channel type — swappable without core changes.

Template Pattern

NotificationTemplate defines a skeleton message with {{variable}} placeholders. The render() method fills in values — same template, different content per user.

Factory Pattern

Notification creation encapsulated in the service. sendNotification() acts as a factory — generates UUID, creates Notification with correct defaults (PENDING status, retryCount=0, timestamp).

Priority Queue Comparator

private final PriorityQueue<Notification> pendingQueue = new PriorityQueue<>(
    Comparator.comparing(Notification::getPriority,
                       Comparator.comparingInt(Priority::getValue).reversed())
);

reversed() ensures URGENT(4) is dequeued before LOW(1). The PriorityQueue is a min-heap by default — reversed makes it a max-heap on priority value.

06 — Key Algorithms

Queue Processing (Producer-Consumer)

Time: O(log n) per enqueue/dequeue (heap operations). Space: O(n) for pending queue.

private void processQueue() {
    while (true) {
        Notification notification = null;

        synchronized (pendingQueue) {
            while (pendingQueue.isEmpty()) {
                try {
                    pendingQueue.wait();  // block until work arrives
                } catch (InterruptedException e) {
                    return;
                }
            }
            notification = pendingQueue.poll();  // highest priority first
        }

        if (notification != null) {
            sendViaChannel(notification);
        }
    }
}

Pattern: Classic wait/notify producer-consumer. Producers call pendingQueue.notifyAll() after offer. Worker blocks on empty queue instead of busy-waiting.

Retry with Exponential Backoff

Time: O(n) scan of all notifications. Backoff: delay × 2attempt

public class RetryPolicy {
    private final int maxRetries;
    private final long retryDelayMs;

    public long getDelayForAttempt(int attempt) {
        return retryDelayMs * (1L << attempt);  // 1s, 2s, 4s, 8s...
    }
}

// In NotificationServiceImpl:
public void retryFailed() {
    for (Notification n : notifications.values()) {
        if (n.getStatus() == NotificationStatus.FAILED
                && n.getRetryCount() < MAX_RETRIES) {
            n.setStatus(NotificationStatus.RETRYING);
            n.incrementRetry();
            synchronized (pendingQueue) {
                pendingQueue.offer(n);
                pendingQueue.notifyAll();
            }
        }
    }
}

Template Rendering

Time: O(t × v) where t = template length, v = number of variables.

public class NotificationTemplate {
    private final String template;

    public String render(Map<String, String> variables) {
        String result = template;
        for (Map.Entry<String, String> entry : variables.entrySet()) {
            result = result.replace(
                "{{" + entry.getKey() + "}}",
                entry.getValue());
        }
        return result;
    }
}

// Usage:
NotificationTemplate tmpl = new NotificationTemplate(
    "welcome", "Welcome", "Hello {{name}}, your order {{orderId}} is confirmed!");
tmpl.render(Map.of("name", "Alice", "orderId", "ORD-123"));
// → "Hello Alice, your order ORD-123 is confirmed!"

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap notifications store Safe concurrent reads/writes from producer and consumer threads
synchronized(pendingQueue) offer + poll + wait/notifyAll PriorityQueue is NOT thread-safe — must synchronize all access for producer-consumer coordination
ExecutorService Worker thread pool (4 threads) Background processing — decouples enqueue (fast) from delivery (slow I/O)
wait() / notifyAll() Queue processing loop Worker blocks when queue empty, wakes when new notification arrives — no busy-waiting
UUID.randomUUID() ID generation Thread-safe unique ID generation without coordination
"The synchronized block is on the PriorityQueue object, not the service. This means multiple notifications on different queues (in a sharded design) can be processed concurrently. The ConcurrentHashMap allows status lookups while the worker thread is processing."

Producer-Consumer Pattern

// PRODUCER (any thread calling sendNotification)
public String sendNotification(...) {
    Notification notification = new Notification(...);
    notifications.put(id, notification);       // ConcurrentHashMap: safe
    synchronized (pendingQueue) {
        pendingQueue.offer(notification);      // O(log n) heap insert
        pendingQueue.notifyAll();               // wake consumer
    }
    return id;
}

// CONSUMER (worker thread in ExecutorService)
private void processQueue() {
    while (true) {
        synchronized (pendingQueue) {
            while (pendingQueue.isEmpty())
                pendingQueue.wait();          // release lock, sleep
            notification = pendingQueue.poll(); // O(log n) heap remove
        }
        sendViaChannel(notification);           // outside lock: I/O
    }
}

09 — Happy Path Flow

Trace this end-to-end in the interview.

Create Notification
id = sendNotification("user-42", "Your order shipped!", EMAIL, HIGH)
UUID generated. Notification created with status=PENDING, retryCount=0, createdAt=now()
Store & Enqueue
notifications.put(id, notification)
ConcurrentHashMap stores for status lookup. PriorityQueue.offer() inserts by priority. notifyAll() wakes worker.
Worker Dequeues
notification = pendingQueue.poll()
Worker thread was blocked on wait(). Wakes up, acquires lock, polls highest-priority notification (URGENT before LOW).
Channel Dispatch
sendViaChannel(notification) — EMAIL adapter
Outside synchronized block (no lock held during I/O). Simulates network call. 90% success rate.
Success Path
notification.setStatus(SENT)
Status updated in-place. getNotificationStatus(id) now returns SENT.
Failure Path (10% chance)
notification.setStatus(FAILED)
Channel threw exception. Notification marked FAILED. Stays in ConcurrentHashMap for retry.
Retry
retryFailed() → re-enqueue with RETRYING status
Scans all notifications. If FAILED and retryCount < 3: incrementRetry(), set RETRYING, offer back to queue. Exponential backoff: delay × 2^attempt.
stateDiagram-v2
    [*] --> PENDING: sendNotification()
    PENDING --> SENT: channel success
    PENDING --> FAILED: channel error
    FAILED --> RETRYING: retryFailed() [count < MAX]
    RETRYING --> SENT: retry success
    RETRYING --> FAILED: retry fails again
    SENT --> [*]
    FAILED --> [*]: max retries exceeded

11 — Interview Follow-up Q&A

How would you handle millions of notifications per minute?
Replace the in-memory PriorityQueue with Apache Kafka or RabbitMQ with priority queues. Partition by userId for ordering guarantees. Scale consumers horizontally. The NotificationService interface stays the same — just swap the implementation.
How do you prevent duplicate notifications?
Store a deduplication key (hash of userId + message + channel + time window) in a ConcurrentHashMap or Redis SET with TTL. Check before enqueue. Idempotency key on the API layer prevents duplicate calls from retrying clients.
Why PriorityQueue instead of a simple LinkedList?
PriorityQueue gives O(log n) insert/remove with ordering by priority. A LinkedList would be O(n) to find the correct insertion point or O(1) append but O(n) to find highest priority. In production with millions of pending notifications, the heap-based priority queue is significantly faster.
How would you implement rate limiting per user?
Add a Map<String, RateLimiter> (e.g., token bucket per userId). Before enqueuing, check rateLimiter.tryAcquire(). If exceeded, either reject with a 429-equivalent or queue for delayed delivery. Different limits per channel (e.g., 5 SMS/hour, 50 push/hour).
Why synchronized on PriorityQueue instead of using BlockingQueue?
PriorityBlockingQueue would work and is cleaner for simple cases. However, explicit synchronized + wait/notify gives finer control: you can batch-drain the queue, add custom wake conditions, and combine with retry logic. In production, I'd use PriorityBlockingQueue for simplicity.
How would you add user notification preferences?
Add a UserPreferences entity with: preferred channels (ordered fallback list), quiet hours, opt-out categories. Before dispatch, the service checks preferences and either routes to the preferred channel, delays for quiet hours, or skips if opted out. The NotificationChannel in the request becomes a suggestion, not a mandate.
How does the retry policy handle thundering herd?
Exponential backoff (delay * 2^attempt) naturally spreads retries over time. Add jitter (random offset) to prevent synchronized retry storms: delay * 2^attempt + random(0, delay). Cap the max backoff to prevent indefinite delays. Circuit breaker pattern stops retries if the channel is down.
How would you support scheduled notifications?
Add a scheduledAt field to Notification. Use a ScheduledExecutorService or a separate DelayQueue<Delayed>. Notifications with future timestamps go to the delay queue. A background thread polls and transfers them to the main PriorityQueue when their time arrives.
How would you track delivery confirmation?
Add DELIVERED status after SENT. For email: webhook from provider (SendGrid/SES delivery events). For push: FCM/APNs delivery receipts. For SMS: carrier delivery reports. For in-app: client-side read receipt. Store timestamps for each status transition for SLA monitoring.
Why Template pattern instead of String.format()?
NotificationTemplate with named placeholders ({{name}}) is more readable and maintainable than positional %s. Templates can be stored in a database and edited by non-engineers. In production, use a proper template engine (Mustache, FreeMarker) for conditionals, loops, and localization.
How would you handle multi-language notifications?
Store templates per locale: Map<String, Map<Locale, NotificationTemplate>>. User preferences include a locale. The service resolves the template by (templateId, locale), falling back to a default locale. Variable substitution stays the same — only the template text changes per language.
What happens if the service crashes mid-processing?
In-memory queue is lost. For production: (1) persist notifications to a database before enqueuing, (2) use a durable message queue (Kafka, SQS), (3) on startup, reprocess all PENDING/RETRYING notifications. The ConcurrentHashMap becomes a database-backed store with write-ahead semantics.
