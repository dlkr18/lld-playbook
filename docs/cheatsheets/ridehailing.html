01 — Requirements

Functional

  • Riders register, request rides with vehicle type preference
  • Drivers register with a Vehicle, go online/offline, accept rides
  • Trip lifecycle: Request → Accept → Start → Complete (or Cancel)
  • Fare estimation on request, actual fare on completion
  • Tiered cancellation fees based on trip status
  • Post-trip bilateral rating (rider ↔ driver)
  • Payment processing after trip completion

Non-Functional

  • Thread-safe — multiple drivers accepting simultaneously
  • Extensible pricing — surge, time-of-day, distance tiers
  • Pluggable matching — nearest, highest-rated, ETA-based
  • Observable events — notifications at each lifecycle step

03 — Design Patterns

The interview differentiator — four patterns, each earning its place.

State Pattern

Trip lifecycle transitions. Each TripStatus is a stateless singleton class defining exactly which transitions are legal. Invalid ops throw immediately.

Strategy Pattern

PricingStrategy (base, surge) and DriverMatchingStrategy (nearest). Injected at construction, swappable at runtime.

Decorator Pattern

SurgePricingStrategy wraps any PricingStrategy with a multiplier. Composable: you can stack surge on top of peak-hour on top of base.

Observer Pattern

Rider & Driver implement TripObserver. Trip holds its observers and fires notifyObservers() on every state change. Three layers: trip observers, ride-request broadcast, direct notify.

04 — Trip State Machine

Why State? Eliminates scattered if/else. Each state class defines EXACTLY which transitions are legal. Adding a new state (e.g. DRIVER_ARRIVING) = adding a new class. Open/Closed Principle. 
stateDiagram-v2
    [*] --> Requested
    Requested --> Accepted: accept(driverId)
    Requested --> Cancelled: cancel [no fee]
    Accepted --> InProgress: start()
    Accepted --> Cancelled: cancel [$25 fee]
    InProgress --> Completed: complete(fare)
    InProgress --> Cancelled: cancel [full fare]
    Completed --> [*]
    Cancelled --> [*]

TripState Interface

public interface TripState {
    TripState accept(Trip trip, String driverId);
    TripState start(Trip trip);
    TripState complete(Trip trip, double actualFare);
    TripState cancel(Trip trip, String cancelledBy, String reason);
    TripStatus getStatus();
}

Trip Delegates to State

// In Trip.java -- one-liner delegation
public void accept(String driverId) {
    this.state = state.accept(this, driverId); // returns next state
}

States are STATELESS singletons (INSTANCE). All mutable data stays on Trip. States access Trip via package-private setXxxInternal() methods.

06 — Observer + Decorator

3 notification layers — State pattern owns transitions, Observer pattern owns WHO gets notified, NotificationService owns HOW delivery happens.

Layer 1: Trip Observer (State Changes)

Dedicated RiderNotifier and DriverNotifier implement TripObserver. Entities stay pure (SRP). Trip holds observers and auto-fires notifyObservers() after every state transition.

interface TripObserver {
    void update(Trip trip); // single method, checks trip.getStatus()
}

// Separate notifier classes -- NOT on the entity (SRP)
class RiderNotifier implements TripObserver {
    private final String riderId;
    private final NotificationService ns;
    void update(Trip trip) { ns.notify(riderId, formatMsg(trip)); }
}
class DriverNotifier implements TripObserver { /* same shape */ }

// In Trip.java (the Subject):
private final List<TripObserver> observers = new CopyOnWriteArrayList<>();

public void accept(String driverId) {
    this.state = state.accept(this, driverId);
    notifyObservers(); // RiderNotifier + DriverNotifier get update(this)
}
EventWho Observes
ACCEPTEDRiderNotifier ("Driver is on the way!")
IN_PROGRESSRiderNotifier + DriverNotifier ("Trip started")
COMPLETEDRiderNotifier ("Fare: $X") + DriverNotifier ("Earnings: $X")
CANCELLEDRiderNotifier ("Cancelled" + fee) + DriverNotifier ("Cancelled by rider/you")

Layer 2: Ride-Request Broadcast

When a ride is requested, no driver notifier is subscribed yet. The service broadcasts directly via NotificationService.

// In requestRide():
trip.addObserver(new RiderNotifier(riderId, notificationService));

// Broadcast to nearby matching drivers (NOT observers yet)
for (Driver d : nearbySedanDrivers) {
    notificationService.notify(d.getDriverId(), "New ride near you...");
}

// In acceptRide():
trip.addObserver(new DriverNotifier(driverId, notificationService));
trip.accept(driverId); // fires notifyObservers()

Key insight: Broadcast and Trip Observer are two separate concerns. Broadcast reaches candidates; Observer reaches participants.

Layer 3: Direct Notification (Non-State Events)

Payment, rating, go online/offline are not Trip state changes. The service calls NotificationService directly.

// Delivery channel interface
interface NotificationService {
    void notify(String userId, String message);
}

// Service uses it directly for non-state events:
notificationService.notify(riderId, "Payment of $5.23 processed");
notificationService.notify(driverId, "You received a 5-star rating");
notificationService.notify(driverId, "You are now online");

"Swap ConsoleNotificationService for SMS, push, or Kafka — zero code changes elsewhere."

Decorator — Surge Pricing

class SurgePricingStrategy implements PricingStrategy {
    private final PricingStrategy delegate;  // wraps base
    private final double surgeMultiplier;

    double calculateFare(double dist, VehicleType type) {
        return delegate.calculateFare(dist, type) * surgeMultiplier;
    }
}

Why Decorator vs subclass? Composes at runtime. You can stack: new Surge(new PeakHour(base), 1.5)

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap All data stores Safe concurrent reads/writes
synchronized(trip) accept, start, complete, cancel Fine-grained lock per trip — only one thread transitions a specific trip at a time
volatile Driver.status, Driver.location Cross-thread visibility without full lock
CopyOnWriteArrayList Trip.observers list Safe iteration while adding/removing TripObservers
AtomicLong ID generation Lock-free incrementing
"I use fine-grained locking on the Trip object, not a global lock. Two different trips can be processed concurrently. Only concurrent operations on the same trip are serialized."

12 — Interview Follow-up Q&A

How would you handle surge pricing?
Already implemented — SurgePricingStrategy is a Decorator wrapping BasePricingStrategy. The service holds a volatile PricingStrategy reference that can be swapped at runtime. In production, a background job monitors demand/supply ratio and activates surge.
How would you find the nearest driver at scale?
In-memory we iterate all drivers. At scale: geospatial index (QuadTree, GeoHash, or Redis GEOSEARCH). The DriverMatchingStrategy interface stays the same — just swap the implementation.
How would you handle ride timeout (no driver accepts)?
ScheduledExecutorService schedules a task to cancel the trip after N seconds if still in REQUESTED state. The state pattern makes this safe — cancel() on a REQUESTED trip transitions to CANCELLED cleanly.
How do you prevent two drivers accepting the same ride?
synchronized(trip) — only one thread enters the block at a time. The first driver to call acceptRide transitions REQUESTED → ACCEPTED. The second driver finds the trip in ACCEPTED state and trip.accept() throws "already accepted".
Why State pattern instead of just if/else on TripStatus?
(1) Each state class is self-documenting — you see exactly what's legal. (2) Adding a new state (e.g., DRIVER_ARRIVING) is a new class, no modification to existing code (Open/Closed). (3) Invalid transitions throw immediately with clear messages — no forgotten else branches.
Why Strategy instead of just a calculateFare method?
(1) Pricing changes independently of the service — new pricing model = new class. (2) SurgePricingStrategy is a Decorator that composes with any base strategy. (3) Swappable at runtime without restarting the service.
How would you handle payment failures?
Payment entity already has a lifecycle: PENDING → COMPLETED / FAILED / REFUNDED. On failure, retry with exponential backoff. After max retries, mark FAILED and alert the rider. In production, integrate with a payment gateway (Stripe/Razorpay) behind a PaymentGateway interface.
How do Driver and Vehicle relate?
One Driver has exactly one Vehicle (composition, set at registration). Vehicle is immutable (type, plate, model, color). Driver matching filters by vehicle type to ensure a SEDAN request only matches SEDAN drivers.
Why three separate notification layers?
Trip Observer (dedicated RiderNotifier/DriverNotifier classes implement TripObserver) handles state-change notifications — auto-fired by Trip. Entities stay pure data (SRP). Ride-request broadcast reaches nearby drivers who aren't subscribed yet. Direct notification handles non-state events (payment, rating, go online/offline) via the service calling NotificationService directly. The State pattern owns transitions, the Observer owns WHO, and NotificationService owns HOW.
Why not put per-state methods on TripObserver (onAccepted, onStarted, etc.)?
That duplicates the state awareness the State pattern already handles. A single update(Trip) method keeps Observer and State cleanly separated. The observer checks trip.getStatus() if it needs to react differently per state.
