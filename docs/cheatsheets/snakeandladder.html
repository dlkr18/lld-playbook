01 — Requirements

Functional

  • Configurable board size (default 100 cells, minimum 10)
  • Place snakes (head > tail) and ladders (bottom < top) on the board
  • Support 2+ players taking turns rolling a dice
  • Landing on a snake head → slide down to tail
  • Landing on a ladder bottom → climb up to top
  • Support snake/ladder chains (land on ladder top which is a snake head)
  • Exact landing required to win — overshoot means stay put
  • First player to reach cell N (board size) wins
  • Game stats tracking: total rolls, snake hits, ladder hits, roll frequency

Non-Functional

  • Thread-safe — synchronized game turns, safe for concurrent access
  • Configurable dice — variable number of faces (default 6)
  • Extensible — easy to add new rules (extra turn on 6, power-ups)
  • Validation — snakes/ladders within bounds, no duplicate positions, no overlap
  • Safety guard — max 10,000 moves to prevent infinite loops

03 — Design Patterns

Four patterns, each earning its place in an interview discussion.

Strategy — Dice Rolling

Dice encapsulates the rolling algorithm. Swap a fair dice for a loaded dice, multiple dice, or a test-friendly deterministic dice — zero changes to the game engine.

Observer — Game Events

GameStats observes every roll, snake hit, and ladder hit. Add loggers, UI updaters, or analytics collectors without touching the game loop.

Factory / Builder — Board Setup

Board.builder(100).addSnake(99,10).addLadder(2,38).build() — fluent API with validation on build. GameConfig.standard() is a static factory for default configs.

Template Method — Game Flow

The playTurn() method defines the skeleton: roll → move → resolve snakes/ladders → check win → advance turn. Subclasses or extensions can override individual steps.

playTurn() — The Core Loop

public synchronized String playTurn() {
    if (gameOver) return "Game is already over. Winner: " + winner.getName();

    Player player = players.get(currentPlayerIndex);
    int roll = dice.roll();
    stats.recordRoll(roll);

    int oldPos = player.getPosition();
    int newPos = oldPos + roll;

    // Overshoot: must land exactly on boardSize
    if (newPos > boardSize) {
        advanceTurn();
        return player.getName() + " stays at " + oldPos + " (overshoot)";
    }

    player.setPosition(newPos);
    newPos = resolvePosition(newPos, log);  // chain snakes/ladders
    player.setPosition(newPos);

    if (newPos == boardSize) {
        gameOver = true;
        winner = player;
        return player.getName() + " WINS!";
    }

    advanceTurn();
}

06 — Key Algorithms

Chain Resolution (Snake/Ladder Chains)

Time: O(k) where k = chain length (capped at 20)  ·  Space: O(1)

private int resolvePosition(int pos, StringBuilder log) {
    int maxChain = 20; // safety guard against infinite cycles
    while (maxChain-- > 0) {
        if (snakeMap.containsKey(pos)) {
            int dest = snakeMap.get(pos);
            log.append(" [SNAKE " + pos + "->" + dest + "]");
            stats.recordSnakeHit();
            pos = dest;
        } else if (ladderMap.containsKey(pos)) {
            int dest = ladderMap.get(pos);
            log.append(" [LADDER " + pos + "->" + dest + "]");
            stats.recordLadderHit();
            pos = dest;
        } else {
            break;
        }
    }
    return pos;
}

Why the cap? If someone misconfigures a circular chain (snake tail = ladder bottom, ladder top = snake head), this prevents an infinite loop.

Full Game Loop

Time: O(turns × chain_length)  ·  Guard: 10,000 move limit

public GameResult playFull() {
    int moves = 0;
    while (!gameOver) {
        playTurn();
        moves++;
        if (moves > 10_000)
            throw new RuntimeException("Exceeded 10000 moves");
    }
    return new GameResult(winner, moves);
}

Turn Advancement

private void advanceTurn() {
    currentPlayerIndex = (currentPlayerIndex + 1) % players.size();
}

Circular rotation via modulo. Supports any number of players.

Extensible Rules (Interview Discussion Points)

RuleHow to Add
Extra turn on 6 After roll, if roll == dice.getSides(), don't call advanceTurn(). Same player goes again.
Three 6s penalty Track consecutive sixes counter per player. On 3rd six, reset to position 0 and advance turn.
Power-ups on cells Add a Map<Integer, PowerUp> alongside snakeMap/ladderMap. Check after chain resolution.
Multiple dice Change Dice to DiceSet that rolls N dice and sums. Strategy pattern keeps game engine unchanged.

08 — Thread Safety

MechanismWhereWhy
synchronized playTurn() method Only one thread can execute a turn at a time — prevents race conditions on position updates and turn order
HashMap snakeMap, ladderMap Read-only after construction — effectively immutable, inherently thread-safe
Collections.unmodifiableList getPlayers() Prevents external mutation of the player list
Fail-fast constructor Snake, Ladder, Dice All validation at construction — objects are valid from birth
"The synchronized keyword on playTurn() ensures turn atomicity. Since snakeMap and ladderMap are never modified after construction, they're effectively immutable and safe for concurrent reads."

12 — Interview Follow-up Q&A

How would you generalize the board size?
Already supported — the constructor takes boardSize as a parameter. Change from 100 to 200, 1000, or any value ≥ 10. The HashMap-based lookup scales linearly with the number of snakes/ladders, not board size. For very large boards (10K+ cells), consider a sorted array with binary search instead of HashMap to reduce memory overhead.
How would you support multiple dice?
Extract a DiceRoller interface: int roll(). The current Dice becomes SingleDice implements DiceRoller. Add MultiDice implements DiceRoller that rolls N dice and sums. The game engine depends only on the interface — Strategy pattern. With two 6-sided dice, the range becomes 2-12 and probability distribution changes (7 is most likely), affecting average game length.
How would you add power-ups (shield, double move, teleport)?
Add a Map<Integer, PowerUp> powerUpMap alongside snakeMap/ladderMap. After chain resolution in resolvePosition(), check for power-ups. PowerUp is an interface with apply(Player, Game). Shield: set a flag on Player that blocks the next snake. Double: give an extra turn. Teleport: move to a random cell. Open/Closed Principle — new power-ups don't modify existing code.
What's the minimum number of rolls to win? (BFS approach)
Model the board as a directed graph: from each cell, you can reach cells +1 through +6 (dice values). Snakes and ladders are additional edges. Run BFS from cell 0 — each level = one dice roll. The first time you reach cell N is the minimum rolls. Time: O(N × D) where D = dice sides. This is a classic interview follow-up. With a standard 10×10 board and well-placed ladders, minimum can be as low as 1 roll (cell 1 has ladder to 100, but that's contrived) or typically 7-8 rolls.
How would you implement save/resume (persistence)?
Serialize the game state: boardSize, snakeMap, ladderMap, players (name + position), currentPlayerIndex, gameOver, stats. Use JSON or Protocol Buffers. On resume, reconstruct the game from the saved state. Key decision: save after every turn (durability) vs. save on demand (performance). For a web game, store in a database with a gameId key. The GameConfig class already separates configuration from runtime state, making serialization clean.
How would you scale to multiplayer (100+ concurrent games)?
Each game is independent — no shared state between games. Run games in separate threads or use an event loop. The synchronized playTurn() ensures per-game thread safety. For a web server: one game object per session, stored in a ConcurrentHashMap<String, SnakeAndLadderGame>. For massive scale, use Redis for game state and Pub/Sub for real-time turn notifications. WebSockets for real-time UI updates.
How do you ensure the dice is fair?
java.util.Random provides uniform distribution via nextInt(faces) + 1. For testing, inject a deterministic dice (Strategy pattern): class FixedDice extends Dice that returns a preset sequence. This lets you write reproducible unit tests for specific scenarios (e.g., always roll 6 to test the exact-landing rule). In production, use SecureRandom for cryptographic fairness if real money is involved.
How would you add "extra turn on rolling 6"?
In playTurn(), after the move: if (roll == dice.getSides()) { /* don't call advanceTurn() */ }. The same player goes again. For the "three consecutive 6s = penalty" rule, add a consecutiveSixes counter per player. On the 3rd six, reset position to 0 and force advance. This is a great place for the Template Method pattern — override just the post-roll hook.
Why HashMap for snakes/ladders instead of a boolean array?
The board is sparse: ~16 snakes/ladders on 100 cells = 16% occupancy. HashMap uses memory proportional to the number of snakes/ladders (O(S+L)), not the board size (O(N)). For a 10,000-cell board with 20 snakes, HashMap uses ~20 entries vs. a 10,000-element array. HashMap also gives O(1) lookup. An int[] board array would also work (cell value = destination or 0) and is cache-friendly — valid trade-off for small boards.
How would you handle disconnections in an online game?
Add a turn timer with ScheduledExecutorService. If a player doesn't roll within N seconds, auto-roll for them (or skip their turn). Track connection state per player. After M consecutive timeouts, mark player as disconnected and either (a) continue without them or (b) pause the game. Use heartbeat pings for detection. The game state is already isolated, so reconnection just means sending the current state to the client.
