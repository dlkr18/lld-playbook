01 — Requirements

Functional

  • Movie Management — Add/update movies with title, genre, language, duration, rating
  • Theater & Screen — Multiple theaters per city, each with multiple screens and configurable seat types (REGULAR, PREMIUM, VIP, RECLINER)
  • Show Scheduling — Schedule shows with start/end time, dynamic pricing per seat type
  • Real-time Seat Map — View available seats instantly, distinguish locked/booked/available
  • Seat Locking — Temporary lock (5 min) during booking; auto-release on timeout
  • Payment & Confirmation — Process payment, confirm booking, release locks on failure
  • Cancellation — Cancel confirmed bookings, auto-release seats back to AVAILABLE
  • Search & Filter — Find movies by city, theater, language, genre, date

Non-Functional

  • Performance — Seat availability < 100ms, booking completion < 500ms, 10,000+ concurrent bookings
  • Concurrency — Per-seat locking (not show-level), thread-safe lock manager, deadlock prevention via sorted seat IDs
  • Reliability — ACID for payment + booking, automatic cleanup of expired locks, rollback on failure
  • Scalability — Distributed locks (Redis) for production, DB sharding by city, caching for metadata

03 — Design Patterns

The interview differentiator — six patterns, each earning its place.

Strategy Pattern

Different payment methods (Card, UPI, Wallet) and pricing strategies. Easy to add new payment gateways without changing booking logic.

Observer Pattern

Notify users on booking confirmation, cancellation, payment status. Decoupled notification system — swap email for SMS or push without touching booking code.

Singleton Pattern

SeatLockManager — single point of lock coordination across all shows. Ensures consistent lock state and prevents split-brain locking scenarios.

Factory Pattern

Create different seat types with preconfigured base prices. Centralized seat creation logic keeps pricing consistent across theaters.

Template Method

Booking flow with hooks — validate → lock → pay → confirm. Subclasses override individual steps (e.g., VIP booking with lounge access).

Repository Pattern

Data access layer abstracts ConcurrentHashMap stores. In production, swap for JPA/Hibernate repositories with zero business logic changes.

04 — Booking State Machine

Why track state? A booking progresses through well-defined states. Each transition has side effects (lock/unlock seats, process payments). Illegal transitions throw immediately — no stale bookings. 
stateDiagram-v2
    [*] --> PENDING: createBooking()
    PENDING --> CONFIRMED: confirmBooking(payment)
    PENDING --> EXPIRED: lock timeout (5 min)
    CONFIRMED --> CANCELLED: cancelBooking()
    EXPIRED --> [*]
    CANCELLED --> [*]
    CONFIRMED --> [*]

Booking Status Transitions

FromToTriggerSide Effects
PENDING CONFIRMED confirmBooking(payment) Process payment, mark seats BOOKED, release locks, send confirmation
PENDING EXPIRED Lock timeout (5 min) Release seat locks, seats → AVAILABLE, notify user
CONFIRMED CANCELLED cancelBooking() Refund payment, seats → AVAILABLE, notify user

Seat Status Transitions

stateDiagram-v2
    [*] --> AVAILABLE
    AVAILABLE --> LOCKED: lockSeats()
    LOCKED --> BOOKED: confirmBooking()
    LOCKED --> AVAILABLE: timeout / cancel / payment failure
    BOOKED --> AVAILABLE: cancelBooking()

Confirm Booking — State Guard

public boolean confirmBooking(String bookingId, Payment payment) {
    Booking booking = getBooking(bookingId);

    if (booking.getStatus() != BookingStatus.PENDING) {
        throw new IllegalStateException("Booking not in PENDING state");
    }

    processPayment(payment);
    booking.setStatus(BookingStatus.CONFIRMED);
    seatLockManager.unlockSeats(showId, seatIds, userId);
    // Seats move from lock map to booked set
    return true;
}

05 — Seat Selection & Locking

The hardest part of BookMyShow. 1,000 users picking seats for the same show — per-seat locking lets them proceed in parallel as long as they pick different seats.

Locking Approaches Compared

ApproachGranularityThroughputVerdict
Show-level synchronized(show) Entire show 1 user at a time Not scalable
DB Row Locking SELECT FOR UPDATE Per seat (DB) Medium High DB load
Per-Seat In-Memory Per seat (memory) High (< 10ms) Chosen approach
Redis Distributed Lock Per seat (distributed) High Production upgrade

SeatLockManager — Core Implementation

public class SeatLockManager {
    // Key: "showId:seatId" → LockInfo
    private final ConcurrentHashMap<String, LockInfo> seatLocks = new ConcurrentHashMap<>();
    private final ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(4);
    private static final long LOCK_TIMEOUT_MS = 300000; // 5 minutes

    static class LockInfo {
        final String userId;
        final long expiryTime;
        final ScheduledFuture<?> unlockTask;
    }
}

Composite key showId:seatId enables per-seat granularity — 1000 users booking different seats for the same show proceed in parallel.

Two-Phase Lock Acquisition

public boolean lockSeats(String showId, List<String> seatIds, String userId) {
    // 1. Sort seat IDs to prevent deadlock
    List<String> sorted = new ArrayList<>(seatIds);
    Collections.sort(sorted);

    // Phase 1: Check all seats are available
    for (String key : seatKeys) {
        LockInfo existing = seatLocks.get(key);
        if (existing != null && existing.expiryTime > now)
            return false; // seat unavailable
    }

    // Phase 2: Atomically lock all seats
    for (String key : seatKeys) {
        LockInfo prev = seatLocks.putIfAbsent(key, lockInfo);
        if (prev != null) {
            rollback(); // undo locks acquired so far
            return false;
        }
    }
    return true;
}

Deadlock Prevention

Always acquire locks in sorted seat ID order. If User A wants seats [S3, S1] and User B wants [S1, S3], both acquire in order [S1, S3] — no circular wait.

// Without sorting: User A locks S3, waits for S1
//                  User B locks S1, waits for S3 → DEADLOCK

// With sorting: Both lock S1 first, then S3 → no deadlock
Collections.sort(sortedSeatIds);

Auto-Expiry with ScheduledExecutorService

// Schedule auto-unlock when lock is acquired
ScheduledFuture<?> task = scheduler.schedule(
    () -> unlockSeat(showId, seatId, userId),
    LOCK_TIMEOUT_MS,
    TimeUnit.MILLISECONDS
);

// Cancel the scheduled task if payment completes before timeout
lockInfo.unlockTask.cancel(false);

07 — Key Algorithms

Seat Availability Check

public List<Seat> getAvailableSeats(String showId) {
    Show show = getShow(showId);
    Screen screen = screens.get(show.getScreenId());
    Set<String> booked = bookedSeats.getOrDefault(showId, Collections.emptySet());

    return screen.getSeats().stream()
        .filter(seat -> !booked.contains(seat.getId()))
        .filter(seat -> !seatLockManager.isLocked(showId, seat.getId()))
        .collect(Collectors.toList());
}
// Time: O(n) where n = seats in screen
// Checks both booked set AND active locks

Show Search by Movie + City

public List<Show> getShowsForMovie(String movieId, City city) {
    return shows.values().stream()
        .filter(show -> show.getMovieId().equals(movieId))
        .filter(show -> {
            Screen screen = screens.get(show.getScreenId());
            Theater theater = theaters.get(screen.getTheaterId());
            return theater.getCity() == city;
        })
        .collect(Collectors.toList());
}
// Traversal: Show → Screen → Theater → City

Booking Amount Calculation

double totalAmount = 0.0;
for (String seatId : seatIds) {

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap All data stores (movies, theaters, shows, bookings, seatLocks) Lock-free reads, safe concurrent writes
putIfAbsent() SeatLockManager lock acquisition Atomic check-and-set prevents double-locking
remove(key, value) SeatLockManager unlock Atomic remove only if same LockInfo — prevents unlocking another user's lock
ConcurrentHashMap.newKeySet() bookedSeats per show Thread-safe Set for tracking booked seats
computeIfAbsent() Initializing booked set per show Atomic creation avoids race on first booking
ScheduledExecutorService Lock timeout auto-release Background thread pool schedules cleanup tasks
Sorted lock order lockSeats() Prevents deadlock when two users select overlapping seats
"I use per-seat locking with composite keys showId:seatId. Two users booking different seats for the same show proceed in parallel. Only overlapping seat selections contend. Combined with sorted acquisition order, deadlocks are impossible."

Double-Booking Prevention

// In lockSeats() -- atomic check-and-set
LockInfo previous = seatLocks.putIfAbsent(seatKey, lockInfo);

if (previous != null) {
    // Race condition: someone locked between check and lock
    task.cancel(false);
    rollbackAllAcquiredLocks();
    return false;
}

// In unlockSeat() -- owner-only release
if (lockInfo != null && lockInfo.userId.equals(userId)) {
    lockInfo.unlockTask.cancel(false);
    seatLocks.remove(seatKey, lockInfo); // only removes if same object
}

Failure Paths

FailureTriggerRecovery
Seat already locked putIfAbsent returns non-null Rollback all acquired locks, return false
Payment fails Gateway error (5% simulated) Payment → FAILED, throw PaymentFailedException

Seat already booked Seat in bookedSeats set Throw SeatNotAvailableException before lock attempt

12 — Interview Follow-up Q&A

How do you handle concurrent bookings for the same seat?
Per-seat locking with ConcurrentHashMap.putIfAbsent(). The composite key showId:seatId provides granular locking. Only one thread can successfully putIfAbsent for a given key — the second thread detects the existing lock and rolls back. This is atomic at the JVM level without explicit synchronized blocks.
How would you implement distributed locking for production?
Replace the in-memory ConcurrentHashMap with Redis using SET key value NX EX 300 (set-if-not-exists with 5-min TTL). For stronger guarantees, use Redlock across multiple Redis nodes. The SeatLockManager interface stays the same — swap InMemorySeatLockManager for RedisSeatLockManager.
What happens if a user's seat lock expires before payment?
The ScheduledExecutorService fires the auto-unlock task after 5 minutes. Seats move back to AVAILABLE. If the user then tries to confirm, confirmBooking() checks that the booking is still in PENDING state and the seats are still locked. If locks expired, confirmation fails and the user must restart the booking flow.
How do you prevent deadlocks when users select overlapping seats?
Always acquire locks in sorted seat ID order. If User A wants [S3, S1] and User B wants [S1, S3], both will try S1 first, then S3. One succeeds, the other fails atomically via putIfAbsent. The loser's already-acquired locks are rolled back immediately. No circular wait is possible.
How would you implement different pricing tiers (peak hours, weekends, offers)?
Use the Decorator pattern on top of a base PricingStrategy. Stack decorators at runtime: new WeekendPricing(new PeakHourPricing(basePricing)). For coupons, create a CouponPricingStrategy that applies flat or percentage discounts after base calculation. The SeatType enum already carries base prices.
How would you implement a waitlist for sold-out shows?
Add a PriorityQueue<WaitlistEntry> per show, ordered by request time. When a seat becomes available (cancellation or lock expiry), dequeue the next user and send a notification with a time-limited offer to book. Use the Observer pattern to fire waitlist notifications on seat release events.
How would you handle cancellation refunds?
Tiered refund policy based on time before showtime: >24h = full refund, 2-24h = 50% refund, <2h = no refund. In cancelBooking(), calculate the time delta, determine refund amount, update Payment status to REFUNDED, and release seats back to AVAILABLE. A RefundPolicy Strategy makes this extensible.
How would you scale to millions of concurrent users?
1) Distributed locking via Redis/Redlock. 2) Database sharding by city or theater_id. 3) Read replicas for seat availability queries. 4) CDN for movie posters, theater images. 5) Message queues (Kafka/RabbitMQ) for async notifications. 6) API gateway with rate limiting. 7) Microservices: separate booking, payment, and notification services. 8) Cache movie/theater metadata with Redis (rarely changes).
Why per-seat locking instead of show-level locking?
Show-level locking (synchronized(show)) means only one user can book at a time for a 1000-seat show. Per-seat locking allows 1000 concurrent bookings as long as they select different seats. Throughput goes from O(1) to O(n) concurrent operations. The tradeoff is higher memory for lock metadata, but lock entries are tiny (<100 bytes each).
How would you handle the "phantom read" problem in seat selection?
Between checking availability and locking, another user might book the seat. The two-phase approach handles this: Phase 1 checks availability (optimistic), Phase 2 uses putIfAbsent (pessimistic). If Phase 2 fails, all Phase 2 locks acquired so far are rolled back atomically. The user sees an "seats no longer available" error and can retry with different seats.
How would you design the seat map UI interaction with the backend?
1) Initial load: GET /shows/{id}/seats returns all seats with status (AVAILABLE/LOCKED/BOOKED). 2) Real-time updates: WebSocket pushes seat status changes. 3) Optimistic UI: mark seats as "selecting" locally, POST to lock endpoint. 4) If lock fails, revert UI and show error. 5) Countdown timer for lock TTL visible to user.
What custom exceptions do you define and why?
Four domain-specific exceptions for clear error handling: SeatNotAvailableException (seat booked/locked), BookingNotFoundException (invalid booking ID), ShowNotFoundException (invalid show ID), PaymentFailedException (gateway error). Each maps to a specific HTTP status code (409, 404, 404, 502) and provides actionable error messages to the client.

Key Takeaways for the Interview

  • Seat locking is the hardest part — per-seat granularity with composite keys, sorted acquisition, atomic putIfAbsent, auto-expiry
  • Two-phase lock — check availability (optimistic), then acquire (pessimistic with rollback)
  • Deadlock prevention — always sort seat IDs before locking
  • Failure recovery — rollback on partial lock failure, auto-release on timeout, refund on cancellation
  • Scale path — in-memory → Redis locks, single DB → sharded, monolith → microservices
