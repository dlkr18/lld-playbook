01 — Requirements

Functional

  • Add elements to the filter
  • Check if element might be present (probabilistic membership test)
  • Configure false positive rate (e.g., 1%)
  • Configure expected number of elements
  • Support for scalable bloom filter (auto-resizing when FPP exceeds threshold)
  • Get filter statistics (size, element count, bits set, fill ratio, FPP)

Non-Functional

  • Performance — O(k) add and contains, where k = number of hash functions
  • Memory — configurable based on false positive rate, far less than HashSet
  • Thread Safety — support concurrent access with BitSet synchronization
  • Scalability — handle millions of elements via ScalableBloomFilter

Use Cases

  • Web Crawlers — avoid re-crawling already visited URLs
  • Database Query Optimization — quick membership check before expensive DB query
  • Spam Filtering — fast spam email detection
  • Cache Filtering — check if item exists before cache lookup
  • Distributed Systems — reduce network calls for non-existent data

03 — Design Patterns

Four patterns — each earning its place in the design.

Strategy Pattern

HashFunction interface with MurmurHashFunction implementation. Swap hash algorithms without changing the core filter. Add CityHash, xxHash, or FNV — zero changes to BloomFilter.

Builder Pattern

BloomFilterConfig computes optimal size and hash functions from expectedElements and falsePositiveRate. Fluent configuration with automatic parameter derivation.

Facade Pattern

BloomFilterService provides a simple interface hiding BitSet manipulation, hash computation, and optimal parameter calculation. Clients call add() and mightContain().

Composite Pattern

ScalableBloomFilter maintains a list of BloomFilter instances. When FPP exceeds the threshold, a new filter (2x size) is added. Queries check all filters.

04 — How Bloom Filter Works

Core idea: A bit array of m bits, all initialized to 0, with k independent hash functions. To add an element, hash it k times and set those bit positions to 1. To query, hash it k times — if ALL bits are 1, "maybe present"; if ANY bit is 0, "definitely not present".

Double Hashing Technique

Instead of k independent hash functions, use two hashes and combine them:

// Generate k hashes from just 2 base hashes
hash_i(element) = hash1(element) + i * hash2(element)) % m

// Where:
//   hash1 = element.hashCode()
//   hash2 = murmurHash(element)
//   i = 0, 1, 2, ..., k-1
//   m = bit array size

This technique by Kirsch & Mitzenmacher (2006) is as effective as k fully independent hash functions.

Key Properties

PropertyValue
False NegativesImpossible — if it says "not present", element is definitely absent
False PositivesPossible — if it says "present", element might not actually be in set
DeletionNot supported — unsetting a bit could affect other elements (use Counting BF)
Space~10 bits per element for 1% FPP vs ~100+ bits for HashSet
TimeO(k) for both add and query, where k is typically 3-10

05 — Formulas & Optimal Sizing

Optimal bit array size (m)
m = −(n × ln(p)) / (ln(2))²
n = expected elements  •  p = desired false positive rate
Optimal number of hash functions (k)
k = (m / n) × ln(2)
m = bit array size  •  n = expected elements
Actual false positive probability after insertions
p = (1 − e−km/size)k
k = hash functions  •  m = elements inserted  •  size = bit array size

Implementation

private static int optimalSize(int n, double p) {
    return (int) Math.ceil(-(n * Math.log(p)) / Math.pow(Math.log(2), 2));
}

private static int optimalHashFunctions(int n, int m) {
    int k = (int) Math.ceil((m / (double) n) * Math.log(2));
    return Math.max(1, k);
}

06 — Counting Bloom Filter & Scalable Variant

Problem: Standard Bloom Filters don't support deletion — clearing a bit could remove other elements. Counting Bloom Filters replace each bit with a counter.

Counting Bloom Filter

OperationStandard BFCounting BF
AddSet bits to 1Increment counters
ContainsCheck bits are 1Check counters > 0
DeleteNot supportedDecrement counters
Space1 bit per slot3-4 bits per slot (4x more)

Trade-off: deletion support at 4x memory cost. Counter overflow is possible but extremely rare.

Scalable Bloom Filter (Implemented)

When the current filter's FPP exceeds the threshold, a new filter with 2x the size is added. Queries check ALL filters.

public class ScalableBloomFilter<T> {
    private final List<BloomFilter<T>> filters;

    public void add(T element) {
        BloomFilter<T> current = filters.get(filters.size() - 1);
        if (current.getFalsePositiveProbability() > errorRate) {
            addNewFilter();  // 2x size of previous
            current = filters.get(filters.size() - 1);
        }
        current.add(element);
    }

    public boolean mightContain(T element) {
        for (BloomFilter<T> filter : filters) {
            if (filter.mightContain(element)) return true;
        }
        return false;
    }
}

07 — Key Algorithms

add(element)

public void add(T element) {
    int[] hashes = getHashes(element);
    for (int hash : hashes) {
        bitSet.set(Math.abs(hash % size));
    }
    elementCount++;
}

O(k) — compute k hashes, set k bits. No data stored, only bit positions.

mightContain(element)

public boolean mightContain(T element) {
    int[] hashes = getHashes(element);
    for (int hash : hashes) {
        if (!bitSet.get(Math.abs(hash % size))) {
            return false;  // Definitely not present
        }
    }
    return true;  // Might be present (possible false positive)
}

O(k) — short-circuits on first 0 bit. If all k bits are 1, returns true (possibly false positive).

getHashes(element) — Double Hashing

private int[] getHashes(T element) {
    int[] hashes = new int[numberOfHashFunctions];
    int hash1 = element.hashCode();
    int hash2 = murmurHash(element.toString());
    for (int i = 0; i < numberOfHashFunctions; i++) {
        hashes[i] = hash1 + i * hash2;
    }
    return hashes;
}

MurmurHash Implementation

public class MurmurHashFunction implements HashFunction {
    public int hash(Object element, int seed) {
        int hash = element.hashCode();
        hash = hash ^ (hash >>> 16);
        hash = hash * 0x85ebca6b;
        hash = hash ^ (hash >>> 13);
        hash = hash * 0xc2b2ae35 * (seed + 1);
        hash = hash ^ (hash >>> 16);
        return hash;
    }
}

Bit-mixing with magic constants for uniform distribution. Seed parameter generates independent hashes.

getCurrentFalsePositiveProbability()

public double getCurrentFalsePositiveProbability() {
    int bitsSet = bitSet.cardinality();
    double ratio = (double) bitsSet / size;
    return Math.pow(ratio, numberOfHashFunctions);
}

Empirical FPP based on actual fill ratio rather than theoretical formula.

08 — Thread Safety

MechanismWhereWhy
BitSet BloomFilter core Not thread-safe by default — wrap access in synchronized or use AtomicIntegerArray
synchronized add(), mightContain() Protect concurrent BitSet reads/writes for correctness
AtomicLong Element count Lock-free incrementing for statistics
ConcurrentHashMap Service filter registry Thread-safe map for managing multiple named filters
ReadWriteLock Production alternative Allow concurrent reads (mightContain) with exclusive writes (add)
"For read-heavy workloads, a ReadWriteLock is ideal — multiple threads can query simultaneously while writes acquire exclusive access. Alternatively, use AtomicIntegerArray instead of BitSet for lock-free CAS operations."

    int elementsAdded;
    int falsePositives;
    int trueNegatives;

12 — Interview Follow-up Q&A

Why use a Bloom Filter instead of a HashSet?
A HashSet storing 1M URLs uses ~50-100 MB (object headers, pointers, string data). A Bloom Filter uses ~1.2 MB for 1% FPP. When false positives are acceptable and memory is constrained (e.g., in-memory caching, mobile devices, network filters), the Bloom Filter wins. Trade-off: you lose the ability to enumerate elements or delete them.
How do you handle deletion in a Bloom Filter?
Standard Bloom Filters don't support deletion — clearing a bit could affect other elements that hash to the same position. Use a Counting Bloom Filter (counters instead of bits, increment on add, decrement on delete). Trade-off: 4x memory. Alternatively, rebuild the filter periodically from the source of truth.
What happens when the filter gets full?
The FPP rises as more bits get set. When fill ratio exceeds ~50%, FPP degrades rapidly. Solutions: (1) ScalableBloomFilter — add a new, larger filter when FPP exceeds threshold; queries check all filters. (2) Rebuild with a larger size from scratch using the source data.
Why double hashing instead of k independent hash functions?
Kirsch & Mitzenmacher (2006) proved that h_i = h1 + i*h2 is as effective as k fully independent hash functions for Bloom filters. This reduces hash computation from O(k) independent functions to just 2 hash computations + k additions, which is faster with the same theoretical guarantees.
Why MurmurHash instead of SHA-256 or MD5?
MurmurHash is a non-cryptographic hash optimized for speed with excellent distribution. SHA-256/MD5 are cryptographic hashes designed for collision resistance, which is unnecessary here — and they're 10-100x slower. MurmurHash provides uniform distribution (minimizes false positives) with minimal CPU cost.
How would you make this distributed?
Two approaches: (1) Partitioned — each node owns a shard of the bit array; queries are scatter-gather. (2) Replicated — each node has a full copy; writes broadcast to all replicas. For write-heavy workloads, partition. For read-heavy, replicate. Redis has built-in Bloom Filter support via the BF.* commands (RedisBloom module).
How does the ScalableBloomFilter maintain FPP guarantees?
Each new filter has 2x the size of the previous one and a tighter error rate. The overall FPP is bounded by a geometric series. Queries check all filters (OR semantics) — if ANY filter says "yes", the answer is "maybe present". This means FPP grows slightly with each added filter, but the exponentially increasing sizes keep it bounded.
Can Bloom Filters give false negatives?
Never. If an element was added, all k bit positions were set to 1. They can never be unset (no deletion). So when querying, all k positions will be 1, and the filter will always return true for elements that were added. This is the fundamental invariant of a Bloom Filter.
What's the optimal fill ratio for a Bloom Filter?
The optimal fill ratio is 50% (ln(2) ≈ 69.3% of bits set when using optimal k). At this point, FPP is minimized for the given parameters. Above 50% fill ratio, FPP increases rapidly. Below 50%, you're using more memory than necessary. The optimal k formula ensures you hit this sweet spot.
How would you persist a Bloom Filter?
BitSet implements Serializable, so you can serialize the entire filter. For production: (1) Serialize the bit array as a byte array. (2) Store size, k, and element count as metadata. (3) Use memory-mapped files (MappedByteBuffer) for large filters to avoid loading into heap. (4) Redis BF module persists automatically.
How does this design follow SOLID principles?
S: BloomFilter handles bits, Config handles sizing, Stats handles monitoring. O: New hash functions via HashFunction interface, no modification to core. L: ScalableBloomFilter can replace BloomFilter (same interface). I: BloomFilterService is minimal — add, contains, clear, getFPP. D: Core depends on HashFunction abstraction, not MurmurHash directly.
When would you NOT use a Bloom Filter?
When you need: (1) 100% accuracy (medical records, financial transactions). (2) Element retrieval — BF doesn't store the actual elements. (3) Frequent deletion — use a Counting BF or Cuckoo Filter instead. (4) Small datasets — a HashSet is simpler and the memory savings are negligible.
