Version Control System // Git

Low-level design covering Repository, Branch, Commit object model with DAG-based history, branching/merging, diff computation, and staging area. 10 files, clean layered architecture.

Repository Pattern DAG Commit Graph Content-Addressable Strategy Pattern ConcurrentHashMap

01 — Requirements

Functional

  • Create and manage repositories with a default main branch
  • Commit snapshots — message, author, timestamp, parent pointer, file map
  • Branch creation from any commit — lightweight pointer to HEAD commit
  • Checkout / switch between branches
  • Commit history traversal — walk parent chain from HEAD
  • Diff computation between any two commits (added / modified / deleted)
  • Merge branches — three-way merge with conflict detection
  • Staging area to selectively add files before committing

Non-Functional

  • Thread-safe — multiple users committing to different repos concurrently
  • Extensible — pluggable merge strategies, diff algorithms
  • Performance — O(1) commit lookup, O(k) history walk (k = chain length)
  • Immutable commits — once created, a commit’s data never changes
  • Content-addressable ready — UUID or SHA-based commit IDs

Commit — Immutable Snapshot

public class Commit {
    private final String id;           // UUID or SHA hash
    private final String message;
    private final String author;
    private final LocalDateTime timestamp;
    private final String parentId;      // null for root commit
    private final Map<String, String> files; // tree: filename → content

    public Commit(String id, String message, String author,
                  String parentId, Map<String, String> files) {
        this.id = id;
        this.message = message;
        this.author = author;
        this.timestamp = LocalDateTime.now();
        this.parentId = parentId;
        this.files = new HashMap<>(files); // defensive copy
    }

    public Map<String, String> getFiles() {
        return new HashMap<>(files); // defensive copy on read
    }
}

Defensive copies on both construction and access — ensures immutability even when callers hold a reference to the original map.

Branch — Mutable HEAD Pointer

public class Branch {
    private final String name;         // immutable
    private String headCommitId;       // moves on every commit

    public void updateHead(String commitId) {
        this.headCommitId = commitId;  // fast-forward
    }
}

Repository — Aggregate Root

public class Repository {
    private final String name;
    private final Map<String, Branch> branches;
    private String currentBranch;      // "main" by default

    public Repository(String name) {
        this.name = name;
        this.branches = new HashMap<>();
        this.currentBranch = "main";
        branches.put("main", new Branch("main", null));
    }

    public void createBranch(String name, String fromCommitId) {
        branches.put(name, new Branch(name, fromCommitId));
    }

    public void switchBranch(String name) {
        if (branches.containsKey(name))
            this.currentBranch = name;
    }
}

04 — Design Patterns

Clean separation of concerns across four key patterns.

Repository Pattern

VersionControl interface abstracts storage. VersionControlImpl uses ConcurrentHashMap. Swap for a DB-backed impl without changing callers.

Strategy Pattern

Merge strategy and diff algorithm can be injected. ThreeWayMerge vs FastForwardMerge. Myers diff vs Patience diff. Swappable at runtime.

Immutable Value Objects

Commit is immutable after creation — all fields are final. Defensive copies on Map<String,String> files ensure no external mutation. Thread-safe by design.

Aggregate Root

Repository owns its branches. All branch operations (create, switch, get current) go through Repository. External code never directly mutates a Branch’s membership.

05 — Commit Graph (DAG)

Why a DAG? Commits form a Directed Acyclic Graph via parent pointers. Each commit points to its parent (or parents, for merge commits). Branches are just named pointers into this graph. This enables O(1) branching — creating a branch is just storing a pointer. 
gitGraph
    commit id: "init"
    commit id: "c1: Add README"
    commit id: "c2: Add feature"
    branch feature-branch
    checkout feature-branch
    commit id: "c3: Work on feature"
    commit id: "c4: Fix bug"
    checkout main
    commit id: "c5: Hotfix"
    merge feature-branch id: "c6: Merge feature"

How commit() Works

public String commit(String repoName, String message, String author) {
    Repository repo = repositories.get(repoName);
    if (repo == null) return null;

    Branch currentBranch = repo.getCurrentBranch();
    String commitId = UUID.randomUUID().toString().substring(0, 8);

    Commit commit = new Commit(
        commitId, message, author,
        currentBranch.getHeadCommitId(), // parent = current HEAD
        new HashMap<>()                  // staged files snapshot
    );

    commits.get(repoName).put(commitId, commit);
    currentBranch.updateHead(commitId);  // advance HEAD
    return commitId;
}

Key: parent pointer is set to the current HEAD before advancing. This forms the linked chain of the DAG.

History Traversal

// Walk the parent chain from HEAD to root
public List<Commit> getHistory(String repoName) {
    List<Commit> history = new ArrayList<>();
    String commitId = repo.getCurrentBranch().getHeadCommitId();

    while (commitId != null) {
        Commit c = commits.get(repoName).get(commitId);
        history.add(c);
        commitId = c.getParentId();  // follow parent pointer
    }
    return history; // newest first
}

Time: O(k) where k = number of commits in chain. Space: O(k) for the result list.

06 — Branching & Merging

Branch Creation — O(1)

A branch is just a named pointer to a commit. Creating one is a single HashMap put:

public void createBranch(String repoName, String branchName) {
    Repository repo = repositories.get(repoName);
    String headCommit = repo.getCurrentBranch().getHeadCommitId();
    repo.createBranch(branchName, headCommit); // pointer copy
}

Branch Switch

public void switchBranch(String repoName, String branchName) {
    Repository repo = repositories.get(repoName);
    repo.switchBranch(branchName); // just changes currentBranch string
}

In real Git, checkout also updates the working directory. In our LLD, working tree state is the staged files map.

07 — Staging Area (Index)

What is it? The staging area (index) sits between the working directory and the repository. It holds the next commit’s snapshot. git add copies working dir → staging. git commit snapshots staging → commit.

Staging Area Implementation

public class StagingArea {
    private final Map<String, String> stagedFiles = new HashMap<>();

    public void add(String filename, String content) {
        stagedFiles.put(filename, content);
    }

    public void remove(String filename) {
        stagedFiles.remove(filename);
    }

    public Map<String, String> snapshot() {
        Map<String, String> snap = new HashMap<>(stagedFiles);
        stagedFiles.clear();  // reset after commit
        return snap;
    }

    public boolean isEmpty() { return stagedFiles.isEmpty(); }
}

On commit, snapshot() drains the staging area into the new Commit's files map, then clears itself. This mirrors git commit semantics.

08 — Diff Computation

Core operation: Compare two commit snapshots (file maps) to produce added, modified, and deleted file lists. Enables git diff, git log -p, and merge conflict detection.

File-Level Diff Algorithm

public class DiffResult {
    private final List<String> added;
    private final List<String> modified;
    private final List<String> deleted;
}

public static DiffResult diff(Commit oldCommit, Commit newCommit) {
    Map<String, String> oldFiles = oldCommit.getFiles();
    Map<String, String> newFiles = newCommit.getFiles();

    List<String> added = new ArrayList<>();
    List<String> modified = new ArrayList<>();
    List<String> deleted = new ArrayList<>();

    // Files in new but not old → ADDED
    // Files in both but content differs → MODIFIED
    for (String file : newFiles.keySet()) {
        if (!oldFiles.containsKey(file))
            added.add(file);
        else if (!oldFiles.get(file).equals(newFiles.get(file)))
            modified.add(file);
    }

    // Files in old but not new → DELETED
    for (String file : oldFiles.keySet()) {
        if (!newFiles.containsKey(file))
            deleted.add(file);
    }

    return new DiffResult(added, modified, deleted);
}

Time: O(n + m) where n, m = number of files in each commit. Uses HashMap lookups (O(1) each).

Line-Level Diff (Myers Algorithm)

For content-level diffs (like git diff output), use the Myers diff algorithm:

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap repositories, commits stores Safe concurrent reads/writes across repos
ConcurrentHashMap (nested) commits per repo Each repo has its own concurrent commit store
synchronized(repo) commit, createBranch, switchBranch Serialize operations within a single repository
Immutable Commit Commit objects All fields final + defensive copies — thread-safe by design
Defensive copies Commit.getFiles(), Repository.getBranches() Callers get a copy, can't corrupt internal state
"Two users committing to different repos proceed in parallel — no contention. Operations on the same repo are serialized at the repo level. Commits are immutable, so any thread can read commit history without locks."

13 — Key Algorithms

LCA (Lowest Common Ancestor) for Merge Base

Finding the merge base of two branches requires finding the LCA in the commit DAG.

public Commit findMergeBase(String commitIdA, String commitIdB) {
    Set<String> ancestorsA = new HashSet<>();

    // Collect all ancestors of A
    String current = commitIdA;
    while (current != null) {
        ancestorsA.add(current);
        current = commits.get(current).getParentId();
    }

    // Walk B's ancestors, first match is the LCA
    current = commitIdB;
    while (current != null) {
        if (ancestorsA.contains(current))
            return commits.get(current);
        current = commits.get(current).getParentId();
    }
    return null; // disjoint histories
}

Time: O(n + m) where n, m = depth of each branch. Uses HashSet for O(1) ancestor lookups.

Content-Addressable Storage (SHA Hashing)

public static String computeHash(String content) {
    MessageDigest md = MessageDigest.getInstance("SHA-1");
    byte[] hash = md.digest(content.getBytes(StandardCharsets.UTF_8));
    StringBuilder hex = new StringBuilder();
    for (byte b : hash)
        hex.append(String.format("%02x", b));
    return hex.toString();
}

In production Git, every blob/tree/commit is addressed by its SHA-1. Identical content → identical hash → automatic deduplication. Our LLD uses UUID for simplicity but the pattern is the same.

Fast-Forward Detection

public boolean isAncestor(String potentialAncestor, String commitId) {
    String current = commitId;
    while (current != null) {
        if (current.equals(potentialAncestor)) return true;
        current = commits.get(current).getParentId();
    }
    return false;
}

// If target HEAD is ancestor of source HEAD → fast-forward (just move pointer)
// Otherwise → three-way merge required

14 — Interview Follow-up Q&A

How does Git store files internally?
Git uses a content-addressable filesystem with three object types: Blob (raw file content), Tree (directory listing mapping names to blob/tree hashes), and Commit (points to root tree + parent commits + metadata). Every object is identified by its SHA-1 hash. In our LLD, we simplify with Commit.files: Map<String, String> which merges blob+tree into a single map, and use UUID instead of SHA.
Why is branching O(1) in Git?
A branch is just a 41-byte file containing a commit SHA. Creating a branch = writing one file. In our LLD, it's new Branch(name, headCommitId) — a single HashMap put. No file copying, no snapshot duplication. The commit DAG is shared between all branches. This is fundamentally different from SVN where branching copies the entire directory tree.
How would you handle merge conflicts?
Three-way merge: (1) Find the merge base (LCA of two branch heads). (2) Diff base vs source and base vs target. (3) If both sides changed the same file differently → CONFLICT. Return a MergeResult with conflictedFiles list. The caller resolves conflicts and re-commits. In our LLD, this is a MergeStrategy interface — Strategy pattern.
How would you implement git log efficiently?
Walk the parent chain from HEAD using a priority queue (ordered by timestamp) for multi-parent merge commits. For single-parent chains, it's a simple linked-list traversal: O(k). For filtered logs (by author, date range), apply predicates during traversal. Our getHistory() does exactly this.
What is the staging area and why does it exist?
The staging area (index) is an intermediate layer between working directory and repository. It lets you selectively commit — add only some changed files. Without it, every commit would include all modifications. In our LLD, StagingArea holds a Map<String,String> that gets drained into the Commit on snapshot().
How would you scale this to millions of files?
Three key optimizations: (1) Content-addressable storage with SHA hashing — identical files across commits share one blob (deduplication). (2) Pack files — delta-compress similar blobs. (3) Tree objects with hierarchical hashing — unchanged subtrees share the same tree hash, so commits only store diffs. In our LLD, you'd replace the flat Map<String,String> with a proper Tree object hierarchy.
How does concurrent access work?
ConcurrentHashMap for the repo/commit stores allows parallel access to different repos. Operations on the same repo can be serialized with synchronized(repo). Commits are immutable (all fields final + defensive copies), so reads never need locks. Branch HEAD updates are the only mutable operation — serialized per-repo.
How would you implement git rebase?
Rebase replays commits from one branch onto another: (1) Find the fork point (merge base). (2) Collect commits from fork to source HEAD. (3) For each commit, create a new commit with the same diff but a different parent (on top of target HEAD). (4) Update the source branch pointer. The original commits remain in the DAG (garbage collected later). This is why rebased commits get new IDs.
Why are commits immutable?
(1) Thread safety — any thread can read any commit without locks. (2) Integrity — SHA-based IDs depend on content; changing content changes the hash. (3) History preservation — you can always trace back to any point. (4) Sharing — multiple branches can reference the same commit safely. In our LLD, all Commit fields are final and Maps use defensive copies.
How would you implement distributed VCS (push/pull)?
Each clone has a full copy of the commit DAG. Push: send missing commits (those in local but not remote) — compare HEAD pointers, walk the DAG to find divergence point, transfer delta. Pull: fetch missing commits from remote + merge. The content-addressable design makes this efficient — objects with the same SHA are identical, skip transfer. Add a RemoteRepository interface with fetch/push methods.
What's the difference between merge and rebase?
Merge creates a new commit with two parents, preserving the branch topology. History is non-linear but accurate. Rebase replays commits linearly on top of the target, creating new commit IDs. History is cleaner but rewrites it. In our LLD, merge would create a Commit with a List<String> parentIds instead of a single parentId.
How would you implement git stash?
Stash = save current staging area and working directory changes to a special stack of commits not attached to any branch. Implement as a Deque<Map<String,String>> on the Repository. stash() pushes current staged files onto the stack and clears the staging area. stashPop() restores them. The stash commits can reference the HEAD at stash time for context.
