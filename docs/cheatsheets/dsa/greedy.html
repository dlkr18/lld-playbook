Greedy Algorithms // Cheat Sheet

Make the locally optimal choice at each step, hoping it leads to a global optimum. Works when a problem has optimal substructure and the greedy choice property.

Greedy Choice Property Optimal Substructure No Backtracking Sort First

01 — Pattern Guide

A greedy algorithm works when both conditions hold:

  1. Greedy Choice Property: A locally optimal choice at each step leads to a globally optimal solution. You never need to reconsider past choices.
  2. Optimal Substructure: An optimal solution to the problem contains optimal solutions to its sub-problems.

How to Spot Greedy Problems

  • "Minimum number of ___" or "Maximum number of ___"
  • Scheduling, interval, or assignment problems
  • Problems where sorting reveals the greedy order
  • Problems where a simple invariant (max-so-far, running sum) suffices
Greedy recipe: (1) Sort or structure the input. (2) Iterate once, making the locally best choice. (3) Never revisit a decision.

02 — Jump Game (LC 55)

Problem: Given nums[i] = max jump length from index i, can you reach the last index?

Key Insight

Track maxReach — the farthest index reachable so far. If at any point i > maxReach, you’re stuck.

class Solution {
public:
    bool canJump(vector<int>& nums) {
        int maxReach = 0;
        for (int i = 0; i < nums.size(); i++) {
            if (i > maxReach) return false;
            maxReach = max(maxReach, i + nums[i]);
        }
        return true;
    }
};

Time: O(n) · Space: O(1)

03 — Jump Game II (LC 45)

Problem: Minimum number of jumps to reach the last index (guaranteed reachable).

Key Insight

Think of it as BFS by levels. Each "level" is the range of indices reachable with the current number of jumps. Expand the frontier greedily.

class Solution {
public:
    int jump(vector<int>& nums) {
        int jumps = 0, curEnd = 0, farthest = 0;
        for (int i = 0; i < (int)nums.size() - 1; i++) {
            farthest = max(farthest, i + nums[i]);
            if (i == curEnd) {       // exhausted current level
                jumps++;
                curEnd = farthest;
                if (curEnd >= (int)nums.size() - 1) break;
            }
        }
        return jumps;
    }
};
BFS analogy: curEnd = end of current BFS layer. farthest = end of next layer. Each layer transition = one jump.

Time: O(n) · Space: O(1)

04 — Gas Station (LC 134)

Problem: N gas stations in a circle. gas[i] fuel at station i, cost[i] to travel to i+1. Find the starting station to complete the circuit, or -1.

Key Insight

Two greedy observations: (1) If totalGas < totalCost, no solution exists. (2) If running sum drops below 0 at station i, reset start to i+1.

class Solution {
public:
    int canCompleteCircuit(vector<int>& gas, vector<int>& cost) {
        int totalSurplus = 0, curSurplus = 0, start = 0;
        for (int i = 0; i < gas.size(); i++) {
            int diff = gas[i] - cost[i];
            totalSurplus += diff;
            curSurplus += diff;
            if (curSurplus < 0) {    // can't reach i+1 from start
                start = i + 1;       // reset: try starting after i
                curSurplus = 0;
            }
        }
        return totalSurplus >= 0 ? start : -1;
    }
};
Why reset works: If the prefix sum [start..i] is negative, every sub-prefix starting within [start..i] is also negative (because we already eliminated earlier bad prefixes). So the first valid start must be after i.

Time: O(n) · Space: O(1)

05 — Task Scheduler (LC 621)

Problem: Given tasks with cooldown n between identical tasks, find minimum intervals to finish all tasks.

Key Insight

The most frequent task dictates the schedule. Build a frame of (maxFreq - 1) chunks of size (n + 1), then fill the last partial chunk.

class Solution {
public:
    int leastInterval(vector<char>& tasks, int n) {
        vector<int> freq(26, 0);
        for (char t : tasks) freq[t - 'A']++;

        int maxFreq = *max_element(freq.begin(), freq.end());
        // count how many tasks share the max frequency
        int maxCount = 0;
        for (int f : freq) if (f == maxFreq) maxCount++;

        // (maxFreq-1) full chunks of (n+1) slots + last partial chunk
        int slots = (maxFreq - 1) * (n + 1) + maxCount;
        return max(slots, (int)tasks.size());
    }
};
Visual: tasks = [A,A,A,B,B,C], n=2 → A _ _ | A _ _ | A → fill B,C into blanks → A B C A B _ A → answer = max(frame, total tasks).

Time: O(n) · Space: O(1) (26-size array)

06 — Partition Labels (LC 763)

Problem: Partition a string so each letter appears in at most one part. Maximize the number of parts.

Key Insight

Record the last occurrence of each character. Greedily extend the current partition’s end to the farthest last-occurrence of any character seen so far.

class Solution {
public:
    vector<int> partitionLabels(string s) {
        vector<int> last(26, 0);
        for (int i = 0; i < s.size(); i++)
            last[s[i] - 'a'] = i;

        vector<int> result;
        int start = 0, end = 0;
        for (int i = 0; i < s.size(); i++) {
            end = max(end, last[s[i] - 'a']);
            if (i == end) {          // all chars in [start..end] don't appear later
                result.push_back(end - start + 1);
                start = i + 1;
            }
        }
        return result;
    }
};

Time: O(n) · Space: O(1) (26-size array)

07 — Hand of Straights (LC 846)

Problem: Given hand of cards, can you rearrange into groups of groupSize consecutive cards?

Key Insight

Sort the cards. Always start a group from the smallest remaining card (greedy). Use a frequency map and consume cards greedily.

class Solution {
public:
    bool isNStraightHand(vector<int>& hand, int groupSize) {
        if (hand.size() % groupSize != 0) return false;

        map<int, int> freq;           // sorted by key
        for (int c : hand) freq[c]++;

        while (!freq.empty()) {
            int start = freq.begin()->first;  // smallest card
            for (int i = start; i < start + groupSize; i++) {
                if (freq.find(i) == freq.end()) return false;
                freq[i]--;
                if (freq[i] == 0) freq.erase(i);
            }
        }
        return true;
    }
};

Time: O(n log n) · Space: O(n)

08 — Assign Cookies (LC 455)

Problem: Children have greed factors g[i], cookies have sizes s[j]. Assign cookies to maximize content children (cookie size ≥ greed).

Key Insight

Sort both arrays. Match the smallest cookie that satisfies the least greedy child first — this wastes the least resources.

class Solution {
public:
    int findContentChildren(vector<int>& g, vector<int>& s) {
        sort(g.begin(), g.end());
        sort(s.begin(), s.end());

        int child = 0, cookie = 0;
        while (child < g.size() && cookie < s.size()) {
            if (s[cookie] >= g[child])
                child++;                // child satisfied
            cookie++;                   // try next cookie either way
        }
        return child;
    }
};

Time: O(n log n) · Space: O(1)

09 — Activity Selection / Non-overlapping Intervals (LC 435)

Problem: Given intervals, find minimum removals so remaining intervals don’t overlap. (Equivalent: maximum non-overlapping intervals.)

Key Insight

Sort by end time. Greedily pick the interval that ends earliest — this leaves the most room for future intervals.

class Solution {
public:
    int eraseOverlapIntervals(vector<vector<int>>& intervals) {
        sort(intervals.begin(), intervals.end(),
            [](const vector<int>& a, const vector<int>& b) {
                return a[1] < b[1];
            });

        int kept = 1, prevEnd = intervals[0][1];
        for (int i = 1; i < intervals.size(); i++) {
            if (intervals[i][0] >= prevEnd) {  // no overlap
                kept++;
                prevEnd = intervals[i][1];
            }
            // else: skip (remove) this interval
        }
        return intervals.size() - kept;
    }
};
Why sort by end? Choosing the earliest-ending interval is provably optimal — it maximizes remaining schedule space. Sorting by start doesn’t work (counter-example: [1,100], [2,3], [4,5]).

Time: O(n log n) · Space: O(1)

11 — When Greedy Fails

Greedy fails when the locally optimal choice doesn’t lead to the global optimum. Use DP instead.

Classic Counter-Examples

ProblemWhy Greedy FailsUse Instead
0/1 Knapsack Taking highest value/weight ratio doesn’t account for item combinations. E.g. items (60,10), (100,20), (120,30) with W=50: greedy takes first two (160), DP finds (100+120=220 doesn’t fit) but (60+120=180) via ratio is also wrong. DP: O(n·W)
Coin Change Greedy (largest coin first) fails for non-canonical systems. E.g. coins [1,3,4], target 6: greedy picks 4+1+1=3 coins, optimal is 3+3=2 coins. DP: O(n·amount)
Longest Increasing Subsequence Greedily picking the next larger element doesn’t guarantee the longest chain. Need to track all possible subsequence endings. DP + Binary Search: O(n log n)
Edit Distance Locally choosing insert/delete/replace doesn’t minimize total operations. DP: O(m·n)
Matrix Chain Multiplication No clear greedy ordering for parenthesization. DP: O(n³)
Rule of thumb: If you can construct a small counter-example where the greedy choice is suboptimal, the problem needs DP. If you can’t find one, greedy likely works — but prove it via exchange argument or induction.
