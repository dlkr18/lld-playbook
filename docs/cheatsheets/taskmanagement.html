01 — Requirements

Functional

  • CRUD — Create, read, update, delete tasks with title & description
  • Task lifecycle — TODO → IN_PROGRESS → IN_REVIEW → DONE (or BLOCKED)
  • Priority levels — LOW, MEDIUM, HIGH, CRITICAL
  • Assignment — Assign tasks to users by assigneeId
  • Tagging — Add multiple tags per task for categorization
  • Due dates — Set deadlines, detect overdue tasks
  • Search & filter — By status, assignee, priority, tag, date range
  • Comments — Add/edit comments on tasks with author tracking
  • Team management — Create teams with members and managers
  • Notifications — Observers fire on every status change

Non-Functional

  • Thread-safe — ConcurrentHashMap + synchronized observer list
  • Extensible — Interface-driven service layer (swap in-memory for DB)
  • Performance — O(1) CRUD via HashMap, O(n) filter via streams
  • Scalable — 10,000+ concurrent users, response < 100ms
  • RBAC-ready — UserRole enum: ADMIN, MANAGER, MEMBER, VIEWER

03 — Design Patterns

Four patterns, each earning its place in the design.

Observer Pattern

TaskObserver interface fires onTaskStatusChanged(task, old, new) on every transition. Decouples notification logic from the board and service.

Repository Pattern

TaskService interface abstracts all data access. InMemoryTaskService implements it with ConcurrentHashMap. Swap for JPA/DB with zero changes upstream.

Stream-Based Filtering

All query methods use Java Streams with filter/collect. Composable predicates for status, assignee, priority, tag, date range, and overdue detection.

Interface Segregation

TaskService defines 15 methods grouped by concern: CRUD, assignment, status/priority, querying. Clean contract for any implementation.

04 — Task Lifecycle

Why 5 states? Covers real-world workflows: backlog (TODO), active work (IN_PROGRESS), code review (IN_REVIEW), dependency wait (BLOCKED), and closure (DONE). Each transition triggers observers. 
stateDiagram-v2
    [*] --> TODO
    TODO --> IN_PROGRESS: start work
    TODO --> BLOCKED: dependency found
    IN_PROGRESS --> IN_REVIEW: submit for review
    IN_PROGRESS --> BLOCKED: dependency found
    IN_PROGRESS --> DONE: fast-track complete
    IN_REVIEW --> IN_PROGRESS: changes requested
    IN_REVIEW --> DONE: approved
    BLOCKED --> TODO: unblocked
    BLOCKED --> IN_PROGRESS: unblocked + resume
    DONE --> [*]

Status Transition in Task.java

public void setStatus(TaskStatus status) {
    this.status = status;
    if (status == TaskStatus.DONE) {
        this.completedAt = LocalDateTime.now();  // auto-stamp completion
    }
}

The completedAt timestamp is auto-set when status transitions to DONE. Enables duration metrics (createdAt → completedAt).

TaskBoard Delegates & Notifies

public void updateTaskStatus(String taskId, TaskStatus newStatus) {
    Task task = tasks.get(taskId);
    if (task != null) {
        TaskStatus oldStatus = task.getStatus();
        task.setStatus(newStatus);
        notifyObservers(task, oldStatus, newStatus); // fires all observers
    }
}

Team Model

public class Team {
    private final String id;
    private String name, description;
    private Set<String> memberIds;   // HashSet -- O(1) add/remove/contains
    private String managerId;        // must be a member to be set as manager

    public void setManagerId(String managerId) {
        if (memberIds.contains(managerId)) {  // guard: must be member first
            this.managerId = managerId;
        }
    }

    public void removeMember(String userId) {
        memberIds.remove(userId);
        if (userId.equals(managerId)) {
            managerId = null;  // auto-clear manager if removed from team
        }
    }
}

07 — Observer Pattern

Why Observer? Decouples "what happened" from "who cares." Adding email, Slack, or audit-log notifications = new observer class, zero changes to TaskBoard or InMemoryTaskService.

TaskObserver Interface

public interface TaskObserver {
    void onTaskStatusChanged(Task task, TaskStatus oldStatus, TaskStatus newStatus);
}

Receives both old and new status — observers can react to specific transitions (e.g., only fire when entering DONE).

Observer Registration & Dispatch

// Two levels: TaskBoard (lightweight) and InMemoryTaskService (full)

// In TaskBoard.java:
private final List<TaskObserver> observers = new ArrayList<>();

public void addObserver(TaskObserver observer) {
    observers.add(observer);
}

private void notifyObservers(Task task, TaskStatus old, TaskStatus next) {
    for (TaskObserver obs : observers) {
        obs.onTaskStatusChanged(task, old, next);
    }
}

// In InMemoryTaskService.java (thread-safe copy):
private void notifyObservers(Task task, TaskStatus old, TaskStatus next) {
    List<TaskObserver> copy;
    synchronized (observers) {
        copy = new ArrayList<>(observers);  // snapshot under lock
    }
    for (TaskObserver obs : copy) {
        obs.onTaskStatusChanged(task, old, next);
    }
}
EventObserver Use Cases
TODO → IN_PROGRESSNotify manager, update sprint burndown
IN_PROGRESS → IN_REVIEWNotify reviewer, trigger CI pipeline
* → BLOCKEDAlert team lead, flag on dashboard
* → DONENotify reporter, update velocity metrics
Task created (null → TODO)Notify board subscribers, log audit trail

Overdue Detection

public List<Task> getOverdueTasks() {
    LocalDateTime now = LocalDateTime.now();
    return tasks.values().stream()
        .filter(t -> t.getDueDate() != null
                  && t.getDueDate().isBefore(now))
        .filter(t -> t.getStatus() != TaskStatus.DONE)
        .collect(Collectors.toList());
}

Two-predicate chain: has a due date in the past AND not yet completed. Null-safe — tasks without due dates are excluded.

Date Range Query

public List<Task> getTasksCreatedBetween(LocalDateTime start, LocalDateTime end) {
    return tasks.values().stream()
        .filter(t -> !t.getCreatedAt().isBefore(start)
                  && !t.getCreatedAt().isAfter(end))
        .collect(Collectors.toList());
}

Inclusive range: start ≤ createdAt ≤ end. Uses negated isBefore/isAfter for inclusive bounds.

Tag-Based Search

// getTags() returns a defensive copy: new ArrayList<>(tags)
public List<Task> getTasksByTag(String tag) {
    return tasks.values().stream()
        .filter(t -> t.getTags().contains(tag))
        .collect(Collectors.toList());
}

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap InMemoryTaskService.tasks Lock-free reads, segment-level writes — safe concurrent CRUD
synchronized(observers) addObserver(), notifyObservers() Protects observer list mutation; snapshot-copy before iteration
AtomicLong idGenerator Lock-free monotonic ID generation: "TASK-1", "TASK-2", ...
Defensive copies Task.getTags(), Team.getMemberIds() Return copies to prevent external mutation of internal collections
"ConcurrentHashMap for the data store gives us safe concurrent reads without blocking. Observer notification uses a snapshot copy under a synchronized block — observers can be added while notification is in progress."

13 — Interview Follow-up Q&A

Why ConcurrentHashMap instead of synchronized HashMap?
ConcurrentHashMap uses segment-level locking (Java 8+: CAS + synchronized on individual bins). Multiple threads can read concurrently and write to different segments simultaneously. A synchronized HashMap locks the entire map on every operation — single-threaded throughput for all callers.
How would you add task dependencies (blocks/blocked-by)?
Add a Set<String> blockedByTaskIds to Task. On updateTaskStatus(), check if all dependencies are DONE before allowing transition out of BLOCKED. Use a topological sort for scheduling. The observer can auto-unblock tasks when a dependency completes.
How would you implement task search at scale (100K+ tasks)?
The O(n) stream filters work for <10K tasks. At scale: (1) Secondary indexes — Map<TaskStatus, Set<String>> for O(1) status lookup. (2) Inverted index for tags — Map<String, Set<String>>. (3) Full-text search with Elasticsearch. The TaskService interface stays the same — just swap the implementation.
Why Observer instead of direct notification calls?
(1) Open/Closed: add new observers (Slack, email, metrics) without modifying TaskBoard or Service. (2) SRP: notification logic lives in observer classes, not in the core service. (3) Testability: mock observers for unit tests. (4) Both TaskBoard and InMemoryTaskService support observers independently.
How would you handle concurrent updates to the same task?
Add synchronized(task) or optimistic locking with a version field. On update: check task.version == expectedVersion, else throw StaleDataException. ConcurrentHashMap handles map-level safety, but field-level mutations on the same Task object need additional protection.
Why String-based assigneeId instead of a User reference?
Loose coupling — Task doesn't depend on User's class or lifecycle. Works with any user system (LDAP, OAuth, microservice). In production, validate the assigneeId exists via a UserService call. The String ID is serialization-friendly (JSON, DB, message queues).
How would you add subtasks?
Add String parentTaskId and List<String> subtaskIds to Task. This creates a Composite pattern. A parent is DONE only when all subtasks are DONE. The observer on subtask completion can check and auto-complete the parent.
How would you make the Comment system thread-safe?
Store comments in a ConcurrentHashMap<String, Comment> keyed by comment ID. For comment editing, the setContent() method auto-updates updatedAt. The isEdited() check compares createdAt != updatedAt. At scale, use a CopyOnWriteArrayList for the comment list per task if reads vastly outnumber writes.
How does Team.setManagerId() ensure data integrity?
Guard clause: if (memberIds.contains(managerId)) — a manager must be a member first. removeMember() auto-clears managerId if the manager is removed. This prevents orphaned manager references without needing a separate validation layer.
How would you persist this to a database?
The TaskService interface stays identical. Create JpaTaskService implements TaskService with JPA/Hibernate. Task maps to a tasks table, tags to a task_tags join table. The stream-based filters become JPQL/SQL queries. Spring's @Transactional replaces the ConcurrentHashMap safety guarantees.
Why defensive copies in getTags() and getMemberIds()?
return new ArrayList<>(tags) prevents callers from modifying the internal list. Without it, a caller could do task.getTags().clear() and destroy the task's tag data. This is encapsulation 101 — especially important in concurrent environments.
How would you add real-time updates (WebSocket)?
The Observer pattern is already in place. Add a WebSocketTaskObserver implements TaskObserver that pushes updates over WebSocket sessions. On onTaskStatusChanged(), serialize the event and broadcast to connected clients. The service code doesn't change at all.
