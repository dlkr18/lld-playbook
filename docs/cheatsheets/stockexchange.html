01 — Requirements

Functional

  • Investors place BUY and SELL orders for listed stocks
  • Support market, limit, and stop order types
  • Order book maintains buy side (bids) and sell side (asks) per symbol
  • Price-time priority matching — best price first, then earliest timestamp
  • Partial fills — large orders can match against multiple smaller orders
  • Order cancellation for open orders
  • Trade execution with trade record generation
  • Best bid/ask price queries per symbol
  • Portfolio and market data management

Non-Functional

  • Thread-safe — multiple traders placing orders concurrently
  • Low latency — response time < 100ms for order operations
  • Scalable — support 10,000+ concurrent users
  • Consistent — no duplicate fills, no phantom orders
  • Reliable — 99.9% uptime, ACID where needed

03 — Design Patterns

Four patterns, each solving a specific exchange concern.

Strategy Pattern

Matching algorithms are swappable. Price-time priority is the default, but FIFO, pro-rata, or auction-based matching can be plugged in without changing the engine.

Observer Pattern

Market data updates, trade notifications, and order status changes fire events. Listeners subscribe to symbols for real-time price feeds.

Repository Pattern

Abstracts data access from business logic. In-memory ConcurrentHashMap stores, easily swappable for database-backed implementations.

Factory Pattern

Order creation with auto-generated IDs, validation, and default status. Centralizes the "how" of order instantiation.

04 — Order Book (Buy/Sell Sides)

Core data structure of every exchange. Each stock symbol has its own OrderBook with two sides: buy orders sorted by highest price first (max-heap), sell orders sorted by lowest price first (min-heap).

OrderBook Structure

public class OrderBook {
    private final String stockSymbol;
    private final PriorityQueue<Order> buyOrders;   // Max heap by price
    private final PriorityQueue<Order> sellOrders;  // Min heap by price

    public OrderBook(String stockSymbol) {
        this.stockSymbol = stockSymbol;
        this.buyOrders = new PriorityQueue<>(
            (a, b) -> Double.compare(b.getPrice(), a.getPrice()));
        this.sellOrders = new PriorityQueue<>(
            (a, b) -> Double.compare(a.getPrice(), b.getPrice()));
    }

    public void addOrder(Order order) {
        if (order.getType() == OrderType.BUY) {
            buyOrders.offer(order);
        } else {
            sellOrders.offer(order);
        }
    }
}

Buy vs Sell Side

SideHeap TypeComparatorTop = Best
BUY (Bids) Max-Heap Double.compare(b.price, a.price) Highest price buyer
SELL (Asks) Min-Heap Double.compare(a.price, b.price) Lowest price seller

Match condition: Best bid price ≥ Best ask price. The spread closes and a trade executes.

05 — Matching Engine

Heart of the exchange. The OrderMatchingEngine implements price-time priority matching. When an order is placed, it immediately attempts to match against the opposite side of the book.

Match Algorithm (Price-Time Priority)

public int matchOrders(String symbol) {
    OrderBook book = orderBooks.get(symbol);
    if (book == null) return 0;

    int matchCount = 0;

    while (true) {
        Order buyOrder  = book.getBestBuyOrder();   // peek max-heap
        Order sellOrder = book.getBestSellOrder();  // peek min-heap

        if (buyOrder == null || sellOrder == null) break;
        if (buyOrder.getPrice() < sellOrder.getPrice()) break;

        // Execute trade at sell order price (price-time priority)
        int qty = Math.min(buyOrder.getRemainingQuantity(),
                         sellOrder.getRemainingQuantity());
        double price = sellOrder.getPrice();

        executeTrade(buyOrder, sellOrder, qty, price);
        matchCount++;

        if (buyOrder.getRemainingQuantity() == 0) {
            book.removeOrder(buyOrder);
            buyOrder.setStatus(OrderStatus.FILLED);
        }
        if (sellOrder.getRemainingQuantity() == 0) {
            book.removeOrder(sellOrder);
            sellOrder.setStatus(OrderStatus.FILLED);
        }
    }
    return matchCount;
}

Trade Execution

private void executeTrade(Order buy, Order sell,
                           int quantity, double price) {
    String tradeId = "TRD-" + tradeIdGenerator.incrementAndGet();
    Trade trade = new Trade(
        tradeId, buy.getSymbol(),
        buy.getId(), sell.getId(),
        quantity, price, LocalDateTime.now()
    );

    trades.computeIfAbsent(buy.getSymbol(),
        k -> new ArrayList<>()).add(trade);

    buy.setFilledQuantity(buy.getFilledQuantity() + quantity);
    sell.setFilledQuantity(sell.getFilledQuantity() + quantity);
}

public class Order {

    public enum OrderStatus { PENDING, FILLED, CANCELLED }

    private final String orderId;
    private final String stockSymbol;
    private final OrderType type;
    private final double price;
    private int quantity;
    private OrderStatus status;
    private LocalDateTime timestamp;

    public Order(String orderId, String symbol,
               OrderType type, double price, int qty) {
        this.orderId = orderId;
        this.stockSymbol = symbol;
        this.type = type;
        this.price = price;
        this.quantity = qty;
        this.status = OrderStatus.PENDING;
        this.timestamp = LocalDateTime.now();
    }
}

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap orders, trades, orderBooks maps Safe concurrent reads/writes across symbols
synchronized(orderBook) matchOrders(), addOrder() Per-symbol locking — AAPL and GOOG match concurrently, only same-symbol ops serialize
AtomicLong orderIdGenerator, tradeIdGenerator Lock-free unique ID generation
computeIfAbsent OrderBook creation per symbol Atomic lazy initialization
volatile Pricing strategy references Cross-thread visibility for runtime swaps
"I use per-symbol locking on the OrderBook, not a global lock. Two different stocks (AAPL, GOOG) can match orders concurrently. Only concurrent operations on the same symbol's book are serialized."

Insert (offer)O(log n)Maintains heap property
Peek bestO(1)Top of heap = best price

11 — Key Algorithms

Price-Time Priority Matching

Time Complexity: O(min(B,S) × log n) per match cycle, where B = buy orders, S = sell orders

// The golden rule of exchange matching:
// 1. PRICE priority: best bid vs best ask
// 2. TIME priority: among same price, earliest order first
// 3. EXECUTION PRICE: use the resting order's price (sell price)

while (buyOrders.isNotEmpty() && sellOrders.isNotEmpty()) {
    if (bestBid.price >= bestAsk.price) {
        qty   = min(bestBid.remaining, bestAsk.remaining);
        price = bestAsk.price;  // resting order price
        executeTrade(bestBid, bestAsk, qty, price);
    } else break;  // spread is open, no more matches
}

Order Placement Flow

public String placeOrder(Order order) {
    // 1. Generate unique ID
    String orderId = "ORD-" + orderIdGenerator.incrementAndGet();
    order.setId(orderId);
    order.setStatus(OrderStatus.OPEN);

    // 2. Store in global order map
    orders.put(orderId, order);

    // 3. Add to symbol's order book (lazy init)
    OrderBook book = orderBooks.computeIfAbsent(
        order.getSymbol(),
        k -> new OrderBook(order.getSymbol()));
    book.addOrder(order);

    // 4. Attempt immediate match
    matchOrders(order.getSymbol());

    return orderId;
}

Order Cancellation

public boolean cancelOrder(String orderId) {
    Order order = orders.get(orderId);
    if (order == null || order.getStatus() != OrderStatus.OPEN)
        return false;

    order.setStatus(OrderStatus.CANCELLED);
    OrderBook book = orderBooks.get(order.getSymbol());
    if (book != null) book.removeOrder(order);

    return true;
}

13 — Interview Follow-up Q&A

Explain price-time priority matching.
Orders are matched first by best price (highest bid vs lowest ask), then by time of arrival among orders at the same price. The trade executes at the resting order's price (the order already on the book). This ensures fairness: earlier orders at the same price level get filled first.
Why PriorityQueue and not TreeMap?
PriorityQueue gives O(log n) insert and O(1) peek for best price, which covers the hot path. However, arbitrary removal (cancel) is O(n). In production, use TreeMap<Double, LinkedList<Order>> where each price level is a FIFO queue — this gives O(log p) for all operations where p = number of distinct price levels.
How do you handle partial fills?
Each order tracks filledQuantity and remainingQuantity. When a match occurs, Math.min(buyRemaining, sellRemaining) determines the trade quantity. If remaining > 0 after the trade, the order stays on the book with PARTIALLY_FILLED status. Multiple trades can fill a single large order.
How would you handle market orders?
Market orders have no price limit — they match at whatever the best available price is. For a market BUY, set price to Double.MAX_VALUE so it always passes the buyPrice ≥ sellPrice check. Market orders never rest on the book. If there's insufficient liquidity, the remaining portion is cancelled (IOC behavior) or queued.
How do you prevent the same order from matching twice?
synchronized(orderBook) ensures only one thread runs the matching loop for a given symbol at a time. Once an order is FILLED, it's removed from the book. The ConcurrentHashMap for the global orders map ensures safe status updates across threads.
How would you scale to millions of orders?
Partition by symbol — each symbol's order book runs on its own thread/core (like LMAX Disruptor). Use a single-writer principle: one thread per symbol eliminates locking entirely. For cross-symbol operations, use message queues. Consider ring buffers for order ingestion instead of ConcurrentHashMap.
What if the database goes down?
The in-memory matching engine is the source of truth during trading hours. Use write-ahead logging (WAL) to persist every order and trade to disk before acknowledging. On recovery, replay the WAL to rebuild the order book. In production: replicated state machines with Raft consensus.
How would you implement stop orders?
Maintain a separate Map<String, TreeMap<Double, List<Order>>> for stop orders, keyed by trigger price. After every trade, check if the trade price crossed any stop triggers. If so, convert the stop order to a market/limit order and inject it into the regular order book.
How do you ensure data consistency between order status and the book?
All mutations (status change + book add/remove) happen inside the same synchronized(orderBook) block. This provides atomicity for the pair of operations. The order is never in an inconsistent state where its status says OPEN but it's not on the book, or vice versa.
Why use AtomicLong for IDs instead of UUID?
AtomicLong is lock-free and generates monotonically increasing IDs — great for ordering and debugging. UUIDs are 128-bit, random, and have higher comparison cost. For a single-JVM exchange, AtomicLong is simpler and faster. In distributed setups, use Snowflake IDs (timestamp + machine + sequence).
How would you add real-time market data feeds?
Implement the Observer pattern: after every trade execution, publish a MarketDataEvent (symbol, lastPrice, volume, bestBid, bestAsk) to registered subscribers. Use CopyOnWriteArrayList for thread-safe iteration. At scale, push events to Kafka topics and let subscribers consume asynchronously.
What are the custom exceptions and when do they fire?
InsufficientSharesException — sell order exceeds portfolio holdings. InvalidPriceException — negative or zero price on a limit order. OrderNotFoundException — cancel/query for a non-existent order ID. Each maps to specific validation in the service layer before reaching the matching engine.
