01 — Requirements

Functional

  • Given a long URL, generate a unique short URL (6–8 chars)
  • Given a short URL, redirect to the original long URL
  • Support custom aliases (user-chosen short codes)
  • Duplicate detection — same long URL returns same short code
  • URL validation & normalization (lowercase domain, strip default ports)
  • Analytics — access count, created-at, last-accessed timestamps
  • Delete a short URL mapping
  • Availability check — is a given short code taken?

Non-Functional

  • Low latency — O(1) read/write for shorten + redirect
  • Thread-safe — concurrent shortening & redirects via ConcurrentHashMap + AtomicLong
  • Scalable namespace — 6-char Base62 = 56.8 billion unique URLs
  • URL-safe — only alphanumeric characters [0-9a-zA-Z]
  • Idempotent — shortening the same URL twice returns the same code
  • Abuse prevention — reserved keywords blocked, length limits enforced

03 — Design Patterns

Three patterns, each earning its place in the design.

Strategy Pattern

Encoding algorithm is the core strategy. Base62Encoder is the current implementation. Swappable for MD5-truncate, Snowflake, or any custom encoder without changing the service.

Repository Pattern

Dual ConcurrentHashMap stores abstract data access. shortToLong for redirect, longToShort for dedup. Swap for Redis/DB without changing business logic.

Factory Pattern

generateShortCode() encapsulates ID creation. Combines AtomicLong counter + Base62 encoding + padding. Callers don't know the generation strategy.

Value Object Pattern

ShortURL, Analytics, Location are immutable with no identity. Thread-safe by design. Equals/hashCode based on value, not reference.

04 — Encoding Algorithms

Why Base62? URL-safe (no special chars), compact (6 chars = 56.8B URLs), human-readable, and reversible. The gold standard for URL shorteners.

Base62 Encode Implementation

public static String encode(long num) {
    if (num == 0) return "0";

    StringBuilder result = new StringBuilder();
    while (num > 0) {
        int remainder = (int) (num % 62);
        result.append(BASE62_CHARS.charAt(remainder));
        num /= 62;
    }
    return result.reverse().toString();
}

// Padded version for fixed-length codes
public static String encode(long num, int minLength) {
    String encoded = encode(num);
    if (encoded.length() >= minLength) return encoded;
    return "0".repeat(minLength - encoded.length()) + encoded;
}

05 — Collision Handling

Our approach: Counter-based generation produces zero collisions by design. Each ID is unique because AtomicLong never repeats. No collision resolution needed.

Strategies Comparison

StrategyHowTrade-off
Counter-based (ours) AtomicLong → Base62. Monotonically increasing. Zero collisions but sequential/predictable. Add random offset to mitigate.
Pre-generated key pool Worker generates keys in advance, service pops from pool. Zero collision at read time. Extra infra for key generation service.
Hash + retry MD5/SHA → truncate. On collision, append counter and rehash. Non-predictable but collision probability grows with scale. Extra DB lookups.
Bloom filter check Check Bloom filter before DB. If probably-exists, rehash. Fast negative lookups, false positive rate needs tuning.

Duplicate URL Detection

// Bidirectional lookup prevents duplicates
String existingCode = longToShort.get(normalizedURL);
if (existingCode != null) {
    return new ShortURL(existingCode, baseUrl); // idempotent
}

The longToShort map ensures the same long URL always maps to the same short code. Normalization (lowercase domain, strip trailing slash) prevents near-duplicates.

Our choice: 302 for analytics tracking. Every redirect hits the server, allowing recordAccess() to update click counts and timestamps.

Redirect Implementation

public String getLongURL(String shortCode) {
    URLMapping mapping = shortToLong.get(shortCode);
    if (mapping == null) {
        throw new URLNotFoundException("Short code not found: " + shortCode);
    }
    mapping.recordAccess();  // AtomicLong increment + volatile timestamp
    return mapping.getLongURL();
}

07 — TTL / Expiration

Why TTL? URLs accumulate over time. Expired links free namespace, reduce storage, and prevent stale redirects. Critical for abuse prevention.

Implementation Strategies

ApproachMechanismPros/Cons
Lazy Expiration Check TTL on read. If expired, delete & return 404. Simple. Dead entries linger until accessed.
Scheduled Cleanup Background thread scans & deletes expired entries periodically. Proactive. Adds background thread complexity.
Redis TTL Set EXPIRE key seconds on insert. Redis auto-evicts. Zero application logic. Requires Redis.
Hybrid Lazy check on read + periodic cleanup for unreached keys. Best coverage. Production approach.

Lazy Expiration Pseudocode

public String getLongURL(String shortCode) {
    URLMapping mapping = shortToLong.get(shortCode);
    if (mapping == null) throw new URLNotFoundException(...);

    if (mapping.isExpired()) {
        deleteURL(shortCode);  // clean up both maps
        throw new URLNotFoundException("URL has expired");
    }

    mapping.recordAccess();
    return mapping.getLongURL();
}

08 — Key Algorithms

Short Code Generation

Counter + Base62 + Padding. O(1) time, zero collisions.

private String generateShortCode() {
    long id = counter.getAndIncrement();  // AtomicLong: lock-free
    return Base62Encoder.encode(id, shortCodeLength);  // pad to 6 chars
}

Custom Alias Validation

public static boolean isValidAlias(String alias) {
    if (alias.length() < 4 || alias.length() > 8) return false;
    if (!alias.matches("^[a-zA-Z0-9]+$")) return false;
    if (isReservedKeyword(alias)) return false;
    return true;
}

// Per-URL atomic tracking on URLMapping
public void recordAccess() {
    this.lastAccessedAt = LocalDateTime.now();  // volatile write
    this.accessCount.incrementAndGet();        // AtomicLong
}

public void trackClick(String shortUrl, String ipAddress) {
    clickCounts.merge(shortUrl, 1, Integer::sum);

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap shortToLong, longToShort, clickCounts, uniqueVisitors Lock-striped concurrent reads/writes, O(1) average
AtomicLong counter (ID generation), accessCount (per mapping) Lock-free CAS-based increment — no contention under high concurrency
volatile URLMapping.lastAccessedAt Cross-thread visibility of timestamp without synchronization overhead
Immutable Objects ShortURL, Analytics Thread-safe by design — no synchronization needed
"No synchronized blocks needed. ConcurrentHashMap handles concurrent map operations, AtomicLong handles counter increments, and volatile handles timestamp visibility. Each mechanism is chosen for minimal contention."

Race Condition: Duplicate Shortening

// Potential race: two threads shorten the same URL simultaneously
// Thread A: checks longToShort → not found
// Thread B: checks longToShort → not found
// Both generate different short codes for same URL

// Fix: ConcurrentHashMap.putIfAbsent() or synchronized block
String existing = longToShort.putIfAbsent(normalizedURL, shortCode);
if (existing != null) {
    shortToLong.remove(shortCode); // rollback
    return new ShortURL(existing, baseUrl);
}

12 — Key Data Structures

// Primary stores (bidirectional lookup)
ConcurrentHashMap<String, URLMapping> shortToLong;  // "abc123" → URLMapping
ConcurrentHashMap<String, String>     longToShort;  // normalized URL → "abc123"

// ID generation
AtomicLong counter = new AtomicLong(1);  // monotonically increasing

// Config
String baseUrl;            // "https://short.ly"
int    shortCodeLength;    // default 6

// Per-mapping analytics (on URLMapping)
AtomicLong     accessCount;     // lock-free counter
volatile LocalDateTime lastAccessedAt; // visible across threads

// Advanced analytics (AnalyticsTracker)
ConcurrentHashMap<String, Integer>      clickCounts;     // total clicks
ConcurrentHashMap<String, Set<String>> uniqueVisitors;  // per-IP tracking

Why Dual Maps?

MapLookup DirectionUsed For
shortToLong short code → URLMapping Redirect: O(1) lookup, analytics, existence check
longToShort long URL → short code Dedup: same URL returns same code (idempotent)

Trade-off: 2× memory for deduplication. Worth it — without longToShort, the same URL could get different short codes on every call.

13 — Interview Follow-up Q&A

When should you use 301 vs 302 redirect?
301 (Permanent): Browser caches the redirect — subsequent requests skip the server entirely. Better for SEO, faster for users, but you lose analytics on repeat visits. 302 (Temporary): Every click hits the server — essential for click analytics, A/B testing, link rotation, and abuse monitoring. Most URL shorteners (bit.ly, TinyURL) use 302 to track clicks.
How do you support custom aliases?
Accept an optional alias parameter. Validate: 4–8 alphanumeric chars, not a reserved keyword (admin, api, www...), not already taken. Check shortToLong.containsKey(alias) — if taken, throw AliasUnavailableException. If the URL was already shortened with an auto-generated code, remove the old mapping and replace with the alias.
How do you track analytics at scale?
Per-mapping AtomicLong handles basic click counts lock-free. For detailed analytics (geo, device, referrer, time series), write click events to Kafka asynchronously and process with a stream pipeline into ClickHouse or Druid. The redirect path stays fast (just enqueue), analytics are computed async. AnalyticsTracker shows the in-memory approach with per-IP unique visitor tracking.
How do you generate unique IDs in a distributed system?
Option 1: Range-based allocation. A coordination service (ZooKeeper) assigns counter ranges [1-1M], [1M+1-2M] to each app server. Each server increments within its range with no coordination. Option 2: Snowflake IDs. 64-bit: timestamp(41) + machineId(10) + sequence(12). ~4096 IDs/ms/machine. Option 3: Pre-generated key service. A separate service generates keys in bulk, app servers request batches.
How would you add a cache layer?
Redis in front of the database: short code → long URL. On redirect: check Redis first (O(1), sub-ms) → on miss, hit DB → populate Redis. Use LRU eviction policy. Hot URLs stay in cache, cold URLs get evicted. Set TTL on cache entries matching URL expiration. Cache hit rate for URL shorteners is typically 80-90% because of Zipf distribution (few URLs get most clicks).
How do you handle hot URLs (viral links)?
Problem: A single viral URL can receive millions of requests/sec, overwhelming a single cache shard. Solutions: (1) Local in-process cache (Caffeine/Guava) with short TTL (30s) on each app server — no network hop. (2) Replicate hot keys across multiple Redis shards. (3) CDN-level redirect — configure the CDN (CloudFront) to cache 302 responses for hot URLs with short max-age. (4) Auto-detect hot keys via access count threshold and promote to local cache.
How do you prevent abuse (spam, phishing)?
Rate limiting: Token bucket per IP/user (e.g., 100 URLs/hour). URL scanning: Check against Google Safe Browsing API before shortening. Reserved keywords: Block admin, api, login, etc. (already implemented in URLValidator). CAPTCHA: For anonymous users creating links. Link preview: Show destination before redirecting. Reporting: Allow users to flag malicious links. Auto-expire: Anonymous links expire after 24h.
How do you handle URL expiration/TTL?
Lazy expiration: Check TTL on every redirect. If expired, delete mapping and return 404. Active cleanup: ScheduledExecutorService scans for expired entries every N minutes. Redis: Set EXPIRE on insert — Redis auto-evicts. Production: Hybrid (lazy + active). Default TTL: 1 year for authenticated users, 24 hours for anonymous. Allow custom TTL on paid plans.
Why counter-based instead of hash-based encoding?
Counter-based: Zero collisions (monotonically increasing), O(1) generation, simple implementation. Downside: sequential/predictable — users can enumerate URLs. Mitigation: Add a random offset to the counter, or XOR with a secret key before Base62 encoding. Hash-based: Non-predictable, but collisions require retry logic, and the hash function adds latency. For an LLD interview, counter-based is the cleaner design.
How would you shard the database?
Shard by short code hash: hash(shortCode) % N → shard. Distributes uniformly. Range-based: codes starting with [0-9] → shard 1, [a-z] → shard 2, [A-Z] → shard 3. Uneven but simple. Consistent hashing: Add/remove shards without rehashing everything. The longToShort map is harder — either replicate across all shards, use a separate dedup service, or accept that different shards may create duplicates (acceptable at scale).
What's the read:write ratio and how does it affect design?
URL shorteners are massively read-heavy — typically 100:1 reads to writes. A URL is created once but clicked thousands of times. This means: (1) Optimize the redirect path (cache, CDN). (2) Write path can tolerate slightly higher latency. (3) Database read replicas are highly effective. (4) Eventual consistency for analytics is acceptable.
How do you ensure the short URL is actually shorter?
Minimum URL length check (MIN_URL_LENGTH = 10). With a 6-char code and base URL "https://short.ly/", the short URL is 24 chars. If the input is shorter than that, shortening it is pointless — but most real URLs are 50-200+ chars. The system validates isValid() to reject malformed inputs but doesn't refuse short URLs — the value is in the permanent, memorable link.
