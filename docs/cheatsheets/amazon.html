01 — Requirements

Functional

  • Product Management — Add/update/delete products with name, description, price, category, images, variants
  • Product Catalog — Categorization, sub-categories, inventory tracking, stock alerts, product status
  • Customer Management — Register/login, profiles with addresses, order history, wishlist, Prime membership
  • Shopping Cart — Add/remove items, update quantity, calculate totals (subtotal + tax + shipping), cart expiry
  • Order Processing — Create from cart, select address, choose payment, order tracking, cancellation & refunds
  • Payment Processing — Card, UPI, Wallet, Net Banking, COD — validation, transaction recording, refunds
  • Review & Rating — 1-5 stars, text reviews, verified purchase badge, helpful voting
  • Search & Discovery — Search by name/description, filter by category/price/rating, sort by price/popularity

Non-Functional

  • Performance — Product search < 200ms, cart ops < 100ms, order placement < 500ms, 100K+ concurrent users
  • Concurrency — Thread-safe cart operations, atomic inventory updates, optimistic locking for orders
  • Reliability — ACID transactions for orders, payment idempotency, order confirmation emails
  • Scalability — Horizontal scaling, DB sharding by category/region, Redis caching, CDN for images

03 — Design Patterns

Seven patterns, each solving a specific e-commerce challenge.

Strategy Pattern

Multiple payment methods (Card, UPI, Wallet, COD). Easy to add new payment providers without modifying service code.

Facade Pattern

AmazonService wraps Product, Cart, Order, Payment, and Customer services into one unified interface for clients.

Repository Pattern

ConcurrentHashMap-backed stores abstract data access. Swap for JPA/MongoDB without changing service logic.

Observer Pattern

Order status updates trigger notifications — email, SMS, push. Decoupled event handling for order lifecycle.

Decorator Pattern

Wrap cart total with promotions, discounts, coupons. Stack: new BulkDiscount(new CouponDiscount(basePrice))

Factory + Command

Factory centralizes Order/Payment creation. Command wraps order operations (Place, Cancel, Refund) for undo/redo and audit trails.

04 — Product Catalog

Key insight: Products are the backbone. They need fast search, category filtering, stock tracking, and running average ratings — all thread-safe.

Product Entity — Core Methods

public class Product {
    private final String id;
    private BigDecimal price;
    private int stockQuantity;
    private double rating;
    private int reviewCount;
    private ProductStatus status;

    public boolean isInStock() {
        return stockQuantity > 0 && status == ProductStatus.ACTIVE;
    }

    public void reduceStock(int quantity) {
        if (quantity > stockQuantity)
            throw new IllegalStateException("Insufficient stock");
        this.stockQuantity -= quantity;
    }

    public void updateRating(double newRating) {
        // Running average: O(1) no need to re-scan all reviews
        this.rating = ((this.rating * reviewCount) + newRating) / (reviewCount + 1);
        this.reviewCount++;
    }
}

ProductService Implementation

public class ProductServiceImpl implements ProductService {
    private final Map<String, Product> products = new ConcurrentHashMap<>();

    public List<Product> searchProducts(String keyword) {
        String lower = keyword.toLowerCase();
        return products.values().stream()
            .filter(p -> p.getName().toLowerCase().contains(lower) ||
                       (p.getDescription() != null &&
                        p.getDescription().toLowerCase().contains(lower)))
            .collect(Collectors.toList());
    }

    public void updateStock(String productId, int quantity) {
        Product p = products.get(productId);
        if (quantity >= 0) p.addStock(quantity);
        else p.reduceStock(Math.abs(quantity));
    }
}

05 — Cart Management

Why Redis-backed? In-memory is fast but lost on restart. DB is persistent but slow (~200ms). Redis gives <10ms ops with 7-day auto-expiry and persistence.

Thread-Safe Cart with synchronized

public class Cart {
    private final Map<String, CartItem> items = new ConcurrentHashMap<>();
    private BigDecimal totalAmount = BigDecimal.ZERO;

    public synchronized void addItem(Product product, int quantity) {
        if (!product.isInStock())
            throw new IllegalStateException("Product is out of stock");
        if (!product.canFulfillQuantity(quantity))
            throw new IllegalStateException("Requested quantity not available");

        CartItem existing = items.get(product.getId());
        if (existing != null) {
            existing.setQuantity(existing.getQuantity() + quantity);
        } else {
            items.put(product.getId(),
                new CartItem(product.getId(), product.getName(),
                             product.getPrice(), quantity));
        }
        recalculateTotal();
    }

    public synchronized void updateQuantity(String productId, int qty) {
        CartItem item = items.get(productId);
        if (item != null) {
            if (qty <= 0) items.remove(productId);
            else item.setQuantity(qty);
            recalculateTotal();
        }
    }

    private void recalculateTotal() {
        totalAmount = items.values().stream()
            .map(i -> i.getPrice().multiply(BigDecimal.valueOf(i.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

Order — State Transitions with Guards

public class Order {
    private OrderStatus status = OrderStatus.PENDING;

    public void confirm(String paymentId) {
        this.paymentId = paymentId;
        this.status = OrderStatus.CONFIRMED;
    }

    public void ship(String trackingNumber) {
        if (status != OrderStatus.CONFIRMED)
            throw new IllegalStateException("Must be confirmed before shipping");
        this.trackingNumber = trackingNumber;
        this.status = OrderStatus.SHIPPED;
    }

    public void deliver() {
        if (status != OrderStatus.SHIPPED)
            throw new IllegalStateException("Must be shipped before delivery");
        this.actualDelivery = LocalDateTime.now();
        this.status = OrderStatus.DELIVERED;
    }

    public void cancel() {
        if (status == OrderStatus.DELIVERED)
            throw new IllegalStateException("Cannot cancel delivered order");
        this.status = OrderStatus.CANCELLED;
    }
}

OrderServiceImpl

public class OrderServiceImpl implements OrderService {
    private final Map<String, Order> orders = new ConcurrentHashMap<>();

    public String createOrder(String userId, List<OrderItem> items, Address addr) {
        String orderId = UUID.randomUUID().toString();
        BigDecimal total = items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
        Order order = new Order(orderId, userId, items, total, addr);
        orders.put(orderId, order);
        return orderId;
    }

    public List<Order> getUserOrders(String userId) {
        return orders.values().stream()
            .filter(o -> o.getUserId().equals(userId))
            .sorted(Comparator.comparing(Order::getOrderDate).reversed())
            .collect(Collectors.toList());
    }
}

Payment Entity — State Machine

public class Payment {
    private PaymentStatus status = PaymentStatus.PENDING;

    public void process(String transactionId) {
        this.transactionId = transactionId;
        this.status = PaymentStatus.PROCESSING;
        this.paymentDate = LocalDateTime.now();
    }

    public void confirm() {
        if (status != PaymentStatus.PROCESSING)
            throw new IllegalStateException("Must be processing first");
        this.status = PaymentStatus.COMPLETED;
    }

    public void refund() {
        if (status != PaymentStatus.COMPLETED)
            throw new IllegalStateException("Can only refund completed payments");
        this.status = PaymentStatus.REFUNDED;
    }
}

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap All data stores (products, orders, carts, reviews) Lock-free reads, segment-level writes
synchronized Cart.addItem(), removeItem(), updateQuantity(), clear() Consistent total calculation — prevents race between add and recalculate
BigDecimal All monetary fields (price, total, subtotal) No floating-point rounding errors ($0.1 + $0.2 ≠ $0.3 with double)
final fields OrderItem (productId, unitPrice, quantity, subtotal) Immutable once created — price locked at order time
Defensive copies getItems() returns new ArrayList/HashMap Callers can't mutate internal collections
@Version (production) Product.stockQuantity updates Optimistic locking prevents overselling under concurrency
"I use synchronized on Cart for consistent totals, ConcurrentHashMap for store-level concurrency, and BigDecimal for monetary precision. In production, inventory gets optimistic locking with @Version."

Inventory: Check-Then-Update vs Atomic Update

// BAD: Race condition -- two users can oversell
if (product.getStockQuantity() >= quantity) {
    product.setStockQuantity(product.getStockQuantity() - quantity);
}

// GOOD: Optimistic locking with @Version
@Version private Long version;

public boolean reserveStock(int quantity) {
    if (stockQuantity >= quantity) {
        stockQuantity -= quantity;  // JPA auto-retries on version mismatch
        return true;
    }
    return false;
}

12 — Key Algorithms

Order Placement Algorithm

// 5-step atomic order placement

// Step 1: Validate Cart
if (cart.isEmpty()) throw new EmptyCartException();
for (CartItem item : cart.getItems()) {
    Product p = productService.getProduct(item.getProductId());
    if (!p.canFulfillQuantity(item.getQuantity()))
        throw new InsufficientStockException();
}

// Step 2: Reserve Inventory (rollback all if any fails)
List<Product> reserved = new ArrayList<>();
try {
    for (CartItem item : cart.getItems()) {
        Product p = productService.getProduct(item.getProductId());
        p.reduceStock(item.getQuantity());
        reserved.add(p);
    }
} catch (Exception e) {
    // Rollback all reserved inventory
    reserved.forEach(p -> p.addStock(/*original qty*/));
    throw e;
}

// Step 3: Create Order (prices locked at this moment)
List<OrderItem> orderItems = cart.getItems().stream()
    .map(ci -> new OrderItem(ci.getProductId(), ci.getProductName(),
                             ci.getPrice(), ci.getQuantity()))
    .collect(Collectors.toList());
Order order = orderService.createOrder(userId, orderItems, address);

// Step 4: Process Payment
Payment payment = paymentService.initiatePayment(order, method);
if (payment.getStatus() == COMPLETED) {
    order.confirm(payment.getId());
} else {
    reserved.forEach(p -> p.addStock(/*qty*/)); // release inventory
    order.cancel();
}

// Step 5: Clear Cart & Notify
cart.clear();
notificationService.send(userId, "Order confirmed!");

Time: O(n) where n = cart items. Space: O(n) for order items. Critical: inventory rollback on any failure.

14 — Interview Follow-up Q&A

How do you prevent overselling of products?
Use optimistic locking with @Version annotation. The DB auto-retries if version mismatches. Two concurrent buyers: first succeeds, second gets OptimisticLockException and retries. No global lock needed — only conflicts on the same product serialize.
How do you handle abandoned carts?
Redis cache with 7-day TTL. After 24h of inactivity, trigger an abandoned-cart email via a scheduled job. After 7 days, Redis auto-evicts. Inventory is NOT reserved at cart-add time — only at order placement. So abandoned carts don't block stock.
What happens if payment fails after order creation?
Rollback inventory for all order items, mark order as CANCELLED, notify customer. The code does this in a try-catch: if payment fails, iterate reserved products and call addStock() to restore quantities. Idempotent payment IDs prevent double-charging.
How to scale for Black Friday traffic (100x normal)?
(1) Pre-warm caches for deal products. (2) Add read replicas for product catalog. (3) Queue-based order processing to absorb spikes. (4) Rate limiting per customer. (5) CDN for static assets. (6) Auto-scale services horizontally. (7) Separate hot-deal inventory into its own service with stricter locks.
How to calculate dynamic pricing with discounts?
Decorator pattern — wrap base price with discount decorators. Stack: new BulkDiscount(new CouponDiscount(new SeasonalDiscount(basePrice))). Each decorator applies its rule and delegates to the wrapped strategy. New discount type = new decorator class, zero modification to existing code.
Why BigDecimal instead of double for prices?
0.1 + 0.2 = 0.30000000000000004 in double. Financial calculations MUST use BigDecimal for exact arithmetic. All prices, totals, and subtotals use BigDecimal. Use BigDecimal.valueOf() (not new BigDecimal(0.1)) to avoid constructor precision issues.
How does the Review system prevent fake reviews?
(1) Verified purchase badge — only customers who bought the product can get the badge. (2) Rating validation: must be 1-5, enforced in constructor. (3) Helpful voting surfaces quality reviews. (4) In production: NLP-based spam detection, one review per product per customer, review moderation queue.
Why Facade pattern for AmazonService?
Without Facade, clients must know about 4+ service interfaces (Product, Cart, Order, Payment). The Facade provides a single entry point with a unified API. Internally it delegates to specialized services. Reduces coupling — clients depend on one interface, not four.
How would you handle order cancellation and refunds?
Order.cancel() checks guard conditions (can't cancel DELIVERED orders). On cancel: (1) Restore inventory via addStock(). (2) If payment was COMPLETED, call payment.refund(). (3) Send notification. Refund processing can be async via a queue for reliability.
How does the category hierarchy work?
Category has parentCategoryId (null for root) and subcategoryIds list. This forms a tree structure: Electronics → Laptops → Gaming Laptops. isRootCategory() checks if it's a top-level category. Products reference a categoryId at any level of the tree.
How would you implement search at scale?
Current: stream filter on ConcurrentHashMap (O(n), fine for LLD). At scale: Elasticsearch with inverted index for full-text search, faceted filtering (category, price range, rating), and relevance scoring. Cache hot queries in Redis. Autocomplete via prefix tries or ES suggest API.
How do you ensure data consistency across services?
For a monolith (LLD scope): transactional order placement — inventory + order + payment in one atomic operation. For microservices: Saga pattern with compensating transactions. Order service orchestrates: reserve inventory → process payment → confirm. On failure at any step, execute compensating actions in reverse.

Key Takeaways for the Interview

  • Inventory + Order placement are the critical path — must prevent overselling with atomic updates
  • BigDecimal for all monetary values — never use double for prices
  • synchronized Cart ensures consistent totals under concurrency
  • Price locking in OrderItem prevents price changes from affecting existing orders
  • Inventory rollback on payment failure maintains data consistency
  • Facade pattern simplifies the client interface for a complex multi-service system
  • Mention Redis for carts, Elasticsearch for search, Saga for distributed transactions
