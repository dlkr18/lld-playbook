01 — Requirements

Functional

  • Book Catalog — Add books with ISBN (unique), title, author, publisher, year
  • Track book status: AVAILABLE, BORROWED, RESERVED, LOST
  • Member Management — Register with auto-generated ID, enforce borrow limit (max 5)
  • Borrow/Return — Validate availability, update status atomically, log transactions
  • Search — By title or author (case-insensitive, partial match)
  • Transaction History — Immutable BORROW/RETURN log with timestamps, query by member
  • Prevent concurrent borrowing conflicts — no double-booking

Non-Functional

  • O(1) book lookup by ISBN, member lookup by ID
  • O(n) search operations (acceptable for < 100K books)
  • Thread-safe — synchronized borrow/return, concurrent reads
  • Scalable — 10K+ books, 1K+ members, 100K+ transactions
  • Data Integrity — ISBN uniqueness, no duplicate borrows, immutable transaction log

03 — Design Patterns

Key patterns that demonstrate design thinking in the interview.

Facade Pattern

LibraryServiceImpl is the single entry point for all operations — borrow, return, search, history. Hides internal data structures and validation logic from clients.

Repository Pattern

ConcurrentHashMap stores for books, members, and transactions. Abstracts persistence — easily swappable for a database-backed implementation.

Interface Segregation

LibraryService interface defines the contract: addBook, registerMember, borrowBook, returnBook, search, history. Implementation is decoupled from API.

Immutable Audit Log

Transactions are append-only. Once created, they are never modified. Provides a complete, tamper-proof borrowing history for reporting and auditing.

LateFeeCalculator

class LateFeeCalculator {
    private static final double FEE_PER_DAY = 0.50;
    private static final int GRACE_PERIOD_DAYS = 14;

    public double calculateFee(Transaction borrowTxn, LocalDate returnDate) {
        LocalDate dueDate = borrowTxn.getTimestamp()
                                      .plusDays(GRACE_PERIOD_DAYS)
                                      .toLocalDate();

        if (returnDate.isAfter(dueDate)) {
            long daysOverdue = ChronoUnit.DAYS.between(dueDate, returnDate);
            return daysOverdue * FEE_PER_DAY;
        }
        return 0.0;
    }
}

// In returnBook():

if (fee > 0) {

}

06 — Search System

Current approach: Linear scan with Java Streams. Case-insensitive partial match on title and author. Good enough for <100K books.

Search by Title

public List<Book> searchByTitle(String title) {
    return books.values().stream()
        .filter(book -> book.getTitle().toLowerCase()
                           .contains(title.toLowerCase()))
        .collect(Collectors.toList());
}

Time: O(n) — scans all books. Space: O(k) where k = matching results.

Search by Author

public List<Book> searchByAuthor(String author) {
    return books.values().stream()
        .filter(book -> book.getAuthor().toLowerCase()
                           .contains(author.toLowerCase()))
        .collect(Collectors.toList());
}

Optimization: Inverted Index

class BookSearchIndex {
    private Map<String, Set<String>> titleIndex = new HashMap<>();

    public void indexBook(Book book) {
        for (String word : tokenize(book.getTitle())) {
            titleIndex.computeIfAbsent(word.toLowerCase(), k -> new HashSet<>())
                      .add(book.getIsbn());
        }
    }

    public List<Book> searchTitle(String query) {
        Set<String> isbns = new HashSet<>();
        for (String word : tokenize(query)) {
            Set<String> matches = titleIndex.get(word.toLowerCase());
            if (matches != null) {
                if (isbns.isEmpty()) isbns.addAll(matches);
                else isbns.retainAll(matches); // AND operation
            }
        }
        return isbns.stream().map(books::get).collect(toList());
    }
}

Time: O(w + k) where w = query words, k = results — vs O(n) linear scan. Build index on addBook().

07 — Key Algorithms

Borrow Book — O(1)

public synchronized boolean borrowBook(String memberId, String isbn) {
    Member member = members.get(memberId);
    Book book = books.get(isbn);

    if (member == null || book == null) return false;
    if (!member.canBorrowMore()) return false;          // limit check
    if (book.getStatus() != BookStatus.AVAILABLE) return false;

    book.borrow(memberId);           // status=BORROWED, borrowedBy=memberId
    member.addBorrowedBook(isbn);    // add to member's list

    Transaction txn = new Transaction(
        UUID.randomUUID().toString(), memberId, isbn, TransactionType.BORROW
    );
    transactions.add(txn);          // immutable log entry
    return true;
}

synchronized ensures atomicity of the 3-step check-update-log sequence. No race condition possible.

Return Book — O(1)

public synchronized boolean returnBook(String memberId, String isbn) {
    Member member = members.get(memberId);
    Book book = books.get(isbn);

    if (member == null || book == null) return false;
    if (!member.getBorrowedBooks().contains(isbn)) return false;

    book.returnBook();               // status=AVAILABLE, borrowedBy=null
    member.removeBorrowedBook(isbn); // remove from member's list

    Transaction txn = new Transaction(
        UUID.randomUUID().toString(), memberId, isbn, TransactionType.RETURN
    );
    transactions.add(txn);
    return true;
}

Key validation: Checks member actually borrowed the book — prevents fraudulent returns.

Check Overdue Books — O(t)

Member History — O(t)

public List<Transaction> getMemberHistory(String memberId) {
    return transactions.stream()
        .filter(txn -> txn.getMemberId().equals(memberId))
        .sorted(Comparator.comparing(Transaction::getTimestamp).reversed())
        .collect(Collectors.toList());
}

Optimization: Maintain Map<MemberId, List<Transaction>> for O(1) lookup.

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap books, members stores Safe concurrent reads/writes for lookups
synchronized methods borrowBook(), returnBook() Atomic check-and-update of book status + member borrow list
final fields Book.isbn, Member.id, Transaction.* Immutable identity fields — thread-safe by construction
UUID.randomUUID() Transaction ID, Member ID Thread-safe ID generation without coordination
ArrayList (synced) Transaction log Append-only under synchronized block — safe
"I use synchronized methods for borrow/return to prevent double-booking. Searches use ConcurrentHashMap iteration and don't block writes. At scale, I'd switch to fine-grained locking per book using Guava's Striped<Lock>."

Fine-Grained Locking (Scale Improvement)

class LibraryServiceImpl {
    private final Striped<Lock> bookLocks = Striped.lock(256); // Guava

    public boolean borrowBook(String memberId, String isbn) {
        Lock lock = bookLocks.get(isbn);
        lock.lock();
        try {
            Book book = books.get(isbn);
            if (book.isAvailable()) {
                book.borrow(memberId);
                return true;
            }
            return false;
        } finally { lock.unlock(); }
    }
}

Different books can be borrowed concurrently. Only same-book operations serialize.

12 — Interview Follow-up Q&A

How would you scale for 1 million books and 100K members?
Move from in-memory maps to PostgreSQL with indexes on ISBN, title (GIN for full-text), and author. Add Elasticsearch for faceted search. Use Redis for caching frequently accessed books (key: book:{isbn}, TTL: 1 hour, invalidate on borrow/return). The LibraryService interface stays the same — only the implementation changes.
Why synchronized methods instead of fine-grained locks?
Simplicity for the current scale (<100 concurrent requests). synchronized on the service serializes all borrow/return ops but reads (search) are unaffected. At scale, switch to Striped<Lock> (Guava) keyed by ISBN — different books can be borrowed concurrently. Alternative: optimistic locking with versioning on Book.
How would you implement a book reservation system?
Add a Queue<String> reservationQueue on Book. When a member reserves a borrowed book, add to queue. On return, check if queue has waiters: if yes, set status=RESERVED and notify the next member. Add reservation expiry (24h) using ScheduledExecutorService. If expired, move to next in queue or set AVAILABLE.
How do you handle multiple copies of the same book?
Change Book to track totalCopies, availableCopies, and Map<String, String> copyIdToBorrower. On borrow, decrement available count and assign a copy ID. On return, increment and release the copy ID. ISBN remains the grouping key, but each physical copy has a unique copyId.
Why ConcurrentHashMap instead of HashMap with synchronized?
ConcurrentHashMap allows concurrent reads without locking. Searches (which iterate values) don't block borrow/return. With a plain HashMap + global synchronized, even reads would be serialized. CHM uses fine-grained segment locking internally.
How would you optimize search from O(n) to sub-linear?
Build an inverted index: Map<String, Set<String>> mapping each word to ISBNs. On addBook, tokenize title/author and index each word. On search, tokenize query and intersect result sets. Time: O(w + k) where w = words, k = results. For production, use Elasticsearch with analyzers.
How do you prevent a member from borrowing the same book twice?
The book's status check handles this: once borrowed, status = BORROWED, so book.getStatus() != AVAILABLE returns false. Additionally, the member's borrowedBooks list tracks current borrows. Double-checking both gives defense in depth.
Why is the transaction log append-only?
Audit trail — every borrow/return is permanently recorded. Transactions are never modified or deleted. This enables: (1) borrowing history reports, (2) overdue detection by scanning BORROW transactions without matching RETURN, (3) analytics on popular books. In production, back by an event store or write-ahead log.
How would you calculate late fees in returnBook()?
Find the BORROW transaction for this member+ISBN. Calculate dueDate = borrowTimestamp + 14 days. If returnDate > dueDate, fee = daysOverdue * $0.50/day. Add fee to member account. Can be extracted into a LateFeeCalculator strategy for different fee policies (progressive, capped, first-time waiver).
How would you add due date reminders and notifications?
Maintain a Map<ISBN, LocalDate> of active borrow due dates. Run a ScheduledExecutorService daily that scans for books due within 3 days and sends reminders via a NotificationService interface (email, SMS, push). On overdue, escalate to librarian dashboard.
What's the tradeoff of maxBorrowLimit = 5?
Prevents book hoarding and ensures fair access. Can be configurable per member type: students (3), faculty (10), premium members (15). Stored on Member, checked in canBorrowMore(). Tradeoff: too low limits frustrate power readers; too high causes scarcity. The limit doesn't account for overdue books — an extension would block borrows if any overdue.
How would you add inter-library loans?
Add a LibraryNetwork interface with requestBook(isbn) that queries partner libraries. If found, create a TRANSFER transaction. The book status at source becomes TRANSFERRED. Add a timeout (7 days) for transfer completion. Use async messaging (Kafka) between libraries for reliability.
