Distributed Message Queue // HLD

Kafka-style distributed log: append-only commit log with partitioning, replication, consumer groups, configurable delivery semantics, and zero-copy transfer for million-message throughput.

Commit Log Partitions Replication Consumer Groups Zero-Copy

01 — Requirements

Functional

  • Producers publish messages to named topics
  • Consumers subscribe and consume; support consumer groups for parallel processing
  • Messages persistent — survive broker restarts
  • Ordering within a partition guaranteed
  • Retention — messages kept for N days, not deleted on consumption
  • Replay — consumers can reset offset to re-read old messages
  • Delivery semantics — at-least-once (default), exactly-once (optional)

Non-Functional

  • High throughput — millions of messages/sec
  • Low latency — < 10ms p99 end-to-end
  • Durability — zero message loss (replicated before ack)
  • High availability — no SPOF
  • Horizontal scalability — add brokers to scale linearly

Why Append-Only Is Fast

  • Sequential disk writes — 600 MB/sec on spinning disk, vs 100-200 random IOPS
  • OS page cache — recent writes served from RAM. Consumers near write-head get data from cache, not disk.
  • Zero-copy (sendfile) — data goes directly from page cache to network socket, bypassing user space. No copy.

05 — Producers

Determine partition
With key: partition = hash(key) % num_partitions → same key always same partition (ordering). Without key: round-robin (max throughput).
Find leader
Fetch metadata from any broker: “Who leads topic=orders, partition=2?”. Cached + refreshed periodically.
Send to leader directly
No intermediary. Producers talk directly to partition leaders.
Wait for ack (configurable)
acks=0 fire-and-forget. acks=1 leader confirms. acks=all all ISR replicas confirm (zero data loss).

Batching

Producers accumulate messages per partition:

  Partition 0 batch: [msg1, msg2, msg3, msg4]  → 1 network request
  Partition 1 batch: [msg5, msg6]              → 1 network request

  Config: batch.size (bytes), linger.ms (max wait)
  Fewer round trips → 10-100x throughput improvement

06 — Replication

Topic "orders", Partition 0, Replication Factor = 3

Broker 1 (LEADER):   [msg0][msg1][msg2][msg3][msg4][msg5]  LEO=5
Broker 2 (Follower): [msg0][msg1][msg2][msg3][msg4]        LEO=4
Broker 3 (Follower): [msg0][msg1][msg2][msg3][msg4][msg5]  LEO=5

High Watermark (HW) = 4
  = last offset ALL in-sync replicas confirmed
  = consumers can only read up to HW

Replication Protocol

  1. Producer sends message to leader
  2. Leader appends to local log
  3. Followers pull from leader (periodic fetch)
  4. Each follower appends, acks back to leader
  5. All ISR confirmed → leader advances High Watermark
  6. Leader acks producer (if acks=all)

Failure Handling

ScenarioWhat Happens
Follower falls behindRemoved from ISR. Writes continue. Re-added when caught up.
Leader diesCoordination service detects (heartbeat timeout). New leader elected from ISR. Producers/consumers discover via metadata refresh. < 5s failover.

07 — Consumers

Consumer Group "payment-service" (3 consumers), Topic "orders" (4 partitions)

  Partition 0 → Consumer A
  Partition 1 → Consumer A   (A gets 2 partitions)
  Partition 2 → Consumer B
  Partition 3 → Consumer C

  Each partition → EXACTLY 1 consumer in the group.
  Consumers > partitions → some sit idle.
Poll
“Give me messages from Partition 2, starting at offset 100” → broker returns batch [msg100..msg149]
Process
Application logic: update DB, send email, aggregate, etc.
Commit offset
Tell broker: “processed up to 149”. On crash + restart, resumes from 149.

Consumer Rebalancing

Before: 3 consumers
  P0 → A,  P1 → A,  P2 → B,  P3 → C

Consumer C crashes → rebalance:
  P0 → A,  P1 → B,  P2 → B,  P3 → A

Consumer D joins → rebalance:
  P0 → A,  P1 → B,  P2 → D,  P3 → A

Read Path: Finding Offset 15432

  1. Binary search segment files by name → falls in segment starting at 10000
  2. Open sparse index: [10000→pos:0] [10050→pos:4096] …
  3. Find largest entry ≤ 15432 → approximate file position
  4. Sequential scan in .log from that position to find exact offset

09 — Delivery Semantics

GuaranteeHowTrade-off
At-most-onceCommit offset BEFORE processing. Crash during processing → message skipped.Fast, may lose
At-least-onceCommit offset AFTER processing. Crash after processing but before commit → re-delivered.Safe, may duplicate
Exactly-onceIdempotent producer (dedup by sequence no.) + transactional writes + consumer commits offset in same TX as output.Slowest, hardest
At-least-once is the default. Handle duplicates in application with idempotency (idempotency key, upsert).

Exactly-Once (Kafka Transactions)

1. Producer assigns each message (ProducerID, SequenceNumber)
2. Broker deduplicates: same (PID, SeqNo) → discard
3. Transactional: begin TX → write to N partitions → commit atomically
4. Consumers in "read_committed" mode only see committed messages
5. Consumer commits offset as part of SAME transaction as output writes

14 — Interview Q&A

How does Kafka achieve such high throughput on commodity hardware?

Four pillars: (1) Sequential I/O — append-only log matches disk access patterns (600 MB/s sequential vs random IOPS). (2) OS page cache — data written to page cache first; consumers near the write head read from RAM. (3) Zero-copy (sendfile) — data goes page cache → NIC, bypassing user space. (4) Batching — producers and consumers batch messages, amortising network round-trip overhead.

What happens when a broker crashes?

Coordination service detects via heartbeat timeout. For each partition where the crashed broker was leader, a new leader is elected from the ISR (in-sync replicas). Producers/consumers discover the new leader via metadata refresh (cached, updated on error). Failover typically < 5 seconds. No data loss if acks=all and min.insync.replicas ≥ 2.

How do you guarantee ordering?

Ordering is guaranteed within a partition. Use a partition key (e.g., user_id, order_id) so all related messages go to the same partition via hash(key) % num_partitions. Global ordering across all partitions would require a single partition (kills parallelism), so it’s avoided. With max.in.flight.requests = 1 or idempotent producer, even retries preserve order.

How does exactly-once delivery work?

Idempotent producer: each message tagged with (ProducerID, SequenceNumber). Broker deduplicates on retry. Transactions: producer begins TX → writes to multiple partitions → commits atomically. Consumers in read_committed isolation only see committed messages. Consumer offset committed in the same TX as output → no double processing.

What is the ISR and why does it matter?

ISR (In-Sync Replicas) = the set of replicas that are fully caught up with the leader. When a follower falls behind (slow network, disk issue), it’s removed from ISR. The High Watermark (HW) = the last offset replicated to ALL ISR members. Consumers only read up to HW. With min.insync.replicas=2 and acks=all, a write is only acked when at least 2 replicas have it → survives 1 broker failure with zero data loss.

How does consumer rebalancing work, and what are its problems?

When a consumer joins, leaves, or crashes, a rebalance redistributes partitions. The Group Coordinator (a broker) detects membership change and triggers the protocol. Problem: stop-the-world. During rebalance, ALL consumers in the group stop processing. Solutions: (1) Incremental Cooperative Rebalancing — only migrates partitions that actually move, others continue processing. (2) Static group membership — consumers have stable IDs, short disconnects don’t trigger rebalance.

Why not use a database instead of a message queue?

Databases do random I/O, maintain indices, and support complex queries — all overhead you don’t need. A message queue only does append + sequential read, which is 100–1000x faster. Databases also struggle with millions of concurrent long-poll connections, partition-based parallelism, and multi-consumer fanout. A queue is a specialised tool for this exact access pattern.

What is log compaction and when would you use it?

Instead of deleting old segments by time/size, compaction keeps only the latest message per key. Use cases: (1) Database changelog — consumers get the latest state of every row. (2) Configuration updates — latest config per key. (3) User profiles — latest profile per user_id. It’s like a key-value snapshot built from a stream of updates.

How do you choose the number of partitions?

Partitions = max parallelism. Formula: partitions ≥ max(target_throughput / per_consumer_throughput, max_consumers_in_group). Start with a reasonable number (e.g., 10-50). Too few → bottleneck. Too many → more file handles, longer rebalances, higher ZK/KRaft load. You can increase partitions later, but never decrease (breaks key-based ordering).
