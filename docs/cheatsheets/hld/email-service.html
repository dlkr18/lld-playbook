Distributed Email Service // HLD

Gmail-scale email: SMTP send/receive, Cassandra metadata + S3 blobs + Elasticsearch search, DKIM/SPF/DMARC deliverability, ML spam filtering, conversation threading, push notifications.

SMTP Cassandra S3 Blobs Spam ML Elasticsearch

01 — Requirements

Functional

  • Send emails — internal and cross-domain
  • Receive emails from external senders
  • Read inbox, folders (Sent, Drafts, Trash, Spam, custom labels)
  • Search — full-text across subject, body, sender, attachments
  • Attachments up to 25 MB
  • Read/unread, starring, conversation threading
  • Spam filtering + virus scanning
  • Push notifications on new email

Non-Functional

  • Scale: 1B users, 100B emails/day
  • Availability: 99.99%
  • Durability: zero email loss
  • Latency: inbox load < 500ms, send < 2s
  • Storage: petabytes to exabytes

05 — Send Path (Outbound)

HTTP POST /v1/emails/send
Auth (JWT), validate recipients, size limits, rate limit.
Store in Sent folder
Metadata → Cassandra. Body → S3. Attachments → S3 (content-hash dedup).
Enqueue to Kafka (outbound topic)
Return 202 Accepted immediately.
Outbound SMTP Sender
DNS MX lookup, TLS connect, SMTP deliver, sign with DKIM.
Handle result
Success: mark Delivered. Failure: retry with backoff, bounce after N retries.

06 — Receive Path (Inbound)

Inbound SMTP Receiver (port 25)
SMTP handshake, verify recipient exists, rate limit per IP, enqueue to Kafka.
Spam filter + virus scan
ML classifier + SPF/DKIM/DMARC check + ClamAV. High score → Spam folder.
Parse MIME
Extract headers, body (text + HTML), attachments.
Store
Metadata → Cassandra. Body → S3. Attachments → S3 (dedup).
Index for search
Async write to Elasticsearch. Searchable in 1-5 seconds.
Deliver + notify
Insert into recipient Inbox. Apply user filter rules. Push notification.

07 — Storage Design

Store 1: Metadata DB (Cassandra / Bigtable)

Table: emails_by_user
  Partition Key:  user_id
  Clustering Key: (folder, timestamp DESC)

  user_id | folder | timestamp   | email_id | from     | subject  | is_read | thread_id
  --------+--------+-------------+----------+----------+----------+---------+----------
  bob     | inbox  | 2026-04-29  | e_abc123 | alice@.. | Hello!   | false   | t_001
  bob     | inbox  | 2026-04-28  | e_abc122 | carol@.. | Meeting  | true    | t_002

  "Bob inbox, latest 20" = single partition read. O(1).

Why Cassandra: partition by user = data locality. Clustering by time = sorted. Write-optimised (LSM trees). Billions of rows.

Store 2: Blob Store (S3 / GCS)

S3://emails/e_abc123/body.html  (5 KB)
S3://emails/e_abc123/att_1.pdf  (3 MB)

Cheap ($0.023/GB/month). Content-hash dedup: same PDF to 100 people = stored once. Tiered: hot (SSD, 30d) + cold (Glacier).

Store 3: Search Index (Elasticsearch)

Full-text on subject, body, sender, attachment names. Partitioned by user_id. Eventually consistent (async, 1-5s). Returns email_ids, fetch from Cassandra + S3.

08 — Conversation Threading

Email headers:
  Message-ID:   <abc123@gmail.com>       (unique ID)
  In-Reply-To:  <xyz789@yahoo.com>       (parent email)
  References:   <xyz789@yahoo.com> ...   (full chain)

Threading algorithm:
  1. On receive, check In-Reply-To and References
  2. Match to existing thread --> add to it
  3. No match --> new thread

Thread t_001:
  msg1: "Hello Bob!"        Message-ID: <m1@gmail>
  msg2: "Hi Alice!"         In-Reply-To: <m1@gmail>
  msg3: "How is it going?"  In-Reply-To: <m2@yahoo>

Inbox shows thread as single row: latest msg, participants.

13 — Interview Q&A

Why three separate stores (metadata, blob, search)?

Different access patterns: Metadata (small, frequent) = Cassandra. Blobs (large, occasional) = S3. Search (full-text) = Elasticsearch. One store for all = suboptimal.

How do you handle attachments at scale?

S3 with content-hash key. Same file to 100 users = stored once (SHA-256 dedup). Metadata stores S3 ref. 25 MB limit at gateway. Virus scan before storing.

How does conversation threading work across providers?

RFC 5322: Message-ID, In-Reply-To, References headers. Match In-Reply-To to known IDs. Fallback: normalised Subject + participants + time window.

How does spam filter avoid false positives?

Conservative thresholds, user "Not Spam" feedback loop, allowlists for contacts, domain reputation, periodic precision/recall audits.

IMAP vs POP3 vs HTTP API?

IMAP: sync across devices. POP3: download and delete. HTTP API: proprietary, richer features. Offer IMAP for compatibility + HTTP for own apps.

How to handle marketing blast (10M users)?

Progressive fan-out: batch 10K to Kafka. Separate topic + pool for bulk. IP rotation. Do not delay transactional emails.

SDE-2 vs SDE-3 differentiator?
SDE-2SDE-3
SMTP, 3-store separation, inbox model, basic spam, folders+ SPF/DKIM/DMARC, IP warming, dedup, threading RFC, Cassandra model, tiered storage, fan-out, greylisting, MIME
