Distributed 64-bit unique ID generation: Snowflake bit layout, clock skew handling, machine ID assignment, sequence overflow, k-sortable IDs for B-tree friendliness, comparison with UUID/ticket-server/range allocation.

Snowflake 64-bit K-Sortable UUID v7 ZooKeeper

01 — Requirements

Functional

  • Generate unique IDs — never collide
  • Generated by distributed cluster (no SPOF)
  • IDs roughly time-sortable (newer > older)

Non-Functional

  • Throughput: 10K+/sec/node, scaling to millions globally
  • Latency: < 1ms per ID
  • Highly available: any node generates independently
  • Compact: fits in 64 bits (long) for DB efficiency

02 — Why It's Hard

Single machine: trivial (counter++).

Distributed: must guarantee uniqueness across N machines
             WITHOUT coordination on every request.

NAIVE 1: DB auto-increment
  Single row contention, ~10K/sec, SPOF.

NAIVE 2: Central Redis INCR
  ~100K/sec, still SPOF, network hop per ID.

NAIVE 3: Random
  Collision probability nonzero forever.

GOAL: each server generates IDs LOCALLY but uniqueness is preserved.

04 — Twitter Snowflake (THE CLASSIC)

64-bit ID layout:

+-+---------------------------------------------+----------+----------+
|0|       41 bits: timestamp (ms since epoch)   | 10 bits  | 12 bits  |
| |                                              | machine  | sequence |
+-+---------------------------------------------+----------+----------+
 sign

Total: 1 + 41 + 10 + 12 = 64 bits
FieldBitsRange
Sign1Always 0 (positive long)
Timestamp41~69 years from custom epoch
Machine ID101024 machines (DC + worker)
Sequence124096 per ms per machine

Throughput per machine: 4096 IDs/ms = 4M IDs/sec/node.
Cluster: 1024 nodes × 4M = 4 billion IDs/sec.
Lifetime: ~69 years before timestamp overflow.

05 — Snowflake Implementation

EPOCH = 1577836800000  # 2020-01-01

def next_id():
  ts = current_ms()
  
  if ts < last_ts:
    raise ClockBackwardError("refusing to generate")
  
  if ts == last_ts:
    sequence = (sequence + 1) & 0xFFF  # 12 bits
    if sequence == 0:
      ts = wait_until_next_ms(last_ts)  # sequence exhausted
  else:
    sequence = 0  # new ms → reset
  
  last_ts = ts
  
  return (
    ((ts - EPOCH) << 22) |
    (machine_id   << 12) |
    sequence
  )

Why It's Unique

  • Same machine: different ts → different timestamp bits. Same ts → sequence increments.
  • Across machines: machine_id bits differ, even if ts + sequence collide.
  • Sortable: high bits = timestamp. Newer IDs > older IDs.

06 — Hard Problems

Clock Skew / Backward Jump

NTP can rewind clock → risk of duplicate IDs.

Detection: if ts < last_ts → PROBLEM.

Options:
  1. REJECT + crash (Twitter style) — safest, hits availability
  2. WAIT until clock catches up — ok for tiny skew
  3. LOGICAL CLOCK — counter that only goes up, persisted

Best practice: NTP slew (gradual), NOT step adjustments.

Sequence Overflow

4096 IDs/ms per machine. If you need more in same ms → busy-wait for next ms (rare). If consistently exceeded → tweak bit allocation.

07 — Database Ticket Server (Flickr)

CREATE TABLE Tickets (
  stub CHAR(1) PRIMARY KEY,
  id   BIGINT NOT NULL AUTO_INCREMENT
);

-- Get an ID:
REPLACE INTO Tickets (stub) VALUES ('a');
SELECT LAST_INSERT_ID();

REPLACE = atomic delete + insert → counter increments.

Throughput: ~10K/sec (single row).

For more: TWO ticket servers (odd/even).
  Server A: 1, 3, 5, ...   (offset 0, increment 2)
  Server B: 2, 4, 6, ...   (offset 1, increment 2)

08 — Range Allocation

Each server reserves a BATCH of IDs from central counter.

  Server 1 → "give me 10000" → gets 1-10000
  Server 2 → "give me 10000" → gets 10001-20000

Server hands out IDs in-process (sub-microsecond).
Coordinator hit only every 10K requests.

Pros: very fast local generation, monotonic per server.
Cons: ID gaps on crash (acceptable). Coordinator still bottleneck at extreme scale.

Used by: Stripe (idempotency keys).

09 — Instagram-Style

64-bit ID generated INSIDE Postgres via stored procedure:

  41 bits: ms timestamp (custom epoch)
  13 bits: shard ID (8192 shards)
  10 bits: per-shard sequence (1024 per ms)

CREATE FUNCTION next_id() RETURNS bigint AS $$
DECLARE seq_id bigint; now_ms bigint;
BEGIN
  SELECT nextval('id_seq') % 1024 INTO seq_id;
  now_ms := FLOOR(EXTRACT(EPOCH FROM clock_timestamp()) * 1000);
  RETURN (now_ms << 23) | (shard_id << 10) | seq_id;
END;
$$ LANGUAGE plpgsql;

ID generated at INSERT time inside DB — no separate ID service.

11 — Why K-Sortable Matters

B-TREE PRIMARY KEY INDEXES:

Random IDs (UUID v4):
  Each INSERT → random page of index → page splits → I/O
  Hot pages constantly shift → cache thrash → slow writes.

Sortable IDs (Snowflake / UUID v7):
  Each INSERT → appends to last page (most recent)
  Hot page stays in memory → fast.

UUID v4 inserts can be 5-10x SLOWER than Snowflake on busy indexes.

13 — Interview Q&A

Why not just use UUID v4?

128-bit (vs 64-bit Snowflake) doubles index/memory cost. Random ordering destroys B-tree write performance — can be 5-10x slower on inserts. No time information. UUID v7 (newer) fixes ordering but is still 128-bit.

Why 41 bits for timestamp?

2^41 ms = ~69 years. Long enough for any system. Fewer bits = shorter lifetime. More bits eats into machine ID and sequence. 41 is the sweet spot.

How does Snowflake handle clock going backward?

Detect on each call (ts < last_ts). Twitter's approach: refuse to generate & alert ops. Alternative: wait for clock to catch up if skew is small. Best prevention: NTP slew (gradual) instead of step adjustments. Persist last_ts across restarts.

How are 1024 machine IDs assigned?

Best: ZooKeeper/etcd — machine creates an ephemeral node claiming a free ID. Released on disconnect. Or use K8s StatefulSet ordinal (pods are numbered 0..N). Avoid hostname hashing (risk of collision).

What if I need MORE than 4M IDs/sec/machine?

Sequence is 12 bits = 4096/ms = 4M/sec. To increase: take bits from machine ID (fewer machines but higher per-machine throughput). Or run multiple Snowflake workers per box with different IDs. Or use range allocation for batch generation.

Are Snowflake IDs strictly monotonic?

Within a single machine: yes, strictly. Across machines: no — only "k-sortable" (roughly time-ordered). Two IDs generated at the same ms on different machines are not totally ordered. For most use cases this is fine.

Why is k-sortable enough for most cases?

Strict monotonicity requires global coordination per ID (slow). K-sortable means "newer IDs are mostly bigger" — enough for: (1) DB index locality (writes hit hot page), (2) approximate time-based queries, (3) user-perceived ordering.
