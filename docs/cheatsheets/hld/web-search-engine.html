Web Search Engine // HLD

Google-lite: crawl 100B pages, build a sharded inverted index in batch, serve 100K QPS with <200ms p99 via scatter-gather across thousands of shards, rank with BM25 + PageRank + ML, augment with two-tower ANN for semantic, personalize at the edge.

Inverted Index BM25 PageRank Two-Tower ANN Scatter-Gather

Whiteboard HLD: three pipelines. Crawling (seed URLs → frontier → fetchers → HTML store). Indexing (offline batch: parse, tokenize, build sharded inverted index, compute PageRank, train embeddings + HNSW). Query serving (online: edge cache → query router → spell correct + expand → scatter-gather to all index shards → merge → ranker BM25+PageRank+ML → personalization rerank → snippet generation).

01 — Requirements

Functional

  • Crawl the public web continuously, respect robots.txt
  • Index pages: text, anchor text, freshness, metadata
  • Serve search queries: top-N results with title + URL + snippet
  • Spell correction ("did you mean…")
  • Query expansion (synonyms, stemming)
  • Personalization (history + location reranks)
  • Real-time freshness for news / breaking events

Non-Functional

  • Low latency: <200ms p99 for the full query path
  • High recall + precision: the relevant doc must be in the result, near the top
  • Massive scale: 100B pages indexed, petabytes of inverted index, 100K QPS
  • High availability: 99.99%; partial degraded results better than no results
  • Crawl politely: respect robots.txt, rate-limit per host
  • Freshness: news pages indexed within minutes, not days

01b — API Design

Hot path (user-facing)

GET /search?q=best+pizza+nyc&p=1&hl=en&loc=US
   -> {
        results: [ { url, title, snippet, score, ranks_used }, ... ],
        spellcheck: { corrected: "best pizza nyc" },
        total: 1_240_000,
        latency_ms: 142
      }

GET /suggest?q=best+pi               (typeahead; hit-rate cached)
   -> { suggestions: ["best pizza", "best pizza nyc", ...] }

Internal (between services)

indexShard.search(query, top_k, filters)            -> [(doc_id, local_score), ...]
ranker.score(candidates, features)                  -> [(doc_id, final_score), ...]
ann.knn(query_vec, top_k)                           -> [(doc_id, similarity), ...]
snippet.render(doc_id, query_terms)                 -> string

# Indexing pipeline (offline)
crawler.fetch(url) -> {html, status, headers, ts}
indexer.build(doc_id, html) -> postings, embedding
The non-obvious contract: queries are non-strict. Some shards may time out (>100ms); we return partial results with a "results may be incomplete" flag rather than failing the whole query.

03 — Why It's Hard

  • Sub-200ms over a petabyte: the only way is massive parallelism (thousands of shards) + WAND-style early termination
  • Long tail of queries: 50%+ of daily queries are never seen before; caching helps only the head
  • Ranking is the product: BM25 alone is mediocre; PageRank + 100s of ML features are what made Google
  • Freshness vs index batch: batch-built indices are days stale; news/sports need a separate realtime index
  • Spam and SEO games: adversaries tune to your ranking signals; must keep evolving
  • Crawl politeness: hit one site too hard and they block you (or you DOS them)
  • Personalization conflicts with caching: if every result is per-user, your cache is useless

05 — Crawler

The crawler is its own beast — see the Web Crawler cheatsheet for the full design. Highlights relevant here:

  • Frontier: priority queue of URLs to fetch; priority = PageRank-ish + freshness need + politeness budget
  • Fetcher fleet: thousands of workers; per-host rate limiting; honor robots.txt
  • Dedup: by content hash (skip pages identical to ones already indexed) and URL canonicalization
  • HTML store: raw HTML kept in S3-like object storage so re-indexing doesn't need a re-crawl
  • Refresh policy: pages re-crawled at a rate proportional to change frequency (news = minutes, encyclopedia = months)

06 — Indexer (Offline Batch)

The pipeline

  1. Parse HTML → strip boilerplate, extract title/anchor/body/meta
  2. Tokenize → word boundaries, lowercase, language detection
  3. Normalize → stem ("running" → "run"), Unicode-fold, drop stopwords
  4. Emit postings → (term, doc_id, tf, positions, fields_mask)
  5. Shuffle + sort by term (or by doc, depending on partition strategy)
  6. Write posting lists → compressed binary, sorted by doc_id, with skip pointers

MapReduce / Spark

Map:    (doc_id, html) -> (term, doc_id, tf, positions, fields_mask)
Reduce: term -> [postings sorted by doc_id]   # the posting list
Output: per-shard inverted index files

Side outputs

  • PageRank: separate iterative job over link graph (50–100 iterations to converge)
  • Anchor text aggregation: for each dst_doc, collect anchor text of inbound links
  • Document embeddings: doc-tower from two-tower model; bulk-write to ANN index (HNSW)
Index build is massively parallel and runs as a periodic batch (e.g., daily for the bulk index, separate near-realtime path for fresh news).

07 — Inverted Index & Sharding

Zoom: inverted index = term → [posting…]. Two sharding strategies. Term-partitioned: each shard owns a slice of the vocabulary. Doc-partitioned: each shard owns a slice of doc IDs and a full inverted index for those docs. Google uses doc-partitioned.

Posting list structure

apple -> [ (doc:7,   tf:3, pos:[1,15,82], fields:title|body),
           (doc:42,  tf:1, pos:[7],        fields:body),
           (doc:105, tf:2, pos:[3,20],     fields:body),
           ... ]
banana -> [ ... ]
Storage: varint/PFOR-encoded delta-gaps + skip pointers for AND/OR.

Term-partitioned vs doc-partitioned

StrategyHowProsCons
Term-partitionedhash(term) → shard; shard owns whole posting listPer-term lookup hits one shardMulti-term query: heavy cross-shard intersection; hot terms = hot shards
Doc-partitionedhash(doc_id) → shard; each shard has full inverted index for its docsMulti-term AND/OR happens locally; load balances naturallyEvery query fans out to every shard (huge scatter-gather)
Google uses doc-partitioned. Scatter-gather is cheaper than cross-shard joins, and you can early-terminate with WAND/MaxScore on each shard locally.

08 — Query Path

Edge cache check
Hot queries (~30% of traffic) hit edge cache with personalized-result-cache key. Miss falls through to router.
Parse + analyze
Tokenize, normalize, detect language, classify intent (informational / navigational / transactional).
Spell correct + expand
Spell: edit distance + query log frequency. Expand: synonyms, stemming, multi-language. Rewrite the query.
Scatter-gather to all index shards
Fan out to thousands of doc-partitioned shards. Each shard intersects posting lists, computes BM25, applies WAND/MaxScore early termination, returns local top-K.
Merge
Heap-merge top-K from each shard into global top-K (~10K candidates).
Tier-1 ranker
BM25 + PageRank + freshness; reduce to top 1K.
Tier-2 ML ranker
Deep model with hundreds of features (CTR, dwell time, query-doc relevance, click history) → top 50.
Personalization rerank
User history + location + language → top 20.
Diversity + snippet
Diversity rerank (no same-domain monopoly). Snippet generator finds best matching passage, bolds matched terms.
Latency budget is brutal. The tail latency of the slowest shard drives p99; we skip outlier shards beyond ~100ms and live with slight recall loss.

09 — Ranking

Zoom: ranking funnel. Candidate generation (millions → 10K) by inverted-index AND ANN. Tier-1 (BM25+PageRank) 10K → 1K. Tier-2 ML deep model 1K → 50. Personalization 50 → 20. Diversity rerank.

10 — Two-Tower / ANN Semantic Retrieval

The model

  • Query tower: NN that maps query → vector[d]
  • Doc tower: NN that maps doc text+metadata → vector[d]
  • Training: positive pairs (clicked query/doc) and hard negatives; loss = contrastive (in-batch negatives + sampled negatives)
  • At index time, embed every doc, push into HNSW ANN index
  • At query time, embed query → HNSW.knn(query_vec, top_k) in <10ms

Why have it alongside inverted index?

  • Inverted index does lexical matching: misses synonyms, paraphrases, multi-language
  • Two-tower handles "best place to eat ramen in NY" matching a doc titled "top NYC noodle shops"
  • Hybrid: union the two candidate sets, let the ranker arbitrate

11 — Caching, Spell, Snippets

Cache layers

  • Edge result cache: full result page for hot queries; key = (query, lang, country, signed-out)
  • Query result cache: top-K doc_ids for the query; per ranker version
  • Posting-list cache: hottest posting lists in RAM at each shard
  • Cache stampede prevention: see Distributed Cache for the techniques

Spell correction

Generate candidates: Levenshtein edit distance ≤ 2 over vocab
Rank candidates:    P(correction | typo) ~ P(correction) * P(typo | correction)
                    Use query log frequency as prior; noisy-channel for likelihood

Snippet generation

  • Find best-matching passage in doc containing query terms
  • Bold matched terms; clip to ~160 chars; pick the densest window
  • Cache snippets per (doc, query-bigram) to avoid recomputation

12 — Scaling

  • Thousands of doc-partitioned shards — the scatter-gather is the work-horse
  • Replicas per shard (3×) for availability and read-throughput
  • Hot-query edge cache absorbs ~30% of traffic before it ever hits the router
  • Geographic data centers serving regional queries with local indices + cross-region failover
  • Realtime news index: separate small fresh index built every few minutes; merged into results
  • Tail-latency mitigation: tied requests (fire to two replicas, take whichever returns first), skip slowest shard, return partial
  • Index update: build new index version offline, swap atomically via shard-level cutover (blue-green)

13 — Edge Cases

  • Slow shard → skip after 100ms; flag result as partial; warn metric
  • Query with zero results → relax query (drop quotes, broader stemming) and retry
  • Malicious query patterns (DOS) → rate limit per IP / user; return cached generic results for abusers
  • Spam pages → offline spam classifier marks docs; rank-pushes them down or removes from index
  • Right-to-be-forgotten → deny-list of doc URLs; filter at query time before snippet generation
  • News surge (election, earthquake) → realtime index promoted, normal index downweighted for freshness
  • Crawler blocked by host → backoff, lower priority, try cached versions, eventually drop
  • Index swap mid-traffic → blue-green at shard level; gradual rollout with canary; rollback on quality drop
  • Personalization signal stale → fall back to demographic / coarse signal; never block the response

15 — Interview Q&A

How do you go from a query to the right document in <200ms over 100B pages?
Inverted index sharded doc-wise across thousands of nodes. Scatter-gather: each shard intersects posting lists for its slice of docs, applies BM25 + WAND/MaxScore for early termination, returns top-K. Merger heap-merges results. Then ranker tiers. Tail-latency mitigation (skip slow shards, tied requests) keeps p99 sane.
Term-partitioned or doc-partitioned index? Why?
Doc-partitioned. Term-partitioned needs cross-shard intersection for multi-term queries (expensive and serial) and creates hot shards for popular terms. Doc-partitioned does all intersection locally on each shard and load balances naturally; the cost is fanning out to all shards, but scatter-gather is cheap with modern networking and WAND early termination.
What's BM25 and why not just TF-IDF?
BM25 is TF-IDF with two improvements: (1) saturating TF — the 100th occurrence shouldn't keep increasing score; (2) document-length normalization — a term in a 100-word doc is more meaningful than in a 10K-word doc. Empirically much stronger than raw TF-IDF on standard IR benchmarks. Default in Lucene/Elasticsearch.
How does PageRank fit in?
PageRank is precomputed offline by iterating over the link graph (a doc's rank = damped sum of inbound docs' ranks). Stored per doc_id. At query time it's just a feature input to the ranker, combined with BM25 and dozens of other signals. By itself it's static and slow to update; it's a feature, not the answer.
How do you handle "best place to eat noodles in NYC" matching "top NYC ramen shops"?
Two-tower neural retrieval. Train query tower and doc tower jointly on click data; at index time embed all docs into an ANN index (HNSW). At query time, embed the query and KNN-search the ANN. Union with the inverted-index candidates and let the ranker arbitrate. Captures synonyms and paraphrase that pure lexical matching misses.
How do you keep results fresh (news, breaking events)?
Separate realtime index built every few minutes from a fast-path crawler that targets news/social. At query time, results are blended from the bulk index + realtime index, with a freshness boost for queries that look time-sensitive. Bulk index is rebuilt daily; realtime index is updated continuously.
What's WAND / MaxScore and why use it?
Algorithms that early-terminate when the remaining docs can't possibly beat the current top-K. Each posting list pre-stores its max possible BM25 contribution; the algorithm tracks an upper bound and skips docs whose upper bound < current threshold. Drops a typical multi-term query's work by 10–100×.
How does personalization not destroy caching?
Cache the impersonal results (top-K doc_ids from ranker) per query; do the personalization rerank cheaply after the cache hit, on a much smaller candidate set. Cache key omits user_id; user_id is only an input to the final rerank. ~80% of the work is cached; only the last 20% is personal.
How would you add spell correction?
Two-stage. (1) Candidate generation: edit-distance-2 candidates from the vocabulary (BK-tree or symspell). (2) Ranking: P(correction|typo) ∝ P(correction) * P(typo|correction). Use query log frequencies as a prior, noisy-channel for likelihood. Threshold confidence to decide auto-correct vs suggest.
How do you handle adversarial SEO and spam?
Offline spam classifier trained on signals like keyword stuffing, link-farm patterns, doorway pages, content quality. Flagged docs are rank-pushed down or removed from index entirely. Continuously evolves — once SEOs learn the signal, you must add new ones. Manual review and penalties for repeat-offender domains.
How is the index updated?
Bulk index rebuilt as a periodic batch job (daily). New version built offline on a parallel cluster; shard-by-shard atomic swap (blue-green) gates rollout with quality canaries. Realtime increment for fresh news lives in a separate small index that's merged at query time. No incremental updates to the bulk inverted index — full rebuilds are simpler and cheaper at scale.
What's the hardest production problem?
Probably ranking quality at the tail of the query distribution. The head (hot queries) is cached and easy; the tail (50% of daily queries never seen before) is where ranking has to generalize. Combined with the adversarial SEO arms race and personalization-vs-cache tension, ranking is where the engineering and ML teams really live.
