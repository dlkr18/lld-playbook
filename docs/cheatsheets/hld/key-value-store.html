01 — Requirements

Functional

  • API: put(key, value), get(key), delete(key)
  • Keys/values are arbitrary bytes (max value ~10 MB)
  • Range queries optional (depends on data model)

Non-Functional

  • Highly available: always responds
  • Low latency: p99 < 10ms
  • Scalable: billions of keys, millions QPS
  • Tunable consistency: strong vs eventual
  • Durable + Fault tolerant

05 — Partitioning (Consistent Hashing)

Hash ring with VIRTUAL NODES (150 per server).
hash(key) → walk clockwise → owner.

Properties:
  • Add/remove server → only K/N keys move
  • Vnodes balance load (~7% std dev)
  • No central routing

See Consistent Hashing cheatsheet for full details.

06 — Replication

Replication factor N = 3.

Key "user_123" hashes to position X.
Walk clockwise → pick N UNIQUE physical servers:
  Server A → primary
  Server B → replica
  Server C → replica

Called "preference list" in Dynamo.

Multi-region: replicas across DCs/AZs.
Cross-region async (latency too high for sync).

07 — Quorum-Based Reads/Writes (KEY)

N = replicas. W = write quorum. R = read quorum.

GUARANTEE: W + R > N → STRONG consistency
  (pigeonhole: write & read sets overlap on >=1 replica)

EXAMPLES (N=3):
  W=2, R=2  → quorum, tolerates 1 down. STANDARD.
  W=3, R=1  → all-write, fast read. Unavailable if any down.
  W=1, R=3  → fast write, slow read.
  W=1, R=1  → fastest, EVENTUAL (W+R=2 < 3).

DynamoDB default: N=3, W=2, R=2 (quorum)
                  Eventually-consistent reads: R=1
Write path (W=2):
  Coord → Replica A: ack
        → Replica B: ack    → 2 acks → SUCCESS
        → Replica C: (slow, async)

Read path (R=2):
  Coord → A: v1
        → C: v2 (stale)
  Compare timestamps → v1 latest → return v1
  Async: read repair C to v1.

08 — Write Path

Coordinator receives put()
Hash key, find preference list (N replicas).
Forward to all N replicas in parallel
Each replica: WAL + MemTable
Append to Write-Ahead Log on disk, update in-memory sorted MemTable, ack.
Wait for W acks → SUCCESS
Remaining replicas write async.
Background: flush MemTable to SSTable
When MemTable full (e.g., 64 MB), flush as immutable sorted file.

09 — LSM Tree Storage

LSM = Log-Structured Merge Tree.
Used by Cassandra, RocksDB, LevelDB, HBase, BigTable.

WHY?  B-trees do random in-place writes (slow at high QPS).
      LSM does SEQUENTIAL writes only → 100x faster writes.

Components:
  +-----------------------------+
  |  In-memory MemTable (64 MB) |  ← active writes (skip list)
  +-------------+---------------+
                | flush
                v
  +-----------------------------+
  |  Level 0: SSTables          |  unsorted files
  +-------------+---------------+
                | compaction
                v
  +-----------------------------+
  |  Level 1: SSTable (10x L0)  |
  +-------------+---------------+
                | compaction
                v
  +-----------------------------+
  |  Level 2: SSTable (10x L1)  |
  +-----------------------------+

Each SSTable: sorted keys, immutable, per-file index + Bloom filter.

WAL: every write logged to disk BEFORE MemTable.
     Crash → replay WAL.

READ: MemTable → L0 → L1 → ... (Bloom filter skips most files).
COMPACTION: merge SSTables, drop overwritten/deleted entries.

11 — Anti-Entropy (Merkle Trees)

Problem: after partition heals, replicas may have diverged.
  Naive: compare every key → TBs of network.
  Smart: Merkle tree (hash tree).

           Root H
           /     \
        H_left  H_right
        /  \    /   \
       hashes of subranges, leaves = hash(key, value)

Comparison:
  1. Exchange ROOTs. Equal → identical, done.
  2. Different → recurse into mismatching subtree.
  3. O(log N) network traffic to find exact diffs.

For 10B keys: ~30 hashes exchanged to find divergence!

Used by Cassandra (anti-entropy repair), DynamoDB, Git, BitTorrent.

14 — Tombstones (Deletes in LSM)

SSTables are IMMUTABLE — can't update in place.

DELETE = WRITE a TOMBSTONE marker.

  put("user_123", "Alice")  → MemTable: { user_123: Alice }
  delete("user_123")        → MemTable: { user_123: TOMBSTONE }

On read: tombstone seen → return "not found".
On compaction: tombstone + data dropped together.

Tombstones must persist long enough to override stale data
on a node that was down.
  Cassandra: gc_grace_seconds = 10 days default.

17 — Interview Q&A

How do you achieve strong consistency in an AP system?

Set W + R > N. Pigeonhole: read and write quorums overlap on at least one replica, which has the latest value. With N=3, W=2, R=2 you tolerate 1 failure and still get strong reads.

Why LSM trees over B-trees?

LSM does SEQUENTIAL disk writes (append to WAL + sorted flush of MemTable). B-trees do RANDOM in-place writes. At high write QPS, LSM is 10-100x faster. Reads are slower (must check multiple SSTables), but Bloom filters mitigate.

How do vector clocks handle conflicts?

Each write tags its value with a vector {(node, counter), ...}. To compare: V1 < V2 if every entry of V1 ≤ V2. If neither dominates → concurrent → CONFLICT. The store returns BOTH versions (siblings) and the application merges (e.g., shopping cart union).

Explain hinted handoff vs read repair.

Hinted handoff: on write, target replica is down → coordinator stores a HINT, satisfies W with others. When target recovers, hint is delivered. Prevents data loss during transient failures. Read repair: on read, mismatch detected → coordinator async-updates the stale replica. Both work together with periodic Merkle tree anti-entropy.

Why are Merkle trees efficient for anti-entropy?

Comparing 10B keys directly = TBs of network. Merkle tree: exchange root hash first (1 hash). Equal → done. Different → recurse into mismatching subtree. Logarithmic network cost — ~30 hashes for 10B keys.

What's the trade-off of W=3, R=1?

Strong reads (any single replica has the latest) and very fast reads. But writes are slow (wait for ALL replicas) and unavailable if ANY replica is down. Good for read-heavy, write-rare workloads.

How does gossip propagate state?

Each node picks a random peer per second, exchanges membership view + heartbeats. After ~log(N) rounds, all nodes converge (epidemic propagation). Failure detection via phi-accrual: track inter-arrival times of heartbeats, mark DOWN when suspicion crosses threshold.
