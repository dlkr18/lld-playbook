Ad Click Aggregator // HLD

Real-time ad click event aggregation: Kafka ingestion, Flink streaming with tumbling windows and watermarks, ClickHouse/Druid for OLAP, batch reconciliation for billing accuracy, and click fraud detection.

Flink ClickHouse Exactly-Once Watermarks Lambda Arch

01 — Requirements

Functional

  • Aggregate clicks in real-time (count per ad per minute/hour/day)
  • Filter by ad_id, campaign_id, advertiser_id, country, device_type
  • Near-real-time dashboards for advertisers
  • Billing queries — exact click counts for charging
  • Click fraud detection — bots, repeated clicks, anomalies
  • Time-series data with configurable granularity (1min, 5min, 1hr)

Non-Functional

  • Scale — ~10 billion clicks/day (~115K/sec avg, ~300K/sec peak)
  • Low latency — aggregated data available within ~1 minute
  • Accuracy — billing must be exact (no lost or double-counted clicks)
  • Availability — always accept clicks (revenue at stake)
  • Idempotency — same click counted exactly once
  • Query latency — dashboard queries < 200ms

05 — Kafka Ingestion

Topic Design

Topic: raw-clicks
  Partitions: 50-100
  Key: ad_id  (all clicks for same ad → same partition → same Flink worker)
  Replication: 3x
  Retention: 7 days

Why partition by ad_id?
  All events for the same ad go to the same Flink instance.
  That instance maintains the COMPLETE count — no distributed coordination.

Dual Consumers

  • Consumer Group 1: Flink (real-time aggregation)
  • Consumer Group 2: S3 sink (archive raw events for batch reconciliation)

Kafka’s consumer groups allow multiple independent readers of the same topic.

06 — Flink Real-Time Aggregation

What Flink Does

Input (raw events):
  { click_id: "c1", ad_id: "ad_123", ts: 10:00:01, country: "US", device: "mobile" }
  { click_id: "c2", ad_id: "ad_123", ts: 10:00:03, country: "US", device: "desktop" }
  { click_id: "c3", ad_id: "ad_456", ts: 10:00:02, country: "UK", device: "mobile" }

Flink: tumbling window 1 min, group by (ad_id, minute, country, device)

Output (aggregated):
  { ad_id: "ad_123", window: "10:00", country: "US", device: "mobile",  count: 1 }
  { ad_id: "ad_123", window: "10:00", country: "US", device: "desktop", count: 1 }
  { ad_id: "ad_456", window: "10:00", country: "UK", device: "mobile",  count: 1 }

Tumbling Window

|----1 min----|----1 min----|----1 min----|
 10:00-10:01   10:01-10:02   10:02-10:03

Fixed-size, non-overlapping. Events fall into exactly one window
based on event timestamp. At window close, emit aggregated count.

Event Time vs Processing Time

Event TimeProcessing Time
WhatWhen click actually happened (client timestamp)When Flink processes it
ProblemLate-arriving events (network delay)Doesn’t reflect reality (out-of-order)
SolutionWatermarks track progress of event timeNot used for accurate aggregation
Always use event time for ad click aggregation. Processing time would miscount clicks into wrong windows.

Exactly-Once in Flink

  • Checkpointing: periodically snapshot state (counts, windows) to S3/HDFS
  • If worker crashes, restart from last checkpoint
  • Combined with Kafka transactional producer → exactly-once end-to-end

07 — Watermarks & Late Events

Watermark: Flink’s way of saying “I believe all events with timestamp < W have arrived.” Used to decide when to close a window.

How Watermarks Work

Strategy: watermark = max_event_time_seen - allowed_lateness (e.g. 10s)

Events arrive:  ts=10:00:01, ts=10:00:05, ts=10:00:02 (out of order!)

After ts=10:00:58:
  Watermark = 10:00:58 - 10s = 10:00:48
  Window [10:00 - 10:01] still open

After ts=10:01:12:
  Watermark = 10:01:12 - 10s = 10:01:02
  Watermark passed 10:01:00 → Window [10:00 - 10:01] FIRES!
  Emit aggregated count for that window.

Handling Late Events

StrategyHowTrade-off
DropDiscard events arriving after watermarkSimple but loses accuracy
Allowed latenessKeep window open extra N seconds; re-emit updated countMore accurate, uses more memory
Side outputRoute late events to late-events topic → reconcile in batchBest of both: real-time is fast, batch fixes gaps
Recommended: Side output for late events + nightly batch reconciliation. Real-time stays fast; billing stays accurate.

08 — Aggregation Database

Option 1: ClickHouse

CREATE TABLE ad_clicks_agg (
  ad_id        UInt64,
  campaign_id  UInt64,
  window_start DateTime,
  country      LowCardinality(String),
  device       LowCardinality(String),
  click_count  UInt64
) ENGINE = SummingMergeTree()
ORDER BY (campaign_id, ad_id, window_start, country, device);

SummingMergeTree: automatically sums click_count for rows with same key during background compaction. Late corrections merge naturally.

Multi-Granularity Rollups

Store finest granularity (1 minute). Roll up periodically:

  ...

Coarser data kept longer. Fine data TTL’d after 7-30 days.

10 — Batch Reconciliation (Billing Accuracy)

Billing must be exact. Real-time is fast but ~99.9% accurate. Batch path is the source of truth.
Lambda Architecture:
  Speed layer (Flink)  → fast, near-accurate
  Batch layer (Spark)  → slow, 100% accurate
  Serving layer (ClickHouse) → merged view

Nightly reconciliation (2 AM):
  1. Spark reads ALL raw events for yesterday from S3
  2. Recompute exact counts per (ad_id, hour, country, device)
  3. Compare with ClickHouse aggregates
  4. If mismatch > threshold:
     a. Write correction to ClickHouse
     b. Alert if discrepancy is large
  5. Billing system reads from RECONCILED data

11 — Click Fraud Detection

Fraud TypeDetectionAction
Repeated clicks (same user, same ad)Count per (user_id, ad_id) in sliding window. >3 in 1 min = suspiciousMark as fraud. Don’t bill.
Bot trafficPerfect intervals, no mouse data, known bot IPs. ML model.Filter at receiver or Flink.
Click farmsMany clicks from same IP range / geo / device fingerprintAnomaly detection on IP/geo clusters.
Competitor clickingSame ads targeted, specific times. Cross-ad correlation.Hard to catch. Pattern analysis.

Flink Fraud Pipeline

raw-clicks → Flink fraud detector (parallel stream):
  - Sliding window per (user_id, ad_id): count > 3/min → flag
  - Sliding window per (ip, ad_id): count > 10/min → flag
  - ML model scoring on features

Output: flagged-clicks topic → review pipeline
Unflagged clicks → normal aggregation pipeline

12 — Scaling

Click Receiver:  Stateless, auto-scale on QPS (300K/sec peak)

Kafka:           50-100 partitions, key=ad_id
                 3x replication, 7-day retention

Flink:           Parallelism = Kafka partitions
                 Each instance handles subset of ad_ids
                 Checkpoint to S3 every 30s
                 Auto-scale on consumer lag

ClickHouse:      Sharded by campaign_id or ad_id
                 2-3 replicas for read scaling
                 ~288 GB/day minute-level aggregates

Query Service:   Stateless, Redis cache (TTL 60s)

Why Partition by ad_id End-to-End?

All events for the same ad go to the same Kafka partition → same Flink worker → that worker maintains the complete count for that ad without distributed coordination. This is the key to accurate counts without distributed transactions.

13 — Deep Dives

Tumbling vs Sliding vs Session Windows

WindowHowUse Case
TumblingFixed size, non-overlapping: [0-1min][1-2min]Click counts per minute (our case)
SlidingFixed size, overlapping: every 30s emit last 1 minMoving averages, trend detection
SessionGap-based: window closes after N sec of inactivityUser session analysis

Flink Checkpointing

Every 30 seconds:
  1. Flink coordinator triggers checkpoint barrier
  2. Barrier flows through the dataflow graph
  3. Each operator snapshots its state to S3:
     - Window contents (partial aggregates)
     - Kafka consumer offsets
  4. On success, checkpoint is committed

If worker crashes:
  1. Restart from last successful checkpoint
  2. Re-read Kafka from checkpointed offset
  3. Reprocess events (idempotent due to state restore)
  4. No duplicates, no data loss

SummingMergeTree (ClickHouse)

A ClickHouse table engine that automatically sums numeric columns for rows with the same primary key during background merge operations.

Insert:  (ad_123, 10:00, US, 245)
Insert:  (ad_123, 10:00, US, 12)   ← late correction

After merge: (ad_123, 10:00, US, 257)  ← auto-summed!

No UPDATE needed. Just INSERT corrections.
ClickHouse merges them in the background.

Parquet File Format (S3 Archive)

Parquet is a columnar file format optimised for analytics workloads. Raw events stored as Parquet in S3:

  • Columnar: Spark reads only needed columns (e.g. ad_id + count), skips the rest
  • Compressed: ~5–10x smaller than JSON/CSV
  • Partitioned: s3://clicks/date=2026-04-04/hour=10/*.parquet — Spark skips irrelevant partitions
  • Predicate pushdown: filter conditions pushed into file reader

Idempotency Across the Pipeline

LayerMechanism
Click ReceiverRedis SETNX on fingerprint (dedup at ingestion)
KafkaIdempotent producer (enable.idempotence=true) prevents duplicate writes
FlinkExactly-once via checkpointing + Kafka transactions
ClickHouseSummingMergeTree (idempotent by design — re-inserting same data just re-merges)

14 — Interview Q&A

Why not just query raw events directly? Why pre-aggregate?

Raw data is 5 TB/day. Scanning that for every dashboard query would take minutes and cost a fortune. Pre-aggregation compresses 10B events into ~2M rows per minute (~288 GB/day) — 20x smaller, queryable in milliseconds.

How do you handle late-arriving events?

Watermarks with allowed lateness (e.g. 10s). Events arriving within the grace period update the window count. Events arriving after → side output to a late-events Kafka topic, reconciled in the nightly Spark batch job. Real-time stays fast; billing stays accurate.

How do you ensure exactly-once counting for billing?

Real-time path: Flink checkpointing + Kafka transactional producer = exactly-once within the stream. Batch path: nightly Spark recomputes from raw events in S3 (source of truth). Reconciliation corrects any discrepancies. Billing reads from reconciled data.

Why partition Kafka by ad_id?

All events for the same ad go to the same partition → same Flink instance. That instance maintains the complete count for that ad in local state. No distributed coordination, no double-counting, no missed events. This is the key design decision.

How do you detect click fraud in real-time?

Flink runs a parallel fraud detection stream: sliding windows per (user_id, ad_id) and per (ip, ad_id) flag anomalies. ML model scores features (click rate, session duration, device fingerprint). Flagged clicks go to a separate topic for review and are excluded from billing counts.

What is the Lambda Architecture and why use it here?

Speed layer (Flink): real-time, ~99.9% accurate. Batch layer (Spark over S3): 100% accurate, source of truth. Serving layer (ClickHouse): merged view. Used because billing requires exact accuracy that real-time alone can’t guarantee (late events, edge cases).

What if Flink crashes? Do we lose data?

No. Flink checkpoints state (window contents + Kafka offsets) to S3 every 30s. On restart, it resumes from the last checkpoint and replays Kafka from the checkpointed offset. Combined with Kafka retention (7 days), no data is lost.

How does SummingMergeTree handle corrections?

Just INSERT a new row with the correction delta. ClickHouse’s background merge automatically sums rows with the same primary key. No UPDATE needed. Example: insert (+3 clicks) correction → merged with original count during compaction.
