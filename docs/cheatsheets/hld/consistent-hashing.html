Consistent Hashing // HLD

Distributed key placement on a circular hash ring. Adding/removing a server moves only K/N keys instead of nearly all. Virtual nodes balance load. Used by Cassandra, DynamoDB, Memcached, CDNs.

Hash Ring Virtual Nodes K/N Movement DynamoDB Cassandra

01 — The Problem

NAIVE: server_id = hash(key) % N

  N=4: hash("user_123") % 4 = 3 → server 3
  
Add 1 server (N: 4 → 5):
  hash("user_123") % 5 = 4 → different server!
  
  ~80% of keys remap. Massive cache miss / data migration.
Goal: when topology changes, move only K/N keys, not nearly all.

02 — Core Idea: A Ring

Hash space = circular [0, 2^32).

Place SERVERS on the ring (hash their IDs).
Place KEYS on the ring (hash them).

Each key is owned by the FIRST server clockwise from its position.

                      0/2^32
                        |
                        |
              +---------+---------+
              |                   |
        Server A                  |
        (hash=10M)                |
              |                   | Server B
              |                   | (hash=2.7B)
              |                   |
              +---------+---------+
                        |
                  Server C
                  (hash=1.3B)

Key "user_123" → hash = 500M → clockwise → Server C
Key "order_99" → hash = 2.0B → clockwise → Server B

03 — Why It Works

Add Server D (hash = 800M):

  Only keys in (10M, 800M] move from C → D.
  All other ranges unchanged.
  
  Total moved: ~1/N of keys (where N = current server count).

Remove Server B:
  Its range merges into the next clockwise server.
  Only B's keys move.

KEY PROPERTY: K/N movement on topology change.

04 — The Skew Problem

3 servers hash to:
  A → 100M
  B → 110M    (close to A)
  C → 3,000M

Result:
  A handles ~32% of keys
  B handles 0.2% of keys
  C handles ~67% of keys

LOAD HORRIBLY UNBALANCED.

Hash function rarely places servers equidistantly.
With few real servers, skew is severe.

05 — The Fix: Virtual Nodes

Place each PHYSICAL server at MANY positions (100-200 vnodes).

Server A → hash("server_A_0"), hash("server_A_1"), ... hash("server_A_199")
Server B → same: 200 vnodes
Server C → same: 200 vnodes

Total: 600 vnodes interleaved on the ring.

Law of large numbers → each REAL server owns ~1/N of ring.
Vnodes / serverStd deviation of load
1~30-40% (terrible)
100~10% (acceptable)
200~7% (good)
1000~3% (excellent)

Production sweet spot: 100-200 vnodes per physical server.

06 — Lookup Algorithm

Compute hash
h = hash("user_123") = 1,500,000,000
Sorted vnode list
[50M:A, 75M:B, 100M:C, ..., 1,520M:B, 1,580M:A, ...]
Binary search clockwise
First entry >= 1,500,000,000 → 1,520M:B. Wrap around if past max.
Return server
Server B owns "user_123". Time: O(log V).

07 — Implementation

import bisect, hashlib

class ConsistentHashRing:
    def __init__(self, vnodes_per_server=150):
        self.v = vnodes_per_server
        self.hashes = []          # sorted hashes
        self.ring = {}            # hash → server

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2**32)

    def add_server(self, server):
        for i in range(self.v):
            h = self._hash(f"{server}#{i}")
            bisect.insort(self.hashes, h)
            self.ring[h] = server

    def get_server(self, key):
        h = self._hash(key)
        idx = bisect.bisect_right(self.hashes, h)
        if idx == len(self.hashes):
            idx = 0
        return self.ring[self.hashes[idx]]

Use MD5 / MurmurHash3 for uniformity. TreeMap (Java) / std::map (C++) / sorted slice + bisect (Python).

08 — Add / Remove Servers

OperationEffect
Add server DCompute 150 hash positions. Insert into ring. Each vnode steals a small range from its clockwise neighbour. Total moved: ~K/N.
Remove server BRemove all 150 of B's vnodes. Each range merges to clockwise neighbour. Only B's keys move.
With vnodes, the moved keys spread across many servers, not just one neighbour.

09 — Replication

Replication factor R = 3.

Key "user_123" → hash position X.
Walk clockwise from X, picking UNIQUE physical servers:
  
  vnodes from X: A, B, A, C, B, D, A
  Replicas:      A (primary), B (1st), C (2nd)
  (skip duplicates: A, B were already chosen)

When a server fails:
  Reads fall back to its replicas.
  Ring redistributes its primary keys to neighbours.
  Replicas re-replicated to maintain R.

Used by Cassandra, DynamoDB.

12 — Common Pitfalls

  • No vnodes → 30%+ load imbalance. Always use 100-200.
  • Bad hash function → clustered vnodes. Use MD5 / MurmurHash3.
  • Hot keys still concentrate on one server. Replicate hot keys or use bounded-load CH.
  • Failed node not rebalanced → neighbour overloaded. Vnodes spread the impact.
  • Metadata sync — nodes need ring topology agreement (ZooKeeper, etcd, gossip).

13 — Interview Q&A

Why not just hash mod N?

Adding/removing a server remaps ~all keys. Consistent hashing moves only K/N. Critical for caches (avoid cold start) and distributed databases (minimise migration).

Why virtual nodes?

Single hash position per server → uneven ring distribution. 100-200 vnodes spread each server uniformly. Std deviation drops from ~35% to ~7%.

What's the lookup time?

O(log V) where V is total vnode count. Binary search on sorted ring. For 10K vnodes, ~14 comparisons. Negligible.

How do you handle hot keys?

Consistent hashing alone won't help — one key always maps to one server. Solutions: (a) replicate hot keys, (b) bounded-load consistent hashing (skip if server full), (c) client-side caching.

How is the ring topology shared?

Coordination service (ZooKeeper, etcd) or gossip protocol (Cassandra). Each node has local copy. Updates propagate within seconds.

Server fails — what happens?

Its vnodes are removed. Each vnode's range flows to its clockwise neighbour. With vnodes, load spreads across many surviving servers, not piled onto one.
