01 — Requirements

Functional

  • Limit requests per client in a time window (e.g., 100 req/min)
  • Support multiple rules — by user ID, IP, API key, endpoint
  • Return HTTP 429 with rate limit headers when exceeded
  • Different limits for different tiers (free / premium / enterprise)

Non-Functional

  • Low latency: < 1ms added per request
  • Distributed: works across multiple API servers
  • Accurate: no significant over/under-counting
  • Highly available: fail-open if rate limiter down
  • Scalable: millions of clients, billions of requests/day

04 — Rate Limiting Algorithms

Token Bucket

Bucket capacity: 10 tokens. Refill: 5 tokens/sec.

Burst of 10 at t=0:
  req 1-10: take token each. bucket: 10 → 0. All ALLOW.
  req 11: no tokens → 429 REJECT.
After 1s: bucket refills +5 → 5 tokens available.

Lazy refill (no background thread):
  elapsed = now - last_refill
  tokens = min(capacity, tokens + elapsed * rate)
  if tokens >= 1: tokens--; ALLOW
  else: REJECT

State: 2 values (tokens, last_refill). Allows bursts up to capacity. Used by AWS, Stripe.

Fixed Window Counter

Window: 1 minute. Limit: 100.

12:00-12:01: count requests. Reset at boundary.
  req at 12:00:30 → count=50 ALLOW
  req at 12:00:59 → count=100 ALLOW
  req at 12:00:59 → count=101 REJECT

BOUNDARY PROBLEM:
  Window 1: 100 reqs in last 30s
  Window 2: 100 reqs in first 30s
  12:00:30 to 12:01:30 = 200 reqs in 1 min! (2x limit)

State: 1 counter. Simple but boundary burst = 2x limit.

Sliding Window Log

Keep sorted log of every request timestamp.

On request:
  1. Remove timestamps older than (now - window)
  2. Count remaining
  3. count < limit? ALLOW + add timestamp : REJECT

Perfectly accurate. But stores every timestamp — 100 req/min x 10M users = 1B timestamps. Not practical at scale.

Sliding Window Counter (THE BEST)

Interpolate between current + previous window.

Limit: 100/min. Current window: 40s elapsed.
  prev_count = 80, curr_count = 30
  weight = 1 - (40/60) = 0.333
  estimated = 80 * 0.333 + 30 = 56.67
  56.67 < 100 → ALLOW

Just 2 counters per window! Cloudflare: 0.003% error.

State: 3 values. Near-perfect accuracy, tiny memory. Used by Cloudflare.

Leaky Bucket

FIFO queue with fixed drain rate.
  Request arrives → add to queue (reject if full).
  Processor drains at constant rate.

  vs Token Bucket:
    Token Bucket: allows bursts
    Leaky Bucket: smooths to fixed rate (no bursts)

Great for traffic shaping. Not common for API rate limiting.

06 — Redis Implementation

-- Sliding Window Counter in Redis (Lua script, atomic)
-- KEYS[1] = "rl:user_123:/api/data"
-- ARGV = {window_secs, limit, now_timestamp}

local window = tonumber(ARGV[1])
local limit  = tonumber(ARGV[2])
local now    = tonumber(ARGV[3])

local curr_win = math.floor(now / window)
local prev_win = curr_win - 1

local curr_key = KEYS[1] .. ":" .. curr_win
local prev_key = KEYS[1] .. ":" .. prev_win

local curr = tonumber(redis.call("GET", curr_key) or "0")
local prev = tonumber(redis.call("GET", prev_key) or "0")

local elapsed = now % window
local weight  = 1 - (elapsed / window)
local est     = prev * weight + curr

if est >= limit then return 0 end  -- REJECT

redis.call("INCR", curr_key)
redis.call("EXPIRE", curr_key, window * 2)
return 1  -- ALLOW

Why Lua? Redis executes Lua atomically. No race between GET and INCR across multiple API servers.

07 — Race Conditions

WITHOUT atomicity (broken):
  Server A: GET counter → 99
  Server B: GET counter → 99  (before A writes)
  Both: 99 < 100 → ALLOW → counter = 100
  Result: 101 requests! Race condition.

WITH Lua (correct):
  Server A: EVAL lua → reads 99, incr to 100, ALLOW
  Server B: EVAL lua → reads 100, >= limit, REJECT
  Result: exactly 100. Correct.

Alternative: INCR is atomic. count = INCR key; if count > limit: REJECT. Works for fixed window.

08 — Rules Engine

rules:
  - key: "user:{user_id}"             limit: 1000/min  (global per-user)
  - key: "user:{user_id}:ep:{path}"   limit: 100/min   (per-endpoint)
  - key: "ip:{client_ip}"             limit: 50/min    (per-IP)
  - key: "user:{user_id}"
    condition: "tier == premium"       limit: 10000/min (tier override)

Evaluate most specific first. ANY rule triggers → reject.

13 — Interview Q&A

Why Sliding Window Counter over Token Bucket?

Sliding Window Counter is simpler (3 values, no float arithmetic), has near-perfect accuracy (0.003% error per Cloudflare), and maps cleanly to Redis INCR + TTL. Token Bucket is equally valid — better when you explicitly want burst control. Both are correct interview answers.

How do you handle race conditions without Lua?

Use INCR which is atomic: count = INCR key; if count == 1: EXPIRE key window; if count > limit: REJECT. Works for fixed window. For sliding window, Lua is preferred because you need to read two keys and compute before incrementing.

What if a single user has different limits per endpoint?

Use composite keys: "rl:user_123:/api/search" with limit 10/min, "rl:user_123:/api/data" with limit 100/min, plus a global "rl:user_123" at 1000/min. Check all applicable rules — any triggered = reject.

How to rate limit across multiple data centers?

Two approaches: (1) Per-region limits — each region has its own Redis, simpler but total can exceed global limit. (2) Quota allocation — split global limit across regions (US: 400, EU: 300), rebalance dynamically based on traffic. Accept slight inaccuracy at boundaries for lower latency.

Fail open vs fail closed?

Fail open (allow all) for most APIs — availability matters more than strict limiting. Fail closed (reject all) only for security-critical paths like login brute-force protection. Best: fail open + local in-memory fallback + alert ops.

How does cost-based rate limiting work?

Assign each endpoint a cost: GET=1, expensive search=10, file upload=5. Use Token Bucket: each request deducts its cost (not 1). Bucket refills at fixed rate. This lets you have one limit that accounts for different resource intensities.
