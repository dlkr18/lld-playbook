Proximity Service // HLD

Design a location-based service like Yelp: find nearby businesses given a user’s location. Covers GeoHash, QuadTree, S2 spatial indexing, read-heavy caching, ranking, and pagination.

GeoHash QuadTree Read-Heavy Caching S2 Geometry

01 — Requirements

Functional

  • Search nearby businesses given (lat, lng) and radius
  • Filter by category (restaurants, gas stations, hospitals)
  • View business details (name, address, photos, hours, reviews, rating)
  • Business CRUD — owners add/update/delete listings
  • Reviews & Ratings

Non-Functional

  • Read-heavy — 100:1 read-to-write ratio
  • Low latency — nearby search < 200ms
  • Availability — search should always work (AP over CP)
  • Scale — ~200 million businesses worldwide
  • Eventual consistency OK — business update takes a few minutes to propagate
Key difference from Uber: businesses are static (they don’t move). This makes the index batch-friendly, heavily cacheable, and eventually consistent on writes.

05 — Spatial Indexing

Why Not Just SQL?

SELECT * FROM businesses
WHERE category = 'restaurant'
  AND distance(lat, lng, 37.7749, -122.4194) < 5000;

→ Computes distance for ALL 200 million rows. Even with B-tree
  on (lat, lng), range queries on two dimensions are inefficient.
→ Need a SPATIAL INDEX designed for 2D queries.

Approach 1: GeoHash

Convert 2D coordinate → 1D string, then use a regular B-tree index.

(37.7749, -122.4194) → "9q8yyk8yuv"

Precision levels:
  4 chars: "9q8y"    → ~40 km cell
  5 chars: "9q8yy"   → ~5 km cell
  6 chars: "9q8yyk"  → ~1.2 km cell
  7 chars: "9q8yyk8" → ~150 m cell

GeoHash Search: Centre + 8 Neighbours

┌───────┬───────┬───────┐
│ 9q8yj │ 9q8ym │ 9q8yt │
├───────┼───────┼───────┤
│ 9q8yh │ 9q8yk │ 9q8ys │  ← User is in 9q8yk
├───────┼───────┼───────┤      Search centre + all 8 neighbours
│ 9q8y5 │ 9q8y7 │ 9q8ye │
└───────┴───────┴───────┘

Query:
  SELECT business_id FROM geo_index
  WHERE geohash LIKE '9q8yk%'
     OR geohash LIKE '9q8yj%'
     OR geohash LIKE '9q8ym%' ...  (all 9 cells)
     AND category = 'restaurant';

Then post-filter by Haversine distance for exact circle.
Why neighbours? A business 2 km away could be in an adjacent cell. Without checking neighbours, you'd miss it at cell boundaries.

Variable Radius: Multi-Precision

500 m search needs finer cells than 20 km search. Two approaches:

ApproachHow
Multiple materialised indexesStore each business at several prefix lengths. Query the right precision for the radius.
One long hash + dynamic truncationStore one full-precision hash. At query time, truncate to the precision matching the radius and do prefix scan.

Radius → Precision lookup table:

RadiusPrecisionCell Size
~500 m7–8~150 m
~5 km5–6~1.2–5 km
~20 km4–5~5–40 km

Approach 2: QuadTree

Recursively split space based on density. Each node splits into 4 children when it has > N businesses (e.g. N = 100).

          ┌─────────────────────────────────────┐
          │              World                   │
          │   ┌──────────┬──────────────────┐   │
          │   │ Manhattan│   Rural NY        │   │
          │   │(dense → │   (sparse →       │   │
          │   │ split    │    NOT split,     │   │
          │   │ many     │    just 3 biz)    │   │
          │   │ times)   │                   │   │
          │   │ ┌──┬──┐ │                   │   │
          │   │ │NW│NE│ │                   │   │
          │   │ ├──┼──┤ │                   │   │
          │   │ │SW│SE│ │                   │   │
          │   │ └──┴──┘ │                   │   │
          │   └──────────┴──────────────────┘   │
          └─────────────────────────────────────┘

QuadTree Search (Range Query)

Start at root
Find leaf containing user’s location
Collect businesses in that leaf within radius
Check if search circle overlaps sibling/parent/neighbour nodes
The search circle (user + radius) may extend beyond the current leaf’s bounding box into adjacent nodes
Recursively search overlapping nodes
Post-filter by Haversine distance, return results

QuadTree Memory Estimation

200M businesses, max 100 per leaf → ~2M leaf nodes
Internal nodes: ~700K
Each node: ~100 bytes
Total: ~270 MB → fits entirely in memory

Build time: O(N log N), ~seconds for bulk build in memory

Approach 3: Google S2 Geometry

Maps the sphere to a Hilbert curve, divides into cells at various levels (0–30). Each cell has a 64-bit integer ID. Nearby cells have nearby IDs.

  • Handles the sphere correctly (no polar distortion)
  • Used by Google Maps, Uber, Tinder
  • More complex but geographically more accurate than GeoHash

Comparison: Which to Use?

ApproachProsConsRecommend When
GeoHashSimple, works with any DB, easy prefix queriesFixed-size grid, polar distortionDefault recommendation in interviews
QuadTreeAdapts to density, efficient non-uniform distributionsMust be in-memory, harder to maintainWhen interviewer asks about density variation
S2Handles sphere, hierarchical cells, accurateComplex to implementMention as “what Google/Uber actually use”

06 — Read Path (Search Flow)

Receive query
(lat=37.77, lng=-122.41, radius=5km, category=restaurant)
Compute geohash + 8 neighbours
Precision 5: "9q8yy" + ["9q8yw", "9q8yx", "9q8yz", ...]
Query geo_index table
SELECT business_id WHERE geohash IN (prefixes) AND category = 'restaurant'
Post-filter by Haversine distance
Remove businesses outside the 5 km radius circle
Fetch full business details
From Business DB or cache (name, rating, photos, hours)
Rank results
Sort by distance, rating, relevance. Apply business rules (promoted, open now).
Return paginated results

Haversine Post-Filter

GeoHash cells return a superset (square cells approximating a circle). Haversine formula computes the great-circle distance between two lat/lng points on Earth. Post-filter drops false positives from the grid.

07 — Write Path (Business Updates)

Business Service validates update
Write to primary Business DB (Postgres)
Publish event to Kafka
{ "event": "business_updated", "business_id": "biz_123", "new_lat": 37.77, "new_lng": -122.41 }
Geo-index updater consumes event
Remove old geohash entry, insert new, invalidate cache
QuadTree rebuild (if in-memory)
Periodic full rebuild every ~15 min OR incremental update
Key insight: Writes are ~6 QPS (negligible). Periodic full rebuild of the QuadTree from DB is totally fine — avoids complex incremental logic.

Cache Key Design

key = "nearby:{geohash_precision}:{category}:{radius_bucket}:{sort}:{page}"
example = "nearby:9q8yy:restaurant:5000:distance:1"
TTL = 5 minutes

Radius must be in the key (different radii return different results).
Bucket radii (500, 1000, 5000, 10000, 20000) to control key cardinality.
Why caching works so well: Many users in the same neighbourhood searching for “nearby restaurants” hit the same geohash cell → same cache key → cache hit. Businesses rarely change.

Cache Invalidation

  • When a business updates, invalidate all cache keys for its geohash cells
  • Updates are rare (~6/sec globally) — invalidation is trivial
  • TTL as safety net (5 min) for missed invalidations

09 — Ranking & Results

Ranking Signals

SignalNotes
DistancePrimary for “nearby.” Decay: 1/distance or exp(-distance)
RatingBayesian/Wilson score so 5.0 with 3 reviews doesn’t beat 4.6 with 2,000
Review countConfidence / demand signal
Open nowHard filter or big boost
Text relevanceIf text query: blend text score with geo score
SponsoredFixed slots (e.g. 2–3) with “Ad” label, or capped boost
DiversityCap per category/chain so top 20 aren’t all one brand (MMR)
score = w1 * f(distance) + w2 * g(rating) + w3 * h(reviews) + boosts(open, sponsored)

Full-Text + Geo (“dim sum near me”)

Use Elasticsearch/OpenSearch with geo_distance filter + full-text query.

Order of operations: Geo filter FIRST to shrink candidate set, THEN text query ranks within it. Not the other way around at Yelp scale.

Pagination: Cursor > Offset

OFFSET/LIMIT while the user moves causes duplicates or gaps.

MethodHow
Cursor / keysetPage 1 ends at (distance=0.8km, id=555). Page 2: WHERE (distance, id) > (0.8, 555) LIMIT 20. Stable ordering.
Frozen searchServer returns search_id + results. Origin lat/lng fixed for that session. Pagination uses frozen origin.

Together: cursor = stable ordering; frozen origin = stable distances.

10 — Scaling

LBS (Read Path) — Stateless Replicas

  ┌─────────────────────────────────┐
  │    LBS (Location-Based Search)  │
  │    - Stateless                  │
  │    - Each replica holds full    │
  │      geo index in memory        │
  │      (QuadTree ~270 MB)         │
  │                                 │
  │  Replica 1  Replica 2  ...  N   │
  └─────────────────────────────────┘
Key insight: The entire geo index is ~270 MB. Each LBS replica holds it all in memory. Just add replicas behind a load balancer for more QPS. No sharding needed for reads!

Business DB

  • 1 Primary (writes) + multiple read replicas
  • 200M rows manageable for single Postgres with read replicas
  • If billions: shard by geographic region

Multi-Region Deployment

  • DNS / GeoDNS routes users to nearest region (US-West, US-East, EU, Asia)
  • Regional LB picks AZ/instance
  • Service mesh keeps dependent calls local
  • Data residency: user PII in user’s region; public business directory can replicate globally

11 — Deep Dives

Geofencing

A geofence is a virtual boundary on a map (circle, polygon, or S2 cell set). When a device enters or exits, trigger logic: send promo, log attendance, unlock scooter, start surge pricing.

Implementation: periodic GPS → point-in-polygon / distance-to-centre → compare to fence definition.

Hilbert Curve (Used by S2)

A space-filling curve that maps 2D → 1D while preserving spatial locality. Nearby points in 2D are usually nearby in 1D. S2 uses this so cell IDs are sorted in a way that nearby cells have nearby IDs — good for range queries, compression.

Blue-Green Deployment

Run two identical environments: Blue (current live) and Green (new version). Switch routing (LB/DNS) from blue → green in one step. If something breaks, switch back.

  • Pros: instant rollback, no half-old / half-new during cutover
  • Cons: double capacity during cutover; DB migrations need backward compatibility

Rate Limiting: Client vs Server

Client-SideServer-Side
Who enforcesApp / SDKAPI gateway / service
TrustUser can bypass (curl, modded app)Authoritative — must enforce here
PurposeUX, save battery, avoid 429sProtect infra, fair use, security

Rule: Server-side is mandatory. Client-side is optional courtesy.

Throttling = intentionally slowing/rejecting traffic when rate exceeds policy. Common algorithms: fixed window, sliding window, token bucket, leaky bucket. Response: 429 Too Many Requests + Retry-After header.

Consistency & Cache Invalidation

  • Write path: Update Postgres (source of truth) → emit Kafka event “business_updated”
  • Consumers: Update geo index rows (old location out, new in), bump version
  • Cache: Delete keys by business_id tag, or invalidate geohash buckets for old + new location
  • CDN: Purge detail URLs for that business
  • TTL (5 min) as safety net, not primary mechanism

13 — Interview Q&A

How do you handle variable search radii (500 m vs 20 km)?

Map radius → geohash precision via a lookup table. Either materialise multiple precision indexes, or store one long hash and truncate by radius at query time. Always query centre + 8 neighbours at the chosen precision, then Haversine post-filter.

How do you rank results beyond distance?

Weighted score: w1*f(distance) + w2*g(rating) + w3*h(reviews) + boosts(open, sponsored). Use Bayesian/Wilson score for ratings (prevents 5.0 with 3 reviews beating 4.6 with 2K). Cap per category/chain for diversity (MMR). Pre-filter by geo + category, then rank top K.

How do you paginate when the user is moving?

Cursor/keyset pagination (not OFFSET): order by (distance, id), next page uses WHERE (distance, id) > (cursor). Combine with frozen search context: fix the origin lat/lng for a search_id session. If user moves significantly, start a new search.

One geohash cell in Manhattan has millions of businesses. How do you handle it?

Use finer geohash precision in dense areas, composite indexes (geohash + category + rating_bucket), hard cap on candidates per cell (e.g. 5K), and pre-materialised “top N per cell + category” from a batch job.

How do you combine full-text search with geo (“dim sum near me”)?

Use Elasticsearch with geo filter first (bounding box / circle) to shrink the candidate set, then text query ranks within it. Geo reduces universe; text scores within it. Not the reverse.

How does cache invalidation work when a business updates?

Event-driven: write to Postgres → Kafka event → consumer updates geo index + deletes cache keys for affected geohash buckets (old + new location). TTL (5 min) as safety net. Updates are ~6/sec globally — invalidation is trivial at this rate.

Could two users searching the same location but different radii share a cache?

No. Radius must be in the cache key (or radius bucket). Key = nearby:{geohash}:{category}:{radius_bucket}:{sort}:{page}. Bucket radii (500, 1K, 5K, 10K, 20K) to control cardinality without infinite unique keys.

Why is this system easier to design than Uber?

Businesses are static — the spatial index can be batch-built, heavily cached, and eventually consistent. Uber’s drivers move every 4 seconds, requiring real-time index updates, strong consistency for matching, and much higher write throughput (250K updates/sec).

Multi-region: how do you handle data residency (EU)?

Partition user data / reviews by region for legal compliance. Public business directory can replicate globally or geo-partition. Cross-border “nearby” is a product/legal choice. DNS/geo-routing picks region; regional LB picks AZ/instance; service mesh keeps calls local.
