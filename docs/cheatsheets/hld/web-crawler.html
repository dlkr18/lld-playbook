Web Crawler // HLD

Distributed system to systematically browse the web, download pages, and build a searchable index. Covers URL frontier, politeness, deduplication, adaptive re-crawl, and scaling to billions of pages.

01 — Requirements

Functional

  • Seed URLs — start with an initial set of URLs
  • Fetch & Parse — download HTML, extract text, links, metadata
  • Discover — follow extracted links to find new pages
  • Store — persist raw HTML and structured metadata
  • Re-crawl — revisit pages to detect content changes
  • Respect robots.txt — honour site-defined crawl rules

Non-Functional

  • Scale — ~1 billion pages/day (~15,000 pages/sec)
  • Politeness — don’t overload any single host
  • Robustness — handle malformed HTML, server failures, crawler traps, timeouts
  • Extensibility — easy to add new content types or processing logic
  • Deduplication — don’t waste resources on identical content
  • Prioritisation — important / frequently-changing pages crawled first

04 — URL Frontier (The Brain)

Key insight: The frontier has two orthogonal concerns — what to crawl (priority) and when to crawl (politeness). Separate them into two layers.

Two-Layer Design

Layer 1: PRIORITY (what)            Layer 2: POLITENESS (when)

  ┌───────────────────┐              ┌────────────────────────────┐
  │ P0 (critical)     │──┐           │  news.site.com:            │
  ├───────────────────┤  │  route    │   [url_7, url_3, url_19]   │
  │ P1 (high)         │──┼─by────►  │  wikipedia.org:            │
  ├───────────────────┤  │ domain   │   [url_2, url_88, url_5]   │
  │ P2 (normal)       │──┤           │  myblog.com:               │
  ├───────────────────┤  │           │   [url_41]                 │
  │ P3 (low)          │──┘           └──────────┬─────────────────┘
  └───────────────────┘                         │ rate-limited
                                         ┌──────▼──────┐
                                         │   Fetcher    │
                                         └─────────────┘

Priority Queue (What to Crawl)

  • Prioritise by: PageRank, freshness, domain importance, content change frequency
  • Multiple priority buckets (P0 critical → P3 low)
  • Weighted scheduling to prevent starvation: P0 gets 60%, P1 25%, P2 10%, P3 5%

Politeness Queue (When to Crawl)

  • One FIFO queue per host/domain
  • Only one request active per host at a time
  • Minimum delay between requests to same host (e.g. 1 second)
  • A queue router maps URL → domain queue
  • A queue selector round-robins across domain queues

Storage for the Frontier

Billions of URLs won’t fit in memory (~100 GB+ for URL strings alone). Use a disk-backed queue:

OptionHow
RocksDBEmbedded KV store on local SSD. Key = priority:domain:timestamp, value = URL + metadata. LSM tree gives fast writes.
KafkaDistributed message queue. One partition per domain (or domain-group). Append-only log on disk. Natural per-domain ordering.

Kafka with Priority: Multiple Topics

Topic: urls-priority-0  (partitioned by domain)
Topic: urls-priority-1  (partitioned by domain)
Topic: urls-priority-2  (partitioned by domain)

Consumer logic:
  while true:
    for priority in [P0, P1, P2, P3]:          // Priority order
      for domain_partition in topic[priority]:   // Round-robin domains
        if rate_limiter.allows(domain):          // Politeness check
          url = poll(domain_partition)
          fetch(url)
          break

05 — Fetcher & DNS Resolver

Fetcher

  • Async HTTP client (event-loop based: Netty / asyncio) — one thread handles thousands of concurrent connections
  • Respects robots.txt — cache per domain, refresh every 24h
  • Handles redirects (limit chain to ~10 hops)
  • Timeout: connect 2s, read 10s
  • Retry with exponential backoff for transient failures (5xx, timeouts)
  • Circuit breaker: blacklist consistently failing hosts

Why Async / Event-Loop?

Crawling is I/O-bound (95% of time is waiting for network). Async lets one thread manage 10,000+ concurrent connections with minimal memory.

Single Thread (Event Loop):
  → Send request to google.com     (non-blocking)
  → Send request to amazon.com     (non-blocking)
  → Send request to wikipedia.org  (non-blocking)
  → [event: google.com responded]  → process it
  → [event: wikipedia.org responded] → process it
  → [event: amazon.com responded]  → process it

OS mechanisms: epoll (Linux), kqueue (macOS), IOCP (Windows)

DNS Resolver

  • DNS lookups are a bottleneck at scale (10–100ms each)
  • Run a local DNS cache (TTL-aware)
  • Use async/batch DNS resolution
  • Pre-fetch DNS for URLs in the frontier

06 — Content Parser & Link Extractor

Parsing

  • Parse HTML (handle malformed HTML gracefully)
  • Extract all <a href="..."> links
  • Extract canonical URL (<link rel="canonical">) — the site’s declared “official” URL for a page
  • Extract metadata: title, description, language, charset

URL Normalisation

The same content can live at multiple URLs. Normalise aggressively:

  • Lowercase the scheme and host
  • Remove fragments (#section)
  • Remove default ports (:80, :443)
  • Sort query parameters
  • Remove tracking parameters (utm_source, etc.)
  • Resolve relative URLs to absolute

Headless Browser Rendering (JS-heavy pages)

ToolCreated ByNotes

Use selectively: Headless browser is 50–100x more expensive than simple HTTP fetch (~200–500 MB RAM, 2–5s per page). Only use for high-value JS-heavy domains.

07 — URL & Content Deduplication

URL-Level Dedup: Bloom Filter

A probabilistic data structure: “Have I seen this URL before?”

PropertyBloom FilterRedis Set
Memory (1B URLs)~1.2 GB (1% FP rate)~60–100 GB
Accuracy~99% (probabilistic)100% (exact)
Network hopNone (in-process)Yes (separate server)
LatencyNanoseconds~0.1–1 ms

Bloom Filter Sizing Formula

m = -(n × ln(p)) / (ln(2))²

n = 1,000,000,000 (1B URLs)
p = 0.01 (1% false positive rate)

m = -(10&sup9; × (-4.605)) / (0.693)²
m = 4,605,000,000 / 0.480
m ≈ 9.6 billion bits ≈ 1.2 GB

Optimal hash functions: k = (m/n) × ln(2) ≈ 7

False Positives vs False Negatives

TypeMeaningConsequence
False PositiveSays “seen” but hasn’tSkip a URL we should crawl (miss a page). Acceptable at 1%.
False NegativeSays “not seen” but hasRe-crawl a page → infinite loops. Unacceptable.

Bloom filters guarantee no false negatives. That’s why they’re perfect here.

Content-Level Dedup: SimHash

Detect near-duplicate content (same article with different ads/headers).

  1. Break page into features (word shingles)
  2. Hash each feature to 64 bits
  3. For each bit position: +1 if bit=1, −1 if bit=0
  4. Final hash: each bit → 1 if sum > 0, else 0

Comparison: Hamming distance between two SimHashes. If ≤ 3 bits differ out of 64 → near-duplicate.

Fast SimHash Lookup: Bit Partitioning

Pigeonhole Principle: If two 64-bit hashes differ by ≤ 3 bits, at least one of four 16-bit blocks must be exactly identical.
SimHash: [Block A: 16 bits][Block B: 16 bits][Block C: 16 bits][Block D: 16 bits]

Build 4 hash tables, one per block:
  Table A: key = Block A → list of SimHashes with that Block A
  Table B: key = Block B → list of SimHashes with that Block B
  ...

Lookup: query all 4 tables, union candidates, check exact Hamming distance.

Naive:         1,000,000,000 comparisons per page
Bit partition:        ~60,000 comparisons per page  ← 16,000x faster

Partitioning Strategy

  • Partition by domain hash — each worker owns a set of domains
  • Naturally enforces politeness (one worker per domain)
  • Consistent hashing for graceful worker add/remove

Handling Domain Skew (Hot Domains)

Wikipedia has 60M pages; myblog.com has 10. The worker that gets Wikipedia is overwhelmed.

SolutionHow
Sub-partition large domainsSplit wikipedia.org into N sub-queues by URL path hash. Multiple workers process the same domain, sharing a rate limiter.
Work stealingIdle workers steal URLs from overloaded workers’ queues. Each worker has a deque: owner pops HEAD, thieves steal from TAIL.
Dynamic rebalancingMonitor queue sizes. Coordinator reassigns domains from large queues to idle workers.

Coordination: ZooKeeper / etcd

A small, highly-reliable, strongly-consistent KV store for all workers to agree on.

  • Distributed locks — only one worker processes a domain at a time
  • Leader election — auto-elect new leader if one dies
  • Service discovery — workers register; others find them
  • Ephemeral nodes — auto-delete when creator disconnects (heartbeat)
  • Watches — push-based notification when a node changes (no polling)

11 — Adaptive Re-Crawl Strategy

Per-URL Metadata

URL: https://news.site.com/homepage
  last_crawled_at:          2026-03-24 10:00:00
  content_hash:             "a3f8b2..."
  change_count:             47     (changed 47 of last 50 crawls)
  crawl_count:              50
  avg_change_interval:      1.8 hours
  current_recrawl_interval: 2 hours
  next_crawl_at:            2026-03-24 12:00:00

Exponential Backoff / Ramp-Up

After each crawl:
  if page CHANGED:
      new_interval = current_interval / 2   (floor: 1 hour)
  if page NOT changed:
      new_interval = current_interval * 2   (ceiling: 30 days)

News homepage:  24h → 12h → 6h → 3h → 1.5h → 1h (floor)
Gov archive:    24h → 2d → 4d → 8d → 16d → 30d (ceiling)

HTTP Conditional Requests (Save Bandwidth)

HTTP FeatureHow It Helps
If-Modified-SinceServer returns 304 Not Modified if unchanged (~200 bytes vs ~100 KB). 500x cheaper.
ETag + If-None-MatchContent fingerprint. Server returns 304 if ETag matches.
Cache-Control: max-ageServer says “won’t change for N seconds”
Sitemap.xml with <lastmod>Site publishes when each page last changed

Budget Allocation

Total budget: 15,000 pages/sec

  40% (6,000/sec) → pages due for re-crawl NOW (overdue)
  40% (6,000/sec) → new URLs never crawled before
  20% (3,000/sec) → forced re-check of “static” pages

Priority score = (current_time - next_crawl_at) / recrawl_interval
  Page A: due 5h ago, interval 2h → score 2.5 (very overdue)
  Page B: due 1h ago, interval 24h → score 0.04 (barely)

12 — Deep Dives

LSM Trees (Log-Structured Merge Trees)

The data structure behind RocksDB, Cassandra, HBase. Turns random writes into sequential writes (100–1000x faster).

WRITE PATH:
  Write("url", data) → MemTable (in-memory skip list, ~64-256 MB)
                         + Write-Ahead Log (WAL, sequential append)
                         
  When MemTable full → flush to SSTable (sorted file on disk)
  
  Background compaction merges SSTables:
    L0 (many small) → L1 (fewer, larger) → L2 (even larger)

READ PATH:
  Check MemTable (memory) → L0 SSTables → L1 → L2
  Bloom filter on each SSTable skips irrelevant files
OperationSpeed
WriteMicroseconds (memory + sequential WAL append)
ReadFast with caching + bloom filters
CompactionBackground; doesn’t block reads/writes

RocksDB

Embedded, persistent KV store by Facebook (forked from Google’s LevelDB). Runs inside your process — no separate server, zero network overhead.

  • Based on LSM trees. Optimised for SSDs.
  • Used by: Kafka Streams (state stores), Flink (streaming state), CockroachDB, TiKV, MyRocks (MySQL + RocksDB)

Async I/O — What “Callback” Means

1. App → OS: "Open TCP to 93.184.216.34:443. Don't block me."
2. Event Loop → OS (epoll/kqueue): "Add fd=7 to watch list.
   Notify me when data is READABLE."
3. Event loop is FREE. Sends requests for 999 other URLs.
4. OS → Event Loop: "fd=7 has data ready!"
5. Event loop resumes the coroutine that was waiting for fd=7.
   The `await` returns with the response.

OS mechanisms:
  Linux:   epoll
  macOS:   kqueue
  Windows: IOCP

Work Stealing

Each worker has a deque (double-ended queue). Owner pops from HEAD, thieves steal from TAIL — minimal contention.

Worker 1's deque:  [A] [B] [C] [D] [E]
                    ↑                 ↑
                 owner pops          thieves steal
                 from HEAD           from TAIL

Worker 2 finishes its tasks → steals E from Worker 1
Result: work completes ~40% faster than without stealing

Java’s ForkJoinPool uses this pattern. In a crawler, idle workers steal URL batches from overloaded workers while respecting shared rate limiters.

ZooKeeper: Ephemeral Nodes + Watches

/crawler/workers/
  ├── worker-1   (ephemeral)
  ├── worker-2   (ephemeral)   ← Worker 2 crashes!
  └── worker-3   (ephemeral)

ZooKeeper detects session timeout → auto-deletes worker-2 node
→ Fires watch notification to coordinator
→ Coordinator reassigns worker-2's domains

Watches = push-based notifications (no polling).
  One-time trigger: fires once, must re-register.
  etcd improvement: persistent watch streams.

13 — Interview Q&A

How do you handle domain skew where one domain has 60M pages?

Sub-partition large domains into N sub-queues by URL path hash. Multiple workers process the same domain in parallel, sharing a distributed rate limiter (e.g. token bucket in Redis). Combine with work stealing (idle workers take from busy workers' deques) and dynamic rebalancing via the coordinator.

Kafka is FIFO per partition. How do you support priority + politeness?

Use multiple Kafka topics by priority (urls-priority-0, urls-priority-1, ...), each partitioned by domain. Consumer reads from P0 first, round-robins domains, and checks a rate limiter before fetching. Weighted scheduling (60% P0, 25% P1, 10% P2, 5% P3) prevents starvation of low-priority URLs.

How do you efficiently find near-duplicate SimHashes among billions?

Bit partitioning (pigeonhole principle): split 64-bit hash into 4 blocks of 16 bits. If two hashes differ by ≤3 bits, at least one block is identical. Build 4 lookup tables keyed by each block. Query all 4, union candidates (~60K), check exact Hamming distance. 16,000x faster than brute force.

Why Bloom filter over Redis for URL dedup?

At 1B URLs: Bloom filter needs ~1.2 GB (in-process, nanosecond lookups). Redis needs ~60–100 GB (separate cluster, ~0.1–1ms per lookup). The 50–80x memory savings is worth the 1% false positive rate. Many systems use both: Bloom filter as fast first check, Redis/RocksDB as definitive backing store.

How do you handle pages that change at different rates?

Exponential backoff/ramp-up: if page changed → halve the interval (floor 1h). If unchanged → double the interval (ceiling 30 days). Track change_count/crawl_count per URL. Use HTTP conditional requests (ETag / If-Modified-Since) to get 304 responses and save 500x bandwidth. Allocate crawl budget by priority score = (now − next_crawl_at) / interval.

What is robots.txt and how do you handle it?

A file at the root of a website (/robots.txt) that specifies which paths crawlers may or may not access. Cache it per domain, refresh every 24h. Parse User-Agent rules, Disallow/Allow directives, and Crawl-delay. Respect it as a matter of politeness and legal compliance.

How do you detect and escape crawler traps?

Set max depth per domain (e.g. 15 levels), max pages per domain cap, detect URL pattern repetition (e.g. same path with incrementing date params), and track unique-content ratio per domain (if many pages have identical content, likely a trap).

What monitoring metrics would you track?
  • Pages crawled/sec, error rate by type (5xx, timeout, DNS failure)
  • Frontier size (growing unboundedly = bad)
  • Average fetch latency per domain
  • Dedup hit rate (Bloom filter + content)
  • Crawl coverage & freshness distribution
  • Top error domains (blacklist candidates)
