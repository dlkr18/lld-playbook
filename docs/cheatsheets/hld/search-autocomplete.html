Search Autocomplete // HLD

Typeahead suggestion system: in-memory Trie with cached top-K per node, multi-layer caching (browser + CDN + Redis), offline pipeline for score computation, real-time trending injection, and client-side debounce.

Trie Top-K Caching CDN Trending Debounce

01 — Requirements

Functional

  • Top K suggestions as user types each character (K=10)
  • Ranked by popularity, recency, and optionally personalisation
  • Prefix matching — “face” → facebook, face mask, facetime
  • Multi-word queries — “how to” → “how to cook rice”
  • Suggestions update over time as trends change

Non-Functional

  • Ultra-low latency — < 100ms (users feel >200ms as laggy)
  • High availability
  • Scale — ~1.5M suggestion requests/sec at peak
  • Freshness — trending topics appear within minutes
  • Read-heavy — 100:1 read-to-write

04 — Trie Data Structure

Basic Trie

Searches: "tree"(1200), "try"(2500), "true"(800), "toy"(500), "top"(3000)

            (root)
           /      \
          t        ...
         / \
        r    o
       / \    \
      e   u    y  [toy: 500]
      |   |    |
      e   e    p  [top: 3000]
  [tree:1200] [true: 800]
  [try: 2500]

User types "tr" → traverse t → r
  → Collect all terms under subtree: tree, try, true
  → Sort by score → return [try(2500), tree(1200), true(800)]

Key Optimisation: Cache Top-K at Each Node

Problem: DFS the entire subtree for each query is O(subtree size) — too slow for popular prefixes like “a” with millions of terms beneath.
Solution: pre-compute and cache top-K at every node.

Node "tr":
  children: {e, u}
  top_k: [(try, 2500), (tree, 1200), (true, 800)]  ← pre-computed!

Node "t":
  children: {r, o}
  top_k: [(top, 3000), (try, 2500), (tree, 1200), ...]

Lookup: traverse prefix chars → return top_k at final node.
  Complexity: O(length of prefix) — no DFS needed.

Trade-off: more memory (top-K list at every node), but queries are O(prefix length).

Build Trie
Insert all terms + scores. Compute top-K at every node. Serialise to binary file.
Distribute to servers
Upload to S3. Each suggestion server downloads and loads into memory. Atomic swap: old Trie → new Trie (zero downtime).

10 — Trending / Real-Time Topics

Trie is rebuilt hourly/daily. But a viral event happens NOW. How to surface it?

Search Logs → Kafka → Flink (real-time)

Flink: sliding window (5 min), count per query term
  If count spikes > 10× normal → mark TRENDING

Trending terms → Redis (trending-terms set)

Suggestion Service on each request:
  1. Look up prefix in Trie (normal results)
  2. Check Redis for trending terms matching this prefix
  3. Merge: trending terms get a big score boost
  4. Return blended top-K
“world cup final score” appears in suggestions within minutes of becoming popular, without waiting for Trie rebuild.

12 — Trie Optimisations

Compressed Trie (Patricia / Radix Tree)

Standard Trie for "facebook":
  f → a → c → e → b → o → o → k  (8 nodes)

Compressed:
  "facebook"  (1 node, stores whole string)

Shared prefixes:
  "face" → "book"
         → "time"
  (3 nodes instead of 12)

50-70% memory reduction. Fewer pointer hops.

Other Optimisations

TechniqueBenefit
Memory-mapped filesTrie as flat binary file, mmap’d. Instant startup (no deserialisation). OS manages paging.
Compact encoding32-bit offsets instead of 64-bit pointers → 50% smaller
Filter unpopular termsOnly include terms with score > threshold. 1B → 100M in Trie.
Trie shardingIf Trie exceeds single server RAM, shard by prefix (first 2 chars → 676 shards)

13 — Interview Q&A

Why a Trie and not a database with LIKE 'prefix%'?

A Trie lookup is O(prefix length) — typically 3–5 character comparisons, nanoseconds. A database LIKE 'fac%' query scans an index, involves I/O, and is orders of magnitude slower. At 1.5M QPS with <100ms requirement, you need in-memory data structures.

How do you handle trending topics that aren’t in the Trie yet?

Flink detects spikes in search logs (sliding window, count > 10× normal). Trending terms are pushed to Redis. Suggestion service merges Trie results with Redis trending terms on each request, giving trending terms a score boost. Appears within minutes without Trie rebuild.

The Trie is 100 GB. How do you update it without downtime?

Atomic swap: Build new Trie offline → upload to S3 → each server downloads new Trie in background → swap pointer from old to new Trie atomically. Old Trie is garbage collected. Zero downtime, zero stale results.

How does the CDN achieve 90% hit rate?

Short prefixes (1–4 chars) have a tiny key space: 26&sup4; ≈ 457K combinations. Millions of users type “fac” every day → always cached at CDN edge. Longer prefixes have lower hit rate but are rarer. CDN + browser cache together absorb 90%+ of traffic.

What if the Trie doesn’t fit on one server?

Shard by prefix: first 2 characters → 676 shards. API gateway routes “fa*” to Shard-FA, “tr*” to Shard-TR. Each shard holds a smaller sub-Trie. Alternatively, filter unpopular terms (keep only top 100M) to fit on one server.

Why debounce on the client? Why not just send every keystroke?

Typing “facebook” generates 8 keystrokes in <1 second. Sending all 8 is wasteful — by the time “fa” response arrives, user has already typed “faceb”. Debounce (wait 50ms after last keystroke) reduces 8 requests to 1–2, cutting server load 4–8x while feeling just as fast.

How do you prevent inappropriate suggestions?

Blocklist of offensive terms, checked during Trie build (filter out) and at serving time (double-check). Updated by content moderation team. Also: ML classifier trained on flagged suggestions. Remove from Trie, not just from display (avoid serving and then hiding).
