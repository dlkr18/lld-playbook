Notification System // HLD

Multi-channel notification delivery: Push, Email, SMS, In-App. Covers priority queues, reliability (DLQ + retry), per-user rate limiting, scheduling, template rendering, and analytics.

Kafka Multi-Channel Reliability Rate Limiting DLQ

01 — Requirements

Functional

  • Multi-channel delivery — Push (iOS/Android), Email, SMS, In-App
  • Notification types — transactional (OTP, order), social (friend request), marketing (promos)
  • User preferences — opt out per channel per type, quiet hours
  • Templating — reusable templates with variable substitution
  • Scheduling — send now or at a future time
  • Tracking — delivery status, open/click rates

Non-Functional

  • High throughput — ~10M/day normal, 100M+ peak (Black Friday)
  • Reliability — transactional notifications must not be lost
  • Low latency — transactional < 5s, marketing can be minutes
  • At-least-once delivery (exactly-once impractical across external providers)
  • No spam — per-user rate limiting per channel
  • Extensible — easy to add new channels (WhatsApp, Slack)

05 — API Service (Entry Point)

Idempotency check
Redis SETNX on idempotency_key with TTL. Seen before? Return cached response.
Validate
Template exists? Users exist? Caller authorised?
Caller rate limit
Marketing Service: max 1K req/min. Auth Service: max 10K req/min (higher for OTPs).
Fan out
For each user in “to”: create message { user_id, template_id, data, channels, priority }
Enqueue to Kafka
Route to priority topic: HIGH → priority-0, MED → priority-1, LOW → priority-2

06 — Message Queue (Kafka)

Why Kafka?

KafkaSQS / RabbitMQ
ReplayYes — consumers re-read for debug/reprocessingNo — deleted after consumption
OrderingPer-partition (partition by user_id)Best-effort (SQS) or per-queue
ThroughputMillions/secModerate
RetentionDays/weeks (configurable)Minutes/hours
Multiple consumersConsumer groups (push sender, analytics, audit)Need separate queues

Topic Design

notifications-priority-0  (HIGH: OTP, auth, order)
  → 32 partitions (partition by user_id % 32)

notifications-priority-1  (MEDIUM: social, friend requests)
  → 16 partitions

notifications-priority-2  (LOW: marketing, weekly digest)
  → 8 partitions

Workers drain priority-0 first, then 1, then 2.
Separate consumer groups per channel sender.

07 — Workers Pipeline

Each worker consumes a message and runs this pipeline:

Preference Check
Has user opted out of this channel + type? Quiet hours in user’s timezone? Account active? If blocked → FILTERED, skip.
Per-User Rate Limit
Redis INCR: max 5 push/hour, 3 SMS/day, 10 email/day. If exceeded → RATE_LIMITED, skip or delay.
Template Rendering
Load template from cache. Fill variables: “Hi {{user_name}}, order {{order_id}} shipped!” Handle localisation.
Channel Routing
Explicit channels in request? Else user prefs. Fallback chain: push → email → SMS.
Dispatch
Enqueue to channel-specific queue: push-send-queue, email-send-queue, sms-send-queue.
Write to DB
Insert notification record: status = SENT/QUEUED (for tracking, analytics, user inbox).

08 — Channel Senders

Push Sender (FCM / APNs)

  1. Read from push-send-queue
  2. Look up device token(s) — user may have multiple devices
  3. Build platform-specific payload (FCM JSON / APNs aps dict)
  4. Send to FCM/APNs API
  5. Handle response: 200 → DELIVERED, token expired → remove stale token, rate limited → retry, server error → retry (max 3)

Email Sender (SendGrid / SES)

  • Handle bounces: hard bounce → remove email address; soft bounce → retry
  • Handle complaints (spam report) → auto-unsubscribe
  • Track opens (tracking pixel) and clicks (redirect URL)

SMS Sender (Twilio / SNS)

  • Most expensive channel — use sparingly
  • Delivery receipts via webhook callback
  • Use provider idempotency key to prevent duplicate sends

In-App Sender

  • Store in notification inbox table (Cassandra / DynamoDB)
  • Push to client via WebSocket if user is online
  • Client polls inbox on app open if offline

09 — Reliability

Transactional notifications (OTP, order confirmation) must not be lost. Design for at-least-once delivery with retry + DLQ.

Failure Points & Solutions

FailureSolution
API crashes after receiving requestKafka producer acks=all. Client retries with same idempotency key.
Worker crashes mid-processingDon’t commit Kafka offset until processing complete. On restart, re-reads message.
Third-party API failsExponential backoff (1s, 2s, 4s, 8s, max 5 retries). After max → DLQ.
Third-party down for hoursDLQ accumulates + alert ops. When provider recovers, replay DLQ. Consider failover provider (SES as backup for SendGrid).
Duplicate deliveryAcceptable for push/email. For SMS (costly), use provider idempotency key.

Dead Letter Queue (DLQ)

Normal:  Kafka → Worker → SendGrid → Success ✓

After 5 failed retries:
  Kafka → Worker → SendGrid → Fail (5th time)
                              │
                              ▼
                    ┌─────────────────┐
                    │  Dead Letter     │
                    │  Queue (DLQ)     │
                    │                  │
                    │  + alert ops     │
                    │  + retry later   │
                    │  + manual review │
                    └─────────────────┘

Circuit Breaker Pattern

After N consecutive failures to a provider, stop trying for M seconds (circuit open). Then probe with one request (half-open). If success, resume (circuit closed). Prevents hammering a dead provider.

CLOSED (normal) —N failures→ OPEN (reject all, wait M sec)
                                      │
                              after M sec ↓
                              
                          HALF-OPEN (try one request)
                            │               │
                         success          failure
                            │               │
                            ▼               ▼
                          CLOSED          OPEN (reset timer)

10 — Rate Limiting

Two Levels

LevelWhereRule
Caller rate limitAPI gatewayMarketing Service: 1K req/min, Auth Service: 10K req/min
Per-user rate limitWorker pipelineMax 5 push/hour, 3 SMS/day, 10 email/day

Per-User Implementation (Redis Sliding Window)

Key: "rl:{user_123}:push:{hour_bucket}"
INCR → if > 5 → skip

Key: "rl:{user_123}:sms:{day_bucket}"
INCR → if > 3 → skip

Key: "rl:{user_123}:email:{day_bucket}"
INCR → if > 10 → skip
Why it matters: A bug in the order service could trigger 1000 “order shipped” notifications to one user. Without per-user limits, user gets spammed and uninstalls.

Notification Collapsing

Android collapse_key / iOS thread-id: multiple notifications of the same type collapse into one (“You have 5 new messages” instead of 5 separate pushes).

11 — Scheduling & Marketing Blasts

Scheduled Notifications

┌────────────────────────────────────────┐
│  scheduled_notifications table          │
│                                         │
│  id | send_at              | payload    │
│  1  | 2026-04-05 09:00 PST | {...}     │
│  2  | 2026-04-05 09:00 EST | {...}     │
└────────────────────────────────────────┘

Scheduler job (runs every minute):
  SELECT * FROM scheduled_notifications
  WHERE send_at <= NOW() AND status = 'PENDING'
  ORDER BY send_at LIMIT 1000
  FOR UPDATE SKIP LOCKED;

  For each: enqueue to Kafka, mark as QUEUED.

SELECT FOR UPDATE SKIP LOCKED

ClauseWhat It Does
FOR UPDATELocks selected rows. No other transaction can claim them.
SKIP LOCKEDIf a row is already locked by another worker, skip it — grab the next available. Zero waiting, zero overlap.
Worker 1: SELECT ... FOR UPDATE SKIP LOCKED → gets rows 1-100 (locks them)
Worker 2: SELECT ... FOR UPDATE SKIP LOCKED → rows 1-100 locked, SKIPS
           → gets rows 101-200 instead. Maximum parallelism.

Marketing Blast (50M Users)

Problem: 50M messages at once overwhelms Kafka and senders.

Solution: Progressive fan-out
  1. Create campaign: { segment: "premium_users", template: "summer_sale" }
  2. Resolve segment → list of user_ids
  3. Batch enqueue to Kafka: 10K messages/batch, throttled at 100K/min
  4. Workers process at their own pace
  5. Channel senders respect third-party rate limits
     (SendGrid: 100K emails/hour on your plan)

Quiet Hours

Store timezone in user preferences. Worker checks: is current time in user’s timezone within quiet hours?

  • Marketing/social: delay to next allowed window (e.g. 8 AM local)
  • Transactional (OTP, security alerts): always send regardless of quiet hours

14 — Deep Dives

ClickHouse (Column-Oriented Analytics DB)

Created by Yandex. Data stored by column, not row. When a query only needs 2 columns out of 20, it reads 10–100x less data.

Row-oriented (Postgres):
  Row 1: [user_1, push, opened,  2026-04-05 10:00, ...]
  Row 2: [user_2, email, clicked, 2026-04-05 10:01, ...]
  → Read entire rows even if query only needs channel + action

Column-oriented (ClickHouse):
  Column "channel": [push, email, sms, push, ...]
  Column "action":  [opened, clicked, delivered, ...]
  → Read only the 2 columns needed. 10x less I/O.

SELECT channel, countIf(action = 'opened')
FROM notification_events
WHERE date = today()
GROUP BY channel;
→ ClickHouse: ~100ms over 1 BILLION rows
→ Postgres: minutes

Append-Only Data Model

Notification events are never updated or deleted. Only new events are appended:

{ notification: ntf_123, action: SENT,      ts: 10:00:00 }
{ notification: ntf_123, action: DELIVERED,  ts: 10:00:02 }
{ notification: ntf_123, action: OPENED,     ts: 10:05:30 }

Ideal for Kafka (append-only log), S3 (immutable files), ClickHouse (bulk inserts), Cassandra (write-optimised). No update conflicts → easiest to scale.

Idempotency

Caller includes idempotency_key in request. API does Redis SETNX "idem:{key}" EX 86400.

  • Returns 1 (new) → process normally
  • Returns 0 (seen) → return cached response, don’t re-enqueue

Prevents upstream bugs (retry storms, double-clicks) from creating duplicate notifications.

15 — Interview Q&A

How do you ensure exactly-once delivery?

You can’t — not across external providers. FCM/APNs/SendGrid don’t guarantee it. Design for at-least-once: retry on failure, accept possible duplicates. For SMS (costly), use provider idempotency keys. The idempotency key at the API layer prevents duplicate requests, but delivery itself is at-least-once.

Marketing blast to 50M users blocks OTPs. How do you prevent it?

Priority queues: OTPs go to priority-0, marketing to priority-2. Workers drain priority-0 first. Better: separate worker pools per priority — dedicated transactional workers never starved by marketing. Also throttle marketing enqueue rate (100K/min).

User gets 20 push notifications in 10 minutes. How to prevent spam?

Per-user per-channel rate limiting (Redis sliding window: max 5 push/hour). Exceeded → drop (marketing) or delay (batch into digest). Notification collapsing: Android collapse_key / iOS thread-id collapses same-type notifications into one.

How do you handle quiet hours across timezones?

Store timezone in preferences. Worker checks: current time in user’s timezone within quiet hours? Marketing/social → delay to next allowed window. Transactional (OTP, security) → always send regardless.

SendGrid goes down for 2 hours. What happens?

Exponential backoff retries → after max retries → DLQ + alert. Circuit breaker stops hammering dead provider. When recovered, replay DLQ. For critical emails, failover to backup provider (SES as secondary).

How do you track if a notification was actually seen?

Delivered: FCM/APNs returns success. Opened/Clicked: push → client-side callback; email → tracking pixel (opens) + redirect URL (clicks). “Delivered” ≠ “seen” — true seen requires client instrumentation, not all platforms support reliably.

What monitoring metrics would you track?
  • Notifications sent/sec by channel (drop = provider issue?)
  • Delivery success rate (alert if < 95%)
  • DLQ depth (growing = provider down?)
  • P99 end-to-end latency (transactional should be < 30s)
  • Per-user rate limit hits (spike = upstream bug?)
  • Template render errors and device token invalidation rate
SDE-2 vs SDE-3: what differentiates the answer?
SDE-2SDE-3
Clean architecture, Kafka for async, priority queues, retry with backoffAll of SDE-2 + per-user rate limiting, DLQ + circuit breaker, idempotency, notification collapsing, quiet hours + timezone, provider failover, analytics pipeline, capacity planning for marketing blasts
