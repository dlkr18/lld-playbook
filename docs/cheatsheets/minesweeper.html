01 — Requirements

Functional

  • Create a board of configurable rows × cols with N randomly placed mines
  • Reveal a cell — if mine → game lost; if zero adjacent mines → auto-expand (flood fill)
  • Flag / unflag a cell to mark suspected mines
  • Win detection — all non-mine cells revealed
  • Support difficulty presets: Easy (9×9, 10), Medium (16×16, 40), Hard (16×30, 99)
  • Game timer tracking elapsed seconds
  • Score recording with player name, time, and difficulty
  • On loss, reveal all mines for post-game inspection

Non-Functional

  • Encapsulation — Cell state only mutated via controlled methods
  • Extensible — new difficulty levels without code changes
  • Efficient reveal — BFS flood fill avoids stack overflow on large boards
  • Immutable coordinates — Cell (row, col) set once at construction
  • Separation of concerns — model vs game logic vs demo

03 — Design Patterns

Focused patterns for a game domain — each solving a concrete problem.

Encapsulation

Cell hides internal state behind controlled mutation methods. No direct field access. reveal() is one-way, toggleFlag() guards against revealed cells. Board enforces all rules.

Enum Configuration

Difficulty encapsulates board dimensions and mine count. Adding a new level = one line. GameStatus enum replaces magic strings.

Aggregate Root

Board is the aggregate root — all cell access goes through it. External code never manipulates cells directly. Board owns initialization, reveal, flag, and win-check.

Value Object

Score is immutable after construction. Cell(row, col) coordinates are final. No identity — equality by value.

04 — Game State Machine

Three states, strict transitions. GameStatus controls what actions are legal. Reveal on a lost game? Silently rejected. All non-mine cells revealed? Auto-transition to WON. 
stateDiagram-v2
    [*] --> IN_PROGRESS: new Board(rows, cols, mines)
    IN_PROGRESS --> WON: all non-mine cells revealed
    IN_PROGRESS --> LOST: mine cell revealed
    WON --> [*]
    LOST --> [*]

GameStatus Enum

public enum GameStatus {
    IN_PROGRESS,   // game is active, reveal & flag allowed
    WON,           // all non-mine cells revealed
    LOST           // mine was revealed, all mines exposed
}

State Transitions in Board

public boolean revealCell(int row, int col) {
    if (!isValid(row, col) || grid[row][col].isRevealed()
            || grid[row][col].isFlagged()) return false;

    Cell cell = grid[row][col];
    cell.reveal();

    if (cell.hasMine()) {
        status = GameStatus.LOST;     // ← transition to LOST
        revealAllMines();              // expose all mines for display
        return false;
    }

    if (cell.getAdjacentMines() == 0) {
        revealAdjacentCells(row, col); // ← flood fill
    }

    if (checkWin()) {
        status = GameStatus.WON;      // ← transition to WON
    }
    return true;
}

05 — Board Generation

Three-phase initialization: (1) create empty cells, (2) randomly place mines, (3) compute adjacent mine counts.

Phase 1: Grid Allocation

private void initializeBoard() {
    for (int i = 0; i < rows; i++)
        for (int j = 0; j < cols; j++)
            grid[i][j] = new Cell(i, j);   // immutable coordinates
    placeMines();
    calculateAdjacentMines();
}

Phase 2: Random Mine Placement

private void placeMines() {
    Random random = new Random();
    int placed = 0;
    while (placed < mineCount) {
        int row = random.nextInt(rows);
        int col = random.nextInt(cols);
        if (!grid[row][col].hasMine()) {
            grid[row][col].placeMine();     // retry-loop avoids duplicates
            placed++;
        }
    }
}

Complexity: Expected O(mines) if mine density is low. Worst case (near-full board) degrades — use Fisher-Yates shuffle for guaranteed O(rows×cols).

Phase 3: Adjacent Mine Counting

private int countAdjacentMines(int row, int col) {
    int count = 0;
    for (int dr = -1; dr <= 1; dr++) {
        for (int dc = -1; dc <= 1; dc++) {
            if (dr == 0 && dc == 0) continue;
            int nr = row + dr, nc = col + dc;
            if (isValid(nr, nc) && grid[nr][nc].hasMine())
                count++;
        }
    }
    return count;  // 0–8 for each non-mine cell
}

Complexity: O(rows × cols) — each cell checks up to 8 neighbors. Runs once at board creation.

06 — Flood Fill Reveal

Core algorithm. When a cell with 0 adjacent mines is revealed, recursively reveal all neighbors. Two implementations: recursive (Board) and iterative BFS (MinesweeperGame). 
flowchart TD
    A["revealCell(r, c)"] --> B{valid & unrevealed?}
    B -->|no| Z[return false]
    B -->|yes| C[cell.reveal]
    C --> D{has mine?}
    D -->|yes| E["status = LOST
revealAllMines()"]
    D -->|no| F{adjacentMines == 0?}
    F -->|yes| G["BFS flood fill
enqueue neighbors"]
    F -->|no| H[show number]
    G --> I["for each neighbor:
reveal & enqueue if 0"]
    I --> J[checkWin]
    H --> J
    J --> K[return true]

Iterative BFS (MinesweeperGame) — Preferred

if (cell.getAdjacentMines() == 0) {
    Queue<int[]> queue = new LinkedList<>();
    queue.add(new int[]{row, col});
    while (!queue.isEmpty()) {
        int[] pos = queue.poll();
        for (int di = -1; di <= 1; di++) {
            for (int dj = -1; dj <= 1; dj++) {
                int ni = pos[0] + di, nj = pos[1] + dj;
                if (ni >= 0 && ni < rows && nj >= 0 && nj < cols) {
                    Cell neighbor = board[ni][nj];
                    if (!neighbor.isRevealed() && !neighbor.isFlagged()
                            && !neighbor.isMine()) {
                        neighbor.reveal();
                        if (neighbor.getAdjacentMines() == 0)
                            queue.add(new int[]{ni, nj});
                    }
                }
            }
        }
    }
}

Why BFS over DFS? Iterative BFS uses a queue on the heap — no stack overflow risk on large boards (e.g., 16×30 = 480 cells). Recursive DFS on Board works for small grids but risks StackOverflow.

Recursive DFS (Board) — Simpler

private void revealAdjacentCells(int row, int col) {
    for (int dr = -1; dr <= 1; dr++) {
        for (int dc = -1; dc <= 1; dc++) {
            if (dr == 0 && dc == 0) continue;
            int nr = row + dr, nc = col + dc;
            if (isValid(nr, nc) && !grid[nr][nc].isRevealed())
                revealCell(nr, nc); // recursive call
        }
    }
}

07 — Flagging & Win Check

Flag Toggle

public void toggleFlag(int row, int col) {
    if (isValid(row, col) && !grid[row][col].isRevealed()) {
        grid[row][col].toggleFlag();  // isFlagged = !isFlagged
    }
}

Guards: Cannot flag a revealed cell. Flagged cells cannot be revealed (protected from accidental clicks). Toggle design allows easy un-flag.

Win Condition

private boolean checkWin() {
    for (int i = 0; i < rows; i++)
        for (int j = 0; j < cols; j++) {
            Cell cell = grid[i][j];
            if (!cell.hasMine() && !cell.isRevealed())
                return false;   // still unrevealed non-mine cells
        }
    return true;  // ALL non-mine cells revealed → WON
}

Note: Flagging mines is NOT required to win. The only condition is that every non-mine cell is revealed.

09 — Thread Safety

ConcernCurrent ApproachProduction Enhancement
Single-player game No concurrency needed (single thread) For multiplayer: synchronized(board) on reveal/flag
Board initialization Happens in constructor (thread-confined) Safe — object not shared until constructed
Timer access Simple start/stop volatile running or AtomicBoolean for cross-thread timer reads
Scoreboard Score is immutable (thread-safe by design) ConcurrentSkipListSet for concurrent leaderboard
Game state GameStatus enum (single field) volatile GameStatus for visibility across threads
"Minesweeper is inherently single-player, so concurrency isn't a primary concern. For a multiplayer variant, I'd add synchronized on the Board for reveal/flag operations and use volatile for the game status field."

13 — Interview Follow-up Q&A

Why BFS instead of DFS for flood fill?
Recursive DFS risks StackOverflowError on large boards (e.g., 30×16 = 480 cells, potentially hundreds of recursive calls). BFS uses a heap-allocated LinkedList queue — bounded only by memory, not call stack depth. Both are O(R×C) time, but BFS is production-safe.
How would you ensure the first click is never a mine?
Defer mine placement until the first revealCell call. Pass the first-click (row, col) to placeMines() and exclude that cell (and optionally its 8 neighbors) from mine candidates. This is how the original Windows Minesweeper works.
Why two Cell implementations?
The root Cell is lightweight (no coordinates) — suitable when the game tracks position externally via 2D array indices. The model Cell carries immutable (row, col) and has toString() for display. This demonstrates that entity design depends on context — the Board aggregate knows cell positions, so model.Cell is self-describing.
How would you handle a multiplayer Minesweeper?
Synchronize revealCell and toggleFlag with synchronized(board). Use volatile for GameStatus. Add player turns or concurrent reveal (cooperative mode). For networked play, introduce a GameServer that serializes game state and broadcasts updates via WebSocket.
What's the mine placement complexity?
The retry loop is expected O(mines) when mine density is low (<50%). For worst-case (near-full board), it degrades. A better approach: create a list of all cell positions, apply Fisher-Yates shuffle, take the first N — guaranteed O(rows×cols) regardless of density.
How would you optimize checkWin()?
Maintain a counter unrevealed = (rows * cols) - mineCount. Decrement on each reveal. When counter reaches 0, the player wins. O(1) per reveal instead of O(R×C) scan. Same correctness, dramatically better for large boards.
Why is Score immutable?
Score is a value object — once recorded, it never changes. Immutability gives inherent thread safety (no synchronization needed for a shared leaderboard), hashability (can be used as Map keys), and eliminates defensive copying.
How would you add an undo feature?
Use the Command pattern. Each action (reveal, flag) becomes a command object with execute() and undo(). Maintain a stack of commands. undo() would need to un-reveal cells, which means storing the set of cells revealed by a single flood-fill action.
How does the Difficulty enum follow Open/Closed Principle?
Adding a new difficulty (e.g., EXPERT(20, 40, 150)) requires adding one enum constant. No existing code changes. The Board constructor accepts raw (rows, cols, mines) — it doesn't depend on the enum, so custom boards work too.
How would you persist game state for save/load?
Serialize the Board: grid dimensions, mine positions, revealed/flagged state per cell, current GameStatus, and elapsed time. Use JSON or Protocol Buffers. On load, reconstruct the Board without re-running placeMines — directly set cell states. Implement a GameRepository interface with save(Board) / load(gameId).
Could you apply Observer pattern here?
Yes — Board could be the Subject, notifying observers on state changes: cell revealed (with coordinates), game won, game lost. A BoardObserver interface with onCellRevealed(row, col), onGameWon(), onGameLost(). UI components subscribe to update the display without polling.
Why is the Board an Aggregate Root?
All cell mutations go through Board methods. External code never calls cell.reveal() directly — only board.revealCell(row, col). This ensures invariants: flagged cells can't be revealed, game state transitions are always checked, and flood fill always fires when needed. The Board is the consistency boundary.
