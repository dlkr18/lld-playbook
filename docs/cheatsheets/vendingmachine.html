01 — Requirements

Functional

  • Users insert money (coins: PENNY, NICKEL, DIME, QUARTER; bills: $1, $5)
  • Users select a product by slot code (e.g., “A1”, “B2”)
  • Machine dispenses product and returns change if overpaid
  • Users can cancel transaction at any point and get a full refund
  • Machine tracks slot inventory (product, quantity, max capacity)
  • Products organized by category (Snacks, Beverages, Candy, Healthy, Coffee, Fresh)
  • Query available products, check slot availability, get slot info
  • Admin can add slots and refill products

Non-Functional

  • Thread-safe — concurrent users inserting money / selecting products
  • Precise money handling — BigDecimal, no floating-point errors
  • State-driven — illegal operations throw immediately with clear messages
  • Extensible — add new states, coin types, product categories without modifying existing code
  • Immutable value objects — Money and Product are thread-safe by design

Money — Immutable Value Object

The cornerstone of precise financial handling. Uses BigDecimal with HALF_UP rounding, 2 decimal scale. Implements Comparable<Money>.

public final class Money implements Comparable<Money> {
    public static final Money ZERO = new Money(BigDecimal.ZERO);
    public static final Money QUARTER = cents(25);
    public static final Money DOLLAR  = dollars(1);

    private final BigDecimal amount; // always scale=2

    public static Money cents(int cents) {
        return new Money(BigDecimal.valueOf(cents)
            .divide(BigDecimal.valueOf(100), 2, HALF_UP));
    }

    public Money add(Money other)     { return new Money(amount.add(other.amount)); }
    public Money subtract(Money other) { /* throws if negative */ }
    public boolean isGreaterThanOrEqual(Money other) { ... }
    public boolean isLessThan(Money other) { ... }
    public boolean isZero() { ... }
}

Key: Private constructor, all factory methods validate. subtract() throws if result would be negative — fail-fast, never a negative balance.

03 — Design Patterns

The State pattern is the star. Every interview answer should reference it.

State Pattern

Machine lifecycle transitions. Each state (IDLE, HAS_MONEY, PRODUCT_SELECTED, DISPENSING) is a stateless singleton class. Invalid ops throw immediately with descriptive messages.

Singleton (per State)

Each state class uses private constructor + static INSTANCE field + getInstance(). States are stateless — all mutable data lives on VendingMachineImpl.

Value Object

Money is immutable with BigDecimal precision. Product is immutable with id-based equality. Both are inherently thread-safe.

Interface Segregation

VendingMachine interface separates customer ops, query ops, state management, and internal ops. Clean contract for testing and extensions.

04 — State Machine

Why State pattern? Eliminates scattered if/else on status checks. Each state class defines EXACTLY which transitions are legal. Adding a new state (e.g. MAINTENANCE) = adding a new class. Open/Closed Principle. 
stateDiagram-v2
    [*] --> Idle
    Idle --> HasMoney: insertMoney()
    HasMoney --> HasMoney: insertMoney() [add more]
    HasMoney --> ProductSelected: selectProduct()
    HasMoney --> Idle: cancel() [refund all]
    ProductSelected --> ProductSelected: selectProduct() [change selection]
    ProductSelected --> ProductSelected: insertMoney() [add more]
    ProductSelected --> Dispensing: dispense()
    ProductSelected --> HasMoney: cancel() [clear selection]
    Dispensing --> Idle: product dispensed + reset

VendingMachineImpl Delegates to State

// Every customer operation delegates to current state
@Override
public synchronized void insertMoney(Money money) {
    currentState.insertMoney(this, money);
}

@Override
public synchronized Product selectProduct(String slotCode) {
    this.selectedSlotCode = slotCode;
    return currentState.selectProduct(this, slotCode);
}

@Override
public synchronized Product dispense() {
    return currentState.dispense(this);
}

States are STATELESS singletons (INSTANCE field). All mutable data (balance, selectedProduct, slots) stays on VendingMachineImpl. States manipulate the machine via its public interface methods.

05 — State Implementations

IdleState — Waiting for Money

public class IdleState implements VendingMachineState {
    private static final IdleState INSTANCE = new IdleState();
    private IdleState() {}
    public static IdleState getInstance() { return INSTANCE; }

    void insertMoney(machine, money) {
        // Validate non-null, non-zero
        machine.addToBalance(money);
        machine.setState(HasMoneyState.getInstance()); // → HAS_MONEY
    }
    Product selectProduct(...) { throw "Please insert money first"; }
    Product dispense(...)       { throw "Please insert money and select a product first"; }
    Money   cancel(...)         { return Money.ZERO; } // nothing to refund
}

HasMoneyState — Money Inserted

void insertMoney(machine, money) {
    machine.addToBalance(money); // accumulate, stay in HAS_MONEY
}

Product selectProduct(machine, slotCode) {
    Slot slot = machine.getSlot(slotCode);
    if (slot == null || slot.isEmpty()) throw "Product not available";

    Product product = slot.getProduct();
    if (balance.isLessThan(product.getPrice()))
        throw "Insufficient funds. Price: X, Balance: Y";

    machine.setSelectedProduct(product);
    machine.setState(ProductSelectedState.getInstance()); // → SELECTED
    return product;
}

Money cancel(machine) {
    Money refund = machine.getCurrentBalance();
    machine.resetTransaction();      // zero balance, clear selection
    machine.setState(IdleState.getInstance()); // → IDLE
    return refund;
}

ProductSelectedState — Ready to Dispense

void insertMoney(machine, money) {
    machine.addToBalance(money); // can still add more money
}

Product selectProduct(machine, slotCode) {
    // Allows CHANGING selection (re-validates availability + funds)
    machine.setSelectedProduct(product);
    return product; // stays in PRODUCT_SELECTED
}

Product dispense(machine) {
    machine.setState(DispensingState.getInstance());
    return machine.getCurrentState().dispense(machine); // delegate to DISPENSING
}

Money cancel(machine) {
    machine.clearSelectedProduct();
    machine.setState(HasMoneyState.getInstance()); // → HAS_MONEY (keep balance)
    return Money.ZERO; // no refund, just deselect
}

DispensingState — Dispensing Product

Product dispense(machine) {
    Product selected = machine.getSelectedProduct();
    // Find matching slot in available slots
    for (Slot slot : machine.getAvailableSlots()) {
        if (slot.getProduct().getId().equals(selected.getId())) {
            machine.deductFromBalance(selected.getPrice()); // charge
            Product dispensed = machine.dispenseFromSlot(slot.getCode()); // qty--
            machine.resetTransaction();  // zero balance, clear product
            machine.setState(IdleState.getInstance()); // → IDLE
            return dispensed;
        }
    }
    throw "Selected product no longer available";
}

// ALL other operations throw IllegalStateException
void insertMoney(...) { throw "Cannot insert money during dispensing"; }
void selectProduct(...) { throw "Cannot select product during dispensing"; }
void cancel(...) { throw "Cannot cancel during dispensing"; }

06 — Key Methods

cancelTransaction() — Smart Refund Logic

@Override
public synchronized Money cancelTransaction() {
    Money refund = currentState.cancel(this);
    // If state's cancel didn't clear balance, handle remainder
    if (!currentBalance.isZero()) {
        Money remaining = currentBalance;
        resetTransaction();
        setState(IdleState.getInstance());
        return remaining.add(refund);
    }
    return refund;
}

Two-phase cancel: first delegates to state (ProductSelected returns ZERO, keeps money), then machine checks for remaining balance as a safety net.

Slot.dispense() — Thread-safe Inventory

public synchronized Product dispense() {
    if (isEmpty()) throw new IllegalStateException("Slot " + code + " is empty");
    quantity--;
    return product;
}

public synchronized void refill(Product product, int qty) {
    this.product = product;
    this.quantity = Math.min(this.quantity + qty, maxCapacity);
}

public enum Coin {

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap VendingMachineImpl.slots Safe concurrent slot lookup/insertion by multiple threads
synchronized methods insertMoney, selectProduct, dispense, cancelTransaction Serializes all customer operations on the machine instance — prevents race between insert and dispense
synchronized methods Slot.dispense(), Slot.refill(), Slot.setProduct() Per-slot lock prevents two concurrent dispenses from over-decrementing quantity
final fields Money (all fields), Product (all fields) Immutable value objects are inherently thread-safe — no synchronization needed
Singleton states All 4 state classes Stateless — no shared mutable data. Eagerly initialized (thread-safe class loading).
"Customer operations are synchronized on the machine. Slot operations are synchronized on the slot. Money and Product are immutable. States are stateless singletons. Four concurrency layers, each appropriate for its scope."

Design choice: Internal methods are on the interface so states can manipulate the machine context. In production, use a separate internal interface or package-private access.

09 — Happy Path Flow

Trace this end-to-end in the interview.

Setup Machine
VendingMachineImpl machine = new VendingMachineImpl();
machine.addSlot(new Slot("A1", cola, 10, 15));
Machine starts in IDLE state, balance = $0.00
Insert Money
machine.insertMoney(Money.DOLLAR); // $1.00
machine.insertMoney(Money.QUARTER); // $0.25
State: IDLE → HAS_MONEY. Balance accumulates: $1.25. IdleState.insertMoney() transitions on first coin, HasMoneyState.insertMoney() adds subsequent coins.
Select Product
Product selected = machine.selectProduct("A1");
HasMoneyState checks: slot exists? not empty? balance ≥ price? Sets selectedProduct, transitions → PRODUCT_SELECTED.
Dispense
Product dispensed = machine.dispense();
ProductSelectedState transitions → DISPENSING, then DispensingState.dispense(): deducts price from balance, calls slot.dispense() (qty--), resets transaction, → IDLE.
Machine Ready
State: IDLE, Balance: $0.00, selectedProduct: null
Machine is clean and ready for the next customer.

Alternative: Cancel Flow

Insert & Cancel
machine.insertMoney(Money.DOLLAR); // HAS_MONEY
Money refund = machine.cancelTransaction();
HasMoneyState.cancel() returns full balance ($1.00), resets transaction → IDLE. Customer gets full refund.
Select & Change Mind
machine.insertMoney(Money.DOLLAR);
machine.selectProduct("A1"); // PRODUCT_SELECTED
machine.cancelTransaction();
ProductSelectedState.cancel() clears product selection → HAS_MONEY. Then machine detects remaining balance and refunds it → IDLE.

11 — Interview Follow-up Q&A

Why State pattern instead of if/else on a status enum?
(1) Each state class is self-documenting — you see exactly what's legal and what throws. (2) Adding a new state like MAINTENANCE is a new class with no modification to existing code (Open/Closed Principle). (3) Invalid transitions throw immediately with clear error messages — no forgotten else branches. (4) States are stateless singletons, so zero memory overhead.
Why is Money immutable and uses BigDecimal?
Immutability: Thread-safe by design, no defensive copies needed. BigDecimal: Floating-point arithmetic causes rounding errors (e.g., 0.1 + 0.2 != 0.3 in double). Financial calculations MUST use BigDecimal with explicit scale and rounding mode. Money.subtract() throws if result is negative — impossible to have a negative balance.
How do you handle change/overpayment?
The balance accumulates all inserted money. When dispensing, deductFromBalance(price) is called, leaving the remainder. In the current implementation, resetTransaction() zeroes the balance. To return change, capture balance - price before reset and return it. The Money class's subtraction handles the math precisely.
What if two users try to use the machine simultaneously?
All customer operations (insertMoney, selectProduct, dispense, cancelTransaction) are synchronized on the VendingMachineImpl instance. Only one thread operates at a time. Physical vending machines are inherently single-user, so instance-level locking is appropriate. For a virtual/API vending machine, you'd use per-session state.
How would you add a new state like MAINTENANCE?
Create MaintenanceState implements VendingMachineState as a singleton. All operations throw "Machine is under maintenance". Add a transition: admin calls setState(MaintenanceState.getInstance()). No existing state classes need modification (Open/Closed). Add exit transition in MaintenanceState to go back to IDLE.
Why are states singletons vs new instances?
States hold ZERO mutable data — they're pure behavior. Creating new instances per transition would waste memory and GC pressure for identical objects. Singleton pattern is natural here: private constructor, static INSTANCE, getInstance(). Eager initialization (static final) is thread-safe via class loading guarantees.
How would you extend this with payment strategies (cash, card, mobile)?
Add a PaymentStrategy interface with pay(Money amount) and refund(Money amount). Implementations: CashPaymentStrategy (current behavior), CardPaymentStrategy, MobilePaymentStrategy. Inject via constructor or setter on VendingMachineImpl. The State pattern remains unchanged — just swap how payment is processed.
How would you handle product expiration?
Add an expirationDate field to Product. In Slot.isEmpty(), also check if the product is expired. A background ScheduledExecutorService periodically scans slots and marks expired products. This keeps the state machine clean — expired products simply appear as "not available" during selection.
Why does ProductSelectedState.cancel() go to HasMoney instead of Idle?
User experience: the customer inserted money and wants to pick a different product, not lose their money. Cancel from ProductSelected means "I changed my mind about THIS product" not "I want my money back." Going to HasMoney lets them select again. To actually get a refund, they cancel again from HasMoney state.
How would you implement the change-dispensing mechanism?
Add a CoinInventory class tracking available coins for change. Use a greedy algorithm: start with largest denomination (QUARTER), compute how many fit into the change amount, then DIME, NICKEL, PENNY. If exact change can't be made, throw InsufficientChangeException before dispensing. This is checked in DispensingState before deducting.

13 — Patterns Summary

PatternWhereWhy
State VendingMachineState → 4 singletons Machine lifecycle without if/else. Legal transitions enforced per-state.
Singleton IdleState, HasMoneyState, ProductSelectedState, DispensingState States are stateless — one instance shared globally. Zero overhead.
Value Object Money (BigDecimal), Product (immutable) Immutable, thread-safe, equality by value. Financial precision.
Enum Pattern Coin, ProductCategory Type-safe constants with behavior (toMoney(), getDisplayName())
Interface Segregation VendingMachine interface sections Customer ops, query ops, state management, internals — clear separation
Repository ConcurrentHashMap<String, Slot> In-memory slot store, easy to swap for DB
