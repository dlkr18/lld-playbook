01 — Requirements

Functional

  • Multiple log levels: TRACE, DEBUG, INFO, WARN, ERROR, FATAL with priority ordering
  • Level-based filtering — only emit logs at or above the configured threshold
  • Pluggable appenders — Console, File, Network destinations via a common interface
  • Centralized log collection from multiple services into a single store
  • Search and filter logs by level, source, keyword, and time range
  • Retention management — auto-evict oldest logs when capacity is reached
  • Log aggregation — ingest, search, and retrieve recent logs across services
  • Structured log entries with timestamp, thread name, logger name, level, and message

Non-Functional

  • Thread-safe — concurrent log writes from multiple threads without corruption
  • Low latency — logging should not bottleneck the calling application (<1ms per log)
  • Extensible destinations — add new appenders (Kafka, Elasticsearch) without modifying core
  • Scalable storage — bounded memory with configurable max capacity
  • Pluggable formatting — swap log format (JSON, plain text, structured) at runtime
  • 10,000+ concurrent producers supported

03 — Design Patterns

Four patterns, each solving a distinct concern in the logging pipeline.

Strategy Pattern

LogAppender (where to write) and LogFormatter (how to format). Add new destinations — FileAppender, KafkaAppender, ElasticsearchAppender — without touching LoggerImpl.

Observer Pattern

Logger holds a List<LogAppender> (observers). Every log call dispatches the entry to ALL registered appenders. Add/remove appenders at runtime.

Chain of Responsibility

Log level filtering. Each log call checks level.getPriority() >= threshold.getPriority(). Below threshold? Silently dropped. Above? Passed to appenders. Chainable with parent loggers.

Singleton Pattern

CentralizedLoggingService — single shared instance collecting logs from all sources. Thread-safe via ConcurrentLinkedQueue. One aggregation point for search and analytics.

04 — Log Levels & Filtering

LogLevel Enum

public enum LogLevel {
    TRACE(0), DEBUG(1), INFO(2), WARN(3), ERROR(4), FATAL(5);

    private final int priority;

    LogLevel(int priority) {
        this.priority = priority;
    }

    public int getPriority() {
        return priority;
    }
}

Level-Based Filtering (Chain of Responsibility)

private void log(LogLevel level, String message) {
    if (level.getPriority() >= this.level.getPriority()) {
        LogEntry entry = new LogEntry(level, message, name);
        for (LogAppender appender : appenders) {
            appender.append(entry);  // dispatch to all registered appenders
        }
    }
    // Below threshold? Silently dropped -- zero overhead
}

If logger level = WARN(3), then DEBUG(1) and INFO(2) are dropped. WARN(3), ERROR(4), FATAL(5) pass through.

CentralizedLoggingService

public class CentralizedLoggingService implements LoggingService {

    private final Queue<LogEntry> logs;
    private final int maxLogSize;

    public CentralizedLoggingService() {
        this(10000); // default max 10K logs
    }

    public CentralizedLoggingService(int maxLogSize) {
        this.logs = new ConcurrentLinkedQueue<>();
        this.maxLogSize = maxLogSize;
    }

    @Override
    public void log(LogLevel level, String message, String source) {
        LogEntry entry = new LogEntry(level, message, source, LocalDateTime.now());
        logs.offer(entry);

        // Evict oldest logs when exceeding capacity
        while (logs.size() > maxLogSize) {
            logs.poll();
        }
    }
}

LogEntry — Immutable Record

public class LogEntry {
    private final LogLevel level;
    private final String message;
    private final LocalDateTime timestamp;
    private final String thread;
    private final String logger;

    public LogEntry(LogLevel level, String message, String logger) {
        this.level     = level;
        this.message   = message;
        this.timestamp = LocalDateTime.now();
        this.thread    = Thread.currentThread().getName();
        this.logger    = logger;
    }

    @Override
    public String toString() {
        return String.format("[%s] %s [%s] %s - %s",
            timestamp, level, thread, logger, message);
    }
}

All fields are final. Thread name captured at creation time — critical for async debugging. No setters — immutable by design.

07 — Thread Safety

MechanismWhereWhy
ConcurrentLinkedQueue CentralizedLoggingService.logs Lock-free FIFO queue — multiple threads can offer/poll concurrently without blocking
final fields LogEntry (all fields) Immutability eliminates data races — safe to share across threads without synchronization
Thread.currentThread() LogEntry constructor Captures calling thread name at log time, not at read time — accurate async tracing
volatile Logger level field (production) Level changes visible across threads immediately without full synchronization
CopyOnWriteArrayList Appender list (production) Safe iteration during log dispatch while appenders are added/removed concurrently
"ConcurrentLinkedQueue gives us O(1) offer/poll without locks. In a distributed logging system, the write path (log ingestion) must NEVER block the calling application. Lock-free data structures are essential."

08 — Querying & Search

Filter by Level

@Override
public List<LogEntry> getLogsByLevel(LogLevel level) {
    return logs.stream()
        .filter(log -> log.getLevel() == level)
        .collect(Collectors.toList());
}

Filter by Source

@Override
public List<LogEntry> getLogsBySource(String source) {
    return logs.stream()
        .filter(log -> log.getSource().equals(source))
        .collect(Collectors.toList());
}

Keyword Search (Case-Insensitive)

@Override
public List<LogEntry> searchLogs(String keyword) {
    String lowerKeyword = keyword.toLowerCase();
    return logs.stream()
        .filter(log -> log.getMessage().toLowerCase().contains(lowerKeyword) ||
                      log.getSource().toLowerCase().contains(lowerKeyword))
        .collect(Collectors.toList());
}

Recent Logs (Tail)

@Override
public List<LogEntry> getRecentLogs(int count) {
    List<LogEntry> allLogs = new ArrayList<>(logs);
    int size = allLogs.size();
    int startIndex = Math.max(0, size - count);
    return allLogs.subList(startIndex, size);
}

09 — Retention & Cleanup

Bounded storage. Without retention, logs consume unbounded memory. The service auto-evicts oldest entries when maxLogSize is exceeded — FIFO eviction via poll().

Eviction Logic

public void log(LogLevel level, String message, String source) {
    LogEntry entry = new LogEntry(level, message, source, LocalDateTime.now());
    logs.offer(entry);

    // Ring-buffer behavior: evict oldest when full
    while (logs.size() > maxLogSize) {
        logs.poll(); // remove head (oldest)
    }
}

public void clearLogs() {
    logs.clear(); // manual purge for maintenance windows
}

Production Extensions

StrategyImplementation
Size-based (current)Evict oldest when count exceeds maxLogSize
Time-basedScheduledExecutorService purges entries older than N days
Level-basedKeep ERROR/FATAL longer than DEBUG/TRACE
Tiered storageHot (memory) → Warm (disk) → Cold (S3/GCS)

End-to-end trace: from logger creation to centralized search.

Logger created with name "PaymentService", default level = INFO
Register Appenders

logger.addAppender(new FileAppender("/var/log/app.log"))
Two observers (appenders) registered. Both will receive every log entry that passes the level filter.

Threshold lowered from INFO(2) to DEBUG(1). Now DEBUG and above will pass through.
Log Messages

logger.info("Payment accepted")
logger.error("Gateway timeout")
Each call: check priority ≥ threshold → create LogEntry (captures timestamp + thread) → dispatch to ALL appenders
Centralized Ingestion
loggingService.log(ERROR, "Gateway timeout", "PaymentService")
Appenders forward to CentralizedLoggingService. Entry added to ConcurrentLinkedQueue. Auto-evicts if > maxLogSize.
Search Logs
loggingService.searchLogs("timeout") → [LogEntry{ERROR, "Gateway timeout", ...}]
Case-insensitive keyword search across message and source fields. Stream + filter + collect.
Filter by Level
loggingService.getLogsByLevel(ERROR) → all ERROR entries
Useful for alerting dashboards — show only errors and fatals.
Recent Logs (Tail)
loggingService.getRecentLogs(50) → last 50 entries
Snapshot of queue, subList from (size-50) to end. Equivalent to tail -n 50.

Why These Choices?

StructureAlternativeWhy Chosen
ConcurrentLinkedQueue LinkedBlockingQueue Lock-free — no blocking on write path. Logging must never slow the caller.
ArrayList<Appender> CopyOnWriteArrayList Simple for single-threaded use. Upgrade to COWAL for concurrent appender registration.
Enum with int priority String constants Type-safe, O(1) comparison, ordinal-based. No string parsing or typo risk.

13 — Interview Follow-up Q&A

How would you handle millions of log events per second?
Use an async appender backed by a ring buffer (like LMAX Disruptor or Log4j2's AsyncLogger). Producers write to the ring buffer in nanoseconds; a dedicated consumer thread drains to appenders. Also shard by service/partition key and use Kafka as the transport layer with batching and compression.
Why ConcurrentLinkedQueue instead of LinkedBlockingQueue?
ConcurrentLinkedQueue is lock-free (uses CAS operations). LinkedBlockingQueue uses two ReentrantLocks (put/take) which cause contention under high throughput. For a logging system, the write path must NEVER block the calling thread. We accept slightly higher memory overhead for lock-free performance.
How would you implement structured logging (JSON)?
Implement a JsonLogFormatter that converts LogEntry to JSON. The LogFormatter interface is already a Strategy — just inject the new formatter. Add structured fields (MDC-style context map) to LogEntry for request IDs, user IDs, trace IDs.
How would you add log correlation across microservices?
Add a correlationId (trace ID) to LogEntry. Pass it via HTTP headers (X-Request-Id) or gRPC metadata. Store in ThreadLocal (MDC pattern). All logs for a single request share the same trace ID, enabling distributed tracing when aggregated in Elasticsearch/Splunk.
How would you implement log rotation for file appenders?
Implement a RollingFileAppender that checks file size or time before each write. When threshold is reached, close current file, rename with timestamp suffix, and open a new file. Use a configurable RolloverPolicy (size-based, time-based, or composite). Delete or compress old files based on retention.
Why Strategy pattern for appenders instead of inheritance?
Strategy allows runtime composition. A single logger can dispatch to Console + File + Kafka simultaneously. With inheritance, you'd need a multi-appender subclass for every combination. Strategy also makes testing trivial — inject a mock appender that captures entries.
How would you implement log level changes at runtime without restart?
The setLevel() method already supports this. In production, expose it via a JMX bean, admin REST endpoint, or watch a config file (e.g., ZooKeeper/Consul). The volatile keyword on the level field ensures cross-thread visibility. Useful for enabling DEBUG on a production service temporarily.
What's the difference between Logger and LoggingService?
Logger is the producer-side API — application code calls logger.info("..."). It handles level filtering and appender dispatch. LoggingService is the backend/centralized API — it stores, indexes, and queries logs from ALL services. Logger writes; LoggingService reads. In production, an appender bridges the two (e.g., KafkaAppender → LoggingService consumer).
How would you handle backpressure when the log consumer can't keep up?
Three strategies: (1) Drop — bounded queue that discards oldest entries (current approach). (2) Block — LinkedBlockingQueue with capacity; producers wait (risky for latency-sensitive code). (3) Sample — probabilistic logging at high throughput (log 1 in N events). The right choice depends on whether you prioritize completeness or latency.
How would you make the keyword search faster than O(n)?
Build an inverted index (like Elasticsearch). Tokenize each log message on write, map tokens → Set<LogEntry>. Keyword search becomes an O(1) map lookup + set intersection. Trade-off: more memory and write overhead for O(1) reads. In production, use Elasticsearch or Apache Lucene.
How would you implement log sampling for high-volume debug logs?
Create a SamplingAppender (Decorator pattern) that wraps any appender and only forwards 1-in-N entries for DEBUG/TRACE levels. Use ThreadLocalRandom for lock-free sampling. ERROR and FATAL always pass through (sample rate = 1.0). Configurable per-level sampling rates.
Why is LogEntry immutable? What if I need to add metadata later?
Immutability makes LogEntry thread-safe by construction — no synchronization needed when shared across appenders running in different threads. For additional metadata, use the MDC (Mapped Diagnostic Context) pattern: add an immutable Map<String,String> context field populated at creation time from a ThreadLocal context map.
