01 — Requirements

Functional

  • Users register with name and email
  • Create groups with multiple members (e.g., Roommates, Trip)
  • Add expenses with different split types: Equal, Exact, Percentage
  • Track pairwise balances between every pair of users
  • Settle up — view who owes whom and how much
  • Debt simplification — minimize number of transactions to settle all debts
  • Support non-group expenses (direct between two users)
  • Record payments/settlements between users

Non-Functional

  • Thread-safe — concurrent expense additions must not corrupt balances
  • Extensible split types — add SHARE or custom strategies without modifying core
  • Accurate arithmetic — handle rounding, remainder assignment, penny-exact splits
  • Scalable — in-memory for LLD, swappable to DB-backed via interface
  • Audit trail — every expense and payment timestamped

03 — Design Patterns

Two core patterns that earn their place in every Splitwise interview.

Strategy Pattern

Split calculation logic varies by type. Each SplitType drives a different calculation in Expense.calculateSplits(). Adding a new type (SHARE, WEIGHT) = adding a new case. In the advanced version, SplitCalculator encapsulates each strategy as a static method.

Observer Pattern

Notify users when expenses are added, balances change, or settlements occur. NotificationService interface decouples delivery (console, push, email) from business logic.

Builder Pattern

Expense.Builder (advanced version) handles complex construction with split type, currency, group assignment, and validation in build().

Repository Pattern

ConcurrentHashMap stores act as in-memory repositories. The SplitwiseService interface abstracts data access — swap to JPA/Mongo without touching business logic.

Split Calculation (Expense.java)

private void calculateSplits() {
    switch (splitType) {
        case EQUAL:
            double share = amount / participants.size();
            for (String userId : participants)
                splits.put(userId, share);
            break;

        case EXACT:
            double total = 0;
            for (String userId : participants) {
                double exactAmt = exactAmounts.getOrDefault(userId, 0.0);
                splits.put(userId, exactAmt);
                total += exactAmt;
            }
            if (Math.abs(total - amount) > 0.01)
                throw new IllegalArgumentException("Exact amounts don't add up");
            break;

        case PERCENTAGE:
            double totalPct = 0;
            for (String userId : participants) {
                double pct = percentages.getOrDefault(userId, 0.0);
                splits.put(userId, amount * pct / 100.0);
                totalPct += pct;
            }
            if (Math.abs(totalPct - 100.0) > 0.01)
                throw new IllegalArgumentException("Percentages don't sum to 100%");
            break;
    }
}

// SCALE=2, HALF_UP rounding — remainder goes to first participant

BigDecimal baseSum = share.multiply(BigDecimal.valueOf(count));

05 — Debt Simplification

Goal: Minimize the number of transactions needed to settle all debts. Without simplification, N users can have up to N(N-1)/2 pairwise debts. The greedy algorithm reduces this to at most N-1 transactions.

Algorithm: Two-Pointer Greedy Matching

  1. Compute net balances — for each user, sum all money owed to them minus all money they owe
  2. Sort by net balance — most negative (biggest debtor) on left, most positive (biggest creditor) on right
  3. Two pointers — left pointer at biggest debtor, right at biggest creditor
  4. Match — settle min(|debt|, credit), update balances, advance pointer if settled
  5. Repeat until all balances are zero
public static List<Transaction> simplifyBalances(Map<String, Double> balances) {
    List<Transaction> transactions = new ArrayList<>();

    List<Map.Entry<String, Double>> sorted = new ArrayList<>(balances.entrySet());
    sorted.sort(Map.Entry.comparingByValue());

    int left = 0, right = sorted.size() - 1;

    while (left < right) {
        double debtAmount  = -sorted.get(left).getValue();   // negative → positive
        double creditAmount =  sorted.get(right).getValue();

        double amount = Math.min(debtAmount, creditAmount);
        transactions.add(new Transaction(debtor, creditor, amount));

        sorted.get(left).setValue(sorted.get(left).getValue() + amount);
        sorted.get(right).setValue(sorted.get(right).getValue() - amount);

        if (Math.abs(sorted.get(left).getValue())  < 0.01) left++;
        if (Math.abs(sorted.get(right).getValue()) < 0.01) right--;
    }
    return transactions;
}

Time: O(N log N) for sorting + O(N) for matching = O(N log N). Space: O(N).

06 — Balance Tracking

Key data structure: Each User holds a Map<String, Double> balances where positive = others owe you, negative = you owe others. The map is symmetric: if A's balance with B is +50, B's balance with A is -50.

How updateBalance Works

// When Alice pays $300 for dinner split equally among Alice, Bob, Charlie:
// Each person's share = $100

// For Bob (non-payer, share = $100):
payer.updateBalance(bob, +100);    // Alice: "Bob owes me $100"
bob.updateBalance(payer, -100);    // Bob: "I owe Alice $100"

// For Charlie (non-payer, share = $100):
payer.updateBalance(charlie, +100); // Alice: "Charlie owes me $100"
charlie.updateBalance(payer, -100); // Charlie: "I owe Alice $100"

// Payer's own share is skipped (you don't owe yourself)

07 — Key Algorithms

1. Add Expense

Time: O(P) where P = number of participants

  1. Create Expense with split calculation based on SplitType
  2. Validate splits sum to total amount
  3. Link expense to group
  4. For each non-payer participant: update pairwise balances (both directions)

2. Calculate Splits

Time: O(P)

TypeAlgorithmEdge Case
EQUALamount / countRemainder penny to first participant
EXACTUse provided amountsMust validate sum = total
PERCENTAGEamount * pct / 100Must validate percentages sum to 100; last participant gets remainder
SHAREamount * share / totalSharesSame remainder handling as percentage

3. Simplify Debts

Time: O(N log N)   Space: O(N)

  1. Aggregate all pairwise balances into a net balance per user
  2. Sort users by net balance (debtors first, creditors last)
  3. Two-pointer greedy: match biggest debtor with biggest creditor
  4. Output minimized list of Transaction(from, to, amount)

4. Settle Up

Time: O(B) where B = number of balance entries for a user

  1. Iterate user's balance map
  2. For each entry: positive → "X owes you $Y", negative → "You owe X $Y"
  3. Skip entries within ±$0.01 (effectively zero)

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap users, groups, expenses stores Safe concurrent reads/writes to data stores
synchronized updateBalances() method Atomic balance updates — two concurrent expenses must not interleave balance writes
Defensive copies getBalances(), getSplits(), getParticipants() Return new HashMap<>(map) to prevent external mutation
final fields Expense, Payment, Group IDs Immutable after construction — safe to share across threads
"The synchronized updateBalances() method ensures that adding two expenses concurrently doesn't corrupt the pairwise balance maps. Each user's balance map is updated atomically for a given expense. In production, I'd use per-user locks or optimistic locking (version field) for finer granularity."

Why Map<UserId, Double> on User?

  • O(1) lookup — check balance between any two users instantly
  • Incremental updates — each expense only touches affected pairs
  • No global recalculation — balances stay current after each expense
  • Symmetric invariant — A.balances[B] = -B.balances[A] always

12 — Interview Follow-up Q&A

Is debt simplification NP-hard? Why use a greedy approach?
Optimal (minimum number of transactions) debt simplification is NP-hard — it reduces to the subset-sum problem. However, the greedy two-pointer approach gives at most N-1 transactions (where N = number of users with non-zero balance), which is optimal for most practical cases. For groups of 3-10 people, the greedy result is nearly always optimal. In production, you'd only need an exact solution for pathological cases with many users.
How would you handle multiple currencies?
Add a currency field to Expense (already in the advanced model). Maintain separate balance maps per currency: Map<String, Map<String, Double>> (currency → userId → balance). Simplification runs independently per currency. For cross-currency settlement, integrate an exchange rate service and convert at settlement time, not at expense time.
Group expenses vs. non-group expenses — what's the difference?
Both use the same balance tracking mechanism. Group expenses are linked to a Group entity and appear in group history. Non-group expenses have groupId = null and only affect the two users involved. The balance update logic is identical — groupId is purely organizational. In the API, non-group expenses pass null for groupId.
How would you implement recurring expenses?
Add a RecurringExpense entity with frequency (WEEKLY, MONTHLY), template fields (amount, participants, splitType), and nextDueDate. A ScheduledExecutorService periodically checks for due expenses and calls addExpense() with the template data. The recurring expense stores a reference to all generated expense IDs for audit.
How do you handle partial settlements?
A partial payment updates balances by the paid amount, not the full owed amount. recordPayment(payerId, payeeId, amount) calls payer.updateBalance(payee, +amount) and payee.updateBalance(payer, -amount). The remaining balance is naturally preserved. The Payment entity records the partial amount for audit.
How would you implement an audit trail?
Every Expense and Payment already has createdAt timestamp. For a full audit trail: (1) store an ActivityLog for every balance-changing operation (expense added, expense deleted, payment recorded), (2) include before/after snapshots of affected balances, (3) make the log append-only (Event Sourcing). This enables "undo expense" by replaying the log without the deleted event.
Why BigDecimal in the advanced version instead of double?
double has floating-point representation errors (e.g., 0.1 + 0.2 ≠ 0.3). For financial calculations, BigDecimal with SCALE=2 and HALF_UP rounding guarantees penny-exact arithmetic. The basic version uses double for simplicity with a 0.01 epsilon tolerance. In production, always use BigDecimal.
How would you handle expense deletion / editing?
Deleting an expense requires reversing the balance updates: for each split entry, subtract instead of add. Store the original splits in the Expense entity (we already do). Editing = delete old + add new. Wrap both operations in a synchronized block to ensure atomicity. Mark deleted expenses as DELETED rather than physically removing them (soft delete for audit).
How would you scale to millions of users?
(1) Database-backed: Replace ConcurrentHashMap with a relational DB. Balance table: (user_a, user_b, amount) with composite primary key. (2) Sharding: Partition by group_id since most queries are group-scoped. (3) Caching: Redis for hot balance queries. (4) Event-driven: Publish expense events to Kafka, update balances asynchronously. (5) CQRS: Separate read (balance queries) from write (expense creation) paths.
What if two people add expenses to the same group simultaneously?
The synchronized updateBalances() method serializes balance updates. Two concurrent addExpense() calls will create their Expense objects concurrently (safe — ConcurrentHashMap), but balance updates are serialized. In production with a database, use optimistic locking (version column on balances) or SELECT FOR UPDATE on the affected balance rows.
