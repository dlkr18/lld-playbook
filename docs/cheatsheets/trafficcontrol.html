01 — Requirements

Functional

  • Model traffic lights with RED, YELLOW, GREEN signals per direction
  • Manage intersections with 4 directional lights (N, S, E, W)
  • Implement signal cycling: RED → GREEN → YELLOW → RED
  • Support fixed-time and adaptive timing strategies
  • Emergency override — set all lights to RED instantly
  • Round-robin scheduling — one direction green at a time, coordinated pairs (N-S / E-W)
  • Start / stop the entire traffic cycle system
  • Real-time status queries for any intersection

Non-Functional

  • Thread-safe — multiple intersections cycled concurrently via ScheduledExecutorService
  • Performance — response time < 100ms for signal changes
  • Extensible timing — swap fixed for adaptive or emergency without modifying controller
  • Scalable — support 10,000+ concurrent intersections
  • Reliability — fail-safe to RED on errors (99.9% uptime)
  • Observable events — signal change notifications for monitoring and logging

03 — Design Patterns

Three core patterns, each earning its place in the design.

State Pattern

Signal enum (RED, YELLOW, GREEN) drives the light phase. Each state defines legal transitions: RED → GREEN, GREEN → YELLOW, YELLOW → RED. Emergency forces all lights to RED unconditionally.

Strategy Pattern

Timing strategies: fixed-interval, adaptive (traffic-density), and emergency override. Injected into the controller, swappable at runtime without code changes.

Observer Pattern

Signal changes fire events for monitoring dashboards, logging systems, and pedestrian signals. Decouples signal state from consumers of signal events.

Composite Pattern

Intersection composes 4 TrafficLights keyed by Direction. Manages them as a unit — coordinated pair cycling (N-S green while E-W red, then swap).

04 — Signal State Machine

Why State? Each signal phase has exactly one legal next phase. State pattern eliminates scattered if/else and makes invalid transitions impossible. Emergency override short-circuits to RED from any state.

05 — Timing Strategies

Why Strategy? Timing algorithms change independently of the controller. New timing model = new class. Swappable at runtime without modifying existing code.

Fixed-Interval Strategy (Implemented)

// ScheduledExecutorService runs at fixed rate
cycleTask = scheduler.scheduleAtFixedRate(() -> {
    for (Intersection intersection : intersections.values()) {
        if (!intersection.isEmergencyMode()) {
            cycleLights(intersection);
        }
    }
}, 0, 10, TimeUnit.SECONDS);
StrategyLogicUse Case
Fixed Interval N-S green 5s → yellow 2s → E-W green 5s, repeating at 10s rate Low-traffic intersections, default mode
Adaptive Adjust green duration based on traffic density sensors per direction High-traffic intersections, rush hours
Emergency Override ALL lights → RED immediately, bypass normal cycling Ambulance, fire truck, accident response

Cycle Coordination: Paired Directions

private void cycleLights(Intersection intersection) {
    Map<Direction, TrafficLight> lights = intersection.getLights();

    // Phase 1: North-South GREEN, East-West RED
    lights.get(Direction.NORTH).changeSignal(Signal.GREEN);
    lights.get(Direction.SOUTH).changeSignal(Signal.GREEN);
    lights.get(Direction.EAST).changeSignal(Signal.RED);
    lights.get(Direction.WEST).changeSignal(Signal.RED);

    Thread.sleep(5000); // green phase

    // Phase 2: N-S YELLOW transition
    lights.get(Direction.NORTH).changeSignal(Signal.YELLOW);
    lights.get(Direction.SOUTH).changeSignal(Signal.YELLOW);
    Thread.sleep(2000); // yellow phase

    // Phase 3: East-West GREEN, North-South RED
    lights.get(Direction.NORTH).changeSignal(Signal.RED);
    lights.get(Direction.SOUTH).changeSignal(Signal.RED);
    lights.get(Direction.EAST).changeSignal(Signal.GREEN);
    lights.get(Direction.WEST).changeSignal(Signal.GREEN);
}

06 — Intersection Management

Intersection: Composite of 4 Lights

public class Intersection {
    private final String id;
    private final String name;
    private final Map<Direction, TrafficLight> lights;
    private boolean emergencyMode;

    public Intersection(String id, String name) {
        this.id = id;
        this.name = name;
        this.lights = new HashMap<>();
        initializeLights(); // auto-creates 4 directional lights
    }

    private void initializeLights() {
        for (Direction direction : Direction.values()) {
            String lightId = id + "_" + direction;
            lights.put(direction, new TrafficLight(lightId, id, direction));
        }
    }
}

Each intersection auto-initializes 4 lights keyed by direction. Light IDs follow pattern: {intersectionId}_{DIRECTION}.

Emergency Mode

public void setEmergencyMode(boolean emergency) {
    this.emergencyMode = emergency;
    if (emergency) {
        for (TrafficLight light : lights.values()) {
            light.changeSignal(Signal.RED); // ALL RED immediately
        }
    }
}

Emergency mode is a boolean flag checked by the scheduler. When enabled, the intersection is skipped during normal cycling and all 4 lights remain RED until emergency is disabled.

Defensive Copy on getLights()

public Map<Direction, TrafficLight> getLights() {
    return new HashMap<>(lights); // defensive copy — prevents external mutation
}

"Returning a copy prevents callers from adding/removing lights directly. Only the intersection manages its own light collection."

07 — Key Algorithms

Round-Robin Light Cycling

Time Complexity: O(d) where d = number of directions (constant 4)

Space Complexity: O(1)

// Coordinated pair cycling: N-S ↔ E-W
// Phase 1: N-S GREEN (5s) → Phase 2: N-S YELLOW (2s) → Phase 3: E-W GREEN
//
// Key invariant: at most ONE pair of opposing directions is GREEN at any time.
// All other directions are RED. This prevents cross-traffic collisions.
//
// Algorithm:
// 1. Set N-S to GREEN, E-W to RED
// 2. Sleep(greenDuration)
// 3. Set N-S to YELLOW
// 4. Sleep(yellowDuration)
// 5. Set N-S to RED, E-W to GREEN
// 6. Next scheduler tick repeats from step 1

Emergency Override

Time Complexity: O(d) — sets all lights to RED

// Algorithm:
// 1. Lookup intersection by ID from ConcurrentHashMap — O(1)
// 2. Set emergencyMode = true
// 3. Iterate all lights, set each to RED — O(d)
// 4. Scheduler skips this intersection on next tick (checks isEmergencyMode())
// 5. To resume: set emergencyMode = false, next tick resumes cycling

Scheduled Cycling with ScheduledExecutorService

private final ScheduledExecutorService scheduler =
    Executors.newScheduledThreadPool(2);
private ScheduledFuture<?> cycleTask;

public void startCycle() {
    if (cycleTask != null && !cycleTask.isDone()) {
        return; // idempotent — already running
    }
    cycleTask = scheduler.scheduleAtFixedRate(() -> {
        for (Intersection i : intersections.values()) {
            if (!i.isEmergencyMode()) {
                cycleLights(i);
            }
        }
    }, 0, 10, TimeUnit.SECONDS);
}

Thread pool of 2 — one for cycling, one for emergency overrides. scheduleAtFixedRate ensures steady cadence regardless of cycle duration.

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap intersections store Safe concurrent reads/writes when adding intersections while cycling
ScheduledExecutorService TrafficControllerImpl Thread pool of 2 for concurrent cycling & emergency handling
ScheduledFuture<?> cycleTask handle Idempotent start — checks isDone() before creating new task
Thread.currentThread().interrupt() cycleLights catch block Proper interrupt handling — restores interrupt flag on InterruptedException
Defensive copy Intersection.getLights() Returns new HashMap to prevent external modification of internal state
cancel(false) stopCycle() Graceful shutdown — lets in-progress cycle complete before stopping
"I use ConcurrentHashMap for the intersection store and a ScheduledExecutorService with 2 threads for cycling. Emergency override is non-blocking — it sets a flag checked by the scheduler, so it doesn't contend with the cycling thread."

12 — Interview Follow-up Q&A

Why use State pattern for traffic signals?
Each signal phase (RED, GREEN, YELLOW) has exactly one legal successor. The State pattern makes transitions explicit and prevents invalid ones (e.g., RED → YELLOW). Adding a new phase like FLASHING_RED means adding one state class — no modifications to existing code (Open/Closed Principle). It also eliminates scattered if/else chains for signal logic.
How would you implement adaptive timing based on traffic density?
Extract a TimingStrategy interface with getDuration(Direction, Signal). The adaptive implementation would receive traffic density data (from sensors, cameras, or loop detectors) and dynamically adjust green duration — busier directions get longer green phases. The TrafficControllerImpl holds a strategy reference, swappable at runtime. No changes to Intersection or TrafficLight.
How does emergency override work without race conditions?
setEmergencyMode(true) sets a boolean flag and immediately sets all 4 lights to RED. The scheduler's cycleLights() checks isEmergencyMode() before processing — if true, it skips the intersection entirely. Since the flag is set atomically (boolean write is atomic in Java) and checked on each tick, there's no window where the scheduler could green-light an emergency intersection.
How do you prevent cross-traffic collisions?
The cycling algorithm guarantees a key invariant: at most ONE pair of opposing directions (N-S or E-W) can be GREEN simultaneously. All transitions go through a YELLOW phase before the other pair goes GREEN. The 2-second yellow gap provides an all-red period where no direction is green. Emergency override forces all to RED as the ultimate safety fallback.
Why ScheduledExecutorService instead of Timer?
ScheduledExecutorService offers: (1) thread pool — Timer uses a single thread, so one slow intersection blocks all others; (2) exception handling — Timer silently dies on uncaught exceptions, ScheduledExecutorService isolates failures; (3) scheduleAtFixedRate vs scheduleWithFixedDelay semantics for precise cadence control; (4) clean shutdown() and ScheduledFuture cancellation.
How would you scale to 10,000+ intersections?
Increase the thread pool size proportionally (e.g., Runtime.getRuntime().availableProcessors()). Partition intersections into groups and assign each group to a thread. For true horizontal scaling, shard intersections across multiple controller instances by geographic zone. Each controller manages its zone independently. A central coordinator handles cross-zone scenarios (e.g., emergency vehicle routing across zones).
How would you add pedestrian crossing signals?
Observer pattern: pedestrian signals register as observers on the intersection. When a traffic light changes to RED (via changeSignal()), it fires an event. The pedestrian signal observer sees RED on the traffic light and shows WALK for its direction. This keeps pedestrian logic decoupled from traffic light logic — the light doesn't know about pedestrians.
Why does Intersection auto-initialize 4 lights?
Encapsulation — an intersection is always a complete unit with exactly 4 directional lights. It prevents partially-constructed intersections where a direction has no light. The initializeLights() method iterates Direction.values(), so adding a new direction (e.g., NORTHEAST for a 5-way intersection) only requires updating the enum. Light IDs follow a deterministic pattern: {intersectionId}_{DIRECTION}.
How would you handle a power failure (fail-safe)?
All lights default to RED on construction (currentSignal = Signal.RED). On system restart, no intersection has a green light until the controller explicitly starts cycling. This is the fail-safe principle: unknown state = all RED. In hardware, this maps to dark/flashing-red mode. The boolean emergencyMode defaults to false, so cycling resumes normally on restart.
How would you implement left-turn signals?
Extend the Direction enum to include NORTH_LEFT, SOUTH_LEFT, etc. Each left-turn direction gets its own TrafficLight. The cycling algorithm adds a protected left-turn phase: after N-S green ends, N-S left-turn gets a brief green window before E-W goes green. This is a new phase in the cycle, not a new state in the signal enum.
Why defensive copy in getLights()?
Returning new HashMap<>(lights) prevents external code from directly adding/removing lights from the intersection. Only the intersection manages its own light collection (encapsulation). Without the copy, a caller could do getLights().remove(Direction.NORTH) and corrupt the intersection's state. The TrafficLight objects themselves are still mutable — in production, return unmodifiable views.
How would you add real-time monitoring?
Observer pattern: create a SignalChangeListener interface. Each TrafficLight fires an event on changeSignal(). Monitoring dashboards, logging systems, and analytics register as listeners. Use an event bus (Guava EventBus or Kafka) at scale to decouple producers from consumers. Current System.out.println in changeSignal() is a placeholder for this.
