Pub/Sub System // Message Broker

SDE3-level low-level design with Observer, Strategy, and Singleton patterns. Async message delivery via BlockingQueue, thread-safe fan-out, and pull-based acknowledgement.

01 — Requirements

Functional

  • Topic management — create, delete topics dynamically
  • Publish — publishers send messages to named topics
  • Subscribe / Unsubscribe — subscribers attach to topics, receive a subscription ID
  • Fan-out delivery — every subscriber of a topic gets every message published to it
  • Pull-based consumption — subscribers pull messages via getMessages(subscriptionId)
  • Acknowledgement — subscribers ack messages to remove them from their queue
  • Async processing — messages queued in a BlockingQueue, processed by background thread
  • Real-time status updates and notification support
  • Search, filtering, and reporting/analytics capabilities

Non-Functional

  • Performance — response time < 100ms for critical operations
  • Thread-safe — multiple publishers and subscribers operating concurrently
  • Scalable — support 10,000+ concurrent users
  • Reliable — 99.9% uptime, message durability guarantees
  • Extensible — pluggable delivery strategies (push, pull, hybrid)
  • Data Consistency — ACID transactions where needed
  • Security — authentication, authorization, data encryption
  • Multi-region — deployment ready for availability

03 — Design Patterns

Three patterns, each earning its place in the interview.

Observer Pattern

Core of the pub/sub paradigm. Topics are Subjects, Subscribers are Observers. When a message is published, the broker fans it out to all registered subscribers. Loose coupling — publishers don't know about subscribers.

Strategy Pattern

Delivery strategies: push-based (callback via onMessage), pull-based (queue + getMessages), or hybrid. Swappable at runtime — new delivery model = new class.

Singleton (Broker)

PubSubSystem is the single broker instance. Owns the topic registry, message queue, and thread pool. Single point of coordination for all publish/subscribe operations.

Repository Pattern

InMemoryPubSubService abstracts data access behind PubSubService interface. Swap for database-backed, hybrid (cache + DB), or distributed implementation without changing business logic.

Push vs Pull Model (Two Implementations)

ModelClassMechanism
Push PubSubSystem Background thread takes from BlockingQueue, calls subscriber.onMessage() via thread pool
Pull InMemoryPubSubService Messages enqueued per subscription in ConcurrentLinkedQueue. Subscribers call getMessages() + acknowledgeMessage()

Push-Based: PubSubSystem Core Loop

private void startMessageProcessor() {
    executor.submit(() -> {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                Message message = messageQueue.take(); // blocks until msg available
                deliverMessage(message);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    });
}

private void deliverMessage(Message message) {
    List<Subscriber> subs = topicSubscribers.get(message.getTopic());
    if (subs != null) {
        for (Subscriber sub : subs) {
            executor.submit(() -> sub.onMessage(message)); // async fan-out
        }
    }
}

Pull-Based: InMemoryPubSubService Publish

public boolean publish(String topicName, Message message) {
    Topic topic = topics.get(topicName);
    if (topic == null) return false;

    // Fan-out: enqueue message for every subscription
    for (String subId : topic.getSubscriptionIds()) {
        Queue<Message> queue = subscriberMessages.computeIfAbsent(
            subId, k -> new ConcurrentLinkedQueue<>());
        queue.offer(message);
    }
    return true;
}

05 — Delivery Semantics

Key interview topic: Understanding the trade-offs between delivery guarantees is critical for system design discussions.
SemanticGuaranteeMechanismUse Case
At-Most-Once Fire-and-forget. Message may be lost but never duplicated. No ack, no retry. Our push model — executor.submit(onMessage). If subscriber crashes, message is gone. Metrics, logs, non-critical notifications
At-Least-Once Message is delivered one or more times. May have duplicates. Queue + ack model. Our pull model — message stays in ConcurrentLinkedQueue until acknowledgeMessage() is called. Redeliver on timeout. Order processing, payments, emails
Exactly-Once Message delivered exactly once. Hardest to achieve. At-least-once + idempotency key on subscriber side. Dedup via messageId + processed-set. Transactional outbox in production. Financial transactions, inventory updates

06 — Key Algorithms

Topic Matching & Message Routing

Time Complexity: O(S) per publish where S = number of subscribers on the topic

// 1. Publisher calls publish(topicName, message)
// 2. Broker looks up topic → O(1) via ConcurrentHashMap
// 3. For each subscriptionId in topic.getSubscriptionIds():
//    - Enqueue message into subscriber's queue → O(1)
// Total: O(1) lookup + O(S) fan-out = O(S)

Topic topic = topics.get(topicName);        // O(1)
for (String subId : topic.getSubscriptionIds()) {  // O(S)
    subscriberMessages.get(subId).offer(message);  // O(1) per sub
}

Retry with Exponential Backoff

// For at-least-once delivery: retry failed deliveries
int maxRetries = 5;
long baseDelay = 1000; // 1 second

for (int attempt = 0; attempt < maxRetries; attempt++) {
    try {
        subscriber.onMessage(message);
        return; // success
    } catch (Exception e) {
        long delay = baseDelay * (1L << attempt); // 1s, 2s, 4s, 8s, 16s
        Thread.sleep(delay + randomJitter());
    }
}
sendToDeadLetterQueue(message); // all retries exhausted

Message Acknowledgement (Pull Model)

public boolean acknowledgeMessage(String subscriptionId, String messageId) {
    Queue<Message> queue = subscriberMessages.get(subscriptionId);
    if (queue == null) return false;
    return queue.removeIf(m -> m.getId().equals(messageId));
}
// O(n) scan of subscriber queue — in production, use a Map<msgId, Message>

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap topics, subscriptions, subscriberMessages stores Safe concurrent reads/writes without global lock
LinkedBlockingQueue PubSubSystem.messageQueue Thread-safe producer-consumer queue. offer() never blocks, take() blocks until message available
ConcurrentLinkedQueue Per-subscriber message queues Lock-free queue for fan-out message storage. Multiple publishers can enqueue concurrently
ExecutorService PubSubSystem (fixed pool of 4) Async message delivery — fan-out to subscribers without blocking the processor thread
AtomicLong subscriptionIdGenerator Lock-free unique subscription ID generation (SUB-1, SUB-2, ...)
computeIfAbsent subscribe(), publish() Atomic check-and-create for subscriber lists and queues
"The BlockingQueue decouples publishers from subscribers. Publishers never block — offer() is O(1). The background processor thread does the heavy lifting of fan-out delivery. Subscribers get notified asynchronously via the ExecutorService thread pool."

Message Ordering Guarantees

  • Per-topic ordering — LinkedBlockingQueue is FIFO, so messages within a topic are processed in publish order
  • Per-subscriber ordering — NOT guaranteed because executor.submit() dispatches to a thread pool. Subscriber A may process message 2 before message 1
  • To guarantee per-subscriber ordering — use a single-threaded executor per subscriber, or a partitioned queue

11 — Interview Follow-up Q&A

How do you guarantee message ordering?
The LinkedBlockingQueue is FIFO, so messages are processed in publish order per topic. However, fan-out via executor.submit() means different subscribers may receive messages out of order. Fix: Use a single-threaded executor per subscriber, or partition messages by key and assign partitions to dedicated threads (like Kafka's partition model). For global ordering, use a single-partition topic.
What is a Dead Letter Queue (DLQ) and when would you use one?
A DLQ stores messages that fail delivery after all retry attempts are exhausted. Implementation: after N retries with exponential backoff, move the message to a separate deadLetterQueue topic. This prevents poison messages from blocking the pipeline. Operators can inspect, fix, and replay DLQ messages manually or via automation. Our acknowledgeMessage() mechanism naturally supports this — unacked messages past a TTL get moved to DLQ.
How would you handle backpressure?
Producer side: Use a bounded ArrayBlockingQueue instead of LinkedBlockingQueue. When full, offer() returns false — the publisher gets a signal to slow down. Consumer side: Rate-limit delivery with a token bucket or leaky bucket per subscriber. Queue size monitoring: Expose messageQueue.size() as a metric. Alert when queue depth exceeds thresholds. In production, Kafka handles this with consumer lag metrics.
How would you partition topics for horizontal scaling?
Split each topic into N partitions. Each message is assigned to a partition via hash(messageKey) % N. Each partition is consumed by exactly one consumer in a consumer group. Benefits: (1) Parallelism — N consumers process concurrently. (2) Ordering — guaranteed within a partition. (3) Scalability — add partitions and consumers independently. This is exactly how Kafka works. In our codebase, we'd add a Partition entity between Topic and Subscription.
What are consumer groups and how do they differ from plain subscriptions?
Plain subscription (our model): Every subscriber gets every message (fan-out / broadcast). Consumer group: A group of subscribers shares the load — each message goes to exactly ONE member of the group (load-balanced). Implementation: add a groupId field to Subscription. During fan-out, for each group, deliver to only one member (round-robin or least-loaded). Different groups still get independent copies of every message.
How do you achieve exactly-once delivery?
Exactly-once is at-least-once + idempotency. Producer side: Assign a unique messageId (already in our Message). Enable producer retries. Broker deduplicates by messageId + topic (keep a Set of recent IDs). Consumer side: Maintain a processedMessageIds set. Before processing, check if messageId was already handled. Use a transactional outbox pattern: write the processed result and ack in the same DB transaction. Kafka achieves this with idempotent producers + transactional consumers.
How would you implement message filtering / content-based routing?
Add a filter predicate to Subscription: Predicate<Message> filter. During fan-out, only enqueue if filter.test(message) returns true. For attribute-based filtering, use message metadata: message.metadata.get("priority").equals("HIGH"). For topic-based filtering, support wildcards: orders.* matches orders.created, orders.cancelled. Implement with a trie-based topic matcher.
How do you handle subscriber failures and retries?
In the push model: wrap subscriber.onMessage() in a try-catch. On failure, schedule a retry with exponential backoff (1s, 2s, 4s, 8s, 16s + jitter). After max retries, send to DLQ. In the pull model (our InMemoryPubSubService): messages stay in the queue until acknowledged. Add a visibility timeout — if not acked within T seconds, the message becomes visible again for redelivery (like SQS). Track delivery attempts in message metadata.
How would you scale the broker for millions of messages/second?
Horizontal partitioning: Shard topics across multiple broker nodes via consistent hashing. Disk persistence: Append messages to a commit log (like Kafka) — sequential disk writes are faster than random. Zero-copy transfer: Use sendfile() syscall to transfer data from disk to network without user-space copies. Batching: Batch messages for network efficiency. Replication: Leader-follower replication per partition for fault tolerance. Consumer groups: Parallelize consumption across group members.
What's the difference between Pub/Sub and Message Queue?
Pub/Sub (our model): Messages are broadcast to ALL subscribers of a topic (fan-out). Each subscriber gets its own copy. Good for event notification, real-time updates. Message Queue: Messages are delivered to exactly ONE consumer (competing consumers). Good for task distribution, work queues. Hybrid: Consumer groups give you queue semantics within a group + pub/sub semantics across groups. Kafka supports both models.
How do you handle message persistence and durability?
Our in-memory implementation loses messages on crash. Production approach: (1) Write-Ahead Log (WAL): Persist every published message to an append-only log before delivering. On recovery, replay the log. (2) Replication: Replicate partitions across brokers. Configurable ack: acks=all waits for all replicas. (3) Retention policy: Keep messages for N days or until consumed. (4) Swap InMemoryPubSubService for a database-backed implementation — the PubSubService interface stays unchanged.
Why use a BlockingQueue instead of direct delivery?
The BlockingQueue decouples publishers from the delivery mechanism. (1) Publishers never block — offer() is O(1), instant return. (2) Back-pressure is natural — bounded queue rejects when full. (3) Async processing — the background thread handles fan-out independently. (4) Burst absorption — queue buffers spikes in publish rate. (5) Easy to add batching, priority ordering, or rate limiting by swapping queue implementations.
How would you implement message TTL (time-to-live)?
Add a ttl field to Message and an expiresAt timestamp. During delivery or getMessages(), check if LocalDateTime.now().isAfter(expiresAt) and skip expired messages. Run a periodic cleanup task via ScheduledExecutorService to purge expired messages from subscriber queues. For the push model, check TTL before calling onMessage(). This prevents stale data from being processed.
How does topic deletion cascade to subscriptions?
Our deleteTopic() implementation cascades cleanly: (1) Remove the Topic from the topics map. (2) Iterate over the topic's subscriptionIds. (3) For each subscription, remove from subscriptions map AND clean up the subscriber's message queue from subscriberMessages map. This prevents orphaned data. In production, you'd also notify active subscribers that their topic was deleted.

13 — Patterns Summary

PatternWhereWhy
Observer Topic (Subject) → Subscribers (Observers) Core pub/sub paradigm: loose coupling, fan-out without publisher knowing subscribers
Strategy Delivery strategy (push via callback, pull via queue+ack) Swap delivery mechanisms without changing broker logic
Singleton PubSubSystem (central broker) Single coordination point for topic registry, queue, and thread pool
Repository PubSubService → InMemoryPubSubService Abstract data access; swap for DB-backed or distributed impl
Producer-Consumer BlockingQueue + background processor thread Decouple publish from delivery; absorb bursts, enable async fan-out
Value Object Message (immutable: final fields, no setters) Thread-safe by design, safe to share across subscriber queues
Factory Subscription ID generation (AtomicLong + "SUB-" prefix) Encapsulated, unique ID creation
