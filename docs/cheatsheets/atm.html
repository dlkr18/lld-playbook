01 — Requirements

Functional

  • Card Authentication — validate card number, expiry, PIN (max 3 attempts), block card after failures
  • Cash Withdrawal — select account, enter amount (multiples of 10, max $10,000), check balance, dispense optimal denominations ($100, $50, $20, $10)
  • Balance Inquiry — display available balance, support multiple account types (Savings, Current, Credit)
  • Cash Deposit — accept denominations, credit to account, generate deposit slip
  • PIN Management — change PIN (old PIN + new PIN twice), 4–6 digit validation
  • Mini Statement — last 5 transactions with date, type, amount, balance

Non-Functional

  • Performance — card auth < 100ms, cash dispensing < 5s
  • Security — PIN encryption (bcrypt), audit logging, session timeout (30s inactivity)
  • Reliability — 99.9% uptime, low-cash alerts, network failure resilience
  • Concurrency — thread-safe cash dispenser, atomic balance updates

03 — Design Patterns

The interview differentiator — five patterns, each earning its place.

State Pattern

ATM lifecycle transitions. Each ATMState defines exactly which operations are legal. IDLE → CARD_INSERTED → PIN_ENTERED → TRANSACTION_SELECTED → CASH_DISPENSED. Invalid ops throw immediately.

Chain of Responsibility

Cash dispensing chain: $100 → $50 → $20 → $10. Each handler dispenses its denomination and passes remaining to the next. Clean separation per denomination.

Strategy Pattern

DispensingStrategy (greedy vs DP) and AuthenticationStrategy (PIN, biometric). Injected at construction, swappable at runtime.

Command Pattern

Each transaction (Withdraw, Deposit, BalanceInquiry) is a Command object with execute() and undo(). Enables rollback on failure and transaction logging.

Singleton Pattern

ATM instance per physical machine. Ensures single point of control for cash inventory and state management. Thread-safe lazy initialization.

Observer Pattern

Notify on low cash, failed auth attempts, suspicious activity. Bank operations center receives real-time alerts without coupling to ATM logic.

04 — ATM State Machine

Why State? Eliminates scattered if/else. Each state defines EXACTLY which operations are legal. Adding a new state (e.g. MAINTENANCE_MODE) = adding a new class. Open/Closed Principle. 
stateDiagram-v2
    [*] --> IDLE
    IDLE --> CARD_INSERTED: insertCard()
    CARD_INSERTED --> PIN_ENTERED: enterPIN() [valid]
    CARD_INSERTED --> IDLE: ejectCard() / 3 failures → block
    PIN_ENTERED --> TRANSACTION_SELECTED: selectTransaction()
    TRANSACTION_SELECTED --> PROCESSING: execute()
    PROCESSING --> CASH_DISPENSED: dispense [withdraw]
    PROCESSING --> IDLE: complete [inquiry/deposit]
    CASH_DISPENSED --> IDLE: ejectCard()

ATMState Enum

public enum ATMState {
    IDLE,              // waiting for card
    CARD_INSERTED,     // card read, awaiting PIN
    PIN_ENTERED,       // authenticated, select transaction
    TRANSACTION_SELECTED, // processing transaction
    CASH_DISPENSED    // cash out, eject card
}

State Transitions in ATM

public boolean insertCard(Card card) {
    if (state != ATMState.IDLE) return false;
    this.currentCard = card;
    this.state = ATMState.CARD_INSERTED;
    return true;
}

public boolean enterPIN(String pin) {
    if (state != ATMState.CARD_INSERTED) return false;
    if (currentCard.validatePin(pin)) {
        state = ATMState.PIN_ENTERED;
        return true;
    }
    return false; // Card tracks failedAttempts internally
}

public void ejectCard() {
    this.currentCard = null;
    this.state = ATMState.IDLE;
}

Each method guards on current state. Invalid transitions return false or throw. All mutable data stays on the ATM object.

CashDispenser — Greedy Largest-First

public class CashDispenser {
    private Map<Integer, Integer> cashInventory; // denomination → count

    public synchronized Map<Integer, Integer> dispenseCash(double amount) {
        Map<Integer, Integer> dispensed = new HashMap<>();
        int remaining = (int) amount;

        Integer[] denoms = {100, 50, 20, 10};
        for (int denom : denoms) {
            if (remaining >= denom && cashInventory.get(denom) > 0) {
                int notes = Math.min(remaining / denom,
                                     cashInventory.get(denom));
                if (notes > 0) {
                    dispensed.put(denom, notes);
                    cashInventory.put(denom,
                        cashInventory.get(denom) - notes);
                    remaining -= notes * denom;
                }
            }
        }
        return remaining == 0 ? dispensed : null;
    }
}

DP-Based Alternative (Optimal Bill Count)

public Map<Integer, Integer> dispenseCashDP(double amount) {
    int amt = (int) amount;
    boolean[] dp = new boolean[amt + 1];
    dp[0] = true;

    for (int denom : denominations.keySet()) {
        for (int i = denom; i <= amt; i++) {
            if (dp[i - denom] && denominations.get(denom) > 0)
                dp[i] = true;
        }
    }

    if (!dp[amt]) throw new InsufficientCashException();

    return backtrack(amt, denominations); // optimal breakdown
}

Time: O(amount × num_denominations)  |  Space: O(amount)  |  Finds a solution even when greedy fails (e.g. $600 with only $500 + $100).

06 — State Pattern Detail

Card — PIN Validation with Auto-Block

public class Card {
    private final String cardNumber;
    private final String pin;
    private CardStatus status;
    private int failedAttempts;

    public boolean validatePin(String inputPin) {
        if (status == CardStatus.BLOCKED) return false;

        if (this.pin.equals(inputPin)) {
            failedAttempts = 0;       // reset on success
            return true;
        }

        failedAttempts++;
        if (failedAttempts >= 3) {
            status = CardStatus.BLOCKED; // auto-block!
        }
        return false;
    }

    public boolean isExpired() {
        return LocalDate.now().isAfter(expiryDate);
    }
}

Account — Synchronized Balance Operations

public class Account {
    private BigDecimal balance;
    private final AccountType type;
    private final List<Transaction> transactions;

    public synchronized boolean withdraw(BigDecimal amount) {
        if (amount.compareTo(balance) > 0) return false;
        balance = balance.subtract(amount);
        return true;
    }

    public synchronized void deposit(BigDecimal amount) {
        balance = balance.add(amount);
    }
}

BigDecimal for money — never double. Synchronized to prevent overdraft from concurrent withdrawals.

07 — Key Algorithms

Cash Dispensing (Greedy)

StepLogicComplexity
1Sort denominations descending: [$100, $50, $20, $10]O(d log d)
2For each denom: notes = min(remaining/denom, available)O(d)
3Deduct from inventory, subtract from remainingO(1)
4If remaining > 0 after all denoms, return null (cannot dispense)O(1)

Overall: O(d) where d = number of denominations. Constant for fixed denominations.

Balance Check

public BigDecimal checkBalance(String accountNumber) {
    Account account = accounts.get(accountNumber);
    if (account == null) throw new IllegalArgumentException("Account not found");

    Transaction txn = new Transaction(
        UUID.randomUUID().toString(), accountNumber,
        TransactionType.BALANCE_INQUIRY,
        BigDecimal.ZERO, account.getBalance());
    account.addTransaction(txn);

    return account.getBalance();
}

PIN Validation Flow

1. Check card status → BLOCKED? reject immediately
2. Compare hashed PIN → match? reset failedAttempts, return true
3. No match? increment failedAttempts
4. failedAttempts ≥ 3? auto-block card, throw CardBlockedException
5. Return false (allow retry)

08 — Transaction Types

Withdraw

  • Validate amount (multiples of $10)
  • Check account balance
  • Check ATM cash availability
  • Dispense optimal denominations
  • Deduct balance, record transaction

Deposit

  • Accept cash/check
  • Validate denominations
  • Credit to account
  • Generate deposit slip
  • Record transaction

Balance Inquiry

  • Retrieve account balance
  • Display on screen
  • Option to print receipt
  • Record as zero-amount transaction

Transfer

  • Select source & destination accounts
  • Validate both accounts exist
  • Atomic debit + credit
  • Record paired transactions

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap cards, accounts stores Safe concurrent reads/writes for card and account lookups
synchronized CashDispenser.dispenseCash() Prevents two withdrawals from dispensing the same physical bills
synchronized Account.withdraw(), deposit() Prevents overdraft from concurrent balance modifications
BigDecimal All monetary values Avoids floating-point precision errors — never use double for money
AtomicInteger Card.failedAttempts Thread-safe increment for concurrent PIN validation attempts
"I use object-level locking on Account and CashDispenser. Two accounts can process concurrently. Only operations on the same account or the same dispenser are serialized."

Enter PIN

User selects: WITHDRAWAL
State: PIN_ENTERED → TRANSACTION_SELECTED. Screen displays options: Withdraw, Deposit, Balance, Transfer.
Enter Amount
amount = $280 (validated: multiple of $10, ≤ $10,000)
Check account balance ($5,000 ≥ $280 ✓). Check ATM inventory ($280 dispensable ✓).
Dispense Cash

synchronized(dispenser) { $100 × 2 + $50 × 1 + $20 × 1 + $10 × 1 = $280 }. Inventory decremented. State → CASH_DISPENSED.
Update Balance

synchronized(account) { $5,000 - $280 = $4,720 }. Transaction record created with UUID, timestamp, balanceAfter.

Receipt shows: date, card number (masked), transaction type, amount, remaining balance.

13 — Interview Follow-up Q&A

How do you handle denomination shortage (e.g. out of $20 bills)?
The greedy algorithm naturally skips denominations with zero inventory. If the remaining amount can't be made with available bills, dispenseCash() returns null and the ATM shows "Cannot dispense this amount." The DP alternative tries all combinations before giving up, making it more resilient. In production, low-inventory triggers an Observer alert to the cash management team.
How do you handle concurrent ATM access (multiple users)?
Each physical ATM is a single-user device, so the ATM object itself doesn't need concurrent access. The shared resources — CashDispenser and Account — are synchronized. If the same account is accessed from two different ATMs, synchronized(account) ensures only one withdrawal completes at a time. ConcurrentHashMap for lookups ensures safe concurrent reads.
How do you enforce daily withdrawal limits?
Add a dailyWithdrawn: Map<LocalDate, BigDecimal> field on Account. Before dispensing, check dailyWithdrawn.get(today) + amount ≤ dailyLimit. The limit is configurable per AccountType (e.g. Savings: $2,000, Current: $5,000). Reset automatically on date change. In production, this would be enforced at the bank's core banking system.
How would you implement receipt printing?
Strategy pattern: ReceiptPrinter interface with printReceipt(Transaction). Implementations: PhysicalReceiptPrinter (hardware), DigitalReceiptSender (email/SMS), NoOpPrinter (skip). The Transaction entity already holds all receipt data: id, type, amount, timestamp, balanceAfter. Mask card number showing only last 4 digits.
How do you handle network failure during withdrawal?
Two-phase approach: (1) Offline mode — queue the transaction locally with a PendingTransaction store. Apply pessimistic balance (deduct locally). (2) Sync on reconnect — replay pending transactions to the core banking system. If the bank rejects (insufficient funds detected server-side), reverse the local deduction and alert the customer. Daily offline limits are stricter ($500 max).
How would you detect fraud (card skimming, unusual patterns)?
Observer pattern: FraudDetectionObserver monitors all transactions. Rules: (1) Multiple failed PINs across different ATMs → block card. (2) Withdrawals from geographically distant ATMs within minutes → flag. (3) Unusual amount patterns (always max withdrawal). (4) Transaction velocity checks (more than N withdrawals per hour). In production, ML models score each transaction in real-time.
Why use BigDecimal instead of double for money?
double has floating-point precision errors: 0.1 + 0.2 = 0.30000000000000004. For financial calculations, this is unacceptable. BigDecimal provides arbitrary-precision decimal arithmetic. Use BigDecimal.valueOf() (not new BigDecimal(double)) to avoid the double constructor's precision trap. Compare with compareTo(), never equals() (scale matters).
How would you handle card retention (ATM eats the card)?
After 3 failed PIN attempts, the card is blocked AND retained. The ATM transitions to a special CARD_RETAINED state. It notifies the bank's security system via Observer. The customer sees "Card retained for security. Contact your bank." The retained card is logged with ATM ID, timestamp, and card number for physical retrieval by bank staff.
Why State pattern vs. simple if/else on ATMState enum?
(1) Each state class is self-documenting — you see exactly which operations are legal. (2) Adding MAINTENANCE_MODE is a new class, no modification to existing states (Open/Closed). (3) Invalid transitions throw immediately with clear messages. (4) In the simple enum approach shown here, the guard checks are manageable for 5 states, but State pattern scales better for 10+ states with complex transition logic.
How would you add multi-currency support?
Extend CashDispenser to hold Map<Currency, Map<Integer, Integer>>. Add a CurrencyConverter strategy interface for exchange rates. The withdrawal flow becomes: select currency → convert amount → dispense from currency-specific inventory. Each Account stores balance in its native currency; conversion happens at transaction time with a locked exchange rate.
What's the ATM's initialization / cash loading process?
CashDispenser.loadCash(Map<Integer, Integer>) is called by authorized personnel. The constructor pre-loads default inventory: {$100: 100, $50: 100, $20: 200, $10: 200}. getTotalCash() returns the sum for reconciliation. A CashLoadEvent is logged for audit. In production, the ATM enters MAINTENANCE state during loading, rejecting all customer operations.
