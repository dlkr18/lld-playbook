Functional

03 — Design Patterns

Five patterns, each earning its place in the interview.

State Machine

Order lifecycle with explicit state transitions: PLACED → CONFIRMED → PREPARING → READY → OUT_FOR_DELIVERY → DELIVERED. Invalid transitions throw immediately. Cancellation gated by current status.

Strategy Pattern

Partner matching algorithms — nearest, highest-rated, ETA-based. Injected at construction, swappable at runtime without code changes.

Observer Pattern

Order status notifications to customers and delivery partners. Status changes fire events to all registered observers — SMS, push, email channels pluggable.

Factory Pattern

Order creation centralized in placeOrder() — validates restaurant is open, calculates totals (subtotal + tax + delivery fee), sets timestamps, generates UUID.

Repository Pattern

HashMap-backed stores behind service interface. Swap to JPA/Redis with zero changes to business logic. Clean separation of data access from domain.

04 — Order Lifecycle (State Machine)

Why a state machine? Prevents invalid transitions (e.g., can't deliver an order that hasn't been picked up). Each status defines exactly which next statuses are legal. Adding a new state (e.g., PAYMENT_PENDING) is isolated. 
stateDiagram-v2
    [*] --> PLACED
    PLACED --> CONFIRMED: restaurant accepts
    PLACED --> CANCELLED: customer cancels
    CONFIRMED --> PREPARING: kitchen starts
    PREPARING --> READY_FOR_PICKUP: food ready
    READY_FOR_PICKUP --> OUT_FOR_DELIVERY: partner picks up
    OUT_FOR_DELIVERY --> DELIVERED: partner delivers
    PLACED --> CANCELLED: timeout
    CONFIRMED --> CANCELLED: restaurant rejects
    DELIVERED --> [*]
    CANCELLED --> [*]

Status Update + Completion Handling

public void updateOrderStatus(String orderId, OrderStatus status) {
    Order order = getOrder(orderId);
    order.setStatus(status);
    if (status == OrderStatus.DELIVERED) {
        order.setActualDeliveryTime(LocalDateTime.now());
    }
}

public void cancelOrder(String orderId) {
    Order order = getOrder(orderId);
    if (order.getStatus() == OrderStatus.DELIVERED) {
        throw new InvalidOperationException("Cannot cancel delivered order");
    }
    order.setStatus(OrderStatus.CANCELLED);
}

Cancellation is gated: delivered orders cannot be cancelled. In production, add time-window checks (free cancellation within 2 min of PLACED).

Restaurant Search Algorithm

public List<Restaurant> searchRestaurants(String query, Address location, double radiusKm) {
    return restaurants.values().stream()
        .filter(r -> r.isOpen())                                         // open only
        .filter(r -> query == null ||
            r.getName().toLowerCase().contains(query.toLowerCase()))     // name match
        .filter(r -> location == null ||
            r.getAddress().distanceTo(location) <= radiusKm)            // within radius
        .sorted((a, b) -> Double.compare(b.getRating(), a.getRating())) // by rating DESC
        .collect(Collectors.toList());
}

Time: O(n log n) — Optimization: geo-spatial index (R-tree, QuadTree, or Redis GEOSEARCH) to reduce candidates to O(log n).

06 — Delivery Partner Matching

Why Strategy? Matching algorithm changes independently of the service. Nearest partner today, ETA-based tomorrow — just swap the implementation. No changes to service code.

Matching Algorithm (Nearest Available)

// Algorithm: FindNearestPartner
// Input: restaurant location, delivery address
// Output: best delivery partner

1. availablePartners = getPartnersWithStatus(AVAILABLE)
2. if availablePartners is empty: return null

3. bestPartner = null, minCost = infinity

4. for each partner in availablePartners:
      // Total: Partner → Restaurant → Customer
      distToRestaurant = haversineDistance(partner.location, restaurant.location)
      distToCustomer   = haversineDistance(restaurant.location, deliveryAddress)
      totalDist = distToRestaurant + distToCustomer

      // Rating-weighted time cost
      timeCost = (totalDist / AVG_SPEED) * (2.0 - partner.rating / 5.0)

      if timeCost < minCost:
         minCost = timeCost
         bestPartner = partner

5. return bestPartner

Time: O(p) where p = available partners. Space: O(1). Higher-rated partners get a slight distance advantage via the weighting factor.

Partner Assignment

public void assignDeliveryPartner(String orderId, String partnerId) {
    Order order = getOrder(orderId);
    DeliveryPartner partner = deliveryPartners.get(partnerId);
    if (partner == null)
        throw new PartnerNotFoundException("Partner not found");
    if (!partner.isAvailable())
        throw new PartnerNotAvailableException("Partner is not available");

    order.setDeliveryPartnerId(partnerId);
    partner.setStatus(PartnerStatus.BUSY);
    partner.setCurrentOrderId(orderId);
}

Partner Availability Query

public List<DeliveryPartner> getAvailablePartners(Address location) {
    return deliveryPartners.values().stream()
        .filter(DeliveryPartner::isAvailable)
        .collect(Collectors.toList());
}

// DeliveryPartner.isAvailable()
public boolean isAvailable() { return status == PartnerStatus.AVAILABLE; }

// Called every time an item is added to the order
private void calculateAmounts() {
    this.subtotal    = items.stream().mapToDouble(OrderItem::getTotal).sum();
    this.deliveryFee = subtotal > 50 ? 0 : 5.0;
    this.tax         = subtotal * 0.08;
    this.totalAmount = subtotal + deliveryFee + tax;
}

// OrderItem.getTotal()
public double getTotal() { return price * quantity; }

Extension point: In production, add platformFee (2% of subtotal) and promoCode discount using a Strategy/Decorator approach — same pattern as ride-hailing surge pricing.

Full Breakdown Algorithm

// Production-grade total calculation
itemsTotal   = ∑(item.price × item.quantity)
taxes        = itemsTotal × TAX_RATE          // 5–8%
deliveryFee  = calculateDeliveryFee(distance) // base + per km
platformFee  = itemsTotal × PLATFORM_FEE_RATE  // ~2%
discount     = applyPromoCode(promoCode, itemsTotal)
total        = itemsTotal + taxes + deliveryFee + platformFee - discount

08 — Haversine Distance Formula

Why Haversine? Euclidean distance on lat/lon is wrong — the Earth is a sphere. Haversine gives great-circle distance in km, critical for delivery radius filtering and ETA estimation.

Implementation (Address.distanceTo)

public double distanceTo(Address other) {
    double lat1 = Math.toRadians(this.latitude);
    double lon1 = Math.toRadians(this.longitude);
    double lat2 = Math.toRadians(other.latitude);
    double lon2 = Math.toRadians(other.longitude);

    double dlat = lat2 - lat1;
    double dlon = lon2 - lon1;

    double a = Math.pow(Math.sin(dlat / 2), 2)
             + Math.cos(lat1) * Math.cos(lat2)
             * Math.pow(Math.sin(dlon / 2), 2);

    double c = 2 * Math.asin(Math.sqrt(a));

    return 6371 * c; // Earth radius in km
}

Time: O(1) — Used by: restaurant search radius filter, delivery fee calculation, partner matching distance. At scale, wrap in a geo-index (QuadTree/GeoHash) to avoid iterating all entities.

13 — Interview Follow-up Q&A

How would you handle peak hours (lunch/dinner rush)?
Pre-assign partners to hotspot areas. Apply dynamic pricing (surge delivery fees) via a SurgeDeliveryFeeStrategy decorator. Add partner incentives for busy times. Use queue management to throttle orders per restaurant. Monitor demand/supply ratio in real-time.
How would you optimize delivery routes?
Batch orders going to the same area for the same partner (multi-stop delivery). Use route optimization algorithms (TSP variants / Google Maps Directions API). Consider real-time traffic data. The DeliveryMatchingService interface stays the same — just swap the strategy implementation.
How to prevent two partners accepting the same order?
synchronized(order) — only one thread enters the assignment block at a time. The first partner call sets deliveryPartnerId; the second finds it already assigned and throws InvalidOperationException. At scale, use optimistic locking with a version field or a distributed lock (Redis SETNX).
How to handle order cancellations?
Time-based cancellation policy: free within 2 minutes of PLACED. Partial refund after CONFIRMED/PREPARING. No cancellation once READY_FOR_PICKUP or later. Compensate partners for wasted trips. Blacklist frequent cancellers. The state machine enforces which statuses allow cancellation.
How to handle payment failures?
Add a Payment entity with lifecycle: PENDING → COMPLETED / FAILED / REFUNDED. Retry with exponential backoff. After max retries, mark FAILED and hold the order. In production, integrate with a payment gateway (Stripe/Razorpay) behind a PaymentGateway interface. Use idempotency keys to prevent double-charging on retries.
How would you scale restaurant search?
Replace the O(n) stream filter with Elasticsearch geo-queries or a PostGIS-backed spatial index. Use Redis GeoSets (GEOADD/GEOSEARCH) for sub-millisecond proximity queries. Cache popular search results with TTL. The service interface stays unchanged — only the repository implementation changes.
How would you track delivery partner location in real-time?
Use WebSocket or Server-Sent Events for push-based updates (5–10 second intervals). Store latest location in Redis (low latency reads). Persist route history to a time-series DB for analytics. The updatePartnerLocation() API is already defined — just change the storage backend.
Why separate OrderItem from MenuItem?
MenuItem is the restaurant's catalog — it can be updated, marked unavailable, repriced. OrderItem captures a snapshot of the item at order time (price, quantity, customization). If the restaurant changes the menu later, existing orders are unaffected. This is the Memento/Snapshot pattern applied to pricing.
How to prevent fraud (fake orders)?
OTP verification at pickup and delivery. Payment pre-authorization before confirming order. Delivery partner photo proof of delivery. Customer verification (phone, address history). Rate-limit order creation per customer. Flag anomalous patterns (many cancellations, new accounts with large orders).
How would Restaurant.preparationTimes work at scale?
The map stores per-item prep time (default 30 min). At scale, use ML to predict prep time based on order complexity, kitchen load, time of day, and historical data. Feed this into ETA calculation: ETA = prepTime + travelTime. Update ETA dynamically as the order progresses through states.
Why RestaurantStatus has ACCEPTING_ORDERS_ONLY?
For restaurants that are technically closed for walk-ins but still accepting online orders (common during COVID or for cloud kitchens). The enum provides fine-grained control: OPEN (full service), CLOSED (no orders), TEMPORARILY_CLOSED (maintenance), ACCEPTING_ORDERS_ONLY (online only). The isOpen() method only returns true for OPEN, but could be extended.
How would you add a recommendation engine?
Add a RecommendationService interface with strategies: collaborative filtering (users who ordered X also ordered Y), content-based (cuisine preference from order history), popularity-based (trending restaurants). The service queries order history from the repository. Start simple with popularity, add ML-based personalization later.
How to handle restaurant going offline mid-order?
If order is PLACED/CONFIRMED: cancel with full refund, notify customer with alternatives. If PREPARING: continue (food already cooking). Add an OrderEventHandler that listens to restaurant status changes and triggers compensating actions. The Observer pattern makes this extensible.
