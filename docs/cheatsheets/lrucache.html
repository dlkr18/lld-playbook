LRU Cache // O(1) Everything

HashMap + Doubly Linked List for O(1) get, put, and evict. Thread-safe variant with ReadWriteLock. Strategy pattern for pluggable eviction policies.

HashMap + DLL Strategy Pattern Sentinel Nodes Decorator Pattern ReadWriteLock

01 — Requirements

Functional

  • Get(key) — retrieve value in O(1), move to MRU position, return Optional<V>
  • Put(key, value) — insert/update in O(1), move to MRU, evict LRU if at capacity
  • Capacity management — fixed at construction, auto-evict when size exceeds capacity
  • Statistics tracking — hits, misses, evictions, hit rate calculation
  • containsKey(key) — check existence without updating access order
  • clear() — remove all entries and reset statistics
  • Reject null keys and null values with IllegalArgumentException

Non-Functional

  • O(1) time for get, put, evict — no scanning, no sorting
  • O(capacity) space — HashMap + DLL nodes
  • Exact LRU — always evict true least-recently-used, not approximate
  • Thread safety — base impl is single-threaded; ConcurrentLRUCache wraps with locks
  • Extensible — pluggable eviction policies (LRU, LFU, FIFO) via Strategy
  • Generic <K, V> — any hashable key, any value type

Out of Scope

  • Distributed caching across multiple nodes
  • Persistent storage (in-memory only)
  • Time-based expiration (TTL) — discussed as extension
  • Cache warming strategies
Real-world: CPU caches, database query caches, web browser caches, CDN edge caching, Redis maxmemory-policy: allkeys-lru.

02 — Core Data Structure

Why HashMap + Doubly Linked List? HashMap gives O(1) lookup. DLL maintains access order and allows O(1) insertion/removal at any position. Together they make every cache operation O(1).

HashMap + Doubly Linked List

// The two data structures work together:
HashMap<K, CacheNode>  →  O(1) key lookup, returns node reference
DoublyLinkedList        →  O(1) order tracking (MRU at head, LRU at tail)

// Visual layout (capacity=3, items A, B, C):

  HashMap                Doubly Linked List
  ┌───────┐
  │ A → ●─┼──┐     Head ←→ [C] ←→ [B] ←→ [A] ←→ Tail
  │ B → ●─┼──┤      MRU                         LRU
  │ C → ●─┼──┘     (sentinel)             (sentinel)
  └───────┘
                     ↑ most recent          ↑ least recent
                       access                  evict this

HashMap values point directly to DLL nodes. No scanning needed — follow the pointer, unlink, relink.

Sentinel Nodes (Head/Tail)

Dummy nodes at both ends eliminate null checks and simplify edge cases.

Without Sentinels (complex)

if (head == null) {
    head = newNode;
    tail = newNode;
} else {
    newNode.next = head;
    head.prev = newNode;
    head = newNode;
}

With Sentinels (clean)

newNode.next = head.next;
newNode.prev = head;
head.next.prev = newNode;
head.next = newNode;

03 — How It Works

Trace every operation step by step.

get(key) — Cache Hit

HashMap Lookup
node = cache.get(key) — O(1) direct access
Remove from Current Position
Unlink node from its neighbors: node.prev.next = node.next; node.next.prev = node.prev
Add to Head (MRU)
Insert right after head sentinel — node is now most recently used
Return Value
statistics.recordHit(), return Optional.of(node.value) 
// Example:
Initial:  Head ←→ C ←→ B ←→ A ←→ Tail
get(A):   Find A in HashMap → unlink A → add A after Head
Result:   Head ←→ A ←→ C ←→ B ←→ Tail

put(key, value) — Eviction

Check Existing
node = cache.get(key) — if exists, update value and moveToHead
Check Capacity
If cache.size() == capacity, evict LRU: tail.prev
Evict LRU
Remove tail.prev from DLL and HashMap. statistics.recordEviction()
Insert New
Create node, add to HashMap, add to head of DLL 
// Example (capacity=3):
Initial:  {A:1, B:2, C:3}   Head ←→ C ←→ B ←→ A ←→ Tail

put(D, 4):
  1. Cache full (size=3, capacity=3)
  2. Evict LRU: A (tail.prev)
  3. Remove A from HashMap and DLL
  4. Add D to head

Result:   {D:4, C:3, B:2}   Head ←→ D ←→ C ←→ B ←→ Tail

CacheNode — The DLL Node

class CacheNode<K, V> {
    final K key;         // immutable — needed for HashMap removal on eviction
    V value;              // mutable — updated on put(existingKey, newValue)
    CacheNode<K, V> prev;
    CacheNode<K, V> next;

    CacheNode(K key, V value) { this.key = key; this.value = value; }
    CacheNode() { this.key = null; this.value = null; } // sentinel constructor
}

Why store key in node? When evicting tail.prev, we need the key to call cache.remove(lruNode.key) on the HashMap.

05 — Design Patterns

Three patterns, each earning its place.

Strategy Pattern

EvictionPolicy determines which item to evict. Different policies (LRU, LFU, FIFO) implement the same interface. Injected at construction, swappable at runtime.

Decorator Pattern

ConcurrentLRUCache wraps any LRUCache to add thread-safety. Composable: you can wrap any implementation without modifying it.

Template Method

Cache<K,V> interface defines the contract. LRUCacheImpl provides the HashMap+DLL skeleton. Subclasses or decorators extend behavior.

Sentinel Pattern

Head/Tail dummy nodes eliminate null checks at boundaries. No special cases for empty cache, single element, or full capacity.

06 — Key Operations (Java Code)

Pointer manipulation is the heart of LRU Cache. Every operation is O(1) because we never scan the list.

get(key) — Retrieve & Promote

public Optional<V> get(K key) {
    validateKey(key);

    CacheNode<K, V> node = cache.get(key);   // O(1) HashMap lookup
    if (node == null) {
        statistics.recordMiss();
        return Optional.empty();
    }

    moveToHead(node);                          // O(1) promote to MRU
    statistics.recordHit();
    return Optional.of(node.value);
}

put(key, value) — Insert/Update & Evict

public void put(K key, V value) {
    validateKey(key);
    validateValue(value);

    CacheNode<K, V> node = cache.get(key);

    if (node != null) {
        // Update existing: change value, promote to MRU
        node.value = value;
        moveToHead(node);
    } else {
        // New key: create node, add to cache and DLL head
        CacheNode<K, V> newNode = new CacheNode<>(key, value);
        cache.put(key, newNode);
        addToHead(newNode);

        // Evict LRU if over capacity
        if (cache.size() > capacity) {
            CacheNode<K, V> lru = removeTail();  // O(1)
            cache.remove(lru.key);                // O(1)
            statistics.recordEviction();
        }
    }
}

Pointer Manipulation Helpers

// Remove node from its current position in DLL
private void removeNode(CacheNode<K, V> node) {
    node.prev.next = node.next;    // prev skips over node
    node.next.prev = node.prev;    // next skips back over node
}

// Insert node right after head sentinel (MRU position)
private void addToHead(CacheNode<K, V> node) {
    node.next = head.next;         // node points to old first
    node.prev = head;              // node points back to sentinel
    head.next.prev = node;         // old first points back to node
    head.next = node;              // sentinel points to node
}

// Promote: unlink + relink at head
private void moveToHead(CacheNode<K, V> node) {
    removeNode(node);
    addToHead(node);
}

// Evict: remove LRU node (before tail sentinel)
private CacheNode<K, V> removeTail() {
    CacheNode<K, V> lru = tail.prev;  // actual LRU (not sentinel)
    removeNode(lru);
    return lru;                        // caller removes from HashMap
}

Constructor — Sentinel Initialization

public LRUCacheImpl(int capacity) {
    if (capacity <= 0)
        throw new IllegalArgumentException("Capacity must be positive");

    this.capacity = capacity;
    this.cache = new HashMap<>();
    this.statistics = new CacheStatistics();

    // Sentinel nodes: head ←→ tail (empty list)
    this.head = new CacheNode<>();
    this.tail = new CacheNode<>();
    head.next = tail;
    tail.prev = head;
}

Alternative: Encapsulated DoublyLinkedList

private static class DoublyLinkedList<K, V> {
    private Node<K, V> head, tail;

    DoublyLinkedList() {
        head = new Node<>(null, null);
        tail = new Node<>(null, null);
        head.next = tail;
        tail.prev = head;
    }

    void addToHead(Node node) { /* same pointer ops */ }
    void remove(Node node)    { /* same pointer ops */ }
    void moveToHead(Node node) { remove(node); addToHead(node); }
    Node removeTail()         { /* remove tail.prev */ }
    void clear()              { head.next = tail; tail.prev = head; }
}

Both approaches are valid. The encapsulated DLL is cleaner; inline is fewer allocations.

07 — Thread Safety

ConcurrentLRUCache (Decorator Pattern)

public class ConcurrentLRUCache<K, V> implements LRUCache<K, V> {
    private final LRUCache<K, V> delegate;
    private final ReadWriteLock lock = new ReentrantReadWriteLock();

    public Optional<V> get(K key) {
        lock.writeLock().lock();     // WRITE lock! get() modifies order
        try {
            return delegate.get(key);
        } finally {
            lock.writeLock().unlock();
        }
    }

    public void put(K key, V value) {
        lock.writeLock().lock();
        try {
            delegate.put(key, value);
        } finally {
            lock.writeLock().unlock();
        }
    }

    public boolean containsKey(K key) {
        lock.readLock().lock();      // Read lock OK: doesn't modify order
        try {
            return delegate.containsKey(key);
        } finally {
            lock.readLock().unlock();
        }
    }
}
MechanismWhereWhy
ReentrantReadWriteLock ConcurrentLRUCache Separates read-only ops (containsKey, size) from mutating ops
writeLock for get() get() method get() modifies access order (moves node to head) — not a pure read!
AtomicLong CacheStatistics Lock-free counters for hits/misses/evictions
Gotcha: get() needs a write lock because it modifies the DLL access order. Using a read lock would allow concurrent get() calls to corrupt the linked list. For read-only checks, use containsKey() which uses a read lock.

Segmented Locking (Scale-up)

// Reduce contention by partitioning into N independent caches
class SegmentedLRUCache<K, V> {
    private final int segments = 16;
    private final LRUCacheImpl<K, V>[] caches;

    private int getSegment(K key) {
        return Math.abs(key.hashCode() % segments);
    }

    public Optional<V> get(K key) {
        return caches[getSegment(key)].get(key);
    }
}

Reduces lock contention by 16x. Each segment is an independent LRU cache with its own lock. Trade-off: not globally LRU, but per-segment LRU.

Workload Comparison

Workload: [A, B, C, D, A, B, E]    Capacity: 3

LRU evicts: D  (least recently used)
LFU evicts: C or D  (lowest frequency = 1)
FIFO evicts: A  (oldest insertion)

LRU is the best general-purpose policy. LFU is better for skewed access patterns. FIFO is simplest but worst hit rate.

10 — Interview Q&A

Why HashMap + DLL instead of just a HashMap with timestamps?
With timestamps, eviction requires O(n) scan to find the minimum. The DLL maintains access order automatically — LRU is always tail.prev, giving O(1) eviction. The HashMap provides O(1) key lookup. Together: O(1) for everything.
Why not just use Java's LinkedHashMap?
LinkedHashMap with accessOrder=true and overridden removeEldestEntry() works perfectly in production (~10 lines). In an interview, the custom implementation demonstrates understanding of DLL pointer manipulation, sentinel nodes, and data structure design. Use LinkedHashMap in production, custom impl in interviews.
How would you add TTL (time-based expiration)?
Add long expiryTime to CacheNode. On get(), check if System.currentTimeMillis() > node.expiryTime — if expired, remove and return empty (lazy expiry). For proactive cleanup, run a ScheduledExecutorService that scans from tail (oldest access) and removes expired entries. Dual eviction: LRU by access order + TTL by time.
Why does get() need a write lock in the concurrent version?
get() is not a pure read — it modifies the DLL by calling moveToHead(). If two threads concurrently call get() with read locks, both would modify the DLL pointers simultaneously, causing corruption. Solution: use write lock for get(), or use containsKey() (which doesn't modify order) for read-only checks with a read lock.
How would you implement a distributed LRU cache?
Three approaches: (A) Consistent Hashing + Local LRU — hash key to a server, each server has independent LRU. Same key always routes to same server. (B) Centralized Metadata — Redis stores global access order, accurate but higher latency. (C) Approximate LRU (Redis approach) — sample 5 random keys, evict the one with oldest access time. Not exact LRU but O(1) and good enough for most workloads.
How do you handle cache misses and cache stampede?
Cache miss: Return Optional.empty() and let the caller fetch from the backing store. Cache stampede: When a popular key expires, many threads simultaneously try to recompute. Solutions: (1) Mutex/Singleflight — only one thread fetches, others wait. (2) Early refresh — probabilistically refresh before expiry. (3) Stale-while-revalidate — serve stale value while background thread refreshes.
Write-Through vs Write-Back caching?
Write-Through: Write to DB first, then update cache. Data consistency guaranteed but higher write latency. Write-Back: Write to cache immediately, flush to DB asynchronously. Lower latency but risk of data loss on crash. Use write-through for financial/critical data, write-back for logs/analytics/high-write workloads.
What are sentinel nodes and why use them?
Sentinel (dummy) nodes at head and tail of the DLL that never hold real data. They eliminate all null checks and edge cases: empty list, single element, adding first node, removing last node — all use the same pointer manipulation code. Without sentinels, every operation needs if (head == null) or if (node == tail) checks. Sentinels simplify code and reduce bugs.
How would you handle cache warming?
On startup, preload popular items: (1) Query analytics/logs for top-K accessed keys. (2) Bulk-load them into the cache with put(). (3) For a new cache, use shadow mode — run the cache alongside the old system, log access patterns, then switch over once warmed. This avoids cold-start latency spikes.
Why store the key inside CacheNode?
When evicting from the DLL (removeTail()), we get a node reference but need to also remove the entry from the HashMap. Without the key stored in the node, we'd need to scan the HashMap to find which key maps to this node — O(n). Storing the key in the node makes it O(1): cache.remove(lruNode.key).
LRU vs LFU — when to use which?
LRU is best for temporal locality (recently accessed items likely accessed again). Good for most general workloads. LFU is best for frequency-based patterns (some items are inherently more popular). Better for CDN caching, API rate limiting. LFU downside: "frequency pollution" — once-popular items stay cached forever. Solution: LFU with aging (decay frequency over time).
How would you test this implementation?
Unit tests: Basic get/put, update existing key, eviction order, null key rejection, capacity=1 edge case. Invariant tests: After every operation, verify HashMap.size == DLL.size, all HashMap entries exist in DLL, head.next is MRU, tail.prev is LRU. Concurrency tests: Parallel get/put with ConcurrentLRUCache, verify no corruption. Property-based: Random operations maintain invariants. The validateInvariants() method checks all these.
