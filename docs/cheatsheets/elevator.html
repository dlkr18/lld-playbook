01 — Requirements

Functional

  • Elevator Control — Move up/down to target floor, stop at requested floors, open/close doors, emergency stop
  • External requests — Floor button press with direction (UP/DOWN)
  • Internal requests — Elevator panel button press (destination floor)
  • Queue management for pending requests, priority handling (emergency, VIP)
  • Dispatch algorithms — FCFS, SCAN, LOOK, SSTF (pluggable via Strategy)
  • State tracking — Current floor, direction (UP/DOWN/IDLE), door state (OPEN/CLOSED)
  • Capacity management — Max passengers / weight enforcement
  • Elevator status — IDLE, MOVING_UP, MOVING_DOWN, MAINTENANCE
  • Multi-elevator coordination — Assign requests to optimal elevator, load balancing
  • Graceful failure handling — Elevator failures reroute pending requests

Non-Functional

  • Performance — Average wait time < 30 seconds
  • Safety — Door sensors, weight limits, emergency stops
  • Scalability — Support 10+ elevators, 100+ floors
  • Reliability — 99.9% uptime, graceful degradation
  • Concurrency — Handle multiple simultaneous requests safely

03 — Design Patterns

The interview differentiator — three core patterns, each earning its place.

State Pattern

Elevator states: IDLE, MOVING_UP, MOVING_DOWN, MAINTENANCE. Each state determines legal transitions. Direction changes happen when destinations are exhausted or bounds are reached — no scattered if/else.

Strategy Pattern

ElevatorScheduler interface: pluggable dispatch algorithms. Current impl uses SCAN. Swap to FCFS, LOOK, or SSTF by providing a new selectElevator() implementation.

Observer Pattern

Floor display boards and waiting passengers are notified when an elevator arrives. The controller broadcasts arrival events. Decouples elevator movement from notification logic.

Step Simulation

Discrete step()-based simulation: each call moves all elevators one floor, processes pending requests, and reassigns unserviced requests. Deterministic and testable.

04 — Elevator State Machine

Why State? Eliminates scattered if/else for direction changes. Each status defines exactly what happens next. Adding DOOR_OPEN or EMERGENCY is a new enum value with localized logic. Open/Closed Principle.

State Transitions in Elevator.moveToNextFloor()

public void moveToNextFloor() {
    if (destinationFloors.isEmpty()) {
        status = ElevatorStatus.IDLE;
        direction = Direction.IDLE;
        return;
    }

    // Determine direction from IDLE
    if (direction == Direction.IDLE) {
        int nextFloor = getNextDestination();
        direction = nextFloor > currentFloor ? Direction.UP : Direction.DOWN;
    }

    // Move one floor
    if (direction == Direction.UP) {
        currentFloor++;
        status = ElevatorStatus.MOVING_UP;
    } else {
        currentFloor--;
        status = ElevatorStatus.MOVING_DOWN;
    }

    // Arrived at destination? Remove it.
    if (destinationFloors.contains(currentFloor)) {
        destinationFloors.remove(currentFloor);
    }

    // Boundary reversal or idle
    if (destinationFloors.isEmpty()) {
        direction = Direction.IDLE;
        status = ElevatorStatus.IDLE;
    } else if (direction == Direction.UP && currentFloor >= maxFloor) {
        direction = Direction.DOWN;
    } else if (direction == Direction.DOWN && currentFloor <= minFloor) {
        direction = Direction.UP;
    }
}

06 — Dispatch Strategy

Core question: When a floor button is pressed, which elevator should respond? The findBestElevator() method implements a two-tier priority system.

Two-Tier Dispatch Logic

PriorityCriteriaWhy
Tier 1: Moving & Same Direction Elevator is moving in the same direction as the request AND is approaching (not past) the request floor Elevator can service the floor en route — zero detour cost
Tier 2: Idle Elevator has no destinations — pick the nearest idle elevator Fallback when no elevator is already heading that way

If neither tier matches, the request is queued in pendingRequests and re-evaluated on every step() call.

findBestElevator() Implementation

private Elevator findBestElevator(int floor, Direction direction) {
    Elevator bestMoving = null;
    int bestMovingDistance = Integer.MAX_VALUE;
    Elevator bestIdle = null;
    int bestIdleDistance = Integer.MAX_VALUE;

    for (Elevator elevator : elevators.values()) {
        int distance = elevator.distanceToFloor(floor);

        if (elevator.isIdle()) {
            if (distance < bestIdleDistance) {
                bestIdle = elevator;
                bestIdleDistance = distance;
            }
        } else if (elevator.getDirection() == direction) {
            boolean movingToward =
                (direction == Direction.UP && elevator.getCurrentFloor() <= floor) ||
                (direction == Direction.DOWN && elevator.getCurrentFloor() >= floor);
            if (movingToward && distance < bestMovingDistance) {
                bestMoving = elevator;
                bestMovingDistance = distance;
            }
        }
    }
    return bestMoving != null ? bestMoving : bestIdle;
}

07 — Key Methods

Elevator — Core Model

public class Elevator {
    private final int id;
    private volatile int currentFloor;
    private volatile Direction direction;
    private volatile ElevatorStatus status;
    private final Set<Integer> destinationFloors; // ConcurrentSkipListSet
    private final int minFloor;
    private final int maxFloor;

    public void addDestination(int floor) {
        if (floor >= minFloor && floor <= maxFloor)
            destinationFloors.add(floor);
    }

    public void moveToNextFloor() { /* see State Machine section */ }

    public boolean isIdle() {
        return status == ElevatorStatus.IDLE;
    }

    public int distanceToFloor(int floor) {
        return Math.abs(currentFloor - floor);
    }
}

    private final Direction direction;
    private final long timestamp;

    public Request(int floor, Direction direction) {
        this.floor = floor;
        this.direction = direction;
        this.timestamp = System.currentTimeMillis();

}

step() — Simulation Tick

public void step() {
    // 1. Move all elevators one floor toward their destinations
    for (Elevator elevator : elevators.values()) {
        elevator.moveToNextFloor();
    }

    // 2. Re-evaluate queued requests
    while (!pendingRequests.isEmpty()) {
        Request request = pendingRequests.peek();
        Elevator best = findBestElevator(request.getFloor(), request.getDirection());
        if (best != null) {
            pendingRequests.poll();
            best.addDestination(request.getFloor());
        } else break;
    }
}

ElevatorMetrics — Performance Tracking

public class ElevatorMetrics {
    private int totalTrips;
    private long totalWaitTime;
    private int floorsTraversed;

    public void recordTrip(long waitTime, int floors) {
        totalTrips++;
        totalWaitTime += waitTime;
        floorsTraversed += floors;
    }

    public double getAverageWaitTime() {
        return totalTrips == 0 ? 0 : (double) totalWaitTime / totalTrips;
    }
}

08 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap elevators store in controller Safe concurrent read/write of elevator map
ConcurrentSkipListSet Elevator.destinationFloors Thread-safe sorted set — multiple threads can add destinations simultaneously
volatile Elevator.currentFloor, direction, status Cross-thread visibility without full lock
Queue<Request> pendingRequests FIFO ordering for unassigned requests (swap to ConcurrentLinkedQueue for production)
Immutable Request Request model All fields final — thread-safe by design
"ConcurrentSkipListSet is the key choice — it gives us a sorted, thread-safe set for destinations. Multiple passengers can press buttons in the same elevator simultaneously without races. The sorted nature ensures SCAN ordering is free."

Why These Choices?

StructureWhy Not Alternatives
ConcurrentSkipListSet TreeSet is not thread-safe. PriorityQueue allows duplicates. SkipList gives O(log n) sorted insert + thread safety.
ConcurrentHashMap HashMap is not thread-safe. synchronized Map has coarse locking. CHM gives segment-level locking.
Queue<Request> FIFO ordering preserves fairness for pending requests. In production, use PriorityQueue with wait-time comparator to prevent starvation.

12 — Interview Follow-up Q&A

How do you handle multiple elevators efficiently?
The findBestElevator() method uses a two-tier dispatch: first prefer an elevator already moving in the same direction toward the request floor (zero detour), then fall back to the nearest idle elevator. This minimizes both wait time and unnecessary travel. At scale, you'd add a WeightedScoreScheduler that factors in current load, remaining stops, and estimated arrival time.
How would you implement a VIP/express elevator?
Add a priority field to Request (NORMAL, VIP, EMERGENCY). Dedicated elevators can be tagged as VIP-only. The scheduler filters: VIP requests only go to VIP-tagged elevators. Alternatively, VIP requests jump to the front of the destination set. The ElevatorScheduler strategy interface handles this cleanly — new class, no existing code changes.
How do you handle overweight / over-capacity?
Add a currentWeight field to Elevator with a maxCapacity. Before doors close, check weight sensor. If overweight: doors stay open, buzzer sounds, last passenger must exit. The elevator skips picking up new external requests until below capacity — it only services existing internal destinations. The dispatch algorithm excludes full elevators from consideration.
What happens during a power failure?
All elevators transition to MAINTENANCE status. Emergency battery moves each elevator to the nearest floor. Doors open for evacuation. On power restore: elevators return to IDLE at their current floor, pending requests from before the failure are re-queued, and normal dispatch resumes. Each elevator's destination set is cleared and rebuilt from the pending queue.
How would you optimize for peak hours (morning rush)?
During peak hours, most requests go from ground floor UP. Deploy a PeakHourScheduler strategy that parks idle elevators at the ground floor instead of wherever they last stopped. In the evening (reverse rush), park at upper floors. The ElevatorScheduler interface makes this a strategy swap. Also: group passengers going to similar floors into the same elevator ("zoning" — evens vs odds, or floor ranges).
How would you add floor display boards (Observer)?
Create a FloorDisplay that implements an ElevatorObserver interface with onArrival(elevatorId, floor, direction). Each floor registers its display as an observer. When an elevator arrives, the controller notifies all observers for that floor. Decouples display logic from elevator movement — you can add sound announcements, mobile push, etc. without touching the controller.
Why SCAN over FCFS?
FCFS causes excessive back-and-forth — imagine requests at floors 2, 10, 3: the elevator goes 2→10→3 (18 floors traveled). SCAN would do 2→3→10 (8 floors). SCAN guarantees every floor is visited at most once per sweep, giving bounded and fair wait times. FCFS only makes sense for a single-elevator, low-traffic building.
How do you prevent starvation in SSTF?
SSTF always picks the nearest request, so distant floors can wait indefinitely. Mitigation: add an age field to requests that increments each step. When age exceeds a threshold, boost the request's priority. This is essentially aging from OS scheduling theory. Alternatively, use LOOK (which has no starvation) and only switch to SSTF for low-traffic periods.
How would you handle elevator maintenance scheduling?
The ElevatorStatus.MAINTENANCE state already exists. A maintenance scheduler (cron job) transitions an elevator to MAINTENANCE after completing its current destinations. The dispatcher excludes MAINTENANCE elevators. For N elevators, schedule maintenance in rotation so at most one is down at a time. Emergency maintenance immediately transitions — pending requests are redistributed to other elevators.
How does the step-based simulation work?
Each step() call is one discrete time unit. All elevators move one floor toward their next destination. After movement, pending requests are re-evaluated — if an elevator has become idle or is now moving in a useful direction, queued requests get assigned. This makes the system deterministic and easy to test. In production, replace with a real-time event loop or timer-based scheduling.
How would you scale to 100+ floors and 20+ elevators?
Zoning: Divide floors into zones (1-25, 26-50, 51-75, 76-100), each served by dedicated elevators. Express elevators: Skip floors, only stop at zone entry points. Sky lobbies: Transfer floors where passengers switch between local and express elevators. Data structure-wise, ConcurrentSkipListSet scales well — O(log n) operations remain fast for 100 destinations.
