01 — Requirements

Functional

  • Support multiple log levels: TRACE < DEBUG < INFO < WARN < ERROR < FATAL
  • Filter log messages based on minimum level threshold
  • Route log output to multiple appenders (console, file, rolling file, async)
  • Pluggable formatters — plain text, JSON, pattern-based
  • Per-class / per-package logger instances with independent configs
  • Thread-local MDC (Mapped Diagnostic Context) for request tracing
  • Structured logging — key-value pairs alongside messages
  • Graceful shutdown with queue flush for async appenders

Non-Functional

  • Thread-safe — concurrent logging from 1000+ threads without data loss
  • Low latency — <1μs for level check, async write decouples caller from I/O
  • Zero allocation on filtered-out messages (level check before formatting)
  • Extensible — new appender/formatter = new class, no existing code modified
  • Minimal footprint — no external dependencies for core
  • Reliable — no log loss on normal shutdown (flush before exit)

03 — Design Patterns

Five patterns, each solving a distinct logging concern.

Singleton (LogManager)

One LogManager per JVM. Double-checked locking with volatile instance. Factory method getLogger(Class) returns cached or new Logger per class name.

Chain of Responsibility

LogFilter chain: LevelFilter → PackageFilter → RateFilter. Each filter decides pass/block independently. Short-circuits on first reject.

Strategy (Formatters + Appenders)

LogFormatter and Appender are pluggable strategies. Swap JSON for pattern-based or XML — zero changes to Logger. Injected at construction.

Observer (Log Listeners)

External systems subscribe to log events. Metrics collectors, alerting engines, and audit trails receive onLog(LogMessage) callbacks without coupling to the logger.

Builder (LoggerConfig)

Fluent API: LoggerConfig.builder().level(INFO).addAppender(console).formatter(json).build(). Immutable config after build, validated at construction.

LogLevel Enum

public enum LogLevel {
    DEBUG(1), INFO(2), WARN(3), ERROR(4), FATAL(5);

    private final int value;
    LogLevel(int value) { this.value = value; }
    public int getValue() { return value; }
}

Level Filtering Logic

// Fast integer comparison — no string ops, O(1)
public boolean shouldLog(LogMessage message) {
    return message.getLevel().ordinal() >= minLevel.ordinal();
}

// In LoggerImpl — level check BEFORE allocation
public void log(LogLevel level, String message) {
    if (level.getValue() >= minLevel.getValue()) {
        queue.offer(new LogMessage(level, message, className));
    }
}

Zero-cost filtering: if DEBUG is disabled, no LogMessage object is ever created.

AsyncAppender — The Implementation We Ship

public class LoggerImpl implements Logger {
    private final BlockingQueue<LogMessage> queue = new LinkedBlockingQueue<>();
    private final ExecutorService executor = Executors.newSingleThreadExecutor();

    public LoggerImpl(String className, LogLevel minLevel) {
        this.className = className;
        this.minLevel = minLevel;
        executor.submit(this::processLogs); // background consumer
    }

    private void processLogs() {
        while (true) {
            try {
                LogMessage msg = queue.take(); // blocks until available
                System.out.println(msg);
            } catch (InterruptedException e) {
                return; // clean shutdown
            }
        }
    }

    public void shutdown() {
        executor.shutdown(); // graceful: processes remaining queue
    }
}

06 — Key Algorithms

Algorithm 1: Level Filtering

Time: O(1) — Space: O(1)

Integer comparison on enum ordinal. Check happens before any object allocation or string formatting, so disabled levels have zero cost.

// Hot path: called millions of times/sec
level.getValue() >= minLevel.getValue()  // single int compare

Algorithm 2: Format String Parsing

Time: O(n) where n = pattern length — Space: O(n)

// Pattern: "{timestamp} [{level}] [{class}] {message}"
// Scan for {} tokens, replace with message fields
public String format(LogMessage msg) {
    return String.format(
        "{\"level\":\"%s\",\"message\":\"%s\",\"timestamp\":\"%s\"}",
        msg.getLevel(), msg.getMessage(), msg.getTimestamp());
}

Algorithm 3: Async Queue with Flush

Producer: O(1) offer — Consumer: O(1) take (blocking)

07 — Thread Safety

MechanismWhereWhy
LinkedBlockingQueue LoggerImpl.queue Thread-safe producer-consumer. Multiple threads offer(), single consumer take(). Lock-free for concurrent puts.
SingleThreadExecutor LoggerImpl.executor Exactly one consumer thread per logger — serializes writes, prevents interleaving.
Immutable LogMessage All fields final Safely shared across threads without synchronization. Published via queue's happens-before guarantee.
ThreadLocal MDC / LogContext Per-thread context (requestId, userId). No contention. Child threads inherit via InheritableThreadLocal.
volatile LogManager.instance Double-checked locking for Singleton. Ensures safe publication of the LogManager instance.
ConcurrentHashMap Logger cache (name → Logger) Lock-free reads for getLogger(). computeIfAbsent() for atomic creation.
"The caller thread does zero I/O. It performs a single int compare + queue.offer(). All formatting and disk/network writes happen on the dedicated consumer thread. This means logging never blocks your business logic."

MDC — Thread-Local Context

public class LogContext {
    private final Map<String, String> context = new HashMap<>();

    public void put(String key, String value) { context.put(key, value); }
    public String get(String key) { return context.get(key); }
    public Map<String,String> getAll() { return new HashMap<>(context); }
    public void clear() { context.clear(); }
}

// Usage: wrap in ThreadLocal for per-request context
private static final ThreadLocal<LogContext> MDC =
    ThreadLocal.withInitial(LogContext::new);

MDC.get().put("requestId", "req-4a2b");
MDC.get().put("userId", "user-123");
logger.info("Processing payment");
// Output: [req-4a2b] [user-123] Processing payment
MDC.get().clear(); // MUST clear in finally block

11 — Interview Follow-up Q&A

How does log rotation work in a RollingFileAppender?
Two policies: size-based (rotate when file exceeds N MB) and time-based (rotate daily/hourly). On rotation: close current writer, rename app.log → app.log.1 (shifting existing archives), open new file. Add a maxHistory to delete archives older than N days. In production, compress rotated files with gzip. The RollingPolicy is a Strategy — swap size for time without changing the appender.
Async vs. Sync appenders — when to use which?
Async (our default): caller never blocks on I/O, best for high-throughput services. Trade-off: risk of log loss on JVM crash (messages still in queue). Sync: every log call blocks until written to disk/network. Use for audit trails where zero loss is mandatory. Hybrid approach: async for INFO/DEBUG, sync for ERROR/FATAL. Our implementation uses LinkedBlockingQueue + single consumer thread.
What is MDC (Mapped Diagnostic Context) and how is it thread-safe?
MDC stores per-thread key-value pairs (requestId, userId, sessionId) using ThreadLocal<LogContext>. Each thread has its own Map — zero contention. On log output, the formatter reads MDC.get() and embeds context into the log line. Critical: always clear MDC in a finally block, especially with thread pools, to avoid context leaking between requests. For async frameworks (reactive), use Context propagation instead of ThreadLocal.
How would you implement structured logging?
Two approaches: (1) Marker-based — pass key-value pairs as method args: logger.info("order.created", Map.of("orderId","ORD-42","amount","99.99")). LogMessage stores a Map<String,Object> fields. (2) Format-based — use SLF4J-style logger.info("Order {} created for ${}", orderId, amount) with parameterized messages. JsonFormatter outputs all fields as JSON. Structured logging enables machine-parseable logs for ELK/Splunk queries.
What is the performance impact of logging?
Level check: ~1ns (integer compare). Filtered-out message: 0 cost (no object allocation). Logged message: ~200ns for LogMessage creation + queue.offer(). Formatting: ~500ns-2μs (string concatenation on consumer thread). Disk write: ~5-50μs (buffered). The key optimization is that the caller only pays the level-check + enqueue cost. All expensive operations happen on the consumer thread asynchronously.
How would you handle log aggregation in a distributed system?
Add a NetworkAppender that sends log events to a central service. Common stacks: ELK (Elasticsearch + Logstash + Kibana), Loki + Grafana, or Splunk. The appender serializes LogMessage to JSON, sends via TCP/UDP/HTTP. Use async with a bounded queue and backpressure (drop oldest on full). Add hostname, serviceId, and traceId to every log entry for correlation. Our Strategy pattern makes this trivial — just add a new Appender implementation.
How would you implement log sampling?
Add a SamplingFilter implementing LogFilter: accept every Nth message or use probability-based sampling. Per-level sampling: 100% for ERROR/FATAL, 10% for INFO, 1% for DEBUG. Use an AtomicLong counter for thread-safe counting. This is critical for high-traffic systems where logging 100% of DEBUG messages would overwhelm storage. The Chain of Responsibility pattern makes this composable with level filtering.
What happens if the async queue fills up?
Three strategies: (1) Block (backpressure) — queue.put() blocks the caller until space is available. Safe but defeats async purpose. (2) Drop — queue.offer() returns false if full, caller discards message. Best for high-throughput where occasional loss is acceptable. (3) Drop oldest — evict head of queue to make room. Our implementation uses unbounded LinkedBlockingQueue (option 2 with offer). In production, use a bounded queue with a configurable overflow policy.
How do you ensure logs are not lost on shutdown?
Register a JVM shutdown hook: Runtime.addShutdownHook(() -> logger.shutdown()). The shutdown() method calls executor.shutdown() which stops accepting new tasks but processes remaining queue. Follow with executor.awaitTermination(timeout). For critical systems, add a flush() method that drains the queue synchronously. FileAppenders must call writer.flush() before close.
Why Singleton for LogManager but not for Logger?
LogManager is the factory — one per JVM, manages the logger cache and global config. Singleton is correct because multiple LogManagers would mean duplicate caches and inconsistent configuration. Logger instances are per-class/per-package — you need different loggers with different levels (e.g., com.db=DEBUG, com.web=INFO). LogManager.getLogger() uses computeIfAbsent on a ConcurrentHashMap for thread-safe lazy creation.
How would you add JSON structured output for ELK?
Already supported via JsonFormatter implementing LogFormatter (Strategy pattern). Output: {"level":"INFO","message":"Order created","timestamp":"2025-12-26T10:30:15","class":"PaymentService","requestId":"req-4a2b"}. MDC fields are merged into the JSON. For ELK, the NetworkAppender sends JSON directly to Logstash — no regex parsing needed. Swap formatters at runtime without restarting.
How does this compare to SLF4J + Logback in production?
Same core architecture. SLF4J = our Logger interface (facade). Logback = our implementation (appenders, filters, formatters). Key additions in Logback: (1) Parameterized messages with {} placeholders (avoids string concat when filtered out), (2) XML/Groovy config (our Builder is the programmatic equivalent), (3) Turbo filters (our Chain of Responsibility), (4) Marker API for semantic tagging. Our design demonstrates understanding of the same patterns at interview scale.
