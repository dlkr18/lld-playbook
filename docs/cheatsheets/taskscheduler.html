01 — Requirements

Functional

  • Schedule one-time tasks with specific execution timestamps
  • Schedule recurring tasks with configurable interval (every N seconds)
  • 4 priority levels: CRITICAL > HIGH > MEDIUM > LOW
  • Execute higher priority tasks first when scheduled for same time
  • Allow runtime priority updates without re-scheduling
  • Cancel scheduled tasks before execution
  • Automatically reschedule recurring tasks after completion
  • Query tasks by time range or list all scheduled tasks
  • Auto-generate unique task IDs (TASK-1, TASK-2, ...)

Non-Functional

  • O(log n) task insertion and removal via min-heap
  • O(1) task lookup by ID via ConcurrentHashMap
  • Thread-safe — concurrent scheduling and execution without races
  • Configurable thread pool — 4 threads default, tunable
  • Graceful shutdown — drain executor, await termination with timeout
  • 10,000+ concurrent scheduled tasks, 1000+ tasks/sec throughput
  • Clean separation: API / Implementation / Models / Exceptions

03 — Design Patterns

Four patterns working together to make the scheduler extensible and production-grade.

Dual Storage (Map + Heap)

ConcurrentHashMap for O(1) lookup by ID. PriorityQueue for O(log n) next-due-task retrieval. Both kept in sync under synchronized(taskQueue).

Interface Segregation

TaskSchedulerService defines the contract. PriorityTaskScheduler implements it. Swap to distributed scheduler without changing callers.

Thread Pool (Executor)

ScheduledExecutorService with fixed pool manages task execution. Built-in recurring scheduling via scheduleAtFixedRate for the polling loop.

Comparator Strategy

Custom Comparator<ScheduledTask> sorts by time first, then by priority. Multi-criteria ordering without modifying the entity.

04 — Task Lifecycle

Why a lifecycle? Tasks transition through well-defined states. Each transition has pre-conditions (e.g., can't cancel a completed task). The status enum makes every task's state explicit and queryable.

05 — Scheduling Strategies

Three scheduling modes — one-time for fire-and-forget, recurring for periodic jobs, and the Schedule value object for richer configuration.

One-Time Tasks

Scheduled for a specific LocalDateTime. Removed from storage after execution.

ScheduledTask task = new ScheduledTask(
    "TASK-HIGH",
    () -> System.out.println("Executed!"),
    now.plusSeconds(2)
);
task.setPriority(Priority.HIGH);
String id = scheduler.scheduleTask(task, now.plusSeconds(2));

Recurring Tasks

Delegates to scheduleTask(), then sets recurring=true and intervalSeconds. After each execution, the task is rescheduled at now + interval.

String id = scheduler.scheduleRecurringTask(task, startTime, 3);
// Internally:
//   scheduleTask(task, startTime)  ← reuses one-time logic
//   task.setRecurring(true)
//   task.setIntervalSeconds(3)

Schedule Value Object

Encapsulates scheduling configuration with richer semantics — type, interval, start/end windows, and next-execution-time computation.

class Schedule {
    private final ScheduleType type;      // ONE_TIME or RECURRING
    private final long intervalSeconds;
    private final LocalDateTime startTime;
    private final LocalDateTime endTime;   // optional expiry

    public boolean isRecurring() {
        return type == ScheduleType.RECURRING;
    }

    public LocalDateTime getNextExecutionTime(LocalDateTime current) {
        if (!isRecurring()) return startTime;
        return current.plusSeconds(intervalSeconds);
    }
}

"In production, extend with cron expressions, time-zone awareness, and end-time-based auto-cancellation."

06 — Priority Queue & Comparator

Why PriorityQueue? The scheduler must always know the next task due. A min-heap gives O(log n) insert and O(1) peek. The custom comparator ensures time-first, then priority-descending ordering.

Comparator: Time first, Priority second

this.taskQueue = new PriorityQueue<>((a, b) -> {
    // Primary: earlier time comes first
    int timeCompare = a.getScheduledTime().compareTo(b.getScheduledTime());
    if (timeCompare != 0) return timeCompare;

    // Secondary: higher priority (larger ordinal) comes first
    return a.getPriority().compareTo(b.getPriority());
});

07 — Execution Engine

Design: A ScheduledExecutorService polls executeDueTasks() every 1 second. Due tasks are pulled from the queue and submitted to the thread pool for concurrent execution.

Scheduler Start / Stop

public void start() {
    if (running) return;
    running = true;
    executor.scheduleAtFixedRate(
        this::executeDueTasks,
        0,             // initial delay
        1,             // period
        TimeUnit.SECONDS
    );
}

public void stop() {
    running = false;
    executor.shutdown();
    try {
        if (!executor.awaitTermination(5, TimeUnit.SECONDS)) {
            executor.shutdownNow();  // force after 5s
        }
    } catch (InterruptedException e) {
        executor.shutdownNow();
        Thread.currentThread().interrupt();
    }
}

Graceful shutdown: shutdown() stops accepting new tasks, awaitTermination gives running tasks 5s to finish, then shutdownNow() interrupts stragglers.

Recurring Task Reschedule

After a recurring task executes, it is immediately re-offered to the PriorityQueue with nextTime = now + intervalSeconds. One-time tasks are removed from the Map.

if (task.isRecurring()) {
    LocalDateTime nextTime = now.plusSeconds(task.getIntervalSeconds());
    task.setScheduledTime(nextTime);
    synchronized (taskQueue) {
        taskQueue.offer(task);  // re-insert with new time
    }
} else {
    tasks.remove(task.getId());  // one-time: clean up
}

08 — Retry Logic

Current implementation: Failures are logged and swallowed. The task stays in the Map for recurring tasks. For production-grade retries, the following pattern applies.

Retry with Exponential Backoff

class ScheduledTask {
    private int maxRetries = 3;
    private int currentRetry = 0;

    public void executeWithRetry() {
        try {
            execute();
            currentRetry = 0;  // reset on success
        } catch (Exception e) {
            if (currentRetry < maxRetries) {
                currentRetry++;
                // Exponential backoff: 2^retry seconds
                scheduledTime = now().plusSeconds((long) Math.pow(2, currentRetry));
                taskQueue.offer(this);
            } else {
                sendToDeadLetterQueue();  // give up
            }
        }
    }
}

Preventing Duplicate Execution

class ScheduledTask {
    private AtomicBoolean executing = new AtomicBoolean(false);

    public void execute() {
        if (!executing.compareAndSet(false, true)) {
            return;  // already running in another thread
        }
        try {
            action.run();
        } finally {
            executing.set(false);
        }
    }
}

compareAndSet is lock-free. If two threads try to execute the same task, only one proceeds.

09 — Key Algorithms

Schedule Task — O(log n)

public String scheduleTask(ScheduledTask task, LocalDateTime scheduledTime) {
    String taskId = "TASK-" + taskIdGenerator.incrementAndGet();  // atomic
    task.setScheduledTime(scheduledTime);
    task.setId(taskId);

    tasks.put(taskId, task);            // O(1) HashMap
    synchronized (taskQueue) {
        taskQueue.offer(task);           // O(log n) heap insert
    }
    return taskId;
}

Execute Due Tasks — O(k log n)

public int executeDueTasks() {
    LocalDateTime now = LocalDateTime.now();
    List<ScheduledTask> dueTasks = new ArrayList<>();

    synchronized (taskQueue) {
        while (!taskQueue.isEmpty() &&
               !taskQueue.peek().getScheduledTime().isAfter(now)) {
            dueTasks.add(taskQueue.poll());  // O(log n) per poll
        }
    }

    for (ScheduledTask task : dueTasks) {
        executor.submit(() -> {
            try {
                task.execute();
                // reschedule or remove...
            } catch (Exception e) {
                System.err.println("Task failed: " + e.getMessage());
            }
        });
    }
    return dueTasks.size();
}

k = number of due tasks. The while-loop peeks at O(1) and polls at O(log n). Total: O(k log n).

Cancel Task — O(n)

public boolean cancelTask(String taskId) {
    ScheduledTask task = tasks.remove(taskId);  // O(1) HashMap
    if (task == null) return false;

    synchronized (taskQueue) {
        taskQueue.remove(task);  // O(n) linear scan
    }
    return true;
}

PriorityQueue.remove() is O(n) because it must scan the heap. Acceptable for low cancel rates. For high cancel rates, use a lazy-deletion approach with a HashSet of cancelled IDs.

Update Priority — O(n)

public boolean updateTaskPriority(String taskId, Priority priority) {
    ScheduledTask task = tasks.get(taskId);  // O(1)
    if (task == null) return false;

    synchronized (taskQueue) {
        taskQueue.remove(task);           // O(n) remove
        task.setPriority(priority);
        taskQueue.offer(task);            // O(log n) re-insert
    }
    return true;
}

Why remove + re-insert? Java's PriorityQueue doesn't support decreaseKey. Remove invalidates old position, offer restores heap invariant.

Query Time Range — O(n)

public List<ScheduledTask> getTasksInRange(LocalDateTime start, LocalDateTime end) {
    return tasks.values().stream()
        .filter(t -> !t.getScheduledTime().isBefore(start) &&
                    !t.getScheduledTime().isAfter(end))
        .collect(Collectors.toList());
}

Scans the Map values. At scale, use a TreeMap keyed by scheduledTime for O(log n + k) range queries.

10 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap tasks store Safe concurrent reads/writes; O(1) lookup
synchronized(taskQueue) offer, poll, remove on PriorityQueue PriorityQueue is NOT thread-safe; all heap operations are serialized
AtomicLong taskIdGenerator Lock-free unique ID generation; CAS-based increment
ScheduledExecutorService Task execution pool Thread pool with work queue; prevents thread exhaustion
AtomicBoolean Duplicate execution guard CAS ensures only one thread executes a given task
"I synchronize on the taskQueue object, not a global lock. The ConcurrentHashMap handles its own locking for the tasks Map. The two data structures have independent lock scopes — only PriorityQueue mutations need serialization."

Why Not PriorityBlockingQueue?

PriorityBlockingQueue is thread-safe but doesn't support peek() without removal or efficient batch draining. Our approach lets us drain all due tasks in a single synchronized block, then execute them outside the lock.

14 — Interview Follow-up Q&A

How would you scale to 1 million scheduled tasks?
Partitioned Queues: Shard by time range (next hour, next day, overflow). DB Persistence: Store in PostgreSQL, load only next N due tasks into memory. Distributed: Use Kafka scheduled messages or Redis sorted sets. Time Wheel: O(1) insertion (used by Kafka, Netty) — buckets for each time slot with overflow list. Trade-off: more memory, fixed granularity.
Why PriorityQueue over Time Wheel?
PriorityQueue: O(log n) insert, dynamic scheduling, general-purpose. Time Wheel: O(1) insert, better for 100k+ tasks/sec, but fixed granularity and more memory. Our choice suits human-scale intervals (seconds to hours). For real-time systems with microsecond precision, use a hierarchical time wheel.
How would you handle task failures and retries?
Add maxRetries and currentRetry fields. On failure, reschedule with exponential backoff: delay = 2^retry seconds. After max retries, send to a Dead Letter Queue for manual inspection. Advanced: circuit breaker pattern to stop retrying if the underlying service is down.
How do you prevent duplicate execution of the same task?
Use AtomicBoolean executing on each task. compareAndSet(false, true) ensures only one thread enters the execution block. In distributed systems, use a distributed lock (Redis SETNX) with TTL, or database row-level locking with a "locked_by" column.
How would you persist tasks across restarts?
On schedule: INSERT INTO tasks. On startup: SELECT * FROM tasks WHERE time > NOW() and reload into the PriorityQueue. On complete: DELETE FROM tasks. Challenges: DB becomes bottleneck, need indexing on scheduled_time, and handling clock skew between instances.
Why ConcurrentHashMap + synchronized(PriorityQueue) instead of PriorityBlockingQueue?
PriorityBlockingQueue doesn't support efficient batch draining or non-removing peek with time comparison. Our approach drains all due tasks in a single synchronized block, then submits them outside the lock. This minimizes lock hold time and allows true concurrent execution.
How would you size the thread pool?
Formula: Threads = CPUs × Utilization × (1 + Wait/Compute). Example: 8 cores, 50% util, 80% I/O → 8 × 0.5 × (1 + 4) = 20 threads. For CPU-bound tasks, use Runtime.getRuntime().availableProcessors(). For I/O-bound, multiply by wait ratio. Make it configurable via constructor.
How would you add cron expression support?
Extend Schedule with a cronExpression field. Parse cron expressions (using a library like cron-utils) to compute the next execution time. Override getNextExecutionTime() to call the cron parser. The rest of the system stays unchanged — it only cares about the next scheduled time.
How would you implement task dependencies (DAG)?
Model dependencies as a directed acyclic graph. Each task has a Set<String> dependsOn of task IDs. Before scheduling, do a topological sort. A task only enters the PriorityQueue when all its dependencies are COMPLETED. Use an in-degree counter: decrement when a dependency completes, schedule when in-degree reaches 0.
How would you add rate limiting?
Use a token bucket or sliding window rate limiter before submitting tasks to the executor. RateLimiter.acquire() (Guava) or a Semaphore with timed permits. This prevents the executor from being overwhelmed. Also add per-task-type rate limits for isolation.
What monitoring would you add in production?
Metrics: tasks scheduled/executed/failed per second, queue depth, execution latency (p50/p95/p99), thread pool utilization, retry rate. Alerts: queue depth > threshold, failure rate spike, execution latency > SLA. Logging: structured logs with taskId, status, duration. Expose via JMX or Prometheus.
How would you make the cancel operation O(log n)?
Use lazy deletion: mark the task as cancelled in the Map but don't remove from the PriorityQueue. When executeDueTasks() polls a task, check if it's cancelled — if so, skip it. This makes cancel O(1) at the cost of phantom entries in the queue. Periodically compact the queue to remove stale entries.
How would you distribute this across multiple instances?
Leader election: Use ZooKeeper/etcd to elect a single scheduler instance. Sharding: Hash task IDs to assign tasks to specific instances. Shared state: Use Redis sorted sets (ZADD by scheduled time) as the shared queue. Each instance polls with ZPOPMIN. Use distributed locks to prevent duplicate execution.
