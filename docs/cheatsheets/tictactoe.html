01 — Requirements

Functional

  • Two players alternate placing X and O on an N×N board (default 3×3)
  • First player to fill a complete row, column, or diagonal wins
  • If all cells are filled with no winner → Draw
  • Move validation — reject out-of-bounds and occupied cells
  • Undo support — revert the last move
  • Game reset without creating a new instance
  • Algebraic notation input (e.g., "a1", "b2", "c3")
  • AI opponent via pluggable strategy (Random, Minimax)
  • Move history tracking and replay
  • Game statistics (wins/losses/draws across sessions)

Non-Functional

  • Immutable board snapshots — getBoard() returns a copy
  • Extensible win checking — O(1) counter-based approach for N×N
  • Pluggable AI — any class implementing AIStrategy can play
  • Thread-safe reads via defensive copies
  • Clean separation — Board, Game, Win logic, AI are independent

03 — Design Patterns

Four patterns, each earning its place in the design.

State Pattern

Game lifecycle transitions. GameState drives which operations are legal. makeMove() refuses if NOT_STARTED or game over. State transitions happen atomically after each move.

Strategy Pattern

AIStrategy interface — pluggable AI opponents. RandomAI for testing, Minimax for unbeatable play. WinChecker can be swapped between O(n) scan and O(1) counter approaches.

Factory Pattern

GameConfig.standard() creates default 3×3 configuration. Extend for N×N boards, tournament settings, or custom rules without modifying game logic.

Command Pattern (Undo)

Each Move is an immutable command stored in moveHistory. Undo replays all moves except the last — enabling full game rewind and replay.

04 — Game State Machine

Why State? Prevents illegal operations at each lifecycle phase. makeMove() returns MoveResult.error() if the game hasn't started or is already over. No scattered if/else — each state knows what it allows. 
stateDiagram-v2
    [*] --> NOT_STARTED
    NOT_STARTED --> IN_PROGRESS: start()
    IN_PROGRESS --> X_WON: X completes row/col/diag
    IN_PROGRESS --> O_WON: O completes row/col/diag
    IN_PROGRESS --> DRAW: board full, no winner
    IN_PROGRESS --> IN_PROGRESS: valid move (switch player)
    X_WON --> NOT_STARTED: reset()
    O_WON --> NOT_STARTED: reset()
    DRAW --> NOT_STARTED: reset()

GameState Enum

enum GameState {
    NOT_STARTED,    // initial — only start() is legal
    IN_PROGRESS,    // moves allowed, checking win/draw after each
    X_WON,          // terminal — X completed a line
    O_WON,          // terminal — O completed a line
    DRAW            // terminal — board full, no winner
}

State Transition in makeMove()

public MoveResult makeMove(Position position) {
    // Guard: game must be active
    if (state == GameState.NOT_STARTED)
        return MoveResult.error("Game not started");
    if (state != GameState.IN_PROGRESS)
        return MoveResult.error("Game is over: " + state);

    board.makeMove(position, currentPlayer);
    moveHistory.add(new Move(currentPlayer, position));

    // Check for winner → X_WON or O_WON
    Optional<Player> winner = board.getWinner();
    if (winner.isPresent()) {
        state = winner.get() == Player.X ? GameState.X_WON : GameState.O_WON;
        return MoveResult.gameOver(true, state, winner.get());
    }

    // Check for draw → DRAW
    if (board.isFull()) {
        state = GameState.DRAW;
        return MoveResult.gameOver(true, state, null);
    }

    // Continue → switch player
    currentPlayer = currentPlayer.opponent();
    return MoveResult.success(currentPlayer);
}

05 — Win Detection

Interview Gold: Discuss both approaches and explain when to pick each. O(1) is the differentiator.

Approach 1: O(n) Scan (Implemented)

After each move, scan all rows, columns, and both diagonals. Simple and correct for 3×3.

public Optional<Player> getWinner() {
    // Check rows
    for (int row = 0; row < SIZE; row++) {
        Player w = checkLine(grid[row][0], grid[row][1], grid[row][2]);
        if (w != null) return Optional.of(w);
    }
    // Check columns
    for (int col = 0; col < SIZE; col++) {
        Player w = checkLine(grid[0][col], grid[1][col], grid[2][col]);
        if (w != null) return Optional.of(w);
    }
    // Check diagonals
    Player w = checkLine(grid[0][0], grid[1][1], grid[2][2]);
    if (w != null) return Optional.of(w);
    w = checkLine(grid[0][2], grid[1][1], grid[2][0]);
    if (w != null) return Optional.of(w);
    return Optional.empty();
}

private Player checkLine(Player a, Player b, Player c) {
    return (a != null && a == b && b == c) ? a : null;
}
MetricValue
Time per checkO(n) — scan 2n+2 lines
SpaceO(1) extra
Best for3×3 boards, simplicity, interviews that don't ask for optimization

Approach 2: O(1) Counter-Based (Optimal for N×N)

Maintain counters per row, column, and diagonal. X adds +1, O adds −1. If any counter reaches +N or −N, that player wins.

int[] rowCount = new int[N];   // one counter per row
int[] colCount = new int[N];   // one counter per column
int diagCount  = 0;            // main diagonal (\)
int antiDiag   = 0;            // anti-diagonal (/)

boolean checkWinO1(int row, int col, Player player) {
    int delta = (player == Player.X) ? 1 : -1;

    rowCount[row] += delta;
    colCount[col] += delta;
    if (row == col)         diagCount += delta;
    if (row + col == N - 1) antiDiag  += delta;

    int target = N * delta;  // +N for X, -N for O
    return rowCount[row] == target
        || colCount[col] == target
        || diagCount     == target
        || antiDiag      == target;
}
MetricValue
Time per checkO(1) — 4 comparisons
SpaceO(n) — 2n+2 counters
Best forN×N boards, performance-critical, impressing interviewers

06 — Board & Move Validation

Board Representation

public class Board {
    public static final int SIZE = 3;
    private final Player[][] grid;  // null = empty cell

    public Board() {
        this.grid = new Player[SIZE][SIZE]; // all null initially
    }

    public Board copy() { return new Board(this.grid); } // deep copy
}

Key design: Player[][] where null means empty. No separate Cell class needed — the enum IS the mark.

public boolean isValidMove(Position pos) {
    return isInBounds(pos) && grid[pos.getRow()][pos.getCol()] == null;
}

private boolean isInBounds(Position pos) {
    return pos.getRow() >= 0 && pos.getRow() < SIZE
        && pos.getCol() >= 0 && pos.getCol() < SIZE;
}

Algebraic Notation

Supports chess-like input: column letter + row number (e.g., "a1" = (0,0), "b2" = (1,1), "c3" = (2,2)).

public static Position fromNotation(String notation) {
    char colChar = Character.toLowerCase(notation.charAt(0));
    char rowChar = notation.charAt(1);
    int col = colChar - 'a';
    int row = rowChar - '1';
    return new Position(row, col);
}

07 — Key Algorithms

makeMove() — Core Game Loop

Time: O(1) with counter-based win check, O(n) with scan
Space: O(1) per move

public MoveResult makeMove(Position position) {
    // 1. Validate game state
    if (state != GameState.IN_PROGRESS)
        return MoveResult.error(...);

    // 2. Validate move legality
    if (!board.isValidMove(position))
        return MoveResult.error("Invalid move");

    // 3. Place mark on board
    board.makeMove(position, currentPlayer);
    moveHistory.add(new Move(currentPlayer, position));

    // 4. Check terminal conditions
    if (board.getWinner().isPresent()) { /* set state, return gameOver */ }
    if (board.isFull())                { /* set DRAW, return gameOver */ }

    // 5. Switch player & continue
    currentPlayer = currentPlayer.opponent();
    return MoveResult.success(currentPlayer);
}

checkWin() — O(1) Counter Technique

Insight: Map X → +1, O → −1. A counter reaching ±N means that player owns the entire line.

// Counters maintained alongside the board
int[] rows = new int[N], cols = new int[N];
int diag = 0, anti = 0;

void placeAndCheck(int r, int c, Player p) {
    int d = (p == X) ? 1 : -1;
    rows[r] += d;  cols[c] += d;
    if (r == c)         diag += d;
    if (r + c == N-1) anti += d;

    if (Math.abs(rows[r]) == N || Math.abs(cols[c]) == N
     || Math.abs(diag) == N    || Math.abs(anti) == N)
        // p wins!
}

undo() — Replay-Based Undo

Strategy: Save move history, then replay all moves except the last one on a fresh board.

public boolean undo() {
    if (moveHistory.isEmpty()) return false;

    // Snapshot history, then restart fresh
    List<Move> history = new ArrayList<>(moveHistory);
    start();  // resets board, clears moveHistory

    // Replay all moves except the last
    for (int i = 0; i < history.size() - 1; i++) {
        Move m = history.get(i);
        board.makeMove(m.getPosition(), m.getPlayer());
        moveHistory.add(m);
    }

    // Restore correct current player
    if (!moveHistory.isEmpty())
        currentPlayer = moveHistory.get(moveHistory.size()-1)
                                   .getPlayer().opponent();
    return true;
}

Trade-off: O(n) per undo, but avoids tracking reverse state. With O(1) counters, you can subtract the delta instead.

12 — Interview Follow-up Q&A

How would you generalize to an N×N board?
Make Board.SIZE configurable via GameConfig. The O(n) scan approach degrades linearly, so switch to the O(1) counter-based win detection: maintain int[N] row/col counters plus 2 diagonal counters. Win condition: any counter reaches ±N. The checkLine() method generalizes naturally but the counter approach is far superior for large N.
How would you support more than 2 players?
Change Player from an enum to a class with a symbol field. Maintain a List<Player> and cycle through them via an index. For O(1) win detection, use a Map<Player, int[]> per player for row/col/diag counters, or use separate counter arrays. Win condition: any player's counter reaches N.
How does undo work with the O(1) counter approach?
Two options: (1) Reverse the delta — subtract instead of add (O(1) undo). (2) Replay from scratch (current approach, O(k) for k moves). Option 1 is preferred for production: pop the last Move, compute negative delta, update counters. Option 2 is simpler to implement and less error-prone.
How would you implement an AI opponent using Minimax?
Implement AIStrategy with a MinimaxAI class. The algorithm: recursively try all empty positions, score terminal states (+10 for AI win, -10 for loss, 0 for draw), pick the move with the max score (for AI) or min (for opponent). Alpha-beta pruning cuts the search space from O(b^d) to O(b^(d/2)). For 3×3, the tree is small (~9! states) so even naive minimax is fast. Use board.copy() for safe exploration.
How would you add a tournament / match mode?
Create a Tournament class that manages multiple TicTacToeGame instances. Track scores via GameStats (already exists). Support best-of-N series. Alternate who plays X first each round. Add a TournamentObserver for live updates. Consider a Mediator pattern to coordinate match scheduling.
How would you make this a real-time multiplayer game?
Add a GameServer that holds a ConcurrentHashMap<String, TicTacToeGame> of active games. Use WebSockets for real-time communication. synchronized(game) around makeMove() for thread safety. Add a GameLobby for matchmaking. Each game gets a unique ID. Player actions are serialized into commands (Command pattern) sent over the wire.
Why use an enum for Player instead of a class?
For 2-player TicTacToe, an enum is perfect: (1) Exactly two instances, enforced by the JVM. (2) Built-in opponent() toggle. (3) Identity comparison with == (faster than equals). (4) Thread-safe singletons. (5) Natural switch/case support. For N-player generalization, refactor to a class.
How would you persist game state for resume-later play?
Serialize the moveHistory list (not the board — the board is derived state). On resume, create a new game, call start(), replay all moves. This is essentially the Event Sourcing pattern. Alternatively, serialize the full game state (board + currentPlayer + state) as a snapshot for instant restore.
What's the time complexity of the full game?
At most 9 moves in a 3×3 game. With O(n) win check: O(9n) = O(9×3) = O(27) = O(1) effectively. With O(1) counters: O(9) total. Space: O(n²) for the board + O(moves) for history. For N×N: max N² moves, so O(N³) with scan vs O(N²) with counters.
Why does getBoard() return a copy instead of the actual board?
Encapsulation. If we returned the real board, callers could mutate it directly (place marks, clear cells) bypassing all validation. The copy ensures: (1) Game invariants can't be broken externally. (2) AI can safely explore moves on the copy without affecting the real game. (3) Observers get a consistent snapshot. The O(n²) copy cost is trivial for small boards.
