01 — The Process (the meta-skill)

Working > complete. A running v1 with two well-chosen patterns beats a half-typed "perfect" design. Never end with code that doesn't compile.

  1. Clarify (3–5 min) — scope, inputs/outputs, scale, single vs multi-user, concurrency? Ask before designing; state assumptions aloud.
  2. Identify core entities (5 min) — nouns → classes. Sketch 4–6 key classes + relationships in comments before typing.
  3. Define contracts (5 min) — the service API + the 1–2 behavior/strategy interfaces.
  4. Walking skeleton (bulk) — simplest end-to-end path that compiles + runs a main within ~40 min. Then flesh out.
  5. Demo + edge cases — print output that proves it works; handle 2–3 edge cases.
  6. Extend live — they will add a requirement. Your design should absorb it with minimal change (that's what the patterns buy you).

02 — OO Principles

SOLID (the 3 that matter)

  • SRP — one reason to change per class. Split Order / PricingCalculator / OrderRepository.
  • Open/Closed — extend by new class, not by editing a switch. This is why Strategy/State exist.
  • Dependency Inversion — depend on interfaces. Field is PricingStrategy, not HourlyPricing.
  • (L & I: substitutability + small interfaces — know the names.)

4 habits interviewers watch for

  • Program to interfaces — List<X> l = new ArrayList<>(); service fields typed as interfaces.
  • Composition over inheritance — inheritance only for true "is-a"; else has-a + delegate.
  • Encapsulation — private fields; return copies / unmodifiable views, never leak mutable internals.
  • Immutable value objects — final fields, constructor-set, no setters (money, coords, IDs).

03 — Pattern Arsenal (the 6 that win ~90%)

Recognize by the smell. Use a pattern only when the smell is real — forcing patterns is penalized.

Situational bonus

Command (undo / queue of actions) · Builder (many optional fields) · Template Method (fixed skeleton, varying steps) · Chain of Responsibility (pipeline of handlers).

Enum-with-behavior — the lightweight State

For a linear lifecycle with no per-state behavior, skip class-per-state; put the transition table on the enum via canTransitionTo(next). Reach for full GoF State only when each state carries real behavior.

Must-know idioms

  • Enums with fields/behavior — cleaner than constant soup.
  • equals + hashCode together for value objects used as map keys; IDs for entities.
  • Comparable vs Comparator — Comparator.comparing(X::getP).reversed().
  • Generics — interface Repository<T, ID>; bounded <T extends Comparable<T>>.

Don't get burned

  • Money = BigDecimal, never double.
  • Time = java.time (LocalDate, Duration, Instant).
  • IDs — AtomicLong counter or UUID.randomUUID().
  • var / record only if the round's Java is 16+; explicit classes always compile — don't gamble.

05 — Concurrency Kit (the likely follow-up)

When asked "make it thread-safe," answer 3 questions aloud: (1) What is the shared mutable state? (2) What is the critical section? (3) What guards it, and at what granularity?

ToolUse it for
synchronized / ReentrantLockmutual exclusion; lock the smallest critical section
ConcurrentHashMapconcurrent map; computeIfAbsent for atomic get-or-create + per-key locks
Atomic* + compareAndSetlock-free single-slot state (e.g. seat AVAILABLE→HELD)
ReadWriteLockmany readers, few writers (caches, filesystem)
ExecutorServicenewFixedThreadPool(n), submit Callable, Future.get(), shutdown()
CountDownLatchwait for N tasks / start them together in tests

Two rules that impress

Lock granularity: a per-entity lock (ConcurrentHashMap<Id, Lock>) beats one global lock — different entities proceed in parallel.
Deadlock avoidance: when locking two things, always acquire in a consistent global order (e.g. by id) — kills the A→B / B→A deadlock.

06 — Testing & Edge Cases

Prove it runs

  • A main demo with 3–4 scenarios printing clear output is mandatory.
  • A few unit tests (happy path + 1–2 edges) is the bonus that impresses.

Edge cases to always name

null / empty input · duplicate · not-found · capacity / full · concurrent race · invalid state transition.

07 — Red Flags & Time Management

Marked down for

  • God class — everything in one file
  • No interfaces; switch on type instead of polymorphism
  • Magic numbers; no demo
  • Over-engineering a toy; forcing 5 patterns
  • Ignoring thread-safety when asked

Time discipline

  • 60% of time gone with nothing running? Stop adding classes — get the skeleton to run.
  • Build breadth-first: one thin end-to-end slice, then deepen.
  • Keep a running main from minute ~20 and grow it.

Interval overlap (TreeMap, O(log n))

// floor = latest booking starting at/before new start
Map.Entry<Integer,Integer> f = calendar.floorEntry(start);
if (f != null && f.getValue() > start) return CONFLICT;   // [s,e) half-open
Map.Entry<Integer,Integer> c = calendar.ceilingEntry(start);
if (c != null && c.getKey() < end)   return CONFLICT;
calendar.put(start, end);

Enum with behavior + transition table

enum Status {
  PENDING, DONE, FAILED;
  boolean canGoTo(Status n) {
    if (this == PENDING) return n == DONE || n == FAILED;
    return false;                     // DONE/FAILED are terminal
  }
}

Thread pool + wait for completion

ExecutorService pool = Executors.newFixedThreadPool(4);
List<Future<?>> fs = new ArrayList<>();
for (Task t : tasks) fs.add(pool.submit(t::run));
for (Future<?> f : fs) f.get();       // join
pool.shutdown();
