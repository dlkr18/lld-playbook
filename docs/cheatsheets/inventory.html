01 — Requirements

Functional

  • Receive stock into warehouses with quantity tracking
  • Reserve inventory for orders — holds from availability, not on-hand
  • Commit reservations on shipment — deducts from on-hand
  • Release reservations on cancellation — returns to availability
  • Adjust stock for shrinkage/cycle counts (positive or negative delta)
  • Transfer stock atomically between warehouses
  • Order lifecycle: place → confirm payment → pick → pack → deliver
  • Product catalog with SKU, pricing, categories, perishability
  • User management with addresses and geo-based warehouse routing
  • Payment processing with multi-method support (UPI, Cards, COD)

Non-Functional

  • Thread-safe — per-(SKU, Warehouse) fine-grained locking
  • Idempotent reservations — each reservation gets a unique ID
  • Invariant: reserved ≤ onHand — enforced on every mutation
  • Deadlock-free transfers — lock ordering by identity hash
  • Overflow-safe — uses Math.addExact() for quantity arithmetic

03 — Design Patterns

Clean separation — each pattern solves a specific inventory challenge.

Reserve-Commit-Release

Two-phase inventory: reserve holds quantity from availability, commit deducts from on-hand, release returns to pool. Prevents overselling while allowing concurrent orders.

Repository Pattern

ConcurrentHashMap stores behind service interfaces. Swap for JPA/Redis without touching business logic. In-memory for demos and tests.

Value Object Pattern

7 typed IDs (SkuId, WarehouseId, etc.) prevent primitive obsession. StockSnapshot, Address, Money are immutable. Thread-safe by construction.

Service Orchestration

OrderService orchestrates InventoryService, payment, and fulfillment. Single entry point coordinates multi-step workflows with rollback on failure.

StockSnapshot — Immutable Read

public final class StockSnapshot implements Serializable {
    private final long onHand;
    private final long reserved;

    public StockSnapshot(long onHand, long reserved) {
        if (onHand < 0 || reserved < 0) throw new IllegalArgumentException(...);
        if (reserved > onHand) throw new IllegalArgumentException(...);
        this.onHand = onHand;
        this.reserved = reserved;
    }

    public long onHand()    { return onHand; }
    public long reserved()  { return reserved; }
    public long available() { return onHand - reserved; }
}

Invariant reserved ≤ onHand enforced at construction. Snapshot is immutable — safe to pass across threads.

Atomic Transfer with Deadlock Prevention

public void transfer(SkuId skuId, WarehouseId from, WarehouseId to,
                     long quantity, String reason) {
    Entry src = getEntry(skuId, from);
    Entry dst = getEntry(skuId, to);

    // Lock ordering by identityHashCode to prevent deadlocks
    Object firstLock = src.lock, secondLock = dst.lock;
    if (System.identityHashCode(firstLock) > System.identityHashCode(secondLock)) {
        Object tmp = firstLock; firstLock = secondLock; secondLock = tmp;
    }

    synchronized (firstLock) {
        synchronized (secondLock) {
            long available = src.onHand - src.reserved;
            if (available < quantity)
                throw new IllegalArgumentException("Insufficient available at source");
            src.onHand -= quantity;
            dst.onHand = Math.addExact(dst.onHand, quantity);
        }
    }
}

Why ordered locks? Thread A transferring BLR-A→MUM-A and Thread B transferring MUM-A→BLR-A would deadlock without consistent ordering. identityHashCode gives a total order.

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap entries, reservations, orders, users, products, payments Safe concurrent reads/writes without global lock
synchronized(entry.lock) receiveStock, reserve, release, commit, adjust Per-(SKU, Warehouse) fine-grained lock — different products updated concurrently
Dual synchronized transfer() Locks both source and destination entries with consistent ordering
computeIfAbsent() getEntry() Atomic lazy initialization of stock entries
Math.addExact() All quantity additions Throws ArithmeticException on overflow instead of silent wraparound
ConcurrentHashMap.remove() commit(), release() Atomic reservation consumption — prevents double-commit
"I use fine-grained per-entry locking, not a global lock. Two different SKUs at different warehouses can be updated concurrently. Only concurrent operations on the same (SKU, Warehouse) are serialized."

Lock Granularity Design

private static final class Entry {
    long onHand;
    long reserved;
    final Object lock = new Object(); // per-entry lock object
}

// reserve() -- only locks THIS entry, not the entire map
public ReservationId reserve(SkuId skuId, WarehouseId whId, long qty, String ref) {
    Entry e = getEntry(skuId, whId);
    synchronized (e.lock) {
        long available = e.onHand - e.reserved;
        if (available < qty) throw new IllegalArgumentException("Insufficient");
        e.reserved = Math.addExact(e.reserved, qty);
        ReservationId id = ReservationId.random();
        reservations.put(id, new Reservation(id, skuId, whId, qty));
        return id;
    }
}

10 — Key Algorithms

Confirm Payment with Rollback

public Order confirmPayment(OrderId orderId, String gatewayTxnId) {
    Order order = orders.get(orderId);
    if (order.status() != OrderStatus.CREATED)
        throw new IllegalStateException("Not in CREATED state");

    try {
        // Reserve inventory for ALL line items
        List<OrderLineItem> reservedItems = new ArrayList<>();
        for (OrderLineItem item : order.lineItems()) {
            ReservationId resId = inventoryService.reserve(
                item.skuId(), order.assignedWarehouse(),
                item.quantity(), "order:" + orderId);
            reservedItems.add(item.withReservation(resId));
        }
        // Record payment, update order → CONFIRMED
        ...
    } catch (IllegalArgumentException e) {
        // Insufficient inventory → mark FAILED, no reservations to release
        orders.put(orderId, order.withStatus(OrderStatus.FAILED));
        throw new IllegalStateException("Insufficient inventory");
    }
}

Cancel with Reservation Release

public Order cancelOrder(OrderId orderId, String reason) {
    Order order = orders.get(orderId);

    // Release ANY reservations that were made
    for (OrderLineItem item : order.lineItems()) {
        if (item.isReserved()) {
            inventoryService.release(item.reservationId(),
                "order_cancelled:" + reason);
        }
    }
    // Update order status → CANCELLED
    ...
}

12 — Interview Q&A

Why reserve-commit instead of directly decrementing stock?
Direct decrement causes problems: if the order is cancelled after payment but before shipping, you've already deducted stock that needs to be added back. Reserve-commit gives a two-phase approach: reserve holds stock from availability (preventing overselling) without touching on-hand. If cancelled, release returns it. If shipped, commit deducts from on-hand. This cleanly separates "promised" from "physically gone".
How do you prevent overselling when two orders race for the last item?
Each (SkuId, WarehouseId) entry has its own lock object. reserve() acquires synchronized(entry.lock), checks available = onHand - reserved, and only increments reserved if sufficient. The first thread to enter wins; the second sees insufficient available and throws. Different SKUs are never contended.
How does the transfer avoid deadlocks?
Two threads transferring in opposite directions (A→B and B→A) would deadlock with naive lock-both. We use lock ordering by System.identityHashCode(): always lock the lower-hash lock first. This guarantees a total order, making cyclic waits impossible. In the rare case of hash collision, the JVM still serializes since they'd be the same lock object.
Why typed IDs (SkuId, WarehouseId) instead of plain String?
Compile-time safety. You can't pass a WarehouseId where a SkuId is expected. With plain strings, reserve("BLR-A", "MILK-1L", 30, "ref") compiles but silently swaps arguments. Typed IDs also centralize validation and provide random() factory methods for testing.
How would you add low-stock alerts and reorder points?
Add a reorderPoint field to a WarehouseConfig per (SKU, Warehouse). After every commit() or adjust(), check if available ≤ reorderPoint. If so, publish an event via an Observer or event bus. A ReorderService subscriber automatically creates a purchase order. The check happens inside the existing lock, so it's atomic with the stock mutation.
How would you handle partial fulfillment (some items out of stock)?
The current confirmPayment() is all-or-nothing: if any item fails to reserve, the order goes to FAILED. For partial fulfillment: (1) reserve each item independently, (2) split the order into sub-orders per reservation success, (3) adjust pricing for the partial order, (4) refund the difference. This requires an OrderSplitter and richer status tracking.
Why ConcurrentHashMap + synchronized instead of just synchronized?
ConcurrentHashMap allows lock-free reads for getStock() and getOrder() — reads don't block writes. The synchronized(entry.lock) is only for mutations that need atomicity (check-then-act). A global synchronized on the map would serialize all operations across all SKUs.
How would you scale this to millions of SKUs?
The in-memory design has the right lock granularity (per-entry). For production: (1) Replace ConcurrentHashMap with Redis (WATCH/MULTI for CAS) or PostgreSQL SELECT FOR UPDATE. (2) Shard by WarehouseId — each warehouse is independent. (3) Use CQRS: write path goes through the service, read path uses materialized views. The InventoryService interface stays identical.
Why does adjust() enforce newOnHand ≥ reserved?
If adjust() could drop on-hand below reserved, the invariant reserved ≤ onHand breaks. This would mean we've promised more stock to orders than physically exists. The adjustment must account for existing holds. For example, if onHand=50 and reserved=30, the maximum negative adjustment is −20 (to keep onHand=30 ≥ reserved=30).
How does OrderService coordinate inventory and payment?
OrderService follows the Saga pattern informally: (1) placeOrder() validates and creates the order in CREATED state. (2) confirmPayment() reserves inventory, then records payment. If inventory reservation fails, order → FAILED (compensating action). (3) On delivery, commit(). On cancellation, release() + refund. Each step has a clear compensating action.
How would you handle reservation expiry (timeout)?
Add a createdAt timestamp to each Reservation. Run a ScheduledExecutorService that periodically scans reservations older than N minutes and calls release(). This prevents inventory being held indefinitely by abandoned carts. The release is safe because it's idempotent (reservation removed atomically via ConcurrentHashMap.remove()).
Why immutable Order objects instead of mutable state?
Each status transition creates a new Order instance with the updated status. This gives: (1) Thread safety — no concurrent mutation of order fields. (2) Audit trail — each version is a snapshot. (3) Functional style — withReservation() on OrderLineItem returns a new copy. The ConcurrentHashMap.put() atomically swaps the reference.
