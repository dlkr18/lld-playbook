Blackjack / Deck of Cards // LLD

A single round of casino Blackjack: shuffle, deal, hit/stand/double-down, Ace soft/hard valuation, a pluggable dealer draw policy, and 3:2 / 6:5 payouts — all reproducible from a seed. State machine for the round, Strategy for the rules that vary.

01 — Requirements

Functional

  • Deck / Shoe — represent a standard 52-card deck (4 suits × 13 ranks, all distinct); support a multi-deck shoe with a reshuffle threshold; shuffle reproducibly from a seed
  • Deal — deal opening hands casino-style: round-robin, two cards to each player then the dealer
  • Hand valuation — compute value with Ace as 1 or 11 (soft/hard), detect bust (>21) and natural blackjack (two-card 21)
  • Player actions — hit, stand, double-down (split scoped out as a follow-up)
  • Dealer policy — draw to 16, stand on 17, with soft-17 behaviour swappable (S17 vs H17)
  • Settlement — each player vs dealer into WIN / LOSE / PUSH / BLACKJACK, with net payout (1:1, 3:2 or 6:5; push returns stake)

Non-Functional

  • Reproducibility — identical seed ⇒ identical game; all randomness confined to the deck/shoe factory
  • Extensibility — new dealer rules, payout rules, and card sources drop in without editing the orchestrator (Open/Closed)
  • Correctness & clarity — state transitions are explicit and guarded; no silent illegal states
  • Testability — hand valuation is a pure function; round logic is exercised with fully scripted decks
  • Single-threaded by design — a table is a turn-based unit with no shared mutable state; concurrency lives one layer up (many independent tables). This is an intentional decision, documented not accidental — no locking anywhere on the table.

03 — Design Patterns

Five patterns, each earning its place — the interview differentiator. Rules that vary are strategies; rules that don't are code.

Strategy — Dealer Policy

DealerStrategy.shouldHit(hand) — the only real dealer variation is soft-17 behaviour. StandOnSoft17 vs HitOnSoft17 swap in without touching the round loop. Alternative (a boolean flag / switch in the orchestrator) reopens the god object for every new rule.

Strategy — Payout

PayoutStrategy.netPayout(bet, outcome) — only the ratios vary between houses (3:2 vs 6:5). StandardPayout and SixToFivePayout are injected; deciding which outcome occurred stays invariant in the orchestrator.

State Machine (enum-guarded)

Round lifecycle: BETTING → PLAYER_TURN → DEALER_TURN → SETTLEMENT → DONE. Every method calls requireState(…) then advances, so "hit before deal" or "settle twice" throw immediately. Chosen over GoF State (class-per-phase) — too much ceremony for five linear transitions.

Factory Method + Builder

Deck.shuffled(seed) / Shoe.of(n, seed) capture the seed & shuffle at one boundary — a caller can't hand a raw unshuffled list. Blackjack.Builder assembles card source + two strategies + seated players, validates once in build().

Dependency Inversion

The orchestrator depends only on CardSource / DealerStrategy / PayoutStrategy — never on a concrete rule. The identical round loop runs against one Deck or a six-deck Shoe.

04 — Round State Machine

Why an enum state machine? Five strictly one-way phases with no shared per-state data. Each public method guards its phase then advances it, so the round sequence is self-enforcing rather than relying on caller discipline. A GoF State class hierarchy would be more indirection than five linear transitions earn. 
stateDiagram-v2
    [*] --> BETTING
    BETTING --> PLAYER_TURN: deal()
    PLAYER_TURN --> PLAYER_TURN: hit() / stand() / doubleDown() [players remain]
    PLAYER_TURN --> DEALER_TURN: last player done
    DEALER_TURN --> SETTLEMENT: playDealer()
    SETTLEMENT --> DONE: settle()
    DONE --> [*]

Phase-by-phase

MethodRequiresDoes & advances
deal()BETTINGDeals two round-robin cards to each player then the dealer; flags any naturals as STANDING (they don't act) → PLAYER_TURN
hit() / stand() / doubleDown()PLAYER_TURNActs on the current player, then advancePastDonePlayers() walks the seat cursor past everyone done; running off the end → DEALER_TURN
playDealer()DEALER_TURNDraws per DealerStrategy (skips entirely if every player busted — result already decided) → SETTLEMENT

The Guard

private RoundState state = RoundState.BETTING;

private void requireState(RoundState expected) {
    if (state != expected) {
        throw new IllegalStateException(
            "Expected " + expected + " but round is in " + state);
    }
}

// Every public transition opens with its guard:
public Card hit() {
    requireState(RoundState.PLAYER_TURN);
    ...
}

05 — Strategy Pattern

Why Strategy? Dealer draw policy and payout ratios are the only dimensions that genuinely vary between real tables. Encapsulating each behind an interface keeps the round loop closed to modification — new rule = new class, no edit to the orchestrator. Open/Closed Principle.

Dealer Strategy

interface DealerStrategy {
    boolean shouldHit(Hand dealerHand);
    String describe();   // e.g. "S17" / "H17" for demo output
}
ImplementationLogic
StandOnSoft17Hit while total < 17; stand on all 17s, including a soft 17 (A+6). The conservative house rule.
HitOnSoft17Hit while total < 17, and hit a soft 17 (value == 17 && isSoft()) — a soft hand can't bust on the next card. Slightly better house edge.

Payout Strategy

interface PayoutStrategy {
    // signed net profit: + on win, 0 on push, - when bet is lost
    Money netPayout(Money bet, Outcome outcome);
    String describe();   // e.g. "3:2" / "6:5"
}
ImplementationBLACKJACKWINPUSHLOSE
StandardPayout+1.5× bet (3:2)+1× (1:1)0 (return stake)−bet
SixToFivePayout+1.2× bet (6:5)+1× (1:1)0 (return stake)−bet

Money multiplies in minor units via basis points (percent) — 150% of $10 is exactly $15, no floating-point drift.

Why isn't outcome resolution a strategy? Deciding which outcome occurred (bust always loses, blackjack beats a three-card 21, compare totals) is invariant game logic — it stays in the orchestrator. Only the amount each outcome pays varies, so only that is delegated. The split between "what happened" (fixed) and "what it pays" (variable) is deliberate.

06 — Key Algorithm: Ace Soft/Hard Valuation

The signature detail. The insight that removes all special-casing: at most one Ace can ever count as 11, because two 11s = 22 is already a bust. So compute the hard total (every Ace = 1), and if the hand holds any Ace and adding 10 doesn't bust, promote exactly one Ace to 11.

The One Expression

int hard = 0; boolean hasAce = false;
for (Card c : cards) {
    hard += c.baseValue();       // Ace counts as 1 here
    if (c.isAce()) hasAce = true;
}
return (hasAce && hard + 10 <= 21) ? hard + 10 : hard;   // promote ONE ace to 11

Why one expression handles every case

HandHard total+10 fits?value()
A + 6717 ≤ 21 ✓soft 17
A + A + 91121 ≤ 21 ✓21 (one Ace 11, one Ace 1 — never 31)
A + 6 + 101727 > 21 ✗17 (Ace auto-demoted)

Derived predicates

// isSoft() is the SAME predicate -- a soft hand can't bust on the next hit,
// which is exactly what the H17 dealer rule keys off.
public boolean isSoft() {
    int hard = hardTotal();
    return hasAce() && hard + 10 <= 21;
}

public boolean isBust()      { return value() > 21; }

// A NATURAL blackjack is a two-card 21 -- kept distinct from a three-card 21
// so it can beat a dealer 21 and pay 3:2.
public boolean isBlackjack() { return size() == 2 && value() == 21; }

Recompute-on-read is O(cards) on a hand of ≤ ~11 cards — negligible, and a pure function is far easier to test than a running counter with Ace bookkeeping.

07 — Deck / Shoe Construction & Seeded Shuffle

Reproducibility by construction. All randomness is a seeded Random created inside the factory. Same seed → identical deal, every run. This is why persistence/replay is trivial: persist the seed + action log and the RNG guarantees an identical replay.

CardSource abstraction

interface CardSource {
    Card deal();            // next card off the top
    int  remaining();       // cards left
    boolean needsReshuffle();  // Deck: only when empty; Shoe: at the cut card
}

Deck — factory methods (never a raw constructor)

public final class Deck implements CardSource {

    // 52 distinct cards in canonical order (suit x rank)
    public static Deck standardOrdered() { ... }

    // Shuffle a standard deck with a seeded RNG -> reproducible
    public static Deck shuffled(long seed) {
        List<Card> cards = standardOrdered().cards();
        Collections.shuffle(cards, new Random(seed));
        return new Deck(cards);
    }

    // Scripted deck -- the test hook (exact deal, no chance)
    public static Deck of(Card... orderedCards) { ... }

    // Front-to-back cursor; a single deck reshuffles only when exhausted
    public boolean needsReshuffle() { return isEmpty(); }
}

Shoe — multi-deck with a cut card

public final class Shoe implements CardSource {

    // N decks shuffled together from one seed; ~75% penetration cut card
    public static Shoe of(int numDecks, long seed) { ... }

    // Fires EARLY -- once penetration passes the cut, not just when empty
    public boolean needsReshuffle() { return remaining <= cutCardPosition; }
}

The orchestrator draws through CardSource.deal() and never knows whether it's one deck or six — the demo proves the identical loop runs against both.

08 — Public API

The orchestrator surface — each call is a guarded state transition. What to write on the whiteboard first.

public final class Blackjack {

    // --- transitions (each guards + advances RoundState) ---
    Blackjack    deal();                  // BETTING     -> PLAYER_TURN
    Card         hit();                   // draw for current player
    void         stand();                 // current player stops
    Card         doubleDown();            // double bet, take exactly one card
    List<Card>   playDealer();            // DEALER_TURN -> SETTLEMENT
    List<RoundResult> settle();          // SETTLEMENT  -> DONE

    // --- queries ---
    Player       currentPlayer();         // whose turn (PLAYER_TURN only)
    boolean      hasPendingPlayer();      // any player still to act?
    RoundState   state();
    Hand         dealerHand();
    Card         dealerUpCard();          // the one visible dealer card
    List<Player> players();
    int          cardsRemaining();

    // --- fluent Builder ---
    static Builder builder();
}

Blackjack game = Blackjack.builder()
    .cardSource(Deck.shuffled(42L))
    .dealerStrategy(new StandOnSoft17())
    .payoutStrategy(new StandardPayout())
    .player("Alice", Money.dollars(10))
    .player("Bob",   Money.dollars(20))
    .build();   // validates once, returns an immutable table

09 — Happy Path Flow

Trace one round end-to-end in the interview.

sequenceDiagram
    participant C as Caller
    participant B as Blackjack
    participant S as CardSource
    participant D as DealerStrategy
    participant P as PayoutStrategy
    C->>B: deal()
    B->>S: deal() x (2 per player + 2 dealer)
    Note over B: naturals flagged STANDING; -> PLAYER_TURN
    C->>B: hit() / stand() / doubleDown()
    B->>S: deal() (on hit / double)
    Note over B: advance seat cursor; last done -> DEALER_TURN
    C->>B: playDealer()
    loop while shouldHit(dealerHand)
        B->>D: shouldHit(dealerHand)
        B->>S: deal()
    end
    Note over B: -> SETTLEMENT
    C->>B: settle()
    B->>P: netPayout(bet, outcome) per player
    B-->>C: List<RoundResult>
Build the round
Builder injects a seeded Deck.shuffled(42L), StandOnSoft17, StandardPayout, and two seated players. build() validates ≥1 player + all collaborators present.
deal() — BETTING → PLAYER_TURN
Round-robin: two cards to each player, then two to the dealer. Any player dealt a natural is flagged STANDING (they don't act).
Player turns
currentPlayer() acts via hit() / stand() / doubleDown(). After each action advancePastDonePlayers() walks the cursor past anyone BUST or STANDING.
A hit that busts sets PlayerStatus BUST; doubleDown doubles the bet, draws exactly one card, then stands.
Last player done → DEALER_TURN
When the seat cursor runs off the end, state flips to DEALER_TURN automatically.
playDealer() — DEALER_TURN → SETTLEMENT
Draw while dealerStrategy.shouldHit(dealerHand). Skipped entirely if every player already busted (result is decided).
settle() — SETTLEMENT → DONE
Ordered outcome checks per player, then payoutStrategy.netPayout(bet, outcome). Returns immutable RoundResults; summing netPayout gives the house result.

Outcome resolution (invariant — in the orchestrator, ordered)

1. player bust        → LOSE   // even if the dealer also busts
2. player natural     → PUSH vs dealer natural, else BLACKJACK
3. dealer natural     → LOSE
4. dealer bust        → WIN
5. otherwise compare totals → WIN / LOSE / PUSH

// Orchestrator state (single-threaded -- no concurrent collections needed)
RoundState          state = RoundState.BETTING;

int                 currentPlayerIndex;  // seat cursor
List<RoundResult>   results;         // filled at settlement

// Deck: a plain List<Card> + a front-to-back int cursor
List<Card> cards; int next;   // deal() returns cards.get(next++)

// Hand: a growing List<Card>; value() recomputes on read (pure)
List<Card> cards;
Why no ConcurrentHashMap / locks? A table is dealt from by one round loop; players act in strict turn order. There is no shared concurrent state, so plain List and int cursors suffice. Parallelism lives one layer up — many independent tables, each confined to its own thread ("shard by table").

11 — Interview Follow-up Q&A

How would you add split pairs?
Break the "one hand per player" assumption: a player holds a list of hands, each with its own bet. Player owns List<Hand> and the turn loop iterates hands, not players. The state machine is unaffected. This is the main reason split was scoped out — it's a data-model change, not a rules change.
How would you add insurance / surrender?
New player options offered when the dealer shows an Ace. Add the actions in PLAYER_TURN and a side-bet resolution step before settlement. Fits as new Outcomes plus a pre-settlement hook — no change to the core phase flow.
How would you support card counting / true count?
The Shoe already exposes penetration. A counter would observe dealt cards — an Observer on deal(). Reshuffle policy is already pluggable via needsReshuffle(), so a counting-aware reshuffle threshold drops in without touching the orchestrator.
How would you scale to many tables / a whole casino?
Each Blackjack is thread-confined with no shared mutable state, so run many on a thread pool. No shared state means no locking — the concurrency story is simply "shard by table." This is why the single-threaded table is a feature, not a limitation.
Why an enum state machine instead of GoF State classes?
Five strictly linear transitions with no shared per-state data. An enum guard (requireState then advance) gives fail-fast errors with clear messages at a fraction of the ceremony of a class-per-phase hierarchy. If actions became wildly phase-dependent, promoting to State classes is the natural next step — noted as a follow-up, not premature now.
Why compute hand value on every read instead of incrementally?
Recomputing from scratch is O(cards) on a hand of ≤ ~11 cards — negligible. A pure function is far easier to reason about and test than a running counter with Ace-promotion bookkeeping that must be kept correct on every add. Simplicity/purity over micro-optimising a trivial computation.
How do you keep games reproducible and testable?
All randomness is a seeded Random created inside Deck.shuffled(seed) / Shoe.of(n, seed) — same seed produces an identical deal every run. Tests use fully scripted Deck.of(...) decks so no assertion depends on chance. Persistence/replay = persist the seed + action log.
Could you reuse this deck for Poker or War?
Yes — Deck / Card / Shoe / Hand are game-agnostic and reusable. Only the orchestrator (Blackjack) and the strategies are Blackjack-specific. A different game supplies its own orchestrator and rules over the same card primitives.
Why is outcome logic in the orchestrator, not a strategy?
Deciding which outcome occurred (bust always loses, natural beats a three-card 21, compare totals) is invariant across houses, so it lives in one readable place. Only the payout amount varies, so only that is a strategy. Trade-off: a truly exotic variant (e.g. a push-22 rule) would touch the orchestrator — judged the right default for clarity.
