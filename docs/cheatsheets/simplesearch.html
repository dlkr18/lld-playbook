01 — Requirements

Functional

  • Index documents with id, title, content, and creation timestamp
  • Full-text search — query returns documents ranked by relevance
  • Pagination — limit + offset support on search results
  • Update documents — re-index with new content atomically
  • Remove documents — clean removal from all index entries
  • Duplicate rejection — indexDocument returns false if ID exists
  • Relevance scoring via TF-IDF (term frequency × inverse document frequency)

Non-Functional

  • Thread-safe — concurrent indexing and searching via ConcurrentHashMap
  • Interface-driven — SearchEngine interface decouples API from implementation
  • Extensible — swap inverted index for Trie, BM25, or Elasticsearch
  • O(1) document lookup by ID, O(k) search where k = matching docs
  • Response time < 100ms for typical queries on in-memory data

03 — Design Patterns

Clean separation of concerns — each pattern addresses a specific design force.

Repository Pattern

SearchEngine interface abstracts data access. In-memory impl today, swap for Elasticsearch or Solr tomorrow. Zero changes to callers.

Strategy Pattern (ready)

Scoring is isolated in calculateScore(). Extract to a ScoringStrategy interface to swap TF-IDF for BM25, PageRank, or ML-based ranking.

Interface Segregation

SearchEngine exposes only 6 methods. Clients depend on the interface, not the ConcurrentHashMap internals. Testable via mocks.

Immutable Value Objects

Document and SearchResult are immutable (final fields, no setters). Thread-safe by design, safe to share across threads without copying.

04 — Inverted Index

Why Inverted Index? Maps every unique term to the set of documents containing it. Turns a full-text search into a hash lookup — O(1) per query term instead of scanning all documents.

How Indexing Works

public boolean indexDocument(Document document) {
    if (documents.containsKey(document.getId())) return false;

    documents.put(document.getId(), document);

    String[] words = tokenize(document.getContent());
    documentWordCount.put(document.getId(), words.length);

    for (String word : words) {
        invertedIndex.computeIfAbsent(word, k -> ConcurrentHashMap.newKeySet())
                     .add(document.getId());
    }
    return true;
}

Key detail: computeIfAbsent atomically creates the Set if missing. ConcurrentHashMap.newKeySet() returns a concurrent set for thread-safe membership updates.

How Removal Works

public boolean removeDocument(String documentId) {
    Document doc = documents.remove(documentId);
    if (doc == null) return false;

    String[] words = tokenize(doc.getContent());
    for (String word : words) {
        Set<String> docIds = invertedIndex.get(word);
        if (docIds != null) {
            docIds.remove(documentId);
            if (docIds.isEmpty()) invertedIndex.remove(word);
        }
    }
    documentWordCount.remove(documentId);
    return true;
}

Removal cleans up ALL posting lists. Empty terms are garbage-collected from the index to avoid memory leaks.

Tokenizer Implementation

private String[] tokenize(String text) {
    return text.toLowerCase()
               .replaceAll("[^a-z0-9\\s]", "")
               .split("\\s+");
}

Production upgrades: stemming (Porter/Snowball), stop-word removal, n-gram tokenization, Unicode normalization. The tokenize() method is a single extraction point for all of these.

06 — TF-IDF Scoring Algorithm

TF-IDF = Term Frequency × Inverse Document Frequency. Rewards terms that appear frequently in a document but rarely across the corpus. The classic information retrieval ranking signal.
TF
1/|d|
Term freq / doc length
IDF
log(N/df)
log(total docs / docs with term)
SCORE
Σ TF·IDF
Sum over all query terms

Scoring Implementation

public List<SearchResult> search(String query, int limit, int offset) {
    String[] queryWords = tokenize(query);
    int totalDocs = documents.size();
    Map<String, Double> docScores = new HashMap<>();

    for (String word : queryWords) {
        Set<String> docIds = invertedIndex.get(word);
        if (docIds != null) {
            double idf = Math.log((double) totalDocs / docIds.size());
            for (String docId : docIds) {
                double tf = 1.0 / documentWordCount.getOrDefault(docId, 1);
                docScores.merge(docId, tf * idf, Double::sum);
            }
        }
    }

    return docScores.entrySet().stream()
        .map(e -> new SearchResult(documents.get(e.getKey()), e.getValue()))
        .sorted((a, b) -> Double.compare(b.getScore(), a.getScore()))
        .skip(offset)
        .limit(limit)
        .collect(Collectors.toList());
}

Scoring Walk-through

Query: "java design" against 5 indexed documents.

"java"3log(5/3) = 0.51d1, d2, d5
"design"2log(5/2) = 0.92d1, d4

d1 matches both terms, so its score = TF("java")×IDF + TF("design")×IDF — ranked highest. Rare terms ("design", fewer docs) contribute more than common terms ("java").

07 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap documents, invertedIndex, documentWordCount Lock-free reads, segment-level writes — multiple threads index/search simultaneously
ConcurrentHashMap.newKeySet() Posting lists (Set<String>) Thread-safe set backed by ConcurrentHashMap — safe concurrent add/remove of doc IDs
computeIfAbsent Index insertion Atomic check-and-create — no race between "does key exist?" and "put new set"
Immutable Document All final fields Safely shared across threads without defensive copies
getOrDefault Word count lookup in scoring Null-safe access to documentWordCount during concurrent removals
"All three maps are ConcurrentHashMaps. Posting list sets use ConcurrentHashMap.newKeySet(). No explicit locks needed — the data structures handle contention internally."

12 — Interview Follow-up Q&A

Why an inverted index instead of scanning all documents?
Linear scan is O(N × L) where N = documents, L = avg length. Inverted index reduces search to O(1) per query term for the posting list lookup, then O(k) to iterate matches. At 1M documents, the difference is orders of magnitude.
How would you handle phrase queries like "design patterns"?
Store positional index: term → List<(docId, position)>. For phrase queries, find docs where "design" appears at position i and "patterns" at position i+1. This is the standard approach in Lucene/Elasticsearch.
How would you improve ranking beyond TF-IDF?
BM25 (Okapi) is the industry standard — it adds term frequency saturation and document length normalization parameters (k1, b). Extract scoring to a ScoringStrategy interface, swap TF-IDF for BM25 without touching the engine. For web search, layer in PageRank or ML-based learning-to-rank.
How would you add stemming?
Insert a stemming step in tokenize() — e.g., Porter Stemmer reduces "running" → "run", "patterns" → "pattern". Must apply the same stemmer at index-time AND query-time. The tokenize method is a single extraction point, so this is a one-line change.
What about stop-word removal?
Maintain a Set<String> STOP_WORDS containing "the", "is", "at", "in", etc. Filter tokens in tokenize() before indexing. Reduces index size by ~30% and avoids useless postings. Trade-off: exact phrase queries like "to be or not to be" break.
How does update work atomically?
updateDocument() calls removeDocument() then indexDocument(). Between these calls, a search might miss the document. For true atomicity, wrap in synchronized(documents) or use a read-write lock. In production, Elasticsearch uses versioned segments with refresh intervals.
How would you scale to millions of documents?
Sharding: partition the index by document ID hash across multiple nodes. Each shard is an independent inverted index. Query all shards in parallel, merge-sort results by score. This is exactly how Elasticsearch works with its shard/replica model.
Why ConcurrentHashMap instead of synchronized HashMap?
ConcurrentHashMap uses lock striping (segment-level locks) — multiple threads can read/write different segments simultaneously. Synchronized HashMap locks the entire map for every operation. Under high concurrency (10+ threads), ConcurrentHashMap is 5-10x faster for mixed read/write workloads.
How would you handle typos and fuzzy matching?
Edit distance (Levenshtein): compute distance between query term and all index terms, return matches within threshold. Too slow at scale. Better: build an n-gram index (bigrams of each term), use it to find candidate terms, then verify with edit distance. Elasticsearch uses this approach with its fuzziness parameter.
How would you add auto-complete / prefix search?
Replace the HashMap index with a Trie (prefix tree). Given prefix "jav", traverse the trie to find all terms starting with "jav" (java, javascript, etc.) and return their posting lists. Time: O(prefix length + results). Elasticsearch uses completion suggester backed by an FST (finite state transducer).
What if the index doesn't fit in memory?
Use disk-based inverted index with memory-mapped files. Posting lists stored as compressed sorted arrays on disk (vByte or PFOR encoding). Term dictionary in a B-tree or FST. Load hot terms into an LRU cache. This is how Lucene works — segments on disk, OS page cache for hot data.
Why are Document and SearchResult immutable?
Immutable objects are inherently thread-safe — no synchronization needed when sharing across threads. They also prevent accidental mutation of indexed data (which would make the index inconsistent). Java's final fields guarantee safe publication in the Java Memory Model.
