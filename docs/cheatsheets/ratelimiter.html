Rate Limiter // API Throttling

Production-grade rate limiting with Token Bucket, Sliding Window, Fixed Window, and Leaky Bucket algorithms. Thread-safe, per-client, with retry-after metadata.

Strategy Pattern Token Bucket Sliding Window Factory Pattern Thread-Safe

01 — Requirements

Functional

  • Request Throttling — check if request allowed based on client ID, track count per client, block requests exceeding limit, return remaining quota
  • Multiple Algorithms — Token Bucket (smooth rate + burst), Sliding Window (most accurate), Fixed Window (simple), Leaky Bucket (smooth output)
  • Client Management — unique client ID (API key, IP, user ID), per-client rate limit config, tiered limits (free vs premium), auto-cleanup of inactive clients
  • Retry Information — calculate Retry-After header value, time until next available slot, remaining requests in current window
  • Limit Reset — manual reset for specific client, automatic reset after window expires, admin override capabilities

Non-Functional

  • Performance — O(1) request check (Token Bucket), sub-millisecond latency, 10K+ concurrent clients, 100K+ req/sec
  • Accuracy — no false negatives (never allow over-limit), minimize false positives, Sliding Window: exact counting, Token Bucket: burst tolerance
  • Scalability — stateless design for horizontal scaling, distributed rate limiting with Redis, efficient memory usage per client
  • Reliability — thread-safe operations, no race conditions in count updates, graceful degradation (allow requests if limiter fails)

03 — Design Patterns

Two core patterns make this extensible and interview-ready.

Strategy Pattern

RateLimiter interface with multiple implementations: TokenBucketRateLimiter, SlidingWindowRateLimiter. Each algorithm is interchangeable. Client code depends on the interface, not the concrete class.

Factory Pattern

Use RateLimitAlgorithm enum + factory method to instantiate the correct limiter. Config-driven: change algorithm by changing a config value, no code changes.

Template Method

All limiters share the same contract: allowRequest() → RateLimitResult. Each fills in algorithm-specific logic. Common metadata (remaining, retryAfter) returned uniformly.

Repository Pattern

ConcurrentHashMap for per-client state (buckets / timestamp queues). In production, swap for Redis. Interface stays the same.

Factory Method Example

public static RateLimiter create(RateLimitAlgorithm algo, RateLimitConfig config) {
    return switch (algo) {
        case TOKEN_BUCKET   → new TokenBucketRateLimiter(
                                    config.getMaxRequests(),
                                    config.getMaxRequests());  // refill = capacity
        case SLIDING_WINDOW → new SlidingWindowRateLimiter(
                                    config.getMaxRequests(),
                                    config.getTimeWindowMillis());
        case FIXED_WINDOW   → new FixedWindowRateLimiter(
                                    config.getMaxRequests(),
                                    config.getTimeWindowMillis());
        case LEAKY_BUCKET   → new LeakyBucketRateLimiter(
                                    config.getMaxRequests());
    };
}

05 — Token Bucket Algorithm

Analogy: A bucket holds tokens. Each request takes one token. Tokens refill at a constant rate. When empty, requests are blocked. A full bucket allows burst traffic up to its capacity. 
graph TD
    A["Bucket: capacity tokens"] -->|"refill rate: R tokens/sec"| A
    B["Incoming Request"] --> C{"tokens > 0?"}
    C -->|Yes| D["Consume 1 token
ALLOW request"]
    C -->|No| E["BLOCK request
retryAfter = 1000/R ms"]

How It Works

  • Bucket starts with capacity tokens (e.g., 10)
  • Background thread refills tokens every 100ms at refillRate/sec
  • Each allowRequest() tries to consume 1 token
  • If tokens available: decrement & return ALLOWED
  • If empty: return BLOCKED with retryAfter = 1000 / refillRate ms
  • Burst: Full bucket allows capacity requests instantly

Timeline Example

Capacity: 10 tokens, Refill Rate: 5 tokens/sec

t=0.0s: 10 tokens (full bucket)
t=0.1s: Requests 1-5 consumed → 5 tokens left
t=1.0s: Refilled +5 → 10 tokens
t=1.1s: Requests 6-15 → 0 tokens, 5 requests BLOCKED
t=2.0s: Refilled +5 → 5 tokens available

TokenBucketRateLimiter.java

public class TokenBucketRateLimiter implements RateLimiter {

    private final ConcurrentHashMap<String, TokenBucket> buckets = new ConcurrentHashMap<>();
    private final int capacity;
    private final int refillRate;          // tokens per second
    private final ScheduledExecutorService refillScheduler;

    public TokenBucketRateLimiter(int capacity, int refillRate) {
        this.capacity = capacity;
        this.refillRate = refillRate;
        this.refillScheduler = Executors.newScheduledThreadPool(1);

        // Refill tokens every 100ms
        refillScheduler.scheduleAtFixedRate(
            this::refillAllBuckets,
            100, 100, TimeUnit.MILLISECONDS
        );
    }

    @Override
    public RateLimitResult allowRequest(String clientId) {
        TokenBucket bucket = buckets.computeIfAbsent(
            clientId, k → new TokenBucket(capacity));

        return bucket.tryConsume() ?
            RateLimitResult.allowed(bucket.getTokens()) :
            RateLimitResult.blocked(calculateRetryAfter());
    }

    private void refillAllBuckets() {
        int tokensToAdd = refillRate / 10; // Per 100ms interval
        for (TokenBucket bucket : buckets.values()) {
            bucket.refill(Math.min(tokensToAdd, capacity - bucket.getTokens()));
        }
    }

    private long calculateRetryAfter() {
        return 1000 / refillRate; // ms until next token
    }
}

TokenBucket Inner Class

private static class TokenBucket {
    private int tokens;

    TokenBucket(int initialTokens) {
        this.tokens = initialTokens;
    }

    synchronized boolean tryConsume() {
        if (tokens > 0) { tokens--; return true; }
        return false;
    }

    synchronized void refill(int count) { tokens += count; }

    synchronized int getTokens() { return tokens; }
}

06 — Sliding Window Algorithm

Analogy: Keep a log of every request timestamp. Count how many fall within the last N seconds. If under limit, allow. The window slides forward continuously with time — no boundary effects. 
graph TD
    A["Incoming Request at time=now"] --> B["Get timestamp queue for client"]
    B --> C["Remove timestamps older than now - windowMs"]
    C --> D{"queue.size < maxRequests?"}
    D -->|Yes| E["Add now to queue
ALLOW request"]
    D -->|No| F["retryAfter = oldest + window - now
BLOCK request"]

How It Works

  • Store timestamp of every request per client in a queue
  • On each request, evict expired timestamps (older than now - windowMillis)
  • If queue size < maxRequests: add timestamp, return ALLOWED
  • If full: compute retryAfter from oldest timestamp + window - now
  • No boundary burst — window slides continuously
  • Tradeoff: Higher memory (stores all timestamps in window)

Timeline Example

Limit: 5 requests per 10 seconds

t= 0s:  Request 1-5 → ALLOWED (timestamps: [0,0,0,0,0])
t= 5s:  Request 6   → BLOCKED (5 in window [0-10s])
t=10.1s: Request 7  → ALLOWED (oldest t=0 expired, window [0.1-10.1s])
         timestamps: [0,0,0,0,10.1] → evict old → [10.1]

SlidingWindowRateLimiter.java

public class SlidingWindowRateLimiter implements RateLimiter {

    private final ConcurrentHashMap<String, Queue<Long>> requestTimestamps
        = new ConcurrentHashMap<>();
    private final int maxRequests;
    private final long windowMillis;

    public SlidingWindowRateLimiter(int maxRequests, long windowMillis) {
        this.maxRequests = maxRequests;
        this.windowMillis = windowMillis;
    }

    @Override
    public RateLimitResult allowRequest(String clientId) {
        long now = System.currentTimeMillis();
        Queue<Long> timestamps = requestTimestamps.computeIfAbsent(
            clientId, k → new ConcurrentLinkedQueue<>());

        synchronized (timestamps) {
            // Evict expired timestamps
            while (!timestamps.isEmpty()
                   && timestamps.peek() < now - windowMillis) {
                timestamps.poll();
            }

            if (timestamps.size() < maxRequests) {
                timestamps.offer(now);
                return RateLimitResult.allowed(
                    maxRequests - timestamps.size());
            } else {
                long oldestTimestamp = timestamps.peek();
                long retryAfter = oldestTimestamp + windowMillis - now;
                return RateLimitResult.blocked(retryAfter);
            }
        }
    }

    @Override
    public void resetLimit(String clientId) {
        Queue<Long> ts = requestTimestamps.get(clientId);
        if (ts != null) ts.clear();
    }

    @Override
    public int getRemainingRequests(String clientId) {
        long now = System.currentTimeMillis();
        Queue<Long> ts = requestTimestamps.get(clientId);
        if (ts == null) return maxRequests;
        synchronized (ts) {
            while (!ts.isEmpty() && ts.peek() < now - windowMillis)
                ts.poll();
            return maxRequests - ts.size();
        }
    }
}

07 — Fixed Window Algorithm

Simplest approach: Divide time into fixed intervals. Count requests per interval. Reset counter at interval boundary. Vulnerable to 2x burst at window edges. 
graph TD
    A["Request arrives at time=now"] --> B["currentWindow = floor(now / windowSize)"]
    B --> C["count = incrementCounter(clientId:window)"]
    C --> D{"count <= maxRequests?"}
    D -->|Yes| E["ALLOW request"]
    D -->|No| F["BLOCK request"]

Boundary Burst Problem

Window 1: [0-60s]     Window 2: [60-120s]
Limit: 10 req/min

t=59s:  10 requests → OK (window 1 full)
t=60s:  10 requests → OK (new window!)
→ 20 requests in 1 second! (burst at boundary)

Mitigation: Use Sliding Window Counter (hybrid) — weighted average of current and previous window counts.

Fixed Window Pseudocode (Redis-friendly)

public RateLimitResult allowRequest(String clientId) {
    long currentWindow = System.currentTimeMillis() / windowSizeMs;
    String key = clientId + ":" + currentWindow;

    int count = redis.incr(key);            // Atomic increment
    if (count == 1)
        redis.expire(key, windowSizeMs / 1000); // TTL for cleanup

    return count <= maxRequests ?
        RateLimitResult.allowed(maxRequests - count) :
        RateLimitResult.blocked(timeUntilNextWindow());
}

Sliding Window Counter (Hybrid Fix)

// Weighted count eliminates boundary burst
double weight = 1.0 - (elapsedInCurrentWindow / windowSize);
double effectiveCount = (previousWindowCount * weight) + currentWindowCount;

if (effectiveCount < maxRequests) allow();
else block();

// Example: 60s window, 30s into current window
// previous=8, current=4, weight=0.5
// effective = 8*0.5 + 4 = 8 (under limit of 10)

08 — Leaky Bucket Algorithm

Analogy: A bucket with a hole at the bottom. Requests pour in from the top. They "leak" out at a constant rate. If the bucket overflows, requests are dropped. Output is always smooth. 
graph TD
    A["Requests pour in (variable rate)"] --> B["Leaky Bucket Queue"]
    B -->|"constant leak rate"| C["Backend (smooth output)"]
    A -->|"bucket full"| D["DROP / 429"]

Token Bucket vs Leaky Bucket

AspectToken BucketLeaky Bucket
ControlsInput rate (how fast you can send)Output rate (how fast processing happens)
BurstsAllows bursts up to capacityNo bursts — output is always constant
AnalogySpending from a refilling walletWater leaking from a bucket at fixed rate
Use CaseAPI throttling (allows occasional bursts)Network traffic shaping (smooth output)
Excess requestsQueued until token availableDropped when bucket full

Leaky Bucket Pseudocode

public class LeakyBucketRateLimiter implements RateLimiter {
    private final BlockingQueue<Runnable> bucket;  // bounded queue
    private final int capacity;
    private final long leakIntervalMs;   // ms between leaks

    public RateLimitResult allowRequest(String clientId) {
        if (bucket.size() < capacity) {
            bucket.offer(request);
            return RateLimitResult.allowed(capacity - bucket.size());
        }
        return RateLimitResult.blocked(leakIntervalMs);
    }

    // Background thread leaks at constant rate
    private void leak() {
        while (true) {
            Runnable request = bucket.poll(leakIntervalMs, MILLISECONDS);
            if (request != null) request.run();  // process at fixed rate
        }
    }
}

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap buckets, requestTimestamps Safe concurrent reads/writes for per-client state lookup
synchronized TokenBucket.tryConsume(), refill(), getTokens() Fine-grained lock per bucket — only same-client ops serialized
synchronized(timestamps) SlidingWindow.allowRequest() Per-client queue lock — evict + count + add must be atomic
computeIfAbsent() Both implementations Atomic get-or-create for new clients, no double-init race
ConcurrentLinkedQueue SlidingWindow timestamp storage Lock-free queue for concurrent timestamp insertion
ScheduledExecutorService Token refill background task Decouples refill from request handling, runs every 100ms
AtomicInteger Fixed Window counter (Redis INCR) Lock-free atomic increment for request counting
ReentrantLock Alternative to synchronized Trylock with timeout, fairness policy, interruptible
"I use fine-grained locking per client, not a global lock. Two different clients can be rate-limited concurrently. Only concurrent operations on the same client are serialized."

Why synchronized vs ReentrantLock?

// synchronized: simpler, sufficient for TokenBucket
synchronized boolean tryConsume() {
    if (tokens > 0) { tokens--; return true; }
    return false;
}

// ReentrantLock: when you need tryLock with timeout
private final ReentrantLock lock = new ReentrantLock();

boolean tryConsume() {
    if (lock.tryLock(10, TimeUnit.MILLISECONDS)) {
        try {
            if (tokens > 0) { tokens--; return true; }
            return false;
        } finally { lock.unlock(); }
    }
    return false; // couldn't acquire lock in time
}

@GetMapping("/api/resource")
public ResponseEntity<?> getResource(HttpServletRequest req) {
    String clientId = extractClientId(req);
    RateLimitResult result = rateLimiter.allowRequest(clientId);

    HttpHeaders headers = new HttpHeaders();
    headers.add("X-RateLimit-Limit", String.valueOf(MAX_REQUESTS));
    headers.add("X-RateLimit-Remaining",
               String.valueOf(result.getRemaining()));

        headers.add("Retry-After",
                   String.valueOf(result.getRetryAfterSeconds()));

            .headers(headers).body("Too Many Requests");
    }
    return ResponseEntity.ok().headers(headers).body(fetchResource());

12 — Complexity Analysis

AlgorithmTime / RequestSpace / ClientBackground WorkRedis-Friendly
Token Bucket O(1) O(1) Refill thread every 100ms Yes (Lua script)
Sliding Window O(k) eviction O(k) timestamps None Yes (Sorted Set)
Fixed Window O(1) O(1) None Yes (INCR + EXPIRE)

k = number of requests currently in the sliding window. In practice, eviction is amortized O(1) since each timestamp is evicted at most once.

Metrics Tracking

public class RateLimiterMetrics {
    private long totalRequests;
    private long blockedRequests;

    public void recordRequest(boolean allowed) {
        totalRequests++;
        if (!allowed) blockedRequests++;
    }

    public double getBlockRate() {
        return totalRequests == 0 ? 0 :
            (double) blockedRequests / totalRequests;
    }
}

13 — Distributed Rate Limiting (Redis)

Why Redis? Shared state across all servers. Atomic operations via Lua scripts. TTL for auto-cleanup. Sub-millisecond latency.

Redis Token Bucket (Lua Script)

-- Atomic Token Bucket in Redis
local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens = tonumber(bucket[1]) or capacity
local last_refill = tonumber(bucket[2]) or now

-- Calculate tokens to add since last refill
local time_passed = math.max(0, now - last_refill)
local tokens_to_add = time_passed * refill_rate / 1000
tokens = math.min(capacity, tokens + tokens_to_add)

if tokens >= 1 then
    tokens = tokens - 1
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, 3600)
    return {1, tokens}   -- allowed, remaining
else
    return {0, 0}        -- blocked
end

Redis Sliding Window (Sorted Set)

def allow_request_sliding(client_id):
    key = f"rate_limit:{client_id}"
    now = time.time_ms()
    window_start = now - window_size_ms

    # Remove expired entries
    redis.zremrangebyscore(key, 0, window_start)

    # Count current requests in window
    count = redis.zcard(key)

    if count < max_requests:
        redis.zadd(key, now, f"{now}:{random_id()}")
        redis.expire(key, window_size_seconds)
        return True
    return False

14 — Interview Follow-up Q&A

How would you implement distributed rate limiting across multiple servers?

Primary approach: Redis centralized store. All servers call Redis for rate limit checks. Use Lua scripts for atomicity (HMGET + HMSET in one round trip). Token Bucket state (tokens, last_refill) stored as a Redis hash with TTL for auto-cleanup.

Alternatives: (1) Sticky sessions — route same client to same server, use local limiter. (2) Gossip protocol — servers share counters periodically, eventually consistent. (3) Sliding window with Redis Sorted Sets (ZADD + ZRANGEBYSCORE).

Tradeoff: Redis adds 1-2ms latency but gives exact global counting. Gossip has 0ms added latency but is eventually consistent.

How do you handle race conditions in rate limiting?

Local: synchronized on per-client state (bucket or timestamp queue). Not a global lock — different clients are processed concurrently. ConcurrentHashMap.computeIfAbsent() prevents double-init race.

Distributed (Redis): Lua scripts execute atomically on the Redis server. All operations (read tokens, calculate refill, decrement, write back) happen in a single atomic operation. No WATCH/MULTI needed.

Worst case: Without synchronization, two threads read tokens=1, both decrement to 0, allowing 2 requests when only 1 should pass (TOCTOU bug).

How do you handle burst traffic with rate limiting?

Token Bucket is designed for bursts. A full bucket with capacity=100 allows 100 requests instantly, then refills at the configured rate. This is ideal for APIs where occasional bursts are acceptable.

If bursts are NOT acceptable: Use Sliding Window (strict counting, no bursts) or Leaky Bucket (smooths output to constant rate). The Strategy pattern lets you swap algorithms based on use case.

Adaptive approach: Monitor system load. When CPU > 80%, reduce effective limits. When load is normal, allow full capacity. Priority queues let critical requests bypass limits.

How would you integrate rate limiting with an API gateway?

Rate limiting at the API gateway layer (Kong, AWS API Gateway, NGINX) is the standard production approach. The gateway sits in front of all microservices and applies limits before requests reach the backend.

NGINX example: limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s; with burst=20 for queueing excess requests.

Multi-layer defense: CDN/Edge (Cloudflare) blocks DDoS, Load Balancer (NGINX) protects backend, API Gateway applies per-client limits, Application layer enforces business-specific limits, Database layer throttles expensive queries.

What happens when the rate limiter (Redis) itself fails?

Graceful degradation strategies:

  • Fail Open: Allow all requests — prevents cascading failure, risks overload
  • Fail Closed: Block all requests — conservative, may cause outage
  • Local Fallback: Fall back to in-memory rate limiter temporarily
  • Circuit Breaker: Detect Redis failures, auto-switch to fallback, auto-recover

Best practice: try { return redisLimiter.allow(); } catch (RedisException e) { return localLimiter.allow(); }

How do you handle clock skew in distributed rate limiting?

Problem: Different servers have different system times. Token refill rates differ. Sliding window boundaries misalign. "Retry-After" values are inconsistent.

Solutions: (1) NTP Synchronization: Keep server clocks in sync (± 10ms). (2) Centralized time: Use redis.time() as canonical clock. (3) Logical clocks: Version numbers instead of timestamps. (4) Redis-based: Let Redis handle all time-dependent logic in Lua scripts using redis.call('TIME').

Token Bucket vs Sliding Window — when to use which?

Token Bucket when: you need burst tolerance, O(1) memory per client, API throttling where occasional spikes are OK (e.g., Stripe, AWS).

Sliding Window when: you need exact counting with no boundary issues, strict rate enforcement, billing/quota systems where accuracy matters more than memory.

Fixed Window when: simplicity is key, Redis-friendly (just INCR), and boundary bursts are acceptable.

Leaky Bucket when: you need smooth constant-rate output, network traffic shaping, or processing queues at a fixed rate.

How do you implement per-tier rate limits (free vs premium)?

Use RateLimitConfig per client tier. Store tier mapping in a config store or database.

// Tier configs
FREE    → RateLimitConfig(10 req/min, TOKEN_BUCKET)
PREMIUM → RateLimitConfig(1000 req/min, TOKEN_BUCKET)
ADMIN   → no limit (bypass rate limiter)

Each tier gets its own RateLimiter instance or the same limiter with different capacity/refill parameters. Use RateLimitRule to define named rules (e.g., "api-free", "api-premium") and resolve by client tier at request time.

How would you rate limit by multiple dimensions (IP + user + endpoint)? 
class MultiDimensionalRateLimiter {
    public boolean allowRequest(Request req) {
        return allowByIP(req.ip)           // 100 req/sec per IP
            && allowByUser(req.userId)     // 1000 req/hour per user
            && allowByEndpoint(req.endpoint); // 10K req/min per endpoint
    }
}

Each dimension uses a separate RateLimiter instance with its own config. A request must pass ALL dimensions. This prevents both DDoS (IP-level) and abuse (user-level) while protecting expensive endpoints.

How would you implement adaptive / dynamic rate limiting?

Adjust limits based on system load:

int baseLimit = 100;
double cpuLoad = systemMetrics.getCPUUsage();

// Reduce limit if system overloaded
int effectiveLimit = cpuLoad > 0.8 ?
    (int)(baseLimit * 0.5) : baseLimit;

Approaches: (1) CPU/memory-based: reduce limits when system stressed. (2) Queue depth: if backend queue grows, throttle harder. (3) Error rate: if 5xx errors spike, reduce load. (4) Priority-based: critical requests (payments) get higher limits than browsing.
