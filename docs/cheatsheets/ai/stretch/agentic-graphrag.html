Agentic RAG & GraphRAG // Stretch

Beyond vector chunk retrieval: knowledge graphs with community summaries (Microsoft GraphRAG), plus agent loops that plan multi-hop retrieval, rewrite queries, and synthesize answers across structured + unstructured sources.

GraphRAG Knowledge Graph Agentic Multi-hop

Dual index: vector chunks for local facts + knowledge graph (entities, relations, community reports) for global themes. Agent orchestrator chooses retrieval mode, iterates tool calls, and synthesizes grounded answers with citations.

01 — What & Why

Problem: Vanilla RAG fails on global questions (“What are the main themes across 10K docs?”) and multi-hop reasoning (“Which vendor contracts affect Project X’s budget?”). GraphRAG builds a graph index; agentic RAG adds planning loops.

Functional

  • Extract entities/relations from corpus at ingest
  • Cluster graph into communities with LLM-generated summaries
  • Route queries to local (entity-centric) vs global (community) search
  • Agent decomposes complex questions into sub-queries and tool calls

Non-Functional

  • Ingest cost: GraphRAG indexing is LLM-heavy (10–100× vanilla RAG)
  • Latency: agent loops add seconds; cap iterations
  • Freshness: graph rebuild on doc change is expensive — incremental updates hard
Interview anchor: GraphRAG = better index structure; Agentic RAG = better query-time strategy. Production systems combine both.

04 — Deep Dive: GraphRAG Pipeline

  1. Chunk documents (same as vanilla RAG)
  2. Extract entities and relationships with LLM prompts
  3. Build graph — nodes=entities, edges=relations, link to source chunks
  4. Detect communities (hierarchical Leiden algorithm)
  5. Generate community reports — LLM summarizes each cluster
  6. Query: local = entity-centric subgraph + chunks; global = map-reduce reports

Graph zoom: Leiden clustering at multiple levels → community reports stored as retrieval units. Global query runs parallel LLM map over top-K reports, then reduce into final answer. Index cost dominated by extraction + report generation LLM calls.

05 — Deep Dive: Agentic RAG

Agent decides which retrieval tool to call and whether context is sufficient (ReAct, Self-RAG, CRAG patterns).

# Self-RAG style decision (conceptual)
steps = [
  "1. Decompose question into sub-queries",
  "2. vector_search(sub_q) OR graph_local(entity)",
  "3. Grade relevance: sufficient / insufficient / ambiguous",
  "4. If insufficient: rewrite query, try global search",
  "5. Synthesize with citations; abstain if still insufficient",
]

Implement with LangGraph conditional edges and max recursion_limit.

Agent zoom: plan node → tool node (vector/graph/SQL) → grader node → loop or generate. Typical 3–6 LLM calls per user question; budget explicitly in product SLA.

06 — Deep Dive: Hybrid with Vector RAG

  • Keep vector index for lexical/semantic fallback
  • Graph for entity disambiguation (“Apple” company vs fruit)
  • RRF merge graph-derived chunks with dense hits — see RAG hybrid section
  • Store entity_id in chunk metadata for filtered search

08 — Production & Ops

Cost

  • GraphRAG index: budget $LLM calls × chunks × extraction passes
  • Agentic query: cap at 5 tool calls; use smaller model for grading
  • Incremental: re-index only changed docs + graph patch (complex)

Quality

  • Human eval on global questions (vanilla RAG baselines often fail here)
  • Entity linking accuracy metrics
  • Faithfulness checks on synthesized community reports

10 — Common Pitfalls

  • GraphRAG for everything: 10× index cost unjustified for simple FAQ
  • Hallucinated graph edges: poison multi-hop reasoning — human audit sample
  • Unbounded agent loops: cost explosion; always set max iterations
  • Stale community reports: doc deleted but report remains
  • Skipping vanilla RAG baseline: compare before adding graph complexity

11 — Interview Q&A

What problem does GraphRAG solve that vector RAG cannot?
Global summarization across a large corpus: vector RAG retrieves top-k similar chunks, which miss corpus-wide themes. GraphRAG pre-computes community summaries via graph clustering, enabling map-reduce answers to “what are the main topics?” questions. Also helps multi-hop entity reasoning by traversing relations explicitly.
Explain local vs global search in GraphRAG.
Local: identify seed entities in the query, expand neighborhood in the knowledge graph, pull linked text chunks — good for specific facts about named entities. Global: retrieve relevant pre-generated community reports, run parallel LLM summarization (map), then consolidate (reduce) — good for thematic questions over entire dataset.
What is agentic RAG vs GraphRAG?
GraphRAG is an indexing + retrieval strategy using knowledge graphs. Agentic RAG is query-time orchestration: an LLM agent plans multiple retrieval steps, chooses tools, and critiques results. They compose: GraphRAG provides graph tools; agent decides when to use local vs global vs vector search.
Why is GraphRAG indexing expensive?
Each chunk needs entity/relation extraction (LLM calls), then community detection, then an LLM summary per community. For 10K documents this can be millions of tokens offline. Amortize over query volume; not suitable for hourly-changing corpora without incremental pipeline investment.
How would you implement multi-hop retrieval without a full graph?
Agentic loop: first retrieval → extract entities from results → second query with entity filters → repeat. Cheaper than full GraphRAG but less systematic. Hybrid: lightweight entity index in metadata + agent planning. Compare quality on golden multi-hop set before committing to graph build.
What graph database would you use?
Neo4j, Amazon Neptune, or in-memory graph for moderate scale. GraphRAG reference often uses parquet + custom index rather than live graph DB for read-heavy QA. Choose graph DB when you need transactional updates, traversals in product features, or existing KG investment.
How do you prevent agentic RAG runaway costs?
Hard cap on tool calls (5–8), recursion limit in LangGraph, cheaper model for grading steps, semantic cache on sub-queries, timeout per turn, user-visible “still searching” only after 2 hops. Log token usage per agent step.
Entity extraction errors — mitigation?
Canonical entity registry (link to CRM IDs), human-in-loop review sample, confidence thresholds, dedup via embedding similarity, constrain extraction schema, cross-check with structured data sources. Bad nodes propagate through graph — quality gate at ingest is critical.
When is vanilla RAG still the right answer?
Single-domain FAQ, <5K docs, mostly local factual queries, tight budget, fast-changing content. GraphRAG shines when stakeholders ask global analytic questions over large static corpora (research, legal discovery, compliance archives).
Compare GraphRAG to fine-tuning on the corpus.
Fine-tune embeds knowledge in weights (stale, no citations, expensive to update). GraphRAG keeps knowledge external with provenance. Fine-tune for style/format; GraphRAG/RAG for factual grounding. Neither replaces the other.
Design agentic GraphRAG for legal contract analysis.
Ingest: chunk by clause, extract parties/obligations/dates → KG. Community reports by contract type. Query agent: classify local (party-specific) vs global (portfolio risk themes). Tools: vector search, graph traverse, SQL on metadata. Citations to clause IDs. HITL for high-stakes answers. Eval: attorney-labeled golden set.
SDE3: draw the whiteboard architecture.
Dual index (vector + graph with community reports), query router, optional LangGraph agent with graded retrieval loop, synthesis with citations, observability on each hop. Call out index cost vs query cost trade-off and when not to use graph. Link to RAG baseline and search HLD for scale analogies.
