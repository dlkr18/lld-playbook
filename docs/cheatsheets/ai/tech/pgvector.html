pgvector // AI Tech

PostgreSQL extension for vector similarity search: store embeddings alongside relational data, HNSW/IVFFlat indexes, SQL WHERE filters, JOINs, and ACID transactions. The default when you already run Postgres. Compare with Pinecone, Weaviate, and Chroma in the Vector DB overview.

Postgres HNSW SQL ACID JOIN

pgvector in Postgres: vector(1536) column, HNSW index, SQL WHERE tenant filter, JOIN with users/orders in one query. Sweet spot: 100K–10M vectors per instance with read replicas for search QPS.

01 — What & Why

pgvector adds a vector column type and distance operators to PostgreSQL. Instead of syncing embeddings to a separate vector SaaS, you store vectors in the same database as users, documents, permissions, and orders — with full SQL, transactions, and JOINs.

Functional requirements

  • Store fixed-dimension embeddings in vector(n) columns
  • Search by cosine, L2, or inner product distance
  • Filter with standard SQL WHERE (tenant, ACL, date)
  • JOIN embeddings with relational tables in one query
  • Index with HNSW or IVFFlat for sub-linear ANN search

Non-functional requirements

  • Latency: p99 < 50 ms for top-k=20 at 1–5M vectors with HNSW + tuned ef_search
  • Consistency: ACID upserts — vector + metadata in same transaction
  • Scale: ~1–10M vectors per instance (RAM-bound HNSW); read replicas for QPS
  • Ops: Standard Postgres backup, replication, PgBouncer — team already knows this
Prerequisite: HNSW theory and embedding model metric choice — use vector_cosine_ops for OpenAI-style cosine embeddings.

01b — API / Interface Surface

Setup

CREATE EXTENSION vector;

CREATE TABLE document_chunks (
  id          BIGSERIAL PRIMARY KEY,
  doc_id      UUID NOT NULL,
  tenant_id   TEXT NOT NULL,
  chunk_idx   INT NOT NULL,
  content     TEXT NOT NULL,
  embedding   vector(1536) NOT NULL,
  metadata    JSONB DEFAULT '{}',
  created_at  TIMESTAMPTZ DEFAULT now(),
  UNIQUE (doc_id, chunk_idx)
);

CREATE INDEX idx_chunks_tenant ON document_chunks (tenant_id);
CREATE INDEX idx_chunks_hnsw ON document_chunks
  USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 128);

Query (retrieve)

SET hnsw.ef_search = 100;

SELECT id, content, metadata,
       embedding <=> $1::vector AS distance
FROM document_chunks
WHERE tenant_id = $2
  AND metadata->>'doc_type' IN ('policy', 'faq')
ORDER BY embedding <=> $1::vector
LIMIT 20;

Upsert (idempotent ingest)

INSERT INTO document_chunks (doc_id, tenant_id, chunk_idx, content, embedding, metadata)
VALUES ($1, $2, $3, $4, $5::vector, $6)
ON CONFLICT (doc_id, chunk_idx) DO UPDATE SET
  content = EXCLUDED.content,
  embedding = EXCLUDED.embedding,
  metadata = EXCLUDED.metadata;

03 — Architecture

Ingest worker
Chunk → embed batch → INSERT/ON CONFLICT into chunks table in transactions.
Primary + replicas
Writes to primary; vector search reads from replica(s) to isolate OLTP from ANN QPS.
Query path
Embed query → SQL WHERE tenant + JOIN ACL → ORDER BY distance LIMIT k.
Rerank
Top-20 from pgvector → cross-encoder → top-5 to LLM.
  App ──write──► Postgres Primary (chunks + users + orders)
  App ──search──► Postgres Read Replica(s)
                     ├── B-tree: tenant_id, doc_id
                     └── HNSW: embedding vector_cosine_ops

pgvector shines when the retrieval query needs relational context — e.g. JOIN user_doc_access to enforce ACL in the same query as vector search.

04 — Deep Dive: HNSW Index

HNSW is the default ANN index for pgvector in production (PostgreSQL 16+). See HNSW overview for graph traversal theory.

ParameterScopeStarting point
mIndex build (CREATE INDEX)16 (32 if RAM allows, better recall)
ef_constructionIndex build quality128–256
ef_searchPer-query (SET or transaction)40–100; increase until recall@10 ≥ 0.92

Build without blocking writes

CREATE INDEX CONCURRENTLY idx_chunks_hnsw ON document_chunks
  USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 128);
  • Monitor index size vs shared_buffers — HNSW should fit in RAM for best latency
  • After bulk delete, run VACUUM; consider REINDEX if bloat severe
  • Compare recall vs brute-force on 1K sample before trusting defaults

HNSW tuning: m and ef_construction at build time; ef_search at query time via SET hnsw.ef_search = 100. Profile recall@k vs p99 before production cutover.

05 — Deep Dive: SQL Patterns

ACL JOIN pattern

SELECT c.id, c.content, c.embedding <=> $1 AS distance
FROM document_chunks c
JOIN user_doc_access a ON a.doc_id = c.doc_id
WHERE a.user_id = $2
  AND c.tenant_id = $3
ORDER BY c.embedding <=> $1
LIMIT 20;

Partition by tenant (large scale)

CREATE TABLE document_chunks (
  ...
) PARTITION BY LIST (tenant_id);

CREATE TABLE chunks_acme PARTITION OF document_chunks
  FOR VALUES IN ('acme');

Distance threshold (quality gate)

SELECT * FROM (
  SELECT id, content, embedding <=> $1 AS distance
  FROM document_chunks WHERE tenant_id = $2
  ORDER BY embedding <=> $1 LIMIT 20
) sub WHERE distance < 0.35;  -- cosine distance cutoff
Interview tip: SQL WHERE on tenant_id uses B-tree; combined with HNSW index scan — this is pgvector’s pre-filter advantage over post-filter in app code. Same lesson as metadata filtering.

SQL-native retrieval: WHERE tenant filter + JOIN ACL table + ORDER BY cosine distance. One round trip — no sync to external vector store.

06 — IVFFlat

IVFFlat partitions vectors into lists clusters. At query time, probe nearest centroids — lower RAM than HNSW at large N, but needs tuning. See IVF overview.

CREATE INDEX idx_chunks_ivf ON document_chunks
  USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

-- After index build, required:
ANALYZE document_chunks;

SET ivfflat.probes = 10;  -- increase for better recall
When IVFFlatWhen HNSW
RAM constrained, N > 10MDefault for RAG < 10M
Can train on representative sampleWant best out-of-box recall
Accept probes tuning burdenWant simpler query tuning (ef_search only)

07 — Hybrid with Full-Text Search

pgvector does not fuse BM25 + vector natively like Weaviate. Combine Postgres full-text search with vector retrieval:

ApproachHow
Dual query + RRFts_rank FTS top-50 + vector top-50 → reciprocal rank fusion in app

Weighted SQLNormalize ts_rank and distance; linear combine (fragile)
-- FTS leg
SELECT id, ts_rank(search_vector, plainto_tsquery('english', $1)) AS rank
FROM document_chunks
WHERE search_vector @@ plainto_tsquery('english', $1)
ORDER BY rank DESC LIMIT 50;

11 — Common Pitfalls

  • Wrong ops class: L2 index with cosine embeddings — ranking broken.
  • No ANN index: sequential scan timeout at 100K+ rows.
  • ef_search too low: recall cliffs on tail queries — always golden-eval.
  • Huge content in vector table: bloats heap; store pointer, fetch text separately.
  • Search on primary under load: ANN QPS competes with OLTP — use replicas.
  • IVFFlat without ANALYZE: poor cluster stats — bad recall.
  • Mixing embed models in one column after partial migration.

12 — Interview Q&A

Why pgvector over Pinecone?
Single Postgres for relational + vector + ACID. JOIN embeddings with user permissions in one query. Team already knows Postgres ops and backup. Pick Pinecone when no PG ops appetite and scale > 10M with many tenants.
HNSW vs IVFFlat in pgvector?
HNSW: default for recall/latency at <10M vectors; tune ef_search. IVFFlat: lower RAM at large N; tune lists + probes; needs ANALYZE. See HNSW and IVF sections.
How filter by tenant in pgvector?
WHERE tenant_id = $1 before ORDER BY embedding <=> $query. B-tree on tenant_id combines with HNSW index scan — true pre-filter, not post-filter in app.
How tune ef_search?
Start 40–100. Increase until recall@10 on golden set ≥ 0.92. Log setting per request. Watch p99 latency — ef_search 200 may double query time vs 64.
Max vectors per Postgres instance?
Roughly 1–10M at 1536-dim HNSW depending on RAM (index ~4–8 bytes/dim × M factor × N). Beyond: partition by tenant, read replicas, or migrate hot tenants to Pinecone.
Embedding dimension mismatch — what happens?
INSERT fails or query errors on type mismatch. Model change requires new column/table, backfill, CONCURRENTLY index, blue/green cutover.
Hybrid search with pgvector?
Dual FTS + vector queries; RRF in app. ParadeDB for unified BM25+vector. Native hybrid DX is why teams pick Weaviate instead — trade SQL JOINs vs hybrid convenience.
Migration path to add pgvector to existing Postgres?
CREATE EXTENSION; ADD COLUMN embedding vector(n); async backfill job; CREATE INDEX CONCURRENTLY; feature-flag read path; golden eval before full cutover.
pgvector vs Pinecone on cost?
pgvector = Postgres infra you already pay — cheaper at moderate scale, more ops time. Pinecone = per read/storage unit — less ops, higher $ at hyperscale QPS.
Backup and DR?
Standard Postgres pg_dump, WAL archiving, replica promotion. Vectors included like any column — no special vector backup tool needed.
Concurrent upsert pattern?
ON CONFLICT DO UPDATE in batched transactions; connection pool; autovacuum tuning after bulk loads; avoid long locks on primary during peak.
SDE3: pgvector RAG for 5M chunks with ACL?
Partition by tenant; HNSW per partition or shared table with tenant_id B-tree; read replicas for search; ingest worker pool with ON CONFLICT; JOIN user_doc_access; hybrid FTS column + RRF; reranker service; weekly golden eval; graduation path to Pinecone for hot shards if p99 degrades. Link RAG end-to-end.
