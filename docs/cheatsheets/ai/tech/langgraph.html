LangGraph // AI Tech

Low-level orchestration for stateful, multi-step LLM agents: explicit graphs with shared state, cycles, checkpointing, and human-in-the-loop interrupts. Built on LangChain Runnable primitives but replaces opaque chains with a controllable state machine you can draw on a whiteboard.

StateGraph Checkpointing HITL Cyclic Agents Tool Loop

Canonical ReAct-style graph: START → agent → (tools?) → tools → agent → END. Cycles are first-class (LangChain LCEL chains are DAGs). compile(checkpointer=...) persists state per thread_id for resume, HITL, and time-travel debugging.

01 — What & Why

Problem: Production agents are not one-shot prompts. They loop (plan → act → observe), branch on tool results, pause for human approval, and must survive process restarts without losing conversation context.

Functional

  • Define a graph of steps (nodes) and transitions (edges)
  • Carry typed shared state across every step (messages, counters, retrieved docs)
  • Support cycles: agent calls tools until done
  • Checkpoint after each super-step; resume by thread_id
  • Interrupt before/after nodes for human review

Non-Functional

  • Deterministic control flow — you own routing, not the model
  • Observability — LangSmith traces each node; state diffs per step
  • Durability — Postgres/Sqlite checkpointers for long-running workflows
  • Latency — each node is usually one LLM or tool call; budget cycles explicitly
Position in stack: LangChain = components (models, retrievers, tools). LangGraph = orchestration runtime on top. For multi-agent crew patterns see CrewAI; for system-level agent design see Agent Architectures.

01b — API / Interface Surface

from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import Command

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

builder = StateGraph(AgentState)
builder.add_node("agent", call_model)
builder.add_node("tools", run_tools)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", should_continue, ["tools", END])
builder.add_edge("tools", "agent")

graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["tools"],  # HITL gate
)

config = {"configurable": {"thread_id": "user-42"}}
result = graph.invoke({"messages": [("user", "What's the weather?")]}, config)

04 — StateGraph Deep Dive

State is a TypedDict (or Pydantic model). Each key is a channel. Updates are merged via reducers:

  • messages — append with dedup via add_messages
  • Scalars — last write wins unless you define a custom reducer
  • Nested dicts — shallow merge per key; design flat state for clarity

State channels and reducers: nodes emit partial updates; add_messages appends AIMessage/ToolMessage without replacing the full history. Each super-step snapshots state to the checkpointer.

from langchain_core.messages import AIMessage, HumanMessage

def call_model(state: AgentState) -> dict:
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

def should_continue(state: AgentState) -> str:
    last = state["messages"][-1]
    if isinstance(last, AIMessage) and last.tool_calls:
        return "tools"
    return END

05 — Nodes & Edges

Nodes

  • Sync or async function; can call LangChain runnables inside
  • Subgraphs: compile a child graph and add as a node for modular teams
  • Send API for map-reduce fan-out (parallel tool calls to branches)

Edges

  • Normal: add_edge("tools", "agent") — always follow
  • Conditional: router returns "tools" or END
  • Entry: START is virtual; only one edge from START
Interview tip: Draw nodes as boxes with side effects (LLM $, tool latency). Draw edges as decisions you control in code, not hope the model picks.

06 — Checkpointing

Checkpointers implement the persistence contract. Production typically uses PostgresSaver or AsyncPostgresSaver; dev uses MemorySaver.

from langgraph.checkpoint.postgres import PostgresSaver

DB_URI = "postgresql://user:pass@localhost:5432/langgraph"
with PostgresSaver.from_conn_string(DB_URI) as checkpointer:
    checkpointer.setup()  # migrations once
    graph = builder.compile(checkpointer=checkpointer)

# Same thread_id always resumes latest checkpoint
config = {"configurable": {"thread_id": "support-ticket-9912"}}
graph.invoke({"messages": [HumanMessage("Follow up")]}, config)
  • Time travel: list checkpoints via get_state_history for debugging bad tool calls
  • Forking: branch from an older checkpoint to A/B prompt changes
  • TTL: your job to prune old threads (GDPR, cost)

07 — Human-in-the-Loop

Declare interrupts at compile time. When the graph hits an interrupt boundary, execution stops and state is persisted. Your API layer surfaces pending tool calls; humans approve, edit args, or reject.

graph = builder.compile(
    checkpointer=checkpointer,
    interrupt_before=["tools"],
)

# First run stops before tools execute
out = graph.invoke(input, config)
# out["__interrupt__"] or inspect pending tool_calls in state

# Human approves in UI
graph.invoke(Command(resume={"action": "approve"}), config)

# Or reject and inject feedback
graph.invoke(
    Command(resume={"action": "reject", "feedback": "Use read-only query"}),
    config,
)

HITL path: agent proposes tool call → interrupt_before tools → checkpoint + UI review → Command(resume=...) → execute or skip. Required for finance, healthcare, and privileged API access.

08 — Cyclic Flows

LangChain Expression Language chains are DAGs — no cycles. Agent loops need cycles: agent → tools → agent. LangGraph makes cycles explicit, so you can cap iterations:

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    step_count: int

def call_model(state: AgentState) -> dict:
    if state.get("step_count", 0) >= 10:
        return {"messages": [AIMessage(content="Max steps reached.")]}
    return {
        "messages": [llm.invoke(state["messages"])],
        "step_count": state.get("step_count", 0) + 1,
    }
  • Always set max iterations in state or conditional edge
  • Detect tool-call loops (same tool + same args) in router
  • Stream stream_mode="updates" so UI shows progress per cycle

10 — Simple Agent Graph (Full Example)

"""Minimal ReAct agent: web search tool + loop until no tool_calls."""
from typing import Annotated, Literal, TypedDict

from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage
from langchain_core.tools import tool
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.checkpoint.memory import MemorySaver
from langgraph.prebuilt import ToolNode

@tool
def search(query: str) -> str:
    """Search the web (stub)."""
    return f"Results for: {query}"

tools = [search]
llm = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools)

class State(TypedDict):
    messages: Annotated[list, add_messages]

def agent(state: State) -> dict:
    return {"messages": [llm.invoke(state["messages"])]}

def route(state: State) -> Literal["tools", "__end__"]:
    last = state["messages"][-1]
    if getattr(last, "tool_calls", None):
        return "tools"
    return END

builder = StateGraph(State)
builder.add_node("agent", agent)
builder.add_node("tools", ToolNode(tools))
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route, ["tools", END])
builder.add_edge("tools", "agent")

graph = builder.compile(checkpointer=MemorySaver())

if __name__ == "__main__":
    cfg = {"configurable": {"thread_id": "demo-1"}}
    out = graph.invoke(
        {"messages": [HumanMessage("Latest LangGraph release notes?")]},
        cfg,
    )
    print(out["messages"][-1].content)

Run with langgraph dev (Studio) for visual debugging, or deploy via langgraph up on LangGraph Platform for managed checkpoints and scaling.

13 — Common Pitfalls

  • No checkpointer with HITL — resume fails across requests
  • Replacing messages instead of appending — forgetting add_messages reducer
  • Unbounded tool loop — no step_count or router guard
  • Huge state blobs — stuffing full PDFs in state; store refs, retrieve in node
  • Wrong thread_id — users see each other’s conversations
  • Interrupt without API contract — frontend must handle pending tool UI
  • Mixing sync/async — use ainvoke / async nodes consistently in FastAPI

14 — Interview Q&A

What problem does LangGraph solve that LangChain chains do not?
LangChain LCEL is great for DAG-shaped pipelines (retrieve → format → generate). Agents need cycles (model → tools → model), durable state, and pause/resume. LangGraph gives an explicit state machine with checkpointing and interrupts; chains would require recursive hacks and external session stores.
Explain StateGraph in one minute.
You define a typed state schema (channels + reducers), register nodes as functions that return partial updates, wire edges (fixed or conditional), then compile() to a Runnable. Each invocation runs super-steps: execute node, merge state, checkpoint, route. Cycles are just edges back to earlier nodes.
What is a reducer and why does messages need one?
Reducers define how concurrent or sequential updates combine into channel values. For chat history, nodes return {"messages": [new_msg]}; add_messages appends and deduplicates by id instead of overwriting the list. Without it, you lose conversation context after the first node.
How does checkpointing work with thread_id?
The checkpointer (Memory, Postgres, etc.) saves state snapshots keyed by config["configurable"]["thread_id"]. Every super-step writes a checkpoint. A new message on the same thread loads the latest snapshot and continues. Enables multi-turn chat, crash recovery, and HITL resume without resending full history from the client.
How do you implement human approval before a dangerous tool runs?
compile(interrupt_before=["tools"], checkpointer=...). First invoke stops with pending tool_calls in state. Your API shows them to a human; on approve, call invoke(Command(resume=...), same config). Postgres checkpointer ensures the pause survives server restarts. For edits, use update_state to patch tool args before resume.
How do you prevent infinite agent loops?
Three layers: (1) step_count in state with max in the agent node or router returning END; (2) detect repeated identical tool_calls; (3) product-level timeout on invoke. Always stream so users see stuck loops. LangGraph doesn’t auto-stop—you encode policy in the graph.
LangGraph vs CrewAI for multi-agent systems?
LangGraph is low-level: you design every node, edge, and state field—maximum control, more code. CrewAI is higher-level role/task/delegation abstractions for research-style crews. Many teams use CrewAI for prototyping and LangGraph (or subgraphs) for production control, HITL, and custom routing. See CrewAI and Agent Architectures.
What is a conditional edge vs a normal edge?
Normal edge always goes A → B. Conditional edge runs a router function on current state that returns the name of the next node (or END). Classic pattern: after agent, if last message has tool_calls go to tools, else END. Keeps routing logic in testable Python, not in the LLM prompt.
Can nodes run in parallel?
Yes, within a super-step LangGraph can schedule multiple nodes if the graph topology allows (e.g. fan-out with Send for map-reduce over documents). The default ReAct loop is sequential. Use parallel branches for independent retrievers or multi-tool calls with merge reducers.
How would you deploy this in production?
Stateless FastAPI/HTTP workers calling graph.ainvoke; Postgres checkpointer; thread_id from auth session; LangSmith tracing; rate limits per tenant; HITL UI for privileged tools. LangGraph Platform offers hosted persistence and horizontal scale; self-host is the same graph with your DB and queue for async long jobs.
How does streaming work for a chat UI?
Use graph.stream(input, config, stream_mode="updates") (or messages for token streams from LLM nodes). Each chunk is a partial state update per node—map to SSE/WebSocket events. Don’t call invoke if you need incremental UI; stream until interrupt or END.
