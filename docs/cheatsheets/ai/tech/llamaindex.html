LlamaIndex // AI Tech

Data framework for LLM apps: document loaders, node parsers, indices (vector, tree, keyword), query engines, and agents. Strongest when ingestion + retrieval complexity dominates.

IndexQuery EngineNodesAgentsIngestion

LlamaIndex pipeline: load documents → parse to Nodes → build VectorStoreIndex → QueryEngine (retriever + synthesizer) → response with source nodes.

01 — What & Why

LlamaIndex focuses on getting data into LLM-ready form: loaders (PDF, Notion, SQL), chunking into Nodes, building Indices, and QueryEngines that retrieve + synthesize answers with citations.

01b — API Surface

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader
from llama_index.core import Settings
from llama_index.embeddings.openai import OpenAIEmbedding

Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
docs = SimpleDirectoryReader("./data").load_data()
index = VectorStoreIndex.from_documents(docs)
engine = index.as_query_engine(similarity_top_k=5)
response = engine.query("What is our refund policy?")
print(response.source_nodes)  # citations

03 — Architecture

Sources -> Reader -> NodeParser -> Embeddings -> VectorStore
Query -> Retriever (top_k) -> ResponseSynthesizer -> Answer + Sources

04 — Deep Dive: Indices

VectorStoreIndex backed by Pinecone/pgvector/Chroma with docstore for full text

Also: SummaryIndex, TreeIndex, KnowledgeGraphIndex. Production RAG almost always VectorStoreIndex + external vector DB.

05 — Deep Dive: Query Engine

Query engine: retriever top_k=20, postprocessor, compact/refine synthesizer modes

Response modes: compact (single prompt), refine (iterative), tree_summarize. Add SimilarityPostprocessor cutoff.

06 — Deep Dive: Ingestion

LlamaHub loaders: 100+ connectors. IngestionPipeline for parallel extract/embed. Link Document Ingestion.

07 — Deep Dive: Agents

LlamaIndex agents use tools + query engines. For production control prefer LangGraph with LlamaIndex retriever as tool.

08 — vs LangChain

LlamaIndex: indexing, query engines, citation-first RAG. LangChain: general chains, broader ecosystem. Can combine: LI retriever in LC chain.

09 — Production & Ops

  • Persist index to Pinecone not local disk
  • Version embed model in index name
  • Eval with golden queries on source_nodes recall

12 — Interview Q&A

Node vs Document?
Document = loaded file. Node = parsed chunk(s) with metadata — what gets embedded and indexed.
VectorStoreIndex vs loading vector DB directly?
VectorStoreIndex wraps vector store + docstore + index struct. Use from_vector_store() to attach existing Pinecone/pgvector.
How get citations?
response.source_nodes lists retrieved nodes with scores and metadata. Pass to UI; log for eval.
Query engine vs chat engine?
QueryEngine: single Q&A. ChatEngine: multi-turn with memory. Production chat often custom history + query engine per turn.
LlamaIndex vs LangChain for RAG?
LI stronger out-of-box ingestion/query/citations. LC more flexible orchestration. Many use LI for index + LC/LangGraph for agent layer.
Persist index?
storage_context.persist() to disk; or write directly to managed vector DB for prod.
Custom node parser?
SentenceSplitter with chunk_size/overlap; SemanticSplitter for embedding-based splits. Match chunking eval.
Hybrid search in LI?
QueryFusionRetriever combines multiple retrievers; BM25 via Elasticsearch retriever + vector.
Agents in LlamaIndex?
OpenAIAgent with query_engine_tools. Prefer LangGraph when need HITL/checkpointing.
Embedding model change?
Full re-index; new collection name; never mix vectors.
Production scale?
IngestionPipeline async; batch embed; external vector DB; query engine as stateless microservice.
SDE3: LI for enterprise doc RAG?
IngestionPipeline: Notion+SharePoint loaders → semantic chunk → Pinecone namespace per tenant → QueryEngine with metadata filters → reranker → LangGraph agent wrapper; golden eval on source node recall.
