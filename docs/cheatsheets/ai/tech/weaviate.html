Weaviate // AI Tech

Open-source vector database with native hybrid search (BM25 + vector), GraphQL/REST APIs, modular embedders, and multi-tenancy. Self-host on K8s or use Weaviate Cloud. Compare with Pinecone, pgvector, and Chroma in the Vector DB overview.

Hybrid GraphQL BM25 Multi-tenant HNSW

Weaviate: GraphQL/REST API → hybrid query (alpha blends BM25 + vector) → HNSW + inverted index → filtered objects with properties and _additional scores. Self-host cluster or managed cloud.

01 — What & Why

Weaviate is an open-source vector database written in Go. Its killer feature is hybrid retrieval in one API call — BM25 keyword scores fused with dense vector similarity via an alpha parameter. It also supports GraphQL for flexible queries, pluggable vectorizer modules, and first-class multi-tenancy.

Functional requirements

  • Schema classes with typed properties and vectorizer config
  • Vector search via nearText / nearVector with HNSW index
  • Keyword search via BM25 inverted index on text properties
  • Hybrid search combining both with tunable alpha
  • Multi-tenancy with isolated tenant shards per class
  • Modules for server-side embedding, reranking, and generation

Non-functional requirements

  • Latency: p99 hybrid query 30–80 ms at millions of objects (cluster-dependent)
  • Availability: 3+ node replication for self-host HA
  • Ops: K8s Helm chart, Prometheus metrics, S3 backups
  • Scale: 1M–100M+ objects; horizontal sharding in enterprise tiers
When Weaviate wins: hybrid BM25+vector is a core requirement and you can self-host or use Weaviate Cloud. For zero-ops pure vector at hyperscale, Pinecone may be simpler.

  "class": "Document",
  "vectorizer": "text2vec-openai",
  "moduleConfig": {
    "text2vec-openai": { "model": "text-embedding-3-small" }
  },
  "properties": [
    { "name": "content", "dataType": ["text"] },
    { "name": "tenantId", "dataType": ["text"] },
    { "name": "docType", "dataType": ["text"] }
  ],
  "multiTenancyConfig": { "enabled": true }
}

Hybrid GraphQL query

{
  Get {
    Document(
      tenant: "acme"
      hybrid: {
        query: "refund policy for enterprise customers"
        alpha: 0.5
      }
      limit: 20
      where: {
        path: ["docType"]
        operator: Equal
        valueText: "policy"
      }
    ) {
      content
      tenantId
      _additional { score distance id }
    }
  }
}

                         ├── HNSW (vector)

04 — Deep Dive: Hybrid Search

Hybrid search is Weaviate’s standout feature. One query runs BM25 on text properties and vector similarity in parallel, then fuses scores with alpha:

  • alpha = 0 — BM25 only (exact keyword queries, SKUs, error codes)
  • alpha = 1 — vector only (semantic paraphrase)
  • alpha = 0.5 — balanced default starting point
Query typeSuggested alphaWhy
SKU / part number lookup0.2–0.3Exact token match dominates
General Q&A0.5–0.6Balance semantics + keywords
Conceptual / paraphrase0.7–0.9Embedding similarity dominates

Tuning workflow

  1. Build golden query set (50–200 questions) with labeled relevant doc IDs
  2. Grid search alpha ∈ {0.3, 0.5, 0.7} — measure recall@5 and MRR
  3. Optionally route alpha by query classifier (SKU detector → low alpha)
  4. Add cross-encoder reranker on top-20 (+30–80 ms)
See also: Hybrid Search system sheet and vector DB hybrid overview.

Hybrid alpha: 0 = BM25 only, 1 = vector only, 0.5 = default fusion. Grid-search alpha on your golden set — do not ship alpha=0.5 blindly.

GraphQL Get query: hybrid block + where filter on tenantId + _additional { score distance } for logging and reranker input.

11 — Common Pitfalls

  • Self-host without K8s expertise: replication and upgrades bite — use Weaviate Cloud or Pinecone.
  • Wrong alpha for query type: SKU queries need low alpha; semantic-only needs high alpha.
  • Module embed model drift: ingest module ≠ query embed path — re-index required.
  • Missing tenant in query: multi-tenancy enabled but query omits tenant → error or empty.
  • No reranker after hybrid: fusion alone often insufficient for top-5 quality.
  • GraphQL over-fetch: returning huge text properties on every query — latency and cost.

12 — Interview Q&A

What is Weaviate’s killer feature vs Pinecone?
Native hybrid search in one GraphQL call — BM25 + vector with alpha tuning. Avoids dual-system (Elasticsearch + vector DB) fusion code. Pinecone wins on pure managed simplicity and hyperscale zero-ops.
Explain alpha in hybrid search.
Weight between dense vector (1) and BM25 sparse (0). Default 0.5; grid-search {0.3, 0.5, 0.7} on golden recall@5. Route low alpha for SKU/error-code queries; high alpha for paraphrase questions.
Self-host vs Weaviate Cloud?
Self-host: K8s, Helm, replication, you patch and backup. Cloud: managed scaling and backups — ops model closer to Pinecone. Pick self-host when data residency or cost at scale requires it.
GraphQL vs REST — when to use which?
GraphQL: flexible property selection, hybrid + where + _additional in one query — best for app retrieval layer. REST: simpler CRUD scripts and batch import. Most production RAG apps use GraphQL for query, REST for ingest.
How does multi-tenancy work?
Enable on class schema; each tenant gets isolated shard. Import and query must specify tenant name. Strong SaaS isolation — analogous to Pinecone namespaces.
Server-side vectorization — pros and cons?
Pros: simpler client — send raw text. Cons: external API dependency at ingest/query, harder to version-control embed model, latency. Production often prefers BYO vectors (vectorizer: none) for control — same as Pinecone upsert pattern.
Weaviate vs pgvector for a Postgres shop?
pgvector: SQL JOINs, ACID, one database. Weaviate: native hybrid, GraphQL, no JOINs. If hybrid is core and you can run K8s, Weaviate; if relational + vector in one TX matters, pgvector + ParadeDB FTS.
How integrate reranking?
Hybrid top-20 → reranker-cohere module or external cross-encoder (rerankers sheet) → top-5 to LLM. Always measure rerank lift on golden set.
Index rebuild on embedding model change?
New class or tenant namespace; re-import full corpus with new vectorizer; blue/green cutover after recall eval. Never mix models in one class.
What metrics to monitor?
Query p50/p99 latency, object count per tenant, hybrid alpha used (A/B log), module API errors, recall@5 weekly golden eval, disk/RAM per node.
When NOT to pick Weaviate?
No K8s ops and no Weaviate Cloud budget → Pinecone. Need SQL JOINs → pgvector. Local notebook → Chroma. Pure vector with no keyword leg → any engine works; Weaviate shines when hybrid matters.
