Pinecone // AI Tech

Fully managed vector database: serverless and pod indexes, namespaces, metadata filtering, sparse-dense hybrid, and upsert/query at scale without operating HNSW yourself.

ServerlessNamespaceHNSWMetadataManaged

Pinecone architecture: client upsert/query → serverless index (S3-backed segments + compute cache) → HNSW ANN → filtered top-k. p99 query ~20–40 ms at millions of vectors.

01 — What & Why

Pinecone removes vector ops: no HNSW tuning, sharding, or RAM planning. Best when you want managed SLAs and multi-tenant SaaS RAG without a platform team. Compare pgvector if already on Postgres.

01b — API Surface

from pinecone import Pinecone
pc = Pinecone(api_key="...")
index = pc.Index("rag-prod")

index.upsert(vectors=[
    {"id": "c1", "values": emb, "metadata": {"tenant": "acme", "page": 3}}
], namespace="acme_v1")

results = index.query(
    vector=query_emb, top_k=20, namespace="acme_v1",
    filter={"tenant": {"$eq": "acme"}},
    include_metadata=True,
)

03 — Architecture

Serverless: vector data on object storage; query nodes load segments + HNSW into cache. Namespaces isolate tenants. See Vector DB overview for HNSW theory.

04 — Deep Dive: Serverless

Serverless index: S3 segments, query compute cache, auto scale read units

Create with cloud and region. Monitor read units and storage billing.

05 — Deep Dive: Metadata Filters

Metadata pre-filter DSL: $eq, $in, $gte on tenant_id and doc_type before ANN

Index metadata fields you filter on. Pre-filter avoids post-filter starvation.

06 — Hybrid Vectors

Sparse-dense vectors for keyword+semantic in one query when enabled. Else dual-query in app layer + RRF. See Hybrid Search.

07 — Index Lifecycle

12 — Interview Q&A

Why Pinecone over self-hosted?
Zero HNSW ops, managed scaling, namespaces for multi-tenant. Trade-off: cost and less SQL join flexibility vs pgvector.
Serverless vs pod?
Serverless: usage-based, auto scale, recommended for new projects. Pod: dedicated hardware, predictable perf for extreme QPS.
How multi-tenant?
Namespace per tenant OR shared index + metadata filter tenant_id. Namespace = stronger isolation.
Metadata filter syntax?
Mongo-style: {"field": {"$eq": "val", "$in": [...], "$gte": n}}. Only indexed metadata fields filter efficiently.
Upsert best practices?
Batch 100–500; idempotent IDs (doc:chunk); include source text pointer in metadata; retry 429 with backoff.
Query latency tuning?
top_k only what you need (20 before rerank); warm index with probe queries after deploy; co-locate region with app.
Embedding model change?
New index or namespace; full re-embed; cutover after golden eval.
Hybrid in Pinecone?
Sparse-dense when available; else BM25 in ES + vector in Pinecone + RRF in app.
Cost drivers?
Storage (vectors×dim), read units (queries), write units (upserts). Compress metadata; prune stale namespaces.
Delete GDPR user?
Delete by metadata user_id filter; verify docstore also purged.
Pinecone vs Weaviate?
Pinecone simpler managed. Weaviate better if native hybrid GraphQL and self-host control needed.
SDE3: 10K tenants 100M vectors?
Serverless index; namespace per tenant OR metadata tenant_id + strict pre-filter; ingest queue; embed service; reranker; per-tenant recall dashboards; blue/green re-index pipeline.
