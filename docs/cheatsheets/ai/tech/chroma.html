Chroma // AI Tech

Embedded open-source vector database for local dev and prototypes: pip install chromadb, in-process or client/server mode, collections, metadata filters, and LangChain/LlamaIndex integration. Graduate to pgvector or Pinecone before production SLAs. See Vector DB overview and compare with Weaviate.

Embedded Collections Dev LangChain Local

Chroma: Python client → Collection (ids + embeddings + documents + metadatas) → HNSW in-process or Chroma server → query top-k. Ideal for notebooks and CI; graduate before paying-customer SLAs.

01 — What & Why

Chroma is the fastest path to a working vector store in Python — zero cloud signup, zero infra. It wraps HNSW indexing, metadata filtering, and optional auto-embedding behind a simple Collection API. It is a development and prototyping tool, not a production multi-tenant vector platform.

Functional requirements

  • Collections — named stores of ids, embeddings, documents, metadata
  • add / query / update / delete by id with metadata filters
  • Embedding functions — auto-embed on add and query, or bring your own vectors
  • Persistence — disk-backed PersistentClient or ephemeral in-memory
  • Client/server — optional Docker server for team-shared dev DB

Non-functional requirements (dev context)

  • Latency: fine for local dev; not tuned for production p99 SLOs
  • Scale: comfortable to ~500K vectors on a laptop; slows beyond
  • HA: none — single process; no replication
  • Multi-tenant: weak — separate collections, not enterprise isolation
Graduation signal: when you need SLAs, >500K vectors, multi-tenant ACL, or hybrid BM25 — move to pgvector, Pinecone, or Weaviate.

01b — API / Interface Surface

Basic usage (persistent + OpenAI embed)

import chromadb
from chromadb.utils import embedding_functions

ef = embedding_functions.OpenAIEmbeddingFunction(
    api_key=os.environ["OPENAI_API_KEY"],
    model_name="text-embedding-3-small",
)
client = chromadb.PersistentClient(path="./chroma_db")
col = client.get_or_create_collection(
    name="docs_v1",
    embedding_function=ef,
    metadata={"embed_model": "text-embedding-3-small"},
)

col.add(
    ids=["handbook:0", "handbook:1"],
    documents=["Refund within 30 days.", "Enterprise SLA is 99.9%."],
    metadatas=[{"type": "policy", "page": 1}, {"type": "policy", "page": 2}],
)

results = col.query(
    query_texts=["What is the refund policy?"],
    n_results=5,
    where={"type": "policy"},
    include=["documents", "metadatas", "distances"],
)

Bring your own vectors (prod model parity)

col.add(
    ids=["chunk-42"],
    embeddings=[precomputed_vector],  # skip EmbeddingFunction
    documents=["..."],
    metadatas=[{"source": "handbook.pdf"}],
)

col.query(
    query_embeddings=[query_vector],
    n_results=20,
    where={"source": {"$eq": "handbook.pdf"}},
)

03 — Architecture

Client modes
PersistentClient (SQLite + HNSW on disk), EphemeralClient (RAM), HttpClient (remote server).
Collection
Logical index: ids, embeddings, documents, metadatas kept in sync.
Embed path
Optional EmbeddingFunction on add/query — or BYO vectors for control.
Query
HNSW ANN + where metadata filter → top-k with distances.
  Notebook/App ──► chromadb Client
                      └── Collection "docs_v1"
                            ├── SQLite (metadata, docs)
                            └── HNSW index (vectors)
  Optional: chroma run ──► HttpClient (team shared dev)

Chroma runs in-process with your Python app — no separate cluster. That simplicity is the feature for dev; it is the limitation for production.

04 — Deep Dive: Collections

A Collection is the unit of organization — analogous to a Pinecone namespace or a pgvector table.

FieldTypeNotes
idsstring[]Unique per collection; use doc_id:chunk_idx
embeddingsfloat[][]Optional if EmbeddingFunction set
documentsstring[]Raw text; returned in query for RAG context
metadatasdict[]Filterable JSON; keep values str/int/float/bool

Metadata filter examples

col.query(
    query_texts=["return policy"],
    n_results=10,
    where={"type": {"$eq": "policy"}, "page": {"$gte": 1}},
)

col.query(
    query_texts=["error"],
    where_document={"$contains": "ECONNREFUSED"},
)
  • col.update() — change document/metadata by id
  • col.delete(ids=[...]) or where= filter delete
  • col.get() — fetch by id without similarity search

Collection schema: parallel arrays for ids, embeddings, documents, metadatas. Query with where metadata filter and n_results top-k.

05 — Deep Dive: Embedding Functions

Chroma can embed for you via EmbeddingFunction plugins — convenient for demos, dangerous if the demo model differs from production.

FunctionWhen
OpenAIEmbeddingFunctionQuick OpenAI embed in notebook
SentenceTransformerEmbeddingFunctionLocal model, no API key
ONNXMiniLM_L6_V2Default local fallback in some versions
BYO embeddings=Production parity testing — recommended
# Production-parity pattern: embed in your pipeline, not in Chroma
from openai import OpenAI
client = OpenAI()

def embed(texts):
    r = client.embeddings.create(model="text-embedding-3-small", input=texts)
    return [d.embedding for d in r.data]

vectors = embed(["Refund within 30 days."])
col.add(ids=["1"], embeddings=vectors, documents=["..."], metadatas=[{}])
Gotcha: default local embed model ≠ OpenAI production model — retrieval eval in Chroma will not predict production recall. See Embedding Models.

Embedding path: document text → EmbeddingFunction (e.g. OpenAI) → stored vector. Query must use the same function or BYO query vector.

07 — Integrations

LangChain

from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings

vectorstore = Chroma(
    collection_name="docs",
    embedding_function=OpenAIEmbeddings(model="text-embedding-3-small"),
    persist_directory="./chroma_db",
)
retriever = vectorstore.as_retriever(search_kwargs={"k": 20})

LlamaIndex

from llama_index.vector_stores.chroma import ChromaVectorStore
import chromadb

db = chromadb.PersistentClient(path="./chroma_db")
chroma_collection = db.get_or_create_collection("docs")
vector_store = ChromaVectorStore(chroma_collection=chroma_collection)

Keep the same chunk IDs and metadata schema when swapping backend to Pinecone or pgvector — only the VectorStore config changes.

08 — Graduation Path

Design your retriever interface so Chroma is swappable from day one:

  1. Stable chunk IDs: {doc_id}:{chunk_idx} across all backends
  2. Metadata schema: tenant_id, source, page, doc_type, embed_model
  3. Embed in pipeline: not inside Chroma — same code paths for prod
  4. Config-driven backend: env var VECTOR_BACKEND=chroma|pgvector|pinecone
  5. Export script: dump ids, embeddings, metadatas → upsert to prod store
Graduate toWhen
pgvectorOn Postgres, <10M vectors, need JOINs/ACID
PineconeMulti-tenant SaaS, managed scale, zero vector ops
WeaviateHybrid BM25+vector is core requirement

09 — Production & Ops

Chroma is not recommended for production multi-tenant SLAs. If forced into a single-tenant internal tool:

  • Persistent volume with regular directory backup
  • Single-tenant only; no shared server with untrusted users
  • Monitor collection size; re-index if >500K vectors feel slow
  • Contract test that retriever swaps to pgvector/Pinecone in prod config
Interview answer: “Chroma in prod with paying users” is a red flag — cite graduation path to pgvector or Pinecone.

11 — Common Pitfalls

  • Chroma in prod with paying users: no HA, weak multi-tenant isolation.
  • Default embed model ≠ prod model: demo recall misleads.
  • Shared HttpClient server without auth: any team member can read/write all collections.
  • No graduation plan: prototype becomes “production” under load.
  • Storing only documents, no metadata: cannot filter by tenant or doc type.
  • Expecting hybrid search: where_document is contains, not BM25 — use Weaviate.
  • One collection for staging + prod: test data pollutes retrieval.

12 — Interview Q&A

What is Chroma best for?
Local dev, notebooks, CI tests, hackathons — vector search in ~5 lines without cloud signup or Postgres setup. Fastest path to prove a RAG idea.
Persistent vs ephemeral client?
PersistentClient: saves to disk at path; survives restarts. EphemeralClient: in-memory; lost on exit — ideal for unit tests. Pick persistent for multi-day notebook work.
How filter metadata in Chroma?
query(where={"field": "value"}) or operators $eq, $gte, $in. where_document={"$contains": "text"} for body search — not full BM25.
Chroma vs FAISS?
Chroma: metadata, persistence, CRUD API, LangChain integration. FAISS: raw index library — faster ANN but DIY everything (no metadata API, no persistence story). Chroma is FAISS + ergonomics for app devs.
Is Chroma production-ready?
Generally no for multi-tenant SaaS — no managed HA, limited scale, weak isolation. OK for single-tenant internal tools with backup. Production = pgvector or Pinecone.
Embedding function gotcha?
Chroma can embed for you — ensure same model as production pipeline. Prefer BYO vectors from your embed service for eval parity. See Embedding Models.
LangChain integration pattern?
from langchain_chroma import Chroma with persist_directory and embedding_function. Retriever as_retriever(k=20) drops into LCEL chain. Swap to PineconePGVector by changing VectorStore class only.
How migrate from Chroma to Pinecone?
Export: col.get(include=["embeddings","metadatas","documents"]). Transform to Pinecone upsert format. Write to namespace on Pinecone index. Switch retriever config; run golden eval before cutover.
Client/server mode — when?
Docker chroma run for team-shared dev database. Not production HA — single server, no replication. Compare Pinecone for managed multi-node.
Collection sizing limits?
Fine to ~500K vectors on a dev machine; query latency and disk grow noticeably beyond. Signal to graduate to pgvector or managed store.
How backup Chroma?
Copy the persist directory while client is closed, or use collection export via get(). No built-in replication or point-in-time recovery.
SDE3: when would you allow Chroma in an architecture review?
Only dev/staging with synthetic data; retriever interface abstracted; contract tests prove swap to pgvector in prod config; explicit graduation criteria (>500K vectors, multi-tenant, hybrid, SLA). Never for paying-customer retrieval without managed backend. Link vector DB comparison.
