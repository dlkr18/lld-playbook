Model Serving (vLLM / Ollama) // AI Tech

Serve open-weight LLMs locally: vLLM for production throughput (PagedAttention, continuous batching), Ollama for dev simplicity, OpenAI-compatible APIs, GPU sizing, and quantization (AWQ/GPTQ).

vLLMOllamaPagedAttentionQuantizationOpenAI API

vLLM serving: OpenAI-compatible /v1/chat/completions → scheduler → PagedAttention KV cache → GPU workers. 8× Llama-3-8B throughput vs naive HF generate on same hardware.

01 — What & Why

Self-host Llama, Mistral, Qwen when data residency, cost at scale, or latency to cloud matters. vLLM for prod GPU serving; Ollama for laptop dev and quick pulls.

03 — Architecture

Load balancer → vLLM replicas (1 GPU each or TP/PP multi-GPU) → model weights + KV cache RAM. Queue depth metric for autoscale.

04 — Deep Dive: vLLM

vLLM PagedAttention: KV cache in fixed blocks, continuous batching scheduler

Flags: --max-model-len, --gpu-memory-utilization 0.9, --enable-prefix-caching for RAG system prompts.

05 — Deep Dive: Ollama

Ollama local stack: pull model, Modelfile, REST API, single-user GPU

Modelfile for custom system prompt and parameters. Not multi-tenant prod server.

06 — Quantization

AWQ 4-bit fits 70B on 2× A100 80GB. Benchmark perplexity/task accuracy vs FP16 on your eval before prod.

07 — GPU Sizing

8B FP16 ~16GB; 8B AWQ ~6GB; 70B AWQ ~40GB+. tensor-parallel-size splits layers across GPUs.

12 — Interview Q&A

vLLM vs HuggingFace generate?
vLLM: PagedAttention + continuous batching = 5–20× throughput. HF generate: simple, low concurrency.
What is PagedAttention?
KV cache stored in non-contiguous blocks like OS virtual memory — less fragmentation, more concurrent requests per GPU.
Ollama use cases?
Local dev, demos, single-user apps. pull/run models in one command; OpenAI-ish API at :11434.
OpenAI-compatible why?
Point LangChain/OpenAI SDK at vLLM with base_url — no code rewrite for self-host migration.
Quantization trade-off?
AWQ/GPTQ 4-bit: ~1–3% quality drop, 4× memory savings. Eval on your tasks before prod.
GPU for Llama-3-8B?
1× A10/L4 24GB FP16; or 1× T4 16GB with AWQ. Headroom for KV cache scales with concurrent users.
Tensor parallel?
Split model across GPUs when single GPU VRAM insufficient — tensor-parallel-size 2 for 70B.
Prefix caching?
vLLM caches identical prompt prefixes (system+RAG context) — big win for shared system prompts.
When still use OpenAI?
Frontier reasoning, zero ops, burst scale without GPU cap. Hybrid: router sends easy to local, hard to GPT-4.
Embed serving?
Use text-embeddings-inference (TEI) not vLLM for embed models — specialized batching.
Autoscale?
K8s HPA on GPU queue depth and latency; cold start slow — min replicas > 0.
SDE3: private LLM platform?
vLLM on K8s GPU pool; model registry; OpenAI gateway routing; quant pipeline; A/B model versions; cost per token internal chargeback; fallback to cloud API.
