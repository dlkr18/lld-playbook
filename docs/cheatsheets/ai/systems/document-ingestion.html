Document Ingestion Pipeline // Systems

Parse, OCR, dedupe, chunk, embed, and index at scale. Ingest is 80% of RAG quality — garbage in, hallucination out.

KafkaOCRCDCDedupNear-line

Document ingestion pipeline: sources (S3, Confluence, Slack) → Kafka queue → parser workers → dedup by content_hash → chunker → embed GPU pool → dual-write indexes + metadata DB.

01 — What & Why

RAG quality ceiling is set at ingest. Parsing failures, bad OCR, and stale indexes cause retrieval misses before the query path runs.

12 — Interview Q&A

Sync vs async ingest?
Always async. POST returns 202 job_id. User queries poll or webhook. Never block on embedding 1M docs.
How handle doc deletes?
Tombstone in metadata DB. Async purge vector+sparse by doc_id. Verify purge in reconciliation job.
content_hash purpose?
Skip re-chunk/re-embed if document unchanged. Saves GPU on CDC noise.
Parser selection?
Route by mime: PDF→Unstructured/OCR, HTML→trafilatura, markdown→direct, Slack→custom thread parser.
Kafka vs SQS?
Kafka for high-throughput ordered CDC streams. SQS simpler for moderate scale job queue.
Ingest lag SLA?
Near-line: p99 <5 min. Batch: <24h. Alert when lag exceeds 2× baseline.
OCR quality?
Sample 1% human review. Confidence threshold; flag low-confidence pages for manual review.
Multi-tenant ingest?
Partition queue by org_id. Fair scheduling — prevent one tenant saturating embed pool.
Re-embed on model upgrade?
Blue/green index: embed to new index version, swap alias after golden eval. Don't delete old until validated.
DLQ handling?
Parser failures → DLQ with raw doc. Replay after fix. Alert on DLQ depth.
Ingest 1M docs sizing?
10M chunks. Embed ~3 GPU-hours one-time. Kafka 3-node. Parser CPU fleet auto-scaled.
SDE3: ingestion platform?
Pluggable parsers, CDC connectors, content_hash dedup, priority queues, embed GPU autoscaler, quality dashboard, GDPR delete pipeline.
