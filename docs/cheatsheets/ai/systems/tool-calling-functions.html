Tool Calling & Functions // Systems

LLM function calling: JSON schema design, tool registry, execution sandbox, parallel calls, error recovery. The bridge between reasoning and the real world.

JSON SchemaSandboxParallelValidationOpenAI

Tool calling loop: LLM receives tool schemas, emits tool_calls JSON, executor runs in sandbox, results appended to messages, loop until final answer.

01 — What & Why

Tools let LLMs query databases, call APIs, run code. Schema is the contract; executor is the trust boundary.

03 — Architecture

LLM -> tool_calls[] -> Validator -> Executor (sandbox) -> tool results -> LLM loop

04 — Deep Dive: Schema Design

JSON schema fields: name, description, parameters with types and enums

Descriptions are prompts. Use enums for finite choices. Required fields only. Example values in description.

05 — Deep Dive: Tool Registry

Central registry maps name → handler + schema + ACL. Dynamic registration per tenant. Max 20 tools per call — prune by relevance classifier.

06 — Deep Dive: Execution Sandbox

Network egress allowlist. Read-only DB creds. Timeout 30s. No filesystem write except /tmp. Firecracker/gVisor for code execution.

07 — Deep Dive: Error Handling

Return structured errors to LLM: {error, retry_hint}. Max 3 retries per tool. Circuit breaker on downstream failures.

08 — Deep Dive: Parallel Tools

LLM emits parallel tool_calls; executor runs concurrently; results merged

OpenAI parallel tool_calls. asyncio.gather for independent searches. Sequential when tool B needs tool A output.

09 — Production & Ops

  • Log every tool call: name, args, latency, success

  • Rate limit per tool per user

11 — Common Pitfalls

  • 20+ tools confuses model — use retrieval over tools
  • Executing unvalidated args (SQL injection)
  • No timeout on tool execution
  • Returning huge raw JSON to context

12 — Interview Q&A

How design a good tool schema?
Clear name, action-oriented description, typed params with descriptions, enums for choices, minimal required fields, example in description.
tool_choice auto vs required?
auto: model decides. required: forces tool call every turn — use when always need retrieval. none: disable tools.
How prevent SQL injection from tool args?
Parameterized queries only. Validate args against schema. Read-only DB user. Never string-concat SQL from LLM output.
Parallel vs sequential tool calls?
Parallel when independent (search 3 sources). Sequential when dependent (get_user then get_orders).
What return on tool failure?
Structured error JSON with retry_hint. Let LLM recover. Don't throw opaque 500 to model.
How limit tool count?
Dynamic tool retrieval: embed query, select top-5 relevant tools from registry of 100+.
Sandbox requirements for code exec?
No network by default, CPU/memory limits, timeout, ephemeral filesystem, audit log.
Anthropic vs OpenAI tool format?
Both JSON schema; Anthropic tool_use blocks vs OpenAI tool_calls. Normalize in adapter layer.
How test tools?
Unit test handlers. Integration test full loop with mocked LLM. Golden trajectories for agent regressions.
Idempotent tools?
Design GET-like tools idempotent. Write tools need confirmation/HITL and idempotency keys.
Tool result size limits?
Truncate/summarize large results before appending to messages. Store full result externally; pass reference.
SDE3: tool platform for 200 internal APIs?
Registry with OpenAPI import, auto-schema, per-team ACL, sandbox executor, unified tracing, HITL gate on write tools.
