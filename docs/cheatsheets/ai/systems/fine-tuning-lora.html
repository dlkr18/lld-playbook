Fine-Tuning (LoRA / QLoRA) // Systems

When to fine-tune vs RAG vs long context, LoRA/QLoRA mechanics, PEFT parameter budgets, dataset sizing, eval gates before deploy, and HuggingFace / Axolotl training patterns — the production path from base model to served adapter.

LoRA QLoRA PEFT Eval Gate Axolotl

End-to-end fine-tuning pipeline: raw data → JSONL prep → frozen base model + LoRA adapters → eval harness gate → merge or vLLM LoRA slots for production serving.

01 — What & Why

Problem

Base LLMs are generalists. Production apps need consistent tone, domain vocabulary, structured output formats, or task-specific behavior that prompting alone cannot reliably achieve. Full fine-tuning updates all 7B+ parameters — expensive, slow, and risks catastrophic forgetting. LoRA/QLoRA trains tiny adapter matrices (~0.1–1% of params) while keeping the base frozen.

Functional Requirements

  • Adapt a base model to a specific task (classification, extraction, chat style, tool-use format)
  • Train on custom JSONL / chat-format datasets with train/val split
  • Support multiple adapter versions per base model (A/B, per-tenant)
  • Eval gate: no deploy without passing golden-set regression
  • Merge adapter into base weights OR serve dynamically via vLLM LoRA slots

Non-Functional Requirements

  • Cost: QLoRA on 1× A100 for 7B model, ~2–8 hrs for 1K–10K examples
  • Storage: adapter checkpoint ~50–200 MB vs full model ~14 GB (fp16 7B)
  • Latency: merged model = same as base; dynamic LoRA adds ~5–15% overhead
  • Reproducibility: pinned base model hash, dataset version, hyperparams in config YAML
  • Safety: eval for hallucination regression, PII leakage, jailbreak resistance
Position in the stack: Fine-tuning sits after pre-training & SFT and alongside RAG. Most production systems use RAG for knowledge + a small LoRA adapter for style/format. See pretraining-sft-rlhf for where SFT fits in the alignment stack.

01a — Core Concepts

TermDefinitionGotcha
LoRALow-Rank Adaptation: inject trainable rank-r matrices A, B into frozen weight W. Update: W′ = W + BA.Higher rank ≠ always better; r=16–64 is the sweet spot for most tasks.
QLoRALoRA + 4-bit NF4 quantized base weights (via bitsandbytes). Trains adapters on consumer/single-GPU hardware.Quantization adds ~1–3% quality loss vs fp16 LoRA; acceptable for most domain tasks.
PEFTParameter-Efficient Fine-Tuning umbrella: LoRA, Prefix Tuning, IA³, AdaLoRA, DoRA.LoRA dominates production; others are research/niche.
Rank (r)Inner dimension of LoRA matrices. Controls adapter capacity vs param count.r=8 for simple format tasks; r=32–64 for complex domain reasoning.
Target modulesWhich layers get adapters: typically q_proj, k_proj, v_proj, o_proj in attention + MLP.More modules = more params but diminishing returns beyond attention layers.
Adapter mergeBake LoRA weights into base: W_merged = W + BA. Single checkpoint, no runtime overhead.Cannot hot-swap adapters after merge; need separate checkpoints per variant.

01b — API / Interface Surface

HuggingFace PEFT — load & train

from peft import LoraConfig, get_peft_model, PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments, Trainer
import torch

BASE = "meta-llama/Llama-3.1-8B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(BASE)
model = AutoModelForCausalLM.from_pretrained(
    BASE,
    torch_dtype=torch.bfloat16,
    device_map="auto",
    load_in_4bit=True,          # QLoRA
    bnb_4bit_compute_dtype=torch.bfloat16,
)

lora_config = LoraConfig(
    r=16,
    lora_alpha=32,              # scaling = alpha / r
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# trainable params: ~4.2M || all params: 8.0B || trainable%: 0.05%

02 — When to Use / When NOT

Decision flow: knowledge freshness and citation needs → RAG; style/format/vocabulary → LoRA; full-doc reasoning in one shot → long context; production default → RAG + small LoRA.

RAG + LoRA (hybrid)Production default: RAG for knowledge, LoRA for style/format/reasoning templateSimple FAQ bot with static answers (RAG alone suffices)Best ROI for enterprise chatbots

Rule of thumb: Try prompt engineering first → RAG if knowledge gap → LoRA if format/style gap persists → long context only if single-doc reasoning and budget allows. Never fine-tune to inject facts that belong in a vector store.

04 — LoRA & QLoRA Deep Dive

LoRA injects low-rank matrices A (d×r) and B (r×d) alongside frozen weight W. Only BA is trained; effective update W′ = W + (alpha/r)·BA.

Hyperparameter guide

ParamTypical RangeNotes
r (rank)8 – 64Start r=16. Increase if underfitting on val set.
lora_alpha2×r (e.g. 32 for r=16)Scaling factor. alpha/r controls adapter magnitude.
lora_dropout0.05 – 0.1Regularization. Higher if small dataset (<1K).
learning_rate1e-4 – 3e-4LoRA tolerates higher LR than full FT. Use cosine + warmup.
epochs1 – 5Watch val loss — 3 epochs is common; stop early on overfit.
sequence_len2048 – 8192Match expected inference context. Longer = more VRAM.
batch_sizemicro=1–4, accum=4–16Effective batch 16–64. QLoRA 7B fits micro=2 on 24GB GPU.

06 — Dataset Requirements

Size guidelines

Task TypeMin ExamplesRecommendedQuality > Quantity

Complex reasoning / extraction1K–3K5K–20KHuman-reviewed; synthetic data needs validation
Multi-turn chat style500–2K3K–15KMatch production conversation patterns

Data prep checklist

  • Format: JSONL chat messages; identical template as inference (ChatML, Llama-3, etc.)
  • Dedup: MinHash or exact hash on normalized text; remove near-duplicates (>0.9 cosine)
  • PII scrub: regex + NER pass before training; audit for leakage in eval
  • Train/val split: 90/10 or 95/5; stratify by task type if multi-task
  • Balance: equal representation of task categories; oversample rare formats
  • Negative examples: include "I don't know" and refusal patterns for safety
  • Version control: DVC or W&B artifact versioning; never retrain on unversioned data

07 — HuggingFace & Axolotl Training Patterns

HF Trainer loop (minimal)

from datasets import load_dataset
from trl import SFTTrainer, SFTConfig

dataset = load_dataset("json", data_files={"train": "train.jsonl", "val": "val.jsonl"})

training_args = SFTConfig(
    output_dir="./outputs",
    num_train_epochs=3,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=2e-4,
    lr_scheduler_type="cosine",
    warmup_ratio=0.03,
    bf16=True,
    logging_steps=10,
    eval_strategy="steps",
    eval_steps=100,
    save_strategy="steps",
    save_steps=200,
    max_seq_length=4096,
    dataset_text_field="messages",
    packing=True,               # pack short examples for throughput
)

trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset["train"],
    eval_dataset=dataset["val"],
    processing_class=tokenizer,
)
trainer.train()
model.save_pretrained("./outputs/lora-final")

08 — Eval Before Deploy

Never deploy a fine-tuned model without an eval gate. Fine-tuning can silently degrade general capabilities, introduce hallucination patterns, or leak training data.

Eval harness structure

Golden set (50–200 Q&A)
Human-curated expected outputs. Exact match or semantic similarity >0.85. Must pass >90% before deploy.
Regression suite
Run base model + new adapter on same prompts. Block if adapter scores < base on general tasks.
Format compliance
JSON schema validation, regex checks, required field presence. 100% on structured output tasks.
Safety checks
PII leakage probe, jailbreak resistance, toxicity classifier. Compare against pre-train baseline.
LLM-as-judge (optional)
GPT-4 scores quality 1–5 on held-out prompts. Use for subjective tasks; not sole gate.

Eval script sketch

import json
from evaluate import load

def run_golden_eval(model, tokenizer, golden_path: str, threshold: float = 0.90):
    cases = [json.loads(l) for l in open(golden_path)]
    passed = 0
    for case in cases:
        prompt = format_chat(case["messages"][:-1])  # all but expected answer
        expected = case["messages"][-1]["content"]
        output = generate(model, tokenizer, prompt)
        if score_match(output, expected) >= 0.85:
            passed += 1
    accuracy = passed / len(cases)
    assert accuracy >= threshold, f"Eval failed: {accuracy:.2%} < {threshold:.0%}"
    return accuracy

Failure modes

  • Overfitting: great on train, fails golden set — reduce epochs, increase dropout, add data
  • Template mismatch: train on ChatML, serve with Llama-3 template — outputs garbled
  • Catastrophic forgetting: adapter breaks general reasoning — eval regression suite catches this
  • Data leakage: eval examples in train set — strict split + hash dedup across sets
  • Adapter bloat: too many vLLM LoRA slots exhaust GPU memory — LRU eviction policy

11 — Common Pitfalls

  • Fine-tuning facts instead of behavior: if the answer changes weekly, use RAG. LoRA bakes knowledge into weights — stale after data drift.
  • Too few examples with high rank: r=64 on 200 examples = guaranteed overfit. Match rank to dataset size.
  • Skipping eval regression: adapter improves target task but breaks general chat — always compare against base model.
  • Chat template mismatch: #1 silent killer. Train and serve must use identical formatting function.
  • Synthetic data without validation: GPT-generated training data propagates model errors. Human-review at least 10% sample.
  • Learning rate too high: LoRA LR >5e-4 often causes loss spikes and degraded outputs. Start 2e-4.
  • Not versioning datasets: cannot reproduce or debug regressions without pinned data artifacts.
  • Deploying without shadow mode: run new adapter alongside production for 24–48 hrs before full cutover.

12 — Interview Q&A

When would you fine-tune vs RAG vs long context?
RAG when knowledge changes frequently and you need citations — it keeps facts external and updatable. Fine-tune (LoRA) when you need consistent output format, domain vocabulary, or structured extraction that prompting can't reliably achieve. Long context (128K+) when a full document fits in one pass and you need cross-section reasoning without retrieval infra — but at ~$0.30–0.50/query for 128K tokens, it's expensive at scale. Production default: RAG for knowledge + small LoRA for style/format. See the decision tree in section 02.
Explain LoRA mathematically. Why does low rank work?
For a frozen weight matrix W ∈ R^(d×d), LoRA adds a low-rank update: W′ = W + (alpha/r)·BA where B ∈ R^(d×r), A ∈ R^(r×d), r << d. Intuition: task-specific adaptation lives in a low-dimensional subspace — you don't need to update all d² parameters. With r=16 on a 4096-dim layer, you train 2×4096×16 = 131K params vs 16M frozen. The alpha/r scaling lets you decouple rank from learning rate magnitude.
What is QLoRA and when would you use it over fp16 LoRA?
QLoRA = LoRA + 4-bit NormalFloat (NF4) quantized base weights via bitsandbytes. Base model stays frozen in 4-bit; LoRA adapters train in bf16. VRAM drops ~50% (7B model fits in ~12 GB vs ~24 GB). Quality is ~97–99% of fp16 LoRA — acceptable for most domain tasks. Use QLoRA as default on single-GPU setups (A10, 4090, A100). Use fp16 LoRA only for final quality pass or when you have abundant GPU memory and quality is critical.
How do you choose LoRA rank, alpha, and target modules?
Start r=16, alpha=32 (2×r), target q/k/v/o_proj. If val loss plateaus above acceptable quality, increase r to 32 or 64. If overfitting (val loss rises), decrease r or increase dropout to 0.1. Adding MLP layers (gate_proj, up_proj, down_proj) adds ~3× params with diminishing returns — only if attention-only adapters underfit. Never set r > sqrt(dataset_size) as a rough heuristic.
How much training data do you need for LoRA fine-tuning?
Format/style tasks: 200–500 minimum, 500–2K recommended. Domain vocabulary/tone: 500–1K min, 2K–10K recommended. Complex reasoning: 1K–3K min, 5K–20K recommended. Quality beats quantity — 500 human-reviewed examples outperform 5K synthetic ones. Include edge cases, negative examples, and refusal patterns. Always hold out 5–10% for validation.
How do you evaluate a fine-tuned model before deploying?
Four-layer eval gate: (1) Golden set — 50–200 human-curated Q&A, require >90% pass rate. (2) Regression suite — compare adapter vs base model on general tasks; block deploy if adapter regresses. (3) Format compliance — JSON schema validation at 100% for structured tasks. (4) Safety — PII leakage probes, jailbreak resistance. Optional: LLM-as-judge for subjective quality. Integrate as CI gate — no artifact promotion without passing. Never skip regression against the base model.
Should you merge LoRA weights or serve them dynamically?
Merge (W_merged = W + BA) when: single adapter per deployment, want zero runtime overhead, deploying to edge/Ollama. Dynamic LoRA (vLLM slots) when: multi-tenant (different adapter per customer), A/B testing adapter versions, need hot-swap without restart. Dynamic adds ~5–15% latency and ~50–200 MB VRAM per slot. Most SaaS products use dynamic LoRA with 4–8 slots per GPU. See model-serving-vllm for slot configuration.
What is PEFT and how does LoRA fit in?
PEFT (Parameter-Efficient Fine-Tuning) is the HuggingFace library and concept for updating <1% of model params. Methods: LoRA (low-rank matrices), Prefix Tuning (learnable prefix tokens), IA³ (activation rescaling), AdaLoRA (adaptive rank), DoRA (decomposed magnitude+direction). LoRA dominates production due to best quality/efficiency trade-off, native vLLM support, and mature tooling (PEFT, Axolotl, TRL). Use PEFT library's LoraConfig + get_peft_model for all LoRA training.
How does fine-tuning relate to SFT and RLHF?
Pipeline: Pre-training (next-token prediction on web scale) → SFT (instruction tuning on curated pairs — LoRA is the efficient SFT method) → RLHF/DPO (preference alignment). For product teams, LoRA-SFT is 90% of what you need. RLHF/DPO is for alignment beyond task adaptation (helpfulness, safety, tone preferences). See pretraining-sft-rlhf for the full alignment stack. Don't jump to RLHF when LoRA-SFT on 2K examples would solve the problem.
Design a fine-tuning pipeline for an enterprise support chatbot.
Hybrid RAG + LoRA. RAG: ingest support docs into vector DB (see rag-end-to-end). LoRA: fine-tune on 3K historical support conversations for tone, escalation patterns, and JSON ticket extraction format. Train QLoRA r=16 on Llama-3-8B via Axolotl. Eval gate: 150 golden Q&A + regression vs base. Deploy: vLLM with dynamic LoRA slot per product line + RAG retrieval injected into system prompt. Monitor: format compliance rate, CSAT proxy, escalation rate. Retrain monthly on new conversations with same eval gate.
What are the most common fine-tuning failure modes in production?
(1) Training/serving template mismatch — garbled outputs. (2) Overfitting on small data — great train loss, fails golden set. (3) Catastrophic forgetting — task improves but general chat breaks. (4) Fine-tuning facts that should be in RAG — stale knowledge after doc updates. (5) Data leakage — eval examples in training set. (6) Skipping regression eval — silent quality degradation. (7) Learning rate too high — loss spikes. Prevention: identical chat templates, eval gate with regression, RAG for facts, strict data versioning.
