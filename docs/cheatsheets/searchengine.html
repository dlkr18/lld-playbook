01 — Requirements

Functional

  • Index documents — add, update, remove documents with title, content, and metadata
  • Full-text search — multi-keyword queries returning ranked results
  • TF-IDF relevance ranking — documents scored by term frequency × inverse document frequency
  • Fuzzy search — typo tolerance via edit-distance matching
  • Field filtering — restrict search to title, content, tags, or metadata fields
  • Boolean queries — AND / OR semantics on multi-term queries
  • Real-time indexing — documents searchable immediately after indexing
  • Pagination — offset + limit on search results

Non-Functional

  • Thread-safe — concurrent indexing and searching without corruption
  • Low latency — search response < 50ms for typical queries
  • Extensible ranking — plug in BM25, PageRank, or custom scorers via Strategy
  • Pluggable tokenizer — swap simple split for stemming, stop-word removal, n-grams
  • Observable events — notify listeners on index changes (doc added/removed)
  • Memory efficient — shared posting lists, no redundant copies

03 — Design Patterns

Four core patterns — each solves a specific search engine concern.

Strategy Pattern

RankingStrategy (TF-IDF, BM25) and TokenizerStrategy (simple, stemming, n-gram). Injected at construction, swappable at runtime. New ranking model = new class.

Observer Pattern

IndexObserver notified on every document add/update/remove. Analytics, cache invalidation, replication — all decoupled from the core index.

Template Method

AbstractTokenizer defines the pipeline: lowercase → split → filter stop-words → normalize. Subclasses override individual steps (e.g., add stemming).

Builder Pattern

SearchQuery.Builder constructs complex queries fluently: .terms("java").field(TITLE).operator(AND).limit(10).build()

04 — Inverted Index

Why inverted index? Forward index (doc → terms) requires scanning every document. Inverted index (term → docs) gives O(1) lookup per term. This is the heart of every search engine from Lucene to Google.

Building the Inverted Index

public void indexDocument(Document doc) {
    documents.put(doc.getId(), doc);

    List<String> terms = tokenizer.tokenize(doc.getContent());
    for (String term : terms) {
        invertedIndex
            .computeIfAbsent(term, k -> ConcurrentHashMap.newKeySet())
            .add(doc.getId());
    }

    // Store term frequencies for TF-IDF
    Map<String, Integer> freqs = computeTermFrequencies(terms);
    termFrequencies.put(doc.getId(), freqs);

    notifyObservers(IndexEvent.DOCUMENT_ADDED, doc);
}

Removing from the Index

public void removeDocument(String docId) {
    Document doc = documents.remove(docId);
    if (doc == null) throw new DocumentNotFoundException(docId);

    // Remove docId from every posting list
    invertedIndex.forEach((term, docIds) -> {
        docIds.remove(docId);
        if (docIds.isEmpty()) invertedIndex.remove(term);
    });

    termFrequencies.remove(docId);
    notifyObservers(IndexEvent.DOCUMENT_REMOVED, doc);
}

05 — TF-IDF Ranking

TF-IDF = Term Frequency × Inverse Document Frequency. Common words like "the" get low IDF; rare domain terms get high IDF. TF rewards documents that mention the query term often.
TF(t, d)
count(t in d) / |d|
How often term t appears in doc d, normalized by doc length
IDF(t)
log(N / df(t))
N = total docs, df(t) = docs containing term t
Score
Σ TF × IDF
Sum over all query terms

TF-IDF Implementation

public class TfIdfRankingStrategy implements RankingStrategy {

    public double score(Document doc, List<String> queryTerms,
                        SearchIndex index) {
        double score = 0.0;
        int totalDocs = index.getDocumentCount();

        for (String term : queryTerms) {
            double tf  = termFrequency(doc, term, index);
            int    df  = index.getDocumentFrequency(term);
            double idf = Math.log((double) totalDocs / (1 + df));
            score += tf * idf;
        }
        return score;
    }

    private double termFrequency(Document doc, String term,
                                  SearchIndex index) {
        Map<String, Integer> freqs = index.getTermFrequencies(doc.getId());
        int count = freqs.getOrDefault(term, 0);
        int docLen = freqs.values().stream().mapToInt(i -> i).sum();
        return docLen == 0 ? 0 : (double) count / docLen;
    }
}

BM25 — Production Alternative

BM25 improves on TF-IDF with saturation (diminishing returns for high TF) and document-length normalization. Same RankingStrategy interface — just a new class.

// BM25 scoring formula:
// score(t,d) = IDF(t) * (tf * (k1+1)) / (tf + k1 * (1 - b + b * |d|/avgDl))
// k1 = 1.2 (term frequency saturation)
// b  = 0.75 (document length normalization)

06 — Indexing Pipeline

Pipeline pattern — raw text passes through a chain of transformations before entering the index. Each step is independent and testable.

Tokenizer Strategy

interface Tokenizer {
    List<String> tokenize(String text);
}

class SimpleTokenizer implements Tokenizer {
    private static final Set<String> STOP_WORDS =
        Set.of("the", "a", "an", "is", "in", "of", "and", "to");

    public List<String> tokenize(String text) {
        return Arrays.stream(text.toLowerCase().split("\\W+"))
            .filter(t -> !t.isEmpty() && !STOP_WORDS.contains(t))
            .collect(Collectors.toList());
    }
}

class StemmingTokenizer implements Tokenizer {
    private final Tokenizer delegate;  // wraps SimpleTokenizer

    public List<String> tokenize(String text) {
        return delegate.tokenize(text).stream()
            .map(this::stem)
            .collect(Collectors.toList());
    }

    private String stem(String word) {
        // Porter stemmer: "running" → "run", "searches" → "search"
        if (word.endsWith("ing")) return word.substring(0, word.length() - 3);
        if (word.endsWith("es"))  return word.substring(0, word.length() - 2);
        if (word.endsWith("s"))   return word.substring(0, word.length() - 1);
        return word;
    }
}

AND vs OR Semantics

OperatorSet OperationBehavior
OR (default) Union of posting lists Doc matches if it contains any query term. More results, broader.
AND Intersection of posting lists Doc matches only if it contains all query terms. Fewer results, precise. 
private Set<String> findCandidates(SearchQuery query) {
    List<Set<String>> postingLists = query.getTerms().stream()
        .map(term -> invertedIndex.getOrDefault(term, Collections.emptySet()))
        .collect(Collectors.toList());

    if (query.getOperator() == QueryOperator.AND) {
        // Intersection: start with smallest set for efficiency
        return postingLists.stream()
            .sorted(Comparator.comparingInt(Set::size))
            .reduce((a, b) -> { a.retainAll(b); return a; })
            .orElse(Collections.emptySet());
    } else {
        // Union: merge all posting lists
        Set<String> result = new HashSet<>();
        postingLists.forEach(result::addAll);
        return result;
    }
}

Fuzzy Matching (Edit Distance)

private Set<String> fuzzyExpand(String term, int maxDist) {
    return invertedIndex.keySet().stream()
        .filter(indexTerm ->
            levenshteinDistance(term, indexTerm) <= maxDist)
        .collect(Collectors.toSet());
}

private int levenshteinDistance(String a, String b) {
    int[][] dp = new int[a.length() + 1][b.length() + 1];
    for (int i = 0; i <= a.length(); i++) dp[i][0] = i;
    for (int j = 0; j <= b.length(); j++) dp[0][j] = j;
    for (int i = 1; i <= a.length(); i++)
        for (int j = 1; j <= b.length(); j++)
            dp[i][j] = a.charAt(i-1) == b.charAt(j-1)
                ? dp[i-1][j-1]
                : 1 + Math.min(dp[i-1][j-1],
                      Math.min(dp[i-1][j], dp[i][j-1]));
    return dp[a.length()][b.length()];
}

08 — Key Algorithms

Core Search Algorithm

Three-phase: Candidate Retrieval → Scoring → Ranking + Pagination

public List<SearchResult> search(SearchQuery query) {
    // Phase 1: Tokenize query terms
    List<String> queryTerms = tokenizer.tokenize(query.getRawQuery());
    if (query.isFuzzy()) {
        queryTerms = queryTerms.stream()
            .flatMap(t -> fuzzyExpand(t, 2).stream())
            .distinct().collect(Collectors.toList());
    }

    // Phase 2: Retrieve candidate documents
    Set<String> candidateIds = findCandidates(query, queryTerms);

    // Phase 3: Score, rank, paginate
    return candidateIds.stream()
        .map(id -> {
            Document doc = documents.get(id);
            double score = rankingStrategy.score(doc, queryTerms, this);
            return new SearchResult(doc, score, queryTerms);
        })
        .sorted(Comparator.comparingDouble(
            SearchResult::getScore).reversed())
        .skip(query.getOffset())
        .limit(query.getLimit())
        .collect(Collectors.toList());
}

Complexity Analysis

OperationTimeSpaceNotes
Index document O(T) O(T) T = number of terms in doc
Search (per term) O(1) lookup O(D) D = docs containing term (posting list size)
Ranking (TF-IDF) O(C × Q) O(C) C = candidates, Q = query terms
Fuzzy expand O(V × L²) O(L²) V = vocabulary size, L = max word length
Remove document O(V) O(1) Scan all posting lists
AND intersection O(min posting list) O(min) Start with smallest set

09 — Thread Safety

MechanismWhereWhy
ConcurrentHashMap invertedIndex, documents, termFrequencies Safe concurrent reads/writes on index structures
ConcurrentHashMap.newKeySet() Each posting list (Set<String>) Thread-safe set backed by CHM — concurrent add/remove on posting lists
computeIfAbsent() Adding terms to inverted index Atomic check-and-create — no lost posting lists under contention
ReadWriteLock Document update (remove + re-add) Write lock for updates, read lock for searches — readers don't block each other
volatile rankingStrategy, tokenizer references Cross-thread visibility when swapping strategies at runtime
CopyOnWriteArrayList IndexObserver list Safe iteration while adding/removing observers
AtomicLong Document ID generation Lock-free monotonic IDs
"Searches are lock-free reads on ConcurrentHashMap. Indexing uses computeIfAbsent for atomic posting-list creation. Only document updates (remove + re-index) take a write lock to prevent readers seeing a partially-updated state."

Index More Documents

Same pipeline. "java" posting list: {doc1}. "concurrency"/"goroutines": {doc2}.
Search Query
results = engine.search("java patterns")
Tokenize query → ["java", "patterns"]. Lookup posting lists: "java" → {doc1}, "patterns" → {doc1}. OR union = {doc1}.
Score & Rank
TF-IDF scores each candidate document
doc1: TF("java")=1/4, IDF("java")=log(2/2)=0. TF("patterns")=1/4, IDF("patterns")=log(2/2)=0. Score combines all terms. Rare terms boost score.
Return Results
[SearchResult(doc1, score=0.82, matched=["java","patterns"])]
Sorted by score descending. Paginated by offset/limit. Includes matched terms for highlighting.
Update Document
engine.updateDocument(doc1.getId(), "Java Design Patterns", "Updated content...")
Write lock acquired. Remove old terms from posting lists. Re-tokenize new content. Re-index. Release lock. Notify observers (DOCUMENT_UPDATED).

"jva" → fuzzyExpand(maxDist=2) → matches "java" (edit dist 1). "paterns" → matches "patterns" (edit dist 1). Proceed with expanded terms.

engine.setRankingStrategy(new BM25RankingStrategy())
volatile reference swap. Next search uses BM25 scoring. Zero downtime, zero code changes elsewhere.

13 — Interview Follow-up Q&A

Why an inverted index instead of a forward index?
A forward index (doc → terms) requires scanning every document for each search — O(N × T). An inverted index (term → docs) gives O(1) lookup per query term, then intersect/union the posting lists. This is the same approach used by Lucene, Elasticsearch, and Google.
How does TF-IDF handle common words like "the"?
Two layers: (1) Stop-word removal in the tokenizer eliminates ultra-common words before indexing. (2) IDF naturally penalizes common terms — if "the" appears in every document, IDF = log(N/N) = 0, contributing zero to the score. Rare, meaningful terms get high IDF.
How would you handle typos and misspellings?
Fuzzy search via Levenshtein distance. For each query term, expand to all index terms within edit distance ≤ 2. At scale, use a BK-tree or SymSpell for O(1) fuzzy lookup instead of scanning the entire vocabulary. Also: n-gram indexing (trigrams) catches many typos automatically.
Why Strategy pattern for ranking instead of just a method?
(1) Ranking algorithms change independently of the index — new algorithm = new class, zero changes to SearchIndex. (2) Swappable at runtime via volatile reference — A/B test TF-IDF vs BM25 without restart. (3) Composable — wrap BM25 with a freshness boost decorator.
How would you scale to millions of documents?
Sharding: partition the index by document ID hash across N nodes. Each shard maintains its own inverted index. A coordinator fans out the query to all shards, merges results, and re-ranks the top-K. This is exactly how Elasticsearch works (shard = Lucene index).
How do you handle document updates without blocking searches?
ReadWriteLock — searches acquire a read lock (non-blocking, concurrent). Updates acquire a write lock (exclusive). Multiple readers proceed in parallel; updates wait for readers to finish. In production: segment-based indexing (Lucene approach) where new docs go to a new segment, old segment stays readable.
How would you implement autocomplete / search suggestions?
Trie (prefix tree) for prefix matching with O(L) lookup (L = prefix length). For ranked suggestions: maintain a frequency-weighted trie or use the inverted index with prefix expansion. At scale: n-gram index + top-K aggregation.
What's the difference between BM25 and TF-IDF?
BM25 adds two improvements: (1) Term frequency saturation — TF-IDF score grows linearly with term count, BM25 uses k1 parameter to cap the benefit (diminishing returns). (2) Document length normalization — parameter b penalizes long documents that match just by having more words. Both use the same RankingStrategy interface.
How would you add phrase search ("exact match")?
Store term positions in the posting list: term → {docId: [pos1, pos2, ...]}. For phrase "design patterns", find docs containing both terms, then check if positions are consecutive (pos("patterns") = pos("design") + 1). This is a positional inverted index.
Why ConcurrentHashMap.newKeySet() for posting lists?
It's a concurrent set backed by ConcurrentHashMap — safe for concurrent add/remove/iterate without explicit synchronization. Regular HashSet would need external locking. Collections.synchronizedSet() would work but has coarser locking (entire set locked per operation).
How would you implement real-time indexing at scale?
Write-ahead log (WAL) for durability. New documents go into an in-memory buffer (memtable). When buffer is full, flush to an immutable on-disk segment. Background merge compacts segments. Searches fan out to all segments + memtable. This is the LSM-tree approach used by Lucene.
How do you handle stemming across languages?
The Tokenizer strategy interface abstracts language-specific processing. English: Porter/Snowball stemmer. Non-stemming languages (Chinese, Japanese): use n-gram tokenization or dictionary-based segmentation. Swap tokenizers per index or per field — the Strategy pattern makes this a one-liner.
